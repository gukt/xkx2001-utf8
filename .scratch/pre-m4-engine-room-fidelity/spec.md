# Pre-M4：引擎房间保真（风景 / 语义色 / 藏书 / 日间店 / 剧情门）

> 依据：[CLAUDE.md](../../CLAUDE.md) Pre-M4 窗口；[CONTEXT.md](../../CONTEXT.md)「房间风景」「语义色」「房间旗标」「藏书」「日间店铺」「剧情门」「Pre-M4 引擎房间保真」；[ADR-0001](../../docs/adr/0001-no-lpc-behavior-equivalence-verification.md)；[ADR-0010](../../docs/adr/0010-room-centric-objects-placement.md)（放置已决，**本批不得重开**）；[ADR-0011](../../docs/adr/0011-semantic-color-tokens.md)；grill 底稿 [session-notes-2026-07-21.md](session-notes-2026-07-21.md)；兄弟 effort [pre-m4-channels-spawn-quest](../pre-m4-channels-spawn-quest/)（**已关闭**）。
>
> **范围边界（供 `/to-tickets` 核对，不可扩大）**：
> - **硬门闩**：房间 `details` + 语义色（ADR-0011）+ 完整藏书（主路径 + 藏书阁禁 `practice` + 房间旗标按命令面生效）。
> - **本波必做、非硬门闩**：`day_shop` 糖衣；剧情门翰林三件套；创作者契约 v0 / `--validate` 只做加法回写。
> - **验收**：扩展官方 `m2_mvp_scene` 扬州子集（藏书阁、打铁铺日间店、翰林三件套房、带色/`details` 户外等）；不新建橱窗包。
> - **OOS**：液体灌装/饮用、防拐带、放置模型、通用脚本改出口 API、升格停机门闩、自动开 M4。见 Out of Scope。
> - 测试接缝已确认：**S1** 命令/会话面 · **S2** 场景加载+校验 · **S3** 官方扬州闭环。

## Problem Statement

M3 停机与 Pre-M4 频道/spawn/任务已关，单机可玩内核具备 Channel、房间 `objects` 与声明式 Quest，但扬州房间「看起来像」仍薄：`look` 碰不到牌子/对联/书架；内容着色要么没有、要么若照搬 LPC 会把 ANSI 宏写进 YAML；藏书阁没有可读闭环；日间店铺夜间仍可进；翰林式耗钥/无向/NPC 挡向接不上声明式内容。若不在 M4 前补这一轮引擎房间表达力，创作者契约会继续对「风景 / 色 / 读书 / 店门禁 / 剧情门」撒谎或只能透传，官方场景也无法用可玩路径证明这些能力。

## Solution

在 M4 之前插入本独立 effort，严格按硬门闩与本波必做项交付：

1. **房间风景**：房间 YAML `details`（键 → 文本）；`look` 先同房实体、再 `details`；不占 `objects`、非实体。
2. **语义色**（ADR-0011）：权威文本用 `<c:name>…</c>`；允许七色；加载/校验拒 ANSI 与 LPC 色宏；CLI 在 TTY/`--color` 映亮色 ANSI，管道与测试剥纯文本。
3. **完整藏书**：题材包书档 + TOC / 缩写 / 选书 / 付费读章 / 分页；扬州藏书阁可玩；同房禁 `practice`；房间旗标 `no_fight`/`no_steal`/`no_sleep_room` 为通用字段且按现有命令面生效。
4. **日间店铺**（非硬门闩）：`day_shop` 加载期编成夜间 `entry_guard`。
5. **剧情门**（非硬门闩）：声明式耗钥、未开无向、NPC 在场挡向；翰林后院（或同构）验收；不做通用脚本 `add_exit`/`remove_exit` 创作者面。
6. **契约与场景**：回写契约 v0 与校验；扩展官方扬州 MVP 场景承载验收，不另建橱窗包。

不做 LPC 行为等价验证。不自动开 M4。

## User Stories

### 房间风景（硬门闩）

1. 作为场景作者，我想在房间上写 `details`（键 → 描述文本，可含语义色），以便牌子/对联/书架等可被 `look` 而不进入 `objects`。
2. 作为终端玩家，我想 `look <键>` 在同房无同名实体时输出该 `details` 文本，以便观察房间陈设。
3. 作为终端玩家，我想同房实体与风景键撞名时优先看到实体，以便活人/真物盖过装饰文案。
4. 作为引擎开发者，我想风景键不占 `objects` 槽位、不可 `get`，以便不污染补刷与背包语义。
5. 作为场景作者，我不需要写 `牌子(paizi)` 括号语法；键名本身即可 `look`，以便语法保持最小。

### 语义色（硬门闩）

6. 作为场景作者，我想在 `long`/`details`/short 等文本里用 `<c:green>…</c>` 着色，且只能使用允许的七色名，以便多客户端可渲染。
7. 作为场景作者，我想 YAML 里写原始 ANSI 或 `HIG`/`NOR` 等 LPC 色宏时加载或 `--validate` 失败，以便错误格式进不了权威内容。
8. 作为场景作者，我想未知色名或未闭合 `<c:…>` 被校验拒绝，以便坏 markup 早失败。
9. 作为引擎开发者，我想命令回文等权威消息保留语义色 token（不在核心层提前染成 ANSI），以便客户端各自渲染。
10. 作为终端玩家（TTY CLI），我想默认（或 `--color`）看到亮色 ANSI 映射，以便可玩时「看得见色」。
11. 作为引擎开发者，我想管道与自动化测试默认得到剥 token 后的纯文本，以便金线不含转义噪声。
12. 作为引擎开发者，我想本波不支持嵌套色 token、背景色、闪烁/粗体 token，以便色表与解析保持最小。

### 房间旗标与藏书阁禁练（硬门闩的一部分）

13. 作为场景作者，我想在任意房间声明 `no_fight` / `no_steal` / `no_sleep_room`，以便约束可复用。
14. 作为终端玩家，我想在 `no_fight` 房间执行 `attack`/`kill` 被拒绝并得到清晰提示，以便旗标对已有命令面生效。
15. 作为引擎开发者，我想 `no_steal` / `no_sleep_room` 在尚无对应命令时可声明并校验，但不假装已禁偷/睡，以便不为旗标补齐子系统。
16. 作为终端玩家，我想在藏书阁（或声明了禁练的同构房）执行 `practice` 被拒绝（「读书还是练功」类提示），以便对齐藏书阁行为约束。

### 藏书（硬门闩）

17. 作为终端玩家，我想 `look` 书架（`details`）看到 TOC（可分页），以便知道有哪些书。
18. 作为终端玩家，我想用书名缩写解析到中文书名/书档 id，以便选书入口可记。
19. 作为终端玩家，我想 `read <id>` 选定一本书，以便后续按章阅读。
20. 作为终端玩家，我想 `read <n>`（或等价按章命令）在铜板足够时付费读出该章正文，不够时失败且不扣费，以便付费读章可测。
21. 作为终端玩家，我想长 TOC/章节正文支持分页展示，以便终端可读完长文。
22. 作为场景作者，我想书档作为题材包内文本资产被场景/包引用，以便不依赖外部 URL。
23. 作为引擎开发者，我想官方扬州藏书阁挂上完整主路径作为硬门闩验收，以便「缺藏书不算关 effort」可客观判定。

### 日间店铺（本波必做、非硬门闩）

24. 作为场景作者，我想房间写 `day_shop: true`（或等价一等字段），以便不必手写完整夜间 `entry_guard`。
25. 作为引擎开发者，我想加载期把 `day_shop` 编成依赖 `is_night` 的拒入 `entry_guard`，以便不平行搞第二套进房系统。
26. 作为终端玩家，我想夜间无法进入打铁铺（或官方日间店），白天可进，以便「晚上不开」可玩。

### 剧情门（本波必做、非硬门闩）

27. 作为场景作者，我想声明某门解锁时消耗钥匙，以便一次性钥可表达。
28. 作为终端玩家，我想未解锁时该向不可走（无出口或等价拒走），解锁后可走，以便动态出口感可测。
29. 作为场景作者，我想声明「某 NPC 在场则挡某向」，以便翰林式挡路可声明式配置。
30. 作为终端玩家，我想在官方翰林后院（或同构房）走通耗钥 + 无向 + NPC 挡向三件套，以便本波剧情门验收闭合。
31. 作为 UGC 创作者，我**不**需要、本波也**不提供**任意脚本随时 `add_exit`/`remove_exit` 的契约级 API，以便突变面保持声明式。

### 契约、校验与官方场景

32. 作为 UGC 创作者，我想创作者契约 v0 与 `--validate`/`--strict` 已知字段集包含本波新增字段（只做加法），以便契约不撒谎。
33. 作为引擎开发者，我想官方 `m2_mvp_scene` 扩展验收房（藏书阁、打铁铺日间店、翰林三件套、带语义色/`details` 的户外等），以便可玩路径而非橱窗包证明能力。
34. 作为项目架构师，我想 GAP 台账在相关能力落地后把对应「表达不了」条改判为已支持（或等价），以便文档与实现同步。
35. 作为项目架构师，我想本 effort 关闭后仍不自动开 M4，以便商业化评估独立拍板。

## Implementation Decisions

### 总则

- 硬门闩三项缺一不可关 effort；日间店与剧情门本波必做但可止损（非硬门闩）。
- 不并入 M3 停机加固；不升格为停机门闩；不重开 ADR-0010 放置模型。
- 已接受 ADR：0001、0010、0011（本批落地语义色）；0008/0009 不因本批改写。
- 领域词以 CONTEXT 为准：`details` 非 `features`/`item_desc`/`scenery`。

### 房间风景

- 房间字段名：`details`（映射：字符串键 → 描述字符串）。
- `look` 解析顺序：同房实体 → `details` 键 → 既有失败提示。
- 不做括号 id 语法；不做风景实体 / `no_get` 假物品方案。

### 语义色

- 语法与色表以 ADR-0011 为准；解析与校验在加载/`--validate` 路径执行。
- 权威消息保留 token；渲染仅在 CLI 适配层（TTY/`--color` → 亮色 ANSI；非 TTY 默认剥除）。
- 禁止嵌套；无 `NOR` 独立 token（`</c>` 复原）。

### 房间旗标

- 通用布尔字段：`no_fight`、`no_steal`、`no_sleep_room`。
- `no_fight` → 拦 `attack`/`kill`。
- 无命令的旗标：可声明+校验，行为 inert 直至有命令面。
- 藏书阁另：同房拦 `practice`（文案对齐「读书/练功」意图即可，不要求字面 LPC 复刻）。

### 藏书

- 声明式配置 + 题材包内书档资产；命令面覆盖 TOC 展示、缩写解析、选书、按章付费阅读、分页。
- 具体 YAML schema / 命令动词在 `/to-tickets` 钉死，须覆盖用户故事 17–23。
- 官方扬州藏书阁为硬门闩验收锚点；书档不必移植全部 LPC `jybooks`，但须有足够章节支撑付费读章与分页可测。

### 日间店铺

- 一等字段 `day_shop`（布尔）；加载期编译为夜间拒入的 `entry_guard`（复用 `is_night`）。
- 与手写 `entry_guard` 并存时的冲突策略在实现票钉死（推荐：二者勿叠加歧义，冲突则加载失败或明确优先级并写进契约）。

### 剧情门

- 在现有门/出口模型上做声明式扩展，不平行第二套门系统。
- 三件套必达：耗钥；未开无向（或等价不可走）；NPC 在场挡向。
- 标准门「钥匙不消耗」默认保持；耗钥为显式声明的剧情门行为。
- 不把通用运行时 `add_exit`/`remove_exit` 脚本 API 写入创作者契约作本波必达。

### 契约与场景

- 回写 `docs/creator-contract-v0.md` 与加载器已知字段；`--validate`/`--strict` 覆盖新字段与语义色校验。
- 扩展 `m2_mvp_scene.yaml`（及必要资产）承载 S3；不新建房间保真橱窗包。
- GAP 台账：本 effort 关闭时改判已落地条目。

### 建议实现顺序（非强制）

1. `details` + `look` 优先级（S1/S2）——大量文案依赖。
2. 语义色解析/校验 + CLI 渲染（S1/S2）——与风景文本交织。
3. 房间旗标 + 藏书子系统 + 官方藏书阁（S1/S2/S3）——硬门闩收口。
4. `day_shop` + 剧情门翰林三件套 + 契约/GAP 回写（S2/S3）。

## Testing Decisions

- 只测外部行为（命令回文、加载成功/失败、校验退出码、官方场景脚本化路径、CLI 着色适配层输入输出），不测私有函数结构。
- **S1 命令/会话面**（`execute_line` + 会话回文）：
  - `look`：`details` 命中；实体优先于同名风景键。
  - `no_fight` 拦攻击；藏书阁拦 `practice`。
  - 藏书：TOC/选书/付费读章/分页；余额不足不扣费。
  - 剧情门：耗钥、无向、NPC 挡向的命令层结果。
  - 权威回文含 `<c:…>` token（核心层不强制已染 ANSI）。
- **S2 场景加载 + 校验**（`load_scene`、`--validate`/`--strict`）：
  - `details` / 旗标 / `day_shop` / 藏书与剧情门声明字段被消费。
  - 非法色名、未闭合 token、ANSI/LPC 色宏 → 失败。
  - `day_shop` 夜间拒入语义（可通过编译后的 guard 或时间条件观测）。
- **S3 官方扬州闭环**（加载扩展后的 MVP 场景 + 命令序列）：
  - 藏书阁完整主路径。
  - 打铁铺（或指定日间店）夜间不可入、白天可入。
  - 翰林三件套可走通。
  - 至少一处户外/房间展示语义色与 `details`。
  - CLI 着色：TTY/`--color` vs 剥除路径的适配层测。
- 全量回归：本批合并后现有引擎测试基线保持绿。

## Out of Scope

- 液体灌装 / `drink` / 醉酒闭环；防拐带（NPC 进玩家容器）`valid_leave`。
- 重开 `placed_in`/`in_room` vs `objects`（ADR-0010）。
- 新建房间保真能力橱窗包作为唯一验收。
- 通用脚本随时改出口的契约级 API；第二套平行门系统。
- 为 `no_steal`/`no_sleep_room`「成真」而本波实装偷/睡眠命令子系统。
- 语义色嵌套、背景色、dim/bright 双轨、YAML 内 ANSI/LPC 宏。
- `features` / `scenery` / `item_desc` 作为本引擎权威字段名。
- 完整移植 LPC 全部 `jybooks` 书库或行为等价验证（ADR-0001）。
- 并入 M3 停机加固 S0/B3；升格为本批停机门闩；自动开启 M4。
- 留言板 / 创作者 Web 平台（ADR-0006 / post-mvp-backlog）。
- 走路体力全套 `cost` 重做（现有 `Terrain` 骑乘语义保持；本批不强制扩展）。
