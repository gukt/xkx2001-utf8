## Problem Statement

M3 停机与两个 Pre-M4 兄弟 effort（频道/spawn/任务、引擎房间保真）已关闭，单机可玩内核具备 Channel、房间中心 `objects` 放置、声明式 Quest、`details`/语义色/藏书/日间店/剧情门（耗钥 + 无向 + NPC 挡向）。但这些能力全部止步于"声明式静态配置"：出口拓扑在加载期就固定、房间不能在运行时因时间/步数/物理交互改变自己的通行规则。《侠客行》LPC 源码里星宿老怪一系列机关房（白玉峰挖洞崩塌、岔路随机出口、玉路刮锈拔斧推门、沙漠步数迷途、jump/climb 技能门槛、玉室日光秘道、玉厅磁力吸铁、山脚劫匪刷拦、日月洞杀令介入、柔丝索捕获）恰好是这类"运行时改世界"机关的代表集合——如果引擎永远不能表达它们，新引擎在"能不能做出星宿同构级机关"这件事上就只是停留在口头判断，创作者契约也无法诚实地说清楚"声明式 YAML 到底表达不到哪、该怎么降级"。同时 [ADR-0005](../../docs/adr/0005-m3-ugc-loop-creation-surface.md) 定下的"创作面声明式"是对 UGC 创作者的承诺，不能因为要支持这类机关就悄悄给 UGC 也开一个可执行脚本口子，否则会在没有沙箱的情况下把不可信内容的执行面焅开。

## Solution

在两个兄弟 effort 关闭之后，用一个独立的 Pre-M4 效果补齐"运行时改世界"这一档能力，严格按 [ADR-0012](../../docs/adr/0012-trusted-room-hooks-narrow-ctx.md) 收窄后的边界交付：

1. **官方可信房间钩子运行时**：新增与 `SkillBehavior` 同信任级的可信 Python 钩子协议 + 全局注册表；房间 YAML 只引用钩子 id（+ 每房间参数配置），钩子实现本身永远是引擎/题材包自带的可信代码，不是 YAML 内联脚本，也不对 UGC 开放。
2. **窄 `ctx` API**：钩子改世界只能经过一组有限方法（增删/隐藏揭示出口、一次性延时回调、房间/玩家播报、房间级自由状态读写、受限的实体移动），不得直接触达 `World` 私有结构。
3. **UGC 边界的机器可检查化**：内容包轨道（带 `manifest.yaml`）声明钩子字段一律加载失败、`--validate`/`--strict` 同路径报错；官方单文件轨道（`xingxiu_mechanics.yaml` 与未来官方切片）才允许挂钩子。
4. **十类代表机关的同构切片**：不整区移植白玉峰/岔路/沙漠等原始地图，而是每类机关各交付一条足够体现该机关"类型"的可玩验收路径，装进独立的 `xingxiu_mechanics.yaml`，不混进官方扬州 `m2_mvp_scene`。
5. **加载期小原语补一个缺口**：岔路机关本质是"加载期随机决定拓扑"，不需要运行时改图能力，作为一个新的声明式出口小原语（`random_of`）落地，不占用钩子信任边界、不进注册表。

不做 LPC 行为等价验证，不整区移植星宿地图，不自动开 M4，本批不引入 UGC 可执行沙箱。

## User Stories

### 信任边界与钩子注册（横切，ADR-0012 落地）

1. 作为引擎开发者，我想有一个与 `SkillBehavior` 同构的可信钩子协议 + 全局注册表，以便房间机关的接线方式与既有招式钩子保持同一套心智模型。
2. 作为官方 / 题材包开发者，我想用可信 Python 代码实现某个钩子并注册一个 id，以便房间 YAML 只需引用该 id 而不重复写逻辑。
3. 作为场景作者（官方单文件轨道），我想在房间上声明引用某个已注册钩子 id（可带每房间参数），以便同一钩子实现复用在多个房间。
4. 作为引擎开发者，我想钩子引用的 id 在注册表里查不到时加载失败（不区分"拼写错"与"未注册"两种诊断），以便未注册即视为不可信/不存在，统一 fail-closed。
5. 作为 UGC 创作者，我**不**能在内容包（带 `manifest.yaml`）里声明任何钩子字段，声明即加载失败，以便声明式创作面的承诺不被本批悄悄打开一个可执行口子。
6. 作为引擎开发者，我想 `--validate` / `--strict` 对内容包轨道的钩子字段与非严格加载路径给出一致的失败判定（不是仅警告），以便"UGC 禁钩子"是信任边界而非风格建议。
7. 作为引擎开发者，我想钩子实现只能通过窄 `ctx` 方法改世界（出口增删/隐藏揭示、延时回调、播报、房间级自由状态读写、受限实体移动），不直接拿到 `World` 或私有组件存储，以便"经 ctx 改世界"这条约束在接口层就成立，不靠人肉 code review 兜底。
8. 作为引擎开发者，我想 `ctx` 暴露的只读信息只到"当前房间 id / 当前玩家 id / 时段等既有只读快照字段"，不透出其他实体的私有组件，以便钩子作者拿到的上下文和 `CombatContext` 一样"够用但不多给"。
9. 作为引擎开发者，我想钩子异常按 fail-fast 传播（不静默吞掉），以便机关 bug 在开发期就暴露，而不是悄悄不生效。

### 动态出口 + 时限崩塌（机关 #1，白玉峰灵感）

10. 作为场景作者，我想给某个房间挂一个"挖洞开新出口、超时未完成则崩塌"的钩子，以便表达"限时开路"体验。
11. 作为终端玩家，我想执行触发动作后房间新增一个此前不存在的出口，以便"挖出来的路"是真实可走的出口而不是文案。
12. 作为终端玩家，我想超过时限后该出口消失（或触发等价的"崩塌"后果），以便"过期作废"可观察。
13. 作为引擎开发者，我想时限判定复用现有 tick 心跳而不新建平行的通用调度服务，以便不为"目前只有两三个机关用到"的能力过度设计。

### 加载期随机出口（机关 #2，岔路灵感；声明式小原语，非钩子）

14. 作为场景作者，我想给某个方向的出口声明"从一组候选目标随机挑一个"，以便表达岔路效果而不需要运行时改图能力。
15. 作为引擎开发者，我想这个随机选择只发生在加载期一次，选定后落地为普通出口（后续行为与静态出口一致），以便它不占用钩子信任边界、不需要进注册表、不带运行时副作用。
16. 作为终端玩家，我想同一张地图重新加载后岔路结果可能不同（体现"随机"），但同一次运行内该出口结果是固定的（不会走到一半又变），以便体验和心智模型一致。

### 多步房间状态机（机关 #3，玉路刮锈拔斧推门灵感）

17. 作为场景作者，我想给房间挂一个"多步顺序动作才能改变房间状态"的钩子（如先做动作 A 才能做动作 B，最后一步才开新出口/门），以便表达"步骤不能跳"的机关体验。
18. 作为终端玩家，我想跳过前置步骤直接做最后一步时被拒绝并有提示，以便"步骤顺序"是真实约束而非仅文案暗示。
19. 作为终端玩家，我想按正确顺序走完全部步骤后房间状态确实改变（如新增/揭示出口），以便"多步机关"闭环可玩。
20. 作为引擎开发者，我想多步进度存在房间级自由状态里（钩子自己定义存什么，引擎不假设结构），以便不同机关的状态形状不互相绑定死。

### `valid_leave` 步数迷途（机关 #4，沙漠灵感）

21. 作为场景作者，我想给某片房间（或单房间）挂一个"离开前必须先在房间内待够/走够若干步"的钩子，以便表达"迷路打转"体验。
22. 作为终端玩家，我想在步数不够时尝试离开被拒绝（或被重新绕回原地等价效果）并有提示，以便"迷途"是可感知的约束。
23. 作为终端玩家，我想步数够了之后可以正常离开，以便迷途机关有明确的退出条件。
24. 作为引擎开发者，我想本批迷途用最小切片验收（不必复刻 LPC 沙漠原版扣水等资源消耗细节），以便机关同构而不是资源系统移植。

### jump / climb 技能门槛（机关 #5，白玉峰 jump、天路灵感）

25. 作为场景作者，我想给某个出口/方向挂一个"需要达到某项技能等级才能通过"的钩子（如 jump / climb），以便表达技能门槛类地形。
26. 作为终端玩家，我想技能等级不够时尝试通过被拒绝并有清晰提示，以便门槛判定可测。
27. 作为终端玩家，我想技能等级够时可以正常通过，以便门槛机关有明确的达成路径。
28. 作为引擎开发者，我想技能等级判定复用既有的角色技能等级读取与条件求值能力，不新造一套判断逻辑，以便与已有的技能/条件体系保持一致。

### 时段耦合秘道（机关 #6，玉室日光灵感）

29. 作为场景作者，我想给房间挂一个"仅在特定时段（如白天）揭示隐藏出口"的钩子，以便表达"日光开密道"体验。
30. 作为终端玩家，我想在非对应时段看不到、走不了该隐藏出口，以便时段约束真实生效。
31. 作为终端玩家，我想到了对应时段后该出口被揭示且可走，以便机关有明确的开启条件。
32. 作为引擎开发者，我想时段判定复用既有的昼夜状态读取能力，出口揭示复用既有的"隐藏出口解锁后迁入可见出口"机制，不新建平行的门/出口系统，以便与剧情门已有能力保持一致心智模型。

### 入室磁力吸铁（机关 #7，玉厅灵感）

33. 作为场景作者，我想给房间挂一个"携带铁质物品进入时触发效果"的钩子，以便表达磁力机关体验。
34. 作为终端玩家，我想携带命中标签的物品进房间时收到磁力效果的播报（本批至少做到可观察的播报级效果，具体是否强制卸除留 `/to-tickets` 钉死强度），以便磁力机关可玩可测。
35. 作为终端玩家，我想不携带命中物品进房间时无异常反应，以便机关只对特定条件生效。

### 劫匪刷拦（机关 #8，山脚 / 星宿路灵感）

36. 作为场景作者，我想给房间挂一个"按条件生成/激活拦路 NPC"的钩子，以便表达劫匪刷拦体验。
37. 作为终端玩家，我想在触发条件满足时该房间出现拦路 NPC 并有播报，以便劫匪出现可观察。
38. 作为终端玩家，我想击退/击杀拦路 NPC 后通行恢复正常，以便机关有明确的解除条件。
39. 作为引擎开发者，我想拦路 NPC 的生成、交战、死亡复用既有的 NPC 补刷与战斗系统，钩子只负责"何时生成/何时解除"的编排，不新建平行战斗或补刷系统。

### 杀令介入（机关 #9，简化，日月洞灵感）

40. 作为场景作者，我想给房间挂一个"满足条件时触发一次简化的敌意介入"的钩子（如特定阵营玩家进房触发一次性遭遇），以便表达"杀令"式介入体验，而不需要完整的通缉/声望系统。
41. 作为终端玩家，我想满足条件进房时触发一次可观察的介入事件（播报 + 可选战斗触发），以便机关同构可玩。
42. 作为引擎开发者，我想本批不建通缉状态机 / 多阵营声望系统，条件判定复用既有的阵营组件读取能力，以便"简化"名副其实。

### 柔丝索跨玩家捕获（机关 #10，柔丝索 + 柔丝室灵感）

43. 作为终端玩家（施法者），我想对同房间另一名玩家使用柔丝索类招式命中后，把目标玩家移动到指定房间（或等价的"捕获"效果），以便表达跨玩家捕获机关。
44. 作为终端玩家（被捕获者），我想被捕获后确实出现在目标房间（可观察的位置变化），以便捕获效果真实生效而非仅播报。
45. 作为测试作者，我想用同一 `World` 内的双 `PlayerSession` seam（复用频道/spawn/任务 effort 落地的能力）驱动"捕获者 + 被捕获者"两个会话，以便柔丝索机关不需要真实联网就能被测试覆盖。
46. 作为引擎开发者，我想跨玩家的位置变更走窄 `ctx` 提供的受限实体移动方法（内部复用既有的"改位置 + 分发进出房间事件"路径），不新建一条通用远程传送命令面，以便本机关的改世界能力仍收在窄 `ctx` 范围内。

### `xingxiu_mechanics` 切片与契约边界

47. 作为项目架构师，我想十类机关的验收资产是独立的 `engine/data/xingxiu_mechanics.yaml`（不带 `manifest.yaml`，走官方单文件轨道），以便机关切片不与官方扬州 `m2_mvp_scene` 混成同一份验收，也不需要为它伪装成一个内容包。
48. 作为项目架构师，我想每类机关至少有一条房间/路径落在该切片包里，覆盖用户故事 10–46 列出的可玩验收点，以便"档 B：原语切片 + 代表机关全套"不是空话。
49. 作为 UGC 创作者，我想创作者契约文档明确写出"`hooks` 字段为官方轨专属、不在创作者契约的加法承诺范围内、UGC 引用即失败"，以便不误以为这是一个可用的声明式字段。
50. 作为项目架构师，我想 GAP 台账新增一行，说明动态出口/多步状态机/迷途/技能门槛/秘道/磁力/劫匪/杀令/柔丝索这类"运行时改世界"机关目前只能靠官方可信房间钩子表达，纯声明式 YAML 独立表达不到，且明示"不做整区星宿移植、不等价 LPC 行为"，以便文档不撒谎也不越界宣称。
51. 作为项目架构师，我想本 effort 关闭后仍不自动开 M4，以便商业化评估独立拍板。

## Implementation Decisions

### 总则

- 只交付 [session-notes-2026-07-22.md](session-notes-2026-07-22.md) 拍板的档 B 十类机关代表切片；不整区移植星宿地图；不做 LPC 行为等价验证（ADR-0001）。
- 信任边界与运行时形态以 [ADR-0012](../../docs/adr/0012-trusted-room-hooks-narrow-ctx.md) 为准：T1 仅官方 / 题材包可信 Python；R1 可信模块 + 窄 `ctx`；本批不引入 UGC RestrictedPython 沙箱。
- 不重开放置模型（ADR-0010）、不重开单进程单 World（ADR-0009）；不因本批引入多 World 或分布式调度。
- 领域词以 CONTEXT 为准：钩子叫「房间钩子」，验收包叫 `xingxiu_mechanics`（非 `m2_xingxiu_mechanics`）。

### 房间钩子协议与注册表

- 新增与 `SkillBehavior` 同信任级、同注册模式的可信钩子协议 + 全局注册表：按 `hook_id` 注册 Python 侧实现，房间 YAML 只引用 `hook_id`（可带每房间参数映射）。
- 钩子协议方法集合按"房间生命周期事件点"设计，而非照搬 `SkillBehavior` 的固定三方法签名——十类机关的触发点差异太大（进房 / 离房 / 心跳到期 / 新命令动词多步序列），协议方法全部可选实现，未实现的钩子在对应事件点上视为不参与，不强制每个钩子实现全部方法。
- 引用校验：`hook_id` 在注册表里查不到时加载 / `--validate` 失败，不区分"拼写错"与"未注册"两种诊断（未注册即不可信，统一 fail-closed）。
- 钩子挂载时机：场景加载完成后统一遍历所有声明了钩子引用的房间，把对应实现按房间 id 订阅到共享的既有事件点（进房 / 离房 / 心跳）上；不用的事件点不订阅，避免所有房间都被无关钩子轮询到。

### 窄 `ctx` API

- `ctx` 是一个只服务钩子实现的轻量外观对象，构造时绑定"当前房间 / 当前触发实体"上下文；不是 `World` 本身，也不透出私有组件存储。
- 方法覆盖范围：出口的增 / 删 / 隐藏 / 揭示；一次性延时回调登记（内部落在房间级自由状态里，由钩子自己的心跳方法在到期时自查执行，不新建引擎级通用调度服务——理由与 [ADR-0007](../../docs/adr/0007-effect-lifecycle-deferred-from-m2-m3-stop.md) Effect 生命周期延后同一取舍：等未来有多个子系统都要用再一并抽象）；房间广播 / 单玩家消息；房间级自由状态读写（钩子自己定义存什么，引擎不假设结构，供多步状态机 / 迷途步数计数 / 到期戳等自行编码）；受限的实体位置变更（复用既有"改位置 + 分发进出房间事件"路径，供柔丝索捕获等跨实体机关使用，不新建通用远程传送命令面）。
- 只读信息只到房间 id / 触发实体 id / 时段等既有只读快照字段，不透出其他实体的私有组件；钩子需要更多判定（技能等级、阵营、物品标签）时复用既有的对应读取 / 条件求值能力，不重新发明。

### 加载期随机出口小原语（岔路，机关 #2）

- 作为出口层面的新声明式字段，加载期从候选目标里选定一个，落地为普通出口；选定后运行时表现与静态出口完全一致。
- 不进钩子注册表、不占用信任边界、无运行时副作用；与"钩子改世界"的十类机关中的其余九类明确区分开，避免把它误算进信任边界讨论。

### 命令动词扩展（dig / jump / climb 等新增动作）

- 十类机关里涉及新命令动词的（挖洞、跳跃、攀爬、多步序列动作等），按现有命令注册机制新增动词处理函数；这些函数属于引擎 / 题材包自带的可信代码，不是 YAML 可声明的行为。
- 新动词在没有相关钩子的房间里执行时返回统一的"这里不能这么做"类拒绝提示，而不是"未知命令"，以便同一动词可能在部分房间生效、部分房间不生效。
- 新动词内部通过窄 `ctx` 完成实际的世界变更，不直接操作组件存储。

### UGC 边界的机器可检查化

- 内容包轨道（带 `manifest.yaml`）与官方单文件轨道共享同一套场景字段解析，但钩子引用字段只在官方单文件轨道允许；内容包轨道声明该字段一律失败——`--validate` / `--strict` 与非严格加载路径给出一致的失败判定，不是仅警告级别的透传处理（这是信任边界而非"未消费字段"场景，不适用创作者契约"加法只警告"的一般规则）。
- 创作者契约文档为此新增一节，明确写"钩子引用字段为官方轨专属、不在创作者契约的加法承诺范围内、UGC 内容包引用即失败"，避免被误读为一个可用的声明式字段。

### `xingxiu_mechanics` 切片包

- 独立场景资产，不带 `manifest.yaml`（官方单文件轨道），不与官方扬州 `m2_mvp_scene` 混成同一份文件；不因本批把星宿机关塞进扬州验收。
- 十类机关每类至少一条可玩验收路径；房间拓扑同构即可，不追求整区移植，也不追求房间数量或美术级描述与 LPC 原图一致。
- 具体房间数量、每类机关的参数字段形状、命令动词的确切拼写与提示文案，在 `/to-tickets` 钉死，不在本 spec 固化。

### 建议实现顺序（非强制）

1. 房间钩子协议 + 注册表 + 窄 `ctx` 基础设施（S0）——后续九类机关（岔路随机出口除外）都依赖它。
2. 动态出口 + 时限崩塌、加载期 `random_of`（S0/S1/S2）——验证"改出口"与"声明式小原语"两条路径分别走通。
3. 多步房间状态机、`valid_leave` 步数迷途、jump/climb 技能门槛（S0/S1）——验证房间级自由状态与既有技能/条件读取的接线。
4. 时段耦合秘道、入室磁力吸铁（S0/S1）——验证与既有昼夜状态、物品标签、隐藏出口机制的复用。
5. 劫匪刷拦、杀令介入（简化）（S0/S1）——验证与既有 NPC 补刷/战斗/阵营能力的编排式复用。
6. 柔丝索跨玩家捕获（S0/S1，依赖双 `PlayerSession` seam）——验证受限实体移动 `ctx` 方法。
7. UGC 边界校验 + `xingxiu_mechanics.yaml` 十类机关整合 + 契约/GAP 回写（S2/S3）。

## Testing Decisions

- 只测外部行为（钩子 `ctx` 调用后的可观察副作用、命令回文、加载成功/失败、校验退出码、官方切片脚本化路径），不测私有函数结构。
- **S0 钩子直调层**（新增；类比现有 `SkillBehavior` 直接构造 `CombatContext` + 直调 behavior 对象的测试写法）：
  - 直接构造窄 `ctx` + 调用某个已注册钩子实现的各生命周期方法，断言 `ctx.add_exit` / `remove_exit` / `schedule` / `message_room` / `move_entity` / 房间状态读写等副作用，不经命令解析。
  - 未注册 `hook_id` 的引用行为、钩子异常传播行为。
- **S1 命令/会话面**（`execute_line` + 会话回文）：
  - 十类机关各自的命令层可观察结果：新增出口的 `go`、随机出口选定后的稳定性、多步序列动作的顺序校验、迷途拒绝离开的提示、jump/climb 门槛拒绝与放行、时段秘道揭示前后的可见性、磁力播报、劫匪出现与解除、杀令简化介入、柔丝索捕获后双方位置变化。
  - 柔丝索复用同 `World` 双 `PlayerSession` seam：一个会话作为施法者、另一个作为被捕获者，断言后者 `Position` 随捕获动作改变。
- **S2 场景加载 + 校验**（`load_scene`、`--validate`/`--strict`）：
  - 钩子引用字段的官方单文件轨道解析成功；内容包轨道声明同字段解析/校验失败。
  - 未注册 `hook_id` 引用失败；`random_of` 小原语加载期落地为普通出口。
- **S3 官方 `xingxiu_mechanics` 切片闭环**（加载切片包 + 命令序列）：
  - 十类机关各至少一条端到端可玩路径全部跑通。
  - 与官方扬州 `m2_mvp_scene` 的加载/测试互不干扰（两个场景文件独立）。
- 全量回归：本批合并后现有引擎测试基线保持绿。

## Out of Scope

- 整区星宿地图移植（白玉峰全山、沙漠全域、日月洞全图等）；LPC 行为等价验证（ADR-0001）。
- UGC 可执行内容层 / RestrictedPython 沙箱（本批不上，仍是 T1/R1 边界内的官方轨专属能力）。
- 液体灌装 / 饮用 / 醉酒闭环；防拐带（NPC 进玩家容器）`valid_leave` 语义的完整版本（本批只交付步数迷途这一种 `valid_leave` 切片）。
- 雪岭比武寄存全套；空能力橱窗包（`xingxiu_mechanics` 是同构可玩切片，不是能力展示柜）。
- 通缉 / 声望 / 多阵营 AI 决策系统（杀令介入本批只做简化的一次性触发编排）。
- 引擎级通用一次性延时调度服务、通用远程传送命令面（本批用房间级自由状态自查 + 受限实体移动方法顶住需求，不新建这两类通用设施）。
- 把 `xingxiu_mechanics` 并入官方扬州 `m2_mvp_scene`，或反过来把扬州验收房迁入本切片包。
- 升格为 M3 停机门闩；自动开启 M4；商业化 / UGC 创作者平台相关能力（post-mvp-backlog 范畴）。
- 重开放置模型（ADR-0010）、重开单进程单 World（ADR-0009）。
