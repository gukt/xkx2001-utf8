# Pre-M4 房间钩子 / 星宿机制：拆票分析笔记（`/to-tickets`）

> 本文件记录对 [spec.md](spec.md) 的拆分逻辑、依据，以及为消除实现歧义所做的决策。11 张票已发布在 [issues/](issues/)（`01`–`11`，编号即依赖顺序）。执行手册见 [implement-plan.md](implement-plan.md)。
>
> **核心校准**：票编号与 spec 用户故事分组严格一一对应（US1–9→`01`；US10–16→`02`；US17–20→`03`；US21–24→`04`；US25–28→`05`；US29–32→`06`；US33–35→`07`；US36–39→`08`；US40–42→`09`；US43–46→`10`；US47–51→`11`）。信任边界与十类机关全部按 [ADR-0012](../../docs/adr/0012-trusted-room-hooks-narrow-ctx.md) 收窄形态交付；不整区移植星宿；不做 LPC 行为等价（ADR-0001）；关完不自动开 M4。

## 勘察方法

- **spec**：[spec.md](spec.md) 全文（范围边界、US1–51、Implementation / Testing / OOS）。
- **上游**：[PROGRESS.md](../../PROGRESS.md)、[CONTEXT.md](../../CONTEXT.md)、[ADR-0012](../../docs/adr/0012-trusted-room-hooks-narrow-ctx.md)、grill 底稿 [session-notes-2026-07-22.md](session-notes-2026-07-22.md)。
- **兄弟批 precedent**：[pre-m4-engine-room-fidelity](../pre-m4-engine-room-fidelity/)（7 票 / 3 Wave / implement-plan 形状，且本批直接复用其 `Exits`/`HiddenExits`/`block_exits`/`entry_guard`/`is_day`/`is_night`/`ItemTags` 等落地能力）与 [pre-m4-channels-spawn-quest](../pre-m4-channels-spawn-quest/)（双 `PlayerSession` seam，柔丝索复用）。
- **代码接缝（拆票时点，见 explore 子代理报告）**：
  - `SkillBehavior` 协议 + 模块级注册表（`engine/src/openmud/skills.py`）是本批「同构协议+注册表+窄 ctx」的直接仿照对象；`CombatContext` 是「窄只读 ctx」形状的先例。
  - `EventBus`（`events.py`）已有 `ON_BEFORE_ENTER_ROOM`（可否决）、`ON_ENTER_ROOM`/`ON_LEAVE_ROOM`（dispatch）、`ON_TICK`；**离房目前只 dispatch、不可否决**——`valid_leave` 迷途需要新增一个否决点，工作量不与其他机关同质，值得单独成票（`04`）。
  - `Exits`/`HiddenExits` 组件与 `unlock` 揭示迁移路径（`commands._cmd_unlock`）可直接复用；**没有** `add_exit`/`remove_exit` 封装函数，惯例是直接改 dict（先例 `ferry.py`）——窄 `ctx` 需要包一层。
  - **没有房间级自由 KV 状态组件**——多步状态机（`03`）、迷途步数（`04`）、崩塌延时戳（`02`）都要用，是本批唯一「完全新建」的数据结构，因此归入基建票 `01`，不重复建。
  - `scene_loader` 已知字段 + `--validate`/`--strict` + `pack_manifest is not None` 双轨判定已有，只需新增「钩子字段」与「UGC 轨拒绝」两条规则,同批落地不留窗口期。
  - `SkillLevels`、`conditions.evaluate`/`is_day`/`is_night`、`ItemTags`、`spawn_scan`/`attach_ai_system`/`try_engage`、`Faction`、双 `PlayerSession` seam（`test_per_session_mailbox.py` 模式）均可直接复用，不新建平行设施。

## 拆分原则

1. **票编号与 spec US 分组一一对应，不重新发明分组。** spec 已经把 51 条用户故事按「信任边界与注册」+「十类机关」+「切片与契约边界」分成 11 组，且组内故事天然构成一个可玩闭环；直接沿用比另起炉灶的分法更不容易漏验收点。
2. **基建单独成票，且本身就是可验证的垂直切片。** `01` 不依赖任何机关即可用 S0（直调 `ctx` + 断言副作用）与一个测试专用哑钩子（不进 `xingxiu_mechanics.yaml`）验证「注册表 + 挂载 + 窄 ctx + UGC 拒绝 + fail-closed + fail-fast」全链路，不是「先攒骨架、后面才补测试」的假垂直切片。
3. **机关粒度按 spec 建议顺序做「合并 vs 拆开」的例外判断，不整批照抄。** spec「建议实现顺序」把 #1+#2、#3+#4+#5、#6+#7、#8+#9 分别列为同一步骤，但拆票时按「每类机关是否需要新增命令动词/新事件点」重新核算工作量：
   - `01`+`02`（机关 #1+#2）按 spec 原样合并——两者都在验证「改出口」的两条不同路径（钩子改图 vs 加载期声明式随机），体量相当，且 `02` 顺带创建 `xingxiu_mechanics.yaml` 文件本体，合并更自然。
   - 机关 `03`/`04`/`05`（spec 原建议合并）**拆开**：`03`（多步状态机）要新增 2–3 个命令动词；`04`（迷途）要新增一个此前不存在的「离房否决」事件点，工程性质与另外两个不同质；`05`（jump/climb）相对独立、体量小。三者合并会让单票同时扛「新命令面 + 新事件点 + 新判据」三种异质工作量，超出「单票适配单个 fresh context window」的量纲，故拆成 `03`/`04`/`05` 三张独立票。
   - 机关 `06`/`07`（时段秘道 + 磁力吸铁，spec 原建议合并）**拆开**：虽然验证目标相近（复用既有昼夜/标签/隐藏出口），但一个改「出口可见性」、一个改「进房播报副作用」，落点房间与命令面互不相关，拆开不增加协调成本，反而让每票的 S1 验收更聚焦。
   - 机关 `08`/`09`（劫匪刷拦 + 杀令介入，spec 原建议合并）**拆开**：理由同上——都编排既有 NPC/战斗/阵营能力，但触发条件与验收房间独立，拆开更利于按 spec「治理止损线」单独核算工作量，不因为绑在一起而被迫一次性重估两个机关。
   - 机关 `10`（柔丝索）保持独立成票——它不是 `RoomHook`（房间生命周期钩子），而是一个新 `SkillBehavior`（招式命中回调）直调 `01` 建的「受限实体移动」方法本体；把它并进任何一张房间钩子票都会混淆两种触发协议，必须单独说清楚。
4. **收口票依赖全部机关票，不提前拆分。** `11` 需要核对十类机关全部落地才能做「整合确认 + 契约/GAP 措辞」，依赖 `02`–`10` 全部完成；UGC 拒绝规则的机器可检查化已经在 `01` 随字段一起落地，`11` 只做复核与文档回写，不是补代码。
5. **Prefactor 克制。** 不为「十类机关」另开一个通用「机关配置 DSL」票——每类机关的房间参数形状在各自票内按需定义，钩子协议本身只规定生命周期方法签名，不规定参数 schema（spec Implementation Decisions 已明确「协议方法全部可选实现」）。

## 关键设计决策（实现时直接采纳）

1. **UGC 拒绝规则随字段一起在 `01` 落地，不留窗口期。** 一旦 `hooks` 字段被 `scene_loader` 认识，同一票就必须让内容包轨（`pack_manifest is not None`）声明该字段时 fail-closed，理由：信任边界是本批的核心交付物，留一个「字段已解析但校验滞后」的中间态会在多票并行开发期造成可被误用的窗口（即使只是开发期风险）。
2. **柔丝索（`10`）复用的是「受限实体移动」方法本体，不经过 `RoomHook` 注册表。** `01` 需要把该方法实现成一个可被 `RoomHook` 窄 `ctx` 与 `SkillBehavior` 双方共享调用的独立函数/方法（内部复用既有「改位置 + 分发进出房间事件」路径），而不是只暴露在 `ctx` 对象上——否则 `10` 无法在不假装自己是房间钩子的前提下拿到同一能力。
3. **劫匪刷拦（`08`）优先复用房间保真已交付的 `block_exits`（NPC 在场挡向），钩子只负责「何时生成/何时清除」的编排。** 不重新发明一套「机关专属阻挡」字段——`block_exits` 已经是「某 NPC 模板在场则挡某向」的通用声明式能力，劫匪刷拦钩子只需要在满足条件时把对应 NPC 蓝图 spawn 到房间（复用 `spawn_scan`/蓝图机制），挡向语义自动成立；解除靠 NPC 死亡/despawn 后 `block_exits` 天然失效，不需要钩子额外清理。
4. **迷途（`04`）新增的「离房否决」事件点是本批唯一的引擎级事件总线小扩展，且只服务这一个机关，不误判为「通用设施」。** 与 [ADR-0007](../../docs/adr/0007-effect-lifecycle-deferred-from-m2-m3-stop.md)「等未来多个子系统都要用再一并抽象」同一取舍——先做成 `ON_BEFORE_LEAVE_ROOM`（可否决，镜像既有 `ON_BEFORE_ENTER_ROOM`），不做成钩子协议里的通用方法族。
5. **杀令介入（`09`）的「一次性」语义用房间自由状态记一个「本次在场期间是否已触发」的标记位，不引入通缉/声望持久状态。** 理由：spec 明确本批不建通缉状态机/多阵营声望，最小切片只需要「同一次进房只介入一次」这个防重复触发的局部状态，房间自由状态组件（`01`）已经够用。
6. **`xingxiu_mechanics.yaml` 由 `02` 创建文件本体，后续机关票逐票追加房间，不建统一「预留全部房间骨架再填内容」的中间态。** 与房间保真兄弟批扩展 `m2_mvp_scene.yaml` 的做法一致；每票落地时若与其他并行票同时改这个文件，由 `/implement` 时序合并处理，不作为票据阻塞边。
7. **`random_of`（机关 #2，随 `02` 一起交付）技术上不依赖 `01` 的钩子基建**，但因为它和机关 #1 共享同一张验收文件、且 spec 把两者列为同一验证步骤，仍合并进 `02` 一起交付，不单独拆票——避免为一个纯声明式小原语开一张几乎没有工作量的票。
