# M1 — 核心引擎骨架：空场景 + 命令-移动-存档最小闭环

> 依据：[CLAUDE.md 架构不变量](../../CLAUDE.md) 全 8 条 + [.scratch/mvp-scope/](../mvp-scope/) 9 票决议，尤其 [04-engine-code-disposition](../mvp-scope/issues/04-engine-code-disposition.md)（整体重写，旧代码只作参考）、[07-governance-cost-tracking](../mvp-scope/issues/07-governance-cost-tracking.md)（M1 里程碑定义）、[08](../mvp-scope/issues/08-subsystem-classification-research.md)/[09](../mvp-scope/issues/09-subsystem-classification-confirm.md)（子系统四档归类）。范围与 seam 已与用户核对确认（2026-07-18，见本 session 对话）。

## Problem Statement

项目已完成从"全量复刻《侠客行》"到"题材无关核心引擎 + UGC 创作层 + 官方轻量题材包"的目标重设，`engine/` 现有约 45k 行代码已拍板整体重写（旧代码只作房间/对象模型骨架形状与 ECS 相关教训的参考，不是重写起点）。但重写尚未有任何落地代码——当前没有一个可运行的、体现新架构不变量的最小引擎骨架，无法验证"组件化对象模型 + 命令调度 + 存档"这套核心机制在新目标下是否成立，也无法作为后续里程碑（M2 单场景可玩、M3 UGC 闭环）的地基。

## Solution

构建一个不含任何具体题材内容（武侠数值、战斗、技能、NPC）的最小引擎骨架：由几个通过出口互联的空房间组成的场景，玩家可以通过 CLI 输入命令移动、查看、拾取/丢弃物品、查看物品栏；引擎周期性心跳驱动存档，退出时立即存档，重启时能从存档恢复到崩溃/退出前的状态。骨架内建的对象模型（组件化 Entity/Component）、命令调度机制、存档机制，形状上均可被 M2（战斗/技能/NPC 等题材内容）直接复用扩展，而不需要推翻重做。

## User Stories

1. 作为终端玩家，我想通过 CLI 输入 `go <方向>` 移动到相邻房间，以便探索由多个房间组成的场景。
2. 作为终端玩家，我想在尝试往一个不存在出口的方向移动时收到明确提示，而不是让程序崩溃或静默无反应。
3. 作为终端玩家，我想输入 `look` 查看当前房间的简述、详细描述、地面物品与可用出口列表，以便判断该往哪走、有什么可拿。
4. 作为终端玩家，我想移动到新房间后自动看到该房间的描述，不需要额外再输入一次 `look`。
5. 作为终端玩家，我想输入 `take <物品>` 把地面上的物品捡到自己的物品栏。
6. 作为终端玩家，我想输入 `drop <物品>` 把物品栏里的物品放到当前房间地面。
7. 作为终端玩家，我想尝试拾取/丢弃一个不存在的物品时收到明确提示，而不是崩溃。
8. 作为终端玩家，我想输入 `inventory/i` 查看自己当前持有的所有物品。
9. 作为终端玩家，我想输入一个引擎不认识的命令时收到"未知命令"提示 + 简短帮助指引，而不是程序异常退出。
10. 作为终端玩家，我想输入 `help/h` 查看当前骨架支持的全部命令列表。
11. 作为终端玩家，我想输入 `quit` 退出游戏，并确信我的当前状态（所在房间、物品栏）已经被保存。
12. 作为终端玩家，我想在异常退出（未执行 `quit`，如直接 Ctrl+C 或进程被杀）后重新启动游戏，能够从上一次周期性自动存档的状态继续，而不是永远丢失进度或读到损坏数据。
13. 作为运维/测试人员，我想验证存档写入过程中途被中断（模拟进程崩溃）不会损坏已有的上一份完整存档，以便确认崩溃恢复的耐久性承诺成立。
14. 作为运维/测试人员，我想验证存档文件损坏（如手动截断/写坏一个存档文件）时引擎重启会跳过该损坏文件并记录警告，而不是直接崩溃拒绝启动。
15. 作为引擎开发者，我想对象模型（Entity/Component）在 M1 阶段只包含移动骨架必需的最小组件集合（身份、位置、物品容器、出口表、可开合/上锁状态、描述——按"是否被多种实体类型复用"拆分，不是按字段数量拆分），以便 M2 引入战斗/技能/状态等新组件时，只需新增组件类型和处理逻辑，不需要重新设计存储机制或牵动已有组件。
16. 作为引擎开发者，我想命令调度机制以"动词 -> 处理函数"的注册表形式组织，以便 M2 新增战斗/技能等命令时只需注册新处理函数，不需要改动调度框架本身。
17. 作为引擎开发者，我想命令处理函数只响应玩家的外部输入意图，不承担"世界状态随时间自然演化"的职责（如未来的生命恢复、NPC 行为），以便后续引入心跳驱动的系统（tick-driven system）时职责边界清晰，不会与命令路径产生冲突或重复触发。
18. 作为引擎开发者，我想心跳循环（tick loop）作为一个独立机制先行搭建，M1 阶段只承载"周期性触发自动存档"这一件事，以便 M2 引入战斗回合结算、生命自然恢复、NPC 行为决策时，直接挂载到已有心跳循环上，不需要重新设计驱动机制。
19. 作为引擎开发者，我想存档机制的持久化边界定义为"崩溃恢复级耐久"（而非"每次写入即权威提交"），内存态是权威状态，以便后续所有游戏逻辑可以直接操作内存对象，不需要为每次状态变更单独考虑持久化往返延迟。
20. 作为引擎开发者，我想房间的出口表在运行时可以被动态增删（而不是启动后固定不变的常量映射），并且每个出口可以选配一个门状态（开/关/上锁，配对钥匙物品），以便"渡口渡船""桥梁坍塌""密道开启"这类地理连接随玩家动作改变的场景内容，未来直接复用这套出口/门机制就能实现，不需要另起一套连接模型。
21. 作为终端玩家，我想遇到关着的门无法直接通过、遇到锁着的门被提示需要钥匙，用 `open`/`close`/`knock`/`unlock` 操作门，成功解锁后才能通行。
22. 作为终端玩家，我想某些出口在游戏过程中会被动态开启或关闭（如触发某个动作后多出一条路/封住一条路），以便验证"出口表可运行时变化"这条机制确实生效，而不只是纸面设计。
23. 作为终端玩家，我想输入命令时可以用简短别名（如 `i` 代替 `inventory`、`l` 代替 `look`、`n` 代替 `go north`），以及用物品/出口的别名而不必输入完全规范的名称，以便交互更顺手，不需要记住每个东西的"标准写法"。
24. 作为引擎开发者，我想命令别名与目标对象别名的解析逻辑是通用、与具体命令/对象类型无关的共享机制（而不是给每个命令、每种目标各写一套特例判断），以便新增命令或新增可被引用的对象类型时，别名支持是"顺带就有"的，不需要重新实现一遍。
25. 作为引擎开发者，我想把"把一行原始文本理解成明确的动作意图"这件事，和"执行这个意图"这件事解耦成两个独立阶段，通过一个稳定的中间表示衔接，以便未来可以在"确定性解析失败"时，接入第二个解析器（用 AI/LLM 理解更自由的自然语言输入）作为兜底，而不需要改动执行层或已有的确定性解析器。
26. 作为引擎开发者，我想 M1 阶段只实现确定性的（基于命令表/别名表/简单语法规则的）解析器，但解析失败时返回的信号足够明确、足够结构化，以便后续接入 AI 兜底解析器时，兜底逻辑能准确判断"该不该、能不能接手"，不需要靠猜测或字符串匹配错误信息。
27. 作为引擎开发者，我想对象模型识别出"会被多种实体类型复用的通用能力"（如"持有一堆物品"这件事，房间地面和玩家物品栏本质相同）并拆成独立组件供不同实体组合，而不是给每种实体各写一份专属同义组件，也不是为了追求"每个组件只有一个字段"而强行拆分本来就内聚的概念（如房间该有的字段可以合理地放在一起），以便未来给某个实体"热挂"一项新能力（如临时中毒、可骑乘）时，能直接复用已有的通用组件形状，不需要牵动已有组件的字段结构或触发大范围改动。
28. 作为未来题材包创作者（M3 UGC 闭环筹备），我想房间、出口、初始物品这些"场景数据"与"命令调度/存储引擎"这些"引擎能力"在代码结构上分离，以便未来题材包只需提供场景数据，不需要触碰引擎内部实现。
29. 作为项目架构师，我想这个骨架从零设计、不逐段复用 `engine/` 现有实现，也不复用旧代码的文件组织结构及命名（以现代游戏最佳实践组件、类、函数、变量命名为主，而不要一味地追求和侠客行源码统一，当然，如果原来的命名已经非常贴切可以复用），只在对象模型的骨架形状与 ECS 相关教训上参考旧代码，以便与"整体重写"的既定决策保持一致。

## Implementation Decisions

**对象模型**

- 采用组件化（ECS 风格）Entity/Component 存储：entity 只是一个不透明 id，行为数据分散在按类型存储的组件里，查询接口至少支持"新建实体 / 挂载组件 / 读取某类型组件 / 判断是否有某类型组件 / 移除组件 / 查询同时拥有多个指定组件类型的实体集合"。这是对旧 `engine/` ECS 实现（SparseSet 存储、`entities_with` 多类型查询）设计形状的参考延续，但内部存储实现从零编写，不搬迁旧代码。
- **组件设计原则：拆分依据是"能否被其他组件/实体类型复用"，不是"字段数量"。** 不是"组件必须小、必须单一属性"这种教条——一个组件把多个字段放在一起是完全合理的，只要这些字段确实只属于同一个内聚概念、不会被别的实体类型复用。真正要拆出来的，是那些**在多种不同实体类型上反复出现的通用能力**：一旦发现某组属性会被两种以上不相关的实体共用，就应该把它提炼成一个独立组件，让不同实体按需组合，而不是在每种实体专属的组件里各写一份。反面例子是旧代码 `Attributes` 那种把气血/内力/年龄/性别等几十个跨领域字段硬凑一起的大杂烩——那不是"字段多"本身的问题，是这些字段分别属于"生命值""基础属性""角色设定"等不同关注点，被错误地拼进了同一个组件。依据是 `engine/prototypes/ecs_ugc/` throwaway 原型对 ECS / 继承树 / Feature 混入三种模型的手感对比：把通用能力提炼成独立组件后，"运行时给已有实体热挂新能力"与"跨实体按能力统一查询"（如"查所有能开合的东西"）这两个 UGC 高频场景摩擦最低；但这不代表所有组件都要拆到只剩一个字段，过度拆分本身也是一种不必要的摩擦。
- 按这条标准（复用性，不是字段数）拆分 M1 阶段骨架必需的最小组件集：
  - 身份（名称 + 别名列表）——房间、物品、（未来）NPC 等一切"需要被玩家用名字/别名指代"的实体通用，明显跨多种实体类型复用，独立成组件。
  - 位置（所在房间 id）——玩家等"存在于某个房间里"的实体使用，同样跨类型复用。
  - 物品容器（持有的物品集合）——**房间的地面与玩家的物品栏本质是同一种通用能力**"这个实体持有一堆物品"，不应该分别做成两个专属命名的组件；用同一种"物品容器"组件，房间实体与玩家实体各挂一份即可。这也顺带给未来的箱子/背包等容器类物品预留了直接复用的空间，不需要另起一套。
  - 出口表（方向 -> 目标房间 id 的映射，运行时可增删条目）——目前只有房间会用到，但"一个地点通向哪些其他地点"这条能力未来也可能被"载具内部的微型空间"之类的实体复用，独立成组件为这类复用留空间；同时它和"这个房间叫什么/长什么样"是两件独立的事（一个密室未来即便没有出口表也不影响它有身份和描述），分开存也让"某个房间暂时没有任何出口"这种状态天然可表达，不需要特殊处理空值。
  - 可开合/上锁状态（按方向索引：开/关/锁 + 匹配钥匙物品 id）——这条能力明显不止用在房间出口上（未来箱子、密室入口等"可以被开关/上锁"的东西都能复用同一个组件形状），因此即便 M1 只有出口用得到，也按"可复用能力"对待，独立成组件，不依附在出口表或房间描述上。
  - 房间描述（简述 + 详细描述文本）——是否要进一步拆分到"物品也复用同一个描述组件"，取决于 M1 是否需要给物品也配简述/长描述；若物品的展示文本需求和房间一致（大概率一致，都是"简述+详细描述"两段文本），应该合并成一个通用的"描述"组件供房间与物品共用，而不是各自专属的 `RoomDescription`/`ItemDescription`。
  这套拆分的判断标准是一致的：先问"这个属性/属性组会不会被其他实体类型用到"，会就独立成组件；不会、且这些字段本身就是同一个概念的自然组成部分（比如"这个门锁着"这件事的开/关/锁三个状态字段），就留在一起，不用强行拆到一个组件一个字段。
- 不引入组件类型的强类型 schema 校验层（旧代码的 SchemaRegistry 概念）。M1 组件数量少、字段简单，校验层收益不足以覆盖引入成本；待 M2 组件数量显著增加、确实出现过拼写错误类故障时再评估是否引入，避免过度设计。
- 对象模型需要天然支持"未来给某个 entity 追加新组件类型"而不影响现有查询逻辑，为 M2 NPC/战斗组件的引入留出扩展点（不要求 M1 就实现 NPC，只要求模型形状不排斥）。

**命令调度**

- 命令处理以"动词字符串 -> 处理函数"的注册表组织，每个处理函数签名统一为"给定当前世界状态与发起命令的玩家实体 id、命令参数，返回要展示给玩家的消息列表"。
- 调度流程保留"识别动词 -> 查找命令 -> 执行"这一最小主干（对应 08 号票"命令解析分发"骨架结论），权限校验作为管线中的一个可选阶段先占位（M1 单机单玩家、无账号分级，该阶段内容为空实现），避免 M2 引入多角色权限时需要改动调度框架本身。
- 命令处理函数只处理"玩家一次性输入触发"的状态变更（移动、拾取、丢弃）；不承担随时间推进的世界演化职责——这条边界延续自旧代码已验证过的不变量（"System tick 派生变更不经 Command"），为 M2 心跳驱动的系统（如战斗回合、NPC 行为）预留清晰的职责分界，避免同一状态变更被两条路径重复触发。
- 未知动词、缺失必需参数、目标不存在（房间无此出口/无此物品）均返回明确的提示消息，不抛出未捕获异常导致 CLI 进程终止。

**命令解析：别名 + 双层解析架构（为未来 AI 兜底解析预留扩展点）**

- 命令动词支持别名（如 `i` -> `inventory`、`l` -> `look`、方向简写 `n`/`s`/`e`/`w` 等 -> `go <方向>`）。别名在注册命令处理函数时随手一并声明，不是调度框架里为每个命令各写一段特例分支——保证以后新增命令时"顺带"就有别名支持，不需要单独设计。
- 命令作用的对象（出口方向、地面物品、物品栏物品，未来会扩展到 NPC）支持按别名匹配，玩家不必输入完全规范的名称。这条"给定候选集合 + 每个候选的规范名与别名列表 + 玩家输入的字符串，找出匹配项"的逻辑做成一个与具体对象类型无关的共享工具，供物品、出口、（未来）NPC 等各类目标解析复用，不为每种目标类型各写一套匹配代码。
- **解析与执行解耦，通过一个稳定的中间表示（"意图"：动词 + 已解析的目标引用 + 参数）衔接**，分两个独立阶段：
  1. 原始输入文本 -> 意图（"解析"阶段，可能失败，失败时返回明确、结构化的"无法理解"结果，而不是笼统的异常或字符串错误）。
  2. 意图 -> 调用对应的命令处理函数并产出消息（"执行"阶段，此前"命令调度"的所有决策都发生在这一阶段，且完全不关心某个意图是被哪个解析器解析出来的）。
  这个"稳定中间表示"的存在，是本节唯一必须在 M1 就做对的设计——一旦这层接口稳定，解析阶段可以是一条解析器链：先尝试最快、最确定的解析器，失败了再尝试下一个。
- M1 阶段只实现并接入**一个**解析器：基于命令别名表 + 简单语法规则（动词 + 位置参数）的确定性解析器。但架构上要把"解析器可以是一条链，各解析器共享同一个输出格式与失败信号"这件事定下来，使其成为一个可插拔的位置，而不是在确定性解析器内部硬编码"这是唯一的解析方式"这条假设。
- 不在 M1 实现第二个（AI/LLM 驱动的自然语言理解）解析器本体——那是 M1 之后的实验性方向，依赖具体的模型接入方式与成本考量，本 spec 不展开。M1 只保证：确定性解析器失败时产出的信号足够清晰、可被判断"要不要交给下一级尝试"，且"意图"这个中间表示的形状足够通用，能装得下未来 AI 解析器的输出，不需要事后回头改执行层或已有解析器的接口。
- 这个分层设计的取舍是刻意的：常规、能被规则覆盖的命令走确定性解析器，不产生任何 AI 调用开销（免费、可预测、可测试）；只有确定性解析器识别不了的输入，才有可能在未来升级路径上被交给能理解更自由表达的 AI 解析器兜底（有成本、但覆盖面更广）。两者互补而不是互斥。

**心跳循环**

- 引入一个独立的 tick 计数机制，作为后续所有"随时间推进"的系统的统一驱动点。M1 阶段该循环只承载一件事：达到设定的 tick 间隔时触发一次存档。
- 心跳的推进时机（每条命令后推进一次，还是独立于命令节奏推进）由实现阶段决定，但需保证：普通玩玩过程中会稳定触发若干次周期性存档，`quit` 时无论是否到达周期都会立即触发一次全量存档。

**存档与崩溃恢复**

- 持久化边界定义为"崩溃恢复级耐久"：内存中的对象状态是权威态，存档只是周期性快照，用于进程重启后恢复到最近一次成功快照的状态；不承诺"每次状态变更立即持久化"，两次快照之间发生的变更在异常崩溃后会丢失（这是有意的设计取舍，不是缺陷）。
- 存档写入必须是原子的：不允许因为写入过程中途中断（进程被杀/断电模拟）而破坏上一次已经成功写入的存档数据。技术手段不限定，但必须满足"写入中断后，重启时至少能恢复到写入开始前的状态，绝不会出现半写状态"这一行为承诺。
- 单个存档文件（或存档条目）损坏时，恢复流程只应跳过该条目并记录警告，不应因为一处损坏拒绝整体启动。
- 存档粒度按 entity（每个实体一份独立存档记录），为后续"只存变更过的实体"（脏标记式增量存档）留出扩展空间，M1 阶段可以先做全量存档，只要不排斥后续升级为增量。
- 存档格式选择需要保证人类可读、便于调试（M1 场景数据量很小，不需要为性能预先优化格式选型）。

**场景数据与引擎能力的边界**

- **2026-07-18 范围修订（偏离本节原决定，用户当场拍板，详见文末「范围修订记录」）**：M1 的空场景数据（房间、物品、静态展示型 NPC）改为用 YAML 文件配置，不再是内嵌 Python 数据结构；YAML 加载器是 M1 骨架的一部分，不再整体推给 M3。这是一份**M1 内部的过渡格式**，目标是"让非工程师也能改场景数据、跑通闭环优先"，不是"现在就设计好未来 UGC 创作者用的正式 DSL"——M3 UGC 闭环里程碑仍然可能整体替换这份格式，本节这条决定不预支 M3 的设计工作。
- YAML 配置的场景元素范围：房间（身份/描述/出口，对应 01 号票已有的 `Identity`/`Description`/`Exits` 组件）、物品（对应 03 号票的物品与容器组件）、**静态展示型 NPC**（只有 `Identity`+`Description`+`Position`，站在某个房间里、有名字有描述，不能对话、不能战斗、没有任何行为逻辑——不是"NPC 与 NPC AI"这个仍然 Out of Scope 的子系统，只是复用已有组件表达"一个有名字的房间摆件"）。是否顺带覆盖 04 号票的门/锁状态由实现该票时决定（如果当时 04 已完成，一并纳入更完整；如果还没完成，先覆盖房间/物品/npc，门/锁状态留一个明确的后续小补丁）。
- 加载器的失败处理延续本 spec 别处已定的原则：YAML 里引用了不存在的房间/方向/物品等结构性错误，启动时应给出清晰的错误定位（哪个文件、大致哪个位置），不是裸 Python 异常堆栈；这不是"崩溃恢复"范畴（那是运行时存档的语义），是"加载期数据校验"，两者不要混为一谈。
- **动态出口机制**：出口表不是启动后固定不变的常量，而是运行时可以被增删的状态（谁来触发增删——命令、还是后续心跳驱动的系统——由实现阶段决定，M1 只要求"至少有一条可验证的路径能把出口表改掉"）。这是给 08 号票坐骑与交通系统提到的"动态出口缝合地理断点"（渡口渡船、桥梁坍塌、密道开启一类"地理连接随玩家动作改变"的内容）预留的通用引擎机制。
- **门机制**：出口可以选配一个"可开合/上锁状态"，与出口表分开存储——因为这条能力预期会被出口之外的其他东西复用（箱子、密室入口等），按"通用能力独立成组件"的标准处理（见"对象模型"）。关着的门挡路；锁着的门需要玩家物品容器里有匹配的钥匙物品才能解锁；开/关/敲门/解锁是独立命令。
- **范围边界（需你确认）**：M1 落地的是"动态出口"与"门"这两个**机制**本身（数据结构 + 至少一条可验证的示例），不是"渡口渡船"这类具体题材内容（船只实体、乘船动作、坐骑加速等）——那些仍然需要有场景内容才谈得上验证，按原计划留给 M2。我把你的要求理解为"把机制提前到 M1"，如果你的本意是 M1 就要有一个可乘坐的渡船场景，请告诉我，我会调整范围。

**CLI 前端**

- **确认：真实命令行终端进程（stdin/stdout），不做 Web 模拟终端。** 提供一个最小命令行交互循环（读取一行输入 -> 走解析/调度 -> 打印返回消息 -> 继续读下一行），用 `python -m ...` 之类方式直接在终端启动，作为 M1 唯一的玩家交互入口。理由：Web 模拟终端本质上需要一条网络通道（HTTP/WebSocket）把浏览器和引擎进程连起来，这属于本 spec 已经明确排除的"网络协议层"，M1 阶段引入会提前背上协议设计的复杂度，与"单机、不搞分布式/网关"的架构不变量方向不符；真实终端零依赖，且天然验证"前端只是核心的一个调用者"这条边界。
- 网络协议（WebSocket/TCP，或给 Web 终端模拟用的通道）不在本里程碑范围，后续里程碑再接入，不影响本骨架的调度/存储核心设计（核心与前端应可分离——这也是为什么前端选型可以晚定/换，因为它不影响核心）。
- CLI 循环本身的职责仅限于"输入行 -> 委托解析与调度 -> 打印输出"，不应把游戏逻辑写在 CLI 模块里（游戏逻辑归命令处理函数）。

## Testing Decisions

- 测试只验证外部可观察行为：给定一行命令输入，断言返回的消息内容/世界状态查询结果，不断言组件存储的内部实现细节（如是否用了某种特定数据结构）。
- 覆盖点：
  - 命令调度：合法命令产生预期消息与状态变更；未知命令、缺参、目标不存在均返回提示而不抛异常。
  - 移动：合法方向移动 + 自动 look；非法方向不移动且给出提示。
  - 拾取/丢弃：物品在房间地面集合与玩家物品栏集合之间正确转移；操作不存在的物品给出提示。
  - 门与动态出口：关着的门挡路且提示；锁着的门无钥匙时拒绝、有匹配钥匙时解锁成功；开/关/敲门行为符合预期；至少一条"出口表运行时被增删"的路径被验证到（增删后 `look`/`go` 能反映最新出口状态）。
  - 别名解析：命令别名（如 `i`/`l`/方向简写）与目标对象别名（物品/出口）都能正确命中对应的规范命令/对象；别名与规范名冲突或歧义时的行为有明确定义（不是未定义行为）。
  - 解析失败信号：确定性解析器识别不了的输入，返回的"无法理解"结果格式一致、可被后续兜底逻辑消费（即便 M1 还没有第二级解析器，这个信号的形状也要被测试锁定，防止后续接入时才发现格式不够用）。
  - 存档与恢复：保存后重新构建一个全新的引擎/世界实例并从存档恢复，恢复后的状态（玩家位置、物品栏、房间地面物品、出口表与门状态）与保存前一致；模拟写入中途中断不破坏上一次已成功的存档；单个存档条目损坏时恢复流程跳过它而不是整体失败。
  - 心跳循环：达到设定间隔后触发存档；`quit` 无论是否到达周期都会触发一次立即存档。
- Prior art：现有 `engine/tests/` 下的 e2e 测试已经建立"构建 world → 调用命令函数 → 断言返回消息与之后可查询的世界状态"这一测试结构，可作为 M1 新测试代码在写法上的参照（不复用具体断言内容，场景内容完全不同）；旧存储模块的"原子写三步 + 崩溃恢复"验证思路同样可作为存档测试设计的参照。

## Out of Scope

- 战斗、技能、状态/效果、死亡与轮回：这四个子系统虽在 08/09 号票中被归类为 MVP 必做，但没有具体题材内容（怪物、技能数值、状态效果）就测不出价值，且部分依赖 02 号票尚未拍板的"战斗流程归引擎"倾向；推到 M2（有内容的题材场景可玩）后再落地。M1 完全不涉及，因此 02 号票是否改判也不影响本 spec。
- 登录/账号系统：M1 是单机单玩家 demo，玩家实体在启动时直接创建/从存档恢复，不设计账号校验、多玩家并发登录、角色创建流程。
- 频道系统、表情系统、金钱系统、**NPC 的行为/AI**（对话、战斗、调度、任何"随时间/随交互产生反应"的逻辑）：均需要具体场景内容或多玩家/多实体交互才有验证意义，推到 M2 及以后。**注意范围收窄**：静态展示型 NPC（无行为，见「场景数据与引擎能力的边界」2026-07-18 修订）已改判为 M1 内，这里排除的是"NPC 有行为"这部分，不是"NPC 这种实体完全不存在"。
- 坐骑与交通系统的**具体题材内容**（坐骑实体、乘船/骑乘动作、移动效率加成、渡船班次等）：M1 只落地"动态出口"与"门"这两个通用机制（见"实现决策"），机制之上的具体载具/坐骑内容仍然推到 M2（需要场景内容才谈得上验证）。
- AI/LLM 驱动的自然语言命令解析器本体：M1 只把架构设计成"可以在确定性解析失败后接一个这样的解析器"，不实现这个解析器本身——具体模型接入方式、成本、prompt 设计留给后续实验性阶段单独定。
- 网络协议层（WebSocket/TCP 多客户端接入，或给 Web 终端模拟用的通道）：M1 只做真实命令行终端前端（见"CLI 前端"决策），多人网络接入是后续里程碑的工作。
- UGC/DSL 场景数据加载格式的**正式设计**（面向未来创作者生态的、有版本兼容承诺的格式）：M1 落地的 YAML 加载器是内部过渡格式（见「场景数据与引擎能力的边界」2026-07-18 修订），够用即可，不追求向后兼容或完整校验；面向创作者的正式设计仍然留给 M3，M3 可能整体替换掉这份格式。
- 组件类型的强类型 schema 校验层：本 spec 明确不引入，见"实现决策"。
