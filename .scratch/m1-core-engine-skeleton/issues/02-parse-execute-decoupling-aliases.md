# 02 — 解析执行解耦 + 别名机制

**What to build:** 把 01 号票里"读一行文本直接查表调用处理函数"的路径,重构成两个独立阶段,通过一个稳定的中间表示"意图"(动词 + 已解析的目标引用 + 参数)衔接:

1. **解析阶段**:原始输入文本 -> 意图。可能失败,失败时返回结构化、格式一致的"无法理解"结果(不是笼统异常或裸字符串错误),这个信号的形状要能被"要不要交给下一级解析器兜底"这个判断消费,即便 M1 还没有第二级解析器。
2. **执行阶段**:意图 -> 调用对应处理函数 -> 产出消息。这一阶段完全不关心某个意图是被哪个解析器解析出来的,01 号票里已有的调度决策(动词查找、参数传递)都归到这一阶段。

M1 只接入**一个**解析器:基于命令别名表 + 简单语法规则(动词 + 位置参数)的确定性解析器。但架构上要把"解析器可以是一条链,各级共享同一输出格式与失败信号"这件事定下来,做成可插拔的位置,不要在这个解析器内部硬编码"这是唯一的解析方式"这条假设——这是为后续(M1 之后)接入 AI/LLM 兜底解析器预留的扩展点,本票不实现那个解析器本体。

同时新增两类别名支持,复用同一套"给定候选集合 + 每个候选的规范名与别名列表 + 用户输入的字符串,找出匹配项"的通用工具(与具体对象类型无关,不为每种目标各写一套匹配代码):

- **命令别名**:如 `i` -> `inventory`、`l` -> `look`、`n`/`s`/`e`/`w` 等方向简写 -> `go <方向>`。别名在注册命令处理函数时随手声明。
- **目标别名**:玩家输入指代某个对象(本票范围内只有"出口方向"这一种目标类型可验证,如某个出口的别名"门"/"洞口"等)不必输入完全规范的名称。这个通用匹配工具本身要做到与目标类型无关,供 03(物品)、04(出口/门)号票直接复用,不需要重新实现。

**Blocked by:** 01 — 需要先有可运行的命令注册表与 CLI 循环才能重构其调度路径。

**Status:** resolved

- [x] 存在一个"意图"的稳定中间表示类型(动词 + 目标引用 + 参数),解析阶段的唯一产出物是它或一个结构化失败信号
- [x] 确定性解析器:文本 -> 意图,覆盖 01 号票已有的 `go`/`look`/`help`/`quit`,且这些命令行为与 01 号票的验收标准保持一致(重构不引入行为回归)
- [x] 解析失败(如完全无法识别的动词)返回结构化的"无法理解"结果,不抛出未捕获异常;这个失败信号的形状有测试锁定
- [x] 执行阶段函数只接收"意图"作为输入,不感知原始输入文本或解析器身份
- [x] 命令别名生效:至少 `i`(部分是 03 号票才有意义,若本票 inventory 尚不存在可暂不测试)、`l`、`n`/`s`/`e`/`w` 可用且与规范命令行为一致
- [x] 存在一个与目标类型无关的通用别名匹配工具函数/接口,并有测试覆盖至少一种目标类型(出口方向别名)的匹配成功与匹配失败(歧义或无匹配)场景
- [x] 别名与规范名冲突或歧义时的行为有明确定义并被测试锁定(不是未定义行为)
