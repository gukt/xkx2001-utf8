# 05 — 心跳循环 + 存档与崩溃恢复

**What to build:** 引入一个独立的 tick 计数机制,作为后续所有"随时间推进"系统的统一驱动点;M1 阶段该循环只承载一件事——达到设定的 tick 间隔时触发一次存档。心跳的推进时机(每条命令后推进一次,还是独立于命令节奏推进)由实现决定,但普通游玩过程中要能稳定触发若干次周期性存档。

**存档与崩溃恢复语义**(持久化边界是"崩溃恢复级耐久",内存态是权威状态,不承诺每次变更立即持久化):

- 存档粒度按 entity(每个实体一份独立存档记录),格式人类可读、便于调试。
- `quit` 无论是否到达 tick 周期都会立即触发一次全量存档。
- 存档写入必须原子:写入过程中途中断(模拟进程被杀)不能破坏上一次已成功写入的存档数据——重启后至少能恢复到写入开始前的状态,绝不出现半写状态。
- 单个存档文件/条目损坏时,恢复流程只跳过该条目并记录警告,不因一处损坏拒绝整体启动。
- 重新构建一个全新的引擎/世界实例并从存档恢复后,恢复出的状态(玩家所在房间、玩家物品栏、房间地面物品、出口表与门状态)与保存前一致——这条要求依赖 03(物品容器)与 04(出口表增删、门状态)已经存在的组件与状态。

**Blocked by:** 03、04 — 存档要覆盖物品容器状态(03)与出口表/门状态(04)才能验证"重启后完整恢复"这条承诺;03、04 互相独立,可并行完成后再开始本票。
