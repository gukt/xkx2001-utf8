# 01 — 引擎骨架起步:ECS 存储 + 命令调度 + CLI 循环 + 静态两房间移动

**What to build:** 一个可以通过真实终端进程运行的最小闭环。玩家启动 CLI 后出现在一个由至少两个互通房间组成的静态场景中的某一间;输入 `go <方向>` 可以在合法出口间移动,移动后自动展示新房间的 `look` 信息;尝试往不存在的出口方向移动会收到明确提示而不是崩溃或静默无反应;`look` 展示当前房间的简述、详细描述与可用出口列表(本票暂不含地面物品,物品容器留给 03 号票);`help`/`h` 列出当前骨架支持的全部命令;输入引擎不认识的命令收到"未知命令"提示 + 简短帮助指引;`quit` 直接退出进程(本票不含存档,退出即丢弃当前状态,存档留给 05 号票)。

## Comments

实现落地在 `engine/src/mud_engine/{world,components,commands,scenes,cli,__main__}.py`,共 38 条测试(`engine/tests/test_{world,commands,scenes,cli}.py`)。`just gate`(ruff + pytest)全绿,并手动跑过一遍真实终端管道验证完整闭环。ECS 存储模块文件名定为 `world.py`(类名仍是 `World`),不叫 `ecs.py`——避免文件名直接暴露实现技术选型这种"技术架构味"的命名。

实现过程中的两个额外决定(影响后续 02~05 号票的写法,不是本票范围内的需求变更):

1. **测试命名改为 BDD 嵌套类风格**(用户当场决定):测试按 Given/When 场景分组成嵌套类,方法名只写 Then,一个方法一个行为焦点。记录在 `engine/README.md`「测试约定」。踩坑记录:这些嵌套类不带 `Test` 前缀,pytest 默认 `python_classes=Test*` 会把它们静默跳过(不报错不警告),因此在 `engine/pyproject.toml` 把 `python_classes` 扩到 `["Test*", "When*", "Given*"]`。
2. **新增 `World.require_component`**(`/code-review` Standards + Spec 双轴都独立发现同一处问题后加的):原来 `commands.py`/`scenes.py` 里多处用 `assert component is not None` 保证场景数据完整性,这个模式重复出现(Duplicated Code),且 `assert` 在 `python -O` 下会被整体剥离,不够稳健。改成 `World.require_component`,缺失时抛 `LookupError`,任何运行模式下都生效;不用于"玩家输入引用的目标可能不存在"这种需要给玩家提示而非抛异常的场景(那部分逻辑不变)。

`/code-review` 还发现 `help` 命令把别名 `h` 直接注册进 `_REGISTRY`(而不是走 `register()` 声明式别名),导致 `h` 会跟 `help` 一起出现在命令列表里;这是本票文字里已经写明的刻意取舍(不为一个特例先建 02 号票的别名机制),补了一条测试把这个行为锁定,没有改行为本身。
