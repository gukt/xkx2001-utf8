# 04 — 门与动态出口机制

**What to build:** 落地"动态出口"与"门"两个通用引擎机制本身(不含渡船/坐骑等具体题材内容,那些推到 M2)。

- **可开合/上锁状态组件**:按方向索引,记录开/关/锁三种状态之一,锁定状态可选配一个匹配钥匙物品 id。与出口表分开独立存储(不依附在出口表或房间描述上),因为这条能力预期会被出口之外的其他实体(箱子、密室入口等)复用。
- **新增命令**:`open`/`close`/`knock`/`unlock`,均作用于当前房间某个方向的出口。关着的门无法直接通过(`go` 该方向被拒绝并提示);锁着的门需要玩家物品容器里有匹配的钥匙物品才能 `unlock` 成功,没有钥匙时收到明确提示。
- **动态出口验证**:出口表不是启动后固定不变的常量,而是运行时可以被增删的状态。本票至少落地一条可验证的路径能把某个房间的出口表改掉(增加或删除一条出口),改动后 `look`/`go` 能立即反映最新的出口集合——路径本身可以很简单(例如某个命令的副作用),重点是证明数据结构支持而不是给出完整的题材内容。

出口方向/门的名称匹配复用 02 号票的通用目标别名匹配工具。

**Blocked by:** 02 — 复用其目标别名匹配工具与"解析 -> 意图 -> 执行"管线来接入这四个新命令;可与 03 号票并行开发(互不依赖)。

**Status:** ready-for-agent

- [ ] 场景数据里至少一对相邻房间之间的出口预置为"关闭"状态,尝试 `go` 该方向被拒绝并提示门是关的
- [ ] `open` 该方向的门后,`go` 该方向能成功移动;`close` 后再次被拒绝
- [ ] 场景数据里至少一个出口预置为"上锁"状态并关联一个钥匙物品;玩家物品栏没有该钥匙时 `unlock` 失败并提示需要钥匙
- [ ] 玩家物品栏持有匹配钥匙时 `unlock` 成功,之后该出口按"开"状态处理(或按设计需要再 `open` 一次,行为需在实现中明确定义并测试锁定)
- [ ] `knock` 命令对关闭/上锁的门产生合理反馈消息(具体文案由实现决定,只要求命令存在且不崩溃)
- [ ] 存在至少一条路径可以在运行时给某个房间的出口表增加或删除一条出口记录,改动后立即体现在该房间的 `look` 输出与 `go` 行为上
- [ ] 可开合/上锁状态是一个独立于出口表的组件(代码结构或测试可体现两者分开存储)
- [ ] 出口/门的目标名称匹配复用 02 号票的通用别名匹配工具,而不是另写一套判断逻辑
