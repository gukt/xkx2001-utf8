# 06 — YAML 场景数据 DSL:房间/物品/静态展示型 NPC

**What to build:** 把场景数据(房间、初始物品摆放、静态展示型 NPC)从 01/03 号票里内嵌的 Python 数据结构,改成从 YAML 文件读取。这是 M1 内部的过渡格式(见 [M1 spec「场景数据与引擎能力的边界」2026-07-18 修订](../spec.md) 与文末「范围修订记录」),目标是"非工程师也能改场景数据、跑通闭环优先",不是现在就设计好 M3 要交给创作者用的正式 UGC DSL——M3 可能整体替换这份格式,本票不追求向后兼容或完整校验。

具体范围:

- **房间**:身份(名称)、描述(简述+详情)、出口表——对应 01 号票已有的 `Identity`/`Description`/`Exits` 组件,YAML 里一份房间条目应该能表达这三者外加它在场景图里的键(用于出口引用其他房间)。
- **物品**:对应 03 号票的物品与容器组件,YAML 应该能表达一件物品的身份/描述,以及它初始摆放在哪个房间的地面容器里。
- **静态展示型 NPC**(本票新增的场景元素,不是新子系统):只用 `Identity`+`Description`+`Position` 三个已有组件表达"站在某个房间里、有名字有描述"的实体,**没有任何行为**——不能对话、不能战斗、不会移动、不会被任何命令特殊处理(玩家甚至可以对它 `look`,但引擎不需要为它专门写一个"npc 命令");这不是把 spec 里明确 Out of Scope 的"NPC 与 NPC AI"子系统提前做了,只是给一个已有的组件组合起了个别名。
- 是否同时覆盖 04 号票的门/锁状态(方向的开合/上锁),由实现本票时 04 号票的完成情况决定:04 已完成就一并纳入 YAML 表达;04 还没完成就先覆盖房间/物品/npc 三类,门/锁状态留一个明确的后续小补丁(不要因为等 04 而阻塞本票)。
- **加载期校验**:YAML 里引用了不存在的房间键(如某个出口指向一个没有定义的房间)、缺失必需字段等结构性错误,启动时要给出清晰的错误定位(哪个文件、大致是哪一条数据),不是让 Python 解析异常的原始堆栈糊到玩家脸上。这属于"加载期数据校验",跟"崩溃恢复级"的运行时存档语义是两件不同的事,不要混着实现。
- 01 号票现有的三房间场景数据(起始庭院/长廊/静室)迁移成等价的 YAML 内容,行为不变(现有 35+ 条测试的场景相关断言应该继续成立,除非测试本身在断言"数据是内嵌 Python 结构"这种实现细节——那种测试本来就不该存在)。

**Blocked by:** 03(需要物品与容器组件已经存在,才能在 YAML 里配置房间地面的初始物品摆放;NPC 与房间所需的组件 01 号票已具备)。可以与 05 号票并行认领(两者互不依赖)。

- [x] 存在至少一份 YAML 文件,定义了 01 号票原有的房间图(房间身份/描述/出口),引擎启动时从这份 YAML 加载,不再依赖内嵌 Python 元组
- [x] YAML 里能定义物品的身份/描述与初始摆放的房间,加载后该物品出现在对应房间的地面容器里,`take`/`look` 等既有命令行为不变
- [x] YAML 里能定义至少一个静态展示型 NPC(身份+描述+所在房间),加载后玩家 `look` 该房间能看到这个 NPC 的存在;这个 NPC 不响应任何专门为它设计的命令,也不会自己产生任何行为
- [x] YAML 引用了不存在的房间键/方向等结构性错误时,启动给出清晰的错误定位(文件+大致位置),不是裸解析异常堆栈
- [x] 场景数据加载逻辑与命令调度/ECS 存储代码保持分离(不因为改成 YAML 就把加载逻辑掺进 `commands.py` 或存储实现里)
- [x] 现有测试套件(01/02/03 号票产出的测试)在场景数据换成 YAML 来源后仍然全部通过,不需要改动测试断言本身(如果某条测试因为断言了"数据来自内嵌结构"这种实现细节而挂掉,应该视为那条测试本身有问题,顺手修掉)
