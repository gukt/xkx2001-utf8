# LPC 任务系统现代设计审视

> 角色：现代任务玩法设计师
> 依据：当前仓库 LPC 源码一手资料
> 范围：只产出设计输入，不输出 engine 接口契约

## 一、LPC 任务系统的核心体验特征

### 1.1 节奏：高频、短循环、强限制

LPC 任务整体呈现"短、频、快"的循环特征：

- **时间压力极紧**：走镖任务从接镖到失败仅 100 ~ 120 秒（`call_out("destroy_it", t, ...)`，`t = 100 + random(20)`），`/d/city/obj/biaohuo1.c`。华山巡山 50 ~ 70 个 MUD 单位时间，`/d/huashan/npc/buqun.c`。
- **接取冷却短**：黄河帮任务要求两次接取间隔不少于 180 秒，`/d/huanghe/npc/bangzhu_duty.h`。
- **每日/每轮硬上限**：走镖个人日限 5 次、NPC 日限 20 次；`ob->query_temp("biao/times") >= 5`、`me->query_temp("biao") >= 20`，`/d/city/npc/biao_assign.h`。
- **Cooldown 作为节奏阀**：少林挑水完成后进入 `ts_pending`（10 ~ 25 个单位时间），期间不可再接，`/d/shaolin/npc/tiaoshui2.h`。

这种设计与早期 MUD 的"在线时长换数值"模型高度吻合：玩家通过反复接取短任务累积经验、潜能，而不是通过长叙事弧获得一次性奖励。

### 1.2 循环：模板化数据表 + NPC 口耳相传

任务内容以"数据表 + 随机抽取"的方式批量生产：

- 黄河帮任务按玩家经验分 8 档（3000 ~ 500000），每档一个 `.c` 数据文件，内部是 `mapping *bangjobs` 数组，包含"杀 / 寻 / 摊费 / 截镖 / 示威 / 送礼 / 护驾"等类型，`/d/huanghe/bangjob/bangjob3000.c`、`/d/huanghe/npc/bangzhu_duty.h`。
- 神龙教任务表同样按经验分档，低等级以"寻物"为主，高等级大量填充 `FORCEJOIN` 与 `PK`，`/d/shenlong/sgjob/sgjob20000.c`、`/d/shenlong/sgjob/sgjob500000.c`。
- 所有任务触发几乎都依赖 `ask <npc> about <topic>` 指令，通过 NPC 的 `inquiry` 映射调用函数或返回字符串，`/cmds/std/ask.c`。

换句话说，任务系统不是"任务设计师写脚本"，而是"程序员在数据文件里填表 + 写 NPC 回调"。这导致内容产量高但叙事一致性弱。

### 1.3 社交：PvP、公会竞争与公共声誉

社交不是协作，而是对抗与标签：

- **走镖可被劫**：其他玩家可以 `open` 红镖，导致原镖客失败并公开羞辱（`rumor` 频道广播），`biao_stealer` 标记使其永久失去接镖资格，`/d/city/obj/biaohuo1.c`、`/d/city/npc/biao_assign.h`。
- **帮派对峙**：黄河帮任务包括"示威"（杀其他帮派成员）、"截镖"（劫系统镖车）、"护驾"（护送帮主 NPC），`/d/huanghe/npc/bangzhu_duty.h`。
- **神龙教 PK / 强拉入教**：玩家可戴面罩匿名追杀指定目标，并在尸体上 `sign` 留名；也可以对玩家或 NPC 使用 `forcejoin` 强迫加入神龙教，目标阵营、中毒状态、善恶值都会影响成功率，`/d/shenlong/obj/sg_mianzhao.c`。目标下线后，系统仍会记录并惩罚任务执行者，`/d/shenlong/junk/cancel_pk.h`。
- ** reputation 公开**：`shen`（正邪值）、`biao_stealer`、`sg/spy` 等标签在 NPC 与玩家交互中被反复校验。

### 1.4 随机性：无处不在的骰子

随机性贯穿接取、执行、奖励全过程：

- **接取随机**：走镖 5 个目的地随机；黄河帮 / 神龙教任务表随机抽取；`/d/city/npc/biao_assign.h`、`/d/huanghe/bangjob/bangjob3000.c`。
- **执行随机**：星宿找毒虫依赖 `kar`（福缘）与 `int/con/str` 综合判定，搜索过程中可能随机换房间、被伏击、捡到钱、摔伤，`/d/xingxiu/xx_job.h`。
- **奖励随机**：`job_server.c` 的核心结算公式为 `exp_reward = exp_limit * exp_rate * (time_now - start_time) / 360000`，其中 `exp_rate` 由具体任务代码传入一个 0 ~ 100 的随机百分比，`/clone/obj/job_server.c`。

随机性在早期 MUD 里既是乐趣来源，也是平衡手段；但对现代玩家来说，过度随机容易变成"赌博感"和"无效劳动"。

## 二、与主流游戏设计对照

### 2.1 开放世界任务 vs. LPC 的"点对点"任务

| 维度 | 《原神》《艾尔登法环》《辐射》式开放世界 | LPC 任务系统 |
|---|---|---|
| 空间引导 | 地图标记、环境叙事、兴趣点（POI） | 依赖 NPC 对话与玩家记忆，无任务日志，无地图标记 |
| 触发方式 | 接近即触发、物品拾取触发、环境观察触发 | 必须 `ask` 特定 NPC 关于特定关键词 |
| 并行度 | 多任务并行，随时切换 | 同一类任务通常互斥（`pending` 标记），如少林挑水、`huashan/job_pending` |
| 失败代价 | 低，可重试 | 高：金钱损失、Reputation 污点、Cooldown、甚至死亡 |

证据：华山巡山需要玩家记住 12 个地点并逐一经过，`/d/huashan/npc/buqun.c`、`/d/huashan/fengding.c` 等房间通过 `init()` 设置临时标记。玩家没有任何可视化进度，只能反复 `ask` 岳不群检查。

### 2.2 动态事件 vs. LPC 的定时/定点刷新

- 北京城门守卫任务具备最接近动态事件的结构：`gen_killer()` 每 10 秒检查一次，根据昼夜切换生成不同等级杀手，玩家需在指定城门站岗并击杀 / 放行，`/d/beijing/outer_gate.h`。同时城门会随时间开关，影响全玩家通行。
- 但这类动态事件与玩家个人任务强绑定：必须先从多大人处接到 `current_job`，再到指定城门 `guard`，并且执勤期间不能离开。它更像"带波次刷新的岗位任务"，而非《激战 2》《魔兽世界》式人人可参与的公开动态事件。

### 2.3 支线叙事 vs. LPC 的"数据表任务"

- 《巫师 3》《博德之门 3》的支线任务通常有完整叙事弧、道德选择、多结局。LPC 中具备叙事元素的任务（如一灯大师线、樵夫守门、丐帮密函）多由房间脚本和 NPC 回调硬编码，每个任务都是一次性手工实现。
- 黄河帮 / 神龙教的"数据表任务"本质上没有叙事，只有"目标 + 奖励"，与现代手游的"日常委托"接近，但缺乏包装和情绪节拍。
- 丐帮密函任务具备分支：目标 NPC 的 `shen` 决定其是收下、退还还是杀人灭口，`/d/gaibang/obj/mihan.c`。这是少数接近现代叙事选择的机制，但实现上完全写死在物品回调里，不可复用。

### 2.4 每日 / 每周循环 vs. LPC 的"日限 + Cooldown"

- 《魔兽世界》《最终幻想 14》《原神》都有每日任务 / 周常 / 时令活动，核心目的是建立登录习惯。
- LPC 已经具备雏形：走镖个人日限 5 次、NPC 日限 20 次；黄河帮 180 秒接取 CD；少林挑水完成后进入 `ts_pending`。但缺少"跨日重置"的显式日历系统，更多依赖 MUD 时间或服务器重启。
- 现代设计会把"每日"包装成"今日委托"并提供保底奖励；LPC 的日限更像"防刷机制"，而非正向激励。

### 2.5 成就追踪 vs. LPC 的隐形计数

- 现代游戏会把成就、进度、收集 explicitly 展示给玩家。
- LPC 也有计数，但全部藏在对象属性里：`huashan/job_comp`（华山任务完成次数）、`sg/failure`（神龙教失败次数）、`bangs/jointime`、`mingjiao/cc`（明教忠诚度）等。玩家需要靠 NPC 对话或自我记忆来感知进度。
- `job_server.c` 提供了 per-job 统计直方图和 per-user 日志，但那是给巫师（GM）看的平衡工具，不是玩家成就界面，`/clone/obj/job_server.c`。

### 2.6 社交协作 vs. LPC 的对抗社交

- 《最终幻想 14》的讨伐战、极神、零式副本强调小队协作；《星露谷物语》强调礼物与友谊进度。
- LPC 的社交设计以对抗为主：劫镖、帮派示威、神龙教 PK、强拉入教。唯一的"护送"（护驾）也是 NPC 跟随玩家，而非玩家组队。
- 明教任务把"五行"劳动（挖矿、造枪、挑水、砍树、挖地道）与忠诚度绑定，`/d/kunlun/npc/mingjiao_job.h`，具备"为组织做贡献"的雏形，但仍是单人循环。

## 三、仍然有效的设计（值得保留或借鉴）

### 3.1 任务作为经济 / 数值平衡的"水龙头"

`job_server.c` 把每个任务注册为每小时经验上限（`exp_limit`）与潜能上限（`pot_limit`），并按实际耗时与完成质量结算。这是非常有远见的设计：

- 任务奖励不再是固定值，而是与"时间投入 * 完成度"挂钩。
- 巫师可以通过 `set_exp_limit` / `set_pot_limit` 在线调节经济水龙头，配合 `job_hist` / `job_stat` 查看分布，便于宏观调控。
- 这对新引擎极具参考价值：把任务奖励抽象为"每小时产出上限 + 完成质量系数"，可抑制脚本刷级、便于后期平衡。

证据：`reward_func()` 中的公式与统计直方图，`/clone/obj/job_server.c`。

### 3.2 道具即任务状态

走镖的红镖、少林挑水的水桶与水瓢、黄河帮的帮令、神龙教的面罩，都是"任务道具即状态机"。这种设计的好处：

- 状态可视化：玩家背包里有实物。
- 状态可丢失 / 可被盗 / 可被毁坏，增加紧张感。
- 便于多人交互：别人可以打开你的红镖、抢走你的帮令。

现代游戏中《逃离塔科夫》《Albion Online》等也强调"物品即状态"，但在主流 MMORPG 中已被任务日志取代。新引擎可以保留"关键任务道具"机制，作为高风险的沉浸元素。

### 3.3 环境 + 属性判定驱动的解谜

一灯大师线中的"举石"（`do_lift`，力量判定）、华山悬崖的"系绳攀爬"（`do_tie`、`do_climb`，负重、力量、敏捷、轻功判定）是北京城门守卫任务中"爬墙"（`do_climb`，轻功判定）的同类设计。它们把角色属性与环境交互结合，是现代沉浸模拟游戏（如《博德之门 3》《塞尔达传说》）仍在使用的核心语言。

证据：`/d/dali/yideng2.c`、`/d/huashan/fengding.c`、`/d/beijing/outer_gate.h`。

### 3.4 声誉系统对任务资格的约束

正邪值 `shen`、帮派忠诚度、是否当过镖贼、是否背叛过门派，都会被 NPC 用来决定接任务资格。这形成了"行为有后果"的世界观一致性。现代游戏中，《辐射》《新维加斯》《博德之门 3》的 faction reputation 仍在使用类似逻辑。

证据：岳不群因弟子 `shen < -100` 直接逐出师门，`/d/huashan/npc/buqun.c`；黄河帮拒绝正派卧底，`/d/huanghe/npc/bangzhu_duty.h`。

### 3.5 多类型任务模板表

黄河帮的 `bangjobs` 与神龙教的 `sgjobs` 已经抽象出"寻 / 杀 / 护送 / 劫镖 / 示威 / 送礼 / 强拉入教"等任务类型，并按玩家等级分档。这是现代任务编辑器的前身。新引擎可以把这些类型化为题材包可配置的"任务蓝本"。

证据：`/d/huanghe/bangjob/bangjob3000.c`、`/d/huanghe/npc/bangzhu_duty.h`、`/d/shenlong/sgjob/sgjob20000.c`。

## 四、明显过时的设计（需要警惕或重新设计）

### 4.1 没有任务日志与进度可视化

所有进度都藏在 `query_temp()` / `query()` 里，玩家没有任何界面查看"当前任务 / 已完成步骤 / 剩余步骤"。现代玩家（包括移动端）已经习惯了任务追踪、地图标记、完成百分比。

风险：新引擎如果复刻这种"全隐藏"设计，会大幅提高认知负荷，劝退非硬核玩家。

证据：华山巡山需要集齐 12 个临时标记，玩家只能通过反复询问岳不群确认，`/d/huashan/npc/buqun.c`。

### 4.2 随机惩罚与高压失败

- 走镖失败直接扣 100 两银行存款，并公开羞辱。
- 星宿找虫可能随机摔倒受伤、被伏击、瓦罐破碎。
- 神龙教 PK 目标下线即判定失败并扣 `sg/exp`。

高压失败在早期 MUD 中制造了紧张感，但现代 F2P / 订阅游戏中，过度惩罚会导致流失。需要把"失败"重新设计为"收益折损 + 叙事后果"，而非"金钱 / 进度损失 + 社交羞辱"。

### 4.3 任务奖励与"刷"绑定，叙事薄弱

黄河帮、神龙教、少林挑水、华山巡山等核心循环的叙事包装极薄：NPC 说一句话，玩家去重复执行，回来领奖励。这与《魔兽世界》早期"击杀 10 只狼"式日常一样，已经被玩家广泛诟病为" padding content"。

新引擎应避免把"刷任务"作为主要内容；应把日常任务压缩为自动化的"离线收益"或"低注意力背景活动"，把玩家注意力留给有叙事密度的内容。

### 4.4 强制 PvP / 强制社交对抗

神龙教的 `forcejoin` 可以强迫其他玩家改变阵营，`/d/shenlong/obj/sg_mianzhao.c`；走镖可被任意玩家劫走，`/d/city/obj/biaohuo1.c`。这种"老 MUD 式自由"在现代设计中风险极高：

- 会造成新手被老玩家碾压，形成阶级固化。
- 强制 PvP 会劝退偏好 PvE 和叙事的玩家群体。
- UGC 题材包如果允许强制改变玩家阵营，会破坏创作者生态信任。

建议：把 PvP 对抗改为"可选择加入"的玩法层，或限制在特定世界 / 服务器规则内。

### 4.5 时间限制过短，与探索冲突

走镖 100 ~ 120 秒的时间限制迫使玩家必须非常熟悉路径，无法边探索边任务。这与现代开放世界"边逛边玩"的节奏相冲突。新引擎应区分：

- **限时挑战**：明确作为高难度可选内容。
- **叙事 / 探索任务**：不设置硬时间限制，或只设置宽松的"日落前到达"式软性限制。

### 4.6 巫师手动平衡，缺少自动化测试

`job_server.c` 提供的数据查看工具（`job_info`、`job_stat`、`job_hist`）很棒，但所有调整都依赖巫师在线手动执行命令。现代游戏会通过 A/B 测试、埋点、自动告警来动态调整任务收益。新引擎应把"任务产出上限"暴露为配置，并预留 telemetry 接口。

## 五、新引擎任务系统应重点补强的方向

### 5.1 题材无关的任务抽象层

从 LPC 的具体实现中抽象出以下原语，供题材包组合：

- **Quest**：一个可接取、可推进、可完成 / 失败 / 放弃的实例。
- **Objective**：目标类型包括 `kill`、`fetch`、`deliver`、`escort`、`patrol`、`interact`、`survive`、`dialogue`、`condition`。
- **Trigger**：`ask`、`enter_room`、`obtain_item`、`time_of_day`、`stat_threshold`、`faction_reputation`。
- **Reward**：经验、潜能、金钱、物品、技能熟练度、阵营声望、解锁内容。
- **Cooldown / Limit**：个人日限、世界日限、接取 CD、并行上限。

### 5.2 玩家可见的任务日志与追踪

必须提供：

- 当前任务列表与步骤高亮。
- 关键地点在地图 / 罗盘上的提示（可配置为弱引导或强引导）。
- 已完成任务的回顾与成就统计。

这是降低门槛、提升留存的最基础设施。

### 5.3 从"数值刷"到"叙事包"

借鉴《博德之门 3》《巫师 3》：把任务分为两层：

- **Routine（日常）**：低注意力、可自动化、收益有上限，明确为"背景资源获取"。
- **Episode（章节任务）**：有起承转合、选择分支、多结局，构成题材包的核心卖点。

MVP 阶段应优先保证 Episode 的创作面，Routine 可以先做极简版本。

### 5.4 失败重设计与 Roguelite 化

把 LPC 的"失败 = 惩罚"改为现代更友好的模型：

- **部分完成奖励**：超时送达也能拿到折扣奖励。
- **分支后果**：任务失败触发另一条叙事线，而不是单纯扣钱。
- **可选高难度**：保留高压限时挑战，但明确标识为"大师 / 传奇"难度，奖励更好。

### 5.5 PvP 与对抗玩法隔离

建议新引擎默认不提供强制 PvP 任务。若题材包需要：

- 必须在包元数据中声明 `pvp_required: true`。
- 提供"安全区"与"对抗区"的清晰边界。
- 强制阵营转换类任务必须得到玩家二次确认。

### 5.6 创作者友好的任务编辑器入口

LPC 的任务本质上是程序员在 `.c` 文件里填表 + 写回调。新引擎的 UGC 层需要提供：

- 声明式任务模板（YAML / JSON / DSL），覆盖 Trigger、Objective、Reward、Failure、Cooldown。
- 可复用的条件判定库（ faction、time、item、location、stat、random ）。
- 测试沙盒：创作者可以单玩家快速模拟任务流程，无需重启世界。
- 版本与溯源：每个题材包任务可追溯创作者、版本、消费埋点，呼应商业化架构支撑点（`docs/adr/` 中关于题材包资产元数据的决策）。

### 5.7 经济水龙头与 telemetry

继承 `job_server.c` 的"每小时产出上限"思想，但升级为：

- 任务收益上限作为题材包配置，可被引擎覆盖。
- 每个任务实例记录耗时、完成度、奖励，写入 telemetry。
- 提供自动告警：当某任务实际产出超过设计上限时自动通知运营 / 创作者。

## 六、结论摘要

LPC 任务系统的最大遗产是**把任务当作经济水龙头 + 社交对抗容器**：`job_server.c` 的每小时上限、道具即状态、声誉约束帮派资格等机制至今仍有参考价值。但它同时带有鲜明的时代烙印——**无日志、高惩罚、重随机、强 PvP、叙事薄弱**。新引擎不应复刻这些过时体验，而应保留其数值调控智慧，用现代任务抽象（Quest / Objective / Trigger / Reward / Cooldown）重新封装，并优先为 UGC 题材包提供声明式创作面、玩家可见的进度追踪，以及温和但有趣的失败处理。
