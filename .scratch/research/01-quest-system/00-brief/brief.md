# 01-quest-system 调研总则

> 本次调研属于 `.scratch/research/` 下首个玩法系统研究主题。目标是为新引擎任务系统提供基于《侠客行》LPC 源码的一手设计输入，同时引入现代游戏设计、玩家心理、商业化与 UGC 扩展视角进行批判性审视。

## 1. 调研目标

1. **忠实还原 LPC 原始细节**：基于当前仓库一手源码，细致梳理《侠客行》任务系统的实现方式、数据结构、调用链与状态流转。
2. **提取设计灵感与风险警示**：从现代任务设计、玩家心理、商业化与 UGC 创作角度，输出对新引擎任务系统可参考的方向、应避免的过时模式以及需警惕的设计陷阱。
3. **不输出 engine 接口草案**：本次调研止步于设计输入层，具体的 engine 抽象与接口设计留待后续任务单独决策。

## 2. 范围边界

### 2.1 纳入范围

- LPC 源码中所有可识别的任务类型：
  - 师门任务
  - 走镖任务
  - 帮派赏金任务
  - 门派日常任务（如明教五系、星宿搜索等）
  - 追杀 / PK 任务
  - 剧情 / 触发任务
- 任务与周边系统的交互：
  - 触发条件（对话、物品、地点、时间、NPC 状态）
  - 状态记录与持久化
  - 奖励结算（经验、潜能、金钱、物品、门派贡献、技能点等）
  - 失败 / 超时 / 重置 / 放弃机制
  - 并发限制与任务限额

### 2.2 不纳入范围

- 不做 LPC 行为等价验证。
- 不把 `engine/src/mud_engine/quest.py` 当作正确形态反向脑补。
- 不依赖旧文档结论（`docs/archive/` 仅作为必要时查阅的二手参考）。
- 不输出可直接落地的 engine 代码或接口契约。

## 3. 调研团队与职责

### 3.1 一手考古组

| 角色 | 职责 |
|------|------|
| LPC 源码考古员 | 逐目录阅读任务相关源码，输出代码清单、调用链、数据结构、关键回调与状态变量 |
| 玩法切片策划 | 从 LPC 中挑选代表性任务类型，做成玩家视角 + 数据流的玩法切片 |

### 3.2 机制抽象组

| 角色 | 职责 |
|------|------|
| 任务机制设计师 | 从源码中抽象通用机制：状态机、触发器、目标判定、奖励结算、失败/重置、并发限制、限额 |
| 引擎架构师 | 把通用机制映射到题材无关 engine 核心与题材包扩展接口，输出设计灵感与可选方向 |
| UGC 游戏专家 | 从创作者视角审视任务可扩展性，评估哪些机制适合暴露为题材包创作面 |
| 横向对比验证员 | 交叉检查各门派任务实现，找出共用模式与特例，验证核心模型的覆盖度 |

### 3.3 现代评审组

| 角色 | 职责 |
|------|------|
| 现代任务玩法设计师 | 对标当前主流 MMO / 开放世界 / 手游任务设计，评估 LPC 机制的当代可玩性与过时风险 |
| 玩家心理与留存专家 | 从动机心理学、留存曲线、心流节奏、社交压力等角度点评玩家体验 |
| 商业化与增长专家 | 从付费设计、UGC 创作者经济、题材包消费、用户增长等角度评估商业潜力 |

### 3.4 终审组（评审委员会）

由 5 人组成：玩法切片策划、引擎架构师 1 人、UGC 游戏专家、现代任务玩法设计师、商业化与增长专家。

职责：
1. 审阅所有初稿；
2. 组织红队对抗；
3. 对分歧做裁决；
4. 最终汇总成 `05-synthesis/`。

## 4. 调研方法

### 4.1 多 Agent 并行 Workflow

- **Phase 1：并行初稿**：各角色同步阅读源码并产出指定章节初稿。
- **Phase 2：红队对抗**：横向对比验证员交叉检查，评审委员会组织质询，各角色回应并修订。
- **Phase 3：评审委员会汇总**：统一文风、消除矛盾、标注未决问题，生成最终报告。

### 4.2 资料来源优先级

1. 最高优先级：当前仓库根目录下 LPC 源码（`adm/`、`cmds/`、`d/`、`kungfu/`、`clone/`、`inherit/` 等）。
2. 必要时查阅：`docs/archive/` 旧文档仅作二手参考，不可作为结论依据。
3. 禁止反向使用：`engine/src/mud_engine/quest.py` 仅允许作为事后对照，不允许用来裁剪 LPC 发现。

## 6. 关键约束

- **基于一手资料**：所有结论必须能从当前仓库源码中找到证据，证据需标注文件路径与函数/对象名。
- **全局与细节兼顾**：既要有任务系统宏观脉络，也要有代表性实例的关键实现细节。
- **现代视角批判**：对过时、不符合当代玩家习惯或商业化潜力的设计要显式标注。
- **User Stories 完整**：覆盖所有玩家可触达路径，以及巫师操作路径与系统/NPC 自动触发路径。
