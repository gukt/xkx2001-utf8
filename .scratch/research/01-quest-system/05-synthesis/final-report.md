# 01-quest-system 调研最终汇总报告

> 评审委员会：玩法切片策划、引擎架构师、UGC 游戏专家、现代任务玩法设计师、商业化与增长专家
> 审阅范围：Phase 1（01-raw-findings、02-user-stories、03-engine-insights）与 Phase 2（04-redteam-review）全部产出
> 报告性质：设计输入层结论，不输出 engine 接口契约或代码

## 执行摘要

本次调研以《侠客行》LPC 源码为唯一一手依据，系统梳理了其任务系统的实现形态、玩家路径、运营路径与自动触发路径，并引入现代任务设计、玩家心理、UGC 创作与商业化视角进行批判性审视。

**核心结论**：LPC 任务系统是一套典型的**分散式、脚本驱动、NPC 即任务机**的古典 MUD 设计。任务状态散落在玩家对象属性、临时属性、任务物品、NPC 自身存档与全局 daemon 中，触发方式覆盖对话、物品交付、击杀、地点、时间、随机事件与 NPC 自主行为，奖励由 NPC 脚本直接修改玩家属性结算。这种模式在表达力上极为自由，但也带来了状态不可见、失败高惩罚、随机性失控、强制 PvP、叙事薄弱等显著的时代烙印。

**对新引擎的启示**：任务系统应作为题材无关的 engine 核心 service 重写，保留生命周期管理、触发器、目标判定、统一奖励通道、限时/计时、持久化与运营观测等最小原语；将门派/帮派/声望等具体内容留在题材包；必须内建玩家可见的任务日志、进度追踪、失败保护、PvP opt-in、难度分层、跨任务配额与创作者经济护栏。声明式 QuestDef 可覆盖 MVP 简单任务，事件驱动状态机与受限脚本钩子应作为复杂任务的必要扩展层，而非远期可选项。

**红队裁决**：红队提出的"状态载体不统一""目标类型伪通用""缺少 UX 质量原语""pay-to-win 风险""UGC 经济护栏"等质疑被**采纳或部分采纳**；对"经济水龙头""道具即状态""声誉约束"等机制的补刀被**部分采纳**，即在保留其设计智慧的同时必须软化玩家侧体验。

## LPC 任务系统现状总览

### 1. 整体特征：分散式脚本驱动

《侠客行》没有统一的"任务对象"或"任务管理器"。任务的发放、判定、奖励全部写在具体 NPC、房间、物品与 condition 脚本中。每个 NPC 都是一台"任务机"，通过 `set("inquiry", ...)` 注册任务话题，玩家输入 `ask <npc> about <topic>` 后进入 NPC 的 `ask_job()` / `accept_object()` 等回调。任务生命周期由脚本自行维护，引擎层只提供底层指令、condition 心跳与 `job_server.c` 宏观调控。

### 2. 核心文件与职责

| 类别 | 代表文件 | 职责 |
|------|---------|------|
| 通用奖励伺服器 | `/clone/obj/job_server.c` | 登记各 job 的 exp/pot 上限，按耗时与完成度计酬，维护 per-user 统计与奖励直方图 |
| 通用对话入口 | `/cmds/std/ask.c` | 玩家 `ask` 指令入口，调用 NPC `inquiry` 回调 |
| 主动性任务系统控制台 | `/d/wizard/center.c` | 巫师开关系统、发布任务、查询玩家任务、删除任务、调门派参数 |
| 门派日常任务 | `/d/kunlun/npc/mingjiao_job.h`、五旗掌旗使 `.c` | 明教五旗任务的公共奖励、放弃逻辑与派发 |
| 走镖/劫镖 | `/kungfu/class/misc/linzhennan.c`、`/d/city/npc/biao_assign.h`、`/d/city/obj/biaohuo.c` | 普通走镖、高级护镖、双人护镖、劫镖、失败/超时处理 |
| 帮派赏金 | `/d/huanghe/bangjob/bangjob*.c`、`/d/huanghe/npc/bangzhu.c`、`bangzhu_duty.h` | 按经验分档的任务池、入帮、派任务、验收、授武 |
| 神龙教教务 | `/d/shenlong/sgjob/sgjob*.c`、`/d/shenlong/obj/sg_mianzhao.c`、`sg_process.h` | 寻物、威逼入教、追杀玩家、自动教务进程 |
| 星宿日常 | `/d/xingxiu/xx_job.h`、`/kungfu/class/xingxiu/ding.c` | 找毒虫、扣虫、修炼成毒丹、交付 |
| 御林军守门 | `/d/beijing/npc/duolong.c`、`helper.c`、`outer_gate.h` | 派守门任务、生成杀手、奖励结算、职级晋升 |
| 任务计时器 | `/kungfu/condition/biaoju.c`、`ypjob.c`、`lyjob.c`、`gb_job.c`、`hz_job.c` | 限时、持续收益、随机事件、失败判定 |

### 3. 主要任务类型

| 任务类型 | 代表实现 | 核心机制 |
|----------|---------|---------|
| 师门派遣 | 少林玄慈推荐信 `/d/shaolin/npc/xuanci.c` | 对话接取、物品交付、门派/辈分/CD 校验 |
| 走镖/护镖/劫镖 | 福威镖局 `/kungfu/class/misc/linzhennan.c` | 限时护送、动态生成镖车/镖头、组队共享、PvP 劫镖 |
| 帮派赏金 | 黄河帮 `/d/huanghe/npc/bangzhu_duty.h` | 按经验分档随机任务池：寻、杀、截镖、示威、送礼、护驾、摊费 |
| 门派生产循环 | 明教五旗 `/d/kunlun/npc/mingjiao_job.h` | 采矿、造枪、挑水、砍树、挖地道，奖励忠诚度 |
| 采集/小游戏 | 星宿抓毒虫 `/d/xingxiu/xx_job.h` | `search` 小游戏、随机遭遇、物品炼制 |
| 追杀/PK | 神龙教教务 `/d/shenlong/obj/sg_mianzhao.c` | 玩家目标、击杀签名、强制入教、离线惩罚 |
| 剧情触发 | 雪山葛伦布 `/d/xueshan/npc/gelun1.c` | 对话触发、物品交付、flag 解锁 |
| 驻守/岗位 | 御林军守门 `/d/beijing/outer_gate.h` | 地点指令、波次刷新、绩效考核、职级晋升 |
| 雇佣追杀 | 谢烟客/李四/丘处机 `xie2.c` / `xiejian.c` / `qiu.c` | NPC 自主执行、写入 NPC 自身存档、到期追杀 |

### 4. 状态存储的分散性

任务状态同时存在于：

- **玩家对象属性**：`mingjiao/job`、`biao/*`、`sgjob/*`、`bangs/fam` 等。
- **玩家临时属性**：`temp/hz_job/sexit`、`temp/bangs/fam` 等。
- **任务物品**：黄河帮"帮令"记录 `job` mapping，少林推荐信记录 `owner`，水桶记录 `water_amount`。
- **NPC 自身存档**：谢烟客、李四、丘处机的任务队列写入自身 `save()` 文件。
- **全局 daemon**：`job_server.o` 保存 `stat/<job>`、`exp_hist/<job>` 等统计。

这种分散性是 LPC 灵活性的来源，也是现代引擎必须统一治理的核心问题。

## 关键发现

以下发现按主题分条列出，每条均给出证据来源。对 Phase 1 初稿之间的矛盾与红队质疑，已在括号中给出裁决。

### 1. 任务触发方式高度多样，但全部依赖对象回调链

- **对话触发**：几乎所有任务通过 `ask <npc> about <topic>` 触发，NPC 的 `inquiry` mapping 注册回调。证据：`/cmds/std/ask.c`、`/d/kunlun/npc/zhuangzheng.c`、`/d/huanghe/npc/bangzhu_duty.h`。
- **物品交付触发**：NPC `accept_object()` 判定任务物品并结算。证据：黄河帮 `/d/huanghe/npc/bangzhu_duty.h`、明教 `/d/kunlun/npc/xinran.c`、星宿 `/kungfu/class/xingxiu/ding.c`。
- **地点触发**：房间 `init()` 设置 temp flag，`valid_leave()` 阻止离开。证据：华山巡逻 `/d/village/sexit.c`、灵隐寺讲经 `/kungfu/condition/lyjob.c`。
- **时间触发**：自然时间 CD、`condition` 心跳、`call_out` 延迟、自然事件（日出日落）。证据：黄河帮 `bangs/asktime + 180`（`/d/huanghe/npc/bangzhu_duty.h`）、御林军 `is_sunrise()`（`/d/beijing/npc/duolong.c`）、枯木禅师 `event_dawn`（`/d/hangzhou/npc/kumu.c`）。
- **NPC 状态触发**：NPC 存活、在场、血量、忙碌状态。证据：黄河帮截镖检查 `children(BIAOTOU)`（`/d/huanghe/npc/bangzhu_duty.h`）、谢烟客 `auto_check()`（`/clone/npc/xie2.c`）。
- **玩家属性触发**：门派、辈分、性别、阶级、经验、技能、声望、忠诚度。证据：少林 `/d/shaolin/npc/xuanci.c`、黄河帮 `/d/huanghe/npc/bangzhu_duty.h`、丘处机 `/kungfu/class/quanzhen/qiu.c`。

> 红队裁定：采纳"触发器不是离散事件，而是对象回调链"的质疑。新引擎事件总线必须订阅玩家动作、世界心跳、NPC AI、房间状态等多源事件，而不能假设只有玩家命令。

### 2. 任务状态没有统一模型，导致可观测性差

- 玩家没有任何统一界面查看当前任务、已完成步骤、剩余时间或目标位置。证据：华山巡逻需反复 `ask 岳不群` 确认（`/d/huashan/npc/buqun.c`、`/d/village/sexit.c`）；黄河帮任务 mapping 写在帮令 `ling->set("job", job)` 中（`/d/huanghe/npc/bangzhu_duty.h`）。
- `job_server.c` 的 `job_stat` / `job_hist` 仅面向巫师，不做玩家展示（`/clone/obj/job_server.c`）。

> 红队裁定：采纳"任务信息黑箱"的质疑。任务日志与进度可视化是新引擎的最低门槛，必须作为一等原语。

### 3. 奖励结算脚本化，缺乏统一通道

- NPC 脚本直接 `player->add("combat_exp")`、`add("potential")`、`add("max_neili")`、`add("shen")`。证据：`/d/huanghe/npc/bangzhu_duty.h`、斧头帮 `/d/city/npc/ftb_zhu.c`、明教 `/d/kunlun/npc/mingjiao_job.h`。
- `job_server.c` 提供了一种半集中的结算方式：任务脚本调用 `start_job()` 与 `reward()`，按 `exp_limit * exp_rate * 耗时 / 360000` 计算经验/潜能，并提供上限保护与统计。证据：`/clone/obj/job_server.c::reward_func()`。
- 各任务奖励公式高度异构：黄河帮击杀 `bonus = job["bonus"] * job["max"] / (combat_exp + 1000)`，黄河帮截镖 `bonus = exp * 120 / (exp + combat_exp)`，明教 `BASE + random(add_cc)`，斧头帮根据搜索范围、击杀比例、牺牲次数动态调整 rate。

> 红队裁定：采纳"奖励通道应强制统一"的质疑。引擎必须禁止脚本直接写玩家核心属性，所有奖励走受控 Effect 通道，并写入审计日志。

### 4. 失败/超时机制以惩罚为主，而非收益折损

- 走镖失败扣 100 两银行存款，公开羞辱并永久标记 `biao_stealer` 失去接镖资格。证据：`/d/city/obj/biaohuo1.c`、`/d/city/npc/biao_assign.h`。
- 神龙教 PK 目标下线后，系统每日 `cancel_pk()` 对追杀者施加 `sg/failure` 与 `sg/exp` 惩罚。证据：`/d/shenlong/junk/cancel_pk.h`。
- 明教放弃任务按工种扣除 `mingjiao/cc` 忠诚度，忠诚度过低则无法放弃。证据：`/d/kunlun/npc/mingjiao_job.h`。
- 一品堂青铁令超时仅增加 `yipin/failure` 计数；灵隐寺讲经超时自然结束；华山巡逻超时结束任务。

> 红队裁定：采纳"高压失败是流失源"的质疑。新引擎必须把失败重新设计为"收益折损 + 叙事后果"，取消公开羞辱与永久资格剥夺。

### 5. 随机性贯穿全程，但缺少玩家可控的风险-收益决策

- 走镖目的地随机、奖励随机、高级护镖有 1/12 概率被转为劫镖。证据：`/d/city/npc/biao_assign.h`、`/kungfu/class/misc/linzhennan.c`。
- 星宿找毒虫依赖福缘与属性判定，搜索中可能随机传送、被伏击、踩石头受伤、瓦罐破碎。证据：`/d/xingxiu/xx_job.h`、`/d/xingxiu/obj/waguan.c`。
- 丐帮密函在背包里有概率随机掉落或被偷。证据：`/kungfu/condition/gb_job.c`。

> 红队裁定：采纳"随机性缺少可控代理感"的质疑。随机事件应给玩家选择（高风险高奖励 vs 低风险低奖励），并移除不可控的随机丢失关键任务物品。

### 6. 强制 PvP / 强制阵营转换是高风险设计

- 神龙教 `forcejoin` 可强迫其他玩家改变阵营。证据：`/d/shenlong/obj/sg_mianzhao.c`。
- 走镖红镖可被任意玩家 `open` 劫走。证据：`/d/city/obj/biaohuo1.c`。
- 黄河帮"示威"任务要求击杀非本帮玩家，"截镖"要求劫系统镖车。证据：`/d/huanghe/npc/bangzhu_duty.h`。

> 红队裁定：采纳"强制 PvP 是负向筛选器"的质疑。新引擎默认不提供强制 PvP 任务，对抗玩法必须 opt-in，并限制在特定对抗区/服务器。

### 7. 任务奖励与经济调控已有雏形，但玩家感知为"刷"

- `job_server.c` 的每小时 `exp_limit` / `pot_limit` 是运营级经济水龙头。证据：`/clone/obj/job_server.c`。
- 斧头帮 `adjust_rate()` 根据 20 分钟无人完成情况自动提高奖励，本质是再激活机制。证据：`/d/city/npc/ftb_zhu.c`。
- 黄河帮、神龙教、明教五行旗、少林挑水等核心循环叙事包装薄弱，本质是"数据表 + 随机抽取"的经验水龙头。

> 红队裁定：部分采纳。经济调控思想值得保留，但必须藏在运营后台，不让玩家感知为"刷"；数据表日常必须包装成叙事才能上线。

### 8. 任务状态载体不统一，不能简单归约为玩家中心 QuestProgress

- 状态同时存在于玩家属性、临时属性、任务物品、NPC 存档与全局 daemon。证据：见"现状总览"第 4 节。
- 雇佣追杀型 NPC 的任务状态持久化在 NPC 自身，执行器是 NPC AI 而非玩家。证据：`/clone/npc/xie2.c`、`xiejian.c`、`/kungfu/class/quanzhen/qiu.c`。
- 组队走镖让两名玩家共享 `biao/dest` 与 `biao/dest2`，镖车所有者为两人。证据：`/kungfu/class/misc/linzhennan.c`。

> 红队裁定：采纳"状态载体必须多元"的质疑。统一模型应承认物品凭证、NPC 侧队列、全局统计、共享任务实例等合法载体。

### 9. 动态对象与空间查询是复杂任务的普遍需求

- 斧头帮 `ftb_zhu.c` 调用 `mapdb` / `traverser` 在世界中随机选房间并生成刺客，记录 `obj_list`。证据：`/d/city/npc/ftb_zhu.c`。
- 黄河帮"截镖"动态创建 `BIAOTOU` 并全局计数限制最多 10 个，"护驾"生成帮主分身并 `set_leader(me)`。证据：`/d/huanghe/npc/bangzhu_duty.h`。
- 走镖生成镖车 `biaoche` 与镖头跟随。证据：`/kungfu/class/misc/linzhennan.c`。

> 红队裁定：采纳"动态对象/空间查询应作为 engine 原子能力"的质疑。任务生成的对象必须注册到任务作用域并在任务结束时自动清理，引擎应提供全局配额管理。

### 10. 时间模型混合多种语义，不能仅支持 wall-clock 定时器

- LPC 使用 `time()` 比较的自然 CD、`condition` 心跳 tick、`call_out` 一次性延迟、`chat_chance` 轮询、自然事件（日出日落、event_dawn）。证据：`/d/huanghe/npc/bangzhu_duty.h`、`/kungfu/condition/*.c`、`/d/hangzhou/npc/kumu.c`、`/d/beijing/npc/duolong.c`。

> 红队裁定：采纳"时间模型不是简单倒计时"的质疑，但拒绝"完全移除限时"的极端建议。Timer 应作为可选挑战模式，默认任务优先使用宽松或软性时间限制。

### 11. 剧情/触发任务与数值任务应区分对待

- 雪山葛伦布"还愿供佛"交付酥油罐后仅获得 `temp("marks/酥", 1)`，用于解锁后续区域/剧情，没有经验/潜能/金钱奖励。证据：`/d/xueshan/npc/gelun1.c`。
- 其真实价值是"世界状态 flag + 区域准入"，更接近叙事锁而非任务实例。

> 红队裁定：采纳"剧情推进型无奖励任务难以被统一奖励模型覆盖"的质疑。引擎应区分 Quest 实例与 Narrative Flag / Region Lock。

### 12. 运营工具已存在但权限模型混乱

- `/d/wizard/center.c` 提供系统开关、手动发布/删除任务、查询玩家任务、调门派参数等命令。证据：`/d/wizard/center.c`。
- `/clone/obj/job_server.c` 提供设置奖励上限、查看统计直方图、模拟任务开始/奖励等命令。证据：`/clone/obj/job_server.c`。
- 权限检查不一致：`start_system` 只检查 `wizardp`，`job_cut all` 还需过 `can_used()` 白名单。

> 红队裁定：采纳"权限分级要明确"的质疑。新引擎应区分系统开关、数据查询、数据修改、奖励发放四类权限，危险操作需二次确认与审计。

## User Stories 汇总

以下提炼为三层故事地图：玩家路径、巫师/运营路径、系统/NPC 自动触发路径。每条故事均已在 Phase 1 初稿中标注证据来源。

### 1. 玩家路径故事地图

#### 1.1 接取任务

- 作为符合门派/声望/经验条件的玩家，我想与 NPC 对话接取任务，以便获得成长资源。
- 作为已有任务的玩家，我想确认是否能同时接取多个任务，以便规划活动。
- 作为不符合条件的玩家，我想了解被拒绝的原因，以便调整角色方向。

#### 1.2 执行任务

- 作为持有任务凭证的玩家，我想将物品交付给目标 NPC，以便完成任务。
- 作为执行护送任务的玩家，我想将目标 NPC/物品安全送达指定地点，以便结算奖励。
- 作为执行巡逻任务的玩家，我想依次经过指定区域并自动打卡，以便推进进度。
- 作为执行采集/生产任务的玩家，我想在指定地点使用工具完成动作，以便积累进度。
- 作为执行限时任务的玩家，我想清楚看到剩余时间与失败后果，以便管理风险。

#### 1.3 完成任务

- 作为完成目标的玩家，我想回到任务发布者处领取奖励，以便获得经验/潜能/金钱/物品/声望。
- 作为组队完成任务的玩家，我想与队友共享进度与奖励，以便协作。
- 作为首次完成某条任务链的玩家，我想获得独占奖励或成就，以便展示身份。

#### 1.4 失败/放弃/超时

- 作为任务失败的玩家，我想获得部分奖励或补救路径，而不是被公开羞辱或永久封禁。
- 作为想放弃任务的玩家，我想了解放弃的代价，以便决定是否继续。
- 作为离线的玩家，我希望断线后任务进度可恢复，而不是直接失败。

#### 1.5 社交/对抗任务

- 作为偏好 PvE 的玩家，我想选择不参与 PvP/对抗任务，而不损失核心进度。
- 作为选择参与对抗的玩家，我想在明确边界内挑战其他玩家，以便获得高风险高回报。

### 2. 巫师/运营路径故事地图

#### 2.1 系统开关与全局控制

- 作为高级巫师，我想在特殊活动或版本更新时开启/关闭全服主动性任务系统。
- 作为巫师，我想按题材包/世界实例/玩家群灰度开关任务系统，以便控制风险。

#### 2.2 监控与调试

- 作为巫师，我想查看当前任务系统是否已开启，以及各门派/任务池的运行状态。
- 作为巫师，我想查看某个玩家当前的任务数据与执行进度，以便定位异常。
- 作为巫师，我想查看所有在线玩家的任务执行情况，以便掌握全局。
- 作为巫师，我想查看某个任务的奖励分布直方图，以便判断数值平衡。
- 作为巫师，我想查看某个任务按玩家的详细完成统计，以便发现刷取行为。

#### 2.3 干预与补偿

- 作为巫师，我想手动发布/停止某个任务，以便处理异常或举办活动。
- 作为巫师，我想删除某个玩家或全服的任务数据，但危险操作需二次确认。
- 作为巫师，我想强制完成/放弃某个玩家的任务并发放补偿，以便原子化处理 bug。
- 作为巫师，我想调整某个任务的奖励上限，以便在线平衡经济。

#### 2.4 配置与日志

- 作为巫师，我想调整门派任务出现比率、贡献度基数、幸运值、金钱系数等参数。
- 作为巫师，我想查看任务系统运行期间的所有开关与参数变更日志，以便审计。

### 3. 系统/NPC 自动触发路径故事地图

#### 3.1 定时刷新与 NPC 自主派生

- 作为系统，我按世界状态动态生成任务目标 NPC（如镖头、刺客、帮主分身）。
- 作为系统，我按玩家在线状态与区域规则自主决定追杀 NPC 是否出手。
- 作为系统，我按自然时间/时辰刷新任务窗口或杀手波次。

#### 3.2 超时清算

- 作为系统，我在任务倒计时归零时判定失败、自然结束或给予完成标记。
- 作为系统，我在任务进行中以小概率触发随机事件（如密函丢失、刺客刷新）。

#### 3.3 限额检查

- 作为系统，我在玩家申请任务时检查个人/世界日限、冷却、等级匹配与职位空缺。
- 作为系统，我在奖励结算时确保单任务每小时产出不超过题材包上限。

#### 3.4 状态重置

- 作为系统，我在任务完成后删除玩家任务键、清空任务物品状态。
- 作为系统，我在任务异常中断或玩家下线时清理动态生成的任务对象。
- 作为系统，我在 NPC 血量过低或目标逃跑后让 NPC 返回老巢并满状态重置。

#### 3.5 地点/环境自动标记

- 作为系统，我在玩家进入巡逻任务房间时自动打卡。
- 作为系统，我在玩家试图离开任务区域时阻止其离开（可配置）。

## 对新引擎任务系统的设计建议

以下按优先级排列，并区分"必须吸收"、"可参考"、"应警惕/废弃"。

### 必须吸收

#### 1. 题材无关的最小核心概念

引擎核心层必须包含以下概念，具体实现由题材包配置：

- **Quest / QuestDef / QuestInstance**：任务定义与玩家侧运行时实例。
- **Objective / Condition**：目标与条件，但不应固化为"寻/杀/护送"等名词，而应抽象为"事件 + 守卫条件 + 副作用"三元组。
- **Trigger**：接取触发、推进触发、完成触发、超时触发，支持多源事件（玩家命令、NPC AI、世界心跳、房间状态）。
- **Reward**：通用奖励表（经验、潜能、金钱、物品、技能熟练度、声望、贡献、称号、外观），必须走引擎统一审计通道。
- **Condition / Timer**：限时与状态条件，但 Timer 应定位为可选挑战模式，而非默认生命周期。
- **Log / QuestProgress**：统一的玩家任务进度组件，进存档，并支持玩家可见的任务日志。

#### 2. 统一奖励通道与审计

- 禁止脚本或 DSL 直接调用 `player->add("combat_exp")` / `add("potential")` 等核心属性写操作。
- 所有奖励通过引擎 Effect/奖励通道下发，自动做上限保护、溢出保护与日志记录。
- 每个任务事件（开始、完成、失败、奖励领取）记录 `pack_id`、`creator_id`、`quest_id`，支撑消费/参与度埋点与创作者分成。

#### 3. 玩家可见的任务日志与进度追踪

- 必须提供当前任务列表、已完成步骤高亮、剩余目标、倒计时、失败后果、奖励区间。
- 提供地图/罗盘弱引导（方向/距离），可选强引导（路径高亮）。
- 已完成任务回顾与成就统计。

#### 4. 失败保护与共情设计

- 超时/失败应提供部分奖励或补救任务，而非资源损失 + 社交羞辱。
- 取消公开羞辱、永久资格剥夺、失败惩罚递增。
- 放弃任务仅损失机会成本，若必须扣除声望/贡献则提供挽回任务。

#### 5. PvP / 对抗玩法严格隔离与二次确认

- 默认不提供强制 PvP 任务。
- 劫镖、雇佣追杀、强制阵营转换、帮派示威必须得到目标玩家二次确认，或限制在特定对抗区/服务器。
- 新手保护期（如经验低于阈值）不可被劫镖/追杀。

#### 6. 动态对象与任务作用域管理

- 任务生成的 NPC/物品必须注册到任务作用域，任务结束/放弃/超时时自动清理。
- 引擎提供全局配额管理（同类型动态对象上限），替代手写 `children()` 过滤。
- 空间查询、区域/房间元数据应通过逻辑 ID 而非物理路径访问。

#### 7. 跨任务配额与经济护栏

- engine 统一维护每日/每周任务配额与跨任务共享冷却，防止多 NPC 轮换刷取。
- rate 强制 clamp 到题材包声明区间，负奖励与超上限奖励需要二次确认/白名单。
- 高价值奖励绑定"首次完成"或"账号级别"，降低工作室套利空间。

#### 8. 权限分级与审计

- 区分系统开关、数据查询、数据修改、奖励发放四类权限。
- `job_cut all`、参数重置、上限修改等危险操作需二次确认并进入不可篡改审计日志。

### 可参考

#### 9. 声明式 QuestDef（方案 A）作为 MVP 默认实现

- 覆盖对话接取、同处一室 NPC 判定、交物品完成、flag 完成、固定货币奖励。
- 对 UGC 创作者最友好，引擎可完全控制生命周期，便于验证、测试与 AI 导航。
- 适合实现 MVP 中的送信/寻物/flag 类简单任务。

#### 10. 事件驱动状态机（方案 B）作为复杂任务扩展层

- 覆盖击杀计数、进入房间、限时超时、随机分支、动态生成 NPC 等。
- 在创作者可控性与表达力之间取得平衡，天然适合可视化编辑器。
- 适合巡逻、限时击杀、护送等多步骤任务。

#### 11. 受限脚本/行为树钩子（方案 C）作为必要补充

- 覆盖动态地图遍历、自适应奖励、复杂小游戏、NPC 自主追杀等极端玩法。
- 必须配套沙箱、资源配额、审计日志，并强制注入 UX 护栏（任务日志、失败缓和、玩家确认）。
- 红队质疑其"可 1:1 复刻高摩擦设计"的风险被采纳，因此方案 C 应受限使用，而非无限制开放。

#### 12. 经济水龙头思想

- 保留 `job_server.c` 的"每小时产出上限"思想，但抽象为通用"奖励速率上限"，作为题材包配置。
- 隐藏在运营后台，不让玩家感知为"刷"。
- 提供自动告警：当某任务实际产出超过设计上限时通知运营/创作者。

#### 13. 声誉/阵营系统对任务资格的约束

- 保留"行为有后果"的世界观一致性，但必须提供：当前声誉值可视化、行为如何影响声誉、如何挽回、锁定内容前的明确警告。
- 阵营驱逐应引入申诉/挽回任务，而非一次性永久惩罚。

#### 14. 道具作为任务状态的可视化提示

- 关键任务道具可保留为视觉/交互提示，但任务进度必须由引擎持久化，道具丢失不应导致进度丢失。
- 适用于高沉浸感题材包，但需明确区分"进度保存"与"交互凭证"。

### 应警惕/废弃

#### 15. 把 NPC 脚本直接作为任务系统

- LPC 中每个 NPC 都自己实现接取、判定、奖励，导致逻辑碎片化、难以检索、无法统一日志。
- 新引擎必须将任务生命周期从 NPC 脚本中抽离。

#### 16. 用玩家身上物品充当唯一任务凭证或进度存储

- 物品丢失/复制/被窃会导致任务状态异常。
- 任务进度应由引擎统一持久化，物品仅作可选的交互凭证。

#### 17. 公开羞辱、永久资格剥夺、失败惩罚递增

- 走镖 `biao_stealer`、神龙教公开羞辱、岳不群直接逐出师门等机制对现代玩家容忍度过高。
- 应废弃或彻底改造为温和的后果表达。

#### 18. 强制 PvP / 强制阵营转换

- 神龙教 `forcejoin`、任意玩家劫镖、帮派示威等机制对新手与 PvE 玩家极不友好。
- 应废弃默认实现，仅作为 opt-in 对抗玩法存在。

#### 19. 随机丢失关键任务物品

- 丐帮密函随机掉落、瓦罐随机破碎等机制触发习得性无助。
- 应废弃，改为玩家可控风险或预警信号。

#### 20. 把任务专属宏观调控（job_server）当作通用任务核心

- `job_server.c` 本质是日常/刷怪任务的运营平衡器，不是通用任务模型。
- 应作为经济护栏能力保留，但不纳入任务定义核心。

#### 21. 把世界级门派政治系统混入任务核心

- `/d/wizard/center.c` 管理的门派幸运、金钱系数、策略、势力、贡献度属于武侠题材包的世界事件/阵营系统。
- 应作为题材包扩展，不是题材无关任务抽象。

#### 22. 反脚本验证转嫁给玩家

- 神鹰镖局镇西北的"数学题"验证思路会把压力直接转嫁给正常玩家。
- 应废弃，反作弊必须在后台完成。

## 未决问题与后续工作建议

### 未决问题

1. **任务状态载体的统一程度**：是否必须将物品凭证、NPC 侧队列、全局统计全部映射到玩家中心 `QuestProgress`，还是保留多元载体但提供统一查询接口？
2. **方案 A/B/C 的边界**：MVP 阶段具体哪些任务走声明式，哪些必须走状态机或受限脚本？需要结合 MVP 场景清单（新手村/城镇/门派/野外/官道/交通/坐骑）进一步拆解。
3. **PvP / 对抗玩法的 opt-in 形态**：是在同一世界内设置"对抗区"，还是通过独立世界实例/服务器规则实现？
4. **订阅与付费的精确边界**：在不 pay-to-win 约束下，"节省时间"与"花钱买进度"的精确界限是什么？需要单独的商业化设计专题。
5. **UGC 创作者经济的分成公式**：平台抽成比例、创作者信誉分、审计后结算的具体算法尚未定义。
6. **任务失败与叙事后果的默认模板**：引擎是否需要提供一套默认的"软失败"模板供题材包复用？
7. **离线/断线进度恢复**：任务计时器在玩家离线时是否暂停、是否支持断点续做？

### 后续工作建议

1. **输出 engine 任务系统接口草案**：基于本报告的最小核心概念，启动专门的接口设计工作，产出 ADR。
2. **MVP 任务清单拆解**：将华山村、扬州、少林寺、野外/官道、渡口/坐骑等 MVP 场景中的任务映射到声明式/状态机/脚本三层。
3. **UX 原型设计**：基于"任务日志、进度追踪、失败保护、难度选择"等必须项，产出玩家界面原型。
4. **经济模型专题**：在 CLAUDE.md 支撑点 6 框架下，细化双货币、订阅、创作者分成的具体规则与护栏。
5. **UGC 安全沙箱设计**：明确受限脚本的能力边界、资源配额、审计日志与自动下线机制。

### 附录 D：补充审阅——题材包创作层表面

本附录在最终报告初稿完成后补充，基于 `03-engine-insights/ugc-surface.md` 对题材包创作层表面做专项审阅。

**核心结论**：LPC 任务创作是"以 NPC 脚本为粘合剂，把玩家状态、任务物品、条件计时器、任务池、全局伺服器拼接起来"的过程。新引擎若想让题材包创作者高效产出任务，必须把高频模式声明化，同时保留受限脚本钩子作为复杂玩法的必要扩展。

**应声明化的高频能力（覆盖约 60%-70% 常见任务）**：
- 任务定义：ID、名称、接取/完成 NPC、是否可重复、互斥关系；
- 目标类型：寻物/交付、击杀指定 NPC、到达/巡逻房间、守卫指定地点；
- 触发器：对话关键词、物品交付、击杀事件、进入/离开房间；
- 奖励：经验、潜能、金钱、物品、门派贡献、忠诚度；
- 限时与失败条件：倒计时、超时清算、放弃逻辑。

**必须保留脚本钩子的复杂形态（约 30%-40%）**：
- 动态目标生成（如按玩家经验自适应选择任务池）；
- 护送/劫镖中的移动对象与 PvP 逻辑；
- `search` 小游戏与随机遭遇；
- NPC 自主任务（谢烟客/李四/丘处机的雇佣追杀）；
- 自适应奖励公式（如 `/d/city/npc/ftb_zhu.c` 的动态地图遍历）。

**创作者友好性评估**：
- 容易声明化：对话接取 + 单一交付判定类任务（葛伦布还愿、普通师门任务）；
- 必须保留代码：跨多个房间/对象的复杂状态机（走镖、帮派赏金、神龙教教务）；
- 需要引擎强力支持：状态统一、进度可见、失败保护、调试工具，否则创作者会把状态重新写回分散的 dbase 键值对，复刻 LPC 的混乱。

**详细分析请阅读**：`03-engine-insights/ugc-surface.md`。
