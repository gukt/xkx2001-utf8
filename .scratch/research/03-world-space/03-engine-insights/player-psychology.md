# 玩家心理与留存：世界空间层体验评估

> 角色表 3.3「现代评审组 - 玩家心理与留存专家」。视角：动机心理学、留存曲线、心流节奏、社交压力。
> 证据规则：每条结论标注 LPC 一手来源（文件路径 + 函数/对象名）。无地图/导航命令的判定基于 `cmds/std/` 全量盘点。
> 关键背景：本引擎单机承载、单进程单 World（ADR-0009），MVP 题材包以「华山村 + 扬州 + 少林 + 官道 + 水陆交通 + 坐骑」为场景清单（CLAUDE.md 架构不变量第 7 条），故「社交压力」一节会显式区分「原版多人语境」与「单机 MVP 语境」的差异，避免把多人机制原样搬入单机。

---

## 摘要（核心结论）

6414 房间的大世界是一把双刃剑：它提供强烈的「江湖辽阔」初见冲击，但在**纯方向移动 + 零地图 + 零导航**的基底上，探索欲会在缺乏渐进反馈时迅速转为迷路挫败；移动系统叠加了「人体力 / 负重 / 马匹体力」三重消耗，且耗尽即 `unconcious()` 硬切断（`feature/damage.c`），无软着陆带，构成 D1 流失高风险点。昼夜时段广播（`natured.c`）是唯一稳定的环境节拍，但天气系统（`weather_msg` 5 档）在当前 daemon 中是**未接线的死代码**，沉浸感打了折扣。交通层（渡船/船只）在原版是多人社交触点，但 `ship.c is_owner()` 霸船与翻船物品全毁（`do_drop()`）的惩罚烈度，不能照搬进单机 MVP。建议守住 6 条体验底线（见第六节）。

## 一、探索动机：6414 房间大世界的探索欲驱动与疲劳阈值

### 1.1 探索欲的驱动源

世界规模本身是第一驱动力。35 个区域命名差异极大（`d/REGIONS.h region_names`）：西域白驼山、关外长白山、东海桃花岛、辽东神龙岛、藏边大雪山、飘渺峰灵鹫宫……区域名本身就是「想去看看」的钩子。最大区域 beijing 625 / dali 467 / city 441 / xingxiu 388 / shaolin 368 房间（brief §2.1 引用），玩家在任何一个区域内部都有足够「未走过」的房间维持新鲜感。

房间 `long` 文案是第二驱动。典型钩子如 `d/village/alley1.c` 的「正人君子是不会往那边走去的」、`d/village/eroad1.c` 的「往东就是出村的路了」——文案主动在出口方向上埋下悬念，诱导玩家朝特定方向移动。这是纯文本 MUD 的探索动机主引擎：**每条 `long` 都是一次手写的小型悬念广告**。

门派归属（少林/武当/华山等）是第三驱动，但属于任务/成长系统范畴，不在世界空间层调研范围，此处仅标注其为探索的长期目标锚点。

### 1.2 疲劳阈值：缺乏渐进式发现反馈

问题在于：6414 房间的世界**没有渐进式发现机制**。玩家走到一个新房间，反馈只有 `look` 输出的 `short` + `long` + 出口列表（`cmds/std/look.c look_room()`）。没有任何「已探索房间数 / 区域」「本区域完成度」「此处距<地标>多远」之类的外在奖励梯度。

这意味着探索欲的衰减速度完全取决于玩家内生好奇心，而外在反馈几乎为零。心理学上，无反馈的持续探索会快速从「好奇」滑向「漫无目的」，再滑向「我到底在干嘛」的疲劳。疲劳阈值在缺乏地图的情况下会被进一步压低（见第二节）：玩家不知道「还要走多远」，不确定性本身就是认知负荷。

### 1.3 疲劳的具体触发点

跨区官道是疲劳的集中爆发点。`d/village/hsroad1.c` -> `hsroad2` -> `hsroad3` -> `sexit`（华山村通往武当的官道链），每段 `cost=1`，纯步行每步消耗 2 精力（`cmds/std/go.c` line 229 `me->add("jingli", - env->query("cost")*2)`）。一条官道动辄数段到十数段，玩家在没有「还差几步」反馈的情况下连续 `go north`，是典型的注意力耗散场景。

## 二、迷路挫败：纯方向移动 + 无地图的迷失感与流失风险

### 2.1 关键事实：玩家没有任何通用导航工具

这是本次调研最硬的一条发现。对 `cmds/std/` 全量盘点（apply/ask/beg/...共约 50 个命令），**没有任何 map / where / locate / path / recall / track 命令**。唯一叫 `locate` 的是船上的 `inherit/room/ship.c do_locate()`，只给海里级模糊方位，且对非巫师有 10% 随机扰动（`ship.c` line 447-450 `locx = (locx * 9 + random(2 * locx))/10`）。

存在一个后台地图生成器 `clone/obj/genmap.c`（BFS 从 `/d/beijing/kedian` 出发遍历，`start_mapping()`），以及查询接口 `clone/obj/mapdb.c`（`query_room_exits()` / `query_map()`）。但二者都是**系统/任务侧**的寻路数据库：`genmap.c` 注释明确写「since it is really expensive and may cause server crash, we try to be very prudent here」（`MAX_NODE=5`，每次 `call_out` 只处理 5 个房间以防崩），`mapdb` 的消费者是 `d/city/npc/ftb_zhu.c`、`d/beijing/gulou2.c`、`d/wudang/sheshenya.c` 等特定 NPC/任务的局部 BFS（`gulou2.c` 还做了 `strsrch(room_path,"/d/beijing/") != 0` 的区域裁剪）。**没有任何玩家命令暴露这套地图**。玩家无法输入「去扬州北门」自动寻路。

### 2.2 迷失感的形成链路

玩家进入房间后能看到的全部信息（`look.c look_room()` line 42-61）：

- `short`（房间名，如「小巷」）
- `long`（描述段落）
- 户外则追加 `NATURE_D->outdoor_room_description()`（时段描述，`natured.c` line 144-147）
- 出口列表：`这里明显的出口是 north 和 east。`（出口是**英文方向名**，`look.c` line 56-60）

注意出口列表**不包含目标房间的任何信息**——你只知道「north 能走」，不知道「north 通向哪」。玩家要判断自己是否走对方向，只能依赖 `long` 文案里的手写线索（如 `eroad1.c`「往西通向村子的中心」），而这条线索是否存在于每个房间完全取决于作者是否写了。

跨区时这个问题被放大。`d/village/hsroad1.c` 的 `long` 只说「南边通向扬州城」，但 `hsroad2` / `hsroad3` 是否都有方向暗示未保证。玩家在 134 房间的扬州城（`d/city/`，`ls | wc -l`）或 625 房间的北京城里，**只能靠出口名逐房试错**，没有任何「我在城内哪个片区」的区域归属提示。

### 2.3 流失风险评级：D1 高危

新手从华山村（33 房间）进入扬州城（134 房间）是典型的从「可记忆规模」跨入「不可记忆规模」的临界点。在无地图、无导航、无区域归属提示的条件下，新手在扬州城内的第一次迷路极可能是流失触发点。这对应留存曲线的 D1 断崖：玩家不是「不喜欢这个世界」，而是「在世界里找不到自己」。

唯一缓解是 `long` 文案手写线索，但这是逐房间的人工兜底，覆盖率和准确率都无机制保证。

## 三、移动疲劳：逐房间移动 + cost + 负重 + 坐骑体力衰减的累积负担

### 3.1 三重消耗叠加

移动消耗在 `cmds/std/go.c main()` 里同时检查/扣减三件事：

1. **负重**（`go.c` line 49-50）：`me->over_encumbranced()` 为真则「负荷过重，动弹不得」。负重逻辑在 `feature/move.c`（`add_encumbrance` / `over_encumbrance` line 16-29），超载即硬阻断移动。
2. **人体力 jingli**（`go.c` line 72-73）：`me->query("jingli") < me->query("max_jingli")/10` 时「精疲力尽，动弹不得」。注意这是 1/10 阈值，不是 0。
3. **步耗**（`go.c` line 228-230）：`userp(me)` 时 `me->add("jingli", - env->query("cost")*2)`，普通房间 `cost=1` 即每步 -2 jingli；`jingli <= 0` 直接 `me->unconcious()`。

骑马时再加一层：

4. **马匹体力**（`go.c` line 225-227）：`rided->add("jingli", - env->query("cost")*2)`，且骑手有 1/5 概率额外 -2（`if(random(5)==0) me->add("jingli",-2)`）；`rided->query("jingli") <= 0` 时马匹 `unconcious()`。

### 3.2 马匹衰减的硬切断

`clone/horse/horse.h condition_check()` 给了三档预警，但档位之间落差过大、且终端状态过于严苛：

- `jingli <= max_jingli/3`：「大口大口地喘着粗气」（`say()`，line 37-40）——预警档
- `jingli <= 30 && > 20`：「只在喘气，渐渐地快跑不动了」（line 32-35）——警示档
- `jingli <= 10`：**马匹昏厥 + 骑手坠落 + `receive_wound("qi", 150, ...)`**（line 18-30），「跌掉两颗门牙」

150 qi 伤害对一个低血量玩家可能是致命的，而坠骑发生在马匹 `jingli<=10` 时——此时玩家上一档看到的是「jingli<=30 喘气」，中间 20 的区间没有任何额外提示，玩家很难判断「还能骑多久」。坠落伤害缺乏软着陆：从「还能骑」到「坠马重伤」之间没有「马匹拒绝前进」的中间态。

### 3.3 昏迷的后果是清零式惩罚

`feature/damage.c unconcious()`（line 105-）的后果：

- `set("jing", 0); set("qi", 0); set("jingli", 0);`——**三条核心属性全部归零**
- `set_temp("block_msg/all", 1)`——屏蔽所有消息
- `disable_player(" <昏迷不醒>")`
- `call_out("revive", random(100 - query("con")) + 30)`——30~120 秒后醒来

对玩家而言，体力耗尽（`go.c` line 230 `jingli<=0 -> unconcious()`）等同于一次小型死亡：醒来时三属性全空，等于被传送回「需要长时间恢复」的状态。这在「我只是在赶路」的语境下，惩罚烈度与行为严重性不匹配——玩家不是战败，只是走累了。

### 3.4 累积负担的心理叠加

三条消耗条 + 一条马匹条 + 出口试错成本，玩家在长途移动时需要同时管理多个数值。认知负荷与「无地图」叠加后，移动从「探索」退化为「数值管理 + 记路」。这是移动疲劳的核心来源：不是单步重，而是**多线程管理 + 无空间反馈**的复合负担。

`horse.h init()` 里有一处缓解：房间有 `resource/grass` 时马匹吃草恢复 jingli（line 48-56），但这是被动触发且依赖房间配置，不是玩家可控的休息机制。

## 四、心流节奏：昼夜 / 天气 / 时段广播对沉浸与节奏的作用

### 4.1 时段广播：唯一稳定的环境节拍

`adm/daemons/natured.c update_day_phase()`（line 54-77）在每个时段切换时 `message("outdoor:vision", day_phase[...]["time_msg"] + "\n", users())` 向**所有户外玩家**广播。8 时段数据见 `adm/etc/nature/day_phase`：

| 时段 | length(游戏分钟) | event_fun |
|------|------|-----------|
| dawn | 240 | event_dawn |
| sunrise | 120 | event_sunrise |
| morning | 180 | event_morning |
| noon | 180 | event_noon |
| afternoon | 180 | event_afternoon |
| evening | 180 | event_evening |
| night | 120 | event_night |
| midnight | 240 | event_midnight |

注释（`natured.c` line 46-48）明确「1 minute == 1 second in RL」，即 `call_out` 的 delay 数值等同于现实秒数。总和 1440 = **24 分钟现实时间一个游戏日**，每时段 2~4 分钟现实。

### 4.2 心流正面

时段文案（`day_phase` 的 `time_msg` / `desc_msg`）质量较高，且带 ANSI 色彩（`[1;36m` 等），如 dawn「东方的天空中开始出现一丝微曦」、evening「太阳的馀晖将西方的天空映成一片火红」。这些广播为户外玩家提供环境节拍感，是纯文本 MUD 沉浸感的重要支柱。`event_sunrise()`（line 83-97）自动存档，给玩家一个隐性的「每日」仪式锚点。

`go.c` line 101-103 还把时段接入玩法：`day_shop` 房间在 `event_night` / `event_midnight` 时「晚上不开，请天亮了再来！」——时段不仅是装饰，还 gating 了商店。这种「环境->玩法」的轻度耦合是心流的好素材。

### 4.3 心流风险

**风险一：天气是死代码。** `natured.c` line 11-17 定义了 `weather_msg` 5 档（万里无云 / 几朵云彩 / 白云飘 / 厚云层 / 乌云密布），但全仓库 `grep weather_msg` **唯一命中就是定义本身**，没有任何函数读取或切换它。`outdoor:vision` 频道的唯一发送者就是 `natured.c` 自己。即当前 daemon **没有动态天气**，环境变量只有 8 个时段。沉浸感因此少了一个本应存在的维度——玩家永远不会遇到下雨、起雾，`long` 里写「黑暗、肮脏的小巷」永远不变。

**风险二：广播频率对单机注意力是负担。** 24 分钟现实一天、每 2~4 分钟现实一次广播，意味着玩家在一段持续的探索/战斗中会被频繁打断。多人 MUD 时代这是「世界在呼吸」的群体氛围；单机 MVP 里同一玩家反复收到「太阳开始西沉」会迅速从沉浸退化为噪音。

**风险三：`event_common()` 的系统负担与心流无关。** `natured.c` line 100-142 的 `event_common()` 每时段遍历所有 `livings()` 做 `environment` 检查、并对所有 `users()` 跑 `UPDATE_D->inventory_check`（line 132-141）。这是纯系统侧的清理/校验，对玩家心流零贡献，反而占用了「时段切换」这个本该沉浸的时刻。这属于性能侧议题（详见性能专家报告），此处仅从体验角度标注：**时段切换的心流价值被系统例行公事稀释了**。

## 五、社交压力：渡船等待 / 船只所有权 / 坐骑竞争等社交触点

> 本节区分「原版多人语境」与「单机 MVP 语境」。原版社交机制在单机下要么失效、要么需重新定位为「与 NPC/世界的压力」。

### 5.1 渡船：被动社交停留

`inherit/room/ferry.c` 的渡船周期：`do_yell("boat")` -> `check_trigger()`（`yell_trigger` 锁防重复叫船，line 80）-> `call_out("on_board", 15)` -> `call_out("arrive", 20)` -> `call_out("close_passage", 20)`，全程约 55 秒（line 90/111/138）。

- **多人语境**：55 秒同步等待窗口是天然的社交停留点，多人共渡形成被动互动。错过上船则 `exits/enter` 被删（`on_board()` line 108），需重新 `yell`，制造「等下一班」的群体节奏。
- **单机语境**：无其他玩家，社交停留失效。55 秒等待退化为纯等待——对单机玩家这是体验空窗，需要用环境叙事（艄公对话、水景描写）或机制（等待时可做的事）填补，否则就是干等。

### 5.2 船只：霸权与毁灭性惩罚

`inherit/room/ship.c` 有两处对玩家体验影响极大的设计：

**霸船机制**（`is_owner()` line 475-482 + `do_start`/`do_go`/`do_stop` line 80-82/293-295/328-330）：船上若有 `combat_exp` 高于你的其他玩家，你无法 start/go/stop，提示「长这么大连一点江湖规矩都不懂？」。这是**明确的社交压迫机制**——高战力玩家可以独占船只。

- **多人语境**：这是江湖权力等级的空间投射，有设计意图。
- **单机语境**：无其他玩家时此机制完全失效；若有 NPC 船客则需重新定义其行为。**不能照搬进单机 MVP**。

**翻船惩罚**（`do_drop()` line 513-537 + `time_out()` line 49-53 + `navigate()` 触礁/暴风 line 129-141）：`time_out` 在 900+random(500) 秒无操作后翻船；`navigate()` 中触礁（`jiaos` 坐标匹配）或远海暴风（`drop_factor=1` 且 `weather==2`）翻船。`do_drop()` 的后果是：所有玩家 `unconcious()` + **物品全毁**（`destruct(invofusr[m])`，唯一保留 `tie lian` 铁链，line 523-526）+ 随机冲上岸。

这是**毁灭性损失**。对一个可能在船上携带全部家当的玩家，一次触礁 = 归零。`navigate()` 里 `navigate/wait > 5` 时「船夫们把大家都扔进了海里」（line 187-191）也是同类硬惩罚。惩罚烈度与「航海」这个行为的可预期性严重不匹配——`do_lookout()` 只给模糊方位（`tot > 72` 时「大海茫茫」，line 386-388），`do_locate()` 非巫师有 10% 扰动，玩家在信息不足的情况下承担全损风险。

### 5.3 坐骑竞争与占有

`clone/horse/` 22 种马匹，`rider`/`rided` 是单一占有关系（`horse.h` + `baima.c`）。高价值马匹如 `baima.c`「白色的大宛马」`set("ability", 4)` / `max_jingli 630` 是稀缺资源。

- **多人语境**：抢马、偷马是社交冲突点。
- **单机语境**：竞争消失，马匹退化为个人财产管理。但 `condition_check()` 的坠骑风险（见第三节 3.2）依旧存在，且玩家无法在 UI 上看到马匹体力数值（只有 `say()` 喘气文案），管理是盲操作。

### 5.4 官道抢劫：携带贵重物品触发

`d/village/hsroad3.c`（`init()` line 28-47）：玩家携带 `value >= 10000` 的非货币物品时 `set_temp("rob_victim", 1)`，`random(3)==1` 触发草寇拦截；`valid_leave()`（line 49-57）则在草寇在场时阻断离开「爽快的将宝贝交出来」。

- 这是「富者自危」的社交压力变体——在多人里它制造「露富被劫」的紧张；在单机里它仍是有效的 PvE 压力，但需注意：玩家若不知「携带贵重物会触发」，突袭式拦截会带来不公平感。触发条件对玩家不透明（无提示「你携带的财物引人注目」）。

## 六、必须保护玩家的体验底线（建议机制）

以下 6 条是基于上述证据的体验底线建议。每条标注触发它的 LPC 证据。这些建议不触及 engine 接口设计（按 brief §2.2 不输出接口草案），仅从玩家心理给出方向性约束。

### 底线 1：最小方位感——房间至少标注当前区域归属

- **证据**：`d/REGIONS.h` 已有 35 区域的 `region_names` 映射；`look.c look_room()` 输出 `short`/`long`/出口，但**无区域归属**。
- **问题**：玩家在 134 房间的扬州城内不知道自己在「城东」还是「城西」。
- **底线**：房间呈现应暴露当前区域（`REGIONS.h` 已有数据），让玩家随时知道「我在哪片」。这不是自动寻路，只是方位感的最低反馈。

### 底线 2：移动疲劳软着陆——耗尽不直接昏迷

- **证据**：`go.c` line 230 `me->query("jingli") <= 0` -> `me->unconcious()`；`feature/damage.c unconcious()` 把 jing/qi/jingli 全部归零。
- **问题**：「走累了」直接等于「三属性清零的小型死亡」，惩罚与行为不匹配。
- **底线**：借鉴 `horse.h condition_check()` 的三档预警思路（max/3 -> 30 -> 10），人体力也应在耗尽前分级提示（「你感到疲惫」->「步履沉重」->「必须休息」），耗尽时进入「走不动需休息」而非 `unconcious()`。`go.c` line 72 已有 `< max/10` 的「动弹不得」阻断，可在此之上补一档软警告，避免从「能动」到「昏迷」无过渡。

### 底线 3：坐骑体力可视化——骑手能看到马匹状态

- **证据**：`horse.h condition_check()` 只有 `say()` 文案（喘气/粗气），无数值暴露；`go.c` line 225-227 骑手扣减马匹 jingli 但骑手面板看不到；`jingli<=10` 坠骑 150 qi 伤害突如其来。
- **问题**：玩家管理一条自己看不见的数值条，坠骑惩罚又重，是盲操作 + 暴击。
- **底线**：骑手应能查询马匹体力（至少分档：充沛/疲惫/危险），并在「危险」档拒绝继续骑乘或强制下马，用机制预警取代 150 qi 的硬着陆。

### 底线 4：交通惩罚分级——取消毁灭性物品损失

- **证据**：`ship.c do_drop()`（line 513-537）翻船 `unconcious()` + 物品全毁（仅留 `tie lian`）；`time_out()`（line 49-53）900~1400 秒无操作即翻船；`do_locate()` 非巫师 10% 扰动，玩家信息不足却承担全损。
- **问题**：信息不足 + 全损惩罚 = 不公平感，是留存杀手。
- **底线**：交通意外应分级（轻损 / 重损 / 全损），全损只在玩家明显冒险（如无视暴风警告仍出海）时触发。`time_out` 的「无操作翻船」在单机下尤其不合理（玩家可能只是去上厕所），应改为暂停或安全靠岸。`tie lian` 保留逻辑可保留为「不可掉落物」机制，但不应作为唯一幸存物。

### 底线 5：导航辅助最低线——主干道给「前往<地标>」的渐进提示

- **证据**：`cmds/std/` 无任何 map/where/path 命令；`genmap.c`/`mapdb.c` 是系统侧数据库不暴露给玩家；`look.c` 出口列表只有英文方向名无目标信息；`long` 文案线索逐房间手写、覆盖不全（`eroad1.c` 有，`hsroad2.c` 等未必有）。
- **问题**：6414 房间 + 零导航 = D1 迷路流失。
- **底线**：不必做自动寻路（那会消解探索），但官道/主干道应在房间呈现里给「前方 X 抵达 <地标>」的渐进提示（距离感 + 方向感），让玩家知道自己「在接近」而非「在乱走」。`genmap.c` 已证明 BFS 距离可算（它就是做这个的），数据层面可行，缺的是对玩家的呈现。

### 底线 6：时段广播频率与天气接线——降低噪音、补回沉浸

- **证据**：`natured.c update_day_phase()` 每 2~4 分钟现实广播一次；`weather_msg` 5 档已定义但全仓库无切换/读取（死代码）；`event_common()`（line 100-142）在时段切换时跑全量 `inventory_check`，系统负担挤占心流时刻。
- **问题**：单机下高频广播变噪音；天气缺失让环境永远不变；系统例行公事与心流混在同一时刻。
- **底线**：单机 MVP 可考虑默认仅重大时段（dawn/sunrise/night/midnight）广播、其余静音或合并；把 `weather_msg` 接线（让天气真正变化）以补回环境沉浸维度；将 `event_common()` 的系统清理与时段广播解耦（不要在「太阳升起」的同时跑库存检查）。
