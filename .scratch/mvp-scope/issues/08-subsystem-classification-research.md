## Question

基于 [01-subsystem-classification-framework](01-subsystem-classification-framework.md) 已确定的四档框架("对新引擎与 MVP 题材包的设计参考价值"),对 [00-子系统发现报告](../../../docs/archive/xkx-arch/_archive/_侠客行%20MUD%20架构拆解说明书/00-子系统发现报告.md)与 13 篇架构拆解文档列出的全部 36 个子系统,起草一份归类初稿:每条子系统标注 MVP 必做 / 可选 / 现代化改造 / 丢弃 + 一行归类理由,供用户在 [09-subsystem-classification-confirm](09-subsystem-classification-confirm.md) 中审阅调整。

起草时参考 [02-engine-boundary-combat-effects](02-engine-boundary-combat-effects.md) 中"流程/机制归引擎、数值/设定归题材包"的暂定倾向,用于判断战斗、技能、条件效果等边界模糊的子系统。

## Answer

依据 [01](01-subsystem-classification-framework.md) 四档框架与 [02](02-engine-boundary-combat-effects.md) "流程/机制归引擎、数值/设定归题材包"的暂定倾向,逐条通读对应架构拆解文档（未覆盖的 14 个系统仅按 00 报告简短描述推断）后给出的归类初稿：

| 子系统 | 归类 | 一行理由 |
|---|---|---|
| 1. 广告系统 | 丢弃 | 定时推送广告消息栏是运营工具而非玩法机制，现代做法是运营后台/CMS 定时任务，对引擎或题材包设计没有独立参考价值。 |
| 2. 别名系统 | 可选 | 全局别名映射 + 单引号转 `say` 的速记设计思路本身没问题，MVP 阶段可以不做，后续按用户可自定义别名的方式直接实现即可，不需要专门重新设计。 |
| 3. 封禁系统 | 可选（2026-07-18 用户确认，原判 MVP 必做） | IP/账号黑名单实现思路本身不过时、不需要按当代用户心理期望重新设计，但与 05 号票"运维观测后置"的优先级顺序一致，可以和运维观测一并后置，不阻塞 MVP。 |
| 4. 频道系统 | MVP 必做 | 频道路由、订阅管理、防刷屏（重复消息拦截、谣言频道限流）这套消息分发机制是任何题材多人 MUD 都需要的核心交互能力，应归引擎；具体频道命名/门派频道限制留给题材包配置。 |
| 5. 角色系统 | MVP 必做 | 角色对象模型 + `heart_beat` 心跳驱动生命周期是引擎骨架，必须做；但双层生命值（气/精/内力等）这套具体属性维度要重新设计成题材包可配置的属性系统，而不是硬编码武侠数值。 |
| 6. 中文处理系统 | 现代化改造 | GB/Big5 编码转换在纯 UTF-8 环境已无意义直接丢弃；但数字转中文、干支纪年这类中文文本格式化函数，值得抽出重做成通用的文本本地化工具库供中文题材包复用，不必再挂成 daemon 形式。 |
| 7. 战斗系统 | MVP 必做 | 战斗结算的流程框架（选择对手→命中判定→闪避→格挡→伤害→状态播报→奖惩）是按 02 号票倾向应归引擎的核心机制，题材无关；具体气血内力数值公式和招式描述归题材包。 |
| 8. 命令调度系统 | MVP 必做 | 命令解析分发（动词识别→路径查找→权限校验→执行）是文字 MUD 交互的核心引擎骨架，必须做；具体"用文件系统扫目录当注册表"的实现方式应现代化为装饰器/插件式命令注册。 |
| 9. 编辑器系统 | 现代化改造 | 游戏内文本编辑功能是"玩家在线创作内容"的雏形，与 UGC 创作层目标方向一致，但要按现代富文本/可视化编辑体验重新设计，不能照搬命令行逐行编辑。 |
| 10. 表情系统 | MVP 必做 | `$N`/`$n`/`$P` 等变量替换的消息模板机制是文字 MUD 表达角色互动的核心引擎能力，任何题材都需要；具体表情词库归题材包内容。 |
| 11. 指纹查询系统 | 丢弃（2026-07-18 用户确认，原判现代化改造） | 与标准命令系统（29）的 who/查询功能重叠，MVP 阶段不需要独立的玩家资料卡/好友列表功能，直接丢弃，不单列一档。 |
| 12. FTP 系统 | 丢弃 | 纯粹是给巫师上传下载文件用的运维/开发工具，现代有 Git/对象存储/CI 等更好的替代方案，对新引擎无设计参考价值。 |
| 13. 查询系统 | 丢弃 | 00 报告描述过于简略（仅"处理玩家各种查询请求"），且其功能在现代架构下可直接由状态查询 API/已有命令覆盖，没有独立保留的理由。 |
| 14. 语言系统 | 丢弃 | 核心是 GB/Big5 编码选择与转换，与中文处理系统的历史遗留问题相同，现代统一 UTF-8 环境完全不需要。 |
| 15. 登录系统 | MVP 必做 | 连接→编码/账号校验→密码→（新玩家）角色创建→进入世界的状态机流程，是任何在线游戏必须的认证与入场引擎能力；具体天赋随机分配、门派选择等属于题材内容。 |
| 16. 婚姻系统 | 现代化改造 | 玩家间正式关系绑定（配偶字段、死亡自动解除）这一社交玩法思路值得借鉴，但现代玩家期望更丰富的关系类型和仪式感，应重新设计成题材包可扩展的"社会关系"框架，不照搬简单 spouse 字段。 |
| 17. 金钱系统 | MVP 必做（基础部分）/ 现代化改造（进阶部分，2026-07-18 用户确认拆分） | 单一货币持有/交易/掉落这一基础闭环仍是 MVP 必做（没钱就玩不下去）；但多币种、订阅、账本抽象这套完整商业化设计，按 06 号票原话"支撑点不要求 MVP 实现，只留位置"，降级为现代化改造，不与 MVP 绑定过紧。 |
| 18. 自然系统 | 现代化改造 | 昼夜/天气这套"心跳驱动的世界定时事件调度"骨架值得保留，但天气文案与触发规则要按现代游玩体验重新设计（不只是装饰性文字）；日出自动存档这一具体功能应归数据存储系统而非自然系统。 |
| 19. 网络守护进程系统 | 丢弃 | 内置 DNS/HTTP/FTP/邮件协议栈在现代架构下应由标准 Web 框架与云基础设施（如 FastAPI + SMTP）承担，自造这套网络协议栈没有设计参考价值。 |
| 20. 动作功能模块（/feature/） | MVP 必做 | Feature/Mixin 组合架构（数据存储/移动/命名/消息/装备等能力按需混入，而非深层继承）是整套 LPC 架构里最值得借鉴的设计思想，直接对应现代 ECS/组件化对象模型，应作为新引擎对象模型的核心参考蓝本。 |
| 21. 护甲继承系统 | 可选 | 防具按类型分槏互斥穿戴 + 属性加成叠加是常见装备系统模式，但 MVP 轻量题材包未必需要完整 9 种槏位，可作为后续扩展深度，不阻塞 MVP。 |
| 22. 角色继承系统 | MVP 必做 | `CHARACTER → NPC → TRAINEE` 继承链体现的"玩家与 NPC 共享核心行为、按需扩展驯服/AI 能力"设计，直接指导新引擎 Entity 模型的分层方式。 |
| 23. 物品继承系统 | MVP 必做 | 物品基类 + 可堆叠物品自动合并/按数量显示的设计，是背包系统的基础能力，任何题材包都需要通用物品模型。 |
| 24. 药品继承系统 | 丢弃 | 本质是"可堆叠消耗品 + 使用后效果"这一已被物品继承系统（堆叠）与状态系统（效果）覆盖的题材特化分支，不需要单独的引擎级继承体系。 |
| 25. 杂项继承系统 | 丢弃 | "杂项"本身即缺乏统一设计的大杂烩分类；其中有独立价值的具体功能（如公告板的异步社交模式）已在多人交互系统中体现，作为一个继承分类整体没有独立保留价值。 |
| 26. 房间继承系统 | MVP 必做 | 房间基类（出口/门系统/物品与 NPC 刷新/负重传递）是文字 MUD 世界构建的引擎骨架；特殊房间子类化（银行/商店/交通枢纽）的模式可直接复用于任意题材包场景设计。 |
| 27. 技能继承系统 | MVP 必做 | 技能作为独立、可扩展对象（标准接口 `valid_learn`/`perform_action`/`exert_function`）配合 `map_skill` 技能复用机制，是 UGC 创作者定义新技能内容的理想模板，应作为引擎技能框架骨架保留。 |
| 28. 武器继承系统 | 可选 | 武器类型标志位（`EDGED`/`TWO_HANDED`/`SECONDARY`）与专属击中特效（如剑破防具）的设计模式有参考价值，但不是 MVP 必须的深度，可作为后续扩展。 |
| 29. 标准命令系统 | MVP 必做 | `look`/`go`/`get`/`give`/`kill` 等基础交互命令集是文字 MUD 的最小可玩单元，必须作为引擎自带能力提供，题材包只需扩展专属命令。 |
| 30. 门派武学系统 | 现代化改造 | "专属技能池 + 身份归属 + 师徒传承"的组织设计模式值得借鉴，但具体 21 个门派的设定和技能是武侠题材内容，应重新设计为题材包可配置的"阵营/流派"框架，不直接照搬旧门派设定。 |
| 31. 状态系统 | MVP 必做 | 状态/效果独立可扩展定义 + 统一 tick 更新 + 标志位控制续存/中断的运行时机制，正是 02 号票所指"效果生命周期管理"应归引擎的典型案例，需作为通用 Effect 系统骨架；具体状态内容（蛇毒/醉拳）归题材包。 |
| 32. 技能系统 | MVP 必做 | 技能等级/学习进度追踪、技能映射复用、"多技能拖慢学习速度"的成长曲线设计，是 RPG 题材角色成长系统的核心机制，应归引擎成长系统框架，具体数值曲线题材化调参。 |
| 33. 克隆对象系统 | 丢弃 | 蓝图 + clone 是 LPC 语言特有的对象实例化机制，在 Python 中就是普通类实例化，没有独立设计参考价值；其中"背包自动加载""游玩时长转年龄"等具体点已被存储/角色系统覆盖。 |
| 34. 数据存储系统 | MVP 必做 | `F_DBASE` 的路径式属性访问（如 `set("family/master_name", ...)`）与临时/持久数据分区设计，直接指导新引擎在"内存数据 + 本地 JSON 定时存档"约束下的对象状态模型实现。 |
| 35. 头文件与配置系统 | 现代化改造 | 集中注册"标准对象/守护进程路径"避免硬编码字符串散落各处的思路值得保留，但要用现代 Python 配置管理/依赖注入机制重新实现，不照搬宏定义 + 全局 include 的方式。 |
| 36. 文档与帮助系统 | 现代化改造 | 新手引导的设计意图（出生点设施齐全、对话暗示、指令提示）值得保留，但现代玩家期望交互式教程/UI 引导而非纯文本帮助文件，需要重新设计呈现形式。 |
| 37. InterMUD 网络系统 | 丢弃 | 跨 MUD 互联协议（白名单 + UDP 广播 + 心跳保活）服务于"多个独立 LPC 站点互联成网"的旧生态，与新架构"题材包数量横向扩展、世界实例互相隔离"的方向相悖，也违反六条约束里的不搞分布式/网关。 |
| 38. 坐骑与交通系统 | MVP 必做（2026-07-18，10 号票场景选型阶段改判，原判现代化改造+MVP 不做） | 10 号票选定的 MVP 场景要求官道/野外可骑乘通行、跨区域要有水陆交通（渡口/渡船）联通，坐骑与交通的运行时机制（消耗资源换移动效率、动态出口缝合地理断点）直接成为场景可玩性的必需部分，不能再延后；具体坐骑种类/镖车任务等题材内容仍留给题材包配置。 |
| 39. 死亡与轮回系统 | MVP 必做 | 昏迷→死亡两段式判定、免死区域豁免、死亡惩罚计算与复活状态机是战斗/风险机制不可分割的一部分，按 02 号票倾向归引擎流程能力；阴间世界叙事、门派特色死亡增益（如丐帮）等题材化细节留给题材包。 |
| 40. 武林大会与竞技系统 | 现代化改造 | 报名条件校验/擂主-挑战者排队/装备公平化仲裁（统一发放标准装备）/观众下注/直播转发，构成一套可复用的"锦标赛框架"，对任何题材都有引擎级参考价值，但需按现代电竞/赛季活动体验重新设计自动化赛程调度，而非巫师手动 `start`。 |
| 41. NPC-AI 与行为系统 | 现代化改造 | `heart_beat` 驱动的概率行为树 + `attitude` 态度分类决策，是性价比很高的轻量级 NPC AI 设计，值得作为引擎骨架保留，但现代玩家期待更丰富的反应性和对话深度（如可配置行为树甚至 AI 生成对话），需要重新设计交互与内容生成方式。 |

### 档位分布（2026-07-18，含 09 号票核对 + 10 号票场景选型触发的后续改判，最终版）

- **MVP 必做**（18）：4 频道、5 角色、7 战斗、8 命令调度、10 表情、15 登录、17 金钱(基础部分)、20 动作功能模块、22 角色继承、23 物品继承、26 房间继承、27 技能继承、29 标准命令、31 状态、32 技能、34 数据存储、38 坐骑与交通、39 死亡与轮回
- **可选**（4）：2 别名、3 封禁、21 护甲继承、28 武器继承
- **现代化改造**（10）：6 中文处理、9 编辑器、16 婚姻、17 金钱(进阶部分)、18 自然、30 门派武学、35 头文件与配置、36 文档与帮助、40 武林大会与竞技、41 NPC-AI 与行为
- **丢弃**（10）：1 广告、11 指纹查询、12 FTP、13 查询、14 语言、19 网络守护进程、24 药品继承、25 杂项继承、33 克隆对象、37 InterMUD 网络

（初稿版分布为 MVP 必做 18/可选 3/现代化改造 11/丢弃 9，共 41 条；金钱系统拆成两半分别计入两档，故档位小计之和为 42；坐骑与交通系统在 10 号票场景选型阶段从"现代化改造"改判回"MVP 必做"，是本效力中唯一一条二次改判的条目。）

### 边界模糊清单——09 号票核对结论（2026-07-18）

- **门派武学系统（30）**：维持"现代化改造"——归类只约束引擎层要做成"阵营/流派"通用框架，21 门派的具体内容仍由 MVP 题材包层直接落地，两者不矛盾。
- **战斗（7）/ 状态（31）/ 技能（32）/ 死亡与轮回（39）**（用户核对时额外发现 39 也隐含同样依赖，一并处理）：维持"MVP 必做"，但明确标注这四条都依赖 [02-engine-boundary-combat-effects](02-engine-boundary-combat-effects.md) 尚未拍板的"流程归引擎"倾向——02 号票若后续改判，这四条需要连带重新评估。
- **封禁系统（3）**：改判"可选"，与 05 号票"运维观测后置"的优先级一并后置，不阻塞 MVP。
- **金钱系统（17）**：拆分为"基础持有/交易"（MVP 必做）与"多币种/订阅/账本抽象等完整商业化设计"（现代化改造，严格对齐 06 号票"支撑点不要求 MVP 实现，只留位置"的原始优先级）。
- **坐骑与交通系统（38）**：维持"现代化改造 + MVP 阶段不做"原判，标记为暂定/待定，本轮未深入裁决。
- **指纹查询系统（11）**：改判"丢弃"，与标准命令系统（29）的 who/查询功能重叠，MVP 不需要独立资料卡/好友列表功能。
- **编辑器系统（9）**：维持"现代化改造"作为方向性结论，具体形态留待 [03-ugc-dsl-design-inheritance](03-ugc-dsl-design-inheritance.md) 细化后回头核对。
