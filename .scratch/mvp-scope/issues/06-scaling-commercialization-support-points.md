## Question

用户提到"未来承载目标、商业化闭环也可以早期考虑,MVP 阶段视情况实现部分或暂不实现,但留有支撑点"——这句话需要具体化:

- "承载目标"具体指什么规模?(在线人数?并发?题材包数量?)
- "商业化闭环"具体指什么形态?(题材包市场付费?UGC 创作者分成?订阅?)
- 这些"支撑点"应该落在架构的哪些层面?(数据模型要不要提前留租户/所有权字段?题材包资产格式要不要走"可发布可计费"的结构?)

## Answer

### 商业化闭环形态

参考调研(2026-07-18,见对话记录):UGC 平台侧(Roblox/Fortnite UEFN)用"参与度分成池 + 内容直销分成"给创作者;文字 MUD 先例(Iron Realms/Achaea,20+ 年活案例)用"双货币(免费金币 + 真钱 premium 点数,可在玩家间市场互换)+ 订阅 + 明确不做 pay-to-win(花钱买加速/便利,不买压倒性优势)"。

**采纳叠加方案**:玩家侧照 Iron Realms(双货币 + 订阅 + 不 pay-to-win),创作者侧照 Roblox/Fortnite(题材包/UGC 内容的玩家消费按比例分给对应创作者)。

### 未来承载目标的规模

不追求单个题材包/单个世界的在线规模远超 MVP(不是"万人同服"路线)。扩展方式是**题材包数量的横向扩展**——很多个中小题材包并存,每个题材包的承载量级类似 MVP(ticket 05 定的 1000 在线 + 100 并发)。这与 ticket 05"不做分布式架构"完全兼容:平台层的扩展问题变成"独立运行更多个单机题材包/世界实例",不是"把一个世界做成分布式系统"。

### 架构支撑点(不要求 MVP 实现,但要留位置)

1. **货币/账本抽象**:至少两种货币(免费 + premium),且每一笔消费要能追溯到"题材包 + 物品 + 创作者"三元,这是分成机制的前提。
2. **题材包资产元数据**:带创作者归属 + 版本来源(provenance),参考旧方案 [01-关键修正与避坑清单.md](../../../docs/archive/xkx-arch/_archive/01-关键修正与避坑清单.md) §24 的 provenance 思路(MVP 阶段简化版本号即可,不强制完整溯源)。
3. **消费/参与度埋点**:能打点到题材包 ID,哪怕 MVP 阶段这个 ID 永远是"官方武侠包"。
4. **世界实例隔离**:引擎要支持"每个题材包/世界实例独立进程运行"这种形态,不是所有题材包挤在一个共享的大世界里。
