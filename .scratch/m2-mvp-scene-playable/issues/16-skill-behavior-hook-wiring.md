# 16 — SkillBehavior 钩子实际接入 + 示范招式

**What to build:** 把 03 号票定义的 `SkillBehavior` Protocol（`hit_ob`/`hit_by`/`post_action`）从"占位调用点"（02/12 号票已留）升级为"真正生效"：12 号票的 tick 战斗调度在 `resolve_attack` 结果 apply 回真实组件后，按本回合命中招式的 id 查 03 号票的 `register_skill_behavior` 注册表，若该招式声明了钩子则依次调用 `hit_ob`（命中后，可修改伤害数值或追加文案）/`hit_by`（被击中后触发副作用）/`post_action`（招式结算收尾）。新增**至少一个**使用钩子的示范招式（如"命中后触发中毒"或"被击中后触发招式特效"，具体效果内容可以是最小占位——本票目的是验证接线正确，不是设计一整套状态效果系统，Effect 生命周期机制若需要真实"中毒每回合掉血"效果，超出本票范围，可以用一句战斗播报文案 + 一次性伤害加成代替，不实现完整 buff/debuff 系统）。多数招式（罗汉拳基础招式等）继续不实现钩子，只填 `SkillData` 数值——本票需要证明"数值型招式"与"带钩子招式"共用同一套调度、互不干扰。

**Blocked by:** 03（`SkillBehavior` 协议与注册表），12（真实战斗调度路径）。

**Status:** ready-for-agent

- [ ] tick 战斗调度在命中判定后按招式 id 查 `SkillBehavior` 注册表，有则调用对应钩子；无钩子的招式行为与 12 号票落地时完全一致（零回归）。
- [ ] `hit_ob(ctx, damage) -> int | str | None`：返回值语义明确（修改伤害数值 vs 追加文案 vs 无操作，三种返回类型分别如何处理需写清楚并测试覆盖）。
- [ ] `hit_by(ctx) -> None`：被击中方触发副作用的调用时机（本回合结算的哪一步之后）明确、可测试。
- [ ] `post_action(ctx) -> None`：招式结算完成后的收尾调用，不影响本回合已产出的 `CombatRoundResult`。
- [ ] 至少 1 个示范招式使用 `hit_ob` 或 `hit_by`（如"淬毒的招式命中后追加一句中毒播报 + 下一回合额外固定伤害"，效果可以极简，重点是验证钩子确实被调度到、确实能影响后续状态）。
- [ ] 命令/tick 层测试：对比"无钩子招式"与"有钩子招式"在同一 `CombatContext` 输入下的行为差异，证明钩子调用确实生效而非空转。
- [ ] 现有测试全绿不回归（尤其 02/12 号票已有的 resolve_attack/战斗调度测试）。
