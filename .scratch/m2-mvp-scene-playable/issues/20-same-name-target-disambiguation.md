# 20 — 同名目标序号消歧：ask/attack 支持 `<名> <序号>`（60a–60c 最小档）

**What to build:** 落地 spec 用户故事 60a–60c 确认的**最小档**范围：同房间有多名同名 NPC（如城门两名"官兵"、`count: 2` 的巡逻兵）时，玩家可用 `ask <名> <序号> about <话题>` / `attack <名> <序号>`（1-based，空格分隔）指到第 N 个实例，对齐 MudOS/FluffOS `present("id N")` 的机制形状（不做行为等价，ADR-0001，只借机制形状）。这是一个**架构改动**而不是简单的字符串处理补丁：M1 的 `matching.Candidate` 现为 `(规范名, 别名序列)`，`Intent.target: str` 只是规范名字符串，执行层按名字符串查找——必须升级到"匹配结果携带具体实体引用"才能真正区分同名实例。实现建议（不强制具体形状，但需要解决同一个问题）：扩展 `matching.match_target` 或新增一个专用于"按名+序号"匹配的函数，让解析层能从"规范名 token 末尾的序号"识别出第 N 个实例的 `EntityId`；`Intent` 需要能携带这个实体引用（新增字段，或者让 `target` 在有序号时装一个可解析出实体的复合值，需要与现有 `Intent.target: str | None` 的契约测试兼容或经过明确的破坏性变更评估）；`ask`/`attack` 的执行层改造为优先使用实体引用而非重新按名查找。**无序号且多命中**时保留 M1"不确定你指的是哪个"提示（比 LPC 静默取第一个更友好，不改变现有行为）；**有序号**才消歧。**不做**全命令 present 等价（`get`/`look`/`drop` 等同名非堆叠消歧，spec 60c 标注为可选加码，不在本票范围，且需注意与 `Stackable` "同名合并" 语义并存的复杂度，留给后续独立评估）。

**Blocked by:** 12（`attack` 命令需要已经存在才能对它做消歧）。

**Status:** ready-for-agent

- [ ] `matching.py` 新增/扩展匹配能力：候选集合从"仅 (规范名, 别名)"升级为携带 `EntityId`（如 `Candidate` 追加实体引用维度，或新增专用函数，实现阶段决定并在本票内部保持内聚，不散改），支持解析"名字 + 末尾数字序号"的 token 形式。
- [ ] `Intent` 的形状变更（新增字段或调整 `target` 语义）需要更新 `test_parsing.py`/`test_commands.py` 里对 `Intent` 形状的契约测试（若存在），且不破坏无序号场景下现有全部命令的行为（回归红线）。
- [ ] `ask <名> <序号> about <话题>`：序号超出实际实例数给出明确提示（不是崩溃或静默取最后一个）；序号为 1 且只有一个实例时行为与不带序号完全一致（不强制玩家必须输入序号）。
- [ ] `attack <名> <序号>`：同上序号语义；消歧后建立交战关系的是序号指定的**具体那个实体**（用两个同名 NPC 分别打，断言两次交战关系分别落在不同 entity id 上）。
- [ ] 无序号且多命中：保留现有 `Ambiguous` -> "不确定你指的是哪个" 提示路径，**不**因为本票改动而意外变成"静默取第一个"（这是明确的行为回归红线，需要专门测试锁定）。
- [ ] 单命中（同名只有一个实例）时序号写不写都能正常工作。
- [ ] 命令层测试覆盖：`count: 2` 巡逻兵夹具下 `ask 巡逻兵 1 about ...`/`ask 巡逻兵 2 about ...`/`attack 巡逻兵 2` 分别命中正确实体；序号越界；无序号多命中仍走 Ambiguous。
- [ ] 现有测试全绿不回归（尤其 `test_matching.py`/`test_parsing.py` 现有别名匹配测试）。
