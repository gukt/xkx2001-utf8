# 03 - 复刻意义与取舍战略

> 完全复刻是否抓核心的论证 + LPC 精华/糟粕分级 + 现代替代方案 + 保真度分级表 + 取舍提案

---

## 一、完全复刻是否抓到核心：综合判定

### 判定结论：引擎行为保真抓到了核心，但存在保真分层缺失导致优先级倒置（置信度 high）

项目的核心价值主张是"DSL+Agent 让非程序员创作可玩世界"（[00](docs/xkx-arch/00-愿景约束与总纲.md) §3）。这一主张的成立依赖三个引擎能力护城河（[00](docs/xkx-arch/00-愿景约束与总纲.md) §7 五护城河前 3 条）：

1. **DSL-as-contract**：玩法规格提取是连接引擎与 UGC 的契约层。[resolve_attack.py](engine/src/xkx/combat/resolve_attack.py) 七步管线 + skill_power 完整公式 + riposte 递归（`_RIPOSTE_MAX_DEPTH=4`）+ hit_ob polymorphic 分发 = 可执行规格。**已抓到**。（置信度 high）
2. **combat 确定性回放**：DeterministicRNG seeded combat-only + CombatBridge tick 驱动。[ADR-0023](docs/adr/ADR-0023-combat-determinism-boundary-simplification-ledger.md) + [ADR-0039](docs/adr/ADR-0039-combat-tick-alignment.md)。**已抓到**（但有保真缺口，见下）。（置信度 high）
3. **themed 治理**：[governance.py](engine/src/xkx/runtime/governance.py) 确认为平台级 fail-closed Python System（治理逻辑硬编码模块级常量，UGC 只能触发不能修改规则）。**框架已抓到**。（置信度 high）

### 但存在三重矛盾（5 视角遮蔽的跨视角分歧）

#### 矛盾 1：保真状态判定矛盾

- **architecture 视角**：称"六项关键架构不变量源码已落地"（arch-c2, high confidence），用 resolve_attack "Effect 交织不 mutate"佐证。
- **lpc-fidelity 视角**：同时揭示多处保真缺口--resolve_attack 纯函数+apply_effects 后置 apply 丢失 receive_damage 即时副作用语义（arch-c1, high）、rng 收口约 15 处 < 29（arch-c1b, high）、wound 触发逻辑简化为固定 1/4 概率（本次直接取证确认）。
- **根因**：对"不得先算后 apply"不变量的解读分歧。ADR-0023 认为保留交织顺序（ledger）即合规，但 lpc-fidelity 指出 ledger 仅保顺序不保即时副作用语义（receive_damage 返回值丢失、victim.qi 快照导致后续判定偏移）。

#### 矛盾 2：themed 治理定性矛盾

- **architecture**：themed 治理=平台级 fail-closed Python 机制。
- **game-design**：themed 治理=图形 MMO 无法复制的想象力（内容层）。
- **game-design 红队 2**：themed 治理框架是机制（[governance.py](engine/src/xkx/runtime/governance.py) 确定性代码），但规则设计（天雷惩罚什么/阴间 5 段剧情/法院判什么）是内容设计。
- **综合裁决**：themed 治理的**框架**是机制（fail-closed Python，可验证确定性），**规则设计**是内容（想象力，题材特定）。两者不互斥但需显式区分。00§7:259 原文"themed 治理范式是想象力治理元宇宙"说的是治理的想象力非文本媒介。（置信度 high）

#### 矛盾 3：铁律优先级倒置（最严重）

- **声明的铁律是 stub**：CLAUDE.md 将"do_attack 七步文本与副作用交织不可分离"和"PronounContext 必须携带 viewer（三元组）"列为铁律不变量。但 [resolve_attack.py](engine/src/xkx/combat/resolve_attack.py) `_render` 是"最小占位符替换" stub（完整代词求值"见子系统 9/13，T6 不做"）。即声明的铁律当前是 stub 实现。
- **应可改编的内容被当铁律完整复刻**：grind（cubic level³/3 + 惩罚式死亡 + potential 门控）被 [ADR-0037](docs/adr/ADR-0037-m3-combat-e2e-demo-design.md) 决策 6 显式列为"保真是项目核心不变量"（针对 darba/samu grind 强度），但同一决策确认 e2e 用直接设玩家属性绕过 grind（"强玩家赢 darba/设 longxiang-banruo 30 拜 samu"），CLI demo"不强求通关"。
- **实践中 grind 确有松动但无原则覆盖**：[death.py](engine/src/xkx/runtime/death.py) L467-514 已对 LPC skill.c L133 的 learned 覆盖 bug 做 greenfield 修正（learned 累加记所有技能进度），证明实践中"LPC=绝对规格"已有松动，但无原则覆盖此松动。
- **即**：声明的铁律（PronounContext）是 stub，应可改编的内容（grind）被当铁律完整复刻且验证中被绕过。此优先级倒置仅 game-design 红队 2 点明。（置信度 high）

## 三、保真度分级表

| 级别 | 定义 | 示例 | 偏离后果 |
|---|---|---|---|
| **必须 (MUST)** | 机制铁律，偏离即破坏引擎核心能力或硬门禁 | tick=1s+compute<100ms / combat-only 确定性 / JSON 原子写 / themed 治理 fail-closed / CombatKernel 主题无关 / 三层粒度 Theme>ModulePack>UGC CPK | 项目存续风险 |
| **应该 (SHOULD)** | LPC 机制精华，保真影响行为等价但非存续级 | do_attack 七步交织顺序（即时副作用语义可接受收敛需 golden trace 验证）/ PronounContext 三元组 / 死亡轮回 / 门派灵魂归题材包 / grind 机制（数值可调） | 行为等价偏差，影响保真判定 |
| **可选 (OPTIONAL)** | 内容层，可改编或简化 | 金庸衍生角色名改编 / 阴间 5 段剧情细节 / 随机文案措辞 / command 提示语 / 房间 long 描述 | 不影响引擎行为，影响内容完整性 |
| **可弃 (CAN-DISCARD)** | 1990s 设计选择/历史包袱 | save_object 无原子写 / 50ms tick / Big5 编码 / phantom call / learned 覆盖 bug / PVP 依赖子集（单机死代码） | 已弃或应弃 |

### 当前实现与分级表的偏差

| 分级项 | 当前状态 | 偏差 | 建议 |
|---|---|---|---|
| do_attack 七步交织（SHOULD） | 顺序保留但即时副作用语义有缺口（receive_damage/wound/中途停战） | SHOULD 级实现未达 SHOULD 级保真 | golden trace 前置评估缺口是否可接受收敛 |
| PronounContext 三元组（SHOULD） | _render stub，完整代词求值未实现 | SHOULD 级实现是 stub | 补全代词求值（含 query_close random(5) 收口） |
| grind 机制（SHOULD，数值可调） | 完整复刻但验证绕过 + 实践已松动（death.py greenfield 修正） | 分级未明确，实践已松动但无原则覆盖 | 显式声明 grind 数值为"可调参数"，external 玩家测试后根据留存数据调整 |
| wound 概率（SHOULD） | 固定 1/4（rng.chance(4)），LPC 有 4 种概率 | SHOULD 级保真缺口 | 补全 4 种 wound 概率分支 |

## 四、取舍提案（哪些本轮可直接实现）

### 提案 1：补全 wound 概率 4 分支（Impact: Medium / Effort: Low / Risk: Low）

[resolve_attack.py](engine/src/xkx/combat/resolve_attack.py) L229 `rng.chance(4)` 改为按 attack_type + kill/non-kill 上下文分 4 支：

- 空手 kill: random(4) -> 1/4
- 武器 kill: random(2) -> 1/2
- 空手非 kill: random(7) -> 1/7
- 武器非 kill: random(4) -> 1/4

依据：[golden trace](engine/tools/golden_trace/baseline/meta.json) lpc_formula_map 明确记录。工程量小，不影响纯函数确定性（都是 rng 调用）。**本轮可直接实现**。

### 提案 2：补全 receive_damage 即时性（Impact: High / Effort: Medium / Risk: Medium）

[resolve_attack.py](engine/src/xkx/combat/resolve_attack.py) L254 exp 判定 `rng.rand(victim.max_qi + victim.qi) < damage` 读快照 qi，应读 receive_damage 后值。

两种方案：
- **方案 A（保守）**：在纯函数内模拟 receive_damage 即时减免（damage 重新赋值 + qi 减免后值用于后续判定），保持纯函数可重放性。
- **方案 B（激进）**：重构为可中途终止的执行模型，破坏纯函数。

**建议**：先评估方案 A 是否可覆盖三类缺口（receive_damage/中途停战/riposte speed），若 golden trace 概率分布对照无显著偏差则确认 ADR-0023 让步合理。**本轮先评估，不直接实现**。

### 提案 3：补全 PronounContext 代词求值 + random(5) 收口（Impact: Medium / Effort: Medium / Risk: Low）

[rankd.c](adm/daemons/rankd.c) L600-604 query_close 无性+eunach 分支含 random(5)（1/5 返回异性称呼）。当前 [resolve_attack.py](engine/src/xkx/combat/resolve_attack.py) `_render` 是占位符 stub。

**建议**：子系统 9/13 实现时补全代词求值，纳入 random(5) 收口到 DeterministicRNG。**本轮不直接实现（T6 不做），但应在保真分层 ADR 中标注为 SHOULD 级待补全**。

### 提案 4：显式声明 grind 数值为"可调参数"（Impact: High / Effort: Low / Risk: Low）

在保真分层 ADR 中显式声明：
- grind 机制（cubic 曲线/potential 门控/惩罚死亡）= SHOULD 级保真（MUD 品类核心）
- grind 数值参数（level³/3 系数/potential 消耗速率/惩罚力度）= 可调参数，external 玩家测试后根据留存数据调整
- 当前 death.py greenfield 修正（learned 累加修 LPC bug）是"修 bug 非改设计"的合理实践，应声明为"bug 修正非保真松动"

**本轮可直接实现**（写 ADR 声明即可，不改代码）。

### 提案 5：修正 04 §三 L179 阵法合击规格来源定位（Impact: Low / Effort: Low / Risk: Low）

[04](docs/xkx-arch/04-迁移路径与避坑清单.md) L179 写"s_combatd.c 提取阵法合击规格"，但 [s_combatd.c](adm/daemons/s_combatd.c) 只有 anubis_attack（武器反击非阵法），grep team_attack/formation/阵法 无命中。阵法规格真实来源在 kungfu/skill/{buzhen,pozhen,heji}.c 等（全仓约 28 文件）。

**本轮可直接实现**（文档修正）。

## 五、"LPC 是规格源、行为等价验证"铁律是否仍合理

### 关键事实（5 视角均未整合）

1. **LPC driver 现已可运行**：[ADR-0009](docs/adr/ADR-0009-original-driver-runnable.md) + [golden trace baseline meta.json](engine/tools/golden_trace/baseline/meta.json) 证实 FluffOS 3.0 Rosetta 2 macOS 实际运行中，录制了 14 回合 combat + 完整登录会话 + 概率统计。运行时对照从"不可能"变为"可能但未充分利用"。
2. **部分 LPC 行为是 1990s 设计选择非"灵魂"**：death.py 已做 greenfield 修正修 LPC learned 覆盖 bug，证明实践中"LPC=绝对规格"已有松动但无原则覆盖。
3. **golden trace 覆盖面极小**：当前仅 1 场景 14 回合 combat + 1 登录会话，对 6414 房间中绝大多数内容的行为等价验证仅靠静态 spec 提取（规格自证）无运行时兜底。

### 综合裁决（置信度 medium）

这条铁律**总体仍合理**，但需显式分层：

- **"LPC 是规格源"作为方法论根基**仍成立：greenfield 重写需理解 why（规约）而非复制 what（行为快照），ADR-0009 决策 3 已明确"golden trace 是辅助非主线"。
- **"行为等价验证"应扩展为三层**：
  1. **规格自证**（当前主线）：从代码提取输入输出契约 + hypothesis 属性测试。适用：源码可读的子系统。
  2. **运行时对照**（当前辅助）：golden trace 录制旧系统运行时行为作为基线测试。适用：难以从代码静态推断的路径（valid_leave 命中行为/combat 七步副作用交织时序）。
  3. **算法级推断**（ADR-0061 三档）：源码缺失但存档可读的子系统，接受算法级推断+基线数据断言。适用：/clone/obj/job/ 等源码缺失场景。
- **应评估扩大 golden trace 覆盖面**：当前仅 combat+login。是否应扩展到 go/move/quest/对话/门派逻辑等非 combat 路径，以替代部分静态 spec 自证？此提案可能动摇项目方法论根基，需用户复审拍板。

**不挑战"LPC 是规格源"本身**，但挑战"行为等价验证=仅静态 spec 提取"的单一验证框架，建议扩展为三层验证。此提案 Impact:High / Effort:High / Risk:High，需用户拍板。
