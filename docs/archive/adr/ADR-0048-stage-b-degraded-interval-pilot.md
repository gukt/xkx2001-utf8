# ADR-0048：抽样校准阶段 B 方案修正（降级区间承诺 + pilot 纠偏 + 回归按需后置）

- 状态：已采纳（工时承诺部分被 [ADR-0056](ADR-0056-abandon-effort-estimation-ai-batched-migration.md) 退役，pilot 副产出保留）
- 日期：2026-07-14
- 阶段：0 任务 6 抽样校准实验阶段 B（方案修正）
- 关联：[ADR-0047](ADR-0047-greenfield-effort-semantics.md)（被修正）/ [ADR-0046](ADR-0046-sampling-calibration-methodology.md) / [05](../xkx-arch/05-第三轮专家对抗复审报告.md) 专家 6 工时模型 dissent / [04](../xkx-arch/04-迁移路径与避坑清单.md) 验收"工时承诺有数据支撑" / 评审团 workflow（4 方案 + 3 评委）

## 背景

[ADR-0047](ADR-0047-greenfield-effort-semantics.md) 决策阶段 B 手工实测 80 样本迁移工时，分层均值 × 层规模 + t 分布置信区间推算全量。阶段 B 启动前调研（2 explorer 实读样本）+ 架构专家评审团（4 方案生成 + 3 维度对抗评审）发现三处承重问题：

1. **总工时远超预期**：仅 58 个 pending/logic 约 76 小时（high tier 单函数 120-190 min），全量 80 样本实测是 80-100 小时 / 10+ session 工程，单 session 仅测 1-3 个样本。
2. **分类有系统性偏差**：implemented/logic"工时≈0"偏低（实际 ~7 min/个，普遍有后置副作用分支待补）；pending/data 有误分类（`snake.c:init` 含战斗触发+新命令，实为 logic，30-45 min）。
3. **80 样本对"点估总工时"过宽**（40-50 够），窄 t 分布置信区间是自加码--[04](../xkx-arch/04-迁移路径与避坑清单.md) 验收只要"工时承诺有数据支撑"，非"窄点估 + CI"。

评审团三方收敛（收益/成本风险/约束契合三评委均排名方案 3 第一、方案 2 垫底），建议降级为区间承诺 + pilot 纠偏。

## 决策

1. **目标降级**：从"80 样本窄 t 分布置信区间"降为"难度分级 + 类比基准 + high-tier 定点抽查的区间承诺"。04 验收"工时承诺有数据支撑"= 有实测锚点 + 类比外推的区间，非窄点估 + CI。承认 80 样本是自加码。

2. **执行载体 = pilot 12-16 样本**：high-tier 过采样，含纠偏样本（`snake.c:init` 误分类、`death_penalty` 后置分支）。pilot SOP 模板固化测量一致性 + 共享桩预建（`SKILL_D`/`recognize_apprentice`/`prevent_learn`/`is_spouse_of`/`receive_damage` 一次性补建，避免每样本重复建桩）。

3. **分类纠偏 = 定量审计**：每样本强制记 `corrected_status`/`corrected_kind`，统计误分类率。把 [ADR-0047](ADR-0047-greenfield-effort-semantics.md) 决策 6"类别级近似可回调阈值"从拍脑袋升级为定量触发（误分类率超阈值则回调分类规则并重分层）。

4. **外推 = pilot 实测 + 类比基准**：pilot 12-16 数据点 + 类比基准（30 文件校准工时 + M3 已实现 C1-C6 工时 + 调研 3 个 high-tier 数据点 120-190 min）-> 区间承诺。

5. **退路**：若纠偏后分类漂移大或 high-tier 尾部离群 -> 升级接力补测（回到 [ADR-0047](ADR-0047-greenfield-effort-semantics.md) 分批实测路径）。

6. **窄区间升级 = 按需后置**：仅当未来出现对外硬交付日期、需窄置信区间时，用 pilot 16 样本作回归起点补样到 40-55，激活协变量回归外推（评审团方案 1）。**按需触发，非现在投**--把 80-100h 的窄区间投入变成"按需触发"而非"现在就投"。

7. **LLM 辅助否决纳入工时承诺主线**：评审团三评委一致否决方案 2（LLM 辅助双层工时模型）。理由：
   - **工时语义污染（触碰 [ADR-0047](ADR-0047-greenfield-effort-semantics.md) 决策 1 红线）**：greenfield 工时语义要回答"人迁完需多少工时"，LLM 辅助回答的是"用这套 LLM 管线时人需校验多少工时"--偷换问题。LLM v0 质量是随模型/prompt 漂移的外生变量，承诺不可复现、不可外推到"不使用该 LLM"的真实迁移场景。
   - **成本被低估（评委实读代码核实）**：方案 2 声称复用 `content_gen`/`conformance`/`measure_revision` 脚手架，但核实发现 `content_gen` 无 logic 函数 prompt builder（只有 npc/skill/quest/room）、`conformance.py` 252 行全 combat 专用（无通用骨架）、`measure_revision` 是 scene/yaml 级修订比非函数级。三处"复用"实为从零新建，脚手架 6-8h 大概率超支。
   - LLM 辅助若仅为加速代码生产，可独立于工时估算单独决策，**不纳入"工时承诺"口径**。

8. **代码处置 = 一次性测量目录**：pilot 阶段产出的 Python 等价代码落 [engine/tools/sampling/pilot/](../../engine/tools/sampling/)，测完即丢，不污染 `src/xkx`。理由：方案已降级为"抽查 + 类比"非全量迁移，代码量小且可能返工（误分类纠偏），落 `src/xkx/content/` 会污染。阶段 B-2 批量补测（若触发退路）再决定是否集成。
