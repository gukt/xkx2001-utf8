# 第三轮专家对抗复审报告

> 本报告是第三轮专家对抗复审的过程记录与决策溯源，与 [02-三个开放架构问题裁决](02-三个开放架构问题裁决.md) 互补：02 是"决定做什么"（权威裁决），本报告是"如何得出"（过程与 dissent）。6 位专家独立复审 + 3 个开放问题交叉质证 + 主持人收敛裁决，共 10 个 agent。原始结构化产出见 `_archive/_第三轮专家复审原始产出/`（10 份 JSON + digest）。

---

## 一、复审背景与方法

### 背景

前两轮（36 Agent 研究 + 8 专家对抗复审）产出 v2 架构，但用户判定"目标虽是标杆+UGC+DSL，但有些结构设计弄复杂了"，提出六条收缩约束（去分布式、运维后置、单机网关、纯 Python、内存+JSON 存档、3 个开放问题待定）。本轮在 v2 基础上，对"收缩后架构 + 3 个开放问题"做对抗复审并收敛。

### 评审团（6 位，精简聚焦）

| # | 专家 | 侧重 |
|---|---|---|
| 1 | 游戏服务器架构师 | 单机分层、1000+100 可行性、JSON 存档崩溃安全、存储策略迁 PG 路径 |
| 2 | 游戏引擎/ECS 专家 | 单进程 ECS、tick 预算、GIL、最小工具链、热重载、确定性范围 |
| 3 | MUD 玩法与文化专家 | Command 保真、规则引擎承载 themed 治理、题材包对玩法深度、文化保真 |
| 4 | UGC 平台/插件架构专家 | 题材包热插拔三层边界、受信任开发者扩展、规则引擎作 UGC 触发层 |
| 5 | DSL/规则引擎专家 | 规则引擎统一 condition-action、与 DSL/Command 关系、性能、是否过度设计 |
| 6 | 游戏制作人/范围管理 | 范围控制、kill criteria、里程碑现实性、可观测范围、工时模型 |

### 方法（3 阶段）

1. **阶段1 并行独立复审**：6 位专家基于同一份"v2 架构摘要 + 6 条收缩约束 + 3 个开放问题"上下文独立复审，产出结构化结论（裁定 / 该砍 / 该留 / 对 3 问题立场 / 缺口 / 承重论断）。硬指令：倾向简化、显式列"不做什么"、不得发散、收敛优先于完备。
2. **阶段2 交叉质证**：对 3 个开放问题各跑 1 个质证 agent，综合 6 份复审，显式呈现 support/oppose/最强反论/未消分歧/收敛建议（不伪造共识）。
3. **阶段3 收敛裁决**：1 个主持人 agent 综合全部复审与质证，产出最终裁决。

### 共识度

6 位专家对 3 个开放问题立场**完全一致**：Q1 有条件采纳 / Q2 否决 / Q3 有条件采纳。共识远大于分歧。整体裁定 5 位 risky + 1 位 solid--risky 不在收缩方向（一致认为正确），而在"收缩后单机细节未落地"。

## 二、6 位专家独立复审

### 专家 1：游戏服务器架构师 -- 裁定 risky

收缩方向正确且必要，v2 的 risky 很大程度源于分布式/Ray/K8s/全量事件溯源等过早复杂度，6 条约束精准拆除了这些承重风险。但单机纯 Python+asyncio+内存 JSON 存档架构在三个承重细节上未落地：

**承重论断（high）**：

1. 1000 在线+100 并发在纯 Python 单进程 asyncio 下成立，但唯一前提是 tick 预算硬性受控（<50ms）+ 非均匀 tick（仅活跃实体）+ 存档与 GC 不阻塞事件循环。否则 GIL 下单进程 tick 计算会冻结全网 I/O，这是收缩后唯一承重可行性风险。
2. 内存权威+JSON 存档 与 PG 权威+Redis 缓存 是两种不同一致性模型，策略模式存储接口必须以"持久化边界"（persist=崩溃恢复级耐久）为抽象而非"save=权威写"，否则迁 PG 时是架构变更非策略切换。
3. JSON 存档必须 write-temp+os.rename 原子写 + 事件循环外 offload + dirty-flag 分摊，否则崩溃即文件损坏或单 tick 冻结。LPC `save_object` 正是无原子全量覆盖，greenfield 不得重复此 bug。

**该砍**：Redis 全部依赖（进程内内存化）、分布式网关全套、`world://` URL 寻址、uvloop 硬依赖、JWT RS256+Redis 黑名单、题材包运行时热插拔。

### 专家 2：游戏引擎/ECS 专家 -- 裁定 risky

收缩方向健康：砍掉分布式/K8s/Ray/PG 权威态，坍缩了上一轮最重的过度设计层，核心（SparseSet ECS、战斗七步、DSL 四层、存储策略模式）在单机+内存+1000 在线下站得住。但 v2 文档未按收缩重新校准，残留四处承重隐患：

**承重论断（high）**：

1. **1s tick（LPC heart_beat 实测 `set_heart_beat(1)`）使 GIL/IO 冲突与 compiled-core 紧迫感在 1000 在线下基本 evaporate**。上一轮的 Rust-panic 是按 20Hz/50ms-tick 校准的，该假设不适用于 MUD；纯 Python 在 1s tick 下有 900ms+ I/O 预算。
2. **全仿真确定性扩展到所有 System 是阶段1 的错误范围**；combat-only 确定性才是正确范围且已是 combat-sim 等价性的前置。
3. **ECS 在 Python 是架构选择非性能优化**--SparseSet 在 C++ 的 cache locality/SIMD 优势在 Python dict 存储下不存在。不应以性能为由选型、不应驱动 compiled-core 决策。
4. **Command 模式必须只覆盖玩家发起+触发器发起的状态变更，绝不能套到 tick 驱动的 System 内部 mutation**--会制造热路径对象 churn（GC 压力）与无收益间接层。

**该砍**：全仿真确定性作为阶段1 里程碑、50ms/20Hz tick 框架、6 件工具链一等公民、热重载安全窗口协议、CombatKernel 跨 4 题材预先抽象、对象池化 GC 微优化。

### 专家 3：MUD 玩法与文化专家 -- 裁定 risky

收缩方向正确，MUD 文化保真的承重设计（七步管线、PronounContext 服务端求值、技能三层、5 灵魂系统、命令中间件管线）在单机纯 Python 下反而比分布式更容易保真--`call_other` 退化为进程内同步调用天然契合 LPC 语义，单房间串行是 MUD 固有硬约束而非缺陷，`previous_object` 信任链在单进程内可干净映射为 ActionContext。但裁定 risky 因战斗手感时序与 PronounContext 求值两个承重细节：

**承重论断（high）**：

1. **do_attack 七步管线的文本与副作用是交织不可分离的**--`hit_ob` 返回 polymorphic（string/int/mapping）可同时追加文案与修改伤害；`exp`/`jingli` 在管线中途直接 mutate；递归 riposte 嵌套产生子回合交织。任何纯函数化必须保留交织时序，不能"先算后 apply"。
2. **RANK_D 的 query_close/query_self_close 是观察者相对的二元关系函数**，依赖 `this_player()`--代词求值不是单实体属性而是 (speaker, viewer, target) 关系。PronounContext 必须携带 viewer。
3. **533 个 valid_leave 触发器证明 DSL 层1（condition->action）是 LPC 事件触发器的正确且充分的抽象层**，无需额外自定义规则引擎。

**该砍**：子系统2 分布式脚手架、ThemeRegistry 运行时热插拔、独立规则引擎、Command 扩张解读、02-v2 中 9 处 PG/Redis/NATS 残留。

### 专家 4：UGC 平台/插件架构专家 -- 裁定 risky

收缩方向对 UGC 平台是净利好：砍掉分布式/分片后，UGC 创作闭环不再背负"先验证分布式再验证创作"的串行债务，单进程纯 Python + 内存 JSON 让 CPK/manifest/依赖图能在阶段 -1 垂直切片里直接验证。

**承重论断（high）**：

1. **三层边界（核心引擎/题材包/UGC）是正确的抽象且必须早期确立**，但"热插拔"应降级为"安全窗口冷插拔"--单进程纯 Python+内存态下运行中热卸载有活跃实体的题材包是悬挂引用研究题，不可承诺。
2. **19 门派是 wuxia 题材下的内容包（module pack）不是独立题材**，v2 把门派当题材会让 schema family 机制退化为扁平命名空间。三层粒度应为 Theme > Module Pack > UGC CPK。
3. **独立的自定义规则引擎是过早抽象**--DSL 层1 condition->action 就是 UGC 触发层，系统级治理策略是题材包内 Python 策略对象，condition-action 逻辑有两处存放地就多了一处。
4. **WASM 沙箱在纯 Python 单进程收缩下不划算**--其价值是多租户进程级隔离，而收缩已排除分布式/多租户。应用 RestrictedPython 统一两层。

**该砍**：WASM 沙箱、分布式燃料聚合器、CRDT/OT 多创作者协作、内容市场/分发/计费实现、Louvain 社区检测衍生概念。

### 专家 5：DSL/规则引擎专家 -- 裁定 solid（唯一）

6 条收缩约束移除了会诱发重型规则引擎机制（分布式 fuel 聚合、PG 权威+Redis 读穿透、多分片规则一致性）的分布式/K8s 复杂度，留下一个本领域内健全的切分：层1 事件规则 DSL 管触发器、ECS System 管 tick/调度/heart_beat 仿真、命令管线中间件管玩家意图源变更、PermissionService 作唯一 fail-closed 策略引擎。

**承重论断（high）**：

1. **独立自定义规则引擎是过度设计**，层1 DSL + Python 策略对象 + ECS System 已按调度语义正确切分 condition-action 空间，无须再叠一层"统一引擎"。
2. **themed 治理（天雷/阴间/vote/法院）必须是非 UGC 可编辑的平台级 fail-closed Python**，不能落入规则引擎或 UGC 可触的规则层。

**该砍**：独立规则引擎框架、RETE/前向链/外部引擎（OPA/Drools）、Command 扩张解读、分布式燃料聚合器、themed 治理作为 UGC 可编辑规则。

### 专家 6：游戏制作人/范围管理 -- 裁定 risky

收缩方向正确--砍分布式/K8s/Rust 正是上一轮制作人警告的"greenfield 最大风险是过度设计"的直接执行，与范围纪律一致。但收缩执行留下三处承重缺口，使风险从"过度设计"转移到"隐性范围变更"：

**承重论断（high）**：

1. **存储收缩（内存+JSON）不是"换存储"而是对 v2 子系统6 的隐性重新设计**--子系统6 把 PG+Redis 贯穿于 EventStore/SnapshotStore/ReferenceStore/SchedulerStore/outbox/CommandBus 乐观并发/CPK 内容寻址/CQRS 投影，策略模式接口是对的但必须逐一标注丢失的语义（事务原子性、崩溃恢复、并发写、CAS、关系完整性）。
2. **1000 在线+100 并发在单进程纯 Python 上是从实测 MAX_USERS=50 的 20 倍跃升**，v2 共识9 已明确 GIL/GC/PYTHONHASHSEED 三重天花板，本轮约束4 又砍掉 Rust/Go 退路，且无"基准达不到"的 kill criteria--这使整个单机验证里程碑成为无 Plan B 的未验证赌注。
3. **收缩砍了基础设施复杂度但未重新排序创作平台复杂度**--4 层 DSL/Agent 编排/6 维评估/CPK+provenance/ThemeRegistry/沙箱+配额/审核 pipeline/创作者经济仍是并行范围，应显式排到引擎+核心循环验证之后而非并行，否则里程碑过载。

**该砍**：全仿真确定性扩展、逻辑热重载协议、分布式机械残留、Redis 必需基础设施、ThemeRegistry 热插拔、自定义规则引擎。

## 三、3 个开放问题交叉质证

### Q1 题材包/模块包热插拔

**support（6 位全部支持三层边界）**：三层边界是 UGC 平台天然结构且 greenfield 下应 Day1 文档化；ThemePack=启动时加载的 Python 包；门派灵魂归属武侠题材包资产；三层粒度 Theme > Module Pack > UGC CPK（human.c 19 门派实证）；ThemeRegistry 已设计为词汇表扩展；核心引擎不能硬编码武侠语义。

**oppose（6 位全部反对运行时热插拔）**：运行时热插拔是过早抽象，会迫使每个 System 从 Day1 就 theme-agnostic + version-aware 稀释武侠深度；CombatKernel 跨 4 题材预先抽象会平庸化武侠；intermud 不应被多题材绑架提前；单进程内存态下热卸载有活跃实体的题材包是悬挂引用噩梦；MVP 只支持热加载与停服卸载；当前 1 题材+1 开发者建热插拔是为不存在的受众建设基础设施。

**最强反论**：UGC 专家与制作人的对冲--三层边界必须 Day1 内建（否则核心引擎继续硬编码武侠），但又无需热插拔机制就能验证主题无关性。真正的承重问题是"CombatKernel 从武侠提取（保深度）与用非武侠微场景验证主题无关（保可移植）的张力"，这个张力不解决，三层边界无论静态还是热插拔都会落空。

**收敛**：有条件采纳三层边界，砍运行时热插拔。取最简（静态加载），冷插拔/热加载均后置。CombatKernel 实现从武侠来、验证用非武侠（阶段 -1 非武侠微场景硬门禁兜底）。

### Q2 自定义规则引擎

**support（3 位软支持统一求值入口）**：DSL/规则引擎专家主张保留"薄 RuleEvaluator 函数"作层1 IR 求值器；引擎专家主张保留固定 dispatch table；UGC 专家主张 UGC 触发器落 DSL 层1。三者都承认需要统一求值机制，但反对命名为"引擎"与建独立框架。

**oppose（6 位全部否决独立引擎）**：经典过早抽象，层1 已是规则机制再建是重复；heart_beat tick 不可被事件驱动替代故统一引擎要么与 tick 重复要么仅覆盖触发器（后者就是层1）；533 valid_leave 已证明层1 是正确抽象层；themed 治理是异构 System 非统一规则；1000 在线下事件触发式求值无 Rete 必要；自建规则引擎求值性能是新增未知数。

**最强反论**：规则引擎核心价值是"让非开发者编写大量同类规则而不碰代码"--这正是 UGC 平台长期愿景。否定独立引擎在当前阶段正确，但"层1+Python 策略对象"两分法把"统一 condition-action"真实诉求推给未来，可能产生四类代价：跨层优先级无法统一表达、审计轨迹割裂、层1 原语蠕变成事实规则引擎、themed 治理全用 Python 硬编码意味着反作弊迭代慢。

**收敛**：否决独立规则引擎（6/6 共识）。但"统一 condition-action"诉求部分成立，载体是层1 IR + 薄求值子模块 + System 三分 + 调度语义四分。themed 治理"反作弊迭代慢"是 fail-closed 的正确取舍。RETE/前向链/外部引擎永久排除。

### Q3 Command 模式

**support（6 位全部有条件采纳）**：Command 统一外部输入边界，与确定性重放 input log 天然对齐，授权与审计有明确挂载点。边界=外部输入，System.update 覆盖派生变更。

**oppose（5 位反对扩张解读与新建类层级）**：8 段管线本质已是 Command 模式不应再新建类抽象；把 tick 驱动 System mutation 包成 Command 会制造热路径 GC 压力；LPC command_hook 只处理玩家输入，强制统一会破坏战斗时序；1000+100 未达标上不应再叠加抽象；Command 须保留 LPC 同步语义。

**最强反论**："Command 仅覆盖外部意图、System tick 不走 Command"的干净边界是理想化，LPC 自身并不遵守它。`force_me`（4 个真实调用点）允许 ROOT_UID 对象调用完整命令路径--系统 daemon 与可信对象在 LPC 中本就混合了"仿真"与"命令"。若映射为 Command 则边界泄漏，若映射为 System.update 则破坏保真。二者不可兼得。

**收敛**：有条件采纳。Command = 8 段管线信封的形式化命名（不新建类层级），仅覆盖外部意图；System.update 覆盖派生变更不经 Command。force_me=PrivilegedAction（Command 变体，ROOT 门控+强制审计），**显式承认是保真让步**--边界在 force_me 处妥协以保 LPC 保真。

## 五、关键 dissent 与未消除风险

复审坚持"不伪造共识"，以下分歧与风险显式记录：

1. **CombatKernel 抽象时机张力**（Q1）：从武侠提取（保深度）与非武侠验证（保主题无关）可能冲突--武侠深度尚未完整实现时，提取的接口可能过窄（锁死武侠语义）或过宽（平庸化）。靠阶段 -1 非武侠微场景硬门禁兜底。
2. **"第二个题材真实存在"触发条件未定义**（Q1）：热插拔后置若无触发点会滑成"永不考虑"。第二题材到来时迁移成本可能被低估（门派灵魂切割、组件 schema 迁移、悬挂引用处理均未预研）。
3. **层1 原语蠕变**（Q2）：反复扩充层1 条件/动作原语可能让层1 逐步长成事实上的规则引擎。靠 KPI<15% + 判定标准（可声明式描述且跨规则复用则扩层1，需图灵完备则层3）护栏。
4. **规则冲突语义漂移**（Q2）：LPC 靠注册顺序隐式覆盖触发器命中，层1 若优先级/deny-wins 语义未严格对齐，迁移后 533 valid_leave 命中行为会漂移。靠基线测试断言原 LPC 命中行为。
5. **call_out 归属未交叉验证**（Q2）：DSL 专家独家把 call_out（694 文件/3109 处）归 ActionScheduler，其他专家未确认。需实现时明确与 ConditionSystem/EventBus 边界。
6. **force_me 边界侵蚀**（Q3）：PrivilegedAction 是"仅外部意图"规则的例外，若未来 NPC AI 或触发器大量使用会侵蚀边界。靠审计/限制调用点。
7. **派生变更审计覆盖缺口**（Q3）：System.update 的 mutation（战斗外 exp/jingli、condition 效果、heal）无 Command 级审计轨迹。战斗有副作用账本覆盖，其他 System 逐个决定审计粒度。
8. **存储收缩丢失语义**（制作人）：内存+JSON 丢失事务原子性/崩溃恢复/并发写 CAS/关系完整性/append-only 可篡改，必须逐一标注+止损线（外部测试前迁 PG）。
9. **1000+100 无 Rust 退路是未验证赌注**（制作人）：从 MAX_USERS=50 的 20 倍跃升，砍掉 Rust/Go 退路且无 kill criteria 时是 Plan B 缺失。已补 kill criteria（降级目标到 500+50 或重新评估 Rust/Go）。
10. **平台特性并行范围过载**（制作人）：4 层 DSL/Agent/6 维评估/CPK/ThemeRegistry/沙箱/审核/创作者经济远超"单机 1000+100 验证"，已显式排到引擎+核心循环验证之后。

## 六、复审结论

第三轮复审的核心价值是**识别过早抽象并砍除**，而非发现新能力。6 位专家在"收缩方向正确"上完全一致，分歧仅在收缩后的单机落地细节。三个开放问题的裁决共享同一条收敛主线：**砍掉为不存在的规模/受众建设的基础设施，把承重设计留在"单机纯 Python + 内存 JSON + 1000 在线"约束内可落地的最小集合**。

被砍的不是能力本身，而是当前阶段不该承担的复杂度；每个被砍项都附"何时重新引入"的后置指引，确保推迟而非取消。最终文档集（README + 00-04）即此裁决的落地，本报告为其决策溯源。
