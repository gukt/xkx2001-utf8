## Q1 题材包/模块包热插拔

### 问题

平台除武侠外，未来可能有学院、科幻、模拟经营等不支持或差异大的题材。是否采用题材包/模块包热插拔或插件形式提供扩展？是否开放给受信任开发者？

### 裁决：有条件采纳--保留三层边界（文档化+最小静态实现），砍运行时热插拔

### 关键设计

**1. 三层边界（文档化 + 最小实现）**

| 层 | 内容 | 可插拔性 |
|---|---|---|
| 核心引擎 | 题材无关骨架：ECS + tick + CombatKernel + DSL 编译器 + 沙箱 | 不可插拔 |
| 题材包 | schema family + 谓词/动词词汇表 + themed 治理策略 + SkillBehavior 带代码资产 | 启动时静态加载 |
| UGC | data-only：层0 YAML + 层1 规则 + 层2 对话树 | 沙箱受限 |

**2. 三层粒度**（采纳 UGC 专家划分，`human.c` 19 门派硬编码实证支撑）

> `adm/daemons/race/human.c` 第 92-185 行对武当/全真/峨嵋/少林/大理段家/雪山/血刀门/丐帮/华山/桃花岛/古墓派等 19 门派做硬编码 if-else 加成--证明**门派是 wuxia 题材下的内容包（Module Pack）不是独立题材**。v2 把门派当题材会让 schema family 退化为扁平命名空间。

- **Theme**（词汇表 + themed 治理策略） > **Module Pack**（门派/区域 CPK，受信任开发者） > **UGC CPK**（创作者内容，沙箱受限）。

**3. 门派灵魂归属武侠题材包资产**（MUD 专家）

门派灵魂（SkillBehavior Python 策略，含隐藏大招 / hit_by / auto_perform）归属武侠题材包资产，**非核心引擎硬编码，非 UGC 可创作**。`human.c` 19 门派 if-else 加成迁移时须从 race/核心层剥离为题材包内容。

**4. ThemeRegistry = 启动时加载的静态注册表**

Python dict：`theme_id -> ThemeDescriptor`。题材包 manifest 声明 `theme+version+dependencies`，CPK manifest 复用做依赖管理。**无运行时 unload/版本协商/隔离**，不另建插件运行时。

**5. CombatKernel 抽象从武侠提取**（引擎专家 + 制作人协调）

`CombatAction`/`CombatResolver`/`DamageModel` 接口**从武侠 `do_attack` 七步提取**（保武侠深度，不从 4 题材委员会预先设计）。阶段 -1 用非武侠微场景验证内核主题无关性（保可移植）--**实现从武侠来，验证用非武侠**。这两者非对立而是互补。

**6. 受信任开发者扩展 = 带代码题材包**

走审查 + 签名 + SemVer + CPK provenance，进程级无沙箱 Python（StdLib 级），非 UGC 沙箱路径。UGC 永不创作题材包本身，只在已注册题材内编辑层1。

### 边界（不做什么）

- **不建运行时插件系统**（register/unload/version/isolation 完整机制）--典型过早抽象，会迫使每个 System 从 Day1 就 theme-agnostic + version-aware，稀释武侠旗舰深度（"共享运行时平庸化"风险真实）。
- **不跨 4 题材预先设计 CombatKernel**--接口从武侠提取，不为泛化而平庸化武侠 70% 脚本灵魂。
- **不做安全窗口冷插拔**--UGC 专家中间路线本轮不采纳，等第二个题材。
- **不把 intermud 联邦绑架进多题材提前实现**--intermud 是外部多世界互联非内部分片，属后置能力。

### 最强反论（已回应）

UGC 专家与制作人的对冲构成最大压力：

- **UGC 专家**：三层边界必须 Day1 内建而非后置--`human.c` 19 门派硬编码证明核心引擎已深度硬编码武侠语义，若不早期确立题材包边界，迁移会继续把武侠烙印焊死进核心引擎。
- **制作人**：真正的风险不在热插拔机制有无，而在内核是否主题无关--三层边界文档化若脱离"内核主题无关性验证"就是纸上契约。

**回应**：两个反论合起来指向真正的承重问题--"CombatKernel 抽象从武侠提取（保深度）与用非武侠微场景验证主题无关（保可移植）的张力"。裁决以**阶段 -1 非武侠微场景硬门禁**兜底：实现从武侠来，验证用非武侠，二者互补而非对立。静态加载既无需热插拔机制就能验证主题无关性，又能阻止核心引擎继续硬编码武侠（三层边界 + 门派剥离）。

### 未消除风险

- CombatKernel 从武侠提取时接口可能过窄（锁死武侠语义）或过宽（平庸化武侠），靠阶段 -1 非武侠微场景硬门禁兜底。
- 门派内容包边界切割（`human.c` race 层加成 + `kungfu/class` 脚本分散多处）须在迁移阶段2 明确切割。
- "第二个题材真实存在"触发条件未定义--热插拔后置若无触发点会滑成"永不考虑"。

### 何时重新引入热插拔

**第二个题材真实存在且需不停服切换时再议**。届时须先预研悬挂引用问题（组件注销但实体持有引用、in-flight Effect 引用已注销 condition schema、活跃对话游标引用已卸载对话树）。当前 1 题材 + 1 开发者，建热插拔注册中心是为不存在的受众建基础设施（YAGNI）。

## Q2 自定义规则引擎

### 问题

是否引入自定义规则引擎，统一管理 condition-action 对，UGC 创作者的触发器也落到这一层？

### 裁决：否决独立自定义规则引擎组件（6/6 共识，源码验证支撑）

> 但这不等于"不做 condition-action 统一处理"--Q2 的核心诉求（统一 condition-action）部分成立，只是载体不是独立引擎，而是**层1 IR + 薄求值子模块 + System 三分**。

### 关键设计

**1. 规则表示层 = DSL 层1**（condition->action 编译为 JSON IR）

唯一规则表示层，只覆盖事件触发器（init/valid_leave/accept_object/add_action/chat_msg）。UGC 触发器只能声明式层1，从题材包注册表查询受限谓词/动词，不能注册新词。

**2. 求值器 = 层1 的薄求值子模块**（不命名为"引擎"、不建独立框架）

事件触发即时求值（非前向链），dirty-flag 缓存，主题注册动词派发。**不设独立配置面、不对外暴露"引擎"概念**，避免命名诱发 scope creep。

**3. 调度语义四分**（源码验证，避免双真相源）

> LPC 的 condition-action 实为 4 种调度语义不同的机制，一个"统一规则引擎"要么误模周期 tick（重蹈"事件驱动替代 heart_beat"覆辙），要么自己长成第二个 ECS 调度器。

| 调度类型 | 归属 | 实证 |
|---|---|---|
| 事件触发器 | 层1 求值 | init 1668 处 / valid_leave 533 处 / accept_object / add_action 1134 处，离散即时 |
| `call_out` | 进程内 ActionScheduler（timer wheel） | 694 文件/3109 处，延迟或自递归 |
| 周期 condition | ConditionSystem tick | CND_CONTINUE/CND_NO_HEAL_UP 位标志，`feature/condition.c:15` 注释"called by heart_beat"坐实 |
| 连续仿真 | Combat/Heal/NPCAI System | tick 驱动 |

**heart_beat 1s tick 绝不被规则引擎吸收**。

**4. themed 治理 = 平台级可信 Python System + PunishmentPolicy 策略对象**

| 治理 | 实现 | 实证 |
|---|---|---|
| 天雷 | 输入管线限流阶段 PunishmentPolicy | `alias.c:41-72` `CMDS_PER_TICK=20`，cnt>3x 触发天雷文本+unconscious+死因标记 |
| 阴间 | DeathSystem 状态机 | d/death 13 房 |
| vote | SocialGovernanceSystem + condition 定时器 | CND_CONTINUE 倒计时 |
| 法院 | scripted NPC 序列 | d/wizard/courthouse |

**fail-closed，不进层1、不对 UGC 开放编辑**。themed 治理是异构的（状态机/输入管线惩罚/NPC 脚本），强行统一要么太平庸（无法表达状态机）要么变图灵完备 DSL（与层3 逃生舱重叠）。

**5. 优先级/冲突 = 显式 priority + deny-wins + 同 action-tag 首匹配**

对齐 `notify_fail`，按事件类型固定策略，**不建通用冲突解决引擎**，写进基线测试断言原 LPC 命中行为（533 valid_leave 及 12 个 imm/wiz 同名动词命中行为）。

**6. 表达力不足应对**

- 先做 30 文件表达力校准实验扩层1-2 原语。
- 层3 逃生舱（RestrictedPython UGC）兜底。
- KPI <15% 护栏。
- **明确禁止"规则脚本"中间层**（层1 与层3 之间）。
- 判定标准：可声明式描述且跨多条规则复用则扩层1；需图灵完备逻辑则层3。

### 边界（不做什么）

- **战斗热路径绝不经规则求值器**（v2 已定，6 专家一致）。
- **不引入 RETE/前向链/OPA/Drools 或任何外部规则引擎**--1000 在线下每事件规则扇出几十条、AST 白名单求值微秒级，无组合爆炸，RETE 网络构建成本反而高于线性求值。
- **heart_beat/call_out/condition tick 不被规则引擎吸收**--它们是 System 调度，不是触发器。
- **PermissionService 仍是唯一 (subject,action,resource) fail-closed 策略引擎**，授权不分裂进第二个"规则引擎"。
- **层1 求值子模块不是独立组件**，避免命名诱发 scope creep。

### 最强反论（已回应）

规则引擎的核心价值场景是"让非开发者编写大量同类规则而不碰代码"--这正是 UGC 平台的长期愿景。否定独立引擎在当前阶段正确，但"层1+Python 策略对象"两分法把"统一 condition-action"这个真实诉求推给了未来，可能产生四类代价：(1) 跨层优先级无法统一表达；(2) 审计轨迹割裂；(3) 层1 原语反复蠕变成长成事实规则引擎；(4) themed 治理全用 Python 硬编码意味着反作弊策略调整需改代码重启。

**回应**：当前阶段（1 题材、1 开发者、1000 在线）四类代价均不成立或可接受。层1 原语蠕变靠 KPI+判定标准护栏；规则冲突语义漂移靠基线测试断言原 LPC 命中行为；themed 治理"反作弊迭代慢"是 fail-closed 的正确取舍。若未来 UGC 规模化后统一诉求重新强烈，再评估是否引入轻量统一求值入口--但本轮明确不建独立引擎。RETE/前向链/外部引擎永久排除（除非 UGC 规模证明线性求值成为瓶颈，此时先优化层1 求值而非换引擎）。

### 未消除风险

- 层1 原语蠕变成长成事实规则引擎（靠 KPI + 判定标准护栏）。
- 规则冲突语义漂移（靠基线测试断言 533 valid_leave 命中行为）。
- call_out 与 ConditionSystem/EventBus 边界需实现时明确。
- 薄求值器命名若失控诱发 scope creep（实现纪律约束其作为层1 子模块身份）。

### 何时重新评估

若未来 UGC 规模化后，"层1+Python 策略对象"两分法的统一 condition-action 诉求重新强烈（跨层优先级无法统一表达、审计轨迹割裂、层1 原语反复蠕变），再评估是否引入轻量统一求值入口。themed 治理运营期反作弊策略调整需改代码重启，接受"反作弊迭代慢"代价，不为此引入可热改的规则配置。

## Q3 玩家行为 -> 世界状态：Command 模式

### 问题

每个玩家的操作是否统一封装为 Command 模式？是否每个操作（含 NPC 行为/系统 tick）都走 Command？

### 裁决：有条件采纳--Command 仅覆盖玩家/管理员外部意图，System.update 覆盖 tick 派生变更

### 关键设计

**1. Command = 8 段中间件管线信封的形式化命名**（不另起类层级）

`{verb, args, ActionContext(actor, source, capability_token, seq), result, effects}` --即现有 `ResolvedAction + ActionContext + ActionResult` 的命名，**零成本形式化**（采纳 DSL 专家反冗余立场）。管线 8 段（解码->限流->历史->别名->路由->授权->执行->审计）即 Command 的生命周期。路由段保真 `command_hook` 四分支（exits 回退 -> verb 路由 -> emote -> channel，`feature/command.c:50-64` 已验证）。

**2. 玩家/管理员输入 -> Command 经完整 8 段管线**

ActionContext + CapabilityToken 替换 `previous_object`/`geteuid` 信任链（155 处实证），仅作用于 Command 路径。

**3. System.update(world, dt, ctx) 直接 mutate ECS 组件覆盖 tick 派生变更**

heal、condition 过期（CND_CONTINUE）、combat `do_attack`/`resolve_attack`（6 处递归 `combatd.c:772/777/804/812/823/831` 已验证为 tick 驱动非命令驱动）、NPC AI。**不经 Command、不经解码/限流/别名/历史中间件**。`resolve_attack` 是 System 内纯函数非 Command。

**4. 确定性重放 = Command 日志（input log）+ RNG seed**

System 变更是派生（从相同输入+seed 重放得出），**不进 input log**。若 System 变更也包装为 Command，input log 会记录派生变更而非纯输入，重放时派生变更被重复执行导致状态分叉。

**5. 触发器（DSL 层1）= Command 执行步内子步骤**

如 go 触发 valid_leave，规则触发的动作附加到父 Command 的 effect 账本，**不另起新 Command**。设最大链深度 + 环检测防止 `valid_leave -> damage -> death -> ...` 无限递归。

**6. force_me -> PrivilegedAction API**

Command 变体，仅可信（ROOT 等价）System 代码可调，强制 authorize + 审计。**LPC 保真**（`force_me` 经 `command_hook`，`feature/command.c:89-95` 已验证）但受能力门控。当前 4 调用点低频，NPC AI 未用它做战斗/移动（走 heart_beat/do_attack），边界泄漏可控可审。

**7. 战斗副作用账本按文本与状态交织真实顺序记录**

`do_attack` 的文本与状态交织变更（`combat_exp`/`jingli` 中途 mutate、`hit_ob` polymorphic 返回）按真实交织顺序记录，非"先算后 apply"（采纳 MUD 专家承重修正）。此为 System 内部，非 Command。

**8. call_other 保真**

Command.execute 同步执行返回结果（不经异步队列），对应 LPC `call_other` 同步语义（保 68771 处同步调用栈语义）。

### 边界一句话

> Command = 玩家/管理员意图信封 + 信任链 + 审计单元 + 重放输入；System = 世界变更执行者 + tick/派生逻辑。Command 产出 Effect 交 System 应用，**System 不反向调 Command**（唯一例外是 force_me/PrivilegedAction，显式门控）。

### 边界（不做什么）

- **反对另起 Command 类层级**（DSL 专家）--8 段管线本质上已是 Command 模式，新类是冗余。Command 保持为管线数据信封，抵抗向其添加方法/继承层级。
- **反对将 Command 扩展到 tick 驱动的 System 内部 mutation**（引擎专家）--把每个 `VitalsSystem.update`/`resolve_attack` 包成 Command 会制造热路径对象 churn（GC 压力）与无收益间接层，且与确定性 System 语义重复。1s tick 下避免热路径对象分配是控制 GC 的关键。
- **反对"每个操作含 NPC 行为/系统 tick/call_out 都走 Command"的扩张解读**（MUD 专家）--LPC `command_hook` 只处理玩家输入，`heart_beat`/`auto_fight`/`condition update`/`call_out` 回调直接 mutate dbase 不经命令路径。
- **反对默认异步化**（UGC 专家）--Command 执行须保留 LPC 同步语义，否则破坏 68771 处同步调用栈语义。

### 最强反论（已回应）

"Command 仅覆盖外部意图、System tick 不走 Command"的干净边界是理想化，LPC 自身并不遵守它。`force_me`（`feature/command.c:89-95`）允许任何 ROOT_UID 对象调用完整命令路径--即系统 daemon 与可信对象在 LPC 中本就混合了"仿真"与"命令"。4 个真实调用点：`updated.c:177` 强制传闻广播（系统发起）、`cost.c:18` 巫师工具、`to.c:20-21` 语音重定向。

若新架构把 `force_me` 映射为 Command（PrivilegedAction，LPC 保真），则"仅外部意图"边界泄漏：系统发起的动作变成了 Command；若映射为 System.update（保边界纯净），则破坏 LPC 保真。**二者不可兼得**。

**回应**：裁决选择前者（PrivilegedAction = Command 变体，ROOT 门控 + 强制审计），**显式承认这是保真让步**。边界在 `force_me` 处妥协以保 LPC 保真，这是显式取舍而非隐藏。泄漏面可控（4 调用点低频，NPC 不用于战斗/移动），靠审计/限制调用点约束。

### 未消除风险

- **force_me 边界侵蚀**：若未来 NPC AI 或触发器大量使用 PrivilegedAction 会侵蚀"仅外部意图"边界。缓解：审计/限制调用点，NPC 行为优先走 System 方法。
- **派生变更审计覆盖缺口**：System.update 的 mutation（战斗外 exp/jingli、condition 效果、heal）无 Command 级审计轨迹。战斗有副作用账本覆盖，其他 System 逐个决定审计粒度。
- **触发器链深度**：Command->规则->规则->... 需最大深度限制 + 环检测。
- **GC 压力**：100 并发玩家 Command 全信封 + 中间件对象。1s tick 下大概率可接受，需 tick profiler 实测确认。
- **System 路径 ActionContext 分歧**：DSL 专家主张 NPC/tick"携带 ActionContext 但无解码/限流/别名/历史"（为审计/能力），其余专家暗示 System.update 无 ActionContext（纯派生）。裁决：System 路径携带轻量 SystemContext（供能力/审计钩子）但派生变更不进 input log。

### 何时重新评估

若未来 `force_me` 调用点增长或 NPC AI/触发器大量使用 PrivilegedAction 侵蚀"仅外部意图"边界，则重新评估是否收紧（强制走 System.update）或显式扩展边界并补审计。派生变更审计粒度逐 System 落地后，若发现关键 System 变更不可审计，再补 System 级审计轨迹。全仿真确定性若 M3 后作为独立决策点重新引入，input log 语义须与此处 Command/System 边界一致（派生变更不进 input log 的规则不变）。

## 三裁决的内在一致性

三个裁决共享同一条收敛主线：**识别过早抽象，砍掉为不存在的规模/受众建设的基础设施，把承重设计留在"单机纯 Python + 内存 JSON + 1000 在线"约束内可落地的最小集合**。

- Q1 砍运行时热插拔（为第二个不存在的题材建插件系统）。
- Q2 砍独立规则引擎（把层1 已有的规则机制拆成两件）。
- Q3 砍 Command 类层级与扩张解读（8 段管线已是 Command，不另起抽象；tick 派生变更不套 Command）。

三者都遵循"收敛优先于完备"--宁可少做不可乱做。被砍的不是能力本身，而是**当前阶段不该承担的复杂度**；每个被砍项都附"何时重新引入"的后置指引，确保推迟而非取消。
