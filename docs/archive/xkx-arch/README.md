# 侠客行 MUD 现代化重构架构方案

> 将经典 LPC MUD《侠客行》（8412 LPC 文件、6414 房间、21 门派）从 0 到 1 重构为全新 Python 项目。LPC 是规格源，新引擎从零按规格实现，行为等价验证。
>
> 本方案经三轮迭代：36 Agent 原始研究 -> 8 专家对抗复审（v2）-> **6 专家对抗复审与收敛裁决（v3 收缩版，当前）**。前两轮文档已归档至 [_archive/](_archive/)。

## 核心立场

1. **单机验证优先**：在验证 UGC 是否成立、改造是否顺利之前，不引入分布式、K8s、Rust/Go、PG/Redis 权威态等任何过早复杂度。先用单机纯 Python + 内存数据 + 本地 JSON 存档，跑通"千人在线 + 百并发"与"DSL+Agent 创作闭环"两件事，再谈扩展。
2. **垂直切片优先**：最高产品风险（DSL+Agent 创作闭环）先验证，再投入引擎重构。
3. **greenfield 框架**：LPC 是规格源不是运行系统，清除所有"适配层/双栈/过渡期"增量假设。
4. **收敛优先于完备**：识别过早抽象，砍掉为不存在的规模/受众建设的基础设施。宁可少做不可乱做。
5. **性能以实测定**：阶段 0 做 μs 级基准 + tick 预算 + 1000+100 负载压测，go/no-go 硬门禁，不达标触发 kill criteria。
6. **标杆靠差异化**：DSL-as-contract Agent 闭环 + combat 确定性 + themed 治理 + 题材包三层边界，而非"把 LPC 迁到 Python"。

## 六条收缩约束（硬边界）

1. 不考虑分布式架构（验证 UGC 成立前）
2. 运维观测后置（仅 OpenTelemetry + Grafana + Langfuse，无 K8s/Helm）
3. 不考虑分布式网关（单机 1000 在线 + 100 并发）
4. 纯 Python（暂不考虑 Rust/Go）
5. 内存数据 + 本地 JSON 定时存档（策略模式，后期可迁 PG/MongoDB）
6. 三个开放问题已裁决（见 [02](02-三个开放架构问题裁决.md)）

## 三个开放问题的裁决速览

| 问题 | 裁决 | 一句话 |
|---|---|---|
| Q1 题材包热插拔 | 有条件采纳 | 保留三层边界 + 静态加载，砍运行时热插拔；门派灵魂归武侠题材包资产 |
| Q2 规则引擎 | 否决（6/6） | 砍独立引擎；层1 IR + 薄求值子模块 + System 三分 + 调度语义四分 |
| Q3 Command 模式 | 有条件采纳 | Command 仅覆盖外部意图；System tick 派生变更不经 Command；force_me=PrivilegedAction 保真让步 |
