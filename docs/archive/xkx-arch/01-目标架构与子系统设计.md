> 本文档是 v3 子系统设计。相对 v2（见 `_archive/02-v2-目标架构与子系统设计.md`），**删除全部分布式脚手架与 PG/Redis/NATS 残留**（子系统2 的 CombatCoordinator/Region Supervisor/handoff outbox/PlacementService、子系统6 的 PG+Redis+NATS），重写存储层为内存+JSON，新增"题材与扩展层"承载三层边界裁决。三个开放问题的裁决详见 [02-三个开放架构问题裁决](02-三个开放架构问题裁决.md)。

## 总体架构分层

```text
┌─ 接入层 ────────────────────────────────────────────────┐
│ 单进程 WS 服务器 · AuthService · SessionManager · InputPipeline│  不持游戏状态
└────────────────────────────┬───────────────────────────┘
                             │ CommandMessage{verb,args,ctx,seq}  ← 玩家/管理员外部意图
┌─ 游戏核心层 (单进程 asyncio + 普通对象) ────────────────┐
│ ECS World(SparseSet) · TickSimulator(1s 非均匀) · CommandRouter(8段中间件)│
│ ActionScheduler(timer wheel) · EventBus(进程内)                      │
│ Systems: Combat/Skill/Condition/Heal/Equip/Movement/Finance/NPCAI    │
│          WorldGovernance/Title                                       │
└────────────────────────────┬───────────────────────────┘
                             │ 事件 / 投影 (tick 边界聚合)
┌─ 存储层（策略模式 · 持久化边界抽象）────────────────────┐
│ 内存权威 + 本地 JSON 定时存档（原子写+offload+dirty-flag 分摊）       │
│ StorageBackend 接口 → 未来策略切换 PG/MongoDB                         │
│ audit_event append-only 接口                                         │
└────────────────────────────┬───────────────────────────┘
                             │
┌─ UGC 与创作层 ──────────────────────────────────────────┐
│ DSL编译器(4层->JSON IR) · CPK+provenance · RestrictedPython 沙箱      │
│ Agent 编排(内存状态机) · MCP验证(world-graph/combat-sim)              │
│ ThemeRegistry(启动时静态加载) · ReviewPipeline                       │
└──────────────────────────────────────────────────────────┘
```

**寻址**：`entity_id: UUID`（64 位整数或 UUID4）+ 进程内对象引用 / SparseSet 查询。`world://<shard>/<region>/<room>#<uuid>` URL 格式仅作分布式阶段"未来寻址"文档后置指引，**单机代码不出现**。

## 子系统 1：网关与会话层（单进程）

**理念**：单进程 WS 服务器，纯协议层，不持游戏状态。

- **认证**：argon2 密码哈希（安全不妥协）+ HS256 或纯内存 session token + 内存吊销集合。砍 JWT RS256 非对称签名 + refresh token + Redis 黑名单（单进程无跨进程鉴权需求）。
- **限流**：进程内令牌桶（替代 Redis 令牌桶）。天雷惩罚保留 themed 文本语义（见子系统 12）。
- **会话**：内存 dict（`session_id -> Session`）。断线只会话挂起（SUSPENDED），角色体留在世界继续被仿真。
- **编码中间件**：Big5 转换保留在 InputPipeline 应用层，传输层统一 UTF-8。
- **客户端延迟隐藏**：移动命令乐观执行（前端本地渲染 + `client_seq` 上报，服务端 ack，不一致发 rollback delta）。
- **崩溃重连语义**：
  - 进程内断线（未崩溃）：in-memory ring buffer 重放（容量按典型战斗 tick × 挂起窗口估算，超容退化 snapshot）。
  - 进程崩溃 = 冷重启：从最近 JSON checkpoint 加载世界 + 客户端连接拒绝/重试 + 重启后重新认证 + 服务端发**全量 snapshot**（非 ring 重放）+ 向玩家声明丢失窗口。**不得假装 ring 跨崩溃有效**，否则静默状态漂移。

## 子系统 2：游戏核心（单进程，删除分布式脚手架）

**理念**：单进程 asyncio + 普通对象 + SparseSet ECS + 1s 非均匀 tick。`call_other` 退化为进程内同步函数调用（与 LPC 语义一致）。

> **v3 删除**（v2 子系统2 的分布式脚手架，单进程下全是死代码）：CombatCoordinator / Region Supervisor / MigrationCoordinator / handoff 两阶段提交 + outbox 协议 / PlacementService 一致性哈希 / BoundaryRoomAgent / ShardSupervisor / cross-shard 消息。02-v2 文档残留 20 处引用须在实现时主动剪除。

### 组件

- **World**：单进程 ECS 世界。asyncio 事件循环 + SimulationTick（1s，非均匀）。
- **TickSimulator**：固定 1s tick（`set_heart_beat(1)` 实证）。非均匀 tick 桶（仅活跃实体进 tick 队列，非全量 1000）。按子系统注册 tick 回调。
- **CommandRouter**：8 段中间件管线（解码 -> 限流 -> 历史 -> 别名 -> 路由 -> 授权 -> 执行 -> 审计）。路由段保真 `command_hook` 四分支（exits 回退 -> verb 路由 -> emote -> channel）。
- **ActionScheduler**：进程内内存 timer wheel，`call_out` 等价（694 文件/3109 处）。
- **EventBus**：进程内，仅非热路径事件路由（战斗热路径绕过）。

### 旧 -> 新映射

- `file_name` 即身份又即路径 -> `entity_id`(UUID) 解耦。
- `object` 指针引用 `this_object()`/`environment()` -> 进程内直接对象引用 / SparseSet 查询。
- `call_other` 同步阻塞 -> 进程内同步函数调用（语义一致，无跨进程）。
- `master.c crash()` 全服宕 -> 进程崩溃冷重启从 JSON checkpoint 恢复（单进程无 supervisor 分层）。
- `epilog/preload` 16 daemon -> 32 守护进程职责重新设计（被 ECS System 取代 / 保留无状态 / 演进）。
- `room.c` 门状态 `find_object` 双面协调 -> 进程内 DoorState 键值（无跨分片）。

## 子系统 3：ECS 实体系统

**理念**：自研 SparseSet ECS（单存储，不用 Archetype）。**ECS 在 Python 是架构选择非性能优化**（dict+dataclass 无 cache locality/SIMD），保留因其支持 UGC 动态挂卸组件。

### 组件

- **ECS 运行时核心**：Entity 为 64 位整数 ID。SparseSet 存储 O(1) 挂载/卸载/查询。
- **组件族**：Identity / Vitals / Attributes / DerivedAttributes / ModifierStack / SkillSet / CombatState(enemy_id[] + ThreatTable + fight/dodge 累积态) / EquipmentSlots / Conditions / Position / Inventory / Wallet / TeamRelation / ActionState / DynamicProps / Effect / Progression / MentorshipRelation。
- **System 调度器**：CombatSystem / VitalsSystem / HealSystem / ConditionSystem / AttributeSystem / SkillSystem / EquipmentSystem / MovementSystem / ActionSystem / FinanceSystem / NPCAISystem / WorldGovernanceSystem / TitleSystem。
- **EventBus**：替换 `this_object()->method()` 隐式耦合。战斗热路径绕过总线。
- **Prefab/Archetype**：替代 `default_ob` 单级回退，支持多级继承。

### 关键接口

- `create_entity(prefab_id, overrides) -> EntityID` / `query(*comp_types) -> SparseSetIter`
- `System.update(world, dt, ctx)` --**直接 mutate 组件**，不经 Command（tick 派生变更路径）。
- `EffectInstance{effect_id, type, source, target, duration_ticks, mode: tick|wallclock, payload, stages, rng_seed, stacking_policy}`。

### 旧 -> 新映射

- inherit F_DBASE + dbase/tmp_dbase -> 类型化 Component schema + DynamicProps
- `default_ob` 单级回退 -> Prefab 多级原型链
- F_ATTACK enemy/killer object 数组 -> CombatState.enemy_id[] + ThreatTable
- `receive_damage` 直接 set dbase -> DamageEvent/HealEvent + VitalsSystem（热路径同步）
- `query_str` 等 apply 聚合 -> Attributes + ModifierStack(分三类) + DerivedAttributes dirty 缓存
- 472 个 SKILL_D 策略对象 -> SkillRef(skill_id, level) 组件 + **SkillData/SkillBehavior/SkillEffect 三层**
- 72 个 condition daemon -> Conditions 组件 + ConditionHandler 协议
- `query_entire_dbase()` 活引用直改 -> 先建字段->组件映射表 + Progression 组件

## 子系统 4：命令与行为系统（Command 形式化）

**理念**：8 段中间件管线 + ActionContext + CapabilityToken。**Command = 管线信封的形式化命名，不另起类层级**。

> v3 裁决（[02](02-三个开放架构问题裁决.md) Q3）：Command 仅覆盖玩家/管理员外部意图；`System.update` 覆盖 tick 派生变更不经 Command。

### 组件

- **Command**：`{verb, args, ActionContext(actor, source, capability_token, seq), result, effects}`，即现有 `ResolvedAction + ActionContext + ActionResult` 的命名，**零成本形式化，不新建类层级**。覆盖玩家/管理员外部意图，走完整 8 段管线（解码 -> 限流 -> 历史 -> 别名 -> 路由 -> 授权 -> 执行 -> 审计）。
- **ActionContext + CapabilityToken**：替换 `previous_object`/`geteuid` 隐式信任链（155 处实证），仅作用于 Command 路径。
- **PermissionService**：唯一 `(subject, action, resource)` fail-closed 策略引擎。**权限策略是启动时加载的可信配置/代码**（非可变存档），进程内求值，永远可用；fail-closed = "策略加载失败则拒绝启动"（Redis 已移除，无"运行时外部依赖不可达"问题）。保留路径分层 ACL 语义（`securityd.c` 实证 trusted_write/exclude_write/authorized_cmds 为 directory->wizlevel 映射），不为通用性引入策略表达式求值。从 `securityd.c` 提取权限规格后从零实现，不修旧代码。
- **PrivilegedAction**：`force_me` 的映射，Command 变体，仅可信（ROOT 等价）System 代码可调，强制 authorize + 审计。**明确承认是保真让步**（LPC `force_me` 走 `command_hook`，`feature/command.c:89-95` 实证）。4 调用点低频（updated.c 传闻 / cost.c 巫师 / to.c 语音×2），NPC 不用于战斗/移动，泄漏面可控可审。
- **ActionScheduler**：3570 字符串 `call_out` 机械转换为进程内 timer wheel；144 闭包型人工迁移为 Effect；Effect 内置 target 存活校验。
- **VerbRouter**：ExitResolver 保留 `!arg` 前置条件。

### Command vs System 边界（关键）

| 路径 | 覆盖 | 经 Command？ | 经中间件？ |
|---|---|---|---|
| 玩家/管理员输入 | 外部意图 | 是（8 段全走） | 是 |
| `force_me`（PrivilegedAction） | 系统发起但走命令路径 | 是（变体，ROOT 门控+审计） | 是（保真让步） |
| `System.update`（heal/condition 过期/resolve_attack/NPC AI） | tick 派生变更 | **否** | 否 |
| 触发器（DSL 层1，如 go 触发 valid_leave） | Command 执行步内子步骤 | 否（附加到父 Command effect 账本） | 否 |

- 确定性重放 = Command 日志（input log）+ RNG seed；System 变更是派生（从相同输入+seed 重放得出），**不进 input log**。
- 触发器链设**最大链深度 + 环检测**，防止 `valid_leave -> damage -> death -> ...` 无限递归。
- `call_other` 保真：Command.execute 同步执行返回结果（不经异步队列，保 68771 处同步调用栈语义）。

## 子系统 5：战斗与玩法引擎

**理念**：保留 `do_attack` 七步管线 + AP/(AP+DP) 概率 + 三层资源，用 CombatKernel 抽象泛化，combat 确定性保证可回放。

### 组件

- **TickSimulator**：1s tick，非均匀分级 tick 桶。
- **CombatKernel**：抽象接口 `CombatAction` / `CombatResolver` / `DamageModel`。**从武侠 `do_attack` 七步提取**（保武侠深度，不从 4 题材预先设计）；阶段 -1 用非武侠微场景验证内核主题无关性。
- **CombatSystem**：`resolve_attack()` 纯函数管线。输入 `CombatContext` 快照，输出 `CombatRoundResult`。
  - **CombatContext 快照边界**：战斗开始时对全体参与者 + 可观察环境做快照。
  - **管线副作用账本**：七步各自的 mutate **按"文本与状态交织真实顺序"记录**（`hit_ob` polymorphic 返回 string/int/mapping 可同时追加文案与修改伤害、`exp`/`jingli` 中途直接 mutate dbase、递归子回合嵌套），重放器按交织顺序回放，**显式否定"先纯计算后批量 apply"**。
  - seeded DeterministicRNG（combat 范围）。
- **DamageSystem**：apply_damage/apply_wound/apply_heal/apply_cure。三层资源不变量 `0<=current<=eff<=max`。
- **HealSystem**：战斗/非战斗不同恢复速率；魔法数字提取 HealRateConfig。
- **ConditionSystem**：ConditionStack + ConditionHandler 协议（`on_tick` 返回组合结构 `continue/suppress_heal/expire`）。
- **EffectSystem**：一等公民。`stacking_policy`(unique/refresh/stack/independent)；mode: tick|wallclock。
- **SkillSystem**：**SkillData(YAML) + SkillBehavior(Python 策略) + SkillEffect(Effect 组件)** 三层。UGC 只编辑 SkillData。`skill_power` 三段式可组合公式管线（基础幂 + 状态修饰链 + 属性系数与封顶）；fight/dodge 累积态为 CombatState 瞬态字段。
- **DeathSystem**：DeathPolicy 策略对象；阴间/轮回作为 DeathSystem 核心流程（见子系统 12）。
- **TargetingSystem**：ThreatTable 替代随机选择；MAX_OPPONENT=4 可配。
- **CombatRenderer**：文案抽离 damage_messages.json；**文本体验流 diff** 验证战斗手感。
- **DamageFormula**：三段式可组合公式管线。

### 数据模型示例

```yaml
# kungfu/skill/yuenu-jian.skill.yaml  (层1 SkillData，UGC 可编辑)
id: yuenu-jian
name: 越女剑法
type: sword
valid_enable: [sword, parry]
actions:
  - action: "$N施展一招「白露横江」，$w横削$n$l"
    force: 100
    dodge: 30
    damage: 20
    damage_type: 割伤
# 题材包级 SkillBehavior 文件（非 UGC 可编辑，门派灵魂在此）
behavior:
  hidden_finisher:
    condition: "str>=30 && int>=36 && random(skill)>80 && force>150 && neili>400"
    damage: 200
    special: true
```

### 旧 -> 新映射

- `do_attack` 直接读写 dbase -> `resolve_attack` 纯函数读组件快照，副作用按交织顺序记入账本
- `char.c heart_beat` 上帝方法 -> TickSimulator 分系统独立 tick
- `skill_power` level³/3 -> DamageFormula 三段式管线
- `s_combatd.c` -> 提取阵法合击/自创连击规格，实现为 CombatModifier
- `perform call_out` 闭包递归 -> EffectSystem + Effect 多阶段 stages
- `condition.c update_condition` + CND_CONTINUE 位标志 -> ConditionSystem + 组合返回值
- `human.c` 门派 if-else 加成 -> **从 race/核心层剥离为题材包 FamilyDefinition**（19 门派实证，见子系统 14）
- MudOS `random()` 全局 -> DeterministicRNG + CombatContext 快照（combat 范围）
- `NewRandom(i,20,level/5)` -> DSL `weighted_random` 原语

## 子系统 6：数据与状态层（重写：内存+JSON）

**理念**：**内存数据权威 + 本地 JSON 定时存档**。StorageBackend 是"持久化边界"抽象。砍 v2 的 PG 唯一权威 + Redis 读穿透 + NATS（02-v2 残留 9 处须剪除）。

> **关键认知**：存储收缩不是"换存储"而是对 v2 子系统6 的隐性重新设计。v2 把 PG+Redis 贯穿于 EventStore/SnapshotStore/ReferenceStore/SchedulerStore/outbox/CommandBus 乐观并发/CPK 内容寻址/CQRS 投影。策略模式接口是对的，但必须逐一标注丢失的语义（事务原子性、崩溃恢复、并发写 CAS、关系完整性、append-only 可篡改）。

### 组件

- **内存权威态**：在线实体全部驻留进程内存（ECS World）。session dict / per-session deque / 令牌桶 / EventBus / timer wheel 全部进程内。
- **JSON 存档**：
  - **原子写**：`write-temp + os.replace/os.rename`（POSIX 同文件系统原子）。
  - **序列化 offload**：orjson + `ThreadPoolExecutor`/`aiofiles`，**绝不在事件循环主线程同步写**。
  - **dirty-flag 分摊**：每 tick 只写 N 个 dirty 实体轮转覆盖；关键事件（死亡/升级/交易/门派变更）强制同步 persist 该实体。
  - **丢失窗口**：30-60s，玩家可见处声明 + 标注"非生产安全"。
  - **启动 schema 校验**：加载 checkpoint 时校验，损坏拒绝启动。
- **StorageBackend 策略接口**（持久化边界抽象，非"save=权威写"）：
  - `persist(entity_id, state)` --契约="达到崩溃恢复级耐久"。JSON 实现=异步 write-behind（游戏不阻塞）；PG 实现=同步 write-through（PG 即权威）。
  - `audit_event(type, payload)` --append-only 接口。JSON 实现=追加日志文件；PG 实现=`INSERT audit_events`。高价值领域事件（死亡/交易/婚姻/升级/门派变更）走此接口。
  - **游戏逻辑只调 `persist`/`audit_event` 不感知后端**。迁 PG 是策略切换非架构变更。
- **EntityStateManager**：load 最近 checkpoint + 重放增量 = 当前状态（单进程冷重启用）。
- **SchemaRegistry**：pydantic v2 类型化组件。
- **ReferenceStore**：实体间关系存 ID 引用（进程内 dict + JSON 存档）；未来 PG 关系表。`marryd` 双方 dbase 双写无事务 -> ReferenceStore 关系原子提交（进程内事务语义）。
- **WorldStateReconciler**：声明式 desired-state + reconcile（替代 reset 全量扫描）。

### 丢失语义标注（迁 PG 时需补回）

| 语义 | JSON 存档 | PG（未来） |
|---|---|---|
| 事务原子性 | 无（单进程串行，靠原子写文件） | WAL 事务 |
| 崩溃恢复 | 最近 checkpoint（丢 30-60s） | WAL 回放（不丢） |
| 并发写 CAS | 无（单进程无并发） | 乐观并发 `expected_seq` |
| 关系完整性 | 无（靠应用层） | 外键约束 |
| append-only 可篡改 | 日志文件可篡改 | audit 表只增 |

### 止损线

**任何外部玩家测试开始前必须迁 PG**（崩溃丢 30s 对外部玩家不可接受）--硬止损线非可选。

四层 DSL 全部编译为 JSON IR（详见 [03-DSL-UGC与Agent协作](03-DSL-UGC与Agent协作.md)）。

> v3 修正：
> - **层3 统一为 RestrictedPython**（砍 WASM，单进程纯 Python 下无多租户进程隔离需求；WASM 留作多租户生产阶段重新评估）。靠审查门 + 能力令牌 + 配额区分信任级，非语言隔离。

> - inquiry：简单字符串响应转层0，带交易流程的转"对话+交易原子节点"混合模型。
> - 逃生舱使用率 KPI <15%，超标触发层1-2 表达力迭代。

## 子系统 9：前端与状态同步

## 子系统 14：题材与扩展层（v3 新增）

**理念**：三层边界 + 静态加载。承载 [02](02-三个开放架构问题裁决.md) Q1 裁决。

### 三层粒度

| 层 | 粒度 | 谁来写 | 执行权限 | 机制 |
|---|---|---|---|---|
| 核心 | 引擎（ECS+tick+CombatKernel+DSL 编译器+沙箱） | 平台核心开发者 | 不可插拔 | -- |
| 题材包 | Theme（schema family+谓词/动词词汇表+themed 治理策略）+ Module Pack（门派/区域 CPK） | 受信任开发者 | 进程级无沙箱 Python（StdLib 级） | 审查+签名+SemVer+provenance |
| UGC | CPK（层0 YAML+层1 规则+层2 对话树） | 创作者 | 沙箱受限 RestrictedPython | 能力令牌+fuel 配额 |

### 组件

- **ThemeRegistry**：启动时加载的静态注册表（Python dict: `theme_id -> ThemeDescriptor`），manifest 声明 `theme+version+dependencies`。**无运行时 unload/版本协商/隔离**。
- **ThemeDescriptor**：注册 `component_schemas` / `condition_predicates` / `action_verbs` / `combat_resolver_impl` / `default_assets` / `themed_governance_policies`。
- **CombatKernel**：抽象接口（`CombatAction`/`CombatResolver`/`DamageModel`），从武侠 `do_attack` 七步提取；阶段 -1 用非武侠微场景验证主题无关性（硬门禁）。
- **受信任开发者扩展模型**：带代码题材包，走审查 + 签名 + SemVer + CPK provenance，进程级无沙箱 Python。UGC 永不创作题材包本身，只在已注册题材内编辑层1。

### 门派灵魂归属

门派灵魂（SkillBehavior Python 策略，含隐藏大招/hit_by/auto_perform）归属**武侠题材包资产**，非核心引擎硬编码，非 UGC 可创作。`human.c` 19 门派 if-else 加成迁移时**须从 race/核心层剥离为题材包内容**（门派内容包边界切割须在迁移阶段2 明确，避免题材包变成大杂烩或核心引擎残留武侠烙印）。

### 后置指引

运行时热插拔（register/unload/version/isolation 完整机制）等**第二个题材真实存在且需不停服切换时再议**。届时须先预研悬挂引用问题（组件注销但实体持有引用、in-flight Effect 引用已注销 condition schema、活跃对话游标引用已卸载对话树）。

## 跨子系统一致性约束

1. **entity_id 全局唯一**：UUID + 进程内对象引用（无 `world://` URL）。
2. **ActionContext 贯穿 Command 路径**：玩家/管理员命令携带 ctx；System tick 派生变更不经 Command。
3. **事件 seq 单调**：每会话/每聚合域单调递增。
4. **@persist/@transient 标注**：决定 snapshot 行为。
5. **Effect 可序列化**：所有持续效果可崩溃恢复（单进程冷重启）。
6. **fail-closed 安全**：PermissionService 策略加载失败则拒绝启动。
7. **greenfield 语义**：无双栈/无过渡期/无适配层/无逐步替换 LPC。
8. **单进程串行**：单房间串行是 MUD 固有硬约束；目标单机千人在线+百并发。
9. **combat 确定性**：CombatContext 快照 + seeded RNG + PYTHONHASHSEED=0 + 禁 set 迭代（全仿真确定性后置 M3 后）。
10. **持久化边界抽象**：游戏逻辑只调 `persist`/`audit_event` 不感知后端。
