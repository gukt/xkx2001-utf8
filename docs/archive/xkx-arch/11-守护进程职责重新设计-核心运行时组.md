## 1. logind.c -- LOGIN_D

- **LPC 职责**：登录状态机，从连接入口 logon 到 enter_world 的完整 13 阶段流程 + 重连路径 reconnect。
- **有状态？**：是。input_to 状态链、wiz_lock_level、当前连接对象映射。关键 state：LoginState 枚举（LOGON -> CONFIRM_BIG5 -> GET_ID -> GET_PASSWD / CONFIRM_ID -> GET_NAME -> NEW_PASSWORD -> CONFIRM_PASSWORD -> GET_GIFT -> GET_EMAIL -> GET_GENDER -> ENTER_WORLD / RECONNECT）。
- **新引擎归属**：**ECS System（LoginSystem）**
- **理由**：登录是跨 tick 的有状态会话，新引擎中映射为 ConnectionSystem + LoginState 状态机，驱动 WS 会话生命周期。
- **阶段**：阶段 1（核心循环）
- **衔接**：层 H 规格（`_logon` / `_get_id` / `_get_passwd` / `_make_body` / `_enter_world` / `_reconnect` / `_check_legal_id` / `_check_legal_name` / `_random_gift` / `_init_new_player`，共 10 个 FunctionSpec）。LoginState 枚举已定义。ADR-0010 层 H 范围。
- **备注**：
  - 层 H 13 阶段状态机完整性是核心不变量。
  - BAN_D / MAX_USERS / wiz_lock / valid_wiz_login / REGBAN_D 五重 fail-closed 前置检查。
  - exec(user, ob) 连接转移在 Python 中映射为 ActionContext 会话切换，不需要 LPC exec 语义。
  - random_gift 天赋生成（str+int+con+dex+end=100, kar+pat+per=60）保留为角色创建随机性。
  - 起始房间选择优先级链：ghost -> DEATH_ROOM; death_count 惩罚; startroom; fallback。

## 2. chard.c -- CHAR_D

- **LPC 职责**：角色初始化（setup_char：种族分派 + 属性钳位 + 默认值填充 + reset_action）+ 华山派师徒解除（break_relation）+ 尸体生成（make_corpse，层 F 已提取）。
- **有状态？**：否（setup_char 是一次性初始化函数，操作传入对象的 dbase，daemon 自身无持久状态）。
- **新引擎归属**：**无状态服务（CharService）**
- **理由**：setup_char 是纯初始化逻辑（种族分派 + 属性钳位 + 默认值），无 tick 驱动、无 daemon 级状态。break_relation 是操作特定 NPC 状态的命令式函数。
- **阶段**：阶段 1（核心循环）
- **衔接**：层 H 规格（`_setup_char` / `_break_relation`）。setup_char 的属性钳位不变量（jing <= eff_jing <= max_jing）衔接层 E 三层资源不变量。make_corpse 属层 F。
- **备注**：
  - 属性钳位仅对玩家生效（userp），NPC 有独立 force 自动设置逻辑。
  - break_relation 仅处理华山派风清扬师徒，其他门派不受影响。
  - break_relation 由 die() 触发（层 F），是平台级 fail-closed 逻辑。

## 3. securityd.c -- SECURITY_D

- **LPC 职责**：权限校验 valid_cmd（每条命令都过）/ valid_write / valid_read + 巫师等级管理 get_status / set_status / valid_wiz_login。
- **有状态？**：是。wiz_status 映射（uid -> WizLevel）、wiz_sites 映射（uid -> IP 正则白名单）、authorized_cmds / exclude_cmds / trusted_write / exclude_write / trusted_read 权限表。F_SAVE 持久化。
- **新引擎归属**：**演进为新能力（PermissionService + CapabilityToken）**
- **理由**：CLAUDE.md 要求 greenfield 重新设计，securityd 的 euid/uid 模型在 Python 中无对应物，需重新设计为 CapabilityToken + PermissionService。类型必须完整（CLAUDE.md 安全相关模块要求）。
- **阶段**：阶段 1（核心循环）
- **衔接**：层 H 规格（`_valid_cmd` / `_valid_write` / `_valid_read` / `_get_status` / `_valid_wiz_login` / `_set_status`）。WizLevel 10 级枚举已定义。valid_cmd 被层 C command_hook 每条命令调用（层 C 规格 `_func_find_command_commandd`）。
- **备注**：
  - **fail-closed 是核心不变量**：euid 为空或权限不匹配时返回 0，不得默认放行。
  - **exclude 优先于 trusted**：先检查 exclude 列表（命中即拒绝），再检查 trusted 列表（命中即允许）。
  - **themed 治理**：securityd 是平台级 fail-closed Python，不落入 UGC（09 文档确认）。
  - 04 §六不做清单明确"不修 securityd 旧代码"，greenfield 重新设计。
  - 新引擎中 euid 概念映射为 CapabilityToken（不可伪造的能力令牌），valid_cmd 映射为命令管线第 1 段权限校验中间件。
  - ROOT_UID 门控映射为系统级 admin 权限，force_me = PrivilegedAction（CLAUDE.md 不变量）。

## 4. natured.c -- NATURE_D

- **LPC 职责**：游戏时间推进（真实 1 秒 = 游戏 1 分钟）+ 8 阶段日间循环 + event_sunrise 全局自动保存 + event_common 通用清理。
- **有状态？**：是。current_day_phase（当前时段索引）、day_phase 表（8 阶段定义）、call_out 队列（下次 update_day_phase 定时器）。
- **新引擎归属**：**ECS System（TimeSystem）**
- **理由**：时间推进是 tick 驱动的有状态系统，8 阶段循环 + 定时事件（sunrise 自动保存、common 清理）需要 System 级调度。
- **阶段**：阶段 1（核心循环）
- **衔接**：层 H 规格（`_update_day_phase` / `_event_sunrise` / `_event_common` / `_init_day_phase` / `_game_time`）。
- **备注**：
  - **时间不变量**：真实 1 秒 = 游戏 1 分钟（TIME_TICK = time() * 60），不得更改。
  - **JSON 存档崩溃安全**：event_sunrise 触发全局自动保存，Python 实现需 write-temp + os.replace 原子写 + offload + dirty-flag 分摊（CLAUDE.md 不变量）。
  - event_common 清理无环境 NPC（destruct）和玩家（move 到安全房间），衔接 ECS 实体生命周期管理。
  - 8 阶段：凌晨(240) -> 日出(120) -> 上午(180) -> 正午(180) -> 下午(180) -> 傍晚(180) -> 夜晚(120) -> 午夜(240)。

## 5. chinesed.c -- CHINESE_D

- **LPC 职责**：中文数字转换（chinese_number）+ 中文干支日期（chinese_date）+ 英中翻译查询（chinese）。
- **有状态？**：是（dict 翻译映射 F_SAVE 持久化，可动态增删），但核心函数 chinese_number / chinese_date 是无状态纯函数。
- **新引擎归属**：**无状态服务（ChineseFormatService）**
- **理由**：chinese_number 和 chinese_date 是确定性映射纯函数，无随机性、无 tick 驱动。chinese 翻译查询是简单字典查找。dict 的持久化属于配置管理，非运行时状态。
- **阶段**：阶段 1（核心循环）
- **衔接**：层 H 规格（`_chinese_number` / `_chinese_date` / `_chinese`）。被 LOGIN_D（在线人数显示）、NATURE_D（game_time 委托 chinese_date）广泛调用。
- **备注**：
  - 确定性映射函数，无随机性，不需要确定性 RNG。
  - 天干 10 个 + 地支 12 个，60 年一循环。十二时辰每 2 小时一个时辰。

## 6. commandd.c -- COMMAND_D

- **LPC 职责**：命令路径查找（find_command）+ 命令缓存（rehash）。
- **有状态？**：是。search 映射（目录 -> 命令文件名列表缓存）。
- **新引擎归属**：**无状态服务（CommandRegistry）**
- **理由**：find_command 是路径查找 + 缓存，无 tick 驱动。search 缓存属于启动时构建的注册表，非运行时变更的状态。在新引擎中映射为 CommandRegistry 模块，命令路径搜索变为命令注册表查找。
- **阶段**：阶段 1（核心循环）
- **衔接**：层 C 规格（`_func_find_command_commandd` / `_func_rehash`）。COMMAND_PATHS 常量已提取。find_command 逆序搜索 + valid_cmd 校验是不变量。
- **备注**：
  - 搜索方向逆序（从 path 尾部向头部，尾部优先）。
  - 每条命令必须过 SECURITY_D->valid_cmd 校验（层 C 规格 invariant）。
  - 新引擎中 LPC 的文件路径查找变为 CommandRegistry 注册表查找，rehash 变为启动时扫描注册。

## 8. combatd.c -- COMBAT_D

- **LPC 职责**：战斗主循环 do_attack 七步管线 + skill_power 公式 + fight/auto_fight + death_penalty/killer_reward + damage_msg/status_msg。
- **有状态？**：否（daemon 自身无持久状态，所有状态在参与者对象的 dbase 上）。但 do_attack 是有副作用的复杂管线。
- **新引擎归属**：**ECS System（CombatSystem）**
- **理由**：战斗是 tick 驱动的核心系统，do_attack 七步管线由 heart_beat -> attack() -> fight() -> do_attack() 链驱动。在新引擎中映射为 CombatSystem，每 tick 遍历有 CombatTarget 组件的实体执行攻击回合。
- **阶段**：阶段 1（核心循环）/ 阶段 2.4（Combat 迁移专项）
- **衔接**：层 E 规格（`_do_attack` / `_skill_power` / `_fight` / `_auto_fight` / `_death_penalty` / `_damage_msg` / `_eff_status_msg` / `_report_status` + attack.c 的 `_attack` / `_fight_ob` / `_kill_ob` / `_select_opponent` / `_reset_action` + damage.c 的 `_receive_damage` / `_receive_wound` / `_receive_heal` / `_receive_curing` / `_die` / `_unconcious` / `_heal_up` + skill.c 的 `_query_skill` / `_improve_skill` / `_skill_death_penalty` + condition.c 的 `_apply_condition` / `_update_condition` / `_clear_condition`）。CombatStep 七步枚举 + ResourceType 四类 + AttackType 三类已定义。
- **备注**：
  - **do_attack 七步副作用交织不可分离**（CLAUDE.md 不变量）：不得"先算后 apply"，SideEffect.order 字段记录 message_output 与 state_mutation 的严格交织顺序。
  - **combat 确定性范围 = combat-only**（CLAUDE.md 不变量）：29+ 处 random() 全部提取为 RandomSpec，含概率模型和 seed_inputs。全仿真确定性后置 M3 后。
  - 闪避判定 `random(ap+dp) < dp`（概率 = dp/(ap+dp)）和招架判定 `random(ap+pp) < pp`（概率 = pp/(ap+pp)）是 combat 确定性的两个核心随机点。
  - **三层资源不变量**：0 <= qi <= eff_qi <= max_qi（jing/jingli/neili 同理）。
  - killer_reward 中 PK 计数 + 通缉 condition 施加是法院系统的触发点（09 文档确认 themed 治理）。
  - riposte 递归 / s_combatd 阵法合击 / perform/exert / condition 具体状态类型 / kungfu 门派武学均后置。
  - CombatKernel 从武侠提取、用非武侠验证（CLAUDE.md 不变量，阶段 -1 硬门禁）。

  - 新引擎中 PronounContext 是三元组 (speaker, viewer, target)，System tick 无"当前说话者"时需定义回退（04 §七避坑重点第 5 点）。
  - query_rank 基于 shen 值分级（旷世侠客/大侠/侠客/少侠/...魔王/魔头），门派称号由 family + class + skills 决定。
  - PKS > 100 且 PKS > MKS 时称号变"土匪"/"土匪婆"（法院系统关联，09 文档确认）。
  - 7 个函数中 query_close / query_self_close 有 random(5)（无性者称谓随机），属非 combat 随机性。

## 15. fingerd.c -- FINGER_D

- **LPC 职责**：玩家信息查询（finger_all 全服在线列表 + finger_user 单玩家详情 + remote_finger_user 跨服查询 + get_killer 通缉榜）。
- **有状态？**：否。所有函数是查询 + 格式化输出，无持久状态。
- **新引擎归属**：**无状态服务（QueryService）**
- **理由**：fingerd 是纯查询函数，无 tick 驱动。finger_all 遍历 users()，finger_user 从存档恢复，get_killer 遍历在线玩家检查 condition。
- **阶段**：阶段 1（finger_all / finger_user）/ 阶段 1（get_killer，法院系统关联）
- **衔接**：层 H 规格标注 FINGER_D 属第二梯队。09 文档题材文化系统台账列 finger 为平台级阶段 1。get_killer 被法院系统通缉告示（bounty_list.c）使用。
- **备注**：
  - finger_user 区分 player 和 wizard 视角（wizard 可见 IP 和上次连线地址），映射为新引擎中基于 CapabilityToken 的信息可见性控制。

## 3 个关键决策亮点

1. **securityd -> PermissionService + CapabilityToken**：greenfield 重新设计，LPC 的 euid/uid 模型映射为不可伪造的 CapabilityToken，valid_cmd 映射为命令管线第 1 段权限校验中间件。fail-closed 是核心不变量（euid 为空或权限不匹配时拒绝，不得默认放行）。exclude 优先于 trusted 的权限模型保留。CLAUDE.md 要求安全相关模块类型完整。themed 治理确认平台级 fail-closed，不落入 UGC。

2. **combatd -> CombatSystem（七步交织 + combat 确定性）**：do_attack 七步管线（选技能 -> 取招式 -> AP/DP 计算 -> 闪避判定 -> 招架判定 -> 伤害结算 -> 经验获得）的 SideEffect.order 字段记录 message_output 与 state_mutation 的严格交织顺序，不得"先算后 apply"。29+ 处 random() 全部提取为 RandomSpec，combat 确定性范围 = combat-only，全仿真确定性后置 M3 后。三层资源不变量（0 <= qi <= eff_qi <= max_qi）是 damage.c 的核心契约。CombatKernel 从武侠提取、用非武侠验证。

3. **rankd -> PronounContext 三元组（speaker / viewer / target）**：query_close 和 query_self_close 使用 this_player() 作为隐式 viewer（对比 ob/target 的 age/mud_age/gender 决定称谓），是 CLAUDE.md 关键不变量"PronounContext 必须携带 viewer"的实证。新引擎中 PronounContext 是三元组 (speaker, viewer, target)，System tick 无"当前说话者"时需定义回退（04 §七避坑重点第 5 点）。04 §三阶段 2.5 明确 TitleSystem 前置条件为 RANK_D 7 函数规格提取（三元组求值）。
