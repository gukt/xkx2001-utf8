# 侠客行 MUD 现代化重构总纲

> 本文档是 v3（收缩版）架构总纲。在前两轮（36 Agent 研究 + 8 专家对抗复审，见 `_archive/`）基础上，经第三轮 6 位专家对抗复审与收敛裁决（见 [05-第三轮专家对抗复审报告](05-第三轮专家对抗复审报告.md)），将项目从"分布式标杆平台"收缩为"**单机验证优先**"的可控范围。
>
> **核心立场**：在验证 UGC 是否成立、改造是否顺利之前，不引入分布式、K8s、Rust/Go、PG/Redis 权威态等任何过早复杂度。先用单机纯 Python + 内存数据 + 本地 JSON 存档，跑通"千人在线 + 百并发"与"DSL+Agent 创作闭环"两件事，再谈扩展。

---

## 1. 愿景与目标

将经典 LPC MUD《侠客行》（8412 LPC 文件、6414 房间、21 门派、68771 处同步调用）从 0 到 1 重构为全新 Python 项目。LPC 是**规格源**（spec/source），新引擎从零按规格实现，行为等价验证，非增量迁移。

| 维度 | 目标（v3 收缩版） |
|---|---|
| 后端 | Python 3.12 asyncio（stdlib 优先，uvloop 可选），单进程 |
| 前端 | 现代 Web（React + TypeScript），文本为基线、图形为渐进增强 |
| 部署 | **单进程起步**；分布式/分片/K8s 全部后置，待单机验证通过且承载达 80% 后再议 |
| 平台 | UGC 平台--用户借 DSL 配置剧情场景，创作者经济 Day1 仅预留字段 |
| 创作 | Agent 协作创作多题材世界；MVP 先验证 DSL+Agent 创作闭环是否成立 |
| 性能 | **单机 1000 人在线 + 100 并发请求**（现状 `MAX_USERS=50` 的 20 倍） |
| 存储 | 内存数据权威 + 本地定时 JSON 存档（策略模式封装），后期可迁 PG/MongoDB |
| 标杆 | "最 MUD 的现代游戏"而非"最现代的 MUD"--保留 themed 治理灵魂，用现代技术放大表现力 |

---

## 2. 六条收缩约束（硬边界，不可突破）

本轮对前两轮方案的最大调整是**收缩**。以下六条约束是专家复审的硬边界，任何设计不得重新突破（仅可在"何时重新引入"层面留后置指引）。

1. **不考虑分布式架构**：验证 UGC 成立、改造顺利之前，不引入分布式。砍掉分片 / Louvain 社区检测 / handoff / Ray / 一致性哈希 / 跨分片消息 / placement。
2. **运维观测后置**：最多先引入 OpenTelemetry + Grafana + Langfuse（追踪 Agent 调用）；K8s / Helm / ArgoCD / OpenSearch / ClickHouse / KEDA 等完全不考虑。
3. **不考虑分布式网关**：重点验证单机支持 1000 人在线 + 100 并发。网关从"无状态水平扩展 + Redis 会话路由"收缩为单进程 WS 服务器。
4. **纯 Python，暂不考虑 Rust/Go**：compiled core 后置，先按纯 Python 规划容量与确定性。仅当"do_attack 微基准 + 1000/100 负载双失败且 Python 优化穷尽"时才重新评估 Rust/Go 热路径。
5. **内存数据 + 本地 JSON 定时存档**：改造期间内存为权威态，定时存档到本地 JSON 文件，用策略模式封装存储接口，后期可迁 PG/MongoDB。**这是"持久化边界"抽象，非"换存储"**。
6. **三个开放问题须架构师定夺**：题材包热插拔、规则引擎、Command 模式--裁决见 [02-三个开放架构问题裁决](02-三个开放架构问题裁决.md)。

## 3. greenfield 语义（不变）

- LPC 是规格源不是运行系统：无适配层 / 双栈 / 过渡期 / 逐步替换 / 修旧 bug。
- 不修 `securityd.c` 旧代码，从其提取权限规格后从零实现 PermissionService。
- 差分测试重定义为"规格提取 + 行为等价验证"：对照提取的形式化规格 + 本地 MudOS spec oracle，非运行中的生产 LPC。
- `s_combatd.c` 不存在"废弃"，只提取阵法合击规格后按规格实现。
- 32 守护进程是"职责重新设计"（被 ECS System 取代 / 保留无状态服务 / 演进为新能力），非 1:1 迁移。

---

## 4. 三条不可妥协边界（保留，语义收缩）

1. **网关纯协议层**：单进程 WS 服务器只做帧编解码 / 握手 / 限流 / 心跳 / 会话管理，**绝不持有游戏状态**。虽不再"无状态水平扩展"，但"不持游戏状态"的纪律不变。
2. **会话与游戏实体解耦**：Session 是一等实体，断线只会话挂起（SUSPENDED），角色体留在世界继续被仿真。
3. **渲染下沉前端**：服务端只产语义事件，`message_vision` 模板 / ANSI 颜色 / `damage_msg` 文案全部下移前端纯函数渲染层；服务端只负责求值 `PronounContext`（代词 10 变量 / RANK_D 7 函数，依赖观察者，是二元关系非实体纯属性）。

```text
┌──────────────────────────────────────────────────────────────────┐
│  现代 Web 前端 (React+TS)  语义事件渲染 / 代词上下文 / 状态投影     │
└──────────────────────────────┬───────────────────────────────────┘
                               │ WebSocket (MsgPack + JSON fallback)
┌──────────────────────────────▼───────────────────────────────────┐

┌──────────────────────────────▼───────────────────────────────────┐

│  未来可切：PostgreSQL / MongoDB（策略切换，非架构变更）              │
│  audit_event append-only 接口（JSON 追加日志 / 未来 PG INSERT）     │
└──────────────────────────────┬───────────────────────────────────┘

│  Agent 编排(内存状态机起步) · MCP 验证(world-graph/combat-sim)      │

└──────────────────────────────────────────────────────────────────┘
```

### 分层职责

- **接入层**：单进程 WS 服务器，协议、认证（argon2 + HS256/内存 token）、限流（进程内令牌桶）、会话（内存 dict）。不持游戏状态。
- **游戏核心**：单进程 asyncio + 普通对象 + SparseSet ECS + 1s 非均匀 tick 仿真。`call_other` 退化为进程内同步函数调用（与 LPC 语义一致）。
- **存储层**：内存权威 + JSON 定时存档。**StorageBackend 是"持久化边界"抽象**：`persist(entity_id, state)` 契约 = "达到崩溃恢复级耐久"，JSON 实现 = 异步 write-behind（游戏不阻塞），未来 PG 实现 = 同步 write-through（PG 即权威）。游戏逻辑只调 `persist` 不感知后端。
- **创作层**：DSL -> JSON IR -> 沙箱执行 -> MCP 验证 -> 修订闭环。Agent 编排与人类协作。

## 6. 核心架构决策（v3 收缩版）

### 6.1 对象模型：SparseSet ECS（保留）

自研 SparseSet ECS（单存储，不用 Archetype）替换 inherit+feature，组件运行时动态挂卸（UGC 多题材必需）。

> **关键认知**：ECS 在 Python 是**架构选择非性能优化**--Python dict+dataclass 无 C++ 的 cache locality / SIMD / 真并行。保留 SparseSet 因其支持 UGC 动态挂卸、组合优于继承，但**不应以性能为由选型、不应驱动 compiled-core 决策、不应在无测量热点前加 SOA/numpy**。

核心组件族：Identity / Vitals(qi/jing/jingli+eff/max 三档) / Attributes / DerivedAttributes / ModifierStack / SkillSet / CombatState(enemy_id[] + ThreatTable + fight/dodge 累积态) / EquipmentSlots / Conditions / Position / Inventory / Wallet / TeamRelation / ActionState / DynamicProps / Effect / Progression / MentorshipRelation。

System 替代 daemon+heart_beat：CombatSystem / VitalsSystem / HealSystem / ConditionSystem / AttributeSystem / SkillSystem / EquipmentSystem / MovementSystem / ActionSystem / FinanceSystem / NPCAISystem / WorldGovernanceSystem / TitleSystem。

### 6.2 战斗引擎：七步管线 + combat 确定性（保留，确定性范围收缩）

保留 `do_attack` 七步管线（选技能 -> 取招式 -> AP/DP 命中 -> PP 招架 -> 伤害 -> 施加 -> 经验/反击）作为 `CombatSystem.resolve_attack()` 纯函数管线。AP/(AP+DP) 概率模型原样保留。

> v3 修正：
> - **确定性范围收缩为 combat-only**：仅保留 `CombatContext` 快照 + seeded RNG + `PYTHONHASHSEED=0` + 禁 set 迭代（已是 combat-sim 前置）。**全仿真确定性（所有 System seeded RNG + input log）显式移到 M3 后作为独立决策点**--单机阶段对每个 System 征收确定性税只换取规模期才兑现的收益，与收缩矛盾。
> - **管线副作用账本按"文本与状态交织真实顺序"记录**：`hit_ob`/`weapon->hit_ob` 返回 polymorphic（string/int/mapping）可同时追加文案与修改伤害，`exp`/`jingli` 在管线中途直接 mutate，递归子回合嵌套。重放器按交织顺序回放，**显式否定"先纯计算后批量 apply"**--那是战斗手感漂移的根源。
> - `resolve_attack` 同步返回 `CombatRoundResult` 并就地应用 + 通知同房间订阅者。
> - Vitals 运行时用裸 dataclass + 断言，Pydantic 仅在 snapshot 落盘 / UGC 上传 / 边界校验时用。
> - `ConditionHandler.on_tick` 返回组合结构 `(continue, suppress_heal, expire)`。

### 6.3 数据与状态：内存权威 + JSON 存档（重写，非 v2 的 PG+Redis）

> v3 重大收缩：权威态从 v2 的"PG 唯一权威 + Redis 读穿透"改为"**内存数据权威 + 本地 JSON 定时存档**"。Redis 全部依赖（session 路由 / ring buffer / 限流 / 缓存 / JobStore）单进程内内存化。

**JSON 存档崩溃安全规范（不可妥协）**：

- **原子写**：`write-temp + os.replace/os.rename`（POSIX 同文件系统原子）。LPC `save_object` 是无原子全量覆盖（`feature/save.c` 确认），greenfield 不得重复此 bug。
- **序列化 offload**：用 orjson，在 `ThreadPoolExecutor` 或独立 `aiofiles` 协程中写盘，**绝不在事件循环主线程同步写**（1000 实体 × ~20KB = ~20MB，同步写 ~100ms 会冻结一个 tick）。
- **dirty-flag 分摊存档**：每 tick 只序列化并写盘 N 个 dirty 实体（N 按基准定，如 20-50），轮转覆盖；关键事件（死亡/升级/交易/门派变更）强制同步 persist 该实体。
- **丢失窗口声明**：存档周期 30-60s，崩溃最多丢 30-60s 进度，**在玩家可见处声明并标注"非生产安全"**。
- **启动 schema 校验**：加载 JSON checkpoint 时校验 schema，损坏则拒绝启动并报错。
- **止损线**：任何外部玩家测试开始前必须迁 PG（崩溃丢 30s 对外部玩家不可接受）--这是硬止损线非可选。

**StorageBackend 策略接口 = "持久化边界"抽象**（非"save=权威写"）：

| 接口 | JSON 实现（当前） | PG 实现（未来） |
|---|---|---|
| `persist(entity_id, state)` | 异步 write-behind（游戏不阻塞） | 同步 write-through（PG 即权威） |
| `audit_event(type, payload)` | 追加日志文件 | `INSERT audit_events` |

> **关键**：游戏逻辑只调 `persist` / `audit_event` 不感知后端。迁 PG 时 `persist` 语义从"checkpoint"自然升级为"authoritative"，**无需改游戏层**--这是策略切换非架构变更。若接口按"save=权威写"语义设计，迁 PG 须重铺每条写路径，是架构变更。

### 6.4 命令与权限：8 段中间件管线 + 能力令牌（保留，Command 形式化）

`command_hook` god-function 拆为显式中间件管线（解码 -> 限流 -> 历史 -> 别名 -> 路由 -> 授权 -> 执行 -> 审计）。`previous_object()` 隐式信任链显式化为 `ActionContext` + `CapabilityToken`（155 处实证）。

> v3 裁决（详见 [02](02-三个开放架构问题裁决.md) Q3）：
> - **Command = 8 段管线信封的形式化命名**（`{verb, args, ActionContext, result, effects}`），**不另起类层级**--它就是现有 `ResolvedAction + ActionContext + ActionResult` 的命名，零成本形式化。
> - **Command 仅覆盖玩家/管理员外部意图**；`System.update(world, dt, ctx)` 直接 mutate 组件覆盖 tick 派生变更（heal / condition 过期 / `resolve_attack` / NPC AI），**不经 Command**。
> - 确定性重放 = Command 日志（input log）+ RNG seed；System 变更是派生，不进 input log。
> - `force_me` = `PrivilegedAction`（Command 变体，ROOT 等价门控 + 强制审计，明确承认是保真让步，4 调用点低频）。
> - 路由段保真 `command_hook` 四分支（exits 回退 -> verb 路由 -> emote -> channel）。
> - PermissionService 仍是唯一 `(subject, action, resource)` fail-closed 策略引擎，授权不分裂进第二个"规则引擎"。
> - **权限策略是启动时加载的可信配置/代码**（非可变存档），进程内求值，永远可用。fail-closed 重定义为"策略加载失败则拒绝启动"，非"运行时外部依赖不可达默认拒绝"（Redis 已移除）。

### 6.5 调度：进程内 timer wheel（收缩，砍 APScheduler+Redis）

> v3 收缩：砍 `APScheduler + Redis JobStore` 持久调度。单进程崩溃 = 冷重启（从最近 JSON checkpoint 加载世界），定时器随状态重建。用**进程内内存 timer wheel**。

调度语义四分（文档化，避免双真相源）：

| 调度类型 | 归属 | 说明 |
|---|---|---|
| 事件触发器（init/valid_leave/accept_object/add_action/chat_msg） | DSL 层1 求值 | 离散即时求值 |
| `call_out`（694 文件/3109 处） | 进程内 ActionScheduler（timer wheel） | 延迟或自递归调度，call_out 等价 |
| 周期 condition（CND_CONTINUE/CND_NO_HEAL_UP 位标志） | ConditionSystem tick | heart_beat 周期驱动 |
| 连续仿真 | Combat/Heal/NPCAI System | tick 驱动 |

> `heart_beat` 1s tick 绝不被规则引擎/事件驱动吸收（LPC 固有，层1 只覆盖触发器）。

### 6.6 DSL 与 UGC：四层 DSL（保留，WASM 砍除）

四层 DSL 全部编译为 JSON IR 作为唯一真相源：

- **层0**：YAML 声明式场景数据（exits/objects/item_desc/cost/outdoors，对应 2482 个 replace_program 纯数据房间）。
- **层1**：事件规则 `condition->action`（对应 add_action/valid_leave/accept_object/chat_msg）--**唯一规则表示层**，UGC 触发器落此层且仅声明式。
- **层2**：对话树（Ink 的 knot/stitch/divert/weave，对应 inquiry）。
- **层3**：脚本逃生舱--**统一为 RestrictedPython**（砍 WASM，单进程纯 Python 下无多租户进程隔离需求；WASM 留作多租户生产阶段重新评估）。

> v3 裁决（详见 [02](02-三个开放架构问题裁决.md) Q2）：**否决独立自定义规则引擎**。层1 是唯一规则表示层；层1 的薄求值子模块（不命名为"引擎"、不建独立框架）管事件触发即时求值 + dirty-flag 缓存；themed 治理（天雷/阴间/vote/法院）是平台级可信 Python System + PunishmentPolicy 策略对象，fail-closed，不进层1、不对 UGC 开放。逃生舱使用率 KPI <15%，超标触发层1-2 表达力迭代而非放松 KPI；明确禁止层1 与层3 之间的"规则脚本"中间层。

### 6.7 题材与扩展：三层边界 + 静态加载（新增，砍运行时热插拔）

> v3 裁决（详见 [02](02-三个开放架构问题裁决.md) Q1）：**有条件采纳三层边界，砍运行时热插拔**。

三层粒度（源码实证 `human.c` 19 门派 if-else 加成证明门派是 wuxia 题材下内容包非独立题材）：

| 层 | 粒度 | 谁来写 | 执行权限 |
|---|---|---|---|
| 核心 | 引擎（ECS+tick+CombatKernel+DSL 编译器+沙箱） | 平台核心开发者 | 不可插拔 |
| 题材包 | Theme（schema family+谓词/动词词汇表+themed 治理策略）+ Module Pack（门派/区域 CPK） | 受信任开发者 | 进程级无沙箱 Python（StdLib 级） |
| UGC | CPK（层0 YAML+层1 规则+层2 对话树） | 创作者 | 沙箱受限 RestrictedPython |

- **门派灵魂**（SkillBehavior Python 策略，含隐藏大招/hit_by/auto_perform）归属**武侠题材包资产**，非核心引擎硬编码，非 UGC 可创作；`human.c` 19 门派加成迁移时须从 race/核心层剥离为题材包内容。
- `ThemeRegistry` = 启动时加载的静态注册表（Python dict: `theme_id -> ThemeDescriptor`），manifest 声明 `theme+version+dependencies`，**无运行时 unload/版本协商/隔离**。
- `CombatKernel` 抽象（`CombatAction`/`CombatResolver`/`DamageModel`）**从武侠 `do_attack` 七步提取**（保武侠深度，不从 4 题材委员会预先设计），阶段 -1 用非武侠微场景验证内核主题无关性（保可移植）。
- 受信任开发者扩展 = 带代码题材包，走审查 + 签名 + SemVer + CPK provenance。
- **运行时热插拔等第二个题材真实存在且需不停服切换时再议**（YAGNI）。届时须先预研悬挂引用问题（组件注销但实体持有引用、in-flight Effect 引用已注销 condition schema、活跃对话游标引用已卸载对话树）。

### 6.8 Agent 协作：DSL 为契约，内存状态机起步（保留，收缩）

以 DSL 为唯一契约层把 LLM 非确定性收敛到可验证终态。单 Orchestrator + 能力清单 + RAG 检索（五角色 Worker 降级为能力清单）。MCP 共享验证基底（world-graph 出口可达性 + combat-sim 伤害/胜率）。

> v3 收缩：
> - combat-sim **直接调用从 LPC 提取的 `resolve_attack` 纯函数**（与未来真实引擎同源，非独立简化模型）。
> - Agent 编排 MVP 用**内存状态机**（砍 PG-backed 状态机的 PG 依赖，状态随进程内存；LangGraph 仅过渡原型，中期迁 durable execution 后置）。
> - **LLM token 预算 + kill criterion**：单 CPK token 预算上限；Langfuse 追踪人工修订量趋势，3 轮迭代后修订率 >40% 则先扩层1-2 表达力再继续 Agent，扩后仍 >30% 则 Agent 降级为辅助。
> - Agent NPC 严格分层（LLM 只出对话文本，DSL 状态机守所有状态变更），先判定 tractable 再排期。

### 6.9 前端与状态同步：语义事件流（保留）

服务端维护单一权威状态树，变更通过对比前后权威树生成 JSON-Patch ops + 单调 seq。渲染层完全下沉前端。

- `PronounContext` = `{speaker_id, viewer_id, target_id, vars[10]}`，vars 由 `TitleResolver` 以 `(speaker, viewer, target)` 三元组求值（`rankd.c` `query_close`/`query_self_close` 实证依赖 `this_player()`，是观察者相对二元关系非实体纯属性）；单机下 `this_player()` 映射为 `ActionContext.actor`；求值结果按三元组缓存。
- `message_vision` 1626 模板从 Day1 就是必须实现的规格。
- emote 保留 7 视角 + 10 代词变量，不降级。
- 崩溃重连语义：进程崩溃 = 冷重启（从最近 JSON checkpoint 加载世界）+ 客户端连接拒绝/重试 + 重启后重新认证 + 服务端发全量 snapshot（非 ring 重放）+ 声明丢失窗口；进程内断线（未崩溃）才走 in-memory ring buffer 重放。

### 6.10 性能与确定性：1s tick + combat 确定性（重写，砍 50ms/20Hz 框架）

> v3 重大修正：**砍 50ms/20Hz or 100ms/10Hz 的 tick 预算框架**。LPC `heart_beat` 实测是 1s（`set_heart_beat(1)` 已验证），该框架在解一个 MUD 不存在的 MMO 问题，并错误驱动 compiled-core 紧迫感。

- **tick = 1s**（LPC heart_beat 实证）+ **compute < 100ms**（10%）+ 硬告警 200ms + I/O 预算 900ms+。
- **非均匀 tick**（仅活跃实体进 tick 队列，非全量 1000）--LPC `set_heart_beat(0/1)` 自适应节流的现代版，是单进程 1000 在线的性能关键。
- **GC 调优**：启动暖机后 `gc.freeze()` 冻结长存对象到 gen2，tick 期间禁 gen0 回收或调高阈值，tick 边界手动 `gc.collect`。
- **PYTHONHASHSEED=0** + 禁 set 直接迭代 + 序列化 key 排序（即使只做 combat 确定性也必需，`do_attack` 29 处 `random()` 消耗序列对迭代顺序敏感）。
- **阶段 0 micro-benchmark 为 go/no-go 硬门禁**：单 `do_attack` μs 基准 + 1000 实体 tick 预算 + 1000 在线+100 并发负载压测；不达标触发 kill criteria 而非继续推进。

### 6.11 引擎工具链：收缩为最小三件（砍 6 件一等公民）

> v3 收缩：砍 v2 的"6 件工具链作为阶段0 一等公民"。单机+内存+1000 在线下只保留最小三件，明确区分"引擎工具链"（本地开发工具，阶段0 交付物）与"运维可观测"（OTel+Grafana+Langfuse，后置）。

- **entity inspector**（最小 REPL / dict-dump）--查任意实体组件树。
- **tick profiler**（per-System `time.perf_counter`）--tick 超时告警。
- **combat replay viewer**（最小，因 combat 确定性保留）--`CombatContext` 快照可视化回放。

砍：world graph visualizer（networkx 一次性 dump 足够）、DSL preview（属 UGC 层非引擎层）、hot-reload console（热重载成真再做）。

### 6.12 逻辑热重载与全仿真确定性：后置（推迟非取消）

> v3 收缩：两项 v2 标杆能力显式后置，附重新评估条件。

- **逻辑热重载协议**推迟到验证期之后：单机验证期用维护窗口重启即可，有在线玩家不能容忍重启时再引入。简化为：System 模块在 tick 边界 `importlib.reload` + entity state 原样保留 + schema 变更需重启；CPK/DSL YAML 数据随时自由重载（trivially safe）。
- **全仿真确定性**移到 M3 后作为独立决策点：单机阶段只保留 combat 确定性。

## 7. 业界标杆的差异化能力（收缩版）

标杆护城河不是"把 LPC 迁到 Python"，而是：

1. **DSL-as-contract 的 Agent 协作创作闭环**--schema 强约束把 LLM 非确定性收敛到可验证终态（combat-only 确定性 + 六维评估矩阵）。
2. **combat 确定性回放**--`CombatContext` 快照 + seeded RNG，反作弊/调试/自动化测试共享底座（全仿真确定性后置 M3 后）。
3. **themed 治理范式**--天雷/阴间/法院/vote 是图形 MMO 无法复制的"想象力治理元宇宙"。
4. **Agent NPC 作为游戏内实体**--LLM 出对话文本、DSL 状态机守状态变更的硬隔离（先判定 tractable 再排期）。
5. **题材包三层边界 + 受信任开发者扩展**--多题材可扩展而不平庸化武侠深度。

## 8. 技术选型总表（v3 收缩版）

| 层 | 选型 | 说明 |
|---|---|---|
| 运行时 | Python 3.12 asyncio（stdlib 优先） | uvloop 降级为可选（按阶段0 基准决定是否启用，避免 C 扩展单点风险） |
| WS 服务器 | 单进程（websockets / aiohttp） | 砍 SO_REUSEPORT 多 worker / 会话粘附路由 / NATS 跨节点扇出 |
| REST/UGC API | FastAPI | 单进程内 |
| ECS | 自研 SparseSet（单存储） | 架构选择非性能优化；不驱动 compiled-core |
| 协议 | MsgPack（默认）+ JSON（UGC） | 统一 `{type,version,seq,trace_id,payload}` 帧 |
| 认证 | argon2-cffi + HS256/内存 session token + 内存吊销集合 | 砍 JWT RS256 + refresh token + Redis 黑名单（单进程无跨进程鉴权需求） |
| 权威存储 | **内存数据 + 本地 JSON 定时存档**（StorageBackend 策略接口） | 原子写 + offload + dirty-flag 分摊；未来策略切换 PG/MongoDB |
| 热状态 | 进程内内存（session dict / per-session deque / 令牌桶 / EventBus / timer wheel） | 砍 Redis 全部依赖 |
| 消息扇出 | 进程内 EventBus | 砍 NATS / Redis Stream |
| 调度 | 进程内内存 timer wheel | 砍 APScheduler+Redis JobStore；崩溃冷重启从 checkpoint 恢复 |
| 前端 | React + TypeScript + Zustand/Jotai + Immer | xterm.js 可选终端模式 |
| 序列化 | orjson（高频）/ msgspec | Pydantic v2 仅边界校验；JSON 存档用 orjson + 原子写 |
| DSL | Lark（EBNF）+ PyYAML + pydantic v2 + jsonschema | 四层编译为 JSON IR |
| 沙箱 | RestrictedPython（UGC + 平台级统一两层） | 砍 WASM（后置多租户生产）；fuel/wall_time/memory 配额 |
| Agent | 内存状态机（MVP）+ Claude API | LangGraph 仅过渡原型；Langfuse 追踪 |
| 可观测 | OpenTelemetry + Grafana + Langfuse | 砍 K8s/Helm/ArgoCD/OpenSearch/ClickHouse/KEDA |
| 部署 | 单进程（systemd / nohup 起步） | 分布式/K8s 后置 |
| 寻址 | `entity_id: UUID` + 进程内对象引用 / SparseSet 查询 | `world://` URL 仅作分布式阶段"未来寻址"文档后置指引，单机代码不出现 |
| 验证 | 单元级行为规约 + 一次性 golden trace + property-based testing | greenfield 主门禁 |
