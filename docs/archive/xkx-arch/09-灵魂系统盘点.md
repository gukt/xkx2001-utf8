### 2. 职责

阴间系统负责玩家死亡后的完整流程：死亡触发、惩罚计算、尸体生成、鬼魂状态管理、阴间世界流程（黑白无常剧情或隐藏路径还阳）、以及还阳后的状态恢复。

### 3. 核心数据流

**死亡 -> 阴间 -> 还阳的完整路径**：

```
heart_beat (char.c)
  ├─ eff_qi<0 || eff_jing<0 ──────────────────────> die()
  └─ qi<0/jing<0/jingli<0
       ├─ living(this_object()) ──> unconcious() ──> revive() (自动苏醒)
       └─ 已昏迷 ──────────────────> die()

die() (feature/damage.c)
  ├─ [no_death 房间] ──> unconcious() + return（不进入阴间）
  ├─ clear_condition()
  ├─ COMBAT_D->announce(ob, "dead")
  ├─ COMBAT_D->death_penalty(ob)           经验/技能/金钱惩罚
  ├─ COMBAT_D->killer_reward(killer, ob)   击杀者奖励/通缉
  ├─ CHAR_D->make_corpse(ob, killer)       尸体生成 + 物品转移
  ├─ 玩家分支:
  │    ├─ qi/jing/eff_jing/eff_qi = 1
  │    ├─ save()
  │    ├─ ghost = 1
  │    ├─ move(DEATH_ROOM) ─────────> /d/death/gate.c
  │    ├─ MARRY_D->break_marriage()
  │    └─ CHAR_D->break_relation()   华山派解除师徒
  └─ NPC 分支: destruct(this_object())

gate.c init()
  ├─ 销毁鬼魂所有 inventory 物品
  ├─ clear_condition()
  └─ 白无常 NPC init() ──> call_out("death_stage", 30, ob, 0)

death_stage() [5 段对话，每段间隔 5 秒]
  ├─ stage 0-4: 白无常对话剧情
  └─ stage 4: reincarnate() + move(REVIVE_ROOM)

还阳 (reincarnate)
  ├─ ghost=0; qi/jing/eff_qi/eff_jing/jingli/neili 恢复到 max
  ├─ DROP_CMD->do_drop() 丢弃所有物品（主路径）
  └─ move(REVIVE_ROOM) 或 move("/d/xiakedao/shatan")

隐藏路径（跳过黑白无常剧情）
  inn1.c do_stuff() ──> reincarnate() + move("/d/city/wumiao")
```

**关键状态变更**：ghost=1 / death_count+=1 / combat_exp 扣减（1/100，上限 5000）/ shen 扣减 1/20 / potential 扣减 1/2 / balance 扣减 / break_marriage / break_relation。

### 4. 与核心循环的关系

- **heart_beat**：死亡触发在 char.c 的 heart_beat() 中检查 eff_qi/eff_jing 阈值，是 tick=1s 核心循环的一部分。
- **move**：die() 中 move(DEATH_ROOM) 触发 gate.c 的 init()，启动 call_out 延时链。
- **valid_leave**：gateway.c 禁止向南返回；road2.c 迷宫计数；blkbot.c 限制非巫师进入 noteroom。
- **visible**：char.c 的 visible() 检查 is_ghost()，鬼魂对非鬼魂不可见（除非 astral_vision）。
- **登录**：logind.c 检查 is_ghost() 决定 startroom（断线重连回到阴间）。

### 5. themed 治理属性

**判断：平台级 fail-closed（不落入 UGC）**

理由：
1. 死亡惩罚公式硬编码在 combatd.c（确定性公式，无 UGC 可配置项）。
2. 阴间流程是固定剧情（黑白无常 5 段对话、延时参数、还阳路径全部硬编码）。
3. ghost 标志和 reincarnate 是系统级状态机（引擎层面状态管理）。
4. death_count 惩罚是平台级安全机制（反刷死亡 fail-closed 处置）。
5. 婚姻解除/师徒解除涉及多系统状态一致性，不属于 UGC 可编辑规则层。
6. 与 CLAUDE.md 不变量一致（阴间明确列为 themed 治理）。

### 7. 后置阶段建议

**建议实现阶段：阶段 1（M1 死亡轮回完整实现）**

注意事项：
1. `start_death` 是 LPC 空调用（源码中未定义，FluffOS 对不存在的方法返回 0），Python 实现通过 room enter 事件触发 NPC 剧情链，不需要 start_death 方法。
2. 阴间流程有两条还阳路径：主路径（黑白无常剧情结束 + DROP_CMD 丢弃物品）和隐藏路径（inn1 ask 回家，不丢弃物品）。Python 实现需统一还阳入口。
3. gate.c 的物品销毁是关键副作用：玩家进入鬼门关时所有非角色物品被 destruct（销毁，不是掉落）。与 make_corpse 的物品转移是两个独立步骤--先转移物品到尸体（留在死亡地点），再将鬼魂（已无物品）传送到鬼门关。
4. no_death 房间是免死区域（擂台/战场），致命伤害只昏迷不死亡。
5. death_count>200 且 combat_exp<50000 的玩家登录时强制移入地狱/死刑室，是反刷死亡安全机制。
6. wgargoyle1.c 的 fightdie 分支引用了不存在的 noloseroom，是未完成特性，Python 实现可忽略。
7. 尸体四阶段腐烂（新鲜->腐烂->骸骨->骨灰消散，每阶段 60-120 秒），层 F 规格标注为后置，建议阶段 1 一并实现。

**与已提取规格的关联**：`engine/src/xkx/spec/layer_f_death.py` 已提取 die()/unconcious()/revive()/reincarnate()/make_corpse()/death_penalty()/killer_reward() 的完整签名和副作用顺序，但明确将"阴间世界流程"标注为"后置到阶段 1"（文件头注释第 16 行）。本次盘点补全了这一缺口。

### 2. 职责

武林大会系统是一个巫师手动发起的全服级 PvP 赛事平台，包含个人赛（8 个年龄组的擂台挑战赛）和团体赛（试剑山庄夺旗积分赛）两种模式，以及附属的比武招亲、赌局下注玩法。

### 3. 核心数据流

**触发条件**：
- 个人赛：巫师在 square.c 执行 start，广播 CHANNEL_D 全服通知，调用 kantai1~8.c 的 do_start 设置各年龄组
- 团体赛：巫师或裁判在 control.c 执行 start，设置 bwdh=1
- 时间门控：square.c 的 create() 按 localtime 判断，仅在 2/4/6/8/10-12 时段开放 sjsz 方向出口

**输入（报名）**：
- 个人赛：玩家在 kantai 房间执行 baoming -> 检查年龄/败场 -> 首位报名者成为擂主，后续为挑战者 -> let() 将玩家移入擂台，物品暂存 cangku
- 团体赛：队长在 control.c 执行 teamup 队名 玩家1...玩家20 -> 写入 bwboard 的 team/N mapping -> challenge N + accept N 配对东西两队

**状态变更（赛程推进）**：
- 个人赛 auto_check()（call_out 30 秒轮询）：检测胜负 -> 龙虎榜排序 -> 安排下一场 -> 无人挑战 3 轮判擂主胜 -> 擂主连胜两场强制休息
- 团体赛 auto_check()（call_out 3 分钟轮询）：倒计时归零 -> do_judge 比较东西队积分 -> clean_sjsz 清理赛场 -> sort_rank 排序 -> make_npc 随机刷新 NPC

**输出（奖励）**：
- 个人赛：龙虎榜排名记录在 kantai 房间的 rank/N 属性中，无自动物品奖励
- 团体赛：积分记录在 bwboard（持久化到 /data/board/bwboard 文件）
- 巫师手动奖励：award 命令授予称号或九阴真经权限
- 比武招亲（mu.c）：胜者获白金戒指 + 六合枪法提升

### 4. 与核心循环的关系

- **combat/kill_ob**：个人赛 start() 调用 kill_ob 互攻启动战斗；团体赛 do_attack 指令触发 kill_ob 且同队成员自动加入
- **heart_beat**：auto_check 使用 call_out 模拟定时器（30 秒/3 分钟），非 set_heart_beat；战斗本身的 heart_beat 照常运行
- **move/valid_leave**：多处 valid_leave 门控（选手不能 out 离开、不能 down 溜走、sjsz 入场/出场触发物品暂存恢复）
- **death/revive**：leitai 设置 no_death=1（昏迷不死亡），auto_check 中 revive(1) 唤醒昏迷者后移回看台
- **物品管理**：let() 将选手所有物品移入 cangku 暂存，离场时从 cangku 取回

### 5. themed 治理属性

**判断：平台级 fail-closed（不落入 UGC 可编辑）**

理由：
1. 公平竞赛规则是平台核心契约：年龄分组、三败淘汰、连胜强制休息、龙虎榜排序、物品暂存/恢复/清零、满血恢复、擂台禁令（no_fight/no_death/no_save/no_quit）--任何 UGC 修改都会破坏公平性。
2. 团体赛积分/排名系统是平台信用：bwboard 持久化积分数据，若落入 UGC 可编辑可被篡改积分。
3. 赌场赔率系统涉及经济系统，必须平台管控。
4. 物品/属性重置是安全操作：full_all 恢复满血、nude 清除物品--若被 UGC 篡改可能导致物品丢失或属性作弊。

**可能 UGC 可编辑的部分**（有限）：赛场房间的描述文本、NPC 对话 flavor text、赛场地图布局。但核心机制不可触碰。

### 7. 后置阶段建议

## 四、vote 投票系统

### 1. 文件清单

| 文件 | 类型 | 说明 |
|------|------|------|
| `include/vote.h` | 宏定义 | 投票常量（V_FAIL/V_MIN/V_ONETHIRD/V_HALF/V_TWOTHIRD/V_FIVE + 动议类型 V_EJECT/V_CHBLK/V_UNCHBLK/V_ROBOT） |
| `cmds/std/vote.c` | 玩家命令 | 投票入口命令，路由到子命令 |
| `cmds/std/vote/chblk.c` | 子命令 | 投票关闭某人频道 |
| `cmds/std/vote/unchblk.c` | 子命令 | 投票打开某人频道 |
| `kungfu/condition/vote_clear.c` | Condition daemon | 投票超时自动取消 |
| `kungfu/condition/vote_suspension.c` | Condition daemon | 滥用投票权暂停期到期恢复 |
| `adm/daemons/channeld.c` | 频道守护进程 | 执行端：检查 chblk_on 标志 |
| `inherit/char/char.c` | 角色基类 | heart_beat 中刷屏自动设置 chblk_on |

### 2. 职责

vote 系统是玩家自治的频道管理机制：玩家通过投票集体决定关闭（chblk）或打开（unchblk）某人的交谈/谣言/门派频道。系统包含投票发起、一人一票防重、计票阈值判定、结果自动执行、超时自动取消、滥用惩罚六个环节。当前仅实现 chblk/unchblk 两种动议；vote.h 中预留 V_EJECT（驱逐）/V_ROBOT（机器人检测）两种未实现动议。

### 3. 核心数据流

**完整路径：投票发起 -> 计票 -> 结果执行 -> 超时清理**

```
玩家执行 "vote chblk <target>"
  -> vote.c::main(me, arg)
    |-- 前置检查：age >= 16、未被剥夺投票权(vote/deprived)、target 存在
    |-- 路由：load_object("/cmds/std/vote/chblk.c")
  -> chblk.c::vote(me, victim)
    |-- 防重：检查 victim->vote/juror 数组中是否已有 my_id
    |-- 记票：victim->add("vote/count", 1)，victim->vote/juror += my_id
    |-- 阈值计算：
    |     vv = valid_voters(me) / 4   (chblk: 1/4 在线合格玩家)
    |     vc = victim->vote/count
    |     df = max(vv, 4) - vc        (最低 4 票)
    |-- 若 df >= 1（未达阈值）：
    |     shout 广播 "还差 N 票"
    |     victim->apply_condition("vote_clear", 10)  -- 10 tick 超时倒计时
    |-- 若 df < 1（达到阈值）：
    |     shout 广播 "频道被关闭了！"
    |     victim->set("chblk_on", 1)                 -- 执行结果

超时清理（未达阈值且无人继续投票）：
  heart_beat -> tick -> update_condition()
  -> vote_clear.c::update_condition(me, duration)
    |-- duration 逐 tick 递减
    |-- duration < 1 且 > -5：广播 "投票取消了"，清理 vote 状态
    |-- duration < -5（已完成投票）：静默清理

滥用惩罚：
  vote.c::main() -- 若 vote() 返回 <= 0（投票失败）
  |-- me->vote/abuse > 50 时：
  |     set("vote/deprived", 1)
  |     apply_condition("vote_suspension", 120)  -- 120 tick 暂停期
  |-- vote_suspension.c::update_condition() 到期恢复
```

**关键数据状态（存储在 victim 的 dbase 上）**：vote/reason / vote/juror / vote/count / vote/abuse / vote/deprived / chblk_on / chblk_rumor / chblk_chat / chblk_menpai / chblk_gchat。

### 4. 与核心循环的关系

vote 系统通过 **condition 系统**间接接入 heart_beat：

- **heart_beat（非均匀 tick）**：char.c 的 heart_beat() 中，tick 计数器每 5+random(10) 个 heart_beat 周期触发一次 update_condition()。vote_clear 和 vote_suspension 作为 condition daemon 在此被调用。
- **channeld**：玩家每次使用频道时 channeld.c::do_channel() 检查 chblk_on 标志，是被动检查。
- **不接入 go/move/combat/valid_leave**：vote 系统完全独立于移动和战斗循环。
- **shout 广播**：投票进展通过 shout() 全局广播，不依赖房间位置。

### 5. themed 治理属性

**判断：平台级 fail-closed（不落入 UGC）**

理由：
1. 投票规则硬编码在平台命令中（阈值比例 1/4 和 1/6、最低票数 4 票、超时时限 10 tick、滥用惩罚阈值 abuse>50、暂停期 120 tick 均不可修改）。
2. 执行结果是平台级强制（投票通过后直接 set("chblk_on", 1)，channeld 每次发言强制检查，玩家无法绕过）。
3. 防滥用机制是 fail-closed 设计（滥用投票权会被剥夺投票权并暂停 120 tick）。
4. 动议类型是封闭枚举（V_EJECT/V_CHBLK/V_UNCHBLK/V_ROBOT），不支持 UGC 扩展新动议类型。
5. vote 系统是玩家自治治理工具，但规则框架本身是平台定义的，UGC 只能使用这个工具（发起投票），不能修改工具的规则。

CLAUDE.md 中"themed 治理（天雷/阴间/vote/法院）是平台级 fail-closed Python"的判断得到验证。

### 3. 核心数据流

**触发条件（通缉施加路径）**：

1. **PK 杀人在城市区域** -- combatd.c:killer_reward() 检测 strsrch(file_name(env), "/d/city/") >= 0，施加 killer condition 100 tick
2. **攻击执法 NPC** -- 巡捕/捕头 accept_kill() 施加 killercondition 100 tick；北京侍卫 accept_kill/accept_fight() 施加 bjkiller condition 100 tick
3. **攻击城市 NPC** -- 大量城市 NPC 的 accept_kill() 施加 killercondition 100 tick
4. **大理/西夏区域犯罪** -- 大理 NPC 施加 dlkiller，西夏同理
5. **PK 红名累积** -- killer_reward() 中 PvP 且有 pking 标记时施加 pkercondition +120 tick

**状态变更**：

| 状态 | 载体 | 效果 |
|------|------|------|
| killercondition (100) | conditions["killer"] | 城市巡捕自动追杀 |
| xakiller / dlkiller / bjkiller | 同上 | 对应区域执法者追杀 |
| pkercondition (累积) | conditions["pker"] | >240 时禁止 kill/feed/wei/throw 对玩家 |
| PKS 属性 | dbase["PKS"] | >19 且低效率时上通缉榜；>100 称号变"土匪" |
| city_jail / dali_jail / bonze_jail | conditions[...] | 禁闭在监狱房间，到期移出 |

**输出（惩罚）**：
- **通缉追杀**：巡捕/捕头/侍卫 init() 检测 query_condition("killer")，call_out("kill_ob", 1, this_player()) 自动攻击；pursuer 标记使 NPC 跨房间追踪
- **审判收监**：李克秀 do_proceed() -- 穿琵琶骨（embedded 物品）、清空 inventory、按 PKS 分级施加 city_jail（200/300/500/600 tick）、转移经验给逮捕者、10 金赏金、全服通报
- **死亡惩罚**（death_penalty()）：combat_exp 扣 1%（上限 5000）、potential 减半、balance 减半、skill_death_penalty 扣技能、death_count+1
- **受贿销案**：李克秀 accept_object() 收钱后将 killercondition 设为 0（金额 >= combat_exp/10）
- **自我投案**：PKS>19 且低效率的玩家找李克秀"工作"时直接被抓，city_jail 50 tick

### 4. 与核心循环的关系

**heart_beat（tick=1s）**：condition.c:update_condition() 在 heart_beat 中每 tick 调用，递减所有 condition 计数器（killer/pker/jail 等）。condition 到期时触发对应效果（如 jail 到期移出监狱房间）。

**combat（kill_ob / fight_ob）**：
- kill_ob() 将目标 id 加入 killer 数组（is_killing(id) 返回真），标记"性命相搏"关系
- fight_ob() 仅加入 enemy 数组（切磋关系），不加入 killer
- remove_enemy() 在 is_killing() 为真时返回 0（不能停止追杀），只有 remove_killer() 才能清除
- die() 中调用 remove_all_killer() 清除所有追杀关系

**PK 杀人后触发法院机制的完整链路**：
```
玩家 A kill 玩家 B
  -> A.kill_ob(B)  [killer 数组标记]
  -> B 被 A 杀死: B.die()
    -> COMBAT_D->killer_reward(A, B)
      -> A.PKS += 1
      -> 若在 /d/city/: A.apply_condition("killer", 100)  <-- 通缉触发
      -> 若 A 有 pking 标记: A.apply_condition("pker", current+120)  <-- 红名叠加
  -> 巡捕 init() 检测 A.query_condition("killer") > 0
    -> call_out("kill_ob", 1, A)  <-- 执法者自动追杀
    -> pursuer=1: follow_me 跨房间追踪
```

**move / valid_leave**：pursuer 标记的 NPC 通过 follow_me() 追踪逃犯移动方向；巡捕限定在 /d/city/ 区域活动，离开时自动 go_home() 回城。

### 5. themed 治理属性

**判断：平台级 fail-closed（不落入 UGC 可编辑）**

理由：
1. **通缉 condition 施加是硬编码逻辑**：killer_reward() 中 strsrch(file_name(env), "/d/city/") >= 0 触发通缉，是 combatd 的 C 级核心逻辑，非 UGC 可配置规则。各区域 condition（killer/xakiller/dlkiller/bjkiller）也是编译期固定。
2. **量刑分级是硬编码**：李克秀 do_proceed() 中 PKS>99 => 500 tick、PKS>74 => 300 tick、PKS>49 => 200 tick，完全是 if-else 硬分支，无配置接口。
3. **执法 NPC 行为不可编辑**：巡捕/捕头的 accept_kill() 施加 killercondition 是 inherit NPC 的方法覆盖，不属于 UGC 可编辑的 room/npc 脚本范畴。
4. **vote 系统是平台级社区治理**：投票关闭频道是平台行为，不是游戏世界内容。vote.c 中 16 岁门槛、滥用投票权剥夺机制均为硬编码。

### 2. 职责

intermud 系统是跨 MUD 网络通信层，负责在多个侠客行节点之间进行服务发现、心跳保活、以及跨 MUD 的频道广播、私语、finger、who、邮件、玩家定位等数据交换。基于 UDP 数据报协议（主信道）和 TCP 流式连接（inetd 辅助信道），通过 dns_master.c 集中管理 MUD 注册表、消息路由和请求序列化。系统通过白名单（LISTNODES/xkx_muds）严格限制可信节点，伪造消息检测（HOSTADDRESS 比对 + warning 协议）提供安全防护。

### 3. 核心数据流

**连接建立**：
```
dns_master.create()
  -> startup_udp()                           // 绑定 UDP socket (port = mud_port + 4)
  -> init_database()                          // 向 LISTNODES 引导服务器发送 startup 消息
      -> send_udp(bootsrv, "@@@startup||NAME:...||PORTUDP:...@@@")
      -> call_out("init_database", 60)        // 60 秒重试直到获得 mudlist
```

**服务发现**：
```
mudlist_a.incoming_request(info)             // 收到 MUD 列表应答
  -> process_list()                           // 逐条解析 MUD 信息
      -> 验证 NAME 在 LISTNODES 白名单中
      -> 验证 HOSTADDRESS 与白名单一致
      -> DNS_MASTER->set_mud_info(name, New)  // 写入 muds 映射
          -> query_services()                 // 查询新 MUD 支持的服务
```

**消息路由**：
```
UDP 数据报到达
  -> dns_master.read_callback(sock, msg, addr)
      -> 解析 "@@@func||key:val||key:val@@@" 格式
      -> 验证 xkx_muds 白名单（NAME/HOSTADDRESS/PORTUDP 三重比对）
      -> (AUX_PATH + func)->incoming_request(args)  // 分发到对应 service daemon
```

**心跳保活**：
```
dns_master.do_pings()                        // 每 30 分钟（PING_INTERVAL）
  -> 遍历所有已知 MUD
  -> muds[name][DNS_NO_CONTACT]++             // 递增无联系计数
  -> PING_Q->send_ping_q(host, port)
  -> 若 DNS_NO_CONTACT >= MAX_RETRYS(3)       // 超过 3 次未回应
      -> zap_mud_info(name)                   // 从数据库删除

dns_master.refresh_database()                // 每 5 分钟（REFRESH_INTERVAL）
dns_master.sequence_clean_up()               // 每 1 小时（SEQ_CLEAN_INTERVAL）
```

### 4. 与核心循环的关系

**intermud 系统与核心游戏循环（go/move/combat/heart_beat/valid_leave）完全无关，是独立的网络通信层。**

- **不接入 heart_beat**：dns_master 不使用 set_heart_beat()，而是使用 call_out() 实现周期任务（refresh_database、do_pings、sequence_clean_up、init_database）。这些是网络维护任务，与游戏 tick 无关。
- **不接入 go/move/valid_leave/combat**：intermud 不涉及房间移动、战斗。
- **唯一的游戏侧接触点是频道系统**：channeld.c 中的 gwiz 和 gchat 频道通过 intermud 字段挂接 GWIZ->send_msg() / GCHANNEL->send_msg()，将本地频道消息广播到其他 MUD。intermud 收到远端消息后通过 CHANNEL_D->do_channel() 投递到本地频道。
- **命令层接触点**：tell user@mud、finger user@mud、who@mud、locate、mudlist 等命令调用 intermud 服务，但这些是用户命令的独立分支，不与核心游戏逻辑交织。

### 5. themed 治理属性

**判断：平台级 fail-closed，不落入 UGC 可编辑规则层。**

理由：
1. **安全敏感性**：intermud 系统直接操作网络 socket，处理跨 MUD 的消息路由和身份验证。dns_master.c 中有明确的 ACCESS_CHECK、白名单（xkx_muds/LISTNODES）、伪造检测（HOSTADDRESS 比对 + warning 协议）。这些安全机制必须由平台控制，不可委托给 UGC。
2. **协议不可变性**：UDP 消息格式（@@@func||key:val||key:val@@@）、服务发现流程、心跳机制都是网络协议层面的约定，UGC 不应修改。
3. **与架构不变量一致**：CLAUDE.md 明确"themed 治理是平台级 fail-closed Python"。intermud 系统虽然不是 themed 治理本身，但其安全边界（可信节点管理、消息验证）同样是平台级关注点。
4. **UGC 可编辑的部分仅限频道名称映射**：channeld.c 中定义 gwiz/gchat 频道的 intermud 字段指向哪个 service daemon，这属于频道配置而非 intermud 协议本身。

### 7. 后置阶段建议

**建议：砍掉或无限期后置。intermud 系统违反项目收缩约束，不应在可预见未来实现。**

#### 违反收缩约束分析

项目六条收缩约束中有两条直接相关：

1. **"不考虑分布式架构（验证 UGC 成立前）"**：intermud 本质上是分布式 MUD 网络架构，需要在多个 MUD 节点间共享状态（MUD 列表、玩家在线状态、频道消息）。直接违反此约束。
2. **"不考虑分布式网关（单机 1000 在线+100 并发）"**：intermud 的 UDP 通信和 TCP inetd 连接构成跨节点网关，dns_master 本身就是一个分布式服务发现网关。违反此约束。

#### 可行性评估

- 侠客行原版的 intermud 网络基于 1990 年代的 TMI-2 / IIPS 协议，依赖 UDP 数据报 + 自定义文本格式（@@@func||key:val@@@），在现代网络环境中已过时。
- LISTNODES 白名单中配置的 IP 地址（202.99.174.190 等）早已失效，原版 intermud 网络实际已不可用。
- 新引擎的目标是单机 1000 在线 + 100 并发的独立 MUD 服务，不需要跨 MUD 通信。

#### 如果未来需要实现

若项目后期确实需要跨 MUD 通信（例如多服联动），建议：
1. **阶段 M4+ 之后**：远在外部玩家测试之后，且仅在 UGC 验证成立后。
2. **使用现代协议**：替换 UDP 文本协议为 gRPC/WebSocket + Protobuf/JSON，不要照搬 LPC 的 @@@ 格式。
3. **独立网络层模块**：intermud 应作为独立的 NetworkService 模块，通过事件总线与游戏核心解耦，不侵入 channeld 等游戏逻辑。
4. **保留平台级安全控制**：可信节点白名单、消息签名验证、速率限制必须由平台 Python 层管理，不落入 UGC。
5. **频道跨服广播可独立实现**：如果只需要跨服聊天，可在频道系统中预留 intermud_relay 接口，具体网络实现后置。这是 intermud 中唯一对游戏体验有实际价值的功能。
