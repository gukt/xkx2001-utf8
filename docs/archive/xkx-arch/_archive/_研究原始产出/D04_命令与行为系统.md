# 方案设计·命令与行为系统

## 设计概述
将 LPC 的"动词->handler->main(me,arg)"同步阻塞管线重构为"输入中间件链->动词路由->能力授权->动作调度"的异步显式管线，与 WebSocket 网关、Player Actor、ECS 世界、事件总线、DSL/UGC 平台对齐。核心理念：① 用显式 ActionContext 替代 this_player()/previous_object() 隐式上下文，能力令牌(CapabilityToken)替代 previous_object/geteuid 隐式信任链；② 用自注册 CommandManifest(声明动词/别名/参数schema/所需能力/冷却/help)替代 COMMAND_D 的 get_dir 目录扫描缓存与 command.h 的 8 套硬编码 *_PATH；③ 用统一中间件管线(解码->限流->历史->别名->路由->授权->执行->审计)拆解 command_hook 这个耦合出口捷径/命令/emote/频道四类路由的 god-function；④ 用能力策略引擎(主体+动作+资源)替代 securityd.valid_cmd 的目录级 ACL，修复 authorized_cmds[\"cmd\"] 拼写 bug 与 cmds/std|skill|usr 硬编码 return 1 绕过；⑤ 把 feature/action.c 的单 busy 标志升级为可序列化、可迁移、可中断的动作队列(ActionScheduler)，替代 931 处非持久 call_out 与闭包回调；⑥ 合并 aliasd 硬编码 global_alias 与个人 alias 为分层 AliasService；⑦ 保留动词->Action(ctx)->result 契约、help() 自文档化、解析回退链、exclude-before-trust 授权顺序、!/!N 与 $1/$* 的 MUD 式 UX、操作审计实践。force_me 的 ROOT_UID 后门收敛为带审计日志的显式 PrivilegedAction API。Big5->GB 转码下沉到 WebSocket 传输层，内核统一 UTF-8。

## 组件
- **InputPipeline（输入管线，替换 process_input）**：WebSocket 输入到路由意图的有序中间件链。拆解 alias.c 的 5 职责为独立中间件：UTF-8 解码中间件（消除 Big5->GB 内联转换，编码下沉到传输层）、限流中间件（Redis 令牌桶替代对象内 cnt 计数器与天雷惩罚，提供全局视角与分布式共享）、历史中间件（保留 !/!N 与 $1/$* 位置参数 UX，持久化按 session）、别名展开中间件（调用 AliasService 统一展开系统/世界/用户三级别名）、归一化中间件（去前导空格、动词/参数切分）。输出标准化的 InputIntent(verb, args[], raw, session_ctx)。无状态可水平扩展，部署在网关层。（技术：Python asyncio 自研中间件链 + aiolimiter(Redis 令牌桶) + structlog）
- **CommandRegistry（命令注册表，替换 COMMAND_D.find_command+rehash）**：命令元数据的权威登记表，替代 get_dir 目录扫描缓存。命令自注册时声明 Manifest(verb, aliases[], category, args_schema, required_capabilities[], cooldown_ms, priority, help{usage,examples,summary}, handler_ref, cpk_id, version)。DB 持久化支持 UGC 动态命令与热重载；版本标签 + 原子切换替代无失效信号的 rehash 缓存。冲突检测：同动词多注册按 priority + cpk 优先级解析，替代 imm/wiz 间 12 个同名命令的隐式 path 覆盖。读多写少，按 shard/世界分区缓存。（技术：PostgreSQL(权威) + Redis(读缓存，版本标签失效) + msgspec(Manifest 序列化)）
- **VerbRouter（动词路由器，替换 command_hook god-function）**：把 command_hook 的四类回退（出口捷径->命令->emote->频道）重构为显式有序的解析器管线（Ordered Resolver Chain），每个解析器是独立可插拔单元。ExitResolver 把出口名转为 go 动作（保留移动捷径语义，下沉为真实动作而非路由特例）；CommandResolver 查 CommandRegistry；EmoteResolver 调 EMOTE 服务；ChannelResolver 调频道服务。返回 ResolvedAction(handler_ref, args, source) 或 null。解析顺序声明式可配，支持多题材不同路由策略。（技术：Python 解析器链 + 声明式顺序配置(YAML)）
- **ActionScheduler（动作调度器/队列，替换 feature/action.c busy+call_out）**：把单状态 busy 标志升级为每 actor 的可调度动作队列。支持 immediate/delayed/repeating 三类（delayed 替代 931 处 call_out，repeating 替代自递归 call_out 链如 natured）。动作是显式可序列化的 Effect 对象（替代闭包回调，解决无法跨进程迁移/对象销毁后悬空指针的核心痛点），随 Player Actor handoff 时序列化进快照迁移。提供 cancel/interrupt（保留 interrupt_me 语义）、前置条件校验（is_busy/query_busy 迁移为队列状态）、tick 推进（替代 heart_beat 中调 continue_action）。确定性随机按战斗/区域分配 seeded RNG，回合结果写事件日志支持回放。（技术：asyncio 定时任务 + APScheduler(持久化延迟作业) + Redis ZSET(分布式调度) + msgspec(Effect 序列化)）
- **PermissionService（权限服务，替换 securityd.valid_cmd 路径 ACL）**：统一策略引擎按 (subject, action, resource) 求值，替代目录级 authorized_cmds/exclude_cmds。单命令粒度授权（修复 valid_cmd 对 cmds/std|skill|usr 硬编码 return 1 绕过 ACL 的缺陷）。保留 exclude-before-trust（拒绝优先）授权顺序。角色(RBAC)+属性(ABAC)混合：role_definition 继承链替代 enable_player 的 8 套 *_PATH 硬编码 switch，角色->能力集合可配置而非改代码。授权结果可缓存（subject+verb+resource 为键，grant/revoke 时按 subject 失效）。修复 securityd 第 757 行 authorized_cmds["cmd"](单数)拼写 bug--改用类型化 schema 校验，拼写错误在启动期即失败。（技术：Python 自研策略引擎 + casbin(RBAC/ABAC 策略建模) + Redis(授权缓存)；OPA/Rego 作为可选重型后端）
- **CapabilityService（能力声明服务，previous_object->显式能力令牌）**：把 previous_object()/geteuid(previous_object) 的隐式调用栈信任链显式化为能力令牌(CapabilityToken)。ActionContext 携带 token 与 trace，危险动作(move_player/destroy/persist/log_file/force)需提升权限并写审计。能力清单显式声明(read_world/say/move_self/spawn_in_scene/schedule/privileged_force 等)，对齐 LPC set_temp 状态机+log_file 审计但显式化。capability_grant 表支持临时授权(expires)与按资源模式(resource_pattern)授权。force_me 的 ROOT_UID 后门（4 处调用点）收敛为带审计日志的 PrivilegedAction API。（技术：Python 能力令牌库 + 结构化审计(append-only PostgreSQL)）
- **AliasService（别名服务，合并 aliasd+个人 alias）**：合并全局 alias(aliasd.c 硬编码 global_alias)与个人 alias(feature/alias.c 的 alias mapping)为单一分层服务：系统默认层 + 世界覆盖层 + 用户覆盖层，按优先级叠加。数据驱动（DB 存储）替代硬编码 mapping，运行时可配置。保留 $1/$* 位置参数与多词动词展开。查询读多写少，缓存于 Redis。（技术：PostgreSQL(分层存储) + Redis(读缓存)）
- **ActionContext / ContextService（上下文服务，替换 this_player/this_object/previous_object 隐式上下文）**：显式上下文对象，替代 LPC 隐式全局上下文。包含 actor_id、session_id、world_id、shard_id、capability_token、call_chain(trace)、request_id、timestamp。每个 action handler 接收 ctx 作为首参，消除 this_object()/this_player(1)/previous_object() 的隐式依赖，使行为可静态分析与跨进程传递。trace 链替代 previous_object 信任链，保留可审计的调用来源。（技术：Python dataclass(msgspec 结构) + 分布式追踪(OpenTelemetry trace context 传播)）
- **AuditService（审计服务，替换 CMD_LOG/READ_LOG/WRITE_LOG/FLOODER 平文件日志）**：结构化审计日志，替代 sprintf 平文件日志。CommandExecuted/CommandRejected/CommandFailed/CapabilityGranted/CapabilityRevoked/PrivilegedAction 事件追加写入事件存储(PostgreSQL append-only)并经 NATS 扇出供实时监控与反作弊。保留并增强 XKX 的操作审计实践（wiz 登录 IP 白名单、promotion_log、操作审计）。（技术：structlog + PostgreSQL(append-only) + NATS JetStream(扇出)）

## 关键接口/事件
- CommandManifest(数据类): verb:str, aliases:list[str], category:str, args_schema:msgspec.Struct, required_capabilities:list[str], cooldown_ms:int, priority:int, help:HelpMeta, handler_ref:str, cpk_id:str|None, world_id:str, version:int - 替代 cmds/*.c 的 main(me,arg)+help(me) 双契约
- async def execute(ctx: ActionContext, args: Any) -> ActionResult - 统一命令 handler 契约，替代 main(object me, string arg)。返回 ActionResult(status: Literal['ok','failed','rejected'], message:str|None, effects:list[Effect], data:Any) 替代 notify_fail/return 0/return 1 的隐式返回值
- async def resolve(intent: InputIntent, ctx: ActionContext) -> ResolvedAction|None - 路由解析器接口，VerbRouter 管线中每个 resolver(ExitResolver/CommandResolver/EmoteResolver/ChannelResolver) 实现；返回 ResolvedAction(handler_ref, args, source) 或 None 交由下一解析器
- InputIntent(数据类): verb:str, args:list[str], raw:str, session_id:str, timestamp:float - InputPipeline 输出、VerbRouter 输入
- ActionContext(数据类): actor_id:str, session_id:str, world_id:str, shard_id:str, capability_token:CapabilityToken, trace:CallTrace, request_id:str - 显式上下文，替代 this_player()/this_object()/previous_object() 隐式全局
- CapabilityToken: subject_id, role:str, capabilities:frozenset[str], expires_at, signature - 替代 previous_object()/geteuid 隐式信任链；危险动作需提升权限(privileged scope)
- ActionScheduler 接口: schedule(actor_id, effect:Effect, delay:float=0, repeat:bool=False) -> action_id; cancel(action_id); interrupt(actor_id, source, how) -> bool; query_state(actor_id) -> ActionQueueState - 替代 start_busy/is_busy/interrupt_me/continue_action 与 931 处 call_out
- Effect(可序列化动作对象,替代闭包回调): effect_type:str, payload:msgspec.Struct, seeded_rng_state:bytes|None, target_actor_id:str|None - 随 Actor 迁移可序列化，解决 LPC 闭包无法跨进程/对象销毁后悬空指针
- PermissionService 接口: async def authorize(ctx: ActionContext, action:str, resource:str) -> Decision(allow:bool, reason:str, matched_policy_id:str); async def grant(subject_id, capability, resource_pattern, effect='allow', expires=None); async def revoke(grant_id) - 替代 valid_cmd(file,user,func)；exclude-before-trust 在 Decision 求值顺序中保留
- CommandRegistry 接口: async def register(manifest: CommandManifest); async def lookup(verb:str, world_id:str, ctx: ActionContext) -> CommandManifest|None; async def hot_reload(cpk_id:str) - 替代 COMMAND_D.find_command+rehash；register 由命令自注册或 CPK 装载触发
- AliasService 接口: async def expand(input:str, ctx: ActionContext) -> str; async def set_alias(subject_id, verb, expansion, scope); async def list(scope, world_id) - 合并 aliasd.process_global_alias 与个人 alias
- 事件: CommandExecuted(actor_id,verb,args,result,shard,ts) / CommandRejected(actor_id,verb,reason) / CommandFailed(actor_id,verb,error,trace) / ActionScheduled(action_id,actor_id,effect,at) / ActionCompleted(action_id) / ActionInterrupted(action_id,source) / CapabilityGranted(subject_id,capability,by) / CapabilityRevoked(grant_id,by) / CommandManifestUpdated(verb,version) - 经事件总线(NATS)扇出，投影到审计/度量/反作弊读模型
- PrivilegedAction API: async def privileged_force(actor_id, command:str, granted_by:CapabilityToken, reason:str) -> ActionResult - 替代 force_me(me,cmd) 的 ROOT_UID 后门；强制写审计日志，reason 必填

## 数据模型
权威存储 PostgreSQL + 热缓存 Redis，按实体 ID 分片。\n\n1) command_manifest（命令注册表，verb 为逻辑主键，支持多世界/CPK 分区）\n   command_id UUID PK, verb, aliases JSONB, category, args_schema JSONB, required_capabilities TEXT[], cooldown_ms INT, priority INT, help JSONB{usage,examples,summary}, handler_module, handler_func, cpk_id, world_id, version INT, deprecated BOOL, created_at, updated_at\n   约束：UNIQUE(world_id, verb, cpk_id)；version 用于原子热重载\n\n2) command_alias（别名分层）\n   alias_id UUID PK, alias, expansion, scope ENUM(system/world/user), owner_id NULLABLE, world_id, created_at\n   索引：(alias, world_id, scope) 按 scope 优先级解析\n\n3) role_definition（角色->能力，替代 wizhood switch）\n   role_id TEXT PK, label, inherits_role TEXT NULLABLE, capabilities TEXT[], is_default BOOL\n   层级：(admin)>(arch)>(wizard)>(creator)>...(immortal)>(player)，沿用 10 级体系但可配置\n\n4) capability_grant（显式授权，替代 authorized_cmds/exclude_cmds）\n   grant_id UUID PK, subject_id, subject_type ENUM(role/user/cpk), capability, resource_pattern, effect ENUM(allow/deny), granted_by, granted_at, expires_at NULLABLE\n   索引：(subject_id, capability, resource_pattern)；deny 优先于 allow\n\n5) command_audit_log（append-only 审计）\n   log_id BIGSERIAL PK, ts, subject_id, verb, args JSONB, result ENUM(ok/rejected/denied/failed), shard_id, world_id, request_id, trace JSONB, latency_ms\n   分区：按 ts 范围分区；归档到对象存储\n\n6) action_queue（每 actor 动作队列，随 Actor 迁移）\n   action_id UUID PK, actor_id, action_type ENUM(immediate/delayed/repeating), state ENUM(pending/running/done/cancelled/interrupted), scheduled_at, payload JSONB(可序列化 Effect), capability_token_ref, source\n   索引：(actor_id, state, scheduled_at)\n\n7) command_history（会话历史，!N 回溯）\n   session_id, seq INT, input, ts；PK(session_id, seq)，TTL 清理\n\n8) rate_limit_state（分布式限流）\n   session_id, bucket, tokens FLOAT, last_refill_ts；Redis 令牌桶\n\n9) command_cooldown（冷却状态）\n   actor_id, verb, world_id, last_executed_at, cooldown_ms；TTL 键\n\n持久化策略：command_manifest/role/capability_grant 为权威真相(PG)；alias/history/rate_limit/cooldown 为热状态(Redis, TTL 或缓存失效)；action_queue 随 Actor 钝化时落 PG snapshots 表、激活时回填(复用 autoload 思想)；审计 append-only 不更新不删除。能力令牌(CapabilityToken)不在 DB 长存，由会话签发(JWT 内嵌 scopes)，吊销走 Redis 黑名单。

## 旧->新映射
- commandd.c find_command + rehash 目录扫描缓存（进程内可变全局，无失效信号） -> **CommandRegistry 自注册元数据表 + DB 持久化 + 版本标签原子切换**（消除 get_dir 扫描；新增/删除命令改注册即可热重载；UGC 命令随 CPK 装载注册）
- command.h 的 8 套 *_PATH 常量(ADM/ARC/WIZ/APR/IMM/PLR/UNR/NPC_PATH) -> **PermissionService 的 role_definition 继承链 + 角色->能力集合配置**（新增等级或调整权限改配置不改代码；NPC_PATH/UNR_PATH 作为角色模板而非硬编码）
- feature/command.c 的 command_hook god-function（出口捷径/命令/emote/频道四类路由全塞一处） -> **VerbRouter 显式有序解析器管线(ExitResolver->CommandResolver->EmoteResolver->ChannelResolver)**（解析顺序声明式可配；移动捷径下沉为真实 go 动作而非路由特例）
- feature/alias.c process_input（洪水控制+历史+Big5转码+个人alias+全局alias 五职责单函数） -> **InputPipeline 中间件链(解码/限流/历史/别名/归一化 各独立中间件)**（Big5->GB 下沉到 WebSocket 传输层，内核 UTF-8；cnt 天雷改 Redis 令牌桶）
- aliasd.c 硬编码 global_alias mapping + feature/alias.c 个人 alias 双存储 -> **AliasService 三层(系统默认/世界覆盖/用户覆盖)统一服务**（数据驱动替代硬编码；运行时可配置；保留 $1/$* 与多词动词展开）
- securityd.c valid_cmd 的目录级 authorized_cmds/exclude_cmds ACL -> **PermissionService 按 (subject,action,resource) 求值的能力策略引擎，单命令粒度**（修复 cmds/std|skill|usr 硬编码 return 1 绕过；保留 exclude-before-trust 顺序）
- securityd.c 第 757 行 authorized_cmds["cmd"](单数)拼写 bug -> **类型化 schema 校验，键名拼写错误在启动期即失败**（根本性消除同类拼写 bug；manifest/role 定义经 pydantic/msgspec 校验）
- feature/command.c enable_player 的 wizhood->path switch(十余分支) -> **PermissionService 角色解析 + 能力集合计算**（新增 wizhood 等级改配置；角色继承链替代 switch）
- feature/command.c force_me 的 ROOT_UID previous_object 后门(无审计) -> **PrivilegedAction API + 强制审计日志(reason 必填)**（4 处调用点收敛；NPC 脚本/系统驱动场景 API 形态对等覆盖）
- previous_object()/geteuid(previous_object) 隐式调用栈信任链(51 处) -> **ActionContext 携带 CapabilityToken + trace 显式化**（调用来源可审计；跨进程可传递；消除隐式依赖便于静态分析）
- feature/action.c 单 busy 标志 + interrupt_me + continue_action(在 heart_beat 调) -> **ActionScheduler 每 actor 可调度动作队列(cancel/interrupt/tick)**（支持多动作排队；队列状态替代单 busy；保留 is_busy 兼容 shim）
- call_out 非持久定时器(931 处，含自递归链如 natured 昼夜) -> **ActionScheduler delayed/repeating + 持久化 Effect 对象**（可序列化可迁移，随 Actor handoff 迁移；崩溃不丢；最大工作量需适配层过渡）
- feature/action.c start_call_out 的闭包回调(捕获对象引用，对象销毁悬空) -> **显式 Effect 数据对象 + seeded RNG 状态**（解决无法序列化/跨进程/悬空指针三大痛点；回放支持反作弊）
- cmds/*.c 单例蓝图对象 + call_other(file,"main",me,arg) 同步调用 -> **无状态 handler 函数 async execute(ctx,args)->ActionResult**（按请求实例化或池化；可测试；返回值语义显式化）
- 命令返回值 notify_fail/return 0/return 1 隐式约定(988 处 notify_fail) -> **ActionResult(status=ok/failed/rejected, message, effects, data)**（保留'未处理交下一解析器'回退语义；message 替代 notify_fail 文案）
- 每命令自带 help(object me) 函数体 -> **CommandManifest.help 声明式元数据(usage/examples/summary)**（随注册一并声明；可视化与文本双向编辑）
- cmds/std|skill|usr 目录硬编码可执行 -> **manifest.required_capabilities 显式声明 + 角色能力集合**（路径不再即权限；每命令独立授权）
- cmds/imm 与 cmds/wiz 间 12 个同名命令(clone/cat/cp/mv/rm 等)靠 path 顺序隐式覆盖 -> **manifest.priority 显式优先级 + 冲突检测**（注册期检测冲突；按 priority 解析；需基线测试断言原命中行为）
- 对象内 cnt 洪水计数器 + 天雷惩罚(单进程无全局视角) -> **网关层 Redis 共享令牌桶 + 退避(per-session/per-IP 双维度)**（分布式共享；可配规则；去除惩罚性逻辑改为限流）
- SECURITY_D wiz_status 纯内存态 + grant.c 改 .o 持久化 -> **role_definition/capability_grant PG 权威 + Redis 缓存**（分布式可共享；grant/revoke 实时失效；安全子系统本分支已损坏需先重建）

## 分布式扩展策略
分层水平扩展，区分无状态与有状态组件：\n\n1) 无状态层（网关侧水平扩展）：InputPipeline、VerbRouter、PermissionService(策略求值)、AliasService(读)、AuditService(写入)。WebSocket 网关用 SO_REUSEPORT 多 worker，单机数万连接；这些组件可任意扩缩容，会话路由表存 Redis(session_id->{node,character,world,shard})支持断线重连定位。\n\n2) 读多写少缓存层：CommandRegistry、role_definition、capability_grant、command_alias 读副本缓存于 Redis，版本标签失效（manifest 更新发 CommandManifestUpdated 事件，各 shard 监听后原子切换本地缓存，替代无失效信号的 rehash）。授权结果缓存键=(subject_id,verb,resource)，grant/revoke 时按 subject_id 批量失效。\n\n3) 有状态层（游戏分片内）：ActionScheduler 每 actor 一实例，与 Player Actor 同分片共置(co-located)。玩家跨分片 handoff 时动作队列序列化进快照写 Redis(TTL)，目标分片 restore 后回填，迁移期间消息日志重放保证不丢--复用现有 go.c 的冻结->快照->迁移->注销 handoff 模式分布化。同房间战斗参与者保持同分片亲和，COMBAT 实例化为区域/战斗级 Coordinator Actor 串行裁决回合，避免分布式事务。\n\n4) 限流分布式化：rate_limit_state 走 Redis 共享令牌桶（跨节点），替代对象内 cnt 计数器的单进程视角；per-session 与 per-IP 双维度，反刷屏规则可配。冷却状态(cooldown)用 Redis TTL 键。\n\n5) UGC/多租户分区：CommandRegistry 按 world_id/CPK 分区，UGC 命令仅在该题材世界可见；capability_grant 按 CPK 隔离，租户边界即能力边界。DSL 编译的命令注册到对应世界 manifest，运行时沙箱按 per-CPK 配额(eval-cost 对应物)隔离。\n\n6) 跨分片命令：纯本地命令(look/go/inventory)在分片内闭环；跨分片效果(tell/频道/远端查询)经事件总线(NATS pub/sub)路由，VerbRouter 的 ChannelResolver/远程 tell 走总线而非本地 call_other。边界房间建立跨分片事件代理避免每条消息走全局总线。\n\n容量估算：单分片 5000-8000 并发，1 万在线玩家需 5-8 分片；命令路由与权限校验 P99<2ms(缓存命中)，动作队列 tick 按 2s 确定性步进。

## 技术选型
- Python 3.11+ asyncio 为运行时主干，与 WebSocket 网关/aiohttp+uvloop 一致，不引入跨语言成本
- 输入管线：自研轻量 async 中间件链（参考 starlette 中间件形态），避免重型框架；限流用 aiolimiter(Redis 令牌桶)支持分布式共享
- 命令注册表：PostgreSQL 权威存储 + Redis 读缓存(版本标签失效) + msgspec 做 Manifest/args_schema 的类型化序列化与校验
- 权限策略：起步用 Python 自研策略引擎 + casbin(RBAC/ABAC 建模)，保留 OPA/Rego 作为规模化后的可选重型后端；不一开始引入 OPA 以控制运维复杂度
- 动作调度：asyncio 定时任务(即时/重复) + APScheduler(持久化延迟作业，替代 call_out) + Redis ZSET(分布式调度排序)；Effect 用 msgspec Struct 保证可序列化可迁移
- 别名/历史/限流/冷却：Redis 为主(读多写少+TTL)，结构化数据用 PostgreSQL 做权威
- 审计：structlog 结构化日志 + PostgreSQL append-only 表 + NATS JetStream 扇出给实时监控/反作弊投影
- 上下文追踪：ActionContext 携带 OpenTelemetry trace context，支持跨分片调用链追踪与性能剖析(替代 PROFILE_COMMANDS 的 rusage 内存计时)
- 序列化协议：内部 MsgPack(msgspec)高频路径，UGC/DSL 用 JSON，前端按 type 路由--与网关层消息协议一致
- 命令 handler 可选 WASM 沙箱(wasmtime-py)承载 UGC 脚本命令，燃料计量+能力化，对齐 DSL 层 3 逃生舱；可信编辑者可用受限 Python
- 不做：不直接用 Akka/Orleans(非 Python)；不引入 Kafka(单区域 PG+NATS 足够，跨区域多租户达规模再议)；不依赖 entt-py/esper 自研 ECS 与命令系统解耦

## 风险
- call_out 迁移规模巨大：931 个文件使用 call_out，许多是自递归链(natured 昼夜、复活计时、门关闭)，迁移到 ActionScheduler 的可序列化 Effect 需逐处重写闭包为数据化 Effect 对象，是本子系统最大工作量。需提供 LPC->Effect 适配桥接层做过渡期兼容，并优先迁移自递归 call_out 链（单点 miss 即整日时序错乱）。
- busy 语义深嵌：46 个命令文件检查 is_busy、heart_beat 调 continue_action、attack.c 把 busy 与战斗耦合。升级为动作队列时需保留 is_busy 语义兼容(shim)，否则命令行为漂移；队列与战斗回合的交互需明确（战斗中能否排队非战斗动作）。
- notify_fail/返回值语义微妙：988 处 notify_fail 与 return 0/1 的隐式约定(0=未处理交下一解析器、1=已处理、notify_fail 设错误文案)映射到 ActionResult(status,message) 时，必须保留'未处理则交由下一解析器'的回退语义，否则 emote/channel 回退链断裂。需建立行为基线测试集。
- previous_object 信任链迁移：51 处依赖 previous_object()/geteuid(previous_object) 做隐式授权。显式化为 CapabilityToken 时需逐一审计调用点，遗漏即产生权限绕过或误拒；尤以 start_call_out 的 ROOT_UID 检查(feature/action.c:64)与 force_me 的 ROOT_UID 后门(command.c:91)为最敏感点。
- force_me 调用点虽仅 4 处但多为 NPC 脚本/系统驱动场景，PrivilegedAction API 必须覆盖全部调用场景且 API 形态对等，否则 NPC 行为脚本断裂；force_me 当前无审计，迁移后强制审计可能暴露历史未记录的特权调用。
- securityd.c 在本分支已损坏(注释化的 create/restore，wiz_status 纯内存)：必须先重建权限数据模型与 wizlist 迁移，否则权限服务无基线可用；authorized_cmds 默认值在 securityd.c:48-53 与 722-727 两处硬编码易漂移，迁移时需以单一权威源合并。
- 动词冲突解析行为变更：imm/wiz 间 12 个同名命令(clone/cat/cp/mv/rm 等)当前靠 path 顺序隐式覆盖，改 priority 显式解析后可能改变实际命中命令。需对每个冲突动词建立基线测试断言'原命中哪个'，再迁移 priority 配置。
- UGC 命令沙箱必须先行：开放 UGC 脚本命令前，资源配额(每脚本每心跳指令上限/墙钟超时/内存/递归深度/call_out 配额)与能力审核必须就位，否则等于开放拒绝服务--直接对齐 config.xkx 的 eval-cost 语义固化为 per-CPK 配额，作为不可妥协硬约束。
- 动作队列跨进程迁移的正确性：Player Actor handoff 时动作队列序列化+恢复必须保证消息不丢、顺序不乱、确定性 RNG 状态随队列迁移；若 Effect 持有跨分片目标引用，需降级为 ID 引用+目标分片代理，否则迁移后动作指向错误分片。
- 能力粒度爆炸：从目录级细化到单命令级能力后，若缺少良好默认与角色模板，可能产生大量 per-user 微管理负担；需提供 role_definition 继承链与批量授权工具，避免权限治理复杂度反噬可用性。
- team.c 第 28 行 return 1 短路使组队子系统半瘫痪已久，新架构重建队伍为显式服务实体时，需确认是'修复到预期行为'还是'保留当前瘫痪态兼容'，避免改变长期玩家的既定体验。

**裁定**：sound（方向正确，分析扎实，可推进）但含多项实现级风险需在实施前解决。设计对旧系统的痛点定位精准——我逐一验证了 18 项具体声明（931 call_out、988 notify_fail 在 cmds/ 下精确匹配、51 previous_object、12 个 imm/wiz 同名命令、8 套 *_PATH、4 处 force_me 调用点、securityd.c:757 的 authorized_cmds["cmd"] 拼写 bug、securityd 已损坏的 create/restore、team.c:28 的 return 1 短路、natured 自递归 call_out、alias.c 的 5 职责、command_hook 四类路由 god-function、commandd.c 的 rehash 无失效信号、config.xkx 的 eval-cost=600000000），全部属实。核心理念（显式 ActionContext 替代隐式上下文、CapabilityToken 替代 previous_object 信任链、声明式 CommandManifest 替代目录扫描缓存、中间件链拆解 god-function、可序列化 Effect 替代闭包、能力策略引擎替代目录级 ACL）均是对症下药。但有若干刻画偏差与架构决策需纠正：最关键的是 call_out 迁移特征被错误归类，以及权限服务缺少 fail-open/fail-closed 策略定义。这些是实施前必须解决的安全与可行性问题，但不改变整体架构方向。

**严重度**：medium

### 问题与修复
- **call_out 迁移特征刻画有误：设计反复强调'闭包回调'为核心痛点，但实际代码中 931 个文件的约 3700 处 call_out 里，3570 处是字符串方法名调用（call_out("method_name", delay, args)），仅约 144 处使用闭包（function 类型，多通过 start_call_out）。字符串型 call_out 本质是 (对象, 方法名, 延迟, 参数) 四元组，天然可序列化，迁移为 Effect 是机械转换。真正的难点是对象生命周期管理（call_out 触发时对象可能已 destruct），而非闭包序列化。设计将 96% 的问题错误归类为闭包问题，导致迁移策略偏差。**
  - 影响：迁移策略可能过度投入在 144 个闭包上而低估 3570 个字符串 call_out 的批量迁移工程量；闭包的真正难点不是序列化而是对象引用有效性——闭包捕获的 me 对象在 call_out 触发时可能已被 destruct，需 target 存活校验而非纯数据序列化解决
  - 修复：重新分类迁移策略：(A) 3570 个字符串型 call_out（call_out("method", delay, args)）本质是 (object_id, method_name, delay, args) 四元组，天然可序列化，迁移到 Effect 是机械转换，可批量自动化；(B) 144 个闭包型 call_out（多为 start_call_out((: call_other, __FILE__, "method", me, args :)）模式规整，可先识别统一模式再转换。真正的难点不是序列化而是对象生命周期管理（call_out 触发时对象已被 destruct），Effect 需内置 target 存活校验 + 降级策略而非假设目标存在。给出按类别的迁移优先级与自动化工具规划。
- **权限服务 fail-open/fail-closed 策略未定义。PermissionService 授权结果缓存于 Redis，当 Redis 不可达时（网络分区、节点故障），设计未规定是 fail-open（放行，安全漏洞）还是 fail-closed（拒绝，拒绝服务）。这是分布式安全系统的基本决策点，缺失即安全隐患。**
  - 影响：分布式环境中 Redis 不可用是必然事件（网络分区、节点故障），无明确的 fail-open/fail-closed 策略意味着要么安全事故（fail-open 绕过权限）要么可用性事故（fail-closed 全员无法操作）。对于安全子系统，这是不可接受的架构级缺陷。
  - 修复：明确声明安全默认策略为 fail-closed（Redis 不可用时拒绝所有命令执行并返回降级提示），仅对 look/inventory 等只读无副作用命令可选 fail-open。PermissionService 内置本地最近快照（内存 LRU 缓存最近的策略快照），Redis 不可达时以快照 + fail-closed 兜底。在架构文档中将此作为不可妥协的安全约束记录。
- **APScheduler + Redis ZSET 双调度系统引入一致性风险。设计同时使用 APScheduler（持久化延迟作业）和 Redis ZSET（分布式调度排序）两套调度机制，但未说明两者的职责边界与一致性保证。一个 delayed Effect 可能同时存在于 APScheduler 的作业队列和 Redis ZSET 的调度排序中，产生重复触发或竞态。**
  - 影响：两个调度系统需保持一致：一个延迟动作可能在 APScheduler 中排队，同时其优先级在 Redis ZSET 中排序，两者更新不同步导致重复执行或丢失。这是分布式系统中最难排查的竞态类 bug。
  - 修复：统一为单一调度后端：APScheduler + Redis JobStore（或直接用 Redis ZSET + 自研轻量 worker 轮询）。APScheduler 的 Redis JobStore 已提供持久化延迟作业 + 分布式去重，无需额外 ZSET 层。若需严格优先级调度，在 JobStore 层按 actor_id 分桶即可。删除 ZSET 方案，避免双系统一致性问题。
- **ExitResolver 遗漏 command_hook 的 !arg 前置条件。原始 command_hook 的出口捷径仅在 !arg（无参数）时触发：'n' 执行 go north，但 'n foo' 会查找名为 n 的命令（参数 foo）。设计的 ExitResolver 描述为'把出口名转为 go 动作'但未保留此条件，可能改变移动捷径的语义边界。**
  - 影响：若 ExitResolver 无条件将方向词转为 go 动作，则 'n foo' 会变成 'go north foo' 而非查找命令 n，改变数千处依赖此行为的玩家习惯与脚本
  - 修复：ExitResolver 实现必须保留 exit-only-when-no-arg 条件：当 InputIntent.args 为空时查环境 exits 匹配 verb 并转为 go 动作；当 args 非空时跳过 exit 解析直接进入 CommandResolver。在基线测试集中加入断言：'n' -> go north，'n foo' -> 查找命令 n（预期未找到或 emote）。
- **authorized_cmds 默认值格式不一致 bug（设计未发现）。securityd.c:48-53 使用数组格式 ({"(admin)", "(arch)"})，但 725-726 行的 fallback 默认值使用逗号字符串格式 "(admin),(arch)"。valid_cmd 的 member_array() 检查期望数组，因此 725-726 行的逗号字符串 fallback 会静默失败——member_array 无法在字符串中匹配。设计提到了 ["cmd"] 拼写 bug 但遗漏了此格式不一致 bug。**
  - 影响：securityd 的授权兜底逻辑本身就有缺陷——当 authorized_cmds 为空时初始化的默认值格式与 member_array 检查不匹配，导致这些默认授权静默失效。迁移时若不修复，新系统的权限基线会继承此隐性缺陷。
  - 修复：新增 authorized_cmds 格式一致性检测：迁移时将所有授权配置统一为数组格式，并以 pydantic/msgspec schema 校验（List[str] 类型），启动期拒绝逗号字符串格式。在迁移检查清单中补充此 bug 为待修复项。
- **ActionScheduler 与 COMBAT Coordinator 的动作队列所有权未定义。设计中 ActionScheduler 每 actor 一实例，但战斗时设计提到 COMBAT Coordinator Actor 串行裁决回合。玩家在战斗中能否排队非战斗动作？战斗回合与动作队列 tick 如何协调？谁拥有 actor 的动作执行权？**
  - 影响：战斗中 ActionScheduler 与 COMBAT Coordinator 争抢同一 actor 的动作执行权会导致回合错乱、动作重复或丢失。MUD 战斗系统对时序高度敏感，此问题会直接影响战斗正确性。
  - 修复：明确 COMBAT Coordinator 与 ActionScheduler 的协议：战斗开始时 Player ActionScheduler 将非战斗动作挂起（suspend），COMBAT Coordinator 接管战斗回合调度（回合内仅允许战斗指令）；战斗结束后恢复（resume）ActionScheduler。队列所有权转移通过 ActionQueueState 状态机显式管理。在数据模型中补充 action_queue.state 增加 suspended 状态。
- **WASM 沙箱与 async handler 接口的桥接方案缺失。命令 handler 接口为 async def execute(ctx, args) -> ActionResult，但 wasmtime-py 承载的 WASM 模块不原生支持 async/await。WASM 命令执行需在事件循环中同步阻塞，与 asyncio 的非阻塞模型冲突。设计提及'可选 WASM 沙箱'但未说明此桥接方案。**
  - 影响：若 WASM-async 桥接无法在计划期内解决，UGC 命令子系统将无法交付，影响 DSL/UGC 平台的核心目标
  - 修复：定义三层降级策略：(1) 可信编辑者用受限 Python 子集（ast 白名单 + 超时）；(2) 不可信 UGC 用 WASM 同步执行，外层包 async wrapper（asyncio.to_thread 或 run_in_executor），超时用线程取消或进程隔离；(3) 明确 WASM 命令的 P99 延迟上限（如 50ms），超时降级为 '命令执行超时'。在 UGC 沙箱设计中将此桥接方案作为前置依赖。
- **Big5 编码转换'下沉到传输层'的架构定位有误。query("language")=="BIG5" 是 per-user 应用层设置，不同用户可能有不同语言偏好。将其下沉到 WebSocket 传输层混淆了用户偏好与传输编码协商。若多路复用连接（单连接多用户），传输层无法区分 per-user 编码。**
  - 影响：若按设计下沉到传输层，多用户共享连接时无法区分编码偏好；且将用户配置与传输协议耦合违反分层原则
  - 修复：保留编码转换为 InputPipeline 的应用层中间件（用户偏好层），不下沉到传输层。传输层统一 UTF-8，编码转换中间件按 ctx.actor_id 查询用户 language 偏好后做 Big5<->UTF-8 转换。明确标注此中间件的位置在限流之前（避免转换开销被限流误计）。
- **角色变更时的权限缓存失效策略不充分。设计提到 grant/revoke 时按 subject_id 失效，但 role_definition 继承链变更时，所有该角色的 subject 都需失效。'按 subject 失效'策略需枚举所有受影响 subject。**
  - 影响：角色级权限变更（如新增等级或调整能力集合）影响所有持有该角色的用户。'按 subject 失效'需先枚举所有受影响 subject，在大规模下可能产生缓存雪崩（数千用户同时重新计算权限）
  - 修复：定义角色级失效广播：role_definition 变更时发布 RoleCapabilitiesUpdated(role_id) 事件，各节点监听后按 role_id 批量失效缓存（本地维护 role_id->subject_ids 反向索引）。对于高频变更，考虑版本号比较（角色版本号嵌入缓存键，变更即版本号递增，旧缓存键自然失效）。在 scaleStrategy 中补充此失效协议。

### 改进建议
- 重新分类 call_out 迁移策略为两条路径：3570 处字符串型用自动化转换工具批量处理（模式固定：(object, method, delay, args) -> Effect），144 处闭包型逐一人工迁移。在迁移计划中给出按类别的工时估算与自动化工具规格，而非笼统称'最大工作量'。Effect 对象必须内置 target 存活校验（触发时检查目标 actor 是否存在，不存在则降级为 no-op 或日志告警），解决对象生命周期问题而非仅序列化问题。
- 在架构文档中明确声明权限服务的 fail-closed 默认策略为不可妥协的安全约束，并设计 Redis 不可达时的本地快照降级路径。这是分布式安全系统的前置决策，应在实现前而非实现后发现。
- 调度系统收敛为单一后端：APScheduler + Redis JobStore 或自研 ZSET worker 二选一，给出选择依据（延迟作业量级估算、优先级调度需求）。删除双系统方案，消除一致性风险。
- 补充 securityd.c authorized_cmds 格式不一致 bug（725-726 行逗号字符串 vs 48-53 行数组格式）到已知 bug 清单。迁移时统一为类型化 schema（List[str]），启动期校验拒绝格式不一致的配置。
- VerbRouter 的 ExitResolver 实现必须保留 command_hook 的 !arg 前置条件，并在基线测试集中加入 'n' -> go north / 'n foo' -> 查找命令 n 的行为断言。所有解析器都需建立此类行为基线测试。
- 在实现前定义 ActionScheduler 与 COMBAT Coordinator 的动作队列所有权协议：战斗中 ActionScheduler suspend、COMBAT Coordinator 接管、战斗后 resume。在 action_queue 状态模型中补充 suspended 状态。
- UGC WASM 沙箱与 async handler 的桥接方案应作为 UGC 子系统的前置 PoC 验证项。若 wasmtime-py 无法在 asyncio 事件循环中高效执行，应提前评估替代方案（如子进程隔离 + 超时杀掉），避免 UGC 命令成为阻塞依赖。
- 保留 Big5 编码转换为 InputPipeline 应用层中间件（按用户偏好），不下沉到传输层。传输层统一 UTF-8，编码转换在应用层按 ctx.actor_id 查询偏好。
- 补充权限缓存的角色级失效协议：role_definition 变更时发布 RoleCapabilitiesUpdated 事件 + 节点本地 role_id->subject_ids 反向索引批量失效，或用版本号嵌入缓存键实现自然失效。在 scaleStrategy 中量化缓存命中率目标与 cold-path P99 延迟目标。
- team.c 'return 1' 短路决策应在设计阶段而非实现阶段确定：调研当前玩家对队伍系统的实际使用情况，决定是修复（重建为显式服务）还是保留兼容（标注为已知缺陷）。此决策影响 TeamService 的接口契约。
