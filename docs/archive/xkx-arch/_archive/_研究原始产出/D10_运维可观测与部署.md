# 方案设计·运维可观测与部署

## 设计概述
当前 XKX 是单进程 MudOS：一个 driver 二进制 + portbind 端口转发，config.xkx 平文件配置，crash 后靠 master.c.crash() 给全服 shout 一句 + 追加 GBK 文本到 log/static/CRASHES，error_handler 把本地调用栈写到 debug.log，94 个 LPC 文件散落调用 log_file() 写 GBK 文本审计日志(WRITE_LOG/READ_LOG/CMD_LOG/grant_log/CALL_PLAYER/USAGE/regban)，无指标/无追踪/无健康检查/无 CI/CD/无灰度/无弹性，max users 硬编码 150，靠人工重启 driver 恢复。OBJ_DUMP 是唯一的实时拓扑视图(refcount + heart_beat flag + parent 指针)。新架构以"有状态分片 + 无状态网关"分层部署到 K8s，用 OpenTelemetry 统一 metrics/logs/traces 三支柱，Argo CD GitOps + Argo Rollouts 做金丝雀与回滚，KEDA + HPA 按游戏语义指标(tick 延迟、在线人数、handoff 队列)弹性伸缩，审计日志迁入 PostgreSQL append-only(复用 ES 设计的 EventStore)，将 master.c 的 crash/error_handler/valid_* 隐式运维钩子显式化为可观测的边界事件与探针。核心原则：(1) 有状态游戏分片不能套用无状态 HPA，需 region 亲和 + handoff 成本感知的弹性；(2) 战斗确定性要求版本门控，金丝雀只在无状态边缘先行，有状态分片用蓝绿；(3) trace context 必须强制注入 actor 消息信封，否则分布式调试不可能；(4) 审计与事件归档复用 ES 的 append-only 表，不另建存储。

## 组件
- **Deploy Orchestrator (部署编排器)**：GitOps 驱动的发布控制器。基于 Argo CD + Argo Rollouts，将 Helm chart 声明同步到集群，管理无状态网关 Deployment、有状态游戏分片 StatefulSet、守护进程服务的滚动/金丝雀/蓝绿发布。它消费 Release 记录，按 strategy(canary|bluegreen|rolling) 编排，并读取 CanaryAnalyzer 的健康判定决定 promote/rollback。替代当前手动重启 driver 二进制的发布方式。（技术：Argo CD + Argo Rollouts + Helm; Rollout AnalysisTemplate 读取 Prometheus 自定义指标）
- **Shard Lifecycle Manager (分片生命周期管理器)**：管理游戏分片(区域共同体)的激活/钝化/迁移。协调 region supervisor 的 placement、玩家 handoff(冻结->快照入 Redis 附 TTL->目标分片 restore->源分片注销)、组队批量迁移。它是唯一有权调用分片激活/钝化的组件，避免多控制器竞争。订阅 ReleaseManager 的事件，在版本切换时按分片粒度做版本门控(新版本分片先暖启动再切流量)。（技术：自研轻量 Operator (kopf 框架) 或先以 Helm + 手动运维起步; 状态依赖 NATS KV 做选主）
- **Metrics Collector (指标采集器)**：每个进程内嵌 prometheus_client 暴露 /metrics。采集三类指标:(1) 资源级(CPU/mem/fd/GC);(2) 游戏语义级(tick 延迟 p50/p99、tick 超限次数、活跃实体数、heart_beat 频次、AOI 订阅数、handoff 队列深度、call_out 积压、事件日志 lag);(3) 业务级(在线人数、战斗回合/秒、频道消息/秒、金币铸造/回收、登录/登出速率)。其中 tick 超限计数与 OBJ_DUMP 的对象图规模直接替代原 'Too long evaluation' 的隐式告警。（技术：prometheus_client (Python); Prometheus + Grafana; 自定义 game-metrics-adapter 暴露外部指标给 HPA/KEDA）
- **Structured Logger (结构化日志器)**：替代 efun log_file() 与 94 处散落的 GBK 文本日志。提供 log.debug/info/warn/error/critical，输出 JSON(UTF-8)。每条日志强制带 trace_id/span_id/shard_id/actor_id/世界标识，按领域路由到不同 Loki label(login/combat/economy/security/world/dsl)。Transition 期对历史 GBK 日志用 Vector 做 GBK->UTF-8 转码镜像，保留旧 grep 工作流一个窗口。（技术：structlog + python-json-logger; 采集用 Vector (支持 GBK 解码+路由) 优于 Promtail; 后端 Loki）
- **Distributed Tracer (分布式链路追踪器)**：替代 master.c standard_trace() 仅有的本地调用栈。OpenTelemetry SDK 自动插桩 asyncio/aiohttp/FastAPI；自研 Actor 运行时强制在消息信封携带 W3C traceparent，actor mailbox 入/出口自动创建 span，使跨分片跨服务的一次玩家命令(网关->player actor->combat coordinator->skill daemon->event store)成为一条完整 trace。采样策略:边界默认采样+慢 span 尾部采样+按 trace_id 调试采样。这是分布式化调试与公平性审计的前提(战斗回合事件可回放)。（技术：OpenTelemetry Python SDK + auto-instrumentation; OTel Collector; 后端 Tempo 或 Jaeger）
- **Health & Readiness Service (健康检查服务)**：暴露 K8s liveness/readiness/startup 三探针。liveness=进程事件循环存活+tick 未僵死(最近 tick 间隔 < 阈值);readiness=依赖(DB/Redis/NATS)可达+actor 运行时已激活+配置版本已加载;startup=分片暖启动完成(蓝图加载、region 注册、AOI 重建)。readiness 失败时网关摘除流量但不杀进程，避免 tick 抖动导致反复重启。替代当前无任何健康端点、崩了只能靠玩家断线感知的现状。（技术：FastAPI /healthz /readyz /startupz 端点; K8s probe）
- **Config Center (配置中心)**：替代 config.xkx 静态平文件 + adm/etc/* 一堆手工编辑的列表(banned_sites/approved_sites/noreg_sites/preload/day_phase)。配置分三层:(1) 启动配置=K8s ConfigMap+env(pydantic-settings 校验);(2) 运行时可热更新配置=NATS KV 或小型 Config Service，带版本号，在 tick 边界/epoch 应用避免与进行中 tick 竞态;(3) IP 规则/封禁表=Config Center 管理，下发到网关层。所有配置变更发 ConfigChanged 事件并审计。（技术：pydantic-settings (本地校验) + ConfigMap + NATS KV (热更新); 可选 etcd）
- **Release Manager (发布管理器)**：管理 Release 生命周期与版本门控。每个发布有不可变 release_id + version + manifest_ref + strategy + status。金丝雀时按分片粒度灰度:先无状态网关切 5%->25%->100% 流量，每阶段经 CanaryAnalyzer(读 tick_p99/错误率/handoff 失败率)判定;有状态分片用蓝绿(新版本分片暖启动 standby，handoff 切换)。回滚=快速切回上一个 release 的 manifest，因状态外部化(PG/Redis)可秒级回退逻辑而无数据丢失。版本门控确保战斗公式变更时同房间参与者版本一致，避免分叉。（技术：Argo Rollouts + 自研 Release API (PostgreSQL 记录) + Canary AnalysisTemplate）
- **Canary Analyzer (金丝雀健康分析器)**：为 ReleaseManager 的 promote/rollback 决策提供量化依据。对比金丝雀组与基线组的关键指标:tick p99 延迟、tick 超限率、HTTP/WS 错误率、handoff 失败率、DSL 校验失败率、WASM 燃料耗尽率、OOM/重启次数。规则可配置(如 tick_p99 > 1.5x 基线且持续 2 分钟则自动 rollback)。也支持 UGC 内容包(CPK)发布的灰度判定:新 CPK 上线后玩家投诉率/异常退出率对比。（技术：Prometheus Query + Argo Rollouts AnalysisTemplate; 规则 DSL 化）
- **Audit & Event Archive (审计与事件归档)**：替代 securityd 的 WRITE_LOG/READ_LOG/CMD_LOG/grant_log 文本审计与 CALL_PLAYER.c 特权调用审计。所有审计事件以结构化 AuditEvent 追加写入 PostgreSQL append-only 表(复用 ES 设计的 events 表基础设施)，按 (世界,租户,时间) 分区。特权动作(force_me 等价、文件写、权限授予、CPK 发布)均强制经此路径。支持按 actor/action/resource/trace_id 检索，供反作弊与合规回放。取代当前 GBK 文本无法聚合查询的现状。（技术：PostgreSQL append-only (分区) + 复用 EventStore; 检索层可选 ClickHouse 做高维查询）
- **Crash & Incident Handler (崩溃与事件处理)**：替代 master.c crash() 的 shout+追加文本。崩溃时:OTel 发 fatal 事件带 trace_id/stack/command_giver/current_object(对应原 crash 参数)->Alertmanager 告警值班->K8s 自动重启 pod(supervision one-for-one)->可选保留'系统核心惨叫'的 RP 风格通知给同分片玩家(沉浸式 vs 运维清晰的权衡项)。维护事件 runbook 与 on-call 轮值。tick 超限(原 'Too long evaluation')不再静默丢逻辑，而是发 TickOverrun 事件并计入指标。（技术：OTel events + Alertmanager + Grafana Oncall/PagerDuty; runbook-as-code）
- **Scheduler Service (持久化调度器)**：替代 natured 自递归 call_out 链(一次 miss 即整日时序错乱)与全库 3728 处非持久 call_out。任务持久化到 NATS JetStream/PG，崩溃重启后恢复；分布式选主保证单 region 单调度器实例；保留 remove_call_out 的取消语义。day_phase 映射为数据驱动的世界事件调度表。heart_beat 的 2s tick 作为确定性仿真步进，其延迟/超限成为一等指标。（技术：APScheduler 或自研 + NATS JetStream 持久化; 选主用 NATS KV lease）
- **CI/CD Pipeline (持续交付流水线)**：代码已在 GitHub。Pipeline: 代码提交->GitHub Actions(lint ruff/mypy + pytest 单测 + DSL schema 校验 + WASM 沙箱燃料/内存上限测试 + 不变量回归)->构建 distroless Python 镜像->推 GHCR->Argo CD 同步。DSL/CPK 内容包走独立 pipeline: schema 校验->依赖图检查->编译 JSON IR->安全能力审核->内容寻址入库->灰度发布。迁移期含 LPC->DSL 转译脚本的回归校验阶段。（技术：GitHub Actions + ruff/mypy + pytest + 自研 DSL 校验器 + GHCR + Argo CD）
- **Autoscaler (弹性伸缩器)**：三层弹性:(1) 无状态网关与守护进程服务:HPA 按 CPU+连接数/RPS;(2) 游戏分片:KEDA 按事件驱动(玩家会话数、handoff 队列深度)扩缩，单分片上限 5000-8000 并发，超阈值触发新 region 分片暖启动而非无限扩单分片;(3) 状态服务(PG/Redis/NATS):专用节点池不横向弹性，PG 读副本、Redis/NATS 集群固定拓扑。关键反抖动:冷却时间 + 仅对持续信号(滑动窗口)反应 + 预留容量，避免短突发导致分片反复迁移抖动。替代 max_users:150 硬编码上限。（技术：K8s HPA + KEDA (事件驱动) + 自定义 game-metrics-adapter）

## 关键接口/事件
- ShardActivated {shard_id, region, release_id, version, node, activated_at} — 分片激活广播，对应原 preload 完成 'Initializations complete'
- ShardDeactivated {shard_id, reason[drain|crash|scale_in|migrate], drained_players, snapshot_ref} — 钝化事件，必须先 drain 玩家再销毁
- PlayerHandoff {player_id, src_shard, dst_shard, snapshot_ref, trace_id, team_ids[]} — 玩家跨分片迁移，组队批量保持顺序；snapshot_ref 指向 Redis 快照
- TickOverrun {shard_id, tick_ms, budget_ms, trace_id, offending_span} — 替代 'Too long evaluation' 的静默丢逻辑，计入指标与告警
- CrashReport {shard_id, error, trace_id, stack, command_giver_id, current_object_id, ts} — 替代 master.crash(error,command_giver,current_object)，发 OTel fatal 事件+告警+自动重启
- HealthProbe {probe: liveness|readiness|startup, checks: {event_loop_alive, last_tick_age_ms, db_ok, redis_ok, nats_ok, config_version_loaded}} -> {status: ok|degraded|fail} — K8s 探针契约
- ConfigChanged {key, old_value, new_value, version, world_id, updated_by, trace_id} — 热更新事件，在 tick 边界 apply
- DeployEvent {release_id, version, strategy, phase: started|canary_5|canary_25|promoted|rolled_back, shard_scope} — 发布编排事件流
- CanaryDecision {release_id, verdict: promote|hold|rollback, reason, metrics: {tick_p99_ratio, error_rate_delta, handoff_fail_rate}} — 金丝雀健康判定
- AuditEvent {actor_id, actor_kind, action, resource, outcome: allow|deny, trace_id, ts, shard_id, context} — 替代 WRITE_LOG/CALL_PLAYER/grant_log，结构化追加
- MetricSample {name, labels{shard_id,world_id,...}, value, type: gauge|counter|histogram, ts} — Prometheus exposition 契约
- TraceSpan {trace_id, span_id, parent_span_id, service, op, start_ts, duration_ms, status, attributes} — OTel span 契约，actor 消息信封强制携带 traceparent
- ScheduleTask {task_id, type, run_at, payload, shard_id} / CancelTask {task_id} — 替代 call_out/remove_call_out，持久化可恢复

## 数据模型
权威存储 PostgreSQL(append-only events + snapshots 表)，热事件 Redis Stream，扇出 NATS JetStream。核心表:\n1. AuditEvent(shard_id, ts, trace_id, span_id, actor_id, actor_kind, action, resource, outcome, context_json, world_id, tenant_id) — 替代 securityd 文本审计与 CALL_PLAYER，按 (world_id, tenant_id, ts) 分区，append-only 不可变。\n2. Release(release_id PK, version, manifest_ref, strategy, status, shard_scope, created_at, promoted_at, rolled_back_at, rollback_of FK, analysis_summary_json) — 发布生命周期记录，manifest_ref 内容寻址保证可复现与回滚。\n3. ShardState(shard_id PK, region, release_id FK, node, status[activating|active|draining|deactivated], player_count, entity_count, tick_p50_ms, tick_p99_ms, tick_overrun_count, last_handoff_at, aoi_subscriber_count, updated_at) — 实时分片拓扑，对应原 OBJ_DUMP 的对象图+refcount 但结构化可查询，作为 Grafana 拓扑面板与 Autoscaler 输入。\n4. ConfigVersion(key, value_json, version, prev_version, updated_by, updated_at, world_id) — 配置版本链，支持回滚到任意版本，热更新在 tick 边界 apply。\n5. ScheduledTask(task_id, type, payload_json, run_at, shard_id, status, created_at) — 替代非持久 call_out，崩溃可恢复，支持取消(cancel 状态)。\n6. Incident(incident_id, started_at, severity, shard_id, trace_id, runbook_ref, status, resolved_at, postmortem_ref) — 崩溃/告警事件记录。\n时序数据: Metrics 进 Prometheus TSDB(默认 15d 热存+长期远程写)、Logs 进 Loki(标签分域 login/combat/economy/security/world/dsl)、Traces 进 Tempo(采样存储，按 trace_id 检索)。Handoff 快照存 Redis 附 TTL(对齐空间分片方案)。

## 旧->新映射
- driver 二进制 + portbind 端口转发 (config.xkx: port 8888) -> **容器化 WS Gateway (aiohttp+uvloop, SO_REUSEPORT) Deployment + Game Shard StatefulSet**（无状态网关与有状态分片分离，独立扩缩容；portbind 的端口复用思想保留为 SO_REUSEPORT）
- config.xkx 平文件 (eval cost 600M / max array 15000 / max users 150 / swap 120s / clean_up 180s / reset 1800s) -> **Config Center: ConfigMap(启动) + NATS KV(热更新) + pydantic-settings 校验**（eval cost 演进为 per-CPK 资源配额(WASM 燃料+超时+调用深度)；max_users 演进为 Autoscaler 弹性上限；swap/clean_up/reset 时间演进为调度器配置）
- adm/etc/preload 服务清单 (16 个 daemon 路径) -> **Helm values + 服务依赖清单 + 启动健康检查清单**（保留显式 preload 的声明式依赖思想，补 health/readiness 探针）
- master.c crash() -> log/static/CRASHES + shout 全服 -> **Crash & Incident Handler: OTel fatal 事件 + Alertmanager + K8s 自动重启 pod + 可选 RP 通知**（崩溃不再只追加 GBK 文本，而是带 trace_id 的结构化事件并自动恢复；保留'惨叫'的 RP 通知作为可选项）
- master.c error_handler()/standard_trace() 本地调用栈 -> debug.log -> **Distributed Tracer (OpenTelemetry) 跨服务跨分片全链路 trace**（standard_trace 的 program/line/object/function 字段映射为 span 属性；trace_id 经 actor 消息信封传播）
- efun log_file() GBK 文本 (94 处调用, 散落 log/ 与 log/static/) -> **Structured Logger (structlog JSON UTF-8) -> Vector -> Loki**（按领域分 label；Transition 期 Vector 对历史 GBK 做转码镜像保留旧 grep 工作流）
- securityd: WRITE_LOG/READ_LOG/CMD_LOG/grant_log 文本审计 -> **Audit & Event Archive (PostgreSQL append-only, 结构化可检索)**（合并 4 套审计文本为一套结构化 AuditEvent 表，支持聚合查询与反作弊回放）
- log/static/CALL_PLAYER.c (force_me ROOT_UID 后门审计) -> **AuditEvent(action=privileged_call) + 显式特权动作 API 带审计**（ROOT_UID 后门替换为带审计日志的显式特权 API，审计走同一 AuditEvent 路径）
- log/USAGE.c 登录/创角事件 + log/user_count 计数器 -> **结构化 login 事件 -> Metrics(gauge) + AuditEvent**（user_count 整数文件升级为 Prometheus gauge；登录事件升级为可查询事件）
- log/regban.log + adm/etc/banned_sites/approved_sites/noreg_sites IP 列表 -> **Config Center IP 规则 + 网关层速率限制 + 审计**（手工编辑平文件升级为配置中心管理并下发；保留 trusted/exclude 双向 ACL 思想）
- OBJ_DUMP 内存对象图 (refcount + HB flag + parent) -> **ShardState 实时拓扑表 + Grafana 拓扑面板 + Metrics**（OBJ_DUMP 的 refcount/HB/parent 字段结构化为 ShardState，可查询可告警）
- natured 自递归 call_out 链 (昼夜相位, 一次 miss 即错乱) -> **Scheduler Service (持久化, 崩溃可恢复, 分布式选主)**（day_phase 配置保留为数据驱动世界事件表；call_out 全库 3728 处映射为持久化 ScheduledTask）
- heart_beat 2s 全局 tick (单点瓶颈, 超限静默丢逻辑) -> **分片级确定性 tick + TickOverrun 指标/告警 + Autoscaler 输入**（tick 延迟/超限成为一等可观测指标，驱动弹性与告警）
- 手工重启 driver 二进制 (无 CI/CD, 无版本管理, 无回滚) -> **CI/CD Pipeline (GitHub Actions) + Argo CD GitOps + Release Manager 回滚**（状态外部化(PG/Redis)使逻辑回滚秒级且无数据丢失）
- max users:150 硬编码上限 (config.xkx line 138) -> **Autoscaler (HPA + KEDA + 自定义游戏指标) 弹性扩缩**（按分片 5000-8000 并发设计，1万在线需 5-8 分片）
- GBK/Big5 混合编码 (日志/传输/存储) -> **全链路 UTF-8 (传输层统一, 日志 JSON UTF-8)**（编码转换下移到采集层 Vector，内核编码无关；过渡期双写）
- queue/register 邮件注册队列 (平文件) -> **异步 onboarding 服务 (NATS 消费者) + 审计**（注册队列升级为消息总线消费者，可水平扩展）

## 分布式扩展策略
分四层差异化弹性，避免对有状态游戏分片套用无状态 HPA 的错误模式:\n1. 无状态边缘层(WS Gateway / Auth / Channel / 排行查询): HPA 按 CPU + WebSocket 连接数 + RPS 水平扩展，SO_REUSEPORT 多 worker，单机数万连接，跨节点会话经 Redis 路由表定位。\n2. 有状态游戏分片(区域共同体): KEDA 事件驱动，输入信号=该 region 在线玩家会话数 + handoff 队列深度 + tick_p99。单分片上限 5000-8000 并发(10万房/20万对象)；超阈值不无限扩单分片，而是暖启动新分片承接新 region 负载。玩家会话尽量粘附所在分片(粘性路由)，跨分片仅迁移玩家实体与必要快照而非世界数据。反抖动: 冷却时间 + 滑动窗口持续信号 + 预留容量，避免短突发触发反复 handoff 抖动。\n3. 有状态共享服务(PG/Redis/NATS): 专用节点池，固定拓扑不横向弹性。PG 用读副本分摊查询、写主单点(分区缓解热点)；Redis Cluster/NATS Cluster 按分片键(world_id/region)分槽。这些是全局协调点，扩容靠垂直+读副本+分区而非加实例。\n4. 调度器/Coordinator(每世界单例): NATS KV lease 选主，horizontal standby(热备不服务)，主故障秒级接管。day_phase/world tick 这类全局唯一调度走此层，避免多实例双跑导致时序错乱(对应原 natured 单点但增加 HA)。\n可观测流水线自身: OTel Collector / Vector 以 DaemonSet 跑在每个节点(本地采集不占应用网络预算)，集中后端(Prometheus/Loki/Tempo)独立集群或托管，避免观测流量与游戏流量竞争。\n分片边界房间(区域共同体割边)建立跨分片事件代理，AOI 跨分片订阅经边界房间转发，避免每条消息走全局总线(对齐空间分片方案)。

## 技术选型
- 编排: Kubernetes + Helm + Argo CD (GitOps 同步声明式期望状态，对应原 reset 的 reconcile 语义但用于基础设施)
- 金丝雀/回滚: Argo Rollouts (Rollout + AnalysisTemplate 读 Prometheus 自定义指标做自动 promote/rollback)，有状态分片用蓝绿而非金丝雀
- 指标: prometheus_client (Python 内嵌 /metrics) + Prometheus + Grafana; 自定义 game-metrics-adapter 把游戏指标(tick_p99/在线人数/handoff 深度)暴露为 K8s 外部指标
- 弹性: HPA (无状态边缘) + KEDA (事件驱动，读 NATS/Redis lag 与玩家会话数) 替代 max_users:150 硬编码
- 日志: structlog + python-json-logger (JSON UTF-8) -> Vector (GBK 解码+路由，优于 Promtail) -> Loki; 按领域 label
- 追踪: OpenTelemetry Python SDK + auto-instrumentation(asyncio/aiohttp/FastAPI) + 自研 actor 信封 traceparent 注入 -> OTel Collector -> Tempo/Jaeger
- 配置: pydantic-settings (本地校验) + ConfigMap + NATS KV (热更新)，替代 config.xkx 与 adm/etc 平文件
- 审计: PostgreSQL append-only 复用 ES 的 EventStore 表，分区按 (world_id,tenant_id,ts)
- CI/CD: GitHub Actions (ruff/mypy + pytest + DSL schema 校验 + WASM 燃料/内存测试 + 不变量回归) -> distroless Python 镜像 -> GHCR -> Argo CD
- 告警: Alertmanager -> Grafana Oncall/PagerDuty; OTel events 发 fatal 事件
- 调度: APScheduler 或自研 + NATS JetStream 持久化 + NATS KV 选主，替代 call_out 非持久定时器
- 镜像: distroless Python slim + uvloop，减小攻击面与镜像体积
- 采集层: OTel Collector + Vector 以 DaemonSet 部署每节点(本地采集不占应用网络预算)

## 风险
- 有状态游戏分片不能简单 HPA 横向扩：placement/region 亲和/handoff 成本使弹性滞后于负载，短突发会触发反复 handoff 抖动。缓解: 冷却时间 + 仅对滑动窗口持续信号反应 + 预留容量 + 优先扩无状态边缘承接新连接。
- 战斗确定性与金丝雀冲突: 两版本同时运行若战斗公式不同，会导致游戏状态分叉与不公平结果。缓解: 战斗/数值变更走版本门控，同房间参与者强制同版本；有状态分片用蓝绿(暖启动 standby + handoff 切换)而非金丝雀；金丝雀先只切无状态边缘。
- OpenTelemetry 在热路径开销: 每个 actor 消息/call_other 等价都建 span 会显著增加延迟与体积。缓解: 默认边界采样 + 慢 span 尾部采样 + 按 trace_id 调试采样按需开启；生产默认关闭逐消息 span，仅保留跨服务边界 span。
- trace context 传播依赖 actor 信封契约: 若 actor mailbox 实现遗漏 traceparent 注入/透传，会产生断裂 trace 使分布式调试失效。缓解: 在 actor 运行时强制信封携带 trace context 作为不变式，加单测校验；这是硬约束不可妥协。
- GBK->UTF-8 历史日志迁移破坏旧 grep/取证工作流: 团队运维肌肉记忆与历史事件调查依赖 GBK 文本。缓解: Transition 期 Vector 双写(GBK 镜像 + UTF-8 Loki)一个窗口，提供迁移工具与对照表。
- 配置热更新与进行中 tick 竞态: 配置变更若在 tick 中途应用会导致不一致读。缓解: 配置版本号 + 在 tick 边界/epoch apply + apply 后发 ConfigChanged 事件确认。
- K8s Operator 复杂度: 自研 Shard Lifecycle Operator(handoff/迁移自动化)是高难度组件，易引入新 bug。缓解: 先以 Helm + 手动运维 + make 一键脚本起步，仅在 handoff 自动化被证明必要时才引入 Operator；保持 docker-compose 本地开发对等性降低学习曲线。
- 审计 append-only 单 PG 写热点: UGC 规模化后审计写入成为瓶颈。缓解: 按 (world_id,tenant_id,ts) 分区；热审计先入 NATS JetStream 再归档 PG；后期可迁 ClickHouse 做高维查询。
- 丢失'单二进制重启即重置'的运维简单性: 团队需学习 K8s/GitOps，初始运维心智成本上升。缓解: 提供 runbook-as-code、一键 make 部署/回滚目标、本地 docker-compose 开发环境与生产对等。
- 崩溃 RP 通知语义变化: master.crash 原本 shout 全服是沉浸式体验一部分，新模型需决定是否保留。缓解: 保留为可配置开关(运维清晰 vs 沉浸式 RP)，默认 OOC 告警+可选同分片 RP 通知。
- TickOverrun 从静默丢逻辑变为显式告警可能引发告警风暴: 原 'Too long evaluation' 静默，新模型若每个超限都告警会噪声过大。缓解: 告警按持续阈值(连续 N tick 超限)与比例阈值触发，单次超限只记指标不告警。

## 🔍 对抗验证

**裁定**：risky — 方案对旧系统的拆解与映射极为扎实（已逐一核验：call_out 3728 处、config.xkx 各项数值、master.c crash()/standard_trace()、securityd 四套审计、natured 自递归、16 个 preload 守护进程均属实），核心原则正确（有状态分片不能套无状态 HPA、trace context 为硬约束、审计复用 ES EventStore）。但多处"分布式/高并发下成立"的承重声明经不起对抗审查：有状态分片弹性存在分钟级暖启动与反应式 KEDA 的根本时延错配；版本门控做战斗金丝雀与玩家流动性自相矛盾；OTel span 策略组件描述与风险节自相打脸；"秒级回滚无数据丢失"是危险过度声明；Loki 在 UGC 多租户高基数日志场景是错误选型；全库 3728 call_out 统一持久化是性能反模式。按原样实现会在生产引发杀死级联、断裂 trace、审计缺口、回滚不一致等真实事故。需修正这些声明后方可进入实现。

**严重度**：high

### 问题与修复
- **有状态分片弹性存在根本性时延错配，且 5000-8000 并发上限是未经验证的假设。设计声称单分片承载 5000-8000 并发(10万房/20万对象)，但未给出 tick 预算推导。一个含 20 万对象、2s tick 的分片，单 tick 必须处理大量 heart_beat/call_out/AOI；tick_p99 约束下的真实并发上限高度依赖仿真负载而非连接数。**
  - 影响：KEDA 读‘在线会话数+handoff 队列’是滞后信号；当队列已深时玩家已在掉线。新分片暖启动(蓝图加载+AOI 重建+region 注册，对 20 万对象可能是分钟级)期间负载峰值已过。叠加反抖动‘冷却时间+滑动窗口’使响应更慢而非更快。结果：突发负载下要么玩家排队掉线，要么反复 handoff 抖动。
  - 修复：为有状态分片建立‘预测性+预留容量’模型而非纯反应式：基线预留容量常驻(如每 region 1 个 standby 暖分片)、按日历/活动预告预扩、用负载趋势(导数)而非滞后值触发。明确区分‘接新连接’(无状态边缘秒级)与‘承载既有 region 仿真’(分片分钟级，不可反应式)。先用单分片压测得出真实 tick 预算下的并发上限，再定分片数，而非先假设 5000-8000。
- **版本门控做战斗金丝雀与玩家流动性自相矛盾。设计称‘同房间参与者强制同版本’以金丝雀战斗公式变更，但未定义强制同版本的机制。**
  - 影响：武侠 MUD 玩家自由在房间/区域间移动。若版本 A 与 B 战斗公式不同，版本 B 玩家进入版本 A 玩家所在房间时：‘强制同版本’机制未定义——是阻断进入(差 UX)、还是强制迁移该玩家到 B 版本分片(handoff 成本+可能脱离队友)？对社交密集型 MUD，战斗变更的金丝雀实际退化为全量发布。
  - 修复：明确承认：战斗/数值类变更本质是‘灰度即全量’。版本门控机制需具体化——若坚持同房间同版本，需定义‘房间版本归属’与跨版本进入的拒绝/迁移策略并评估 UX。更现实的做法：战斗公式变更走蓝绿全量(暖启动 standby 集群→handoff 整批切换)，放弃在分片粒度金丝雀战斗逻辑。
- **OpenTelemetry span 策略自相矛盾，削弱核心可观测承诺。组件描述称‘actor mailbox 入/出口自动创建 span’使玩家命令成完整 trace、支撑‘战斗回合事件可回放’；但风险节又称‘生产默认关闭逐消息 span’。**
  - 影响：若生产关闭逐消息 span，则‘一次玩家命令(网关->player actor->combat coordinator->skill daemon->event store)成为完整 trace’不成立——服务内多个 actor 跳转坍缩为单个不透明 span，分布式调试与‘战斗回合可回放’承诺落空。若开启，热路径开销与体积显著上升。此外，尾采样需缓存完整 trace 至结束才决策，对跨越多 tick 的长时战斗(可能数分钟)会长期占内存。
  - 修复：统一表述并明确分层采样：跨服务/跨分片边界 span 默认开；同一服务内 actor 跳转用‘共享 trace 的子 span’轻量记录(只记 op+duration，不抓属性)或用 span links 关联而非完整 span。对‘战斗回合可回放’需求，战斗关键步骤单独显式 span(非全量 mailbox 插桩)。长 trace 用基于 trace_id 的概率采样替代尾采样，或限制尾采样缓冲时长并设上限。
- **trace context 传播难度被低估为‘信封契约+单测’。**
  - 影响：asyncio 的 contextvars 是 task 作用域，create_task/gather/ensure_future 默认不自动透传 OTel context；任何裸 create_task、回调注册、call_out 等价物若未显式 attach context 即断链。单测校验‘信封携带’只能覆盖契约入口，覆盖不了所有异步分叉点。
  - 修复：提供 contextvars 传播助手并禁止裸 create_task(经 lint 强制)；为 call_out 等价物/调度回调显式 propagate context 并加契约测试覆盖所有异步边界。定义‘trace 完整率’指标(有父 span 的子 span 占比)持续监控断裂。
- **调度器把全库 3728 处 call_out 统一映射为持久化 ScheduledTask，是性能反模式。**
  - 影响：大量 call_out 是单对象内亚秒~秒级延迟(如‘0.5s 后触发某效果’)，并非需跨重启恢复的调度。全部持久化会产生每秒数千小任务写入，NATS/PG 承压、取消语义从内存操作退化为 DB 写。这与‘高性能 tick 仿真’目标冲突。
  - 修复：分层处理：短延迟(如 <30s)对象内定时器留在 Actor 运行时内存(崩溃丢失可接受，由对象重建恢复)；仅跨重启/全局/长周期定时器走持久化调度器。remove_call_out 对应内存定时器的内存取消(零开销)，仅持久任务才走 DB cancel。明确统计 3728 中各档占比以定分层阈值。
- **‘状态外部化使秒级回滚无数据丢失’是危险过度声明。**
  - 影响：(1) Redis handoff 快照带 TTL 可被淘汰，回滚时可能已无快照；(2) 新版本代码可能已写入旧版本不认识的字段/状态，逻辑回滚后旧代码读到不一致状态——schema 漂移；(3) 进行中战斗在旧版本上，回滚中途会产生不公平结果，与‘公平性审计前提’冲突。
  - 修复：改为‘逻辑回滚≠状态回滚，可能产生数据不一致’。回滚前必须评估新旧版本对世界状态 schema 的兼容性；不兼容时需先做状态迁移或回滚到匹配的状态快照。战斗回滚应限定‘回合边界’+无进行中战斗。Redis 快照设保留下限并监控淘汰。
- **Loki 是 UGC 多租户高维日志检索的错误选型。**
  - 影响：设计要求每条日志带 trace_id/span_id/shard_id/actor_id/world_id 并按域 label。Loki 标签基数是已知陷阱：actor_id/trace_id 作 label 会让 Loki 索引爆炸；作日志字段(非 label)则 Loki 无法高效按其检索。UGC 多租户下 tenant_id/world_id 同样高基数。
  - 修复：重新选型：日志后端用 OpenSearch/Quickwit 或 ClickHouse(支持全文+高维结构化检索)，Loki 仅保留低维 access log。若坚持 Loki，明确 trace_id/actor_id 为日志字段(非 label)，依赖与 Tempo 的 trace_id 关联查询(TraceQL+LogQL join)而非 Loki 自身检索——但这限制了大量‘按 actor 检索日志’场景，需在需求中明确取舍。
- **liveness 探针基于 tick 新鲜度会引发杀死级联。**
  - 影响：若 liveness 关联 tick 新鲜度，则合法繁忙分片(大战斗/GC/慢 tick)会被判死->K8s 杀->重启暖启动(分钟级)->流量与负载转移到其他分片->其他分片也变慢->级联重启。设计自己说要避免‘tick 抖动导致反复重启’，但 liveness 设计恰恰会触发它。
  - 修复：liveness 仅判‘事件循环死锁’(极保守，如 30s+ 无任何事件处理)，绝不基于 tick p99/p50。tick 质量全部归 readiness(摘流量不杀)。设 startup probe 阈值足够长覆盖 20 万对象暖启动，避免 startup 未过即被 liveness 杀。加 PDB 防止多分片同时被杀。
- **配置热更新在分片边界 apply 产生跨分片一致性窗口。**
  - 影响：2s tick 意味配置变更最多 2s 延迟；更关键是各分片在各自 tick 边界 apply，两分片可能短暂持有不同配置版本。对 IP 封禁是安全窗口(玩家可连到未更新分片)；对游戏数值是同版本门控问题在配置层的复现。
  - 修复：全局安全/数值配置走‘两阶段提交’语义：ConfigCenter 发布版本->各分片确认 apply 后才标记版本生效；未确认分片 readiness 降级。IP 封禁这类安全规则下发到网关层(网关统一读最新版本)而非依赖每分片 tick 边界 apply，消除安全窗口。
- **审计‘热路径 NATS 缓冲’与安全审计完整性冲突。**
  - 影响：设计用‘热审计先入 NATS JetStream 再归档 PG’缓解写热点，但安全审计(特权授予/文件写)若异步缓冲，则动作与审计之间崩溃=审计缺失，违背审计目的。安全审计常需‘审计先于提交’语义，此时 NATS 缓冲不解决同步写入延迟。
  - 修复：区分审计等级：特权授予/文件写/CPK 发布等安全关键动作走‘审计先于提交’(同步写 PG 或同步 NATS 确认后才执行动作)；高频低风险审计走异步缓冲。承认‘缓冲优化不适用于安全关键审计’。审计写瓶颈最终迁 ClickHouse(与日志/检索统一)。
- **GBK->UTF-8 过渡假设编码统一，与‘GBK/Big5 混合’自相矛盾。**
  - 影响：设计自己在摘要/映射中注明‘GBK/Big5 混合编码’，但 Vector GBK 解码器不处理 Big5。CRASHES 文件实测为 ISO-8859(即非纯 GBK)。统一按 GBK 转码会产出乱码，破坏旧 grep 取证工作流——正是该过渡期要保护的东西。
  - 修复：Vector 用确定性字符集探测(GBK/Big5/UTF-8 多次尝试+置信度)而非固定 GBK 解码。先全库扫描统计实际编码分布。提供混合编码对照表。考虑过渡期对历史日志做一次性批量转码归档(而非长期双写)。
- **可观测/编排技术栈规模与团队现状严重脱节，无托管决策。**
  - 影响：12+ 基础设施系统对一支从零起步(无 Docker/Python/K8s 痕迹)的团队是巨大运维与认知负担。设计提‘托管’为可选项但未承诺。UGC 平台收入未明时自托管全栈是财务与运维风险。
  - 修复：定义‘MVP 可观测栈’(Prometheus+单一日志后端+Tempo，全部托管优先)与‘完整栈’两阶段。优先用云托管(Grafana Cloud/等效)降低自托管负担，明确成本上限。docker-compose 本地对等必须覆盖可观测端到端路径。加‘自托管 vs 托管’决策矩阵与成本模型。
- **distroless Python 镜像与崩溃事后可调试性冲突。**
  - 影响：无 shell 的 pod 在崩溃事后调查时缺少现场工具，需依赖 ephemeral debug container，对刚从‘手动重启 driver’过渡的团队是陡坡。与设计强调的崩溃/事件响应可观测目标存在张力。
  - 修复：对 distroless 镜像配 ‘debug variant’(含 busybox shell)供事故时 kubectl debug 切换；或用 wolfi-base 保留最小 shell。明确崩溃现场寄存器/栈通过 OTel fatal 事件+coredump 落对象存储而非 pod 内现场调试的流程。
- **day_phase 世界单例 coordinator 混淆了仿真 tick 与表现层时钟。**
  - 影响：day_phase 原始语义是全局世界时钟广播消息给所有用户。分片后各分片有各自用户。若 day_phase 走‘每世界单例 coordinator’广播，是对纯表现层特性的过重协调；若每分片自跑则‘每世界单例’表述错误。
  - 修复：分两类：‘仿真 tick’是每分片本地确定性步进(无需全局单例)；‘世界级广播事件’(昼夜/季节/公告)由轻量广播服务扇出，各分片订阅本地分发。day_phase 不必占 HA 单例槽位。

### 改进建议
- 先做单分片真实压测得到 tick 预算下的并发上限与暖启动耗时，再定分片数与弹性阈值；将分片弹性改为预测性(预留 standby 容量+趋势触发)而非纯反应式 KEDA。
- 战斗/数值变更放弃分片粒度金丝雀，明确走蓝绿全量；版本门控机制(房间版本归属、跨版本进入策略)在实现前必须先定具体语义并评估 UX。
- 统一 OTel span 策略表述并分层采样：边界 span 默认开，服务内 actor 跳转用轻量子 span 或 span links，战斗关键步骤显式 span；长 trace 改概率采样替代尾采样。
- trace 传播提供 contextvars 传播助手 + lint 禁裸 create_task + 覆盖所有异步边界的契约测试，并监控‘trace 完整率’指标。
- 调度器分层：短延迟对象内定时器留内存(零开销取消)，仅跨重启/全局/长周期走持久化；先统计 3728 call_out 各档占比。
- 修正‘秒级回滚无数据丢失’为‘逻辑回滚≠状态回滚，需评估 schema 兼容性’，战斗回滚限定回合边界+无进行中战斗，Redis 快照设保留下限。
- 日志后端改 OpenSearch/Quickwit/ClickHouse 支持高维检索；Loki 仅留低维 access log。
- liveness 仅判事件循环死锁(极保守)，tick 质量全归 readiness(摘流量不杀)；startup 阈值覆盖 20 万对象暖启动；加 PDB。
- 全局安全/数值配置走两阶段 apply(各分片确认后生效)；IP 封禁下发网关层而非依赖分片 tick 边界。
- 安全关键审计走‘审计先于提交’同步路径，异步缓冲仅用于高频低风险审计。
- Vector 用确定性多编码探测而非固定 GBK；先全库扫描编码分布。
- 定义 MVP 可观测栈与完整栈两阶段，优先云托管，docker-compose 本地对等覆盖端到端观测路径，加自托管 vs 托管决策矩阵。
- distroless 镜像配 debug variant 供事故 kubectl debug，或用 wolfi-base 保留最小 shell；崩溃现场走 OTel fatal 事件+coredump 落对象存储。
