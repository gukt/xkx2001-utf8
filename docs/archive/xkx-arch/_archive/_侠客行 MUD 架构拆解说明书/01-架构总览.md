# 侠客行 MUD 架构总览

本文档详细描述了侠客行（XKX）MUD 的整体架构设计，包括驱动层、Mudlib 层、对象模型、继承体系以及核心系统的组织方式。

---

## 1. 驱动层与 Mudlib 边界

### 1.1 驱动程序（Driver）

侠客行 MUD 基于 **MudOS** 驱动程序运行，这是一个用 C 语言编写的 LPC 虚拟机。驱动程序的主要职责包括：

- **LPC 代码执行引擎**：解释执行 LPC 语言编写的游戏逻辑
- **网络连接管理**：处理玩家的 Telnet 连接、输入输出
- **对象生命周期管理**：对象的创建、销毁、交换（Swap）
- **Eval Cost 机制**：限制单个函数的执行时间，防止死循环
- **文件系统访问**：提供安全的文件读写接口
- **内存管理**：管理对象、字符串、映射等数据结构的内存分配

驱动程序二进制位于根目录下的 `driver` 文件，配置文件为 `config.xkx`。

#### 配置文件关键参数（config.xkx）

```c
// 游戏名称与端口
name : 侠客行
port number : 8888

// 核心对象路径
master file : /adm/single/master
simulated efun file : /adm/single/simul_efun
global include file : <globals.h>

// 内存管理参数
time to clean up : 180        // 对象清理时间（秒）
time to swap : 120            // 对象交换时间（秒）
time to reset : 1800          // 房间重置时间（秒）
maximum evaluation cost : 600000000  // 最大执行开销

// 目录配置
mudlib directory : .
include directories : /include
log directory : /log
```

### 1.2 Master 对象

Master 对象（`/adm/single/master.c`）是驱动程序与 Mudlib 之间的桥梁，负责处理驱动的回调请求。其主要职责包括：

#### 1.2.1 连接处理

```c
object connect()
{
    // 新玩家连接时创建 login 对象
    object login_ob;
    mixed err;
    err = catch(login_ob = new(LOGIN_OB));
    if (err) {
        write("现在有人正在修改使用者连线部份的程式，请待会再来。\n");
        destruct(this_object());
    }
    return login_ob;
}
```

当玩家通过 Telnet 连接到游戏时，驱动调用 `connect()` 函数创建一个登录对象来处理后续的认证流程。

#### 1.2.2 错误处理

```c
string error_handler(mapping error, int caught)
{
    // 处理运行时错误
    if (this_player(1)) {
        if(wizardp(this_player(1)))
            tell_object(this_player(1), standard_trace(error, caught));
        else
            tell_object(this_player(1), "你发现事情不大对了，可是又说不上来。\n");
    }
    return standard_trace(error, caught);
}
```

当 LPC 代码运行出错时，`error_handler()` 被调用来格式化错误信息并决定显示给玩家的内容。

#### 1.2.3 安全验证

Master 对象实现了一系列安全验证函数：

- `valid_seteuid()`：验证对象是否有权限设置有效用户 ID
- `valid_override()`：控制哪些 simul_efun 可以被覆盖
- `valid_write()`：验证文件写权限
- `valid_read()`：验证文件读权限
- `valid_shadow()`：控制对象是否可以被 shadow
- `valid_hide()`：控制对象是否可以隐藏

#### 1.2.4 预加载机制

```c
string *epilog(int load_empty)
{
    // 返回需要在启动时预加载的对象列表
    string *items;
    items = update_file(CONFIG_DIR + "preload");
    return items;
}

void preload(string file)
{
    // 预加载单个对象
    int t1;
    string err;
    if (file_size(file + ".c") == -1) return;
    t1 = time();
    write("Preloading : " + file );
    err = catch(call_other(file, "??"));
    if (err)
        write(" -> Error " + err + " when loading " + file + "\n");
    else
        write(".... Done.\n");
}
```

驱动启动完成后，调用 `epilog()` 获取需要预加载的守护进程列表，然后逐个调用 `preload()` 加载它们。

### 1.3 Simul Efun（模拟外部函数）

Simul Efun 对象（`/adm/single/simul_efun.c`）扩展了驱动提供的原生 efun（外部函数），为整个 Mudlib 提供全局可用的工具函数。

#### 1.3.2 核心功能示例

**对象操作（object.c）**：

```c
varargs int getoid(object ob)
{
    // 获取对象的唯一 ID
    int id;
    if (!ob) ob = previous_object();
    sscanf(file_name(ob), "%*s#%d", id);
    return id;
}

void destruct(object ob)
{
    // 封装驱动的 destruct，调用对象的 remove() 先清理
    if (ob) {
        if( previous_object() ) ob->remove( geteuid(previous_object()) );
        else ob->remove(0);
    }
    efun::destruct(ob);
}
```

**文件所有者（object.c）**：

```c
string file_owner(string file)
{
    // 根据文件路径判断所有者
    string name, rest, dir;
    if (file[0] != '/') file = "/" + file;
    if (sscanf(file, "/u/%s/%s/%s", dir, name, rest) == 3) {
        return name;
    }
    return 0;
}

string domain_file(string file)
{
    // 判断文件所属的域名（区域）
    string domain;
    if( sscanf(file, "/d/%s/%*s", domain) )
        return domain;
    return ROOT_UID;
}
```

## 2. 核心对象继承链

侠客行 MUD 采用了精心设计的多重继承与 Mixin 模式，通过继承基类和组合 Feature 来构建复杂的对象模型。

### 2.1 Mixin 模式：Feature 机制

Feature 是侠客行架构的核心设计思想，它将可复用的功能模块化，每个 Feature 实现一个独立的功能点，对象通过继承多个 Feature 来组合所需功能。

#### 2.1.1 F_DBASE：数据库特性

`/feature/dbase.c` 提供了通用的键值对数据存储功能：

```c
mapping dbase;          // 持久化数据存储
static mapping tmp_dbase;  // 临时数据存储（不保存）

mixed set(string prop, mixed data) { ... }
varargs mixed query(string prop, int raw) { ... }
int delete(string prop) { ... }
mixed add(string prop, mixed data) { ... }

mixed set_temp(string prop, mixed data) { ... }
varargs mixed query_temp(string prop, int raw) { ... }
int delete_temp(string prop) { ... }
mixed add_temp(string prop, mixed data) { ... }
```

支持路径式访问，如 `set("family/master_name", "张三丰")`。

#### 2.1.2 F_MOVE：移动特性

`/feature/move.c` 处理对象的移动和负重：

```c
static int weight = 0;          // 对象自身重量
static int encumb = 0;          // 当前负重
static int max_encumb = 0;      // 最大负重

varargs int move(mixed dest, int silently)
{
    // 移动对象到目标位置
    object ob, env;
    // ... 检查目标是否可容纳 ...
    // ... 更新负重 ...
    move_object(ob);  // 调用驱动的移动
    // ... 玩家自动 look ...
}
```

#### 2.1.3 F_NAME：名称特性

`/feature/name.c` 处理对象的命名和标识：

```c
static string *my_id;

void set_name(string name, string *id)
{
    set("name", name);
    set("id", id[0]);

int id(string str)
{
    // 判断字符串是否能标识此对象

    return 0;
}

varargs string name(int raw) { ... }
varargs string short(int raw) { ... }

```

## 3. 对象生命周期

### 3.1 玩家从连接到进入游戏的完整流程

#### 3.1.1 连接建立阶段

```
玩家 Telnet 连接
    |
    v
Driver 调用 master->connect()
    |
    v
创建 LOGIN_OB (/clone/user/login.c)
    |
    v
login_ob->logon() 被调用
    |
    v
调用 LOGIN_D->logon(login_ob)
```

登录对象代码 (`/clone/user/login.c`)：

```c
inherit F_DBASE;
inherit F_SAVE;

void logon()
{
    call_out("time_out", LOGIN_TIMEOUT);
    LOGIN_D->logon(this_object());
}

string query_save_file()
{
    // 登录对象的存档路径
    string id;
    id = query("id", 1);
    if( !stringp(id) ) return 0;
    return sprintf(DATA_DIR "login/%c/%s", id[0], id);
}
```

#### 3.1.2 登录认证阶段 (LOGIN_D)

`/adm/daemons/logind.c` 处理完整的登录流程：

```c
void logon(object ob)
{
    // 1. 检查 IP 是否被封禁
    if (BAN_D->is_banned(query_ip_name(ob)) == 1) {
        destruct(ob);
        return;
    }
    
    // 2. 显示欢迎信息
    cat(WELCOME);
    
    // 3. 询问编码（GB/BIG5）
    write_ob(ob, "\nDo you want to use BIG5 code?(y/n)\n");
    input_to("confirm_big5", ob);
}

void confirm_big5(string arg, object ob)
{
    // 设置编码
    if( arg == "y" || arg == "yes")
        ob->set("language","BIG5");
    else
        ob->set("language","GB");
        
    // 询问英文 ID
    write_ob(ob, "您的英文名字（新玩家可选一喜欢的名字）：");
    input_to("get_id", ob);
}

void get_id(string arg, object ob)
{
    // 验证 ID 合法性
    if( !check_legal_id(arg, ob)) {
        input_to("get_id", ob);
        return;
    }
    
    ob->set("id", arg);
    
    // 检查是否已存在存档
    if( file_size(ob->query_save_file() + __SAVE_EXTENSION__) >= 0 ) {
        // 老用户：验证密码
        ob->restore();
        write_ob(ob, "请输入密码：");
        input_to("get_passwd", 1, ob);
    } else {
        // 新用户：确认创建
        write_ob(ob, "使用 " + arg + " 将会创造一个新人物，确定吗(y/n)？");
        input_to("confirm_id", ob);
    }
}

void get_passwd(string pass, object ob)
{
    string my_pass, file, *tmp;
    object user;
    
    // 验证密码
    my_pass = ob->query("password");
    if( crypt(pass, my_pass) != my_pass ) {
        write_ob(ob, "密码错误！\n");
        destruct(ob);
        return;
    }
    
    // 创建用户身体对象
    if( objectp(user = make_body(ob)) ) {
        if( user->restore() ) {
            user->set("language", ob->query("language"));
            enter_world(ob, user);
            return;
        }
    }
}
```

#### 3.1.3 新玩家创建流程

```c
void confirm_id(string yn, object ob)

    input_to("get_name", ob);
}

void get_name(string arg, object ob)
{
    // 设置中文名
    if( !(arg = check_legal_name(arg, ob)) ) {
        input_to("get_name", ob);
        return;
    }
    ob->set("name", arg);
    
    // 设置密码
    write_ob(ob, "请设定您的密码：");
    input_to("new_password", 1, ob);
}

void new_password(string pass, object ob)
{
    if( strlen(pass) < 5 ) {
        write_ob(ob, "密码至少要 5 个字符！\n");
        input_to("new_password", 1, ob);
        return;
    }
    ob->set("password", crypt(pass, 0));
    
    // 确认密码
    write_ob(ob, "请再输入一次您的密码：");
    input_to("confirm_password", 1, ob);
}

void confirm_password(string pass, object ob, mapping my)
{
    if( crypt(pass, ob->query("password")) != ob->query("password") ) {
        write_ob(ob, "密码不一致！\n");
        input_to("new_password", 1, ob);
        return;
    }
    
    // 随机生成天赋
    mapping gift = allocate_mapping(5);
    random_gift(gift);
```

#### 3.1.4 进入游戏阶段

```c
varargs void enter_world(object ob, object user, int silent)
{
    // 1. 关联连接对象和身体对象
    user->set_temp("link_ob", ob);
    ob->set_temp("body_ob", user);
    
    // 2. 将交互转移到身体对象
    exec(user, ob);
    
    // 3. 设置角色
    user->setup();
    
    // 4. 保存登录对象
    ob->save();
    
    // 5. 登录检查
    UPDATE_D->login_check(user);
    
    // 6. 移动到起始房间
    string startroom = user->query("startroom");
    if( !startroom ) startroom = "/d/xiakedao/shatan";
    user->move(startroom);
    
    // 7. 广播上线消息
    if( !silent ) {
        tell_room(startroom, user->query("name") + "连线进入这个世界。\n", ({user}));
        CHANNEL_D->do_channel(this_object(), "sys",
            sprintf("%s(%s)由%s连线进入。",
                user->name(), user->query("id"), query_ip_name(user)));
    }
}

object make_body(object ob)
{
    // 创建用户身体对象
    object user;
    user = new(ob->query("body"));  // 默认为 USER_OB
    
    // 设置 UID
    seteuid(ob->query("id"));
    export_uid(user);
    export_uid(ob);
    seteuid(getuid());
    
    user->set("id", ob->query("id"));
    user->set("language", ob->query("language"));
    user->set_name(ob->query("name"), ({ ob->query("id") }));
    
    return user;
}
```

#### 3.2.2 对象交换（Swap）

当对象长时间不活动（超过 `time to swap` 配置，默认 120 秒），驱动会将其数据交换到磁盘以节省内存：

```
对象活动
    |
    v
空闲计时开始
    |
    v
达到 swap 时间
    |
    v
驱动交换对象到磁盘
    |
    v
对象被访问时自动恢复
```

#### 3.2.3 对象清理（Clean Up）

空闲时间更长（超过 `time to clean up`，默认 180 秒）时，驱动调用对象的 `clean_up()` 函数：

```c
// /feature/clean_up.c
int clean_up()
{
    // 默认实现
    if( !clonep(this_object()) ) return 0;  // 蓝图对象不清理
    if( query("no_clean_up") ) return 1;    // 标记不清理，重新计时
    
    // 如果有环境或被其他对象引用，保留
    if( environment() || !query_temp("destruct_me") )
        return 1;
        
    // 否则销毁
    destruct(this_object());
    return 0;
}
```

#### 3.2.4 对象销毁

```c
// 调用 destruct()
destruct(ob);

// 1. simul_efun 的 destruct() 被调用
// 2. 对象的 remove() 函数被调用
// 3. 环境的 add_encumbrance() 被调用以更新负重
// 4. 驱动真正销毁对象
```

```c
varargs int save(object me)
{

varargs int restore(object me)

    string file;

}
```

## 7. 设计模式与思想

### 7.1 Mixin 模式

侠客行大量使用 Mixin 模式，通过继承多个 Feature 来组合功能，而非深层单一继承：

```c
// 传统继承：
// Char <- Living <- Player <- ... (问题：功能耦合)

// Mixin 模式：
inherit F_DBASE;     // 数据存储
inherit F_MOVE;      // 移动功能
inherit F_NAME;      // 命名功能
inherit F_ATTACK;    // 战斗功能
// ... 按需组合
```

优点：
- 功能模块化，易于理解
- 灵活组合，避免继承爆炸
- 代码复用性高

### 7.2 约定优于配置

大量使用约定来简化代码：

- 目录结构约定：/d/ 为区域，/cmds/ 为命令
- 函数名约定：create() 初始化，setup() 设置，reset() 重置
- 变量名约定：me 指当前对象，ob 指其他对象

### 7.3 数据驱动设计

- 房间描述、出口、NPC 都通过数据配置
- 武功通过数据定义等级、伤害、消耗等
- 角色属性完全存储在 dbase 映射中

### 7.4 守护进程模式

全局服务由单例守护进程提供，避免重复代码：

```c
// 不这样做：
// 每个对象自己实现频道广播

// 而是这样：
CHANNEL_D->do_channel(ob, "chat", "大家好！");
```

## 8. 开发指南要点

### 8.1 添加新房间

1. 在 `/d/<区域>/` 下创建 `.c` 文件
2. 继承房间基类：`inherit ROOM;`
3. 在 `create()` 中设置描述、出口、物品
4. 调用 `setup()` 完成设置

### 8.2 添加新 NPC

1. 创建继承 NPC 基类的文件
2. 设置名称、描述、属性
3. 添加对话、战斗行为

### 8.3 添加新武功

1. 在 `/kungfu/skill/<类别>/` 创建技能文件
2. 实现 `valid_learn()`、`do_effect()` 等函数
3. 在门派中添加可学习技能

### 8.4 添加新命令

1. 在 `/cmds/<类别>/` 创建命令文件
2. 实现 `main(object me, string arg)` 函数
3. 返回 1 表示成功，0 表示失败

## 9. 总结

侠客行 MUD 的架构设计体现了以下核心思想：

1. **清晰的分层**：驱动层、Mudlib 层、对象层职责分明
2. **灵活的组合**：通过 Mixin 模式实现功能的灵活组合
3. **集中的服务**：守护进程提供全局服务
4. **数据驱动**：游戏内容通过数据配置而非硬编码
5. **规范的约定**：统一的命名和结构约定降低学习成本

这套架构在 90 年代设计，至今仍展示出良好的可扩展性和可维护性，是学习游戏服务器架构设计的优秀范例。
