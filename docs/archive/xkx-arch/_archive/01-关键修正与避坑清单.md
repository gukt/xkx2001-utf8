# 关键修正与避坑清单

> 本文档汇总对抗验证阶段发现的 **40 个承重级假设错误**。每个方案设计 Agent 产出方案后，由独立的评审 Agent 做"默认怀疑"的对抗审查（逐项核验代码声明、评估分布式/高并发成立性、迁移可行性、反模式）。
>
> **这些修正是整个研究最高价值的产出**--若按天真的初版实施，会在生产中表现为杀死级联、断裂 trace、数据不一致、迁移超期、UGC 安全边界崩塌等真实事故。本清单按子系统组织，每条含「问题 / 影响 / 修正」。

---

## 一、规模与容量假设

### §1 现有并发上限是 50 不是 150/230
- **问题**：多处方案以 `config.xkx` 的 `maximum users:150` 或 LPC 注释的 230 为旧系统上限做容量基线。
- **影响**：容量基线误判，扩缩容阈值与迁移优先级设定偏离。实际 `include/login.h` 中 `MAX_USERS=50`（已验证），moneyd 注释亦假设"about 200 ppl online"。
- **修正**：迁移期以 **50** 为容量基线对照。目标 1 万在线是 200x 跨度。**阶段一先单进程 asyncio + 单 PG + 单 Redis 跑通核心改造，不要为 50 用户预先分布式化**。分片上线条件设为"单节点实测承载达 80%"而非"设计图纸支持 5000"。Louvain 分片降级为规模化阶段的执行细节。

### §40 迁移面被低估一个数量级
- **问题**：方案标称"3450 处 environment() 直接指针引用"。
- **影响**：排期与工作量被低估一个数量级，承诺交付期几乎必然超期。
- **修正**：实测 `environment()`=664、`this_object()`=4214、**箭头方法调用=68771**（仅 /d 就 36333）、all/deep_inventory=520、query/set dbase 访问=84136。**以箭头调用面（而非 environment）作为主线工作量指标**，先迁移 go/move/combat/channel 四条高频跨对象调用图。

## 二、ECS 实体系统

### §2 砍掉 SparseSet + Archetype 双存储
- **问题**：方案同时声称用 SparseSet 做 O(1) 查询 + Archetype 做同组件集聚簇遍历。两者是互斥存储策略，混用意味着组件 add/remove 时在两套结构间同步迁移。
- **影响**：实现复杂度高，对 5000-8000 实体规模 Archetype 聚簇的 cache locality 收益微乎其微（L1 cache 对万级对象无感），徒增框架复杂度。
- **修正**：只用 SparseSet 单一存储。enTT 也是 SparseSet 为主。若需批量战斗遍历优化，在 CombatSystem 内对 query 结果做一次排序聚簇缓存即可。

### §3 query_entire_dbase() 活引用直改破坏组件边界
- **问题**：combatd.c（行 356-357, 441-455, 696-712）通过 `my=query_entire_dbase()` 拿到整个 dbase 映射的活引用后直接交叉读写 `my["combat_exp"]`、`my["potential"]`、`my["jingli"]` 等 11+ 字段。这些字段在方案 15 个组件中无归属。
- **影响**：组件拆分后 CombatSystem 需同时 get_component 多个组件且对两实体都写。`combat_exp`（被查 1275 次，全库第 3 高频键）和 `potential`/`max_potential` 会落入 DynamicProps，彻底抵消类型化收益。
- **修正**：迁移 CombatSystem 前必须先用静态分析枚举全部 `query_entire_dbase` 调用点的读写键集，建字段->组件完整映射表。补 `Progression` 组件（combat_exp/potential/max_potential/death_times/PKS/MKS/behavior_exp）。对 `my['int']`/`my['dex']` 这类绕过 query_int() 读裸值的点，区分读基础值(Attributes)还是派生值(DerivedAttributes)。

### §4 Effect 必须是一等公民组件
- **问题**：perform/exert 用 `start_call_out(闭包)` 实现持续效果（196 个技能文件用 call_out），闭包捕获 `this_object()` 引用无法序列化/跨进程/断线重放。方案在 risks 中正确识别但未在组件族给出 Effect 设计。
- **影响**：这是分布式化最关键改造，断线重放和跨分片迁移的前提，不能延后。
- **修正**：引入 `EffectList(skill_id, effect_type, remaining_ticks, payload, source_id)` 可序列化可中断组件 + `EffectSystem` tick 衰减 + 回调 `SkillStrategy.on_tick_effect()`。`start_call_out`/`eval_function` 闭包逐个翻译为 Effect 实例，建闭包->Effect 映射表与等价回归测试（特别是 huashan/feng.c 的 auto_perform 链）。

### §S 下调 Python 单分片承载目标
- **问题**：方案以"单分片 5000-8000 并发"作为 5-8 分片估算基准，未做基准测试。
- **影响**：Python 单线程 asyncio + GIL 下，CombatSystem 每次 do_attack 产生 20-30 次组件查询（运行时 dict 查找），战斗是 per-fight 串行不可向量化。真实单分片容量可能是 1000-2000。
- **修正**：先按 **2000-3000 并发**规划，1 万在线需 4-5 分片。CombatSystem 优化重点放组件引用缓存（回合内一次取齐、本地变量访问）和 skill_power 预计算缓存。架构定型前做单 Player Actor 事件环（Pydantic 边界校验+delta 编码+msgpack+AOI 扇出）p99 延迟与吞吐实测。

## 三、Actor 与分布式

### §5 放弃自研虚拟 Actor 运行时，Ray 起步
- **问题**：自研 asyncio ActorRuntime（含 placement/激活钝化/迁移 handoff/消息序/监督树）是本方案最大工程风险，却被作为"借鉴 Orleans 语义而非从零发明"轻描淡写。同时排斥 Akka/Orleans（非 Python）与 Ray（仅可选）。
- **影响**：自研运行时是整个方案的成败支点。若按现方案自研，2-3 年内主循环稳定性、消息序、激活/钝化正确性都难收敛。等于在 8400 文件迁移之外再背自研分布式框架债务。
- **修正**：**明确放弃"从零实现虚拟 actor 运行时"**。阶段一用 Ray 的 actor 模型做最小可用验证（Python 原生、支持 placement/调度/容错），证明 LPC↔Actor 同构映射在真实负载下成立后再决定是否自研薄壳。不要在原型前承诺自研运行时。无状态 HTTP（登录/查询/排行榜）、批量计算走传统微服务/批处理，仅在"有状态、可寻址、需激活与迁移"的实体上用 actor。

### §6 热路径去 Redis 化
- **问题**：会话路由、实体路由、AOI 订阅、门状态、迁移快照、频道 pub/sub 全部走 Redis。任何一次 Redis 抖动都冻结全服跨区操作。
- **影响**：Redis 即游戏的单点命脉，既限吞吐又是爆炸半径上限。每条跨分片消息过 Redis 是吞吐上限。
- **修正**：区分热路径与控制路径：热路径（会话/实体路由、AOI 订阅）用**一致性哈希环**（`shard=hash(entity_id)`）无外部依赖解析，Redis 仅做控制面（placement 变更通告、handoff 暂存）。门/迁移快照等强一致需求走 PostgreSQL 事务。明确 Redis 降级时游戏降级为"仅本分片可玩"而非全停。

### §7 区域监督 one-for-one 非 all-for-one
- **问题**：RegionSupervisor 用 all-for-one（区域崩溃重启整个区域子树），理由"类似 LPC 重载区域"不成立--LPC 并无"区域崩溃重启"语义。
- **影响**：all-for-one 意味着一个房间 actor 异常连带重置同区域数千房间/NPC/玩家，丢失内存状态，分布式下比单进程更脆弱。
- **修正**：区域层默认 **one-for-one + 持久化重建**。仅在能证明子实体间存在不可恢复的共享可变状态时才对该子树用 all-for-one。区域崩溃触发从 PostgreSQL 快照+事件日志重建，而非丢弃在线状态。先确认多数区域状态实为 per-room/per-npc，很可能根本不需要 all-for-one。

### §V 缺少行为等价性验证（差分测试）
- **问题**：8400 LPC 文件迁移到 Python，如何证明新版与旧版行为一致？方案全文未提差分测试。
- **影响**：无差分测试，行为漂移要等玩家投诉才暴露，届时已积累大量隐性 bug 且难定位根因。
- **修正**：阶段一同步立项差分测试基建：录制 LPC 侧命令流（玩家输入序列+tick 边界）为 golden trace，Python actor 版逐条重放并 diff 输出帧与状态快照。先覆盖 go/move/combat/channel 四主线。**无差分测试通过的子系统不进阶段二**。

### §G 补全 32 守护进程全量分类表
- **问题**：GlobalStateService 三分类只点名了 weapond/emoted/chinesed（无状态）、natured/commandd（只读）、channeld/securityd/marryd（可变），遗漏 adsd/band/aliasd/fingerd/inquiryd/editord/ftpd/languaged/moneyd/pigd/profiled/rankd 等。
- **影响**：遗漏守护进程会导致迁移后部分子系统静默失效。
- **修正**：补一张"32 守护进程全量分类表"，逐个标注分类与迁移方式。

## 四、战斗与玩法引擎

### §8 heart_beat 是 tick 驱动器，不可被事件驱动替代
- **问题**：DSL/数据层方案称"替代 heart_beat 全局心跳改为按需事件驱动"。但 `feature/attack.c` 注释明确"This is called in heart_beat() to perform attack action"--战斗回合由 heart_beat 驱动；busy 状态由 heart_beat 驱动；`condition.c` 注释"update_condition is called by heart_beat"。
- **影响**：若按"事件驱动替代全局心跳"实施，战斗回合、NPC AI、持续伤害/治疗等核心模拟失去驱动机制，需事后补回周期调度，架构返工。
- **修正**：保留周期性 tick 作为一等公民。ConditionScheduler 同时支持事件触发与周期 tick 两种模式。heart_beat 重新定位为"distributed tick scheduler with per-scene/per-actor frequency control"。auto_perform AI 循环（128-189 处）建模为 ECS ContinuousBehaviorSystem。

### §9 战斗确定性需 CombatContext 快照
- **问题**：方案称"每场战斗独立 seeded RNG 保证回合可回放"。但 do_attack 持续读写参与者实时状态，仅记 seed 不在战斗开始时快照全体参与者状态，回放会分叉。
- **影响**：反作弊审计与 bug 复现失败，且这类失败极难发现。
- **修正**：确定性边界明确为"战斗开始时对全体参与者+可观察环境做状态快照 = CombatContext"，seeded RNG 只是其一部分。回放从 CombatContext 起点重放，禁止战斗中途引入未在快照中的输入。先抽样 2-3 场真实战斗导出 do_attack 读取的 dbase 字段集，据此定义 CombatContext 最小边界。

### §10 确定性回放分层而非全量承诺
- **问题**：do_attack 内 29 处 random() 调用含可变迭代次数 while 循环与递归 riposte/double_attack，纯函数管线重构会改变 RNG 消耗序列，"纯函数管线"与"确定性回放"两个目标直接冲突。
- **影响**：反作弊审计基线对迁移内容失效。
- **修正**：分层：(1) 新内容/UGC 强制走 CombatRNG 契约保证可回放；(2) 遗留内容采用"行为等价而非位等价"迁移标准--保留概率分布与伤害量级，放弃逐 random 对齐；(3) 保留原 random() 全局序列的"黄金战报"做统计回归（伤害均值/方差/命中率）而非逐回合比对。

### §11 战斗热路径绕过事件总线
- **问题**：do_attack 一次攻击发 5+ 事件，5000 实体×1攻击/2s≈12500 事件/s/分片全经 asyncio.Queue 分发，每个事件 dataclass 分配；同时 Pydantic 每次 mutate 校验不变量。
- **影响**：原 receive_damage 是 O(1) 进程内直接 mutate，新模型改为发布+分配+订阅，Python 单线程 asyncio 下显著性能回退。宣称的单分片 5000-8000 并发战斗实体有落空风险。
- **修正**：(1) resolve_attack 同步返回 CombatRoundResult（含 damage_delta/messages/effects），CombatSystem 同步应用并就地通知同房间订阅者，仅在 tick 边界发一次聚合事件到总线供投影层；(2) Vitals 运行时用裸 dataclass+断言，Pydantic 仅在 snapshot 落盘/UGC 上传/跨分片反序列化时校验；(3) 给单分片定实测基线（单 do_attack μs 目标）再定容量上限。

### §12 ConditionHandler.on_tick 不能用互斥枚举
- **问题**：方案设计 on_tick 返回互斥枚举 CONTINUE/EXPIRE/NO_HEAL。原 `condition.h` 中 `CND_CONTINUE=1`、`CND_NO_HEAL_UP=2` 是可组合位标志，跨 condition 做 OR 聚合（中毒时抑制恢复）。
- **影响**：中毒等状态在原系统通过位聚合抑制 heal_up，新模型无法表达"多个 condition 中任意一个抑制恢复"，导致中毒/重伤时仍恢复血量，破坏战斗平衡。
- **修正**：on_tick 返回组合结构 `ConditionTickResult(continue, suppress_heal, expire)`，显式定义聚合规则：`aggregate.suppress_heal = any(c.suppress_heal)`；`aggregate.continue = any(c.continue)`。72 个 condition 逐个迁移前必须先落定此契约。

### §C CombatCoordinator 热门房间串行瓶颈
- **问题**：同房间战斗参与者同节点亲和、单 Coordinator 串行裁决。对 50 人 PK 混战或 20 人 boss 战，该房间所有战斗回合排队过单 Actor。
- **影响**：原系统每实体 heart_beat 天然跨实体并行；新模型对单房间强制串行，热门房间反而性能回退。
- **修正**：当房间参与者超阈值（如 >30）时，将房间战斗拆为多个子 Coordinator（按对手对分组），或对超大规模群战启用"简化回合模式"（批量结算而非逐对裁决）。

### §E EffectSystem 需声明 stacking_policy
- **问题**：原 perform 用 set_temp 旗标控制效果互斥（xianglong.c 防重复施展）。Effect 模型按 effect_id 管理，缺少"同类型效果唯一"约束。
- **影响**：降龙一击类效果可能重复施加（parry buff 翻倍、伤害结算多次）。
- **修正**：EffectDefinition 声明 `stacking_policy(unique|refresh|stack|independent)`，EffectRegistry.apply 时按策略处理。把 set_temp 旗标映射为 unique 策略由 Registry 统一管理，调用方不再手工维护互斥。

### §T Effect 需时间模式字段与崩溃恢复策略
- **问题**：Effect.duration_ticks 以 tick 为单位，服务器宕机期间不推进 tick，恢复后按 tick 数继续；原 call_out 基于墙钟，宕机时长会扣减剩余 delay。跨战斗延续效果用已结束战斗的种子在审计上产生歧义。
- **修正**：Effect.duration 增加时间模式字段 `mode: tick|wallclock`，wallclock 按真实经过时间衰减。崩溃恢复：按 downtime 上限 clamp 跳过的 tick 数，超限按比例衰减剩余 duration。跨战斗延续时切换为房间级/全局 RNG。

### §F 门派/伤害公式面向 UGC 须进沙箱
- **问题**：UGC 技能脚本用 WASM 沙箱，但 family_definition.yaml 的加成公式用"纯 Python lambda 或受限表达式 AST"。
- **影响**：UGC 作者可通过公式注入恶意或病态表达式（level**100 数值溢出、无限循环拖垮 tick），绕过 WASM 沙箱。
- **修正**：统一安全边界：family/skill 公式若面向 UGC 全部进 WASM 或受限 AST（白名单运算符、无循环、深度上限），Python lambda 仅限可信内部配置。

## 五、数据与状态层

### §13 不要全量事件溯源，用权威状态库 + 选择性审计
- **问题**：把 ES 当作所有状态变更的默认持久化模式。代码库 52985 处同步调用、654 处 environment() 立即检查，典型 `ob->receive_damage(50); if(ob->query("qi")<0) ob->die();`。CQRS 最终一致意味着写后读走"强一致读 EventStore"，每次读都打 PG。
- **影响**：对 50 用户系统引入 ES+CQRS 复杂度税远超收益，是数据库瓶颈不是高并发。
- **修正**：降级为"权威状态库 + 选择性审计日志"。状态变更直接写 PostgreSQL entity_state 表（行级权威）+ Redis 热缓存；仅对高价值领域事件（死亡/交易/婚姻/升级/门派变更）写 append-only audit_events 表。ES 仅在需严格回放的子域（PvP 战报归档）单独应用，不作为全系统默认持久化。

### §14 明确 Redis 与 PG 写入顺序协议
- **问题**：同时声明 Redis 为"热状态权威"、PG 为"append-only 权威持久层"、快照为"加速点"，却未定义写入顺序。
- **影响**：若先写 Redis 再异步发 PG 事件，PG 写失败则崩溃即永久丢失；若先写 PG 再更新 Redis，52985 处同步调用无法等待异步投影。被"写后读走强一致读"一句带过。
- **修正**：二选一并写明：(a) Redis 仅读穿透缓存，写先入 PG 再回填（PG NOTIFY 回填）；(b) Redis 为热态权威 + PG 为审计源，放弃"PG 事件可重建 Redis"承诺，热恢复依赖 Redis 持久化(AOF/RDB)+多副本。必须写明一致性强度（线性化？读己之写？）与故障窗口数据丢失边界。

### §15 砍掉误用的外部依赖
- **问题**：Neo4j/Apache Age 用于社交图谱（关系类型<10 种，单玩家基数极低）；APScheduler + Redis ZSET 双调度系统；MMO 式"1跳邻域 AOI"。
- **修正**：Neo4j 改 PostgreSQL 关系表（models: relation_type PK, src, dst, props JSONB）；APScheduler 与 Redis ZSET 二选一明确主调度；AOI 退化为"同房间实体集合 + 邻域静态元数据只读缓存"，边界房间代理仅在玩家跨越分片边界这一低频事件触发。

### §D CombatRoundResolved 须内嵌逐击明细
- **问题**：回合聚合事件丢失逐击归因（死亡原因/PvP 击杀归属/连击检测）。
- **影响**：死亡归因是审计/反作弊的关键诉求，聚合后无法还原"谁最后一击致死"。
- **修正**：CombatRoundResolved payload 内嵌该回合逐击明细数组（[{attacker, target, damage, cause, crit}...]）。定义回合事件大小上限（如 >64KB）自动落 S3 引用、事件只存指针。

## 六、命令与权限

### §16 call_out 迁移要分类
- **问题**：方案反复强调"闭包回调"为核心痛点。但 931 个文件约 3700 处 call_out 中，**3570 处是字符串方法名调用**（`call_out("method", delay, args)` 即四元组，天然可序列化，机械转换可自动化），仅约 144 处使用闭包。
- **影响**：迁移策略过度投入在 144 个闭包上而低估 3570 个字符串 call_out 的批量迁移工程量。闭包真正难点不是序列化而是对象生命周期（触发时目标可能已 destruct）。
- **修正**：(A) 3570 字符串型用自动化工具批量处理（模式固定：`(object, method, delay, args) -> Effect`）；(B) 144 闭包型逐一人工迁移。Effect 必须内置 **target 存活校验**（触发时检查目标 actor 是否存在，不存在则降级为 no-op 或日志告警）。

### §17 PermissionService 必须 fail-closed
- **问题**：授权结果缓存于 Redis，未规定 Redis 不可达时 fail-open（放行，安全漏洞）还是 fail-closed（拒绝，拒绝服务）。
- **影响**：分布式环境 Redis 不可用是必然事件，缺失策略意味着安全事故或可用性事故。
- **修正**：明确**安全默认 fail-closed**（Redis 不可达拒绝所有命令执行并返回降级提示），仅 look/inventory 等只读无副作用命令可选 fail-open。PermissionService 内置本地策略快照（内存 LRU），Redis 不可达时快照+fail-closed 兜底。架构文档将此作为不可妥协安全约束。

### §18 调度系统收敛为单一后端
- **问题**：同时用 APScheduler（持久化延迟作业）和 Redis ZSET（分布式调度排序），未说明职责边界与一致性保证。
- **影响**：一个延迟动作可能同时存在于两套调度，产生重复触发或竞态（最难排查的分布式 bug）。
- **修正**：统一为单一调度后端：APScheduler + Redis JobStore（已提供持久化+分布式去重），或直接 Redis ZSET + 自研轻量 worker 轮询。删除 ZSET 方案。

### §19 分离 wizard ACL 与 UGC 能力沙箱
- **问题**：把损坏的 wizard ACL（securityd.c 的 authorized_cmds，决定谁能跑 /cmds/adm）与 UGC 不可信代码沙箱（WASM capability tokens）合并为"能力令牌显式授权"。两类不同威胁模型。
- **影响**：若用 UGC 能力模型替代运维 ACL，巫师命令权限丢失导致平台无法运维；若运维 ACL 渗入 UGC 沙箱则攻击面评估错位。两者绑死拖延"先重建 securityd 再开放协作"前置门。
- **修正**：分离两个安全域：(a) 运维侧保留简单角色门（admin/arch/wiz/player 命令 ACL），先修好 `authorized_cmds` 拼写 bug（第 757 行 `["cmd"]` 应为 `["cmds"]`）与 725-726 行格式不一致（逗号字符串 vs 数组格式），恢复巫师可用；(b) 能力令牌仅作用于 UGC WASM/RestrictedPython 沙箱。securityd 先修复恢复运维可用，**作为 UGC 协作开放的前置阻塞项**。

### §P ExitResolver 须保留 !arg 前置条件
- **问题**：原 command_hook 出口捷径仅在 `!arg`（无参数）时触发（'n' -> go north，但 'n foo' 查找命令 n）。设计未保留此条件。
- **修正**：ExitResolver 实现必须保留 exit-only-when-no-arg：args 为空时查环境 exits 匹配 verb 转 go 动作；args 非空跳过 exit 解析。基线测试加断言：'n' -> go north，'n foo' -> 查找命令 n。

### §A ActionScheduler 与 COMBAT Coordinator 所有权协议
- **问题**：ActionScheduler 每 actor 一实例，战斗时 COMBAT Coordinator 串行裁决。玩家战斗中能否排队非战斗动作？谁拥有动作执行权？
- **修正**：战斗开始时 Player ActionScheduler 将非战斗动作挂起（suspend），COMBAT Coordinator 接管战斗回合调度；战斗结束后恢复（resume）。action_queue.state 增加 suspended 状态。

### §W WASM 与 async handler 桥接
- **问题**：命令 handler 接口 `async def execute(ctx, args)`，但 wasmtime-py 的 WASM 模块不原生支持 async/await。
- **修正**：三层降级：(1) 可信编辑者用受限 Python 子集（ast 白名单+超时）；(2) 不可信 UGC 用 WASM 同步执行，外层包 async wrapper（`asyncio.to_thread`），超时用线程取消或进程隔离；(3) 明确 WASM 命令 P99 延迟上限（如 50ms）。

### §B Big5 编码转换是应用层中间件不下沉传输层
- **问题**：`query("language")=="BIG5"` 是 per-user 应用层设置，下沉到 WebSocket 传输层混淆了用户偏好与传输编码协商。多路复用连接无法区分 per-user 编码。
- **修正**：保留编码转换为 InputPipeline 应用层中间件（按 `ctx.actor_id` 查询用户 language 偏好），传输层统一 UTF-8。位置在限流之前（避免转换开销被限流误计）。

### §R 角色级权限缓存失效协议
- **问题**：role_definition 继承链变更时，所有该角色的 subject 都需失效。"按 subject 失效"需枚举所有受影响 subject，大规模下缓存雪崩。
- **修正**：role_definition 变更时发布 `RoleCapabilitiesUpdated(role_id)` 事件，各节点按 role_id 批量失效（本地维护 role_id->subject_ids 反向索引）。或用版本号嵌入缓存键，变更即版本号递增，旧缓存键自然失效。

## 七、DSL 与 UGC

### §21 inquiry 是交易状态机非对话树
- **问题**：inquiry 本质是"询问->设旗标->收物->校验->交付"的原子交易流程（101 个 NPC 同时耦合 inquiry+accept_object+set_temp）。Ink 的 knot/stitch/divert/weave 是叙事分支模型，不天然表达原子交易。
- **影响**：强行套 Ink 会丢失交易原子性。迁移 424-434 个 inquiry 文件时大量需手写适配。
- **修正**：层2引入"交易原子节点"语义：dialogue node 可挂载原子交易（give_item + take_money + decrement_stock + set_flag 为单一原子单元），与 Ink 纯流程 divert 区分。或更激进：简单字符串响应转层0，带交易的转"对话+交易"混合节点。先做 inquiry 子分类统计：纯字符串响应 vs 函数回调带交易。

### §22 逃生舱使用率 KPI + 表达力校准实验
- **问题**：方案自评约 70% 逻辑需规则或脚本，但实测复杂面更大（189 个 auto_perform NPC AI 循环、101 个 inquiry+accept_object 交易 NPC、75 个 call_out 闭包、172 个 perform 状态机、40 个 CND_CONTINUE）。层1规则 DSL 动词集缺少多 actor buff/监控共同战斗/加权随机等原语。
- **影响**：逃生舱滥用使"声明为主脚本为辅"退化为"脚本为主"，UGC 安全模型与可视化编辑目标落空。
- **修正**：加大层1规则 DSL 表达力投入，优先实现 perform 效果原语与状态机模式。设逃生舱使用率 KPI（如 <15% 逻辑走 WASM/脚本），超标触发 DSL 表达力迭代。**先做表达力校准实验**：取 30 个代表性文件（含 hebi.c 双人和合、auto_perform AI、inquiry 交易、condition 状态机）人工转译，统计落入各层分布与无法表达比例。层3占比超 20% 则先扩充层1-2 原语（apply_buff_to_actors/weighted_random/monitor_cooperation）再推进迁移。

### §23 UGC 脚本用受限 Python 非 WASM
- **问题**：层3 WASM 逃生舱要求作者写 Rust/AssemblyScript/WAT，而目标 UGC 作者是非程序员中文 MUD 爱好者。
- **影响**：UGC 创作者触顶即卡死，无中间路径，平台活跃度受限。
- **修正**：分层创作体验：层0-2 声明式覆盖 80%；层3 用**受限 Python 子集**（RestrictedPython 或 Pyodide 沙箱执行）降低门槛；WASM 仅保留给平台级已审计扩展（可信编辑者）。为 UGC 作者提供层2->层3 平滑过渡（表达式 DSL 扩展点）。RestrictedPython 非安全边界，需叠加运行时资源配额(fuel)+能力令牌校验作为纵深防御。

### §24 provenance 后移到首次发布前
- **问题**：方案坚持"从第一个 CPK 即内建内容寻址与 license，不后补"。MVP 开发期尚未开放发布，没有外部 UGC 需审计。
- **影响**：开发期每条资产创建付溯源开销，拖慢迭代。
- **修正**：把 provenance 强制点从"第一个开发 CPK"后移到"首次对外发布前（审批门3开启前）"。开发期用简单版本号快速迭代；门3开启前再强制全量资产回填 provenance 并固化 CPK 格式。

### §H WASM 沙箱与 perform 多 actor 有状态协调
- **问题**：WASM 沙箱"天然无状态可水平扩展"与 perform 状态机逃舱自相矛盾。hebi.c 双刀和璧是深度有状态多 actor 协调（共享 buff、递归 check_fight 监控双方、跨对象消息协调）。
- **修正**：明确将 WASM 定位为**无状态计算单元**（pure function：input state -> compute -> output action），所有状态（Effect、双人和合协调态）外部化为可序列化 Effect/QuestState 实体存 Redis/PG；WASM 每次调用只做单步状态转移，多步状态机由 ConditionScheduler 驱动 tick 反复调用。或对这类强状态多 actor 协调扩展层1规则 DSL 的 Effect 原语，承认它是规则而非脚本。

### §L Checkpoint 纳入 NPC 本地持久态
- **问题**：Checkpoint 模型只覆盖玩家临时旗标与对话游标，未纳入 NPC 本地 dbase 持久字段（如 qiu.c 的 wan_count/dan_count 库存计数）。
- **影响**：热重载后 NPC 库存等本地状态重置，玩家可重复刷物品。
- **修正**：扩展 Checkpoint 模型为 `{scene_id, temp_flags, npc_local_state, dialogue_cursor, quest_states, active_effects, version}`。战斗中（任何 actor 处于 combat）禁止热重载。

### §M 分布式 fuel 聚合
- **问题**：单进程 eval_cost 由驱动统一强制；分布式 WASM fuel 由 wasmtime 按 module-invocation 计。CPK 逻辑跨多模块/多 actor/多节点时谁聚合 fuel？恶意 CPK 可发起大量小调用，每次低于单次上限但累计无界。
- **修正**：设计分布式燃料聚合器：按 CPK 维度聚合跨调用 fuel（Redis 计数器），设 CPK 级滑动窗口配额，超限熔断。call_out_quota 明确为"CPK 可挂起的未完成延迟任务总数"上限，调度器入队时校验。

### §X Louvain 静态分片 vs UGC 动态新增
- **问题**：UGC 作者持续加房，出口图动态变化，每次 re-run Louvain 并 re-shard 开销大且打断在线玩家。
- **修正**：改用增量/在线社区检测（动态 Louvain 或标签传播），CPK 变更只影响局部社区。明确 re-shard 期间玩家迁移协议（粘性会话+旧分片宽限期）。定义区域共同体稳定性阈值避免高频 re-shard。

## 八、Agent 协作创作

### §25 combat-sim 循环依赖，用独立 Python 数值模型起步
- **问题**：mcp-verification 的 combat-sim 声称"复用重构后 ECS 战斗引擎"，但仓库中不存在任何 ECS 实现，形成循环依赖。
- **影响**：闭环不收敛即"以验证终态收敛 LLM 非确定性"核心论点无法落地；combat-sim 缺位时 Balancer 角色与门派技能数值不变量无法验证，Agent 产出质量无客观信号。
- **修正**：将 ECS 战斗引擎重构显式列为前置里程碑并设交付标准；在此之前 combat-sim 用**独立纯 Python 数值模型**（直接从 LPC `*action` 招式表与 feature/attack.c 伤害公式 seed 出来），不耦合运行时子系统。world-graph 可独立先行（纯 networkx 图分析，无 ECS 依赖）。

### §26 LangGraph 是单进程库，多实例需分布式锁
- **问题**：LangGraph 是单进程状态机库，checkpoint 持久化到 PG 只支持崩溃后恢复，不提供多实例协调。声称"编排器可水平扩展多副本"不可用。
- **影响**：两个编排器实例同时恢复同一 workflow 会重复分派 Worker、双花 LLM token、产出冲突。
- **修正**：明确分布式协调机制：用 Postgres advisory lock 或 Redis SETNX 给 workflow 加租约（lease），任一编排器持锁后才能续跑；或务实接受"单编排器 + 故障转移（standby 从 checkpoint 接管）"而非 active-active 多副本。MVP 直接单进程 asyncio 编排。

### §T4 技术栈砍到 MVP 最小集
- **问题**：方案涉及 14+ 技术组件（Lark+pydantic+Ink+wasmtime-py+LangGraph+LangChain+Celery+Redis+FastAPI+WebSocket+PostgreSQL+MinIO+networkx+MCP+tree-sitter）。git 历史显示实质单人重构。
- **影响**：团队精力消耗在基础设施集成与运维而非验证核心闭环；故障域过大。
- **修正**：砍 MVP 到最小可验证收敛集：DSL 编译器（Lark+pydantic）+ 单进程 LangGraph 编排 + SQLite/PG + 一个 MCP 检查（world-graph）+ 最简评审 UI。Celery/MinIO/分布式 WASM 池/NATS 等单进程验证语义正确后再按需引入。用"是否阻塞验证收敛闭环"为唯一引入判据。

### §M4 迁移工具不押注 tree-sitter-lpc
- **问题**：tree-sitter-lpc 并非成熟维护的 grammar。LPC 的 heredoc 语法、globals.h 自动全局包含（157 行宏）、simul_efun 的 destruct 覆写、this_object()-> 动态分派（274 文件）、previous_object() 信任链（51 文件）都需流分析而非解析。
- **修正**：不押注 tree-sitter-lpc；采用"正则抽取结构化字段（exits/objects/inquiry/marks）+ 人工标注集覆盖动态分派长尾"混合策略。先用单区域（d/city 约 90 房）做端到端迁移试点，度量语义偏差率。this_object()/previous_object 长尾明确划为人工标注而非自动化范围。

### §B4 纯数据房间估算被高估
- **问题**：即便"纯"replace_program 房间仍含 set("objects") 生成引用、set("resource/grass") 资源采集、set("cost")、set("outdoors")。6414 房中仅 2482 是 replace_program（38.7%），1113 房（17.3%）带自定义 add_action 属非声明式。
- **修正**：层0 schema 固化前先穷举环境/语义字段（objects spawn、resource、cost、outdoors、no_clean_up、resource/* 等）为显式 schema 字段；把 1113 个带 add_action 的房间单独列为层1迁移批次而非"纯数据"。迁移排期按"schema 字段完备度"而非"文件数"估算。

### §S4 技能数估算偏差显著
- **问题**：方案称"359 技能有 *action 招式表"，实测 kungfu/skill 下 472 个 .c 文件，但含 'mapping *action' 模式的仅 163 个；query_action 出现在 165 文件。误差近 2 倍。
- **修正**：迁移前先跑精确基线统计（实测：*action 163、query_action 165、NewRandom 131、auto_perform 128、call_out 1011、replace_program 2482、带 add_action 房 1113/6414），用实测值重算层3/WASM 工作量与人力预算。

## 九、前端与状态同步

### §28 tmp_dbase≠delta（概念错误）
- **问题**：方案反复声称"dbase=snapshot、tmp_dbase=delta 直接对应"。但代码证实 dbase 与 tmp_dbase 是两个并行命名空间（分别经 query()/query_temp() 访问），tmp_dbase 不持有针对 dbase 的差分补丁，而是独立瞬时键（apply/attack、pending/exercise、netdead）。apply/ 修饰符是渲染/查询时叠加，不是 diff 流。
- **影响**：若按字面把 tmp_dbase 流式化为 patch，会构建出与真实状态变更方式不匹配的 delta 引擎。补丁生成逻辑需从零设计。
- **修正**：放弃"tmp_dbase=delta"映射话术。明确 dbase 与 tmp_dbase 是并行命名空间。StateTree+delta 作为全新设计：服务端维护单一权威状态树（整合 dbase 与 tmp_dbase 的有效值），变更通过对比前后两次权威树生成 JSON-Patch ops。

### §29 dbase 函数值/闭包键击穿 snapshot
- **问题**：dbase.c 第 51 行 query() 返回 `evaluate(data, this_object())`，实测 104+ 文件 set 了闭包值（如 `set("long", (: long_desc :))`、`set("actions", (: call_other, WEAPON_D, "query_action" :))`）。方案把 dbase 当纯数据快照树。
- **影响**：snapshot 序列化会在闭包键上静默丢弃或崩溃。战斗 actions 是要事件化的核心，却是最典型的闭包值场景，方案内部矛盾。
- **修正**：建立 dbase 键分类台账：data-valued（可快照）vs function-valued（服务端求值后下发）。战斗 actions 等闭包必须在服务端 resolve 成动作描述符再进事件，绝不进 snapshot。对 query() 的 evaluate 路径做统一拦截，标记哪些键是 dynamic，快照时显式跳过或求值。

### §30 渲染层迁移规模被严重低估
- **问题**：message_vision 用于 1626 个文件、message() efun 1070 处、tell_room 317 处。每个 message_vision 调用点把 $N/$n/$P/$p 模板内联在游戏逻辑里。
- **影响**：模板池抽离是数年级工作量，currentToNewMapping 把它呈现为一行映射，严重误导工时与风险预期。
- **修正**：分级迁移子计划：过渡期保留 LPC message_vision 经 Telnet 适配器作为真相源；仅先对最高频的战斗/社交事件抽 msg_key 模板；把 1626 处显式列为多年期 backlog 设优先级；明确"模板池是渐进替换非一次性取代 message_vision"，配合 lint 防 combatd 与模板池双份维护。

### §31 战斗是 tick 流非回合制
- **问题**：方案假设存在离散"回合"可做 CombatRoundResolved 聚合。但 XKX 战斗是连续 tick（每个 heart_beat 一次攻击机会），没有天然 round 边界。
- **影响**：强行造 round 要么改变战斗时序语义，要么事件量接近 tick 级。
- **修正**：把战斗建模为 tick 流，显式定义聚合窗口（按防御方/按时间窗/N-ticks），不假设 round 存在。让 round 成为前端 UI 概念，不作为协议边界。连续态（qi/jing）的周期快照保留。

### §31b apply/ 修饰符两类语义
- **问题**：apply/ 是重载的--`apply/name`/`apply/short`/`apply/long` 是数组栈（last-wins 覆盖），`apply/attack`/`apply/damage`/`apply/armor` 是标量累加槽。方案用单一 ModifierStack 混淆。
- **修正**：统一建模为 Component：presentation 型（覆盖，last-wins）、stat 型（累加，add 语义）、transform 型（数组栈）。前端按类型分别聚合。

### §31c 重连降级路径进协议
- **问题**：ring buffer 满后旧 delta 被驱逐，客户端 last_seq 过旧需降级全量 snapshot 重发，但 ReconnectResult 接口只有 replay_buffer 字段，无"改发 snapshot"模式。
- **修正**：增加降级路径：服务端检测 `last_seq < ring_buffer_oldest_seq` 时发 `type:'snapshot_resend'` 而非 replay_buffer，客户端按全量快照重建。协议接口补这个分支并文档化触发条件。

### §31d 双栈过渡须一致性测试
- **问题**：若 Telnet 适配器走 LPC message_vision、Web 适配器发语义事件，是两条会漂移的代码路径，无不变量保证。
- **修正**：把"每条 Telnet 输出 = render(某条语义事件)"固化为契约，加一致性测试（Telnet 与 Web 同一事件必须互为投影）。

### §31e 天雷惩罚是业务玩法非限流
- **问题**：alias.c 的 cnt 计数超阈值不只限流，还会对玩家造成 qi/jing 伤害（天雷劈）、unconcious、踢出--这是 themed 惩罚玩法。
- **修正**：把"天雷惩罚"显式建模为业务级反作弊/惩罚子系统（独立于网关限流），保留 themed 惩罚语义。网关令牌桶只负责基础设施级限流。

### §31f heart_beat 实测为 1s
- **问题**：代码用 set_heart_beat(1)，MudOS 默认即 1s。方案把"2s tick"当作既定事实。
- **修正**：核实驱动配置实际值后再定 batch 边界数学；或明确"2s"是目标配置而非现状。

### §31g default_ob 是活动主对象非静态 prefab
- **问题**：LPC default_ob 是可挂 heart_beat、可被修改、会 add("no_clean_up",1) 副作用的 live master。方案映射为静态 Prefab 资源回退，丢失活动主对象语义。
- **修正**：default_ob 映射为"活动主对象代理"：前端 prefab 仅作只读默认值快照；任何 default_ob 的可变/计算默认值必须由服务端 resolve 后再进投影，不交给前端静态 prefab。

## 十、运维可观测与部署

### §32 有状态分片用预测性弹性非反应式
- **问题**：KEDA 读"在线会话数+handoff 队列"是滞后信号；队列已深时玩家已在掉线。新分片暖启动（蓝图加载+AOI 重建+region 注册，对 20 万对象可能分钟级）期间负载峰值已过。叠加反抖动使响应更慢。
- **修正**：建立"预测性+预留容量"模型：基线预留容量常驻（如每 region 1 个 standby 暖分片）、按日历/活动预告预扩、用负载趋势（导数）而非滞后值触发。区分"接新连接"（无状态边缘秒级）与"承载既有 region 仿真"（分片分钟级，不可反应式）。

### §33 战斗变更灰度即全量
- **问题**：武侠 MUD 玩家自由移动，两版本战斗公式不同时"强制同房间同版本"机制无法成立（阻断进入是差 UX，强制迁移该玩家脱离队友）。
- **修正**：明确承认战斗/数值变更本质是"灰度即全量"。走蓝绿全量（暖启动 standby 集群->handoff 整批切换），放弃分片粒度金丝雀战斗逻辑。金丝雀先只切无状态边缘。

### §34 liveness 仅判事件循环死锁
- **问题**：若 liveness 关联 tick 新鲜度，合法繁忙分片（大战斗/GC/慢 tick）被判死->K8s 杀->重启暖启动->流量转移->其他分片也变慢->级联重启。
- **修正**：liveness 仅判"事件循环死锁"（极保守，如 30s+ 无任何事件处理），绝不基于 tick p99/p50。tick 质量全部归 readiness（摘流量不杀）。设 startup probe 阈值足够长覆盖 20 万对象暖启动。加 PDB 防多分片同时被杀。

### §35 日志后端用 OpenSearch/ClickHouse 非 Loki
- **问题**：Loki 标签基数是已知陷阱：actor_id/trace_id 作 label 让 Loki 索引爆炸；作日志字段（非 label）则 Loki 无法高效按其检索。UGC 多租户下 tenant_id/world_id 同样高基数。
- **修正**：日志后端用 OpenSearch/Quickwit 或 ClickHouse（支持全文+高维结构化检索），Loki 仅保留低维 access log。若坚持 Loki，明确 trace_id/actor_id 为日志字段（非 label），依赖与 Tempo 的 trace_id 关联查询。

### §36 "秒级回滚无数据丢失"是危险过度声明
- **问题**：(1) Redis handoff 快照带 TTL 可被淘汰；(2) 新版本代码可能已写入旧版本不认识的字段（schema 漂移），逻辑回滚后旧代码读不一致状态；(3) 进行中战斗在旧版本上回滚中途产生不公平结果。
- **修正**：改为"逻辑回滚≠状态回滚，可能产生数据不一致"。回滚前必须评估新旧版本对世界状态 schema 的兼容性；不兼容时需先做状态迁移或回滚到匹配的状态快照。战斗回滚限定"回合边界"+无进行中战斗。Redis 快照设保留下限并监控淘汰。

### §37 call_out 不应统一持久化（分层）
- **问题**：大量 call_out 是单对象内亚秒~秒级延迟，并非需跨重启恢复的调度。全部持久化产生每秒数千小任务写入，NATS/PG 承压，取消语义从内存操作退化为 DB 写。
- **修正**：分层：短延迟（<30s）对象内定时器留 Actor 运行时内存（崩溃丢失可接受，由对象重建恢复）；仅跨重启/全局/长周期定时器走持久化调度器。remove_call_out 对应内存定时器的内存取消（零开销）。

### §38 安全关键审计先于提交
- **问题**：用"热审计先入 NATS JetStream 再归档 PG"缓解写热点，但安全审计（特权授予/文件写）若异步缓冲，则动作与审计之间崩溃=审计缺失。
- **修正**：区分审计等级：特权授予/文件写/CPK 发布等安全关键动作走"审计先于提交"（同步写 PG 或同步 NATS 确认后才执行动作）；高频低风险审计走异步缓冲。审计写瓶颈最终迁 ClickHouse。

### §39 trace context 传播助手 + lint
- **问题**：asyncio 的 contextvars 是 task 作用域，create_task/gather/ensure_future 默认不自动透传 OTel context；裸 create_task/回调注册/call_out 等价物若未显式 attach context 即断链。单测只能覆盖契约入口。
- **修正**：提供 contextvars 传播助手并禁止裸 create_task（经 lint 强制）；为 call_out 等价物/调度回调显式 propagate context 并加契约测试覆盖所有异步边界。定义"trace 完整率"指标（有父 span 的子 span 占比）持续监控断裂。OTel span 策略统一：跨服务/跨分片边界 span 默认开；服务内 actor 跳转用轻量子 span 或 span links；战斗关键步骤显式 span；长 trace 改概率采样替代尾采样。

### §40b 配置热更新两阶段 apply
- **问题**：各分片在各自 tick 边界 apply，两分片可能短暂持有不同配置版本。对 IP 封禁是安全窗口。
- **修正**：全局安全/数值配置走"两阶段提交"语义：ConfigCenter 发布版本->各分片确认 apply 后才标记版本生效；未确认分片 readiness 降级。IP 封禁这类安全规则下发到网关层（网关统一读最新版本）而非依赖每分片 tick 边界 apply，消除安全窗口。

### §40c GBK->UTF-8 编码探测
- **问题**：设计自己在摘要注明"GBK/Big5 混合编码"，但 Vector GBK 解码器不处理 Big5。CRASHES 文件实测为 ISO-8859（非纯 GBK）。统一按 GBK 转码会产出乱码。
- **修正**：Vector 用确定性字符集探测（GBK/Big5/UTF-8 多次尝试+置信度）而非固定 GBK 解码。先全库扫描统计实际编码分布。考虑过渡期对历史日志做一次性批量转码归档（而非长期双写）。

### §40d MVP 可观测栈两阶段
- **问题**：12+ 基础设施系统对从零起步（无 Docker/Python/K8s 痕迹）的团队是巨大运维与认知负担。
- **修正**：定义"MVP 可观测栈"（Prometheus+单一日志后端+Tempo，全部托管优先）与"完整栈"两阶段。优先用云托管（Grafana Cloud/等效）降低自托管负担。docker-compose 本地对等必须覆盖可观测端到端路径。加"自托管 vs 托管"决策矩阵与成本模型。

### §40e distroless 镜像配 debug variant
- **问题**：无 shell 的 pod 在崩溃事后调查时缺少现场工具。
- **修正**：对 distroless 镜像配 debug variant（含 busybox shell）供事故时 kubectl debug 切换；或用 wolfi-base 保留最小 shell。崩溃现场通过 OTel fatal 事件+coredump 落对象存储。

### §40f day_phase 不必占 HA 单例槽位
- **问题**：day_phase 原始语义是全局世界时钟广播。若走"每世界单例 coordinator"广播，是对纯表现层特性的过重协调。
- **修正**：分两类："仿真 tick"是每分片本地确定性步进（无需全局单例）；"世界级广播事件"（昼夜/季节/公告）由轻量广播服务扇出，各分片订阅本地分发。day_phase 不必占 HA 单例槽位。
