# 迁移路径与里程碑

> 本路线图以 [01-关键修正与避坑清单](01-关键修正与避坑清单.md) 的承重修正为驱动，按"风险递增、差分测试门禁、单进程起步"原则编排。**核心立场：不要为 50 用户的系统预先分布式化，不要在原型前承诺自研运行时，不要在差分测试通过前推进下一阶段。**

---

## 一、总体原则

1. **风险驱动排序**：先低耦合后高耦合（Query/索引 -> Identity/Position -> Vitals/Heal -> Combat）。Combat 是最高风险（活引用/递归/闭包全在此），放最后。
2. **差分测试门禁**：每个子系统迁移必须先录 golden trace，无差分测试通过的子系统不进下一阶段。
3. **单进程起步**：阶段一单进程 asyncio + 单 PG + 单 Redis 跑通核心改造，分布式留到单节点达 80% 承载后。
4. **原型先行**：Ray actor 原型验证 LPC↔Actor 同构映射，再决定是否自研薄壳。
5. **容量以实测定**：单分片承载上限由压测得出，按 2000-3000 并发规划而非假设 5000-8000。
6. **MVP 最小技术栈**：用"是否阻塞验证核心闭环"为唯一引入判据，砍掉所有"显得先进而非解决痛点"的选型。

## 二、分阶段路线图

### 阶段 0：现状基线与差分测试基建（1-2 月）

**目标**：建立迁移地基与安全网，不写新功能。

| 任务 | 产出 | 验收标准 |
|---|---|---|
| 精确统计迁移面 | 68771 调用点分类表、3570 字符串 call_out + 144 闭包清单、1626 message_vision 清单、104 闭包 dbase 键台账、989 dbase 键读写点映射 | 全库扫描完成，数字可复现 |
| 修复 securityd.c 已知 bug | authorized_cmds 拼写（第757行 `["cmd"]`->`["cmds"]`）+ 725-726 格式不一致 | 巫师命令权限恢复可用 |
| 差分测试基建 | golden trace 录制框架（LPC 命令流+tick 边界）+ Python 重放 diff | go/move/combat/channel 四主线可录制可重放可 diff |
| 单区域迁移试点 | d/city 约 90 房端到端迁移 | 度量语义偏差率，校准全量迁移可行性 |
| 现有系统行为基线 | 关键战斗/经济/社交的输入输出快照 | 数值回归基线建立 |

**不可跳过**：差分测试基建是后续所有迁移的安全网，无此则行为漂移不可见。

### 阶段 1：单进程 asyncio 核心循环（3-6 月）★ 不可跳过

**目标**：单进程跑通核心改造，验证语义正确性。

```
┌─────────────────────────────────────────┐
│ 单进程 asyncio (Ray actor 模型起步)      │
│  ├─ SparseSet ECS 骨架                   │
│  │   ├─ Identity/Vitals/Position 核心组件│
│  │   └─ Progression 组件 (combat_exp等)  │
│  ├─ SchemaRegistry (类型化组件, 拼写拦截) │
│  ├─ ReferenceStore (ID 引用, PG关系表)   │
│  ├─ SchedulerStore (持久调度, 分层)       │
│  └─ WorldStateReconciler (desired-state)  │
├─────────────────────────────────────────┤
│ 无状态 WS 网关 + 会话 + JWT/argon2       │
│  └─ ring buffer 重连                      │
├─────────────────────────────────────────┤
│ 权威状态库 PG + Redis 热缓存 (非全量ES)   │
│  └─ Redis/PG 写入顺序协议 (明确一致性强度)│
└─────────────────────────────────────────┘
```

| 里程碑 | 验收 |
|---|---|
| Ray actor 原型验证 LPC↔Actor 同构映射 | go/move 语义在 Ray actor 上跑通，延迟+消息序可接受 |
| SparseSet ECS 骨架 + 核心组件 | Identity/Vitals/Position/Progression 可创建/查询/迁移 |
| SchemaRegistry 类型化组件 | query('xxx') 拼写错误编译期拦截 |
| 字段->组件映射表 | query_entire_dbase 调用点读写键集枚举完成（CombatSystem 迁移前置） |
| 无状态 WS 网关 + 认证 + 重连 | 单进程承载会话，断线重连 ring buffer 重放 |
| 权威状态库 + 审计 | PG entity_state + audit_events（仅高价值事件）+ Redis 热缓存 |
| Redis/PG 写入顺序协议 | 一致性强度（线性化/读己之写）与故障窗口数据丢失边界写明 |
| 差分测试 go/move/combat/channel 四主线通过 | golden trace 重放零差异 |

**阶段一硬约束**：
- Effect 提升为一等公民组件（可序列化可中断可迁移）
- ConditionHandler.on_tick 返回组合结构（continue/suppress_heal/expire）契约落定
- 无差分测试通过的子系统不进阶段二

### 阶段 2：子系统逐个迁移（6-12 月，按风险递增）

**目标**：逐子系统替换 LPC 行为为 Python System，每步保持游戏可运行。

| 顺序 | 子系统 | 风险 | 前置条件 |
|---|---|---|---|
| 2.1 | Query/索引层 + Identity/Position/Inventory | 低 | 分片框架验证 |
| 2.2 | Vitals/Heal/Condition | 低（只读为主） | ConditionHandler 契约（修正 §12） |
| 2.3 | Attribute/Skill/Equipment | 中 | ModifierStack 三类语义（修正 §31b） |
| 2.4 | **Combat**（最高耦合） | **高** | 数值回归基线 + CombatContext 快照边界 + Effect 组件 |

**Combat 迁移专项（2.4）**：
- 迁移前必须先建 per-attribute/per-skill 输入输出快照对比基线
- CombatContext 快照边界定义（战斗开始快照全体参与者+可观察环境）
- do_attack 29 处 random() 调用点收口到 CombatRNG
- 闭包型 call_out（144 处）逐个翻译为 Effect
- s_combatd.c 废弃，仅提取阵法合击/自创连击为 CombatModifier
- 跨分片战斗降级语义明确（同分片同步递归，跨分片异步 RiposteEvent）

### 阶段 3：分布式与多分片（按需，单节点达 80% 承载后）

**目标**：当单进程承载达到 80%（实测约 2000-3000 并发）时才进入分布式。

| 里程碑 | 验收 |
|---|---|
| Ray 多节点 + placement + handoff | actor 适配器验证跨节点迁移不丢消息 |
| 区域共同体分片（Louvain） | 5-8 共同体分片，边界对齐社区割边 |
| 跨区迁移 handoff | freeze->snapshot->activate->replay->deactivate，每步超时/补偿定义 |
| 边界房间代理 + 一致性哈希热路径 | 热路径去 Redis 化，Redis 仅控制面 |
| 区域监督 one-for-one + 持久化重建 | 单房间崩溃只影响自身 |
| 32 守护进程全量分类迁移 | 无遗漏子系统 |

**分布式硬约束**：
- 热路径一致性哈希去 Redis 化（修正 §6）
- 区域监督 one-for-one 非 all-for-one（修正 §7）
- handoff 管线每步超时/补偿/最终一致目标规约（修正 §V 补充）
- CombatContext + seeded RNG 支持回放（修正 §9）

### 阶段 4：UGC 与 DSL 平台

## 四、风险驱动的"不做"清单

这些是研究过程中识别出**看似先进但不解决痛点、或风险远超收益**的选型，明确放弃：

| 不做 | 理由 | 替代 |
|---|---|---|
| 从零自研 asyncio 虚拟 Actor 运行时 | 多年级工程 | Ray 起步，验证后再决定是否自研薄壳 |
| SparseSet + Archetype 双存储 ECS | 互斥策略，徒增复杂度 | 只用 SparseSet |
| 全量事件溯源(CQRS+ES)作为默认持久化 | 52985 处同步调用与最终一致根本冲突 | 权威状态库 + 选择性审计 |
| Neo4j/Apache Age 社交图谱 | 关系类型<10，过度工程 | PostgreSQL 关系表 |
| MMO 式"1跳邻域 AOI" | MUD 是房间作用域非连续空间 | 同房间实体集合 + 邻域静态元数据只读 |
| ModifierStack 用 Redis ZSET | 丢失修饰符组合语义 | modifier 链表/栈（分三类） |
| APScheduler + Redis ZSET 双调度 | 一致性风险 | 单一后端 + 分层（短延迟留内存） |
| 全库 3728 call_out 统一持久化 | 性能反模式 | 分层：短延迟(<30s)留内存 |
| 直接用 Akka/Orleans | 非 Python 增跨语言成本 | Ray + 自研薄壳（如需） |
| 过早引入 Kafka | 单区域 PG+NATS 足够 | 跨区域多租户达规模再议 |
| tree-sitter-lpc 批量解析 | 非成熟 grammar，动态分派需流分析 | 正则+AST+人工标注混合 |
| 层3 UGC 用裸 WASM | UGC 作者是非程序员 | RestrictedPython（WASM 仅平台级） |
| provenance 从第一个开发 CPK 内建 | MVP 拖慢迭代 | 门3开启前才强制 |
| combat-sim 硬等 ECS 引擎 | 循环依赖 | 独立纯 Python 数值模型起步 |
| LangGraph active-active 多副本编排 | 单进程库，重复分派 | 单编排器+故障转移（如需多实例加分布式锁） |
| Loki 做高维日志检索 | 标签基数爆炸 | OpenSearch/ClickHouse |
| 有状态分片用 HPA 反应式扩缩 | 分钟级暖启动与滞后信号时延错配 | 预测性+预留容量 |
| liveness 关联 tick 新鲜度 | 杀死级联 | 仅判事件循环死锁 |
| 战斗变更分片金丝雀 | 玩家流动，灰度即全量 | 蓝绿全量 |
| 秒级回滚无数据丢失（过度声明） | schema 漂移导致不一致 | 逻辑回滚≠状态回滚，需评估 schema 兼容性 |
| 全量事件驱动替代 heart_beat | heart_beat 是 tick 驱动器 | 保留周期 tick，事件仅覆盖触发器 |

## 七、决策检查点

每个阶段结束前，必须回答以下问题，任一为否则不推进：

**阶段 0 -> 1**：
- 差分测试基建是否覆盖 go/move/combat/channel？
- securityd 已知 bug 是否修复？
- 迁移面统计是否完成？

**阶段 1 -> 2**：
- Ray actor 原型是否验证 LPC↔Actor 同构映射成立？
- Redis/PG 写入顺序协议是否写明？
- Effect/ConditionHandler 契约是否落定？

**阶段 2 -> 3**：
- 单进程是否达 80% 承载？
- Combat 迁移是否通过差分测试（CombatContext + seeded RNG）？
- 32 守护进程是否全量分类迁移？

**阶段 3 -> 4**：
- 单分片容量是否实测？
- 分布式 handoff 是否规约化（超时/补偿/最终一致）？

**阶段 4 -> 5**：
- securityd 能力模型是否重建？
- 沙箱配额是否就位？
- DSL 表达力校准是否通过（层3<20%）？
