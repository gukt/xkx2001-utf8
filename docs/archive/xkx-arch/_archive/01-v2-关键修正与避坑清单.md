# 关键修正与避坑清单（v2）

> 本文档汇总 v2 视角下的承重级假设错误。v1 版本（[01-关键修正与避坑清单](01-关键修正与避坑清单.md)）经 8 位专家对抗复审后，发现存在"greenfield 与增量重构概念混淆"这一承重级缺陷，并新增大量源码级修正（代词 10 变量、emote 7 视角、5 个灵魂系统遗漏、技能三层、Python 性能天花板、combat-sim 双重标准、验证覆盖度幻觉等）。
>
> **本清单以 [05-专家对抗复审报告](05-专家对抗复审报告.md) 的交叉质证结论为准**，对 v1 保留的修正不再重复标注；重点补充 v2 必须纳入的 greenfield 语义、阶段 -1 垂直切片、Ray 非游戏服务器框架、性能基准与工具链等。

## 一、greenfield 语义：清除所有增量重构假设

### §G1 阶段 0 不是"录制运行中 LPC 命令流"

- **问题**：v1 把阶段 0 定义为"修复 securityd.c 已知 bug、录制 LPC 命令流为 golden trace、单区域端到端迁移试点"，默认旧 LPC 系统是当前运行真相源。
- **影响**：greenfield 项目不存在"修旧 bug / 双栈过渡 / 逐步替换 / 适配层"概念。若按增量重构实施，会在阶段 0 陷入"让 25 年前的 MudOS 在现代 Linux 跑起来"的数周级工程，且概念上把 LPC 当作运行系统而非规格源。
- **修正**：
  - 阶段 0 改为"**LPC 规格提取与验证基建**"（spec extraction + equivalence verification）。
  - 删除所有"迁移适配层 / 双栈过渡 / 过渡期 / 逐步替换"概念。
  - securityd **不修旧代码**，从 `securityd.c` 提取权限规格后从零实现新 PermissionService。
  - 差分测试重定义为"**规格提取 + 行为等价验证**"：对照提取的形式化规格 + 本地 MudOS spec oracle，而非运行中的生产 LPC。
  - `s_combatd.c` 不存在"废弃"概念，只提取阵法合击规格后按规格实现。

### §G2 32 守护进程不是"全量分类迁移"

- **问题**：v1 要求"32 守护进程全量分类迁移"，隐含按名称一一对应。
- **影响**：greenfield 下大量守护进程职责会被 ECS System / 无状态服务 / 新能力取代，强行"迁移"会制造不必要的历史包袱。
- **修正**：改为"32 守护进程的**职责重新设计**"——标注每个 daemon 是被 ECS System 取代 / 保留为无状态服务 / 演进为新的平台能力（如 TitleSystem / WorldGovernanceSystem）。

### §G3 迁移内容 Day1 以 CPK 形式作为官方 StdLib 入库

- **问题**：v1 把"先迁移后 UGC"串行化。
- **影响**：延迟享受 greenfield 红利，StdLib 与 UGC 执行路径隔离不清。
- **修正**：迁移内容 Day1 以 CPK 形式作为官方 StdLib 入库，UGC 平台架构并行设计。StdLib 是可信 Python（无沙箱限制），UGC 是沙箱受限；CPK manifest 区分执行路径权限。

## 二、MVP 顺序：阶段 -1 垂直切片平台验证

### §V1 最高产品风险最后验证是顺序反了

- **问题**：核心价值主张"DSL+Agent 让非程序员创作可玩世界"要到阶段 4-5（14+ 月后）才被验证，前 14+ 月全在验证工程可行性。
- **影响**：若 DSL+Agent 创作闭环在工程或产品上成立，14 月引擎投入全部沉没。
- **修正**：在阶段 0 前插入"**阶段 -1：垂直切片平台验证**"（2-3 月）：
  - 5-10 房间 + 2 NPC + 1 战斗 + 1 任务 + 1 对话，全部 DSL 定义。
  - Agent 部分生成，验证 copilot 模式 + 产出可用性。
  - 跑在**真实引擎最小子集**（`resolve_attack` 纯函数 + stub 运行时），不依赖完整 ECS 运行时但保证与未来真实引擎同源。
  - 验证三件事：DSL 能否表达、Agent 产出是否可用、玩家是否觉得好玩。
  - **通过后才投入引擎重构**。

### §V2 垂直切片必须用真实引擎最小子集而非 stub

- **问题**：制作人曾建议用"最小 stub 运行时"，但 stub 语义与真实引擎不一致会制造 false confidence。
- **影响**：验证信号失真比没验证更危险。
- **修正**：架构师与 AI 专家裁定用"**真实引擎最小子集**"——提取 `resolve_attack` 为无副作用纯函数。纯函数提取是 Combat 迁移阶段 2.4 的前置工作，提前做不浪费。

## 三、Actor 与分布式：Ray 不是游戏服务器框架

### §A1 阶段 1 不引入 Ray 做实体运行时

- **问题**：v1 建议"阶段一用 Ray 的 actor 模型做最小可用验证"。但 `do_attack` 在 `combatd.c` line 772/777/804/812/823/831 递归调用自身（riposte/double_attack/follow_up），分支由 `random()` 决定。Ray actor 默认 FIFO 非重入，actor A 调 B、B 回调 A 会死锁；`ray.get()` 阻塞事件循环使分布式化为零。
- **影响**：68771 处同步调用的调用栈语义与递归回溯无法在 Ray actor 中正确映射。
- **修正**：
  - 阶段 1 用**纯 asyncio + 普通实体对象**（同步方法调用，与 LPC `call_other` 语义一致），**完全不引入 Ray**。
  - **分片内同步、分片间异步**：同房间/同区域实体 colocate 到同一进程，`call_other` 退化为进程内同步函数调用，仅在跨分片边界用异步消息。
  - Ray 推迟到阶段 3 真正需要分布式**计算**时引入，且仅用于无状态/可并行计算（Agent 编排、combat-sim 批量、离线迁移工具），**不做游戏实体运行时**。
  - greenfield 下"自研薄壳"选项彻底剔除。

### §A2 handoff 必须写成可执行协议规约

- **问题**：v1 只写"freeze→snapshot→activate→replay→deactivate，每步定义超时/补偿/最终一致目标"，但无失败语义定义。snapshot 写 Redis 附 TTL，过期且激活未完成 = 实体永久丢失；战斗中 handoff，freeze 期间对端分片攻击回合如何处理？
- **修正**：采用**两阶段提交 + outbox 模式**：
  - 源分片将 handoff 作为持久化事务写入 outbox（PG）。
  - 目标分片从 outbox 拉取并确认；确认后源分片才删除旧实体。
  - handoff 期间实体对外"只读冻结"（写入命令排队）。
  - 组队批量 handoff 用一个 outbox 事务包裹全队，原子提交。
  - 每步的幂等性、重试策略、超时降级必须写成可执行协议规约。

### §A3 单房间串行是分布式无法突破的硬约束

- **问题**：v1 提出">30 人拆子 Coordinator"，与 3000 人聚集差两个数量级。
- **影响**：MUD 核心社交魅力是"全服同房"——主城广场喊话全服可见、PK 围观者实时看到战斗、武林大会千人同场。实例化分桶会摧毁这个体验。
- **修正**：
  - 不用实例化分桶。
  - 务实下调目标为"**万人注册 + 千人同时在线**"（MUD 的合理量级）。
  - 热点房间用"**专用分片**"（整个分片只服务一个主城房间）。
  - 房间内并行化战斗裁决（按战斗组分而非玩家分桶）。
  - 明确"单房间串行是 MUD 固有硬约束，分布式无法突破"。

### §A4 客户端延迟隐藏缺失

- **问题**：v1 未正面设计移动命令的乐观执行。
- **影响**：现代游戏 UX 基线缺失。
- **修正**：移动命令乐观执行——前端本地渲染 + client_seq 上报，服务端确认回 ack，不一致发 rollback delta。

## 四、差分测试与验证：从 golden trace 到规格 + 统计回归

### §D1 差分测试 P0 门禁有未声明硬依赖

- **问题**：当前仓库 `./driver` 是 Mach-O（macOS）二进制，目标平台是 Linux（WSL2）。要录制 golden trace 必须先在 Linux 上重新编译 FluffOS 驱动并让 XKX 可启动。系统当前并非完整可运行状态。
- **影响**："让 25 年前的 MudOS 在现代 Linux 跑起来"本身可能是数周级工程，被当作阶段 0 顺手做掉。
- **修正**：
  - 阶段 0 第一步显式加入"**FluffOS Linux 编译 + XKX 可运行性验证**"前置任务（保守估 2-4 周）。
  - 备选：在 Mac 上一次性录制 golden trace（不需持续运行）；或降级为"**单元级行为规约**"（从代码人工推导期望行为写断言，不依赖运行时）。
  - **双轨制**：(A) 单元级行为规约先行（从代码提取 go/move/combat 输入输出契约，不依赖运行 LPC，2-3 周可覆盖核心路径，是 greenfield 主门禁）；(B) 对可运行 FluffOS 录制一次性 golden trace 做统计回归（伤害分布/频率，非逐位对齐）。

### §D2 combat-sim 不能是独立简化模型

- **问题**：v1 建议 combat-sim 用"独立纯 Python 数值模型"顶上，但这与"迁移用 golden trace 差分测试保证行为等价"自相矛盾。独立模型若简化 riposte 递归、fight/dodge 累积态、condition 位聚合、Effect stacking，验证通过的内容上线后可能失衡。
- **影响**：验证信号失真比没验证更危险。
- **修正**：combat-sim **直接调用从 LPC 提取的 `resolve_attack` 纯函数**（输入组件快照 + seed，输出 CombatRoundResult），不依赖完整 ECS 运行时但保证验证与未来真实引擎同源。greenfield 重写的 Python 代码本就用 seeded RNG，运行时放弃确定性不应被延伸。

### §D3 "可验证终态"是覆盖度幻觉

- **问题**：v1 用"可验证终态"暗示内容质量已验证，实际 DSL schema 只收敛结构合法性，world-graph 只收敛图可达性，combat-sim 只收敛数值分布。叙事连贯性、对话语义死锁、任务绕过、经济平衡、趣味性全部在覆盖外。
- **修正**：
  - 显式分层声明验证覆盖度：
    - **可机器自动验证**：结构 / 可达性 / 数值分布 / 不变量回归。
    - **需半自动或人工**：叙事连贯（LLM-as-judge）、对话死锁（flag 状态机模型检查 nuXmv/SPIN）、任务绕过（状态空间探索）、经济平衡（多 agent 玩家行为仿真）、趣味性（playtest agent + 人工抽样）。
  - 对后者逐项声明 gap 与目标覆盖度，不掩盖。

## 五、文本叙事保真：代词与 emote

### §P1 代词是 10 变量，不是 4

- **问题**：v1 把 message_vision 代词替换当作纯渲染函数，默认 4 视角。
- **影响**：源码实测代词是 10 个变量（`$N/$n/$P/$p/$C/$c/$R/$r/$S/$s`），其中 6 个由 RANK_D 的 7 个动态业务函数求值，依赖年龄/性别/职业/门派/官职/武功等级/善恶/鬼魂状态。这是业务逻辑求值，不是纯渲染函数。
- **修正**：
  - 服务端必须为每条事件的 me/you 预先求值 RANK_D 7 函数，产出结构化 `PronounContext{name, pronoun, respect, rude, self, self_rude, close, self_close}` 作为事件 payload 下发。
  - 前端只做 `$X -> context[X]` 的纯字符串替换。
  - RANK_D 7 函数单独列为"**称谓系统迁移**"子任务，规则进 ThemeRegistry。

### §P2 emote 是 7 视角而非 3

- **问题**：v1 按 3 视角设计渲染层。
- **影响**：源码实测 emote 是 7 视角变体（`myself/others/target/myself_target/others_target/myself_self/others_self`），且每条带 `updated` 作者署名（20+ 位 wizard）。
- **修正**：
  - emote 数据保留全 7 视角 + 10 代词变量，不降级。
  - `updated` 作者署名作为 legacy provenance 回填到 CPK（`author.type='legacy_wizard'`）。

### §P3 message 渲染门控是状态依赖

- **问题**：v1 未显式处理 outdoor 子类消息可见性、blind condition 随机丢弃消息。
- **修正**：服务端在事件 payload 附 RenderGating 上下文；blind 的随机丢弃由服务端用 CombatRNG 预计算，保证三端一致。

## 七、技能三层与战斗引擎

### §K1 门派灵魂 70%+ 在脚本不在数据

- **问题**：v1 假设"技能=行为定义非数据"与"数据驱动 *action 迁移 YAML"同时成立，自相矛盾。
- **影响**：以独孤九剑为例：`query_action` 有玄铁重剑隐藏大招（6 条件触发），`hit_by` 有 4 级递进反击，`valid_learn/valid_enable` 有门派绑定，`feng.c` 的 `auto_perform` 按对手武器类型切换破式。**YAML skill 表只能覆盖约 30% 技能行为**。
- **修正**：
  - 明确技能三层：
    - **SkillData**（YAML action 数组）
    - **SkillBehavior**（Python 策略对象，隐藏大招 / hit_by / valid_learn / auto_perform）
    - **SkillEffect**（Effect 组件 perform 状态机）
  - UGC 只编辑层 1，层 2 由平台级可信作者维护；**坦白承认门派特色是平台资产非 UGC 可创作内容**。
  - `skill_power` 建模为三段式可组合公式管线：
    1. 基础幂（level³/3 可配）；
    2. 状态修饰链（jingli_bonus 动态 / fight-dodge 累积态 / apply 修饰 / busy 减半）；
    3. 属性系数与封顶。
  - fight/dodge 累积态单独建模为 CombatState 瞬态字段，不进 YAML。
  - 战斗七步管线的副作用时序必须显式保持，建模"管线副作用账本"，按 step 顺序严格 apply。

### §K2 CombatKernel 抽象

- **问题**：v1 把 do_attack 七步当作武侠特色硬编码。
- **影响**：航海海战/书院比试/现代格斗本质同构，需复用战斗骨架。
- **修正**：提取抽象 `CombatAction` / `CombatResolver` / `DamageModel` 接口，武侠七步是其具体实现。

### §K3 战斗手感验证维度缺失

- **问题**：v1 所有专家谈确定性/handoff/combat-sim/性能，但没人问"新引擎战斗玩家打起来感觉对不对"。
- **修正**：建"**文本体验流 diff**"——录制 message_vision 输出顺序 + 代词替换，前端渲染后做阅读体验 diff。

### §K4 NewRandom 加权随机必须精确复刻

- **问题**：v1 未显式处理自定义偏置函数。
- **修正**：`NewRandom(i, 20, level/5)` 是 base=20、d=level/5 的加权随机；DSL 原语 `weighted_random` 必须精确复刻。

## 八、引擎工具链

### §T1 引擎工具链完全缺失

- **问题**：v1 仅有 Prometheus metrics + OTel traces（运维可观测，不是引擎可观测）。没有场景编辑器 / 实体检视器 / tick 时间线回放器 / ECS 调度图可视化 / AI 行为树调试器 / 性能 profiler。
- **影响**：没有 inspector 和 replay scrubber，调试 8400 文件迁移的行为偏差只能靠 printf。
- **修正**：引擎调试工具链作为**阶段 0 一等公民**，至少包含：
  - **entity inspector**：WebSocket 实时查任意实体组件树。
  - **tick profiler**：per-system 耗时火焰图。
  - **combat replay viewer**：CombatContext 快照可视化回放。
  - **world graph visualizer**：房间出口图 + 实时实体分布。

## 九、Python 性能与 compiled core

### §PY1 Python 性能天花板必须正面回应

- **问题**：v1 未正面回应纯 Python 3.12 的三重天花板：GIL、GC、PYTHONHASHSEED 随机化。
- **影响**：
  - GIL：tick 计算期间全网 I/O 冻结（架构天花板）。
  - GC：每 tick 千级短命对象触发 5-20ms 不可控暂停，破坏 tick 确定性。
  - PYTHONHASHSEED：set 迭代跨进程不稳定，CombatContext+seeded RNG 确定性回放有隐性分叉风险。
- **修正**：
  - 阶段 0 必须做：
    - (a) 单 do_attack μs 级 micro-benchmark + tick 预算定义（50ms/20Hz 或 100ms/10Hz）。
    - (b) 固定 `PYTHONHASHSEED=0` + 禁用 set 迭代 + 序列化 key 排序。
    - (c) GC 基准测试。
  - 按基准结果 go/no-go compiled core。
  - 架构师裁定采用**分阶段 compiled core**：阶段 0 纯 Python 原型 + 单 do_attack μs 级基准 + tick 预算拆解 → 按基准结果决定热路径是否用 Cython（比 Rust 更平滑）→ 仅当 Cython 不够才引入 Rust（仅核心热路径）。
  - 不引入 compiled core 则诚实承认 Python 是原型验证语言而非标杆引擎实现语言。Rust 核心最终决策延后到有实测数据后。

### §PY2 性能达标标准必须定义

- **问题**：v1 未定义 tick 预算含/不含 I/O、单房间容量 p99 阈值。
- **修正**：阶段 0 输出明确的性能达标标准，否则无法 go/no-go compiled core。

## 十、标杆差异化能力

### §B1 标杆定位不是"最现代的 MUD"而是"最 MUD 的现代游戏"

- **问题**：v1 偏重现代工程改造，有抽空文化灵魂换现代工程皮的倾向。
- **修正**：保留侠客行玩法设计灵魂，用现代技术放大其表现力与触达，同时加入 LPC 时代做不到的设计可能。

### §B2 标杆资产拆为独立可交付里程碑

- **M1：确定性回放引擎开源**（3 月）——全行业最缺的反作弊/数值审计基础设施。
- **M2：垂直切片证明 DSL+Agent 创作闭环**（2-3 月）——核心创新点的最小验证。
- **M3：单题材（武侠）完整可玩 demo**（6-8 月）——M3 之后才投入全量迁移与多题材。

### §B3 Agent NPC 是研究问题不是 feature

- **问题**：v1 把 Agent NPC 当作普通功能排期。
- **影响**：LLM 推理延迟（500ms-3s）与 tick 仿真（1s 周期）根本冲突；游戏内 LLM NPC 是 research problem。
- **修正**：立项但严格分层：
  - (a) tick 仿真内 NPC AI 用确定性行为树/状态机（快、确定、可回放）。
  - (b) 实时深度对话走独立"对话服务"（不在 tick 内），结果异步注入。
  - (c) **LLM 出对话文本，DSL 状态机 guardrail 守所有状态变更**——NPC 幻觉不能 mutate 游戏世界（不能刷金币/破坏任务逻辑）。
  - 先判定 tractable 再排期，不能当 feature 轻松加。

## 十一、开放问题与盲区

这些问题在专家交叉质证中涌现，暂无定论，阶段 0-1 必须给出数据或决策：

1. compiled core 引入时机与边界：μs 基准达到什么阈值触发？Cython 还是 Rust？PyO3 边界开销实测多少？
2. do_attack 同步递归调用图在分布式下的处理：分片内同步是否覆盖所有递归？跨房间追杀/组队跨区域 PK 的 CombatContext 如何在分片间传递？

4. 规格提取形式化程度：property-based testing 能覆盖多少 condition 叠加/effect stacking 组合空间？
5. StdLib 与 UGC 执行路径隔离如何在 ECS 层实现：同 World 不同 System 权限，还是不同 World 实例？
6. 热重载安全窗口精确定义：condition 堆叠 / call_out 定时器 / AOI 订阅在热重载期间如何处理？
7. in-game LLM NPC 的"LLM 出文本/DSL 守状态"硬隔离是否足以阻止所有状态泄露？
8. 世界圣经 schema 如何既作 Agent 共享上下文又作可验证规格？
9. 战斗手感等价性如何验证：文本体验流 diff 的粒度与可接受偏差阈值？
10. 经济系统从 200 人到千人在线如何重新平衡？分布式下"分区货币总量+跨区贸易汇兑"具体设计？
