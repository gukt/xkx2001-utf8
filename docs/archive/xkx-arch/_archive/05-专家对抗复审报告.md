### 共识 2：MVP 顺序反了--最高产品风险最后验证

制作人核心论断，AI 专家、主策划、架构师强力支持：

核心价值主张是"**DSL+Agent 让非程序员创作可玩世界**"，但这个假设要到阶段 4-5（14+ 月后）才被验证。前 14+ 月全在验证工程可行性（能否重建引擎）。**这是把最高风险的产品假设放到最后验证，顺序完全反了**。若 DSL+Agent 创作闭环在工程或产品上不成立，14 月引擎投入全部沉没。

**裁决修正**：在阶段 0 前插入"**阶段 -1：垂直切片平台验证**"（2-3 月）--5-10 房间 + 2 NPC + 1 战斗 + 1 任务 + 1 对话，全部 DSL 定义 + Agent 部分生成，验证三件事：DSL 能否表达、Agent 产出是否可用、玩家是否觉得好玩。**通过后才投入引擎重构**。

> ⚖️ **分歧裁决（stub 运行时 vs 真实引擎最小子集）**：制作人建议用"最小 stub 运行时"；架构师与 AI 专家修正为"**真实引擎最小子集**"--提取 `resolve_attack` 为无副作用纯函数（不依赖完整 ECS 运行时但保证验证与未来真实引擎同源）。理由：stub 语义与真实引擎不一致会制造 false confidence（验证信号失真比没验证更危险）。纯函数提取是 Combat 迁移阶段 2.4 的前置工作，提前做不浪费。

### 共识 3：Ray 是 ML 框架非游戏服务器框架

架构师 + 分布式专家 + 引擎专家三方源码验证：

`do_attack` 在 `combatd.c` line 772/777/804/812/823/831 递归调用自身（riposte/double_attack/follow_up），分支由 `random()` 决定。**Ray actor 默认 FIFO 非重入，actor A 调 B、B 回调 A 会死锁**；`ray.get()` 阻塞事件循环使分布式化为零。方案零处论证 68771 处同步调用如何映射到异步而不破坏调用栈语义与递归回溯。

**裁决修正**：
- 阶段 1 用**纯 asyncio + 普通实体对象**（同步方法调用，与 LPC `call_other` 语义一致），**完全不引入 Ray**
- **分片内同步、分片间异步**：同房间/同区域实体 colocate 到同一进程，`call_other` 退化为进程内同步函数调用（保持 68771 处调用栈语义），仅在跨分片边界用异步消息
- Ray 推迟到阶段 3 真正需要分布式**计算**时引入，且仅用于无状态/可并行计算（Agent 编排、combat-sim 批量、离线迁移工具），**不做游戏实体运行时**
- greenfield 下"自研薄壳"选项彻底剔除

### 共识 4：差分测试 P0 门禁有未声明硬依赖

制作人 + 引擎专家实测发现：

当前仓库的 `./driver` 是 **Mach-O（macOS）二进制**，目标平台是 Linux（WSL2）。要录制 golden trace 必须先在 Linux 上重新编译 FluffOS 驱动并让 XKX 可启动。README 也承认 `securityd.c` 不完整导致巫师命令不可用，**系统当前并非完整可运行状态**。"让 25 年前的 MudOS 在现代 Linux 跑起来"本身可能是数周级工程，却被当作"阶段 0 的 1-2 月里顺手做掉"。

**裁决修正**：
- 阶段 0 第一步显式加入"**FluffOS Linux 编译 + XKX 可运行性验证**"前置任务（保守估 2-4 周）
- 备选方案：在 Mac 上一次性录制 golden trace（不需持续运行）；或降级为"**单元级行为规约**"（从代码人工推导期望行为写断言，不依赖运行时）
- **双轨制**：(A) 单元级行为规约先行（从代码提取 go/move/combat 输入输出契约，不依赖运行 LPC，2-3 周可覆盖核心路径，是 greenfield 主门禁）；(B) 对可运行 FluffOS 录制一次性 golden trace 做统计回归（伤害分布/频率，非逐位对齐）

### 共识 5：代词是 10 变量非 4，emote 是 7 视角非 3

MUD 历史专家源码验证，架构师、主策划、AI 专家、引擎专家一致同意：

- emote 实测是 **7 视角变体**（`myself/others/target/myself_target/others_target/myself_self/others_self`），且每条带 `updated` 作者署名（20+ 位 wizard）。
- "代词替换纯函数下移前端"的承重论断（[00-重构总纲.md](00-重构总纲.md) §4.8、[02-目标架构与子系统设计.md](02-目标架构与子系统设计.md) 子系统 9）**在代词维度不成立**。

**裁决修正**：
- 服务端必须为每条事件的 me/you 预先求值 RANK_D 7 函数，产出结构化 `PronounContext{name, pronoun, respect, rude, self, self_rude, close, self_close}` 作为事件 payload 下发
- 前端只做 `$X -> context[X]` 的纯字符串替换
- RANK_D 7 函数单独列为"**称谓系统迁移**"子任务，规则进 ThemeRegistry
- emote 数据保留全 7 视角 + 10 代词变量，不降级
- `updated` 作者署名作为 legacy provenance 回填到 CPK（`author.type='legacy_wizard'`）

### 共识 7：门派灵魂 70%+ 在脚本不在数据

主策划源码验证，架构师、AI 专家、引擎专家一致支持：

以独孤九剑 `dugu-jiujian.c` 为例：
- `query_action` 有玄铁重剑隐藏大招（`str>=30 && int>=36 && random(skill)>80 && force>150 && neili>400 && 武器特定`，6 条件触发，返回 damage:200 特殊招）
- `hit_by` 有 4 级递进反击（连续 random，每级概率门槛递增，直接 mutate neili/jingli/busy）
- `valid_learn/valid_enable` 有门派绑定（`master_id=='feng qingyang'` 且未学过 12 种其他剑法）
- `feng.c` 的 `auto_perform` 按对手武器类型切换破式

**YAML skill 表只能覆盖约 30% 的技能行为**。方案"技能=行为定义非数据"与"数据驱动 mapping *action 迁移 YAML"自相矛盾。

`skill_power` 也不是"可配置表达式"而是**状态依赖的累积函数**（`combatd.c` line 288-333）：level³/3 仅一部分，叠加 jingli_bonus（实时动态 50-150）、apply/attack、apply/defense、**fight/dodge 累积态**（line 305 `level*((100+fight/dodge/10)/100)`，每次命中重置）、str/dex 系数封顶 300。

**裁决修正**：
- 明确技能三层：**SkillData**（YAML action 数组）+ **SkillBehavior**（Python 策略对象，隐藏大招/hit_by/valid_learn/auto_perform）+ **SkillEffect**（Effect 组件 perform 状态机）
- UGC 只编辑层 1，层 2 由平台级可信作者维护，**坦白承认门派特色是平台资产非 UGC 可创作内容**
- `skill_power` 建模为三段式可组合公式管线：(1) 基础幂（level³/3 可配）；(2) 状态修饰链（jingli_bonus 动态/fight-dodge 累积态/apply 修饰/busy 减半）；(3) 属性系数与封顶。fight/dodge 累积态单独建模为 CombatState 瞬态字段，不进 YAML
- 战斗七步管线的**副作用时序必须显式保持**（do_attack 命中/未命中/招架/伤害四分支各自 mutate，且有分支内时序依赖）--显式建模"管线副作用账本"，按 step 顺序严格 apply

### 共识 8：引擎工具链完全缺失

架构师 + 引擎专家 + 主策划一致：

标杆引擎（Unity/Unreal/Bevy/Godot）与玩具的分水岭是工具链。方案仅有 Prometheus metrics + OTel traces（**运维可观测，不是引擎可观测**）。完全没有：场景编辑器 / 实体检视器 / tick 时间线回放器 / ECS 调度图可视化 / AI 行为树调试器 / 性能 profiler。没有 inspector 和 replay scrubber，调试 8400 文件迁移的行为偏差只能靠 printf。

**裁决修正**：引擎调试工具链作为**阶段 0 一等公民**而非 backlog，至少：entity inspector（WebSocket 实时查任意实体组件树）、tick profiler（per-system 耗时火焰图）、combat replay viewer（CombatContext 快照可视化回放）、world graph visualizer（房间出口图 + 实时实体分布）。

### 共识 9：Python 性能天花板未被正面回应

引擎专家 + 架构师 + 分布式专家三方论证：

纯 Python 3.12 存在三重天花板：
1. **GIL**：tick 计算期间全网 I/O 冻结（不是性能优化问题，是架构天花板）
2. **GC**：每 tick 千级短命对象触发 5-20ms 不可控暂停，破坏 tick 确定性
3. **PYTHONHASHSEED 随机化**：set 迭代跨进程不稳定（实测两次运行 `list(s)[:5]` 完全不同），CombatContext+seeded RNG 确定性回放有隐性分叉风险

> ⚖️ **分歧裁决（语言选型与时机）**：
> - **引擎专家**主张阶段 0 严肃评估 **Rust 核心（PyO3）+ Python 脚本**混合架构（标杆 vs 玩具的分水岭决策）
> - **架构师**修正为**分阶段 compiled core**：阶段 0 纯 Python 原型 + 单 do_attack μs 级基准 + tick 预算拆解 → 按基准结果决定热路径是否用 Cython（比 Rust 更平滑）→ 仅当 Cython 不够才引入 Rust（仅核心热路径）。理由：单人/小团队引入双语言栈是认知负荷灾难（构建/调试/类型边界/CI 全部翻倍），PyO3 跨语言边界开销在 do_attack 20-30 次组件查询场景下可能抵消 Rust 优势
> - **主策划**认为语言是实现细节不是标杆分水岭--"一个 Rust 核心如果把 skill_power 简化为 level³/3，比一个忠实实现三段式的 Python 核心更糟糕"
>
> **最终裁决**：采纳架构师的分阶段方案。**阶段 0 必须做**：(a) 单 do_attack μs 级 micro-benchmark + tick 预算定义（50ms/20Hz 或 100ms/10Hz）；(b) 固定 `PYTHONHASHSEED=0` + 禁用 set 迭代 + 序列化 key 排序；(c) GC 基准测试。按基准结果 go/no-go compiled core。**不引入 compiled core 则诚实承认 Python 是原型验证语言而非标杆引擎实现语言**。Rust 核心的最终决策延后到有实测数据后。

### 共识 10：combat-sim 双重标准

AI 专家提出，分布式、架构师、主策划、引擎专家一致支持：

combat-sim 用"独立纯 Python 数值模型从 LPC seed"，但方案自身要求 LPC→Python 迁移用 golden trace 差分测试保证行为等价。**验证信号失真比没验证更危险**。独立模型若简化了 riposte 递归、fight/dodge 累积态、condition 位聚合、Effect stacking，验证通过的内容上线后可能失衡。

**裁决修正**：combat-sim **直接调用从 LPC 提取的 `resolve_attack` 纯函数**（输入组件快照 + seed，输出 CombatRoundResult），不依赖完整 ECS 运行时但保证验证与未来真实引擎同源。这是 Combat 迁移阶段 2.4 的前置工作，提前做不浪费。greenfield 重写的 Python 代码本就用 seeded RNG，§10 的"放弃位等价"只适用于"与原 LPC 逐位对照验证"，不应延伸为"运行时放弃确定性"。

### 共识 11："可验证终态"是验证覆盖度幻觉

AI 专家核心论断，UGC 专家、架构师支持：

DSL schema 只收敛"结构合法性"，world-graph 只收敛"图可达性"，combat-sim 只收敛"数值分布"。但 UGC 内容质量的承重维度--**叙事连贯性、对话语义死锁、任务绕过、经济平衡、趣味性**--全部在验证覆盖外。方案用"可验证终态"暗示"内容质量已验证"，实际只验证了"结构合法 + 数值在分布内"。

**裁决修正**：
- 显式分层声明验证覆盖度：**可机器自动验证**（结构/可达性/数值分布/不变量回归）vs **需半自动或人工**（叙事连贯=LLM-as-judge、对话死锁=flag 状态机模型检查 nuXmv/SPIN、任务绕过=状态空间探索、经济平衡=多 agent 玩家行为仿真、趣味性=playtest agent + 人工抽样）
- 对后者逐项声明 gap 与目标覆盖度，不掩盖
- 五角色 Worker 降级为"**能力清单**"而非"固定进程"：单 Orchestrator 持世界圣经摘要（RAG 检索解决 200k+ token 超窗口）按需唤起专项 worker。Balancer 角色对战斗域几乎无产出能力（skill_power 是平台级 Python）、Continuity 应改"生成时约束"（structured output + 实体关系约束注入）替代事后审查

### 共识 12：handoff partial-failure 是承诺非规约

分布式专家 + 架构师：

文档只写"freeze→snapshot→activate→replay→deactivate，每步定义超时/补偿/最终一致目标"，但**没有任何失败语义定义**：snapshot 写 Redis 附 TTL，过期且激活未完成=实体永久丢失；战斗中 handoff，freeze 期间对端分片的攻击回合如何处理？

**裁决修正**：采用**两阶段提交 + outbox 模式**--源分片将 handoff 作为持久化事务写入 outbox（PG），目标分片从 outbox 拉取并确认，确认后源分片才删除旧实体。handoff 期间实体对外"只读冻结"（写入命令排队）。组队批量 handoff 用一个 outbox 事务包裹全队，原子提交。每步的幂等性、重试策略、超时降级必须写成可执行协议规约。

### 共识 13：单房间串行是分布式无法突破的硬约束

分布式专家 + 架构师 + 引擎专家：

MUD 玩家高度聚集（主城/PK 场可能聚集超单分片容量），房间是单 CombatCoordinator 串行裁决，分布式化无法解决。§C 的">30 人拆子 Coordinator"与 3000 人聚集差两个数量级。

> ⚠️ **分歧裁决（房间实例化分桶）**：
> - **分布式专家**主张房间级水平扩展（多实例 + 玩家分桶，按 player_id hash 分配）
> - **主策划强烈反对**：MUD 核心社交魅力是"全服同房"--主城广场喊话全服可见、PK 围观者实时看到战斗、武林大会千人同场。实例化分桶会摧毁这个体验，把 MUD 变成普通图形 MMO 的"频道制"
>
> **最终裁决**：采纳主策划的红线。**不用实例化分桶**。改为：(a) 务实下调目标为"**万人注册 + 千人同时在线**"（MUD 的合理量级）；(b) 热点房间用"**专用分片**"（整个分片只服务一个主城房间）；(c) 房间内并行化战斗裁决（按战斗组分而非玩家分桶）；(d) 明确"单房间串行是 MUD 固有硬约束，分布式无法突破"。

### 共识 14：全仿真确定性 + 逻辑热重载是标杆分水岭

架构师 + 引擎专家：

- **全仿真确定性**：方案只在战斗子域做了 CombatContext + seeded RNG，但标杆引擎应支持"任意 tick 快照 + 输入日志 = 完整重放"。这是时间旅行调试、反作弊、自动化测试的共享底座。
- **逻辑热重载**：LPC 的 `destruct()+clone()+call_other` 本质是热重载机制，MUD 运营高度依赖"巫师在线改代码即时生效"。方案只有数据热重载（DamageFormula/Checkpoint），缺 Python System 代码热重载。

**裁决修正**：
- 所有 System 的 RNG 走 seeded DeterministicRNG（不仅战斗），记录所有外部输入为 input log
- 设计 System 热重载管线（`importlib.reload` + entity state migration + schema migration），定义"热重载安全窗口"（无进行中战斗、无跨分片 handoff）

## 三、业界标杆的真正差异化能力（专家共识）

8 位专家虽视角不同，但收敛到几个核心差异化点。**这些才是"业界标杆"的真正护城河，而非"把 LPC 迁到 Python"**：

### 标杆定位重定义
- **主策划**：标杆应定位于"**最 MUD 的现代游戏**"--保留侠客行玩法设计灵魂用现代技术放大其表现力与触达，同时加入 LPC 时代做不到的设计可能。不是"最现代的 MUD"（抽空文化灵魂换现代工程皮）
- **制作人**：标杆资产拆为独立可交付里程碑（M1 确定性回放引擎开源 / M2 DSL+Agent 创作闭环 demo / M3 单题材可玩 demo）
- **AI 专家**：业界标杆 = "首个把确定性世界仿真（combat/经济）与 LLM 非确定性创作用验证契约深度耦合、且内置创作者经济与版权链的 UGC 平台"
- **引擎专家**：Rust 确定性仿真核心 + Python 创作层 + 完整工具链 + UGC/Agent 协作（这个组合业界无人做到）

### 五大差异化能力
1. **DSL-as-contract 的 Agent 协作创作闭环**（架构师/制作人/UGC/AI 共识）--schema 强约束把 LLM 非确定性收敛到可验证终态，Roblox/Core/Rec Room 都没有。但必须补齐"语义/叙事/平衡"三维评估矩阵
2. **全仿真确定性 + 时间旅行调试**（架构师/引擎/分布式）--任意 tick 快照 + input log = 完整重放，反作弊/调试/自动化测试共享底座
3. **逻辑热重载不停服**（架构师/引擎）--MUD 引擎 vs 普通 WebSocket 服务器的核心差异
4. **themed 治理范式 + intermud 联邦演进**（MUD 历史专家）--天雷是 themed 反作弊、阴间是 themed 死亡惩罚、法院是 themed 反外挂、vote 是 themed 自治。这是图形 MMO 无法复制的"想象力治理元宇宙"
5. **Agent NPC 作为游戏内实体**（主策划提出，最大但最难）--LPC 做不到的动态对话/涌现叙事

> ⚠️ **分歧裁决（Agent NPC）**：
> - **主策划**主张立项作为革命性体验（LPC 做不到、现在能做、无人做成的核心卖点）
> - **架构师/引擎专家**挑战：LLM 推理延迟（500ms-3s）与 tick 仿真（1s 周期）根本冲突，不能走"与玩家相同的 ECS tick 驱动"
> - **AI 专家**深化：游戏内 LLM NPC 是 research problem 不是 feature--离线创作是温和问题（批处理、可逐项审阅），游戏内 NPC 是实时、不可逐句审阅、直接影响世界状态的研究级问题
>
> **最终裁决**：立项但**严格分层**--(a) tick 仿真内 NPC AI 用确定性行为树/状态机（快、确定、可回放）；(b) 实时深度对话走独立"对话服务"（不在 tick 内），结果异步注入；(c) **LLM 出对话文本，DSL 状态机 guardrail 守所有状态变更**的硬隔离--NPC 幻觉不能 mutate 游戏世界（不能刷金币/破坏任务逻辑）。这是 research problem，先判定 tractable 再排期，不能当 feature 轻松加。

### 标杆资产拆分独立交付（制作人）
不要把标杆能力埋在 14+ 月的大路线图里。拆为独立可交付物早获社区反馈：
- **M1：确定性回放引擎开源**（3 月）--全行业最缺的反作弊/数值审计基础设施，最低成本拿到的标杆资产
- **M2：垂直切片证明 DSL+Agent 创作闭环**（2-3 月）--核心创新点的最小验证
- **M3：单题材（武侠）完整可玩 demo**（6-8 月）--M3 之后才投入全量迁移与多题材

## 五、各专家专项发现

### 游戏主策划专项：玩法设计灵魂
- **战斗手感验证维度缺失**：所有专家谈确定性/handoff/combat-sim/性能，但没人问"新引擎战斗玩家打起来感觉对不对"。需建"文本体验流 diff"（录制 message_vision 输出顺序 + 代词替换，前端渲染后做阅读体验 diff）
- **经济系统规模化重新设计**：moneyd MAX_CASHFLOW_ALLOWED=400000 按 200 人调参的通缩模型，放大到目标千人在线需完全重新设计货币产出/回收/通胀。分布式下用"分区货币总量 + 跨区贸易汇兑"替代 children() 全库扫描。**独立游戏设计专项非迁移子任务**
- **death_count 师徒版本号**：`feng.c` 用 death_count 作为师徒关系失效信号（check_student 检测变化则逐出师门）。这是用死亡次数作关系版本号，"可声明策略"无法表达。建 MentorshipRelation 状态机，bond_version 由可配置失效信号驱动
- **NewRandom(i,20,level/5) 加权随机**：自定义偏置函数非 uniform random，weighted_random DSL 原语要精确复刻（base=20, d=level/5）

### MUD 历史专家专项：文化保真
- **condition 拆三类**：CombatCondition（中毒/流血）+ SocialCondition（killer 通缉/vote_suspension 剥权/city_jail 监禁）+ LifecycleCondition（pregnant/坐禅）。治理 condition 的"期满解除并广播"语义需单独建模
- **天雷是 themed 反作弊范式**：保留 message_vision 文学化文本 + unconscious + 死因标记"被天雷劈死了"。抽象为 PunishmentPolicy，UGC 作者能定义自己题材的 themed 惩罚（航海=海怪袭击、书院=戒尺责罚）。这是"想象力治理"
- **message 渲染门控是状态依赖**：outdoor 子类消息只在户外房间可见、blind condition 随机丢弃消息。服务端在事件 payload 附 RenderGating 上下文，blind 的随机丢弃由服务端用 CombatRNG 预计算保证三端一致
- **day_phase 不只是广播**：event_sunrise 自动保存所有玩家、event_common 遍历 livings() 检查位置清理无环境对象。拆为世界时钟服务 + 时段事件钩子 + 表现层广播

### UGC 平台专家专项：平台能力（对标 Roblox 成熟度约 35%）
- **可视化创作工具是第一护城河**：Roblox Studio/Core Creator/Rec Room Maker 全部以可视化编辑器为创作入口。四层 DSL 对非程序员门槛远超 Lua+Studio
  - ⚖️ **裁决**：UGC 专家主张 UGC 阶段首要交付；架构师修正为引擎定型后做（编辑器操作的数据模型依赖 ECS schema 稳定）。**最终**：先稳定数据模型（阶段 0-1），再做可视化编辑器（阶段 2+），DSL 文本格式作过渡。编辑器作为引擎工具链一部分
- **创作者经济模型完全缺失**：即使 MVP 不实现支付，Day1 在 CPK manifest 预留 author_id/revenue_share/price 字段。阶段 4 交付内容市场 MVP（浏览/搜索/安装/评分）
- **内容审核严重不足**：实测 832 文件含暴力描述、122 文件含政治敏感词、28 文件含赌博、71 文件含金庸版权角色。分层审核 pipeline：自动化预检 + 社区众审 + 专家审核 + 平台终审。加内容分级 + 举报下架申诉
- **版权法律框架空白**：金庸衍生内容需授权或改编化（角色改名/门派虚构化）或标注同人非商用；AI 生成内容版权政策；provenance 扩展为版权链
- **内容发现与分发缺失**：CPK manifest 扩展市场元数据（title/description/tags/screenshots）

### 分布式专家专项
- **PG 唯一权威 + Redis 读穿透**：greenfield 红利，直接选，不犹豫
- **客户端延迟隐藏缺失**：移动命令乐观执行（前端本地渲染 + client_seq 上报，服务端确认回 ack，不一致发 rollback delta）。业界标杆 UX 基线
- **路由模型规约**：一致性哈希只解析"分片当前 owner"，placement 是 owner 的可变覆盖，迁移中实体路由到确定"权威分片"（旧或新二选一）不接受双写

### 引擎专家专项
- **environment() 统计争议**：引擎专家称 3777 处（§40 错误）；架构师核实为两种计数（独立调用 654 vs 字符串出现 3777），§40 核心论点"以箭头调用面为主线指标"仍成立。**裁决**：文档应明确 664 指"独立 environment() 调用"
- **闭包迁移面统计不全**：call_out 闭包 147 处，但全库 `(: ... :)` 闭包语法共 1979 处/897 文件。补全闭包全量分类台账（call_out/dbase/perform/event_handler 四类）
- **范式收敛**：ECS（仿真核心）+ DSL（创作契约）两范式。EntityRef 是 ECS 实体寻址层（call_other 等价物，点对点消息），ECS query 管批量遍历，两种通信模式不可互替。Agent 是 DSL 生产端消费者（离线工具），不是运行时范式
- **CombatKernel 抽象**：do_attack 七步不是"武侠特色"而是"回合制 MUD 战斗通用骨架"。提取抽象 CombatAction/CombatResolver/DamageModel 接口，武侠七步是其具体实现。航海海战/书院比试/现代格斗本质同构

### AI 专家专项：Agent 架构重设
- **离线创作 vs 游戏内 NPC 必须拆为两个非确定性管理问题**（见上文分歧裁决）
- **combat-sim 调真实 resolve_attack 纯函数**（见共识 10）
- **五角色降级为能力清单**（见共识 11）
- **LangGraph 仅过渡原型**：中期迁 Temporal/DBOS durable execution 承载创作工作流的中断-恢复-人类介入
  - ⚖️ **裁决**：AI 专家主张中期 Temporal；架构师修正为 PG-backed state machine 先满足 MVP，Temporal 推迟到多创作者并发需求出现（阶段 3+）。**最终**：MVP 用 PG-backed state machine，明确标注 LangGraph 非终态
- **LLM 成本模型缺失**：五角色 × 多轮 × 世界圣经上下文，单 CPK 可能 75 万-数百万 tokens（$2-40）。建立成本模型 + 分层模型策略（编排/审查用强模型，生成/校验用经济模型）+ 创作者配额
- **世界圣经 schema 与自洽性验证**：WorldBible 是所有 Worker 共享上下文，其错误级联。定义门派关系图/时间线/能力体系的结构化 schema 与自洽规则（如"互敌门派不能有师承关系"），用"生成时约束注入"取代事后 Continuity 审查

## 六、专家遗漏的盲区（交叉质证涌现）

1. **战斗手感验证维度**--所有技术专家无人问"新引擎战斗玩家打起来感觉对不对"
2. **游戏设计标杆维度系统性缺席**--除 Agent NPC 外，无人提出"LPC 做不到但现在能做"的设计创新
3. **经济系统规模化重新设计**--只有主策划提及无人响应
4. **门派师徒关系状态机建模**--只有主策划提及
5. **greenfield 最大风险是"过度设计"**--没有运行中系统约束，容易在无性能基准/容量验证下堆砌四范式。应从最小可行架构起步（单进程 asyncio + 普通对象 + 纯函数 resolve_attack），按实证瓶颈逐层引入复杂度
6. **StdLib 与 UGC 执行路径隔离**--StdLib 是可信 Python（无沙箱限制），UGC 是沙箱受限。方案的 CPK 只区分内容包层面没区分代码执行路径
7. **规格提取的形式化程度**--轻量级（JSON schema + property-based testing）vs 重量级（TLA+/Alloy + 模型检查）。重量级对单人项目不可持续
8. **性能达标标准缺失**--tick 预算含/不含 I/O？单房间容量 p99 阈值？无标准无法 go/no-go

## 七、开放问题清单

1. compiled core 引入时机与边界：μs 基准达到什么阈值触发？Cython（平滑）还是 Rust（彻底）？PyO3 边界开销实测多少？
2. do_attack 同步递归调用图在分布式下的处理：分片内同步是否覆盖所有递归？跨房间追杀/组队跨区域 PK 的 CombatContext 如何在分片间传递？
3. 代词 10 变量求值的服务端性能影响：3000 并发每 tick 数百条事件，RANK_D 7 函数是否成热路径？需预计算+缓存？
4. 规格提取形式化程度：property-based testing 能覆盖多少 condition 叠加/effect stacking 组合空间？
5. StdLib 与 UGC 执行路径隔离如何在 ECS 层实现：同 World 不同 System 权限，还是不同 World 实例？
6. 热重载安全窗口精确定义：condition 堆叠/call_out 定时器/AOI 订阅在热重载期间如何处理？旧 System in-flight 事件如何迁移到新 System？
7. in-game LLM NPC 的"LLM 出文本/DSL 守状态"硬隔离是否足以阻止所有状态泄露？需先判定 tractable 再排期
8. 世界圣经 schema 如何既作 Agent 共享上下文又作可验证规格？
9. 战斗手感等价性如何验证：文本体验流 diff 的粒度（逐行 vs 语义等价）与可接受偏差阈值？
10. 经济系统从 200 人到千人在线如何重新平衡？分布式下"分区货币总量+跨区贸易汇兑"具体设计？
