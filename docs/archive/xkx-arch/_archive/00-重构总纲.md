# 侠客行 MUD 现代化重构总纲

> 本文档由研究 Agents Team（36 个 Agent，分现状分析 / 模式研究 / 方案设计 / 对抗验证四阶段）产出结论汇总而成。
> 对抗验证阶段对每个设计方案做了"默认怀疑"的审查，发现了多处承重级假设错误。本总纲**以修正后的假设为准**，凡修正点均以 ⚠️ 标注并指向 [01-关键修正与避坑清单](01-关键修正与避坑清单.md)。

---

## 1. 愿景与目标

将经典 LPC MUD《侠客行》（8412 个 LPC 文件、6414 房间、21 门派、22+ 守护进程、单进程 MudOS）重构为：

| 维度 | 目标 |
|---|---|
| 后端 | Python + WebSocket，asyncio + uvloop |
| 前端 | 现代 Web（React + TypeScript），文本为基线、图形为渐进增强 |
| 部署 | 分布式大规模，承载高并发同时在线（单分片实测后定上限，初版单进程起步） |
| 平台 | UGC 平台--用户借助 DSL 配置剧情场景 |
| 创作 | Agent 协作创作多题材世界（武侠 / 大航海时代 / 书院 / 穿越 / 现代剧情） |

### 三条不可妥协的边界
1. **网关无状态、纯协议层**：WSGateway 只做帧编解码 / 握手 / 限流 / 心跳 / 会话粘附路由，绝不持有游戏状态。
2. **会话与游戏实体解耦**：Session 提升为一等实体，断线只会话挂起（SUSPENDED），角色体留在世界继续被仿真。
3. **渲染下沉前端**：服务端只产语义事件，`message_vision` 代词体系 / ANSI 颜色 / `damage_msg` 文案全部下移前端纯函数渲染层。

## 2. 现状一句话总结

XKX 是"单进程对象世界 + 蓝图/克隆原型 + Feature/Mixin 组合 + 守护进程单例 + heart_beat 全局 tick + .o 平文件持久化"。这套体系在 90 年代是精巧的，但单进程单线程、`file_name` 即身份又即路径、`call_other` 同步阻塞、`object` 指针跨进程不可用，是分布式化的核心障碍。

> ⚠️ **承重修正 1**：现有实际并发上限是 `MAX_USERS=50`（`include/login.h` 已验证），而非 `config.xkx` 的 150 或 LPC 注释的 230。**不要为 50 用户的系统预先分布式化**。详见 [01-关键修正与避坑清单](01-关键修正与避坑清单.md) §1。

## 3. 总体目标架构（分层）

```

                               │

│  ThemeRegistry(多题材) │ ReviewPipeline(人机协作审批门)           │
└──────────────────────────────────────────────────────────────────┘
```

### 分层职责
- **网关层**：协议、认证、限流、会话路由。无状态水平扩展。
- **分片层**：有状态。Actor（Ray 起步）+ ECS（SparseSet）+ tick 仿真。按区域共同体分片。
- **状态层**：权威状态库（PostgreSQL）+ 热缓存（Redis）+ 消息扇出（NATS）。**非全量事件溯源**。
- **创作层**：DSL → CPK → 沙箱执行 → MCP 验证 → 修订闭环。Agent 编排与人类协作。

## 4. 核心架构决策（含修正）

### 4.1 对象模型：ECS 替换 inherit+feature
XKX 的 Feature 混入已是"组合优于继承"的雏形（F_DBASE 即数据袋、heart_beat 即系统循环、daemon 即无状态逻辑服务）。用自研 **SparseSet ECS** 替换编译期静态混入，实现组件运行时动态挂载/卸载（UGC 多题材必需）。

> ⚠️ **承重修正 2**：不要做 SparseSet + Archetype 双存储（互斥策略，徒增复杂度，5000 实体规模下 Archetype 收益微乎其微）。只用 SparseSet。详见 01 §2。
>
> ⚠️ **承重修正 3**：`query_entire_dbase()` 活引用直改模式（combatd.c 交叉读写 11+ 字段）破坏组件边界。迁移 CombatSystem 前必须先建字段→组件映射表，并补 `Progression` 组件（combat_exp/potential）。详见 01 §3。
>
> ⚠️ **承重修正 4**：`Effect` 必须是一等公民组件（可序列化、可中断、可跨分片迁移），替代 144 处闭包型 `call_out` 与 perform 多阶段效果。详见 01 §4。

**核心组件族**（按领域聚合）：Identity / Vitals(qi/jing/jingli+eff/max 三档) / Attributes / DerivedAttributes / ModifierStack / SkillSet / CombatState(enemy_id[]) / EquipmentSlots / Conditions / Position / Inventory / Wallet / TeamRelation / ActionState / DynamicProps(UGC 动态键) / **Effect**(持续效果) / **Progression**(战斗成长)。

**System 替代 daemon+heart_beat**：CombatSystem / VitalsSystem / HealSystem / ConditionSystem / AttributeSystem / SkillSystem / EquipmentSystem / MovementSystem / ActionSystem / FinanceSystem / NPCAISystem。非均匀 tick（高频系统每 tick，低频系统每 N tick）。

### 4.2 分布式：Actor 模型，Ray 起步而非自研
LPC 对象世界与 Actor 高度同构（dbase=封装状态、feature=行为、call_other=位置透明寻址雏形）。

> ⚠️ **承重修正 5**：**放弃"从零自研 asyncio 虚拟 Actor 运行时"**。这是多年级工程（placement/激活钝化/handoff 不丢消息/消息序/分布式死锁检测每一项都是数月级）。方案同时排斥 Akka/Orleans（非 Python），等于在 8400 文件迁移之外再背一个自研分布式框架债务。**阶段一用 Ray 的 actor 模型做最小可用验证**（Python 原生、自带 placement/调度/容错），证明 LPC↔Actor 同构映射在真实负载下成立后再决定是否自研薄壳。详见 01 §5。

**分片单元=区域共同体**：对 6414 房间出口图跑 Louvain 社区检测得 5-8 个共同体作为初始分片（边界对齐社区割边使跨分片移动最小化）。AOI 退化为"同房间实体集合 + 邻域静态元数据只读缓存"（MUD 是房间作用域，非 MMO 连续空间 AOI）。

> ⚠️ **承重修正 6**：热路径去 Redis 化。会话/实体路由用一致性哈希环（`shard=hash(entity_id)`）无外部依赖解析，Redis 仅作控制面（placement 变更通告、handoff 暂存）。Redis 降级时游戏降级为"仅本分片可玩"而非全停。详见 01 §6。
>
> ⚠️ **承重修正 7**：区域监督默认 **one-for-one + 持久化重建**，而非 all-for-one。多数区域状态实为 per-room/per-npc，无真正区域共享可变状态，根本不需要 all-for-one。详见 01 §7。

### 4.3 战斗引擎：保留七步管线，tick 驱动，确定性分层
保留 do_attack 七步管线（选技能→取招式→AP/DP 命中→PP 招架→伤害计算→施加伤害→经验/反击）作为 CombatSystem.resolve_attack() 纯函数管线。AP/(AP+DP) 概率模型原样保留。

> ⚠️ **承重修正 8**：**heart_beat 是战斗回合驱动器（feature/attack.c 注释明确），不是"可替代的周期清理"**。不可用"事件驱动替代全局心跳"。ConditionScheduler 必须同时支持事件触发与周期 tick 两种模式。auto_perform NPC AI 循环（128-189 处）建模为 ECS ContinuousBehaviorSystem。详见 01 §8。
>
> ⚠️ **承重修正 9**：战斗确定性**不能只靠 seeded RNG**。do_attack 持续读写参与者实时状态，仅记 seed 而不在战斗开始时快照全体参与者状态，回放会分叉。确定性边界明确为"战斗开始时对全体参与者+可观察环境做状态快照 = CombatContext"，seeded RNG 只是其一部分。详见 01 §9。
>
> ⚠️ **承重修正 10**：确定性回放**分层而非全量承诺**：新内容/UGC 强制走 CombatRNG 契约保证可回放；遗留内容采用"行为等价"标准（概率分布与伤害量级对齐，放弃逐 random 位对齐），用黄金战报做统计回归。详见 01 §9。
>
> ⚠️ **承重修正 11**：战斗热路径**绕过事件总线**。resolve_attack 同步返回 CombatRoundResult 并就地应用+通知同房间订阅者，仅在 tick 边界发一次聚合事件到总线供投影层。Vitals 运行时用裸 dataclass+断言，Pydantic 仅在 snapshot 落盘/UGC 上传/跨分片反序列化时校验。否则单分片容量不达标。详见 01 §10。
>
> ⚠️ **承重修正 12**：ConditionHandler.on_tick 返回值**不能是互斥枚举**。原 `CND_CONTINUE=1`/`CND_NO_HEAL_UP=2` 是可组合位标志（跨 condition 做 OR 聚合）。改为组合结构 `(continue, suppress_heal, expire)` + 显式聚合规则。详见 01 §11。

### 4.4 数据与状态：权威状态库 + 选择性审计，非全量事件溯源

> ⚠️ **承重修正 13**（最重要之一）：**不要把事件溯源(ES)当作所有状态变更的默认持久化模式**。代码库有 68771 处同步 `->` 调用、654 处 `environment()` 立即检查，典型模式 `ob->receive_damage(50); if(ob->query("qi")<0) ob->die();`。CQRS 最终一致意味着写后读需走"强一致读 EventStore"，即每次读都打 PostgreSQL--这是数据库瓶颈不是高并发。
> 降级为"权威状态库 + 选择性审计日志"：状态变更直接写 PostgreSQL entity_state 表（行级权威）+ Redis 热缓存；仅对高价值领域事件（死亡/交易/婚姻/升级/门派变更）写 append-only audit_events 表。ES 仅在需严格回放的子域（PvP 战报归档）单独应用。详见 01 §13。
>
> ⚠️ **承重修正 14**：必须明确 Redis 热态 vs PostgreSQL 权威之间的**写入顺序与一致性强度**。二选一：(a) Redis 仅读穿透缓存，写先入 PG 再回填；(b) Redis 为热态权威 + PG 为审计源，放弃"PG 事件可重建 Redis"承诺。无论哪种都要写明一致性强度与故障窗口数据丢失边界。
>
> ⚠️ **承重修正 15**：放弃误用的外部依赖--Neo4j 改 PG 关系表（关系类型<10 种）；APScheduler 与 Redis ZSET 二选一明确主调度机制（不双系统）；MMO 式 AOI 退化为房间作用域。

### 4.5 命令与权限：显式管线 + 能力令牌
command_hook god-function 拆为显式中间件管线（解码→限流→历史→别名→路由→授权→执行→审计）。`previous_object()` 隐式信任链显式化为 ActionContext + CapabilityToken。securityd 的目录级 ACL 升级为按 (subject, action, resource) 求值的能力策略引擎。

> ⚠️ **承重修正 16**：`call_out` 迁移要分类。**3570 处是字符串方法名调用**（`call_out("method", delay, args)` 即 `(object_id, method, delay, args)` 四元组，天然可序列化，机械转换可自动化）；**仅约 144 处使用闭包**。真正难点不是序列化而是对象生命周期管理（触发时目标可能已 destruct）。Effect 必须内置 target 存活校验。详见 01 §16。
>
> ⚠️ **承重修正 17**：PermissionService 必须明确 **fail-closed 默认策略**（Redis 不可达时拒绝所有命令，仅 look/inventory 等只读无副作用命令可选 fail-open），内置本地策略快照兜底。这是不可妥协的安全约束。
>
> ⚠️ **承重修正 18**：调度系统收敛为**单一后端**（APScheduler + Redis JobStore 或自研 ZSET worker 二选一），不要双系统。
>
> ⚠️ **承重修正 19**：分离两个安全域--运维侧 wizard ACL（admin/arch/wiz/player 命令权限）与 UGC 不可信代码沙箱能力（WASM/受限 Python capability tokens）是两类不同威胁模型，**不要用一套能力模型同时覆盖**。securityd.c 已损坏（第 757 行 `authorized_cmds["cmd"]` 应为 `"cmds"` 拼写 bug + 725-726 行格式不一致），**先修复恢复运维可用，再开放 UGC**。详见 01 §19。

### 4.6 DSL 与 UGC：分层混合 DSL，声明为主脚本为辅

四层 DSL，全部编译为 JSON IR 作为唯一真相源：
- **层0**：YAML 声明式场景数据（exits/objects/item_desc/cost/outdoors，对应 2482 个 replace_program 纯数据房间）
- **层1**：事件规则 `condition→action`（对应 add_action/valid_leave/accept_object/chat_msg）
- **层2**：对话树（Ink 的 knot/stitch/divert/weave，对应 inquiry）
- **层3**：脚本逃生舱（WASM 用于平台级已审计扩展 / RestrictedPython 用于 UGC 作者）

> ⚠️ **承重修正 20**：**heart_beat 不可被事件驱动替代**（见 §4.3 修正 8），故层1事件规则只覆盖触发器（进房/命令/对话），战斗回合/NPC AI/持续效果保留周期 tick 语义。
>
> ⚠️ **承重修正 21**：inquiry 本质是**交易状态机**（101 个 NPC 同时耦合 inquiry+accept_object+set_temp），Ink 的叙事分支模型不匹配交易原子性。层2需引入"交易原子节点"语义，或对带交易的 inquiry 用"对话+交易"混合节点，而非强行套 Ink。详见 01 §21。
>
> ⚠️ **承重修正 22**：逃生舱使用率将远超预期。层1规则 DSL 须扩充原语（apply_buff_to_actors/weighted_random/monitor_cooperation），设逃生舱使用率 KPI（<15%），超标触发 DSL 迭代。**先做表达力校准实验**（取 30 个代表性文件人工转译）再启动 8400 文件批量迁移。
>
> ⚠️ **承重修正 23**：层3 UGC 脚本用**受限 Python 子集**（RestrictedPython/Pyodide）而非裸 WASM（UGC 作者是非程序员中文 MUD 爱好者）。WASM 仅留作平台级已审计扩展。RestrictedPython 非安全边界，仅用于可信编辑者，需叠加运行时资源配额。
>
> ⚠️ **承重修正 24**：provenance 强制点**后移到"首次对外发布前（审批门3开启前）"**，而非"第一个开发 CPK"。开发期用简单版本号快速迭代，发布前再回填全量溯源并固化 CPK 格式。

### 4.7 Agent 协作创作：DSL 为契约，单进程起步
以 DSL 为唯一契约层把 LLM 非确定性收敛到可验证终态。编排者-工作者分工（Worldbuilder/Narrator/Behaviorist/Balancer/Continuity 五角色）+ 生成-评审-修订循环。MVP 先跑通这两条主干，红蓝对抗作为后期质量放大器。MCP 共享验证基底（优先 world-graph 出口可达性 + combat-sim 伤害/胜率）。

> ⚠️ **承重修正 25**：**combat-sim 存在循环依赖**--它声称"复用重构后 ECS 战斗引擎"，但该引擎尚未存在。先用**独立纯 Python 数值模型**（从 LPC `*action` 招式表与 attack.c 伤害公式 seed 出来）顶上，world-graph 可独立先行（纯 networkx 图分析无 ECS 依赖）。详见 01 §25。
>
> ⚠️ **承重修正 26**：**LangGraph 是单进程状态机库**，checkpoint 持久化只支持崩溃恢复，不提供多实例协调。声称的"编排器水平扩展多副本"不可用（会重复分派 Worker、双花 LLM token、产出冲突）。MVP 直接单进程 asyncio 编排；若需多实例用 Postgres advisory lock / Redis SETNX 给 workflow 加租约串行化认领，或务实接受"单编排器 + 故障转移"。详见 01 §26。
>
> ⚠️ **承重修正 27**：技术栈砍到 MVP 最小集。Lark+pydantic + 单进程 LangGraph + SQLite/PG + 一个 MCP 检查（world-graph）+ 最简评审 UI。Celery/MinIO/分布式 WASM 池/NATS 等单进程验证收敛后再按需引入。

### 4.8 前端与状态同步：语义事件流 + 客户端投影渲染

> ⚠️ **承重修正 28**：**`dbase=snapshot`、`tmp_dbase=delta` 的直接对应是概念错误**。dbase 与 tmp_dbase 是两个并行命名空间（分别经 query()/query_temp() 访问），tmp_dbase 不持有针对 dbase 的差分补丁，而是独立瞬时键。放弃"tmp_dbase=delta"映射话术，StateTree+delta 作为全新设计：服务端维护单一权威状态树，变更通过对比前后权威树生成 JSON-Patch ops。详见 01 §28。
>
> ⚠️ **承重修正 29**：dbase 有**函数值/闭包键**（104+ 文件 set 闭包值如 `set("actions", (: call_other, WEAPON_D, "query_action" :))`），无法跨网络序列化。建立 dbase 键分类台账：data-valued（可快照）vs function-valued（服务端求值后下发）。战斗 actions 闭包必须在服务端 resolve 成动作描述符再进事件，绝不进 snapshot。
>
> ⚠️ **承重修正 30**：渲染层迁移规模被严重低估--`message_vision` 用于 1626 个文件、`message()` efun 1070 处。模板池抽离是**数年级 backlog**，不是一次性取代。过渡期保留 LPC message_vision 经 Telnet 适配器作为真相源，先对最高频战斗/社交事件抽 msg_key 模板，加 lint 防 combatd 与模板池双份维护。
>
> ⚠️ **承重修正 31**：`apply/` 修饰符语义被错误统一。`apply/name`、`apply/short` 是数组栈（last-wins 覆盖），`apply/attack`、`apply/damage` 是标量累加槽。两类必须分别建模：presentation 型（覆盖）/ stat 型（累加）/ transform 型（数组栈）。

### 4.9 运维可观测与部署

K8s + Helm + Argo CD GitOps，OpenTelemetry 统一 metrics/logs/traces。

> ⚠️ **承重修正 32**：**有状态游戏分片不能套用无状态 HPA**。KEDA 反应式扩缩存在分钟级暖启动与滞后信号的时延错配（队列已深时玩家已在掉线）。改为"预测性 + 预留容量"模型：基线预留 standby 暖分片常驻、按日历/活动预告预扩、用负载趋势（导数）触发。
>
> ⚠️ **承重修正 33**：**战斗/数值变更本质是"灰度即全量"**。武侠 MUD 玩家自由移动，两版本战斗公式不同时"强制同房间同版本"机制无法成立。战斗公式变更走蓝绿全量（暖启动 standby 集群→handoff 整批切换），放弃分片粒度金丝雀战斗逻辑。
>
> ⚠️ **承重修正 34**：**liveness 探针仅判"事件循环死锁"（极保守，如 30s+ 无任何事件处理），绝不基于 tick p99/p50**。否则合法繁忙分片被杀→重启暖启动→流量转移→级联重启。tick 质量全部归 readiness（摘流量不杀）。加 PDB 防多分片同时被杀。
>
> ⚠️ **承重修正 35**：**Loki 是 UGC 多租户高维日志检索的错误选型**（标签基数爆炸）。日志后端用 OpenSearch/Quickwit/ClickHouse（支持全文+高维结构化检索），Loki 仅留低维 access log。
>
> ⚠️ **承重修正 36**：**"状态外部化使秒级回滚无数据丢失"是危险过度声明**。逻辑回滚≠状态回滚：Redis 快照带 TTL 可被淘汰；新旧版本 schema 漂移会导致旧代码读不一致状态；进行中战斗回滚产生不公平结果。回滚前必须评估 schema 兼容性，战斗回滚限定回合边界。
>
> ⚠️ **承重修正 37**：`call_out` 全库 3728 处**不应统一持久化**。短延迟（<30s）对象内定时器留 Actor 运行时内存（崩溃由对象重建恢复，零开销取消）；仅跨重启/全局/长周期走持久化调度器。
>
> ⚠️ **承重修正 38**：安全关键审计（特权授予/文件写/CPK 发布）走"**审计先于提交**"同步路径（同步写 PG 或同步 NATS 确认后才执行动作），异步缓冲仅用于高频低风险审计。
>
> ⚠️ **承重修正 39**：trace context 传播难度被低估。asyncio 的 contextvars 是 task 作用域，裸 `create_task`/回调不自动透传。提供 contextvars 传播助手 + lint 禁裸 create_task + 覆盖所有异步边界的契约测试，监控"trace 完整率"指标。

## 5. 迁移路径（风险驱动，分阶段）

> ⚠️ **承重修正 40**：迁移面被严重低估一个数量级。真实同步调用面是全库 **68771 处 `->` 方法调用**（仅 /d 就 36333 处）+ 520 处 all/deep_inventory 同步图遍历 + 84136 处 query/set dbase 访问，而非标称的 3450 处 `environment()`（实测仅 664 处）。**以箭头调用面（而非 environment）作为主线工作量指标**，先迁移 go/move/combat/channel 四条高频跨对象调用图。

```
阶段0  现状基线与差分测试基建（1-2月）
  ├─ 精确统计迁移面（68771 调用点 / 3570 字符串call_out / 144 闭包 / 1626 message_vision / 104 闭包dbase键）
  ├─ 修复 securityd.c 已知 bug（拼写+格式不一致）恢复运维可用
  ├─ 录制 LPC 命令流为 golden trace，建立差分测试框架（go/move/combat/channel 四主线）
  └─ 单区域（d/city 约90房）端到端 MVP 迁移试点，度量语义偏差率

阶段1  单进程 asyncio 核心循环（3-6月）★ 不可跳过
  ├─ Ray actor 模型重写核心循环验证 LPC↔Actor 同构映射（语义+延迟+消息序）
  ├─ SparseSet ECS 骨架 + Identity/Vitals/Position/Progression 核心组件
  ├─ SchemaRegistry 类型化组件 + ID 引用 ReferenceStore + 持久调度 SchedulerStore + desired-state reconcile
  ├─ 无状态 WS 网关 + 会话 + JWT/argon2 认证 + ring buffer 重连
  └─ 权威状态库 PG + Redis 热缓存（非 ES）；Redis/PG 写入顺序协议
  ★ 无差分测试通过的子系统不进阶段二

阶段2  子系统逐个迁移（6-12月，按风险递增）
  ├─ Query/索引层 → Identity/Position/Inventory（低耦合）
  ├─ Vitals/Heal/Condition（只读为主）★ 先修 ConditionHandler 数据模型
  ├─ Attribute/Skill/Equipment（中等耦合）
  └─ Combat（最高耦合：活引用/递归/闭包全在此）★ 先建数值回归基线
  ★ 战斗迁移前必须先建 CombatContext 快照边界 + Effect 一等公民组件

阶段3  分布式与多分片（按需，单节点达80%承载后）
  ├─ Ray 多节点 + placement + handoff（先 actor 适配器验证再切异步）
  ├─ 区域共同体分片（Louvain）+ 跨区迁移
  └─ 边界房间代理 + 一致性哈希热路径

阶段4  UGC 与 DSL 平台
  ├─ 先转 2482 个纯数据房间到层0（schema 字段完备后）
  ├─ securityd 能力模型重建（wizard ACL 与 UGC 沙箱分离）
  ├─ WASM/RestrictedPython 沙箱 + per-CPK 配额（不可妥协硬约束）
  └─ 单进程 LangGraph 编排 + world-graph MCP 验证

阶段5  Agent 协作创作 + 多题材
  ├─ provenance 在门3开启前回填
  ├─ combat-sim（独立纯Python数值模型起步，ECS 落地后替换内核）
  └─ ThemeRegistry 多题材（wuxia/nautical/academy/modern）
```

### 阶段性硬约束（不可妥协）
1. **差分测试**：8400 文件迁移无差分测试通过的子系统不进下一阶段。
2. **表达力校准**：DSL 8400 文件批量迁移前必须先做 30 文件人工转译验证，层3占比超 20% 则先扩充层1-2 原语。
3. **securityd 先修**：UGC 协作开放前必须先重建能力权限模型。
4. **沙箱配额先行**：开放 UGC 脚本前 per-CPK 资源配额必须就位，否则等于开放拒绝服务。
5. **容量以实测定**：单分片承载上限由压测得出（建议先按 2000-3000 并发规划，而非假设 5000-8000），再定分片数。

## 6. 技术选型总表

| 层 | 选型 | 说明 |
|---|---|---|
| 运行时 | Python 3.12 asyncio + uvloop | 统一异步运行时 |
| WS 网关 | aiohttp/websockets + SO_REUSEPORT | 无状态多 worker |
| REST/UGC API | FastAPI | 独立扩缩容 |
| Actor | **Ray（起步）** | Python 原生，验证后再决定是否自研薄壳 |
| ECS | 自研 SparseSet（**单存储，不用 Archetype**） | 针对 MUD 访问模式优化 |
| 协议 | MsgPack（默认）+ JSON（UGC）+ Protobuf（仅瓶颈） | 统一 {type,version,seq,trace_id,payload} 帧 |
| 认证 | argon2-cffi + PyJWT(RS256) | 替代 DES crypt() |
| 权威存储 | PostgreSQL 16（状态+审计，**非全量ES**） | WAL 强一致，JSONB 灵活 |
| 热状态 | Redis 7 Cluster | 路由/AOI/会话/修饰符 |
| 消息扇出 | NATS JetStream（可选，Redis Stream 起步） | 不过早引入 Kafka |
| 调度 | **APScheduler + Redis JobStore（单一后端）** | 分层：短延迟留内存 |
| 前端 | React + TypeScript + Zustand/Jotai + Immer | xterm.js 可选终端模式 |
| 序列化 | msgspec（高频）/ orjson | Pydantic v2 仅边界校验 |
| DSL | Lark（EBNF）+ PyYAML + pydantic v2 + jsonschema | 四层编译为 JSON IR |
| 沙箱 | **RestrictedPython（UGC）/ WASM（平台级）** | wasmtime-py 燃料计量 |
| Agent | LangGraph（**单进程起步**）+ Claude API | MCP 验证基底 |
| 可观测 | OpenTelemetry + Prometheus + **OpenSearch/ClickHouse** + Tempo | 非 Loki 做高维检索 |
| 编排 | K8s + Helm + Argo CD + Argo Rollouts | GitOps |
| 弹性 | HPA（无状态）+ KEDA（事件驱动，**预测性+预留**） | 有状态分片蓝绿不金丝雀 |
| 迁移 | LPC 静态分析（正则+AST+人工标注）+ 差分测试 | golden trace 回归 |
