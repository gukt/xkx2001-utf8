# 10. 坐骑与交通系统

《侠客行》的世界幅员辽阔，从西北大漠到东南海滨，从中原腹地到海外仙岛，玩家需要借助多种交通工具才能高效穿行。本章将深入拆解游戏的坐骑系统、水上交通系统、特殊交通工具以及它们与世界构建的关联。

---

## 10.1 坐骑系统

### 10.1.1 继承关系与对象模型

坐骑在《侠客行》中并非普通物品（ITEM），而是具有完整生命周期的 NPC 对象。所有可骑乘的动物统一继承自 `NPC_TRAINEE` 基类，其继承链如下：

```
NPC (/inherit/char/npc.c)
  └── NPC_TRAINEE (/inherit/char/trainee.c)
        ├── 马匹 (/clone/horse/*.c)
        ├── 骆驼 (/clone/horse/camel.c)
        ├── 野兽 (/clone/beast/laohu.c, lion.c)
        └── 其他可驯服生物
```

`NPC_TRAINEE` 在 `trainee.c` 中定义，核心职责是为可被驯服的生物提供统一的行为框架：

- **`train_it()`**：驯服机制。通过 `query("wildness")` 和 `query("loyalty")` 计算驯服难度，累计训练点数超过 100 后，生物归顺玩家，调用 `set_lord(ob)` 建立主从关系。
- **`do_gen()` / `do_ting()` / `do_fang()`**：跟随、停留、释放命令，仅对主人有效。
- **`do_yao()`**：指令生物攻击指定目标。
- **`is_trainee()`**：类型标识，供外部系统识别。

坐骑头文件 `/clone/horse/horse.h` 为马匹类生物补充了通用逻辑：

- **`condition_check()`**：核心精力监控。当 `jingli <= 10` 时，坐骑强制昏迷，骑手摔落并受伤；`jingli <= 30` 时发出喘息警告；`jingli <= max_jingli/3` 时大口喘气。
- **`init()`**：自动在草地房间吃草恢复 `food` 和 `jingli`；绑定 `look`、`ting`、`fang`、`gen` 等命令。
- **`do_look()`**：重写 `look` 命令，查看马匹时额外显示其背负的物品清单。

值得注意的是，`horse.h` 采用 `#include` 而非继承的方式嵌入逻辑，这是 LPC 中一种常见的代码复用模式，与 [[04-对象与继承体系]] 中介绍的 `inherit` 机制形成互补。

### 10.1.2 坐骑属性体系

每匹坐骑通过 `create()` 中的 `set()` 方法定义一组关键属性：

| 属性 | 含义 | 典型取值范围 |
|------|------|-------------|
| `ridable` | 是否可骑乘 | 1（可骑）/ 0（不可骑） |
| `ability` | 通行能力 | 2 ~ 10，决定可通过的最大地形 `cost` |
| `str` / `con` / `dex` / `int` | 四维属性 | 影响战斗与负重 |
| `max_qi` / `qi` | 气血上限与当前值 | 400 ~ 2000 |
| `max_jingli` / `jingli` | 精力上限与当前值 | 400 ~ 3000 |
| `combat_exp` | 战斗经验 | 500 ~ 100000 |
| `wildness` | 野性 | 3 ~ 20000，影响驯服难度 |
| `value` | 价值（银两） | 10 ~ 500 |
| `weight` | 重量 | 5000 ~ 200000 |

`ability` 是坐骑系统最核心的设计之一。在 [[05-世界构建系统]] 中，每个房间设有 `cost` 属性，表示该地形的通行难度（平原通常为 1 ~ 2，山地、沙漠可达 3 ~ 5，大海为 5）。在 `/cmds/std/go.c` 中，移动前会做如下校验：

```c
if( me->query("race") == "人类" && objectp(rided) ) {
    if( !obj->query("outdoors") )
        return notify_fail("你不能骑着" + rided_name + "进去。\n");
    if( obj->query("cost") > rided->query("ability") )
        return notify_fail(rided_name + "到了这地方好像走不动了！\n");
}
```

这意味着：
- `ability = 2` 的小毛驴只能在 `cost <= 2` 的平坦道路行走；
- `ability = 8` 的骆驼可以穿越沙漠（`cost = 4 ~ 5`）；
- `ability = 10` 的小红马（汗血宝马）几乎畅行无阻。

### 10.1.3 骑乘/下骑命令与状态变化

骑乘指令由 `/cmds/std/ride.c` 实现，流程如下：

1. **目标校验**：检查参数是否为环境中存在的生物、非人类种族、标记 `ridable`、处于存活状态。
2. **所有权校验**：必须是坐骑的 `lord`（主人），通过 `ob->query_lord() == me` 判断。
3. **状态互斥**：玩家 `me->query("rided")` 和坐骑 `ob->query("rider")` 双向绑定，同一时间只能骑乘一匹。
4. **建立关系**：设置 `me->set("rided", ob)`、`ob->set("rider", me)`，并调用 `ob->set_leader(me)` 使坐骑自动跟随。

下骑指令 `/cmds/std/unride.c` 则简单得多：

```c
me->delete("rided");
ob->delete("rider");
```

同时清除双向引用。若玩家移动后坐骑未能跟随（如被障碍物阻挡），`go.c` 在移动末尾会自动检测并解除 `rided`/`rider` 关系，防止状态悬空。

### 10.1.4 坐骑对移动的影响

骑乘不仅改变移动描述（"骑着 xx 往 xx 飞驰而去"），还在机制层面产生三方面影响：

1. **精力消耗转移**：
   ```c
   if( objectp(rided) ) {
       if( random(5) == 0 ) me->add("jingli", -2);
       rided->add("jingli", - env->query("cost")*2);
       if( rided->query("jingli") <= 0 ) rided->unconcious();
   }
   ```
   玩家自身几乎不消耗精力，坐骑代为承担 `cost * 2` 的消耗。这是骑乘最核心的收益——长距离高速移动。

2. **路口冲撞机制**：
   当出口被其他玩家阻挡时，骑乘方若经验值占优，可直接将挡路者撞飞，甚至把对方从坐骑上撞下来。这引入了交通层面的 PvP 互动。

3. **逃跑优势**：
   战斗逃跑时，骑乘状态的描述为"伏在 xx 身上往 xx 落荒而逃"，虽然机制上仍受 `dex` 判定约束，但视觉表现强化了坐骑的机动性。

### 10.1.5 获取、喂养与丢失机制

**获取途径**

1. **马厩购买**：`/d/city/majiu.c` 是城中马厩，常驻马夫 NPC（`/d/shaolin/npc/ma-fu.c`）。玩家通过 `ask ma fu about 买马` 浏览在售马匹，说出马名后支付对应银两即可成为主人。马匹价格由 `value` 属性决定，例如小毛驴 10 两、伊犁马 140 两、小红马 500 两。

2. **野外驯服**：部分坐骑（如普通白马）或野兽（老虎、狮子）可通过 `train` / `tame` 命令驯服。驯服成功率受 `wildness` 影响，白龙马的 `wildness` 高达 10000，几乎无法野外驯服。

3. **任务/剧情获取**：小红马、白龙马等特殊坐骑通常通过任务链或剧情事件获得，其 `do_tame()` 被重写为直接返回失败，杜绝了野外驯服的可能。

**喂养与恢复**

- **草地自食**：当坐骑处于标有 `resource/grass` 的房间且 `food < max_food_capacity` 时，自动吃草恢复 `food` 和 `jingli`。骆驼在沙漠中因 `resource/grass` 稀缺，需要更长周期恢复。
- **马夫代喂**：玩家可向马夫支付银两（100 ~ 600 文不等）托管喂养，马夫会将 `food`、`water`、`jingli` 恢复至 80%。
- **昏迷急救**：若坐骑精力耗尽昏迷，马夫可以用水泼醒（`revive()`），但 `jingli` 仍需后续恢复。

**丢失机制**

- **死亡丢失**：坐骑在战斗中死亡后，玩家永久失去该对象。
- **离线丢失**：若玩家离线后坐骑未被妥善安置（如寄存在马厩），可能因房间刷新（`reset`）或 `clean_up` 机制导致对象销毁。
- **被抢/被盗**：虽然代码中马夫会对 `train` 陌生马匹的玩家施加 `killer` 标记，但理论上若主人昏迷或死亡，坐骑处于无主状态，可能被他人驯服。

## 10.2 水上交通系统

水上交通是连接大陆与海岛、河流两岸的关键基础设施，分为渡口（FERRY）和海港（HARBOR）两套子系统。

### 10.2.1 渡口（FERRY）的实现模式

渡口基类定义在 `/inherit/room/ferry.c`，采用**虚拟渡船 + 定时器**模式：

```c
void check_trigger()
{
    if( this_object()->query("exits/enter") ) {
        // 船已在岸边，直接提示
        return;
    }
    // 加载对岸渡船对象
    if( !(room = find_object(myboat)) ) room = load_object(myboat);
    room->set("yell_trigger", 1);
    this_object()->set("exits/enter", myboat);
    room->set("exits/out", base_name(this_object()));
    // 15 秒后离岸
    remove_call_out("on_board");
    call_out("on_board", 15);
}
```

玩家在渡口执行 `yell boat`（或 `yell 船家`）后：

1. **0 秒**：系统检查是否有船在岸边。若无，加载渡船对象，在渡口设置 `exits/enter` 指向渡船，同时在渡船设置 `exits/out` 指向渡口。
2. **15 秒**：`on_board()` 收起跳板，删除双向出口，广播"扁舟向江心驶去"。
3. **20 秒**：`arrive()` 在对岸渡口设置 `exits/out` 指向渡船，玩家可下船。
4. **再 20 秒**：`close_passage()` 船夫驾船返回江心，删除出口，重置 `yell_trigger`。

渡口房间通过 `set("boat", __DIR__"duchuan")` 和 `set("opposite", __DIR__"hanshui2")` 指定渡船对象与对岸房间。典型应用包括汉水（少林区域）、黄河（西夏区域）、太湖、峨眉解脱坡等。

FERRY 模式**不收取费用**，完全免费，适合短距离河流摆渡。

### 10.2.2 海港（HARBOR）与海船（SHIP）的移动机制

对于跨海航程（如从塘沽口到桃花岛、神龙岛、冰火岛），系统采用更复杂的坐标导航模式。

**海港（HARBOR）**

`/inherit/room/harbor.c` 是海港基类，核心特征：

- 玩家通过 `yell chuan` 呼叫渔船。
- **收费系统**：`valid_leave()` 中拦截 `enter*` 方向，调用 `MONEY_D->player_pay(this_player(), 1000)` 收取 10 两银子（1000 文）。若余额不足或仅有银票无法找零，则拒绝登船。
- 多船调度：系统扫描 `/clone/ship/` 目录下以 `seaboat` 开头的文件（`seaboat1` ~ `seaboat3`），循环寻找未被占用的船只。

```c
switch (MONEY_D->player_pay(this_player(), 1000)) {
case 0: return notify_fail("穷光蛋，一边呆着去！\n");
case 2: return notify_fail("您的零钱不够了，银票又没人找得开。\n");
default: // 扣费成功，允许登船
}
```

**海船（SHIP）**

`/inherit/room/ship.c` 实现了完整的航海模拟：

1. **开船（`start`）**：删除与港口的 `exits/out`，从港口获取初始坐标 `navigate/locx` 和 `navigate/locy`，启动 `shipweather` 和 `navigate` 定时器。

2. **导航（`go`）**：玩家指定方向（东/南/西/北），存入 `navigate/dir`。每 2 秒执行一次 `navigate()`，坐标步进 1 格。

3. **天气系统**：
   - `shipweather()` 定时触发，有概率进入彤云密布（1）或巨浪滔天（2）状态；
   - 暴风雨中，若 `locx > 50 || locy > 50`（远海），有概率直接翻船。

4. **危险事件**：
   - **触礁**：坐标匹配 `/clone/ship/seashape.h` 中定义的暗礁数组 `jiaos` 时，直接沉船。
   - **翻船**：`time_out()` 在 900 ~ 1400 秒后强制触发翻船；暴风雨中随机翻船。
   - **随机遭遇**：大雾幽灵船（Titanic）、燃烧大鸟（凤凰？）、美人鱼、海怪眼瞳、极光等 10 种事件，纯叙事无战斗。

5. **靠岸判定**：
   - `locx < 1` 时匹配大陆港口（`harbors` mapping）；
   - `locx` 匹配海岛坐标（`islands` mapping）时，设置 `exits/out` 并允许上岸。

6. **翻船惩罚（`do_drop`）**：
   船上所有玩家 `unconcious()`，随身物品（除铁链外）全部 `destruct()`，玩家被 `move()` 到随机港口海岸，广播"浑身水淋淋的家伙被海水冲上岸来"。

### 10.2.3 渡河/渡海的完整流程

以从汉水南岸到北岸为例（FERRY 模式）：

```
玩家位于 /d/shaolin/hanshui1（汉水南岸）
  │
  ▼ yell boat
系统加载 /d/shaolin/duchuan（渡船对象）
  │
  ▼ 0 秒
渡口新增 exits/enter → duchuan
渡船新增 exits/out → hanshui1
  │
  ▼ 15 秒（on_board）
删除上述出口，广播离岸
  │
  ▼ 20 秒（arrive）
对岸 /d/shaolin/hanshui2 新增 exits/out → duchuan
渡船新增 exits/out → hanshui2
  │
  ▼ 玩家执行 out
回到对岸渡口
  │
  ▼ 再 20 秒（close_passage）
船驶回江心，重置触发器
```

以从塘沽口到桃花岛为例（HARBOR + SHIP 模式）：

```
玩家位于 /d/beijing/tanggu（塘沽口，locx=0, locy=0）
  │
  ▼ yell chuan
系统分配空闲 seaboat（如 seaboat1）
塘沽口新增 exits/enter1 → seaboat1
seaboat1 新增 exits/out → tanggu
  │
  ▼ 玩家执行 enter1
扣除 10 两银子，进入海船
  │
  ▼ start
开船，删除港口出口，初始化坐标 (0, 0)
  │
  ▼ go east（多次）
坐标递增，如 (5, 0)、(10, 0)...
  │
  ▼ navigate 判定
当坐标匹配 islands["/d/taohua/haitan"] = ({30, -180}) 时
  │
  ▼ 靠岸
seaboat1 设置 exits/out → /d/taohua/haitan
桃花岛海滩设置 exits/enter1 → seaboat1
```

## 10.3 特殊交通工具

### 10.3.1 镖车系统

`/clone/npc/obj/xbiaoche.c` 实现了任务导向的镖车运输系统。与坐骑不同，镖车是 `ITEM` 对象而非 `NPC`，玩家通过 `ride`/`shang` 命令"坐在镖车上"，通过 `gan che <方向>`（或缩写 `g che east`）驾驶。

**核心机制**：

- **双人协作**：支持 `owner` 和 `owner2` 两名玩家同乘，移动时两人一并转移。
- **Busy 惩罚**：每次驾车后玩家进入 `busy` 状态 4 ~ 8 秒，模拟镖车笨重迟缓。
- **劫镖事件**：定时器 `auto_check()` 在野外有概率召唤 `robber` NPC 劫镖。强盗强度根据玩家经验动态生成，可触发战斗或抢夺。
- **路线终点**：镖车到达目的地后，`greeting()` 触发奖励结算，给予黄金并清除任务状态。

注意：`/cmds/std/drive.c` 并非镖车驾驶命令，而是雪山派降伏术 `necromancy` 的附属技能，用于将被诅咒的玩家驱赶至指定方向，与交通系统无关。

### 10.3.2 特殊坐骑的扩展能力

部分顶级坐骑拥有超越常规 `ability` 校验的特殊技能：

- **白龙马（`bailong.c`）**：`ability = 5`，附带 `duhe` 命令，可在汉水、黄河、峨眉解脱坡、太湖等特定渡口直接纵马渡河，无需等待渡船。
- **小红马（`xiaohongma.c`）**：`ability = 10`，除 `duhe` 外还拥有 `whistle`（吹口哨）逃生技能。当主人遇险时，小红马可主动叼起主人狂奔至相邻房间。

这些设计将坐骑从单纯的"脚力"提升为具有场景交互能力的伙伴，与 [[01-架构总览]] 中"万物皆对象"的设计理念一致。

### 10.3.3 其他可骑乘生物

除马匹外，系统中还散布着一些非常规坐骑：

- **骆驼（`camel.c` / `btcamel.c`）**：`ability = 8`，高精力（3000），专为大沙漠等高 `cost` 地形设计。
- **老虎（`laohu.c`）、狮子（`lion.c`）**：`ability = 4`，属于 `race = "野兽"`，攻击性生物，驯服后可骑乘。但因其攻击性强，只有在成为 `lord` 后才不会反噬主人。
- **小毛驴（`donkey.c`）**：`ability = 2`，最低档坐骑，价格低廉（10 两）。

目前系统中**不存在飞行类交通工具**（如风筝、翅膀、腾云驾雾）。虽然存在 `fly` 关键词的文件（如 `dufeng.c` 毒蜂、`yufeng.c` 玉蜂），但均为战斗宠物或场景生物，不具备载人飞行功能。

## 10.4 交通与世界的关系

### 10.4.1 交通对区域连接的架构意义

在 [[05-世界构建系统]] 中，世界由房间（ROOM）通过 `exits` 静态连接构成。然而，河流与海洋天然切断了这种连接——两岸房间无法直接通过 `exits` 指向彼此。交通系统正是在此缺口上架设的动态桥梁：

- **FERRY**：在运行时动态注入 `exits/enter` 和 `exits/out`，将原本断开的两岸临时缝合。这相当于一种"运行时补丁"，避免了两岸房间直接硬编码跨水域出口导致的世界逻辑混乱。
- **HARBOR + SHIP**：更进一步，引入了坐标空间（`locx`, `locy`）。海港房间只负责上下船，真正的"世界地图"由 `harbor.h` 和 `seashape.h` 中的坐标表维护。这使得大海成为一片可探索的开放空间，而非简单的点对点传送。

```c
// /clone/ship/harbor.h
mapping harbors = ([
    "/d/beijing/tanggu"    :    0,      // 大陆港口，仅 locy
    "/d/jiaxing/zhoushan"  : -200,
    "/d/quanzhou/yongning" : -280,
]);
mapping islands = ([
    "/d/shenlong/beach"    : ({ 30,   20}),
    "/d/island/icefire1"   : ({100,  600}),
    "/d/taohua/haitan"     : ({ 20, -210}),
]);
```

### 10.4.2 交通费用系统

游戏中存在多层费用体系：

1. **海船渡资**：`harbor.c` 统一收取 10 两银子，通过 `MONEY_D` 处理，支持铜钱、碎银、银锭、银票的组合支付与找零。
2. **草地自食**：免费，但受限于地形分布。
3. **马夫托管**：100 ~ 600 文不等，属服务消费。
4. **马匹交易**：购买按 `value` 全价，出售按 `value/2` 折价，存在明显的买卖价差。
5. **镖车任务**：无直接交通费，但隐含风险成本（战斗损耗、装备耐久）。

费用系统的设计逻辑是：短程河流摆渡免费（FERRY），远程跨海航程收费（HARBOR），陆地移动成本转嫁给坐骑精力（时间成本），而高端坐骑（小红马、白龙马）通过特殊技能绕过部分等待时间，形成"付费换时间"或"投入换效率"的经济循环。

---

## 10.5 与其他系统的关联

- **[[01-架构总览]]**：坐骑与交通系统是玩家感知世界规模的核心通道。没有马匹，中原到西域可能需要现实时间数小时；没有海船，桃花岛、冰火岛等剧情区域将完全与世隔绝。
- **[[04-对象与继承体系]]**：坐骑展示了 LPC 中 `inherit` 与 `#include` 两种复用模式的混用——`NPC_TRAINEE` 提供行为框架，`horse.h` 提供横切逻辑（精力检查、草地吃草）。海船则展示了 `ROOM` 对象的扩展形态：`SHIP` 继承 `ROOM`，但内部运行坐标导航与定时器驱动的状态机。
- **[[05-世界构建系统]]**：`cost` 属性将房间地形与坐骑 `ability` 耦合，形成了"地形 - 交通 - 探索"的三位一体。高 `cost` 区域（如大漠、雪山）天然成为低等级坐骑的禁区，引导玩家升级交通能力或选择绕行。

---

## 10.6 小结

《侠客行》的交通系统呈现出一套层次分明、互相补充的架构：

- **坐骑层**：以 `NPC_TRAINEE` 为根基，`ability` 为核心属性，覆盖绝大多数陆地移动需求；
- **渡船层**：以 `FERRY` 为原型，零费用、定时触发，解决河流阻断问题；
- **海航层**：以 `HARBOR + SHIP` 为组合，引入坐标导航与天气风险，连接大陆与海岛；
- **特殊层**：镖车、神驹渡河等机制，为任务和剧情提供定制化交通体验。

整个系统没有采用现代 MUD 中常见的"传送门"或"飞行点"简化设计，而是坚持在 LPC 的对象模型与定时器机制上模拟真实的物理移动过程——等待、消耗、风险、探索感，这正是经典中文 MUD 独特魅力的技术来源。
