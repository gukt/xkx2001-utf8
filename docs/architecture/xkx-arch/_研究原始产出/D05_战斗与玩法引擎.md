# 方案设计·战斗与玩法引擎

## 设计概述
核心理念：保留 XKX 战斗系统的"七步管线 + AP/(AP+DP) 概率 + 三层资源 + 数据驱动招式"心智模型，但用 ECS（组件+系统）替换 inherit/feature 静态混入，用固定 tick 仿真器替换 heart_beat 上帝方法，用可序列化的 Effect 对象替换 call_out 闭包，用事件总线解耦 do_attack 中 feature 间的直接 dbase 读写。

设计遵循三条主线：(1) 数据与行为分离--do_attack 中的伤害公式/魔法数字外提为可配置公式表，技能 mapping *action 外提为 JSON/YAML Prefab，文案 damage_msg/eff_status_msg 外提为数据文件，门派专属 max_qi/max_jing 加成从 human.c 的 if-else 链外提为门派数据声明；(2) 同步调用改事件驱动--do_attack 直接读写 victim->receive_damage/my['jingli']-= 改为发 DamageEvent/VitalsChangedEvent，技能 hit_ob 钩子链改为 Effect Pipeline，跨系统通信走事件总线消除 this_object() 隐式耦合；(3) 状态外部化与可序列化--enemy/killer 的 object 指针改为 entity_id 引用，call_out 持续效果改为可持久化的 Effect 对象，每场战斗独立 seeded RNG 支持回放审计。

七步管线（选技能->取招式->AP/DP命中->PP招架->伤害计算->施加伤害->经验/反击）作为 CombatSystem 的核心 resolve_attack() 方法保留分层，但每步是纯函数读取组件快照、返回事件，而非直接 mutate dbase。skill_power 的 level³/3 三次方公式保留为可配置表达式（支持 UGC 多题材调整曲线），AP/(AP+DP) 与 PP/(AP+PP) 概率模型原样保留（双方大于零故总有命中/被命中可能）。三层资源（qi/jing/jingli + eff/max 双层）与 perform/exert/hit_ob 钩子链作为战斗引擎的差异化资产完整保留。

## 组件
- **TickSimulator（固定 tick 仿真调度器）**：替换 char.c heart_beat 的单一全局 2s 上帝方法。职责：(1) 维护仿真步进循环，固定 tick 周期（默认 2s 保留手感，可配置）；(2) 按子系统注册 tick 回调而非每实体独立 heart_beat，实现批处理；(3) 保留 set_heart_beat(0/1) 的按需启停思想--仅活跃实体（战斗中/有 condition/非满血）进 tick 队列，空闲实体出队省 CPU；(4) 保留 char.c 的 tick=5+random(10) 非均匀调度思想--高频 tick 跑战斗/恢复，低频 tick 跑 condition 衰减/衰老，用 tick_phase 计数器降频；(5) 仿真步进产生事件批边界，每 tick 结束 flush 事件总线。（技术：asyncio 事件循环 + 优先级 tick 队列（heapq）。tick 调度用分级桶：tick0=战斗/恢复(每 tick)，tick1=condition(每 5-15 tick 随机抖动)。用 SimulationClock 单调时钟驱动，断线挂起时暂停推进而非销毁。）
- **CombatSystem（战斗管线系统）**：替换 COMBAT_D 的 do_attack 七步管线与 fight() 调度。保留七步心智模型但解耦 dbase 直接读写：(0) 选技能（weapon skill_type/prepare/unarmed）-> (1) 取招式（查 SkillDefinition 而非 call_other SKILL_D）-> (2) 算 AP/DP（调 DamageFormula 而非内嵌 level³/3）-> (3) 命中判定 random(ap+dp)<dp 闪避 -> (4) 招架判定 random(ap+pp)<pp -> (5) 伤害计算（Effect Pipeline 而非嵌套 if stringp/intp/mapp）-> (6) 施加伤害（发 DamageEvent 而非直接 receive_damage）-> (7) 经验/反击（riposte 递归保留为 TYPE_RIPOSTE）。fight() 的时机判定（victim busy/aggressive guard/random(dex*3)<str*2+speed）保留为回合调度策略。战斗结果通过事件发布，文案渲染下沉到 CombatRenderer。（技术：纯函数管线 resolve_attack(ctx)->CombatRoundResult，每步是可单测的 stage 函数。攻击类型 TYPE_REGULAR/RIPOSTE/QUICK 保留为 enum。双击/双手互博保留为 CombatModifier 链。）
- **DamageSystem（伤害施加系统）**：替换 F_DAMAGE 的 receive_damage/receive_wound/receive_heal/receive_curing。职责：(1) 维护三层资源一致性（current 不超 eff、eff 不超 max）；(2) receive_damage 扣 current 并发 DamageApplied 事件 + 唤醒 tick（原 set_heart_beat(1) 语义）；(3) receive_wound 扣 eff（创伤降低上限），保证 current<=eff 不变量；(4) 精力/内力资源（jingli/neili）的消耗与恢复；(5) pay_money 不足不再 error() 抛异常，改为返回结果对象。关键不变量：0<=current<=eff<=max（qi/jing），jingli/neili 仅 current<=max。（技术：Vitals 组件 + 原子操作 apply_damage()/apply_wound()/apply_heal()/apply_cure()。每次修改经事件总线广播，投影层订阅。用 Pydantic 校验不变量。）
- **HealSystem（恢复系统）**：替换 damage.c heal_up()。职责：(1) 战斗中/非战斗不同恢复速率（原 is_fighting 时 jing+=con/9+max_jingli/30，非战斗 con/3+max_jingli/10）；(2) water/food 消耗与饥饿判定（无水食不恢复）；(3) eff 缓慢自愈（eff<max 时每 tick +1）；(4) jingli 恢复受 str+dex 影响，可超 max*2 缓冲；(5) neili 恢复受 force 技能影响。恢复逻辑拆为独立可测函数，魔法数字（/9,/30,/3,/10,/12,/4,/6,/2）提取为 HealRateConfig。（技术：HealSystem.tick(entity, in_combat) 纯函数，读取 Vitals+Attributes 组件。恢复速率配置化。）
- **ConditionSystem（状态效果系统）**：替换 F_CONDITION 的 update_condition + 72 个 condition 守护进程。职责：(1) 维护实体 ConditionStack（活跃 buff/debuff 列表）；(2) 每 tick 按 tick_interval 驱动每个 condition 的 on_tick 回调；(3) 保留 CND_CONTINUE 生命周期协议--condition 自主决定存活/过期（返回 continue 则保留）；(4) 保留 CND_NO_HEAL_UP 标志抑制恢复（如中毒状态）；(5) condition 应用时支持覆盖/叠加策略（原 apply_condition 由给予者决定覆盖）；(6) 死亡/昏迷时 clear_condition。每个 condition 从 LPC 对象迁移为 ConditionHandler 策略对象（数据驱动 + 行为注册表，天然适配 UGC DSL）。（技术：ConditionHandler 协议 + 注册表（condition_id -> handler）。condition 实例为 Effect 数据对象（见 EffectSystem）。handler 可为纯 Python 策略类或 WASM 脚本（UGC）。）
- **EffectSystem（持续效果系统，替换 call_out）**：替换 perform/exert 中用 call_out(回调函数) 实现的持续效果（如降龙一击的 msg_display->remove_effect 递归 call_out 链）。职责：(1) 将 call_out 闭包提升为可序列化的 Effect 对象（携带 effect_id/duration/tick_handler/payload/source_entity）；(2) Effect 由 TickSimulator 驱动而非绑定对象生命周期，对象销毁/迁移不丢失效果；(3) 支持取消（remove_effect 对应 remove_call_out 语义）；(4) 支持跨 tick 的多阶段效果（如降龙一击：蓄力阶段->命中阶段->结算阶段，每阶段一个 Effect）；(5) Effect 持久化到快照，崩溃恢复后继续。这是分布式化最关键的改造--call_out 闭包捕获对象引用无法跨进程迁移。（技术：Effect dataclass + EffectRegistry。Effect.tick(ctx) 驱动，可注册到期回调。用 asyncio.Task 但状态外部化到 Effect 对象，Task 仅是驱动器。）
- **SkillSystem（技能管理系统）**：替换 F_SKILL 的 skills/learned/skill_map/skill_prepare + inherit/skill/skill.c 基类 + 798 个技能文件。职责：(1) 技能数据驱动化--mapping *action 迁移为 SkillDefinition Prefab（JSON/YAML），含 actions[]招式表（action/force/dodge/damage/lvl/skill_name/damage_type）；(2) query_skill 保留 apply+skills/2+skill_map 聚合计算；(3) map_skill/prepare_skill 保留为技能绑定机制；(4) improve_skill 保留经验积累升级 + spi=30 学习惩罚；(5) skill_death_penalty 保留；(6) perform_action/exert_function 分发保留为技能行为查找（perform/exert 子文件迁移为 SkillEffect 策略对象）；(7) 359 个技能的行为部分（hit_ob/perform_action/exert_function）保留 OOP 策略模式，不做纯组件。技能定义（数据）与技能行为（逻辑）分离。（技术：SkillDefinition（Pydantic 模型，数据驱动）+ SkillBehavior（策略类注册表，按 skill_id 查找）。数据部分可 DSL/UGC，行为部分纯 Python。NewRandom 加权随机保留为可配置概率表。）
- **EquipmentSystem（装备系统）**：替换 F_EQUIP 的 wield/wear/unequip + apply/ 修饰符堆叠。职责：(1) 装备槽管理（weapon/secondary_weapon/armor dict）；(2) wield/wear 时把 weapon_prop/armor_prop 叠加到 ModifierStack（对应原 owner->add_temp('apply/...')）；(3) unequip 时减去（保留对 array apply 的支持）；(4) 双手武器/副手武器/盾牌的手部占用逻辑保留；(5) reset_action 装备变更时触发招式重算。装备变更加 EquipChanged 事件，EquipmentSystem 监听更新 ModifierStack 后标记 AttributeSystem dirty。（技术：EquipmentSlots 组件 + ModifierStack 组件。装备定义为 Prefab（weapon_prop/armor_prop/skill_type/flag）。）
- **AttributeSystem（派生属性系统）**：替换 F_ATTRIBUTE 的 query_str/int/con/dex/kar/per。职责：(1) 派生属性计算（query_str = str + apply/strength + max(武术技能)/10 等）保留为 dirty 缓存标记--ModifierStack 变更时标记 dirty，下次查询重算；(2) 消除散落全库的字符串键 query('str') 无类型检查，改为类型化 Component schema；(3) 门派专属加成从 human.c 的 if-else 链迁移为门派数据声明 Prefab（family_definition.yaml 声明 max_qi/max_jing 加成公式）。（技术：Attributes 组件 + dirty flag + property 描述符缓存。派生属性用纯函数 compute(attr_data, modifiers, skills)。）
- **DeathSystem（死亡/惩罚/奖励系统）**：替换 combatd.c 的 death_penalty/killer_reward + damage.c 的 die/unconcious/revive + announce。职责：(1) 死亡处理链拆为可配置策略--death_penalty 从硬编码 combat_exp/shen/behavior_exp/balance/potential/skills 多维惩罚改为可声明策略（支持 UGC 多题材不同规则）；(2) killer_reward 的 PKS/MKS/shen/killer condition/free_rider 保留为事件驱动；(3) free_rider 抢杀检测保留（受害者非最后击伤则无奖励）；(4) announce 的 dead/unconcious/revive 公告保留为事件；(5) unconcious/revive 的昏迷-苏醒状态机保留，revive 延迟 = random(100-con)+30 迁移为配置；(6) 死亡惩罚策略按题材可配（武侠重惩罚/现代剧情轻惩罚）。（技术：DeathPolicy 策略对象（按题材/世界配置）。death_penalty/killer_reward 拆为独立可配置函数。PLAYER_DEATH/PKILL_DATA 日志并入事件溯源审计流。）
- **TargetingSystem（敌人选择/仇恨系统）**：替换 attack.c 的 select_opponent（随机选 MAX_OPPONENT=4 之一）+ enemy/killer 数组。职责：(1) 仇恨表替代随机选择--记录每个敌人的 threat 值（受伤害量/连击/嘲讽影响），选最高仇恨或按权重；(2) enemy_ids/killer_ids 从 object 指针改为可序列化的实体 ID 引用；(3) clean_up_enemy 的环境感知清理（不在同房间/已死亡/非 killer 自动移除）保留；(4) MAX_OPPONENT=4 上限保留为可配置；(5) remove_enemy/remove_killer 的互斥逻辑保留（killer 不可 remove_enemy）。（技术：ThreatTable 组件（entity_id -> threat_score）。select_target() 按 threat 排序或加权随机。enemy 关系用 ID 集合而非指针。）
- **CombatRenderer（战斗渲染/文案系统）**：替换 combatd.c 内嵌的 damage_msg/eff_status_msg/status_msg/guard_msg/catch_hunt_msg/winner_msg 文案数组。职责：(1) 战斗文案与逻辑分离，迁移为数据文件（damage_messages.json 等）；(2) 表情代词替换体系（$N/$n/$P/$p/$l/$w/$C 称谓）保留为纯函数渲染层；(3) ANSI 颜色码移至 Web 前端渲染层，传输纯语义事件；(4) damage_msg 的 8 种伤害类型分档文案保留为数据表；(5) 支持多语言/换肤/A-B 测试。（技术：MessageTemplate 数据文件 + render(event, context) 纯函数。代词替换用 token 替换表。前端接收语义事件自行渲染。）
- **DamageFormula（伤害公式引擎）**：替换 combatd.c 满地魔法数字（/30,/10,/100,/3,/25,damage/3 减免循环）。职责：(1) skill_power 公式（原 level³/3 + combat_exp）配置化为可调表达式，消除 level³ 三次方导致的数量级碾压（可配置为 level² 或线性）；(2) jingli_bonus 两版不一致（combatd /20 vs s_combatd /2）统一为单一配置；(3) 伤害减免循环（while random(defense_factor)>my_exp: damage-=damage/3）保留为可配置减免函数；(4) 所有魔法数字提取为 DamageFormulaConfig，平衡调整改配置而非改代码；(5) 公式求值支持纯函数与表达式 DSL 两种模式。（技术：DamageFormulaConfig（Pydantic）+ 公式求值引擎（纯 Python lambda 或受限表达式 AST）。公式可热重载。）
- **CombatCoordinator（战斗协调 Actor）**：分布式化时将 COMBAT_D 实例化为 region/战斗级 Coordinator Actor。职责：(1) 同房间战斗参与者保持同节点亲和，由 Coordinator 串行裁决回合避免分布式事务；(2) 持有该战斗的独立 seeded RNG，保证回合结果可回放可审计；(3) 战斗事件写 append-only 日志；(4) 战斗结束/参与者迁移时协调 handoff。（技术：自研 asyncio Actor，按 region 寻址。seeded RNG（random.Random(seed)），每回合推进种子。事件日志写入 PostgreSQL/NATS。）

## 关键接口/事件
- AttackContext（攻击上下文，显式化原 previous_object 隐式信任链）: attacker_id, defender_id, weapon_ref, attack_type(Enum: REGULAR/RIPOSTE/QUICK), rng_seed, tick_seq
- CombatRoundResult（替换 hit_ob 的 string|int|mapping 多态返回）: result(Enum: HIT/DODGE/PARRY), damage_applied, wound_applied, messages(list[MessageToken]), modifier_deltas, follow_up(可选反击 AttackContext)
- SkillEffect 协议（替换 inherit/skill/skill.c 的 hit_ob/query_action/perform_action/exert_function）: query_action(level, weapon)->Action | None; hit_ob(ctx, damage_bonus, factor)->EffectResult; hit_by(ctx, damage)->EffectResult; perform(ctx, target)->EffectResult; exert(ctx, target)->EffectResult
- EffectResult（统一返回类型）: damage_delta(int), messages(list[str]), effects_to_apply(list[EffectInstance]), conditions_to_apply(list[ConditionInstance])--消除原 5 层 if(stringp)/else if(intp)/else if(mapp) 嵌套
- ConditionHandler 协议（替换 condition 守护进程的 update_condition）: on_apply(entity, info)->None; on_tick(entity, info, tick_count)->CNDFlag(Enum: CONTINUE/EXPIRE/NO_HEAL); on_expire(entity)->None
- DamageFormula 接口: skill_power(entity, skill, usage)->int; compute_damage(ctx, action, base)->DamageBreakdown; mitigate(damage, attacker_exp, defender_exp)->int
- 事件: AttackResolved(attacker, defender, weapon, attack_type, result, ap, dp, pp, damage)--对应原 do_attack 全流程
- 事件: DamageApplied(target_id, vitals_type(qi/jing/jingli), amount, source_id, is_wound, reason)--替换 receive_damage 的 set_temp last_damage_from
- 事件: HealApplied(target_id, vitals_type, amount, heal_kind(damage/wound/cure))
- 事件: ConditionApplied(entity_id, condition_id, duration, info) / ConditionTicked / ConditionExpired
- 事件: EffectStarted(effect_id, source, target, duration) / EffectTicked / EffectCancelled / EffectExpired--替换 call_out 的隐式调度
- 事件: SkillPerformed(entity_id, skill_id, perform_name, target_id) / SkillExerted / SkillImproved(entity_id, skill_id, new_level)
- 事件: EntityDied(victim_id, killer_id, is_pk) / EntityUnconscious(victim_id, defeater_id) / EntityRevived(entity_id)
- 事件: EquipmentChanged(entity_id, slot, item_ref, equipped)--触发 ModifierStack 更新 + AttributeSystem dirty
- 事件: CombatRoundResolved(combat_id, round_num, all_events[], rng_state)--回合聚合事件用于事件溯源回放，而非逐 tick 爆炸
- 事件: ThreatChanged(entity_id, target_id, delta, reason)--仇恨表变更
- Targeting 接口: select_target(entity)->EntityID | None（基于 ThreatTable）; add_threat(target, amount, reason); remove_enemy(entity_id); clean_up_enemies()

## 数据模型
## 核心数据模型

### 1. 三层资源模型 Vitals（保留 damage.c 设计）
```
Vitals:
  qi: int          # 气血当前值（受击扣减）
  eff_qi: int      # 气血创伤上限（receive_wound 扣减，<= max_qi）
  max_qi: int      # 气血上限
  jing: int        # 精神当前值
  eff_jing: int    # 精神创伤上限
  max_jing: int
  jingli: int      # 精力当前值（可超 max*2 缓冲）
  max_jingli: int
  neili: int       # 内力当前值（<= max_neili）
  max_neili: int
  water: int       # 饥渴值（heal_up 消耗）
  food: int         # 饥饿值
# 不变量：0<=qi<=eff_qi<=max_qi；jingli<=max_jingli*2
```

### 2. SkillDefinition（数据驱动，替换 mapping *action）
```yaml
# kungfu/skill/yuenu-jian.skill.yaml

  - action: "$N施展小巧腾挪之技，一招「白露横江」，$w横削$n$l"
    force: 100
    dodge: 30
    damage: 20
    lvl: 0
    skill_name: 白露横江
    damage_type: 割伤
  - action: "$N忽发一招「电照长空」..."
    force: 120
    dodge: 20
    damage: 30
    lvl: 10
    damage_type: {random: [刺伤, 劈伤]}  # 原 random(2)?:刺伤:劈伤
# perform/exert 子招式引用

  leiting: {file: yuenu-jian/leiting, min_skill: 200}
```

### 3. ConditionDefinition + EffectInstance（替换 condition 对象 + call_out）
```python
@dataclass
class EffectInstance:        # 可序列化的持续效果（替换 call_out 闭包）
    effect_id: str            # 唯一标识
    type: str                 # perform_effect / condition / dot / buff
    source_entity: EntityID   # 施加者（替换闭包捕获的对象引用）
    target_entity: EntityID
    duration_ticks: int       # 剩余 tick
    tick_interval: int        # 每 N tick 触发一次
    payload: dict             # 携带数据（lvl/jiali/jiajin 等）
    stages: list[Stage]       # 多阶段（如降龙一击：蓄力->命中->结算）
    rng_seed: int             # 所属战斗的 RNG 种子（确定性）

@dataclass
class ConditionInstance:      # 状态效果实例
    condition_id: str        # killer/bandaged/snake_poison...
    duration: int            # 原始 duration
    info: Any                # 携带信息
    no_heal_up: bool         # CND_NO_HEAL_UP 标志
```

### 4. DamageFormulaConfig（替换魔法数字）
```yaml
skill_power:
  base: "level**3 / 3"        # 可改为 level**2
```

## 分布式扩展策略
## 分布式扩展与分片策略

### 1. 战斗亲和与 Coordinator Actor
- 同房间战斗参与者保持同节点亲和，COMBAT_D 实例化为 region/战斗级 CombatCoordinator Actor 串行裁决回合，避免分布式事务
- 玩家进入新 region 时迁移 Player Actor（先激活后注销，消息日志重放），战斗中迁移需冻结回合 -> 快照 combat state -> 目标分片恢复
- 组队战斗批量迁移保持队形，Coordinator 跟随迁移

### 2. 确定性随机与回放
- 每场战斗分配独立 seeded RNG（random.Random(combat_seed)），回合结果写 append-only 事件日志
- 这是分布式下调试与公平性的前提：任意时刻回放重建战报，反作弊审计
- 原 random() 调用点全部收口到 CombatRNG，按调用顺序推进种子，保证可复现

### 3. tick 分片并行
- TickSimulator 按 region/分片独立推进，单分片内串行保证战斗一致性，跨分片并行
- 分片内 tick 按子系统优先级批处理：战斗 tick -> 恢复 tick -> condition tick，单分片单线程串行
- 单分片容量上限 5000-8000 并发战斗实体，以此估算分片数

### 4. Effect 持久化支持迁移
- EffectInstance 可序列化，对象迁移时 Effect 随实体快照迁移到目标分片
- 替换 call_out 闭包（捕获对象引用无法跨进程），Effect 由目标分片 TickSimulator 重新接管驱动
- Effect 的 rng_seed 跟随战斗，迁移后确定性不变

### 5. 事件日志跨节点
- 战斗事件（CombatRoundResolved）写入分片本地 + 聚合到全局事件存储（PostgreSQL append-only）
- 投影层（战报/排行/反作弊看板）独立部署，订阅事件流热更新
- 跨分片 tell/广播经 Redis Pub/Sub 或 NATS，战斗结算不需跨节点（同房间同分片）

### 6. 战斗负载隔离
- 高频战斗 tick 与低频 condition/恢复 tick 分级调度，避免慢 condition 阻塞战斗
- 每请求/每战斗超时 + 资源配额（替换 eval_cost 6 亿全局预算为 per-combat 配额），熔断器保护分片

## 技术选型
- Python 3.11+ asyncio + uvloop 作为仿真与 Actor 运行时主循环，保留 2s tick 为确定性仿真步进单位
- 自研轻量 SparseSet ECS（不依赖 entt-py/esper），按 MUD 访问模式优化，与 WebSocket 后端深度集成
- Pydantic v2 做数据模型校验（SkillDefinition/EffectInstance/Vitals），保证不变量与 schema 演进
- 技能数据驱动用 YAML 定义（mapping *action 的直接映射），行为逻辑用纯 Python 策略类（SkillBehavior 注册表），UGC 脚本逃生舱用 WASM
- 伤害公式引擎用受限表达式 AST 或纯 Python lambda 求值，DamageFormulaConfig 可热重载
- 事件总线用 dataclass 事件 + asyncio.Queue 单分片内分发；跨分片用 Redis Stream（热）+ NATS JetStream（扇出）
- 确定性 RNG 用 random.Random(seed) 每场战斗独立实例，调用点全部收口到 CombatRNG
- 事件溯源存储 PostgreSQL append-only events 表 + snapshots 表；战斗回合聚合 CombatRoundResolved 而非逐 tick
- WASM 沙箱（wasmtime-py）承载 UGC 技能脚本，燃料计量 + 能力化，对齐原 eval-cost 语义但可配置 per-CPK 配额
- CombatRenderer 文案用 Jinja2 模板或纯数据表 + token 替换函数，代词替换（$N/$n/$P/$p）保留为纯函数渲染层

## 风险
- 数值平衡迁移风险：skill_power 的 level³/3 三次方增长导致等级差放大成数量级碾压，新架构虽配置化但若直接复刻原公式则平衡问题不变；需在迁移期用原公式保证手感一致，再逐步调参。jingli_bonus 两版不一致（combatd /20 vs s_combatd /2）需先确定哪个是'正确'行为
- perform call_out 效果迁移工作量极大：359 个技能中大量 perform 用递归 call_out 实现多阶段效果（如降龙一击的 msg_display->remove_effect 链），每个需手动分析闭包捕获的状态、重建为 Effect 对象的多阶段定义，易遗漏隐式状态
- hit_ob 多态返回迁移脆弱：原 hit_ob 返回 string|int|mapping 三态，combatd 中 5 层 if(stringp)/else if(intp)/else if(mapp) 嵌套，迁移到统一 EffectResult 需穷举所有技能的返回形态，遗漏即静默丢失伤害/文案
- 确定性随机与原 random() 行为对齐：原 random() 调用顺序散布在 do_attack 全流程（limbs/ap+dp/ap+pp/damage/damage_bonus/defense_factor 循环），迁移时调用顺序若不一致则回放结果与原系统不符，反作弊审计基线失效
- 战斗回合计时与原 heart_beat 2s 节奏：原 fight() 的时机判定（random(dex*3)<str*2+speed 决定攻击/防御/双击）依赖 heart_beat 2s 节拍，固定 tick 仿真需保持相同节奏否则战斗手感变化，玩家可能感知差异
- call_out 替换为 Effect 后取消语义差异：原 remove_call_out 按 callback+target 取消，Effect 取消需按 effect_id，perform 中若用 set_temp 旗标控制效果流转（如 xianglong 旗标），迁移时旗标与 Effect 生命周期需精确对应
- 门派加成 if-else 迁移数据化可能遗漏边界：human.c:100-199 十余分支含嵌套条件（年龄/技能等级/死亡次数组合），迁移为 FamilyDefinition 时需逐分支验证，遗漏即门派属性错误
- s_combatd.c SRPG 原型共存决策：s_combatd 95% 重复但含未采用的双手互博/阵法合击原型，迁移时需判断哪些特性要保留（如 double_attack 已被 combatd 采用），哪些废弃
- 分布式战斗迁移的回合一致性：战斗中玩家跨分片迁移需冻结回合，但对方可能正在 do_attack 中段，需保证迁移窗口选择在回合边界（CombatRoundResolved 后），否则状态撕裂
- ECS 组件粒度与查询性能：战斗热路径每 tick 查询 Vitals/Attributes/ModifierStack/CombatState 多组件，SparseSet 设计需保证缓存局部性，否则 Python 解释器开销下高频 tick 性能不达标（需 uvloop + __slots__ 优化）

**裁定**：sound（方向正确，但执行层有 1 个已验证的数据模型缺陷 + 多个高影响未解细节，需在落地前修订）

### 问题与修复
- **ConditionHandler.on_tick 返回值设计为互斥枚举 CONTINUE/EXPIRE/NO_HEAL，与原系统位标志语义冲突。原 condition.h 定义 CND_CONTINUE=1、CND_NO_HEAL_UP=2 为可组合位（feature/condition.c:64 `update_flag |= flag` 跨所有 condition 做 OR 聚合，char.c:149 `cnd_flag & CND_NO_HEAL_UP` 检查聚合后的位）。即 condition A 返回 NO_HEAL_UP、condition B 返回 CONTINUE 时，聚合 flag=3 同时表示'继续存活'且'抑制恢复'。互斥枚举使二者不可共存，丢失跨 condition 的位聚合语义。**
  - 影响：中毒等状态在原系统中通过 update_flag |= flag 跨 condition 聚合后抑制 heal_up（char.c:149），新模型无法表达'多个 condition 中任意一个抑制恢复'的聚合语义，会导致中毒/重伤时仍恢复血量，破坏战斗平衡与原手感。这是已验证的数据模型缺陷。
  - 修复：将 ConditionHandler.on_tick 返回值改为位标志或组合结构，如 on_tick(...) -> ConditionTickResult(continue: bool, suppress_heal: bool, expire: bool)，并显式定义 ConditionSystem 聚合规则：aggregate.suppress_heal = any(c.suppress_heal); aggregate.continue = any(c.continue)。不要用互斥 enum。这是数据模型必须先修正的设计缺陷。
- **确定性随机回放（seeded RNG 按 random() 调用顺序推进）与纯函数管线重构存在不可调和的张力，且设计未给出取舍。do_attack 内有 29 处 random() 调用（combatd.c 实测），含可变迭代次数的 while 循环（combatd.c:637-641 `while random(defense_factor)>my_exp: damage-=damage/3; defense_factor/=2`），以及 riposte/double_attack 递归（combatd.c:772,777,812）在同 tick 内顺序消耗 RNG。任何控制流简化、提前返回、follow_up 异步化都会改变 RNG 消耗序列，导致回放结果与原系统不符。**
  - 影响：反作弊审计基线（设计宣称的核心卖点）对迁移内容失效；任何为可读性做的控制流简化都会改变 RNG 消耗，使'纯函数管线'与'确定性回放'两个目标直接冲突。要么牺牲代码清晰度复刻原控制流，要么放弃回放保真--设计同时承诺两者但未给出取舍。
  - 修复：明确分层：(1) 新内容/UGC 强制走 CombatRNG 契约，保证可回放；(2) 遗留内容采用'行为等价而非位等价'迁移标准--保留概率分布与伤害量级，放弃逐 random 对齐；(3) 保留一份原 random() 全局序列的'黄金战报'做回归基线，用统计分布比对（伤害均值/方差/命中率）而非逐回合比对。在 risks 中将此项从'对齐风险'改写为'显式放弃逐位复现'的决策。
- **战斗热路径事件总线 + Pydantic 校验存在叠加性能风险，未量化。do_attack 一次攻击会发 AttackResolved/DamageApplied/SkillImproved/ThreatChanged/多条 message 等约 5+ 事件，5000 实体×1 攻击/2s≈12500 事件/s/分片，全部经单 asyncio.Queue 分发，且每个事件是 dataclass 分配。同时设计称'用 Pydantic 校验不变量'，若 apply_damage() 每次 mutate 都触发 Pydantic 校验，在战斗热路径上是灾难性的。**
  - 影响：原系统 receive_damage/message_vision 是 O(1) 进程内直接 mutate+渲染，新模型改为每次受击走 发布+分配+订阅，在 Python 单线程 asyncio 下是显著性能回退。若同时叠加 Pydantic 每次变更校验不变量，热路径吞吐可能不足以支撑宣称的单分片 5000-8000 并发战斗实体，'高并发承载'目标有落空风险。
  - 修复：(1) 战斗热路径绕过事件总线：resolve_attack 直接返回 CombatRoundResult（含 damage_delta/messages/effects），由 CombatSystem 同步应用并就地通知同房间订阅者，仅在 tick 边界发一次 CombatRoundResolved 到总线供投影层；(2) Vitals 运行时用裸 dataclass + 断言，Pydantic 仅在 snapshot 落盘/UGC 上传/跨分片反序列化时校验；(3) 给单分片定一个实测基线（如单 do_attack 端到端 μs 级目标）再定容量上限，不要先承诺 5000-8000。
- **CombatCoordinator 同房间串行裁决在热门房间是串行瓶颈，且 MAX_OPPONENT=4 不约束房间人口。原系统每实体独立 heart_beat 天然并行；新模型要求同房间战斗参与者同节点亲和、由单 Coordinator 串行裁决回合以避免分布式事务。对一个 50 人 PK 混战或 20 人 boss 战的房间，该房间所有战斗回合排队过单 Actor，单房间吞吐上限远低于分片均值。**
  - 影响：原系统每实体 heart_beat 独立、天然跨实体并行；新模型对单房间强制串行，在热门房间反而是性能回退。5000-8000/分片假设负载均匀分布，单个热门房间的串行瓶颈会拖垮该分片响应延迟。
  - 修复：明确 hot-room 退路：当房间参与者超阈值（如 >30）时，将房间战斗拆为多个子 Coordinator（按对手对分组），或对超大规模群战启用'简化回合模式'（批量结算而非逐对裁决）。在容量规划中给出单房间串行上限与降级策略，不能假设房间负载均匀。
- **call_out 迁移为 Effect 的取消/互斥语义不完整。原 perform 用 set_temp 旗标控制效果互斥（xianglong.c:33-35 `if me->query_temp("xianglong") return` 防止重复施展，xianglong.c:58 set_temp、xianglong.c:155 delete_temp 配对）。Effect 模型按 effect_id 管理，缺少'同类型效果唯一'约束。降龙一击的 parry 加成 Effect 若被重复施加，apply/parry 修饰符会错误叠加。设计仅提到'支持取消（按 effect_id）'，未定义互斥/覆盖语义。**
  - 影响：若 EffectSystem 仅按 effect_id 管理而无'每类型唯一'约束，降龙一击类效果可能重复施加（parry buff 翻倍、伤害结算多次），或前一个 Effect 被覆盖时其 apply 修饰符未正确回滚，造成属性永久漂移。这是正确性问题。
  - 修复：将唯一性/覆盖策略作为 EffectSystem 一等公民：EffectDefinition 声明 stacking_policy(unique|refresh|stack|independent)，EffectRegistry.apply 时按策略处理（unique 则拒绝/替换已存在的同 type effect，并负责清理前者的 apply 修饰符）。不要把 set_temp 旗标留给调用方手工维护。
- **跨战斗延续的 Effect 与 rng_seed=所属战斗 模型冲突，且 tick 计时与原 call_out 墙钟计时的崩溃恢复语义未定义。sanhui.c:192 `call_out("receive_heal", amount*sanhui_busy/3, me)` 是跨战斗延续的恢复效果；heal_up 等也用全局 random()。设计将 Effect.rng_seed 绑定到'所属战斗'，但效果可能在战斗结束后继续 tick。同时 duration_ticks 以 tick 为单位，服务器宕机期间不推进 tick，恢复后按 tick 数继续；而原 call_out 基于墙钟，宕机时长会扣减剩余 delay。**
  - 影响：断电重启后，基于 tick 的中毒/增益会按'服务器在线时长'而非'真实经过时长'结算，与原 call_out 行为不一致，影响平衡（原 2 分钟毒在重启后实际毒更久）。跨战斗延续效果用已结束战斗的种子在确定性审计上产生歧义。
  - 修复：(1) Effect.duration 增加时间模式字段(mode: tick|wallclock)，wallclock 模式按真实经过时间衰减，适配 sanhui/receive_heal 类效果；(2) 崩溃恢复策略明确：按 downtime 上限 clamp 跳过的 tick 数（如最多补跑 N tick），超限则按比例衰减剩余 duration，避免瞬间结算；(3) EffectInstance 增加字段记录'所属战斗是否存活'，跨战斗延续时切换为房间级/全局 RNG 而非沿用已结束战斗的种子。
- **TickSimulator '仅活跃实体进队'缺少 IdlePredicate 的明确归属，且 set_heart_beat(0) 自停语义碎片化。原 char.c:149-157 集中判定 `!is_fighting() && !interactive() && heal_up() 返回无更新` 时 set_heart_beat(0) 让实体睡眠。新架构中'是否空闲'需聚合 Vitals（满血？）、ConditionStack（无活跃 condition？）、CombatState（脱战？）、连接状态（非交互？），这些状态分属不同系统，设计中无组件负责计算该聚合谓词。**
  - 影响：缺少明确的 idle 判定归属会导致：要么每 tick 全量扫描所有实体判定是否出队（CPU 浪费），要么各系统各自维护导致判定不一致（实体已满血但仍在战斗队列，或脱战但 condition 未清空仍 tick）。原 char.c 集中判定语义在新架构中无处安放。
  - 修复：TickSimulator 持有 IdlePredicate，查询组件聚合判定：idle = vitals_full AND no_active_condition AND not_in_combat AND not_interactive。由 DamageSystem（满血时）/ConditionSystem（清空时）/CombatSystem（脱战时）发 ActivityChanged 事件，TickSimulator 据此入队/出队，避免每 tick 全量扫描。明确此判定归 TickSimulator 所有。
- **rioste/double_attack 的 follow_up 与迁移回合边界存在状态撕裂窗口。设计称迁移选在'CombatRoundResolved 后'的回合边界，但 CombatRoundResult.follow_up 字段表示同 tick 内还有待处理的反击/连击（combatd.c:772,812 递归 do_attack）。follow_up 是回合内的子步骤，在主 CombatRoundResolved 与 follow_up 处理之间存在非原子窗口。**
  - 影响：若迁移发生在主攻击结算后、follow_up 处理前，反击上下文丢失或在不同分片重建，导致 RNG 序列与状态撕裂，回放与公平性均受影响。设计将'回合边界=CombatRoundResolved 后'作为安全点，但 follow_up 使该边界不原子。
  - 修复：迁移前明确：只有当 follow_up 队列清空（resolve_attack 返回的 CombatRoundResult.follow_up 为 None）且当前 tick 所有参与者的攻击均结算完成，才算安全迁移点。CombatCoordinator 暴露 in_flight_attacks 计数，迁移协调器等待其归零。在 CombatRoundResolved 之外增加 RoundQuiesced 事件作为迁移信号。
- **门派 FamilyDefinition 公式求值的安全面与 UGC 沙箱策略不一致。设计对 UGC 技能脚本用 WASM 沙箱（正确），但 family_definition.yaml 的 max_qi/max_jing 加成公式用'纯 Python lambda 或受限表达式 AST'。UGC 平台目标下，门派/题材定义最终由用户配置，若用 Python lambda 求值则在主进程执行不可信代码。**
  - 影响：UGC 作者可通过门派/伤害公式注入恶意或病态表达式（如 level**100 制造数值溢出、无限循环拖垮 tick），绕过设计为技能脚本准备的 WASM 沙箱。公式求值是 UGC 攻击面的薄弱环节。
  - 修复：统一安全边界：family_definition/skill_definition 的公式若面向 UGC，同样进 WASM 沙箱或受限 AST（白名单运算符、无循环、深度上限）；Python lambda 仅用于可信内部配置。对所有用户可配置的表达式统一燃料计量+能力化，对齐 eval_cost 语义。

### 改进建议
- 先修 ConditionHandler 数据模型：on_tick 返回组合结构 (continue/suppress_heal/expire) 而非互斥枚举，并显式定义跨 condition 位聚合规则。这是迁移前必须落定的契约，否则 72 个 condition 逐个迁移时会埋入隐性 bug。
- 将'确定性回放'分层而非全量承诺：新内容/UGC 强制 CombatRNG 契约保证可回放；遗留内容采用'行为等价'标准（概率分布与伤害量级对齐，放弃逐 random 位对齐），用黄金战报做统计回归而非逐回合比对。把 risks 中该项从'对齐风险'改写为'显式放弃逐位复现'的决策。
- 战斗热路径绕过事件总线：resolve_attack 同步返回 CombatRoundResult 并就地应用+通知同房间订阅者，仅在 tick 边界发一次 CombatRoundResolved 到总线供投影层。Vitals 运行时用裸 dataclass+断言，Pydantic 仅在 snapshot 落盘/UGC 上传/跨分片反序列化时校验。给单分片定实测基线（单 do_attack μs 目标）再定容量上限，不要先承诺 5000-8000。
- 定义 EffectSystem 一等公民的 stacking_policy(unique|refresh|stack|independent)，迁移时把 set_temp 旗标（如 xianglong）映射为 unique 策略由 Registry 统一管理，调用方不再手工维护互斥。
- Effect 增加时间模式字段(mode: tick|wallclock) 与跳过 clamp 策略，跨战斗延续效果切换为房间级/全局 RNG；迁移协调器等待 RoundQuiesced（follow_up 清空）而非仅 CombatRoundResolved。
- TickSimulator 持有 IdlePredicate 并通过 ActivityChanged 事件驱动入队/出队，明确该判定归 TickSimulator 所有，避免每 tick 全量扫描。
- 补齐 s_combatd 迁移清单：double_attack/pixie-jian 已在 combatd.c 内联，仅从 s_combatd 提取阵法合击(teamwork)/自创连续技(combo)为 CombatModifier，其余废弃。
- 统一安全边界：family/skill 公式若面向 UGC 全部进 WASM 或受限 AST（白名单运算符、无循环、深度上限），Python lambda 仅限可信内部配置。
- 建议在正式动手前做一个最小战斗 spike：单房间 50 实体群战在 asyncio 单线程 + 事件总线 + Pydantic 边界校验下的实测 tick 耗时，用数据验证 2s tick 预算与 5000/分片假设，再决定是否需要 tick 内分片并行或事件路径短路。
