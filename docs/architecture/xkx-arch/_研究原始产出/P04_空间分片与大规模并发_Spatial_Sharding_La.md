# 模式研究·空间分片与大规模并发 (Spatial Sharding & Large-Scale Concurrency for XKX MUD)

## 概述
基于对XKX的6414房间/41区域出口图分析，设计区域共同体分片+房间图AOI方案：将强连通区域聚成分片单元，以1跳房间邻域为AOI，go命令跨区时做玩家状态快照迁移。采用无状态WebSocket网关+有状态游戏分片架构，全局守护进程(channeld/natured)迁出为共享状态服务，热区city做微分区。

## 模式
- **区域共同体分片 (Area-Community Sharding)**：XKX世界为房间图(exits映射)而非坐标空间，41个区域间出口密集交错(如city连接20个区域、beijing连接7个)。直接按单区域分片会产生大量跨分片出口。正确做法是对房间出口图做社区检测(如Louvain模块度算法)，将强连通的区域聚成区域共同体作为分片单元。例如可将[city+gaibang+village+huanghe]聚为中原分片，[shaolin+zhongnan+emei+wudang]聚为武林分片，[beijing+changbai+taihu+shenlong]聚为北方分片。分片边界对齐社区割边，使跨分片移动最小化。分片单元可配置(从单区域到多区域共同体)，支持按负载动态调整粒度。
  - 适用性：适用于单节点上限2000-5000同房在线时，将41个区域按出口图凝聚成5-8个区域共同体，每个共同体为一个分片。city作为枢纽需单独处理(见热区微分区)。
- **房间图AOI + 九宫格混合兴趣区域 (Graph-Grid Hybrid AOI)**：MUD无几何坐标，AOI本质是'可见范围'。现有tell_room(房间广播)天然就是1跳AOI单元，say()/message_vision也限定在environment(当前房)。重构时采用双层AOI：(1)室内区域用房间图AOI——玩家订阅以当前房间为根、N跳邻域(默认1跳)内的房间事件，跨过N跳边界的事件不推送；(2)户外/野外区域引入坐标，用九宫格(3x3)AOI，玩家订阅所在格+8邻格的房间事件。N跳图AOI直接对应go.c的move广播(源房mout+目标房min+follow)，迁移天然落在1跳边界上。订阅表存于分片内存(房间ID->{session_id集合})，跨分片AOI通过边界房间的事件代理转发。
  - 适用性：默认AOI半径=1跳(覆盖move广播的双房)；战斗/PK高密度区可降到0跳(仅本房)；空旷野外可放宽到2跳。配合发送端过滤即可生效，无需改造房间数据模型。
- **跨区迁移Handoff (Cross-Shard Player Handoff)**：复刻go.c现有流程但分布化。当前流程：查exit→load_object(dest)→me->move(obj)→广播mout/min→follow_me。分布式版：(1)源分片解析exit，发现dest_room属于另一分片；(2)源分片冻结玩家(暂停输入队列、保存战斗状态、清理heart_beat)；(3)快照玩家状态(dbase+inventory+combat+temp状态)序列化为handoff_token，写入共享Redis并附TTL；(4)通知目标分片接管，目标分片restore玩家到dest_room、订阅AOI、重放排队输入；(5)源分片注销会话路由。组队迁移(follow_me)是批量handoff：队长先迁移，成功后按队形批量迁移队员，保持到达顺序。门状态(create_door跨房联动)通过共享状态保证双分片一致。
  - 适用性：玩家跨区域边界移动(go命令出口指向另一分片房间)、传送技能(teleport)、死亡轮回到/d/death、组队跟随批量迁移。
- **热区微分区与负载迁移 (Hot-Zone Micro-Sharding)**：city是星型枢纽，若整区归一个分片则该分片承载所有跨区中转流量，成为瓶颈。方案：(1)将city内部按房间簇(街道/广场/客店等社区)切成多个子分片，各子分片可分布在不同节点；(2)设置'城市入口路由'——所有指向/d/city/的外部出口先到入口路由，由路由按房间簇定位目标子分片；(3)子分片间用本地handoff(同进程或同机房低延迟)。同时启用热点检测：监控每分房在线玩家数/CPU负载，当某子分片超阈值时，动态将该子分片(整房间簇)迁移到空闲节点——迁移时对该房间簇做短暂'冻结迁移'，比逐玩家handoff高效。
  - 适用性：对city(扬州枢纽,连通20区域)、beijing(625房最大)、shaolin、dali等高流量枢纽启用。普通区域保持整区分片。分片迁移粒度为房间簇，可在不停服下进行。
- **无状态网关 + 有状态游戏服 (Stateless Gateway / Stateful World Shard)**：解耦连接与游戏逻辑。网关层(无状态、可水平扩展)：负责WebSocket终结、TLS、认证(LOGIN_D逻辑)、会话(session_id↔当前分片的路由表)、限流、命令协议解析。网关不持有房间/NPC状态，收到玩家命令后查路由表转发到当前所在的游戏分片。游戏分片(有状态)：持有房间对象、房间内NPC/物品、房间内玩家实体、AOI订阅表、房间局部heart_beat。玩家会话绑定到一个游戏分片(当前所在区域的owner)，迁移时变更绑定。网关间共享session路由表(Redis)，支持断线重连时定位玩家所在分片。单网关宕机不影响游戏分片状态，玩家重连即可。
  - 适用性：任何规模部署的标配。网关无状态可任意扩缩；玩家增长先扩网关，分片增长按区域共同体扩。session路由表是唯一协调点，用Redis集群承载。
- **守护进程状态解耦 (Daemon State Decomposition)**：原32个单例守护进程是全局共享状态，是分片最大障碍。按状态性质分三类重构：(A)纯计算无状态(combatd伤害计算、moneyd兑换、chinesed中文、weapond)→每分片本地副本，无需协调；(B)全局只读副本(natured时间天气)→单权威写入+只读副本广播，各分片订阅时钟滴答；(C)全局可变状态(channeld聊天、securityd权限、marryd婚姻、band封禁、fingerd在线状态)→抽离到独立状态服务(Redis/专用微服务)，分片通过远程调用访问。channeld特别处理为pub/sub：玩家发频道消息→发到Redis topic→各分片订阅后向本分片在线玩家投递，实现全局聊天与分片隔离兼容。users()/shout等76处全局遍历调用需逐一改造为pub/sub或查询在线状态服务。
  - 适用性：所有LPC单例守护进程迁移到Python架构时必须分类。无状态类(战斗/金钱/中文)优先副本化；全局状态类(频道/自然/安全/婚姻)必须先迁共享存储再分片。这是分片的硬前置依赖。
- **分层存储与缓存策略 (Tiered Storage & Caching)**：存储按读写模式分层：(1)静态内容(房间蓝图/NPC定义/武功技能=原LPC文件)→对象存储+CDN，各分片只读加载，对应原load_object编译；(2)玩家持久态(dbase存档)→Redis Hash(player:{id})热缓存+PostgreSQL冷存储，原/data/user/<首字母>/<id>.o的分目录结构由哈希分片替代；(3)房间运行态(门状态、掉落物、NPC重置计时)→存于拥有该房间的分片内存，跨界门状态写共享Redis保证双分片一致；(4)全局协调态(分片所有权表room→shard映射、session路由表、在线玩家位置)→etcd/Redis强一致存储。房间reset(1800s)仍由各分片本地调度。
  - 适用性：房间/NPC蓝图(只读)适合CDN缓存；房间运行态(门、掉落物)需共享存储保证一致；玩家实体可缓存但权威在分片。跨界门/传送点是高频共享态。

## 适用性
- 房间图AOI(1跳邻域)：直接落地，与MUD原有tell_room/move广播语义一一对应，无坐标改造即可实现跨房间消息边界控制
- 九宫格AOI：仅适用于户外/荒野区域且需在DSL中为房间补坐标，城市/室内区域不适用
- 区域共同体分片：是MUD分片的主方案，单房间/单玩家为粒度过细、单区域为边界切割过密
- 跨区Handoff：适合玩家跨区域边界移动、组队跟随批量迁移、传送类出口（如vortex、enter到/d/bwdh）
- 热区微分区：仅对city/beijing/shaolin/dali等高连通枢纽区域启用，普通区域无需
- 无状态网关：是连接层水平扩展的必备前置，任意玩家数增长先扩网关
- 守护进程解耦：是分片前必须完成的债，无状态守护进程可每分片副本，全局状态守护进程必须先迁出共享存储

## 权衡
- 分片粒度权衡：区域共同体分片跨分片移动少但单分片大(迁移慢、热点难拆)；单房间分片负载均衡细但跨分片移动多、协调开销大。MUD房间连通性强，共同体分片是合理折中
- 图AOI vs 网格AOI：图AOI贴合MUD语义无需坐标改造，但订阅计算需图遍历(1跳尚可，多跳昂贵)；网格AOI高效但要求房间有坐标，需DSL改造。建议混合：室内图AOI+户外网格AOI
- Handoff延迟 vs 一致性：快照迁移若同步等待目标分片确认会增加移动延迟(玩家感觉'卡顿')；若异步则短暂窗口内玩家可能在双分片同时存在(需要单写者锁防双写)。组队批量迁移放大此问题
- 热区微分区复杂度：city拆子分片虽均衡负载，但城市内部移动频繁(handoff多)，且入口路由是新的单点。需权衡拆分收益与路由开销，可对city先观察负载再决定是否拆
- 守护进程远程化性能：channeld改pub/sub增加一跳Redis延迟(原是进程内调用)；combatd若远程化则每次战斗计算都跨网络。应将纯计算(combatd/moneyd)保留本地副本，仅全局状态走远程
- 无状态网关的会话粘性：网关需查路由表才能转发命令，每次命令一次路由查询(可Redis缓存本地)。若玩家在分片间频繁迁移，路由表抖动；建议会话粘附到分片而非每命令重路由
- 门状态一致性：原create_door是同进程双房联动，分片后跨界门需分布式锁或共享存储，增加跨分片写延迟。可接受因为门操作低频
- DSL/UGC与分片协同：用户用DSL配置的新剧情区域需能动态分配到分片(按预估玩家数选分片或新建分片)，分片方案需暴露动态注册房间的接口给UGC平台

## 推荐
- 分片单元定为'区域共同体'(出口图社区凝聚)，而非单区域或单房间：用Louvain对6414房间出口图跑社区检测得到5-8个共同体作为初始分片，边界对齐社区割边使跨分片移动最小化
- AOI默认半径=1跳房间图邻域(匹配现有tell_room/move广播语义)，户外野外区域在DSL补坐标后用九宫格；无需先做坐标化改造即可落地图AOI
- 跨区Handoff复刻go.c流程并分布化：冻结玩家→快照dbase/inventory/combat/team写入Redis附TTL→目标分片restore→源分片注销；组队迁移按队形批量handoff保持顺序；门状态跨分片走共享存储保证一致
- 对city(扬州)热区做内部微分区，切房间簇成子分片并设城市入口路由；对beijing(625房)、shaolin、dali等高连通枢纽同样处理；动态负载迁移以房间簇为粒度做'冻结迁移'
- 连接层用无状态WebSocket网关(认证/限流/会话路由)水平扩展，session↔分片路由表存Redis支持断线重连定位；游戏分片有状态持有房间/NPC/玩家/AOI
- 分片前置债务：将32个守护进程按无状态(副本)/全局只读(natured)/全局可变(channeld/securityd/marryd)三类解耦，全局状态迁Redis，channeld改pub/sub替代users()全局遍历
- 单分片容量上限设计为5000-8000并发(10万房/20万对象/Python async+uvloop)，以此估算1万在线玩家需5-8分片；玩家会话尽量粘附所在分片，跨分片仅迁移玩家实体与必要快照而非世界数据
- 分片边界房间(区域共同体割边上的房间)建立跨分片事件代理，AOI跨分片订阅通过边界房间转发，避免每条消息都走全局总线
