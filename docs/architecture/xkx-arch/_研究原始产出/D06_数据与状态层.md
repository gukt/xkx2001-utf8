# 方案设计·数据与状态层

## 设计概述
以"事件溯源(CQRS+ES)+ 双层热冷状态缓存"替换 LPC 的 save_object/.o 全量覆盖快照与 dbase 无 schema 内存映射。核心理念：把"写一个变量"从直接改内存解耦为"发布一个领域事件 -> 投影更新状态"，使状态变更可审计、可回放、可回滚。状态分三层：Redis 热态(在线实体内存态，对应 LPC 进程内对象)、PostgreSQL 权威层(append-only 事件流 + 周期快照)、S3/对象存储冷归档(长期离线存档)。dbase 的两层(dbase 持久 / tmp_dbase 瞬时)映射为快照(持久) + Redis TTL key(瞬时)；default_ob 原型回退映射为 Prefab/Archetype 模板克隆；autoload 的 path 数组映射为实体引用快照+懒加载；call_out 非持久定时器映射为持久化调度器；heart_beat 单一全局 tick 拆分为解耦的 System 调度器(批处理+非均匀节流)；reset/clean_up 映射为 desired-state reconcile + 缓存逐出。实体身份与文件路径解耦：world://<region>/<entity-uuid> 统一地址替代 file_name，支持热重载与 UGC 多版本共存。过渡期采用双写旁路：保留 save_object 为真相，在关键写点(receive_damage/pay_money/death_penalty)挂事件发布 shim 零风险积累结构化审计，再逐步切为事件驱动权威。

## 组件
- **EventStore 事件存储**：append-only 领域事件权威持久层。替代 save_object 全量覆盖写。所有状态变更先发事件再投影为状态。事件按聚合(玩家/房间/经济交易)分片，带单调递增 seq 与 seeded RNG 种子，支持任意时刻回放重建状态(反作弊/调试/崩溃恢复)。战斗用回合聚合事件 CombatRoundResolved 而非逐 tick 事件，避免事件爆炸。对应 LPC 的 log_file(PLAYER_DEATH/PKILL_DATA) 升级为结构化事件。（技术：PostgreSQL append-only events 表(events: id, aggregate_id, aggregate_type, event_type, seq, payload JSONB, meta JSONB, causation_id, ts) + 索引; 写入走 WAL 强一致; 高频域(经济)走 Redis Stream 做缓冲再批量落 PG）
- **SnapshotStore 快照存储**：周期性/关键节点全量状态快照，作为事件回放的加速起点。复用 dbase 持久层 vs tmp_dbase 瞬时层分离的思想：@persist 字段进快照，@transient 不进。玩家每30s或关键事件(升级/死亡/退出)存快照，房间在钝化(actor 钝化对应 clean_up)时落盘。快照与事件配合：恢复=最近快照+后续事件重放。对应 LPC save_object 的全量快照语义但增量可回溯。（技术：PostgreSQL snapshots 表(aggregate_id, seq, state JSONB, ts) + S3/MinIO 对象存储存大快照(房间/NPC 复杂状态); 热快照 Redis 缓存）
- **HotStateCache 热状态缓存**：在线实体(活跃玩家/已加载房间/NPC)的内存态缓存，对应 LPC 单进程内存中的对象世界。实体激活时从冷存储加载到 Redis，钝化时回写。dbase=持久 snapshot(可重建)、tmp_dbase=易失 delta(对应 Redis TTL key)。apply/ 修饰符堆叠存为 Redis Sorted Set 实现 ModifierStack 聚合。AOI(1跳房间邻域)内实体状态热缓存，跨分片通过边界房间代理订阅。对应 LPC 内存态 + swap/clean_up 的缓存逐出语义。（技术：Redis Cluster + Hash(实体组件状态) + Stream(热事件扇出) + Pub/Sub(跨节点广播/频道); uvloop 内存态 actor 持有热副本）
- **EntityStateManager 实体状态管理器**：状态调和器：从 SnapshotStore 加载最近快照 + 重放 EventStore 增量事件 = 当前状态。对外提供类型化组件读写接口(替代散落的 set/query('xxx') 字符串键)。写入时发布事件并更新投影；读取时聚合 base+modifier(对应 query 的 raw 与 evaluate 两层)。管理实体激活/钝化生命周期(对应 LPC clone/restore 与 destruct/clean_up)。实现 default_ob 原型回退：组件查询时先查实例，未命中回退 Prefab 模板。（技术：Python asyncio + 自研 SparseSet ECS + dataclass/pydantic 类型化组件; 状态调和用纯函数 fold(events) -> state）
- **SchemaRegistry 组件模式注册表**：类型化组件 schema 定义与校验，替代 dbase 无 schema 的字符串键约定。每个组件(Identity/Vitals/Attributes/SkillSet/CombatState/EquipmentSlots/ModifierStack/Position/Conditions/Inventory)声明字段类型+@persist/@transient 标注。UGC DSL 场景定义注册为 schema 变体。query('combat_exp') 拼写错误在 schema 层即被拦截(原系统静默失败)。支持 schema 版本演进与迁移。对应 dbase.h 的 DEFAULT_DBASE_SIZE 升级为显式类型契约。（技术：pydantic v2 模型(类型校验+JSON schema 导出) + 版本号字段; DSL schema 用 JSON Schema 扩展点）
- **ReferenceStore 引用与关系存储**：替代 LPC object 指针引用与 autoload 重建。实体间关系(队伍/leader/enemy/spouse/师徒/门派/背包内容)存为 ID 引用而非指针，跨进程可寻址。autoload 的 path[:param] 数组升级为实体引用快照+投影：背包存为 inventory_refs 组件，恢复时按引用懒加载实体而非全量 new()+move()。关系图(婚姻/师徒/门派成员)存为图结构支持原子事务提交(替代 marryd 双方 dbase 双写无事务)。（技术：PostgreSQL 关系表 + Neo4j/Apache Age(可选，社交图谱) + Redis Set(在线关系集合); 实体引用为 world://<region>/<room>#<uuid> 统一地址）
- **SchedulerStore 持久化调度器**：替代 call_out 单进程非持久定时器(931文件使用)。延迟任务持久化存储，崩溃不丢失，可跨节点调度。natured 昼夜相位、复活计时、门关闭、状态效果衰减等定时任务存入调度表。保留 remove_call_out 的取消语义(取消复活/关门计时)。任务与实体解耦：对象析构不自动丢失定时器(显式取消)，解决原系统'对象销毁即静默丢定时器'问题。call_out 回调函数升级为可序列化的 Effect 对象。（技术：PostgreSQL scheduled_tasks 表(任务表+乐观锁) + Redis ZSET(近实时触发) + APScheduler/Celery beat(周期任务); 分布式锁防多节点重复触发）
- **ProjectionStore 投影与读模型**：CQRS 读侧：从事件流构建查询优化的读模型，替代 log_file 的 append 日志与 moneyd 的 children() 全量扫描。经济域：Wallet 账本双式记账，cashflow 聚合查询替代 O(n) children() 扫描。排行榜/银行流水/战报/背包投影独立部署，热更新=重建投影不触写侧。反作弊投影：扫描异常(余额负/转账不平/短时暴富)产出看板。前端走 WebSocket 订阅投影变更。（技术：PostgreSQL 物化视图 + Redis 缓存读模型 + 异步 ProjectionRunner 消费事件流更新投影; 投影逻辑热更新(Python 重载模块)）
- **WorldStateReconciler 世界状态协调器**：替代 reset 的'声明式对象清单+系统收敛到期望状态'语义。读取房间的 objects 声明(desired state)，对比当前 inventory(actual state)，reconcile 差异(spawn 缺失/destruct 多余)。分片/按需/事件驱动刷新替代全量扫描所有已加载房间(原 reset 1800s 全扫)。门状态跨分片走集中存储保证双面一致(替代运行时 find_object 双面协调)。（技术：声明式 desired-state 配置 + diff/reconcile 循环; 分片级调度器按区域负载分布触发）
- **ArchiveMigrationService 归档与迁移服务**：处理冷数据归档(长期离线玩家存档下沉对象存储)与 schema 版本迁移。.o 存档迁移脚本：批量解析 245 个 .o 文件为结构化事件+快照入仓。schema 演进：事件版本号 + upcaster 链升级旧事件到新 schema(无此机制则旧存档不可读)。历史事件 TTL 归档到冷存储(S3 Glacier)。（技术：Python 迁移脚本 + Alembic schema 迁移 + S3 生命周期策略; 事件 upcaster 模式）

## 关键接口/事件
- CommandBus.dispatch(aggregate_id, command, ctx) -> Result: 命令写入入口，替代 save()/set() 直接改库。命令经校验后产生事件。ctx 携带能力令牌(替代 previous_object 隐式信任链)
- EventStore.append(aggregate_id, events[]) -> seq: 原子追加事件，返回单调 seq。带乐观并发控制(expected_seq)防并发写冲突
- EventStore.load(aggregate_id, from_seq?) -> events[]: 加载事件流用于状态重放
- SnapshotStore.save(aggregate_id, seq, state) / load(aggregate_id) -> (seq, state): 快照存取，作为回放加速点
- HotState.get(aggregate_id, component_type) -> Component / set(aggregate_id, component) / invalidate(aggregate_id): Redis 热态读写，实体激活/钝化时与冷存储同步
- EntityStateManager.activate(entity_id) -> Entity / passivate(entity_id): 实体激活(冷->热，加载快照+重放事件)与钝化(热->冷，存快照+逐出Redis)，对应 LPC clone/restore 与 destruct/clean_up
- EntityStateManager.get_component(entity_id, ComponentType) -> Component: 类型化组件查询，替代 query('str')。内部聚合 base+modifier(对应 query raw/evaluate 两层)与 default_ob 原型回退
- SchemaRegistry.register(component_schema) / validate(component) -> errors: 组件 schema 注册与校验，替代无类型字符串键
- ReferenceStore.resolve(ref) -> entity_id / link(src, rel, dst, txn) / unlink(...): 实体引用解析与关系事务操作，替代 object 指针与 move()
- SchedulerStore.schedule(task_id, at, effect, owner_id) / cancel(task_id) / reschedule(...): 持久化定时任务，替代 call_out/remove_call_out
- ProjectionStore.subscribe(projection_name, event_types[]) / rebuild(projection_name): 投影订阅与热重建
- WorldStateReconciler.reconcile(room_id, desired_state): 房间状态收敛，替代 reset() 全量扫描
- 关键事件类型: CombatRoundResolved(回合聚合)、DamageDealt/Healed、CharacterDied/Unconcious、MoneySpent/Received/Transfer、SkillUp/LevelUp、StatChanged、ItemEquipped/Unequipped、Moved(room->room)、ConditionApplied/Expired、TeamFormed/Dismissed、MarriageEstablished/Dissolved、DoorStateChanged、RoomReset、DayPhaseChanged

## 数据模型
见上方各组件 tech 字段。核心表：events(append-only 权威)、snapshots(加速点)、entity_state(热投影)、scheduled_tasks(持久定时器)、projections(读模型)、references(关系图)。冷热分层：Redis 热态(在线实体+AOI+会话) -> PostgreSQL 温态(活跃事件流+快照+关系) -> S3 冷态(归档事件+长期离线存档)。dbase 的两层(dbase持久/tmp_dbase瞬时)映射为 snapshot(persist) + Redis TTL(transient)。default_ob 原型回退映射为 Prefab 查询时的实例->模板回退链。autoload 的 path 数组映射为 inventory_refs 组件+懒加载实体。query 的 functionp 自动 evaluate 映射为计算型组件 property 描述符。

## 旧->新映射
- save_object(file) 全量覆盖写 .o 文本快照(quit/death/save触发) -> **SnapshotStore.save(aggregate_id, seq, state) + EventStore.append 变更事件; 自动定期快照替代手动触发**（原系统崩溃即丢未保存进度; 新架构事件实时落盘+周期快照, 崩溃只丢最后一个 tick delta）
- restore_object(file) 从 .o 恢复全部 dbase -> **EntityStateManager.activate = SnapshotStore.load 最近快照 + EventStore.load(from_seq) 重放增量**（快照加速避免全量重放; 原系统无增量可重放）
- F_DBASE dbase 持久 mapping(无 schema 字符串键) -> **类型化 ECS 组件(Identity/Vitals/Attributes...)经 SchemaRegistry 校验, @persist 字段进快照**（query('combat_exp') 拼写错误原静默失败; 新架构 schema 层即拦截）
- F_DBASE tmp_dbase static 瞬时 mapping(不持久) -> **Redis TTL key 热态(transient) + @transient 组件标注不进快照**（保留持久/瞬时两层分离思想; tmp_dbase 的 apply/ 修饰符堆叠存为 Redis Sorted Set）
- F_DBASE '/' 路径嵌套访问 _set/_query/_delete -> **嵌套 dataclass/pydantic 模型 + property 描述符; 复杂路径用 JSON-Patch delta 同步前端**（路径访问语义保留但类型化; 前端状态同步复用此路径键发 snapshot+delta）
- default_ob 原型回退(clone 查 blueprint 未命中属性) -> **Prefab/Archetype 模板: 组件查询先查实例副本, 未命中回退只读 Prefab 定义**（升级为完整原型链; UGC DSL 场景定义即 Prefab, 多题材(武侠/航海)为不同组件集）
- query(prop) 对 functionp 自动 evaluate(惰性计算) -> **计算型组件用 property 描述符/@computed 装饰器延迟求值**（保留惰性/计算型组件思想）
- autoload path[:param] 数组 + new(file)+move() 重建背包 -> **inventory_refs 组件存实体引用 ID, 恢复时懒加载实体而非全量实例化**（原系统对象引用不存档变0靠autoload重建; 新架构引用可序列化跨进程）
- object 指针引用(队伍/leader/enemy/spouse/environment) -> **world://<region>/<room>#<uuid> 统一地址 + ReferenceStore 关系表**（分布式核心障碍: 指针不能跨进程; 改 ID 引用是迁移前提）
- call_out(回调函数, delay) 非持久定时器(931文件) -> **SchedulerStore.schedule(task_id, at, Effect对象, owner_id); Effect 可序列化**（原系统对象销毁/崩溃即丢定时器; 新架构持久化+显式取消。回调闭包升级为 Effect 对象）
- remove_call_out(name) 取消定时器 -> **SchedulerStore.cancel(task_id)**（取消语义保留(取消复活/关门计时)）
- heart_beat 2s 单进程全局 tick(战斗/治疗/条件/衰老混做) -> **解耦的 System 调度器: CombatSystem/HealSystem/ConditionSystem 独立步进, 非均匀调度(tick=5+random(10) 降频保留)**（原单点瓶颈; 新架构按系统分拆可并行可分布式分片。set_heart_beat(0) 熄火映射为事件驱动仅活跃实体 tick）
- reset() 1800s 全量扫描已加载房间 repop inventory -> **WorldStateReconciler 分片/按需/事件驱动 reconcile(desired_state vs actual)**（保留'声明式清单收敛到期望状态'思想; 替代全量扫描瓶颈）
- clean_up 180s destruct 空闲对象 + swap 120s 换出 swapfile -> **Actor 钝化(存快照+逐出Redis); 无状态 pod + 外部状态存储, 无本地 swapfile**（保留 clean_up 环境感知回收(检查 deep_inventory 有无玩家); 映射为带活跃度感知的缓存逐出）
- log_file(PLAYER_DEATH/PKILL_DATA/WRITE_LOG) append 文本审计 -> **结构化领域事件入 EventStore + ProjectionStore 构建审计读模型**（零风险双写旁路过渡: 保留 log_file 为真相, 挂事件发布 shim）
- moneyd.query_total_xkx_cashflow children() O(n) 全量扫描 -> **ProjectionStore 经济投影: Wallet 账本双式记账聚合查询 O(1)**（原系统每次NPC/银行发放前都 O(n) 扫描; 新架构账本聚合）
- death_penalty 直接改 dbase 多维属性 + victim->save() -> **CharacterDied/DeathPenalty 事件 + 投影更新; 惩罚策略可声明配置(支持UGC多题材)**（原硬编码于 COMBAT_D; 新架构拆为可配置策略）
- marryd 双方 dbase 双写无事务 -> **ReferenceStore 关系事务原子提交(MarriageEstablished 事件)**（原系统 break_marriage 单边解除即不一致）
- save_data()/make_data_dir() 死代码路径(inherit/save.c) -> **废弃; 由 SnapshotStore 统一管理持久化路径**（确认死代码: MASTER_OB->make_data_dir 从未定义, 注释 'never called'）
- data/user/<首字母>/<id>.o 文件路径分片 -> **PostgreSQL 分片键(aggregate_id hash) + Redis 一致性哈希路由; 玩家会话粘附所在分片**（保留首字母分片思想扩展为分布式分片键）
- data/npc/<name>.o NPC 存档 / data/securityd.o 守护进程存档 -> **统一为 entity/aggregate 快照; securityd 状态迁 RBAC 服务+DB**（原仅 securityd 持久化; 新架构所有守护进程状态外部化）

## 分布式扩展策略
分片单元定为'区域共同体'(出口图社区凝聚)，而非单区域或单房间：用 Louvain 对 6414 房间出口图跑社区检测得到 5-8 个共同体作为初始分片，边界对齐社区割边使跨分片移动最小化。每个分片是独立的有状态游戏 worker(持有该区域房间/NPC/玩家/AOI)，PostgreSQL 与 Redis 按分片键(aggregate_id hash 或 region_id)水平分库。玩家进入新 region 时迁移其 Player Actor 到目标节点(先激活目标节点->快照写入 Redis 附 TTL->目标分片 restore->源分片注销)，对应 LPC move() 的分布式化。1跳房间邻域 AOI 的状态热缓存，跨分片事件通过边界房间代理转发避免每条消息走全局总线。全局只读状态(natured 昼夜相位/channeld 频道配置)迁出为共享状态服务存 Redis，channeld 改 pub/sub 替代 filter_array(users()) 全量遍历。单分片容量上限设计为 5000-8000 并发(10万房/20万对象)，1万在线玩家需 5-8 分片。玩家会话粘附所在分片，session↔分片路由表存 Redis 支持断线重连定位。经济域(高频写)与社交域(低频高价值)可按域独立分片到不同库。冷热分层：热(在线实体 Redis) -> 温(活跃事件流+快照 PG) -> 冷(归档 S3)。读模型投影独立部署可水平扩展读。

## 技术选型
- PostgreSQL 16: 权威事件存储(events/snapshots 表 append-only + JSONB) + 关系数据; 成熟 WAL 强一致、JSONB 灵活 schema、运维成本低。单区域 MUD 用 PG 足够，不过早引入 Kafka
- Redis 7 Cluster: 热状态缓存(实体组件 Hash + 修饰符 Sorted Set) + 会话路由表 + Pub/Sub 跨节点广播 + Stream 热事件扇出; 替代 LPC 单进程内存态
- S3/MinIO 对象存储: 大快照(房间/NPC 复杂状态)与冷归档(长期离线玩家); 低成本海量
- Python asyncio + uvloop: 自研轻量 Actor 运行时 + SparseSet ECS; 实体激活/钝化与状态调和用纯函数 fold(events)->state
- pydantic v2: 类型化组件 schema 校验 + JSON schema 导出; 替代 dbase 无类型字符串键, 拼写错误编译期拦截
- APScheduler + Redis ZSET: 持久化调度器替代 call_out; 近实时触发走 ZSET score, 周期任务走 beat
- NATS JetStream(可选): 跨分片事件扇出与 UGC 多租户隔离; 单区域可用 Redis Stream 起步, 规模化后引入 NATS
- Alembic: schema 版本迁移; 事件 upcaster 链处理旧事件版本演进
- 不选 entt-py/esper: 自研 ECS 针对 MUD 访问模式优化, 与 WebSocket 后端深度集成, 避免外部依赖风险
- 不选 Akka/Orleans(非Python): 跨语言成本高; 借鉴 Orleans Virtual Actor 语义(按ID寻址/按需激活/钝化)自研轻量运行时

## 风险
- 同步返回值迁移成本极高: LPC 大量代码依赖 move() 后立即检查 environment()、call_other 同步返回值, 改异步消息/事件后需重写调用链。缓解: 阶段一单进程 asyncio actor 适配器封装现有同步语义, 逐子系统替换而非全量重写
- 事件爆炸风险: 战斗逐 tick 事件会产生海量数据。缓解: 战斗用回合聚合 CombatRoundResolved + 连续态快照, 不逐 tick 发事件; 经济域先行验证 ES 可行性
- 状态一致性窗口: 事件写入到投影更新存在延迟(CQRS 最终一致), 玩家可能读到旧状态。缓解: 写后读走强一致读(读 EventStore 最新 seq)+投影缓存短 TTL; 关键操作(交易/存档)用同步投影
- schema 迁移断裂: 8400 LPC 文件的 dbase 无 schema, 迁移时字段名/类型缺失会静默失败。缓解: 先解析全量 .o 存档建立字段基线集, schema 注册表渐进式补全; upcaster 链处理旧事件版本
- Redis 单点/容量瓶颈: 热状态全在 Redis, 单实例容量与故障影响全服。缓解: Redis Cluster 分片 + Sentinel 哨兵; 热点玩家/房间 key 分散到不同 slot; 钝化策略控制 Redis 内存增长
- 分布式随机确定性: 战斗 random() 跨节点不可复现, 影响反作弊回放与公平性。缓解: 每场战斗/region 分配独立 seeded RNG, 种子与结果写入事件日志
- call_out 迁移遗漏: 931 文件使用 call_out, 逐一改造工作量巨大且易遗漏导致定时器丢失。缓解: 静态分析全量提取 call_out 调用点建清单; 适配器层先兼容旧语义再逐步迁移到 SchedulerStore
- default_ob 原型回退语义丢失: 克隆查蓝图未命中属性的隐式回退, 迁移时若 Prefab 与实例组件边界划分不当会改变行为。缓解: 严格区分 @persist(实例可变) vs Prefab(只读默认) 字段; 迁移测试对比查询结果
- 双写旁路期数据不一致: 阶段0保留 save_object 为真相同时双写事件, 若 shim 异常会导致审计事件丢失。缓解: shim 失败只告警不阻断主流程; 定期对账脚本校验事件流与 .o 快照一致性
- UGC 沙箱脚本逃逸: 开放 UGC 脚本(WASM)若资源限制不严可耗尽存储/事件表。缓解: per-CPK 配额(事件数/快照大小/调度任务数)硬约束, 对齐原 eval-cost 但可配置可按作者隔离

## 🔍 对抗验证

**裁定**：risky — 设计对现有 LPC 系统的映射分析极其扎实（已逐项代码验证：8412 文件、245 存档、931 call_out 文件、6414 房间、moneyd children() O(n) 扫描、marryd 双方双写无事务、dbase/tmp_dase 两层、default_ob 回退、query functionp evaluate、autoload path 数组均属实），且 SchemaRegistry/ReferenceStore/SchedulerStore/WorldStateReconciler 等组件方向正确。但核心持久化模型存在根本性问题：把事件溯源(ES)当作所有状态变更的默认持久化模式，在当前系统 MAX_USERS=50（已验证 include/login.h）、目标 1 万在线的 200x 跨度下属于过度工程；Redis 热态权威与 PG 事件权威之间的写入顺序未定义（分布式系统的核心一致性问题被'写后读走强一致读'一句带过）；52985 处同步 call_other 调用与 CQRS 最终一致根本冲突，asyncio 适配器封装无法解决语义级同步依赖。这些不是实现细节而是架构层缺陷，会在生产中表现为数据不一致。

**严重度**：high

### 问题与修复
- **事件溯源(ES)被当作所有状态变更的默认持久化模式，对 MUD 场景属于过度工程化，且与代码库的同步语义根本冲突。**
  - 影响：ES 要求所有写先发事件再投影为状态，但代码库有 52985 处同步 `->` 调用（已验证）与 654 处 environment() 立即检查，典型模式是 `ob->receive_damage(50); if(ob->query("qi")<0) ob->die();`。CQRS 最终一致意味着写后读需走'强一致读 EventStore 最新 seq'，即每次读都打 PostgreSQL——这不是高并发而是数据库瓶颈。对 50 用户系统（MAX_USERS=50 已验证）引入 ES+CQRS 的复杂度税远超收益。
  - 修复：降级为'权威状态库 + 选择性审计日志'而非全量事件溯源。状态变更直接写 PostgreSQL entity_state 表（行级权威）+ Redis 热缓存；仅对高价值领域事件（死亡/交易/婚姻/升级/门派变更）写 append-only audit_events 表做审计。这保留了对 PLAYER_DEATH/moneyd/marryd 的结构化审计诉求，却避免对每一次 qi 变化、每一次 move 都产生事件。ES 仅在需严格回放的子域（如 PvP 战报归档）单独应用，不作为全系统默认持久化。
- **Redis 热态权威 与 PostgreSQL 事件权威 之间的写入顺序未定义，存在根本一致性缺口。**
  - 影响：设计同时声明 Redis 为'热状态权威'、PG 为'append-only 权威持久层'、快照为'加速点'，却未定义写入顺序。若先写 Redis 再异步发 PG 事件：PG 写失败则 Redis 有状态而 PG 无记录（崩溃即永久丢失，违背'事件实时落盘崩溃只丢一 tick'承诺）；若先写 PG 再更新 Redis：52985 处同步调用无法等待异步投影。这是分布式系统的核心一致性问题，被'写后读走强一致读'一句带过。
  - 修复：明确二选一：(a) Redis 仅作读穿透缓存，所有写先入 PG 再回填 Redis（牺牲写延迟换强一致，用 PG NOTIFY 回填）；(b) 接受 Redis 为热态权威 + PG 为审计源，但放弃'PG 事件可重建 Redis'的承诺，改为'快照+事件仅用于冷恢复与审计'，热恢复依赖 Redis 持久化(AOF/RDB)+ 多副本。无论哪种，都必须在设计里写明一致性强度（线性化？读己之写？）与故障窗口数据丢失边界。
- **分布式随机确定性(seeded RNG 回放)在 asyncio 异步调度下不可实现，反作弊回放承诺落空。**
  - 影响：设计承诺 seeded RNG 写入事件以支持'反作弊回放'。但战斗涉及跨实体 RNG（攻击掷骰/闪避/掉落），异步调度不保证 RNG 调用顺序稳定，因此即使种子相同、单场战斗结果也不可复现。'反作弊回放'这一核心诉求在异步模型下不可达，除非引入确定性调度器——而这会抵消 asyncio 的并行收益。
  - 修复：删除分布式随机确定性这一目标，或将其限定为'单分片内同步战斗回合内确定性'（回合内串行执行 RNG 调用，种子写入回合事件），明确放弃跨分片/跨回合的端到端回放保证。反作弊改用'结果合理性范围校验'（如伤害落在统计分布内）而非'逐事件复现'。
- **过早分布式化：为 50 用户的系统设计 5-8 分片的 Louvain 社区分片架构。**
  - 影响：当前 MAX_USERS=50（已验证 include/login.h），moneyd 注释亦假设'about 200 ppl online'。设计目标 1 万在线是 200x 当前量级，却已设计 5-8 分片 Louvain 社区检测、区域共同体迁移、跨分片边界代理。这是典型的为 50x-aspirational 负载预先分布式化，增加大量分布式一致性复杂度（跨分片门状态、边界房间代理、会话粘附路由）却无当前负载压力验证其必要性。
  - 修复：阶段一先单进程 asyncio + PG 单库 + Redis 单实例跑通'类型化组件+ID引用+持久调度+reconcile'四项核心改造（这些才是分布式的前提），验证游戏可玩后再谈分片。分片上线条件应设为'单节点实测承载达到 80%（如 3000 并发）'而非'设计图纸支持 5000'。Louvain 分片留作规模化阶段的执行细节，不进入初期架构。
- **双写旁路迁移策略自相矛盾，无法实现'零风险'过渡。**
  - 影响：设计称'保留 save_object 为真相，挂事件发布 shim 零风险积累'。但若 save_object 为真相，事件即非权威，无法真正验证 ES 路径作为权威的可行性，切流前等于运行两套系统。'shim 失败只告警不阻断'意味着审计事件可静默丢失，与'可审计/可回放'目标矛盾。这是分布式双写反模式：要么有事务要么没有，shim 必然发散。
  - 修复：明确分阶段：阶段0用'影子审计'模式——保留 save_object 为权威，在关键写点(receive_damage/pay_money/death/setup_marriage)旁路写结构化事件做审计积累，明确事件非权威、仅用于离线对账与投影建设；阶段1在审计事件覆盖率达标后，选择单一低风险子域(经济)试点事件权威切换；全量切换需在对账脚本连续 30 天零差异后才推进。删去'零风险双写旁路'措辞——双写必然有窗口风险，应诚实标注。
- **call_out 迁移与 clean_up 环境感知回收的复杂度被低估。**
  - 影响：设计称'静态分析全量提取 call_out 调用点建清单'、'逐子系统替换'。但已验证有 3570 个字符串回调站点(931 文件)+1436 个 clean_up 文件。clean_up 含环境感知游戏逻辑(检查 deep_inventory 有无玩家)，非纯缓存逐出。将'逐文件改造 3570+1436 站点'表述为可'逐步替换'低估了工作量，且 clean_up 的游戏逻辑不能简单映射为'钝化策略'。
  - 修复：精确统计闭包形式（已验证 3570 字符串回调可机械映射，3 处闭包需人工重写），按'函数名+绑定的对象'清单驱动迁移工具生成 Effect 对象。对 clean_up，先抽取其环境感知判定逻辑为声明式配置（如'仅在 deep_inventory 无玩家时可钝化'规则），而非逐文件重写。给出量化迁移预算而非笼统'逐步替换'。
- **ModifierStack 用 Redis Sorted Set 建模是错误的数据结构，丢失修饰符组合语义。**
  - 影响：apply/ 修饰符是带优先级的函数链，query() 按链求值(已验证 evaluate() 调用)，涉及加性/乘性混合组合。Redis Sorted Set 仅给出优先级排序，无法表达'伤害减免 = base * (1-0.3) + 5'这类组合。2052 处 apply_ 实例(已验证)若用 ZSET 建模会丢失组合语义。
  - 修复：建模为带优先级的 modifier 链表/栈(每实体内存态)，落盘时序列化；查询时按 priority+type 聚合(base * prod_mods + sum_additive_mods)。Redis 仅缓存聚合后结果或缓存 modifier 列表，不强行用 ZSET 表达组合语义。
- **'1跳房间邻域 AOI'误用了 MMO 空间 AOI 概念到房间制 MUD。**
  - 影响：MUD 交互是房间作用域(同房间内所有人互见)，非 MMO 空间 AOI。缓存'1 跳邻域房间状态'无必要——只需邻域房间的 exits/描述等静态元数据。这是把 MMO 的连续空间 AOI 硬套到房间拓扑上，增加无谓的跨分片状态同步。
  - 修复：AOI 退化为'同房间实体集合缓存'，跨房间仅缓存静态 exits/description 元数据(只读、可全局共享、无需一致性)。边界房间代理仅在玩家'跨越分片边界'这一低频事件上触发，而非为每个邻域房间状态热缓存。
- **Neo4j/Apache Age 用于社交图谱是过度工程。**
  - 影响：婚姻/师徒/门派等关系类型<10 种，单玩家基数极低(1 配偶/1 师父/少数徒弟)。对此引入图数据库是重度过度工程，PostgreSQL 关系表加索引完全胜任。'可选'措辞掩盖了'是否需要'的根本判断缺失。
  - 修复：全部用 PostgreSQL 关系表(models: relation_type PK, src, dst, props JSONB, created_at)实现，低频查询加复合索引即可。删除 Neo4j 依赖。如未来社交复杂度真正上升(如好友推荐/圈子发现)再评估图库引入，当前连'可选'标注都应移除以免误导选型。
- **CombatRoundResolved 回合聚合丢失逐击归因，削弱设计自身强调的审计/反作弊能力。**
  - 影响：聚合为回合事件后，逐击伤害归因(死亡原因/PvP 击杀归属/连击检测)丢失。receive_damage 有 878 处调用点(已验证)带具体 cause，聚合后投影无法还原'谁最后一击致死'，而死亡归因正是审计/反作弊的关键诉求。
  - 修复：CombatRoundResolved 事件 payload 内嵌该回合的逐击明细数组([{attacker, target, damage, cause, crit}...])，保留归因粒度供投影拆解。同时定义'回合事件大小上限'，超过(如>64KB)自动落 S3 引用、事件只存指针。

### 改进建议
- 核心改造优先级应重排为：先做'类型化组件 SchemaRegistry + ID 引用 ReferenceStore + 持久调度 SchedulerStore + desired-state reconcile'四项（这些是分布式的前提且风险低），ES/CQRS 仅作为经济与 PvP 子域的试点审计机制，而非全系统默认持久化。验证四项在单进程 asyncio + 单 PG + 单 Redis 上跑通后再谈分片。
- 在设计文档中明确写出 Redis 与 PG 之间的写入顺序协议与一致性强度（线性化/读己之写/最终一致），并给出故障窗口下的数据丢失边界。当前'热态权威 vs 事件权威'的并列是设计缺陷而非实现细节。
- 修正'1 万在线需 5-8 分片'的容量规划：当前 MAX_USERS=50，应设分片上线触发线为'单节点实测承载达 80%（约 3000-4000 并发）'而非'设计图纸支持 5000'。Louvain 社区分片降级为规模化阶段的执行细节，不进入初期架构决策。房间数 6414 已验证准确，可保留作为分片输入。
- 对 52985 处同步 call_other 调用做量化迁移预算：统计其中'写后立即读'模式（如 combatd 单文件 66 处已验证）的比例，为该类调用设计同步投影 shim（写后同步等投影刷新）而非依赖最终一致。诚实承认'逐子系统替换 5 万调用点'是年级工作量，在路线图中给出阶段性里程碑而非笼统'逐步迁移'。
- 删除/降级误用的外部依赖：Neo4j 改 PG 关系表；APScheduler 与 Redis ZSET 二选一明确主调度机制；MMO 式 AOI 退化为'同房间实体集合 + 邻域静态元数据只读缓存'。这些是'用了显得先进'而非'解决痛点'的选型。
- 双写旁路阶段应明确标注为'影子审计'（事件非权威、仅离线对账），切流条件设为'对账脚本连续 30 天零差异'，删去'零风险'措辞。
- 保留设计中最有价值的部分：LPC->new 映射表经验证高度准确（save_object/dbase/default_ob/autoload/call_out/marryd/moneyd 全部对得上代码），SchemaRegistry 的拼写错误编译期拦截、ReferenceStore 的 ID 化、SchedulerStore 的持久化与显式取消、WorldStateReconciler 的 desired-state 抽象都是对症且正确的，应作为改造主干。
