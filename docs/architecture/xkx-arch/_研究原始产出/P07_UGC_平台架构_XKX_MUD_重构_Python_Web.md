# 模式研究·UGC 平台架构（XKX MUD 重构：Python+WebSocket 后端 + Web 前端，多题材内容创作平台）

## 概述
将 XKX 现有"文件即对象、目录即命名空间、inherit+feature 组合"的内容模型，重构为以"内容包(CPK)+清单+依赖图"为核心的 UGC 平台：声明式 DSL 描述场景、沙箱脚本处理动态行为、能力权限替代 wizlist、内容寻址版本库管理生命周期、市场分发与计费，多主题世界共享运行时隔离。

## 模式
- **内容包(Content Package, CPK)模型**：将 XKX 现有'文件即对象、目录即命名空间'模型升级为以 ContentPackage(CPK) 为单元。每个 CPK 含 manifest.yaml(元数据: id/name/owner/theme/version/license)、scene/(声明式场景 DSL)、entity/(NPC/物品定义)、script/(沙箱脚本)、data/(存档模板)、asset/(二进制素材: 美术/音效/字体)、deps.yaml(依赖声明)。CPK 是创作、版本、审核、分发、计费的最小单元，取代当前散落在 /d、/clone、/kungfu 的零散文件。包内路径用相对引用(./scene/kedian)，跨包引用用 @pkg:scene/x 形式，彻底消除现有 __DIR__ 硬编码导致的强耦合。
  - 适用性：直接映射现有 /d/<region>/ + /clone/ + /kungfu/ 组织方式；改造时把每个 /d/city、/kungfu/skill/* 这类目录整体打包为一个 CPK，原 __DIR__ 引用升级为包内引用、跨目录引用升级为包间依赖。
- **声明式 DSL + 沙箱脚本双轨**：形式化 XKX 已有的数据驱动模式：房间用声明式 YAML/DSL 描述(set short/long/exits/objects/item_desc/outdoors/cost 直接对应现有 ROOM 属性)。凡需要动态行为的(如石门 do_open、桃花岛 do_yell 召船、valid_leave 门派限制)，用受控脚本钩子(hook)实现而非内嵌 LPC。DSL 层保证 95% 场景无需写代码，创作者只需填表;脚本钩子作为'逃逸口'处理复杂逻辑。DSL 经编译器生成中间表示(IR)，运行时解释执行 IR，天然支持静态校验与可视化编辑。
  - 适用性：迁移时把现存的 set("exits"/"objects"/"item_desc") 与 set("long", @LONG) 批量转译为声明式 DSL；保留 (: closure :) 的动态逻辑迁入沙箱脚本。DSL 需为多题材预留扩展词汇(武侠招式/航海航路/书院课业)，通过 schema 插件分主题加载。
- **资源命名空间 + 能力权限(Capability)隔离**：废弃现有 wizard 等级体系(particle/(arch)/(wiz)/(imm))，改为资源寻址 @owner:pkg 与能力(Capability)令牌模型。每个 CPK 在沙箱中获得的能力显式声明: time-travel(改全局时间)、cross-pkg-read(读他包数据)、net(网络)、fs(文件)、combat-affect(影响他人战斗状态)。能力在审核时被裁剪，运行时由沙箱强制执行。命名空间隔离保证一个用户创作的武侠世界不会污染另一个用户的大航海世界数据，跨包调用必须经显式导入与对端授权。
  - 适用性：替代现 adm/single/master.c + securityd.c 的 valid_write/valid_read + wizlist 机制(本分支 securityd.c 已损坏)。能力权限天然适配多租户与多主题隔离。
- **生命周期状态机 + 审核门**：内容生命周期为 draft(草稿)→test(测试服试运行)→review(人工+自动审核)→published(线上)→deprecated(宣布下架)→archived(归档只读)。每个状态对应不同可见性与运行时沙箱: draft 仅作者可见、test 在镜像世界试跑、review 冻结快照供审核、published 热加载到生产、deprecated 仍可运行但停止推荐、archived 摘出运行时仅留存档。审核包含自动门(依赖完整性/配额预检/敏感词/能力审计)与人工门(剧情合规/版权)。下架触发依赖方告警与优雅迁移。
  - 适用性：直接借鉴 config.xkx 的 maximum evaluation cost、time to reset、time to swap、time to clean up 作为配额起点，固化为 per-CPK 的可配置配额，而非全局常量。
- **内容寻址版本库 + 依赖图**：每个 CPK 发布时打不可变快照(SemVer + content hash)，存入内容寻址存储(类 git object store)。依赖在 deps.yaml 中用 pkg@^1.2 或 pkg@sha256:abc 形式固定，支持范围约束与锁定文件(lockfile)保证可复现构建。依赖图在发布前静态分析，检测循环引用、缺失引用、能力越界。这取代当前'任意 .c 文件 new() 任意路径对象'的隐式依赖，让重构与下架的影响面可计算。
  - 适用性：解决现存痛点: 大量 __DIR__ 引用在文件改名/移动时静默断裂无感知;内容无版本、无法回滚、无法分发历史版本。内容寻址让'依赖某包的特定内容快照'成为一等公民。
- **资产分层与官方/第三方能力市场**：把现有 inherit/(基类)、feature/(mixin)、clone/(模板)重组为分层的可复用资源库: Runtime(官方核心: 战斗/存档/移动/消息 守护进程，不可改)→StdLib(官方标准组件库: 房间/武器/NPC 基类与常用模板，版本化可依赖)→Marketplace(社区第三方包)。创作者的 CPK 只能依赖 StdLib 与 Marketplace，禁止直接触碰 Runtime，保证平台升级不破坏内容。StdLib 自身按 SemVer 演进并发布 deprecation 时间表。
  - 适用性：继承现有 inherit/ + feature/ 组合思想(ROOM←F_DBASE←F_CLEAN_UP)，但把'编译期继承'变为'运行期可热插拔的能力模块'，使普通创作者能组合官方/第三方能力而无需碰核心代码。
- **运行时沙箱 + 配额治理**：每条内容运行在隔离沙箱: 进程/协程隔离(沙箱脚本崩不波及主进程)、eval-cost 配额(对应 config.xkx maximum evaluation cost，防死循环)、内存与对象数上限、reset 调度(对应 time to reset 1800s，房间定时刷新)、clean_up 调度(对应 time to clean up 180s，闲置对象回收)、存档配额(/data 下 .o 体积上限)。配额分级: 个人免费档/付费档/认证创作者档。沙箱在触发配额时优雅降级而非硬杀，并记录事件供创作者诊断。
  - 适用性：对齐现有 reset() 周期重生物品、clean_up() 回收闲置对象、save/restore_object 持久化玩家/NPC 状态的机制，把它产品化为可观测、可调度的资源治理层。
- **分支协作 + 结构化合并**：每个 CPK 有独立 git 式分支历史;创作者 fork/branch 编辑，提交 pull-request 经评审合入 main。对场景图等结构化并发编辑采用 CRDT 或操作变换(OT)减少冲突;DSL 文件走文本 diff + 语义合并辅助。合并到 published 前必须过自动门(编译/依赖/配额)。冲突解决策略: 文本冲突人工 merge、语义冲突(如两人改同一出口目标)由校验器标注交回作者。支持多人协作角色: owner/maintainer/contributor/reviewer，映射现有 wiz 等级但更细粒度。
  - 适用性：从现有'单巫师改文件、手动协作'升级为多人可并行创作同一区域(如多人共建华山派)，并通过评审合并机制保证质量，而非依赖文件锁。
- **内容市场与分发**：平台内建内容市场: 浏览/搜索/评分/安装/订阅。市场记录每个 CPK 的安装量、活跃度、依赖被引次数、评分。分发支持'主题合集'(如武侠全包、大航海时代包)与'单作品'两种粒度。市场作为官方与第三方资源的统一入口，官方 StdLib 也经市场分发版本。安装即声明依赖并拉取内容寻址快照，卸载即移除依赖并检测反向依赖防止误删。
  - 适用性：把现存的'核心包 + 16个门派 + 多城市'拆为可独立分发、可组合安装的内容包，让用户能像装 mod 一样装配自己的世界，而非全量部署整个 /d。
- **版权溯源与计费**：每份 CPK 声明 license(CC-BY/CC-BY-SA/自定义/付费)。衍生作品( fork 或依赖)自动建立 provenance 链: 记录基于哪个 hash、哪些资源被引用。付费包走分账: 平台抽成 + 上游依赖按 deps.yaml 分润 + 创作者收益。版权投诉走 DMCA 式流程: 申诉→临时下架→申诉方举证→恢复/永久下架，全程留痕并影响该包与衍生链。市场提供'商用授权'标记便于筛选。
  - 适用性：补齐现有完全缺失的版本控制、分发与计费能力。版权溯源尤其重要: XKX 本身即源于 mudchina 开源代码并标注'仅供学习'，平台需为衍生内容建立清晰的授权链。
- **多主题世界组合**：平台运行多个世界实例(武侠/大航海/书院/穿越/现代)，共享同一运行时与 StdLib 但加载不同主题 DSL 词汇表与渲染规则。世界间通过受控'跨世界传送'机制互联(对应现有 enter:/d/bwdh/square 这类跨区域出口，但升级为显式声明的跨世界门)。每个世界有自己的市场分区、排行榜与运营策略。Agent 协作: AI 创作助手作为特殊协作者角色参与 CPK 的草稿生成，产出物同样走生命周期审核门，人类作者对其负责。
  - 适用性：支撑任务中'引入 DSL 让用户配置剧情场景，借 Agent 协作创作多题材游戏世界'的目标: Agent 即可被建模为拥有受限能力令牌的特殊协作者角色，其产出物走与人类相同的审核与版本流程。

## 适用性
- 适用于内容量极大、由大量原子单元(房间/NPC/物品/技能)组合而成的 MUD 类文本世界；现有 XKX 约8400个LPC文件、186个区域目录，正是'文件即对象'模型的典型，向'内容包'迁移收益最高
- 适用于多题材并行(武侠/大航海/书院/穿越/现代)的世界扩张：各主题用独立 DSL 词汇表与渲染规则，共享同一运行时与市场，避免为每个题材重建一套引擎
- 适用于需要把核心巫师创作权下沉给普通用户的演进路径：DSL+可视化降低门槛，沙箱+能力权限保证安全，让 UGC 从'少数人改源码'变为'多数人配场景'
- 适用于追求长线运营与商业化：版本库+市场+计费支撑内容生态自循环，版权溯源与下架流程满足合规要求
- 当且仅当运行时能提供确定性沙箱(脚本隔离、配额、热重载)时才可安全开放 UGC 脚本能力；若沙箱不可靠，应退化为纯声明式 DSL 不开放脚本逃逸口

## 权衡
- 声明式 DSL 极大降低创作门槛(覆盖95%场景无需写代码)，但任何 DSL 表达力不足处都必须回退到沙箱脚本，增加双轨维护成本与'DSL 边界在哪'的持续争论
- 内容寻址版本库带来精确依赖与可复现构建，但与传统文件系统直觉相悖，对从 LPC 时代迁移过来的创作者有学习曲线，且 lockfile 管理增加复杂度
- 强隔离沙箱(进程/能力/配额)保障平台与多租户安全，但增加运行时开销与跨包协作的握手成本，可能限制某些需要紧密耦合的复杂剧情(如跨门派大战)
- 能力权限模型比 wizlist 更细更安全，但权限设计一旦过度严格会抑制创作自由，过度宽松又失安全;需持续校准默认能力集
- 市场分账与版权溯源支撑商业化，但分账规则复杂(多级依赖链分润)易引发争议，且版权审核在 AI 协作创作场景下判定'原创 vs 衍生'会非常困难

## 推荐
- 第一步先固化'内容包(CPK) + manifest + 依赖声明'格式，把现有 /d、/clone、/kungfu 用迁移脚本批量转译为初始 CPK 集合并建立依赖图基线;这是后续版本/市场/计费的地基，不可跳过
- DSL 先从房间(exit/objects/long/item_desc)这类最规整、覆盖面最广的对象做起，验证'声明为主、脚本为辅'双轨可行后再扩展到 NPC/技能/剧情;沙箱脚本逃逸口默认关闭、按能力审核逐项放开
- 运行时沙箱与配额治理必须先行且作为不可妥协的硬约束，直接对齐 config.xkx 的 eval-cost/reset/clean-up 语义固化为 per-CPK 配额，否则开放 UGC 脚本即等于开放拒绝服务
- 先重建并替代 securityd.c 的能力权限模型(本分支已损坏)，再开放协作;否则多人创作与跨包引用的安全边界无法成立
- 市场与计费作为后期阶段，但版权溯源(provenance 链)与 license 声明要从第一个 CPK 就内建进 manifest，避免后期补登成本过高;Agent 协作产出物也必须走与人类相同的审核与版本流程
- 为多题材预留主题 DSL 词汇表插件机制，使武侠/大航海/书院/穿越/现代可共享运行时但各自演进，避免为每个题材重建引擎
