# 迁移路径与里程碑（v2）

> 本路线图以 [[05-专家对抗复审报告]] 的修正结论为驱动，按"垂直切片优先、greenfield 重写、规格提取先行、性能基准与工具链一等公民"原则编排。**核心立场：最高产品风险（DSL+Agent 创作闭环）先验证；没有旧 LPC 系统需要逐步迁移，LPC 是规格源而非运行系统。**

---

## 一、总体原则

1. **垂直切片优先**：阶段 -1 先验证 5-10 房间 + 2 NPC + 1 战斗 + 1 任务 + 1 对话的 DSL+Agent 闭环，不通过则不投入引擎重构。
2. **greenfield 重写**：无双栈、无过渡期、无适配层、无逐步替换 LPC。LPC 代码是规格源。
3. **规格提取先行**：阶段 0 的核心工序是从 LPC 提取形式化规格契约，而非录制运行中系统。
4. **单进程 asyncio 起步**：阶段 1 纯 asyncio + 普通对象 + SparseSet ECS，不引入 Ray 做实体运行时。
5. **性能基准与工具链一等公民**：阶段 0 必须定义单 do_attack μs 级基准、tick 预算、GC 基准、PYTHONHASHSEED=0，并完成引擎工具链 PRD。
6. **分片内同步、分片间异步**：分布式阶段只在跨分片边界用异步消息，同房间保持同步调用栈语义。
7. **目标务实下调**：万人注册 + 千人同时在线；单房间串行是硬约束。

## 二、独立可交付里程碑（M1/M2/M3）

制作人建议把标杆能力拆为独立可交付物，早获社区反馈：

| 里程碑 | 时间 | 交付物 | 价值 |
|---|---|---|---|
| **M1** | 3 月 | 确定性回放引擎开源 | 全行业最缺的反作弊/数值审计基础设施；最低成本拿到的标杆资产 |

| **M3** | 6-8 月 | 单题材（武侠）完整可玩 demo | M3 之后才投入全量迁移与多题材 |

## 三、分阶段路线图

### 阶段 -1：垂直切片平台验证（2-3 月）★ 新增，最高优先级

**目标**：在投入引擎重构前，验证核心价值主张"DSL+Agent 让非程序员创作可玩世界"。

```
5-10 房间 + 2 NPC + 1 战斗 + 1 任务 + 1 对话
         │
         ▼
    全部 DSL 定义
         │
         ▼
    Agent 部分生成（copilot 模式）
         │
         ▼
    跑在真实引擎最小子集
    （resolve_attack 纯函数 + stub 运行时）
```

| 任务 | 产出 | 验收标准 |
|---|---|---|
| DSL 表达力验证 | 垂直切片全部 DSL 定义 | 不依赖逃生舱即可完成 5 房间+2 NPC+1 战斗+1 任务+1 对话 |
| Agent 产出可用性 | Agent 生成的 DSL 初稿 + 人工修订量度量 | 人工修订量可接受，无结构性错误 |
| 玩家可玩性验证 | 3-5 名目标玩家试玩反馈 | "觉得好玩"评分 ≥ 可继续投入阈值 |
| resolve_attack 纯函数提取 | 无副作用纯函数 + 属性测试 | 输入组件快照 + seed → 输出 CombatRoundResult；与未来真实引擎同源 |

**不可妥协**：本阶段失败则停止项目，不投入引擎重构。

**目标**：建立 greenfield 重写所需的形式化规格与验证基建，不写新功能。

| 性能基准 | 单 do_attack μs 级报告 + tick 预算（50ms/20Hz 或 100ms/10Hz）+ GC 基准 + PYTHONHASHSEED=0 | 有数据支持 compiled core go/no-go |

- 不修 securityd 旧代码。

### 阶段 1：单进程 asyncio 核心循环（3-6 月）

**目标**：纯 Python 单进程跑通核心循环，验证语义正确性。

```
┌─────────────────────────────────────────┐
│ 单进程 asyncio（普通对象 + SparseSet）    │
│  ├─ ECS 骨架                             │
│  │   ├─ Identity/Vitals/Position 核心组件│
│  │   ├─ Progression 组件                 │
│  │   └─ Effect 一等公民                   │
│  ├─ SchemaRegistry（类型化组件）          │
│  ├─ ReferenceStore（ID 引用，PG 关系表）  │
│  ├─ SchedulerStore（单一后端，分层）       │
│  ├─ 全仿真确定性（DeterministicRNG）      │
│  ├─ 逻辑热重载协议（System 热替换）        │
│  └─ 引擎工具链同步开发                    │
├─────────────────────────────────────────┤
│ 无状态 WS 网关 + 会话 + JWT/argon2       │
│  └─ 乐观执行 + rollback delta             │
├─────────────────────────────────────────┤
│ PG 唯一权威 + Redis 读穿透缓存            │
└─────────────────────────────────────────┘
```

| 里程碑 | 验收 |
|---|---|
| ECS 骨架 + 核心组件 | Identity/Vitals/Position/Progression 可创建/查询/序列化 |
| SchemaRegistry 类型化组件 | query 拼写错误启动期失败 |
| 字段→组件映射表 | query_entire_dbase 调用点读写键集枚举完成 |
| 无状态 WS 网关 + 认证 + 重连 | 单进程承载会话，断线重连 ring buffer 重放 / snapshot_resend 降级 |
| PG 权威 + Redis 读穿透 | 写 PG 再回填 Redis，Redis 降级不丢数据 |
| 全仿真确定性 | 单 tick 快照 + input log 可重放 |
| 逻辑热重载 | System 代码 reload + entity state migration |
| 引擎工具链同步 | entity inspector / tick profiler / replay viewer 可用 |
| combat-sim 调 resolve_attack | 纯函数行为等价验证通过 |

**阶段 1 硬约束**：
- 不引入 Ray 做实体运行时。
- Effect 必须可序列化、可中断、可迁移。
- ConditionHandler.on_tick 返回组合结构契约落定。
- 性能基准未达标前标记为原型验证语言。

## 五、风险驱动的"不做"清单

| 不做 | 理由 | 替代 |
|---|---|---|
| 从零自研 asyncio 虚拟 Actor 运行时 | 多年级工程 | 纯 asyncio + 普通对象起步 |
| 阶段 1 引入 Ray 做实体运行时 | Ray actor 非重入，do_attack 递归死锁 | Ray 仅阶段 3 计算层 |
| 双栈过渡 / 过渡期 / 适配层 | greenfield 项目不存在这些概念 | 规格提取后直接实现 |
| 修 securityd.c 旧代码 | 不修旧代码，从规格实现 | 提取权限规格后从零实现 |
| 录制运行中 LPC 命令流为 golden trace | 旧系统在当前 Linux 不可运行 | 单元级行为规约 + 一次性 golden trace |
| s_combatd.c "废弃" | 不存在废弃，只提取规格 | 提取阵法合击规格 |
| 32 守护进程全量分类迁移 | greenfield 下职责重新设计 | 被 ECS System 取代 / 保留无状态 / 演进 |
| SparseSet + Archetype 双存储 ECS | 互斥策略，徒增复杂度 | 只用 SparseSet |
| 全量事件溯源(CQRS+ES)作为默认持久化 | 与同步调用语义冲突 | PG 唯一权威 + Redis 读穿透 |
| Neo4j/Apache Age 社交图谱 | 关系类型<10，过度工程 | PostgreSQL 关系表 |
| MMO 式"1跳邻域 AOI" | MUD 是房间作用域非连续空间 | 同房间实体集合 + 邻域静态元数据只读 |
| ModifierStack 用 Redis ZSET | 丢失修饰符组合语义 | modifier 链表/栈（分三类） |
| APScheduler + Redis ZSET 双调度 | 一致性风险 | 单一后端 + 分层 |
| 全库 call_out 统一持久化 | 性能反模式 | 分层：短延迟(<30s)留内存 |
| 过早引入 Kafka | 单区域 PG+NATS 足够 | 跨区域多租户达规模再议 |
| tree-sitter-lpc 批量解析 | 非成熟 grammar，动态分派需流分析 | 正则+AST+人工标注混合 |
| 层3 UGC 用裸 WASM | UGC 作者是非程序员 | RestrictedPython（WASM 仅平台级） |
| provenance 从第一个开发 CPK 内建 | MVP 拖慢迭代 | 门3开启前才强制 |
| combat-sim 用独立简化模型 | 验证信号失真 | 直接调用 resolve_attack 纯函数 |
| LangGraph active-active 多副本编排 | 单进程库，重复分派 | PG-backed 状态机 / 单编排器+故障转移 |
| Loki 做高维日志检索 | 标签基数爆炸 | OpenSearch/ClickHouse |
| 有状态分片用 HPA 反应式扩缩 | 分钟级暖启动与滞后信号时延错配 | 预测性+预留容量 |
| liveness 关联 tick 新鲜度 | 杀死级联 | 仅判事件循环死锁 |
| 战斗变更分片金丝雀 | 玩家流动，灰度即全量 | 蓝绿全量 |
| 实例化分桶突破单房间容量 | 摧毁 MUD "全服同房"社交魅力 | 热点房间专用分片 + 按战斗组并行 |
| "秒级回滚无数据丢失" | schema 漂移导致不一致 | 逻辑回滚≠状态回滚 |
| 全量事件驱动替代 heart_beat | heart_beat 是 tick 驱动器 | 保留周期 tick |

## 八、决策检查点

每个阶段结束前，必须回答以下问题，任一为否则不推进：

**阶段 -1 -> 0**：
- DSL 能否表达垂直切片全部内容？
- Agent 产出的人工修订量是否可接受？
- 目标玩家是否觉得好玩？
- `resolve_attack` 纯函数是否提取成功且与未来引擎同源？

**阶段 0 -> 1**：
- LPC 规格提取是否覆盖 go/move/combat 核心路径？
- FluffOS Linux 编译是否可行或已降级为单元级行为规约？
- 性能基准是否支持阶段 1 的 Python 原型决策？
- 引擎工具链 PRD 是否评审通过？

**阶段 1 -> 2**：
- 单进程 asyncio 核心循环是否跑通？
- 全仿真确定性是否可重放？
- Effect/ConditionHandler 契约是否落定？
- PG 唯一权威 + Redis 读穿透是否稳定？

**阶段 2 -> M3 / 阶段 3**：
- Combat 迁移是否通过行为等价验证 + 文本体验流 diff？
- 技能三层是否明确？
- 称谓系统、世界观治理层是否落地？

**阶段 3 -> 4**：
- 单分片容量是否实测？
- handoff outbox 是否规约化？
- 单房间串行约束是否被接受？

**阶段 4 -> 5**：
- securityd 权限规格是否提取并重新实现？
- 沙箱配额是否就位？
- DSL 表达力校准是否通过？
- 版权清洗是否完成？
