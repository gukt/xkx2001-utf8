# 交叉质证·资深游戏主策划（15年游戏设计经验，深耕武侠题材与多人交互系统设计，懂经典MUD文本叙事也懂现代游戏设计）

## 同意的论断
- **greenfield 与增量重构的根本性概念混淆：方案全文以「迁移」为主框架，但用户明确这是 0-1 全新项目、LPC 是规格源不是运行系统，「迁移适配层/双栈过渡/逐步替换/修旧 bug」全是增量重构概念**（资深游戏架构师）：完全同意且从游戏设计视角补充：这个混淆不仅是工程概念错误，更是设计方法论错误。增量重构思维会让人误以为'忠实迁移=逐行对照运行中系统'，而 greenfield 的正确思路是'LPC 是规格源，先提取玩法设计规格再从零实现'。我做源码验证时发现，skill_power 三段式公式、death_count 师徒版本号、fight() 硬编码门派特例这些'玩法设计灵魂'恰恰需要从源码静态提取为设计规格，而非对照运行时输出。增量重构假设会误导团队把精力放在'逐位对齐'而非'理解设计意图'。
- **当前 MVP（阶段0-1）验证的是引擎可行性而非产品/市场价值。核心价值主张'DSL+Agent让非程序员创作可玩世界'要到阶段4-5（14+月后）才被验证，把最高风险放到最后验证，顺序完全反了**（资深游戏制作人）：这是我作为主策划最强烈同意的论断。从游戏设计看，'DSL+Agent 能否表达侠客行玩法'是整个项目的产品假设核心，但方案把它排在最后。更尖锐的是：我的源码验证证明侠客行的核心玩法（战斗七步+skill_power三段式+门派隐藏大招+师徒death_count版本号+死亡惩罚反作弊）70%+在脚本里，DSL 极可能表达不了。如果14个月后才发现'DSL 表达不了门派灵魂'，前面引擎投入全沉没。制作人建议的'阶段-1垂直切片验证'是正确的产品决策。
- **LPC 同步 call_other 与 Ray 异步 actor 消息根本不同构，do_attack 的同步递归+random 分支在 Ray 下大概率死锁或退化为单线程**（分布式系统资深专家）：同意且从游戏设计视角强化：这不只是技术问题，更是战斗手感问题。我验证 combatd.c 第765行 riposte、807/826行 double_attack+辟邪剑法硬编码，这些是'同步递归+random分支+跨实体回调'的最难场景。战斗手感本质是'副作用时序'，如果 Ray 异步化破坏了 riposte 的同步回溯语义，玩家会感到'反击节奏不对'。分布式专家指出的'分片内同步、分片间异步'是正确方向，与 LPC 同房间强同步语义一致。
- **代词体系实测是 10 个变量而非方案的 4 个，emote 是 7 视角而非 3 视角，RANK_D 7 个函数是业务逻辑求值非纯渲染**（MUD/LPC 历史与社区专家）：完全同意并经源码验证：emoted.c 注释明确列出 $N/$n/$P/$p/$C/$c/$R/$r/$S/$s 十变量，rankd.c 有 query_rank/query_respect/query_rude/query_self/query_self_rude/query_close/query_self_close 七函数。从游戏设计看，这不是技术细节而是文本叙事核心：7 视角 emote 覆盖了'发起者对自己''旁观者看发起者对自己'等微妙叙事场景，降为3视角等于阉割中文 MUD 文学性。这是'战斗手感'之外的另一个'玩家可感知的保真度'维度。
- **5 份文档完全遗漏至少 5 个灵魂系统：阴间/轮回、武林大会、vote 玩家自治、法院审判、intermud 跨服网络**（MUD/LPC 历史与社区专家）：强烈同意且补充更严重的数据：我实测 d/bwdh 有 276 个 .c 文件（252 活跃），远不止'40+房'，这是包含擂台/看台/仓库/伤害计算的完整社区竞技系统。vote 系统（cmds/std/vote+chblk/unchblk）、courthouse、d/death 15房、network 41文件全部验证存在。这些不是边缘玩法而是 MUD 社区治理骨架。从游戏设计看，bwdh 这么大的系统被完全遗漏，说明迁移面估算本身就有结构性盲区——只数了技术调用面（68771），没数文化系统面。
- **combat-sim 用独立纯 Python 数值模型与方案自身的差分测试原则形成双重标准，验证信号失真比没验证更危险**（AI/Agent 工程资深专家）：同意并从游戏设计视角连接：这与我提的'战斗手感漂移'是同一个问题的两面。独立模型若简化了 riposte 递归、fight/dodge 累积态、condition 位聚合，验证通过的内容上线后战斗手感会变。AI 专家说'应直接提取 resolve_attack 纯函数'方向对，但我补充：纯函数化必须显式保持'分支内 mutate 时序'，否则同样会漂移。两者要同时解决。
- **引擎工具链完全缺失，应作为阶段0一等公民交付**（游戏引擎资深专家）：同意但理由不同：引擎专家从技术调试角度要工具链，我从游戏设计调试角度更要。8400 文件迁移中，战斗手感偏差（精力消耗节奏、经验获取曲线、招式触发频率）只能靠'战斗回放可视化+文案时序 diff'发现，不能靠 metrics。没有 combat replay viewer 和文本体验流 diff，调试战斗手感偏差只能靠人肉打，这在单人团队下不可持续。

## 挑战的论断
- **Python 纯解释器无法达到业界标杆，greenfield 应严肃评估 Rust 核心(PyO3)+Python 脚本混合架构，这是标杆 vs 玩具的分水岭决策**（资深游戏架构师+游戏引擎资深专家）
  反对：语言选型是实现细节不是游戏设计标杆的分水岭。真正的分水岭是'是否忠实保留玩法设计灵魂'。一个 Rust 核心如果把 skill_power 简化为 level³/3 可配置表达式（如02文档第167行所做的），它比一个忠实实现三段式（基础幂+状态修饰链+属性系数封顶）的 Python 核心更糟糕。源码验证 skill_power 第288-333行：level³/3 仅一部分，叠加 jingli_bonus(50-150动态)、apply/attack、fight/dodge 累积态(305行 level*((100+fight/dodge/10)/100))、str/dex 系数封顶300。Rust 让这跑得快10倍但不改变'是否理解并保留这个公式'的问题。标杆是'do_attack 七步管线的副作用时序是否忠实'+'门派隐藏大招是否完整迁移'+'战斗手感是否与原作一致'，不是'单 tick 处理多少实体'。架构师和引擎专家把性能当标杆核心是工程师视角，主策划视角下性能是约束不是目标。
  理由：源码证据：combatd.c skill_power() 第288-333行证明这是状态依赖的累积函数，不是可配置表达式。02文档第167行'DamageFormula 配置化表达式（level³/3 可改 level²/线性）'严重失真。无论用 Rust 还是 Python，如果设计规格提取阶段就把三段式简化为 level³/3，战斗手感必然漂移。语言不决定设计保真度，设计规格提取质量决定。
- **单房间串行是分布式无法突破的硬约束，需房间级水平扩展：热门房间多实例+玩家分桶（按 player_id hash 分配到房间实例）**（分布式系统资深专家）
  反对：从游戏设计强烈反对房间实例化分桶。MUD 的核心社交魅力恰恰是'全服同房'——主城广场喊话全服可见、PK 围观者实时看到战斗描述、武林大会千人同场。实例化分桶会摧毁这个体验：玩家在实例A喊话实例B听不到，围观者看不到另一实例的战斗。这是把 MUD 变成普通图形 MMO 的'频道制'，丢失了 MUD '单一共享世界'的本质。正确的游戏设计方向是：承认单房间串行是容量天花板，要么接受'千人同时在线'而非'万人'的务实目标，要么用'热点房间专用分片+房间内并行化战斗裁决（按战斗组分而非玩家分桶）'。但绝不能用实例化分桶——那是在解决技术问题的同时制造更大的设计问题。
  理由：MUD 社交模型是'单一共享空间+文本广播'，与图形 MMO 的'多频道副本'根本不同。武林大会（实测276文件）就是为'全服同场竞技'设计的，分桶后失去意义。这是游戏设计红线，不是技术权衡。
- **四层 DSL 心智负担远超 Roblox Lua+Studio，应将可视化场景编辑器作为首选创作入口而非让作者直接写 DSL**（UGC 平台产品专家）
  反对：部分同意可视化编辑器重要，但反对其隐含前提：'侠客行核心玩法可被非程序员 UGC 创作'。我的源码验证证明恰恰相反——dugu-jiujian 的 query_action 隐藏大招（6条件触发）、hit_by 4级递进反击、valid_learn 门派互斥、feng.c 的 death_count 师徒版本号、combatd fight() 硬编码辟邪/double_attack——这些玩法灵魂 70%+ 在脚本里且需要设计决策。可视化编辑器能降低'剧情/场景/简单NPC/对话'的创作门槛（这是对的），但不能假装非程序员能可视化编辑'战斗七步管线的副作用时序'或'门派师徒关系状态机'。正确的框架是'分层能力域'：UGC 限定剧情/场景/简单 NPC 创意域；战斗/门派/师徒/反作弊/经济调参为平台核心域由可信作者维护。可视化编辑器服务前者，后者用 Python 策略对象。
  理由：dugu-jiujian.c query_action 第262-272行玄铁重剑大招需要 str>=30&&int>=36&&random(skill)>80&&force>150&&neili>400&&武器特定 6 条件——这不是任何可视化编辑器能表达的，是硬编码设计决策。UGC 平台专家用 Roblox 类比，但 Roblox 的核心战斗是平台提供的基础机制，创作者只做内容；侠客行的战斗机制本身就是内容且高度门派特化。
- **combat-sim 应改为直接调用真实 ECS 的 resolve_attack 纯函数而非独立简化模型，保证验证与未来真实引擎同源**（AI/Agent 工程资深专家）
  反对：方向同意但挑战其框架：真正的问题不是'纯函数 vs 独立模型'，而是'副作用时序是否显式保持'。纯函数化若只是把 do_attack 读快照返回事件，但不显式建模'分支内 mutate 时序'，同样会漂移。我验证 combatd.c 第454-459/503-508/696-712行：未命中扣 jingli 与命中扣 jingli 是同一变量不同分支的互斥 mutate，且分支内有时序依赖。纯函数化要把这些副作用收集为带时序标记的 SideEffect 列表，按 step 顺序严格 apply。AI 专家的'resolve_attack 纯函数优先'只解决了'与引擎同源'，没解决'与原作手感同源'。两者必须同时满足：既是真实引擎的纯函数子集，又显式保持原 do_attack 的分支内副作用时序。
  理由：战斗手感 = 副作用时序 + 文本节奏 + 戏剧性。纯函数化只解决架构清晰度，不解决手感保真。必须额外建'管线副作用账本'，把原 mutate 时序固化为不变量回归断言。这是游戏设计验证维度，AI 专家和架构专家都未覆盖。
- **全仿真确定性是标杆引擎的核心能力，所有 System 的 RNG 走 seeded DeterministicRNG + input log = 完整重放，这是标杆 vs 够用的分水岭**（资深游戏架构师）
  反对：部分同意确定性是重要能力，但反对将其定位为'标杆分水岭'。确定性是手段不是目的。目的是'战斗手感可回放、bug 可调试、反作弊可审计'。过度追求全仿真确定性有两个游戏设计风险：(1)可能为了确定性而简化非确定性设计（如 blind 的 random 消息丢弃、NewRandom 加权分布），反而损害手感；(2)把确定性当目标会导致团队在'技术完美'上耗时而忽视'游戏是否好玩'。真正的标杆分水岭是'游戏设计深度+文化保真+玩家可感知的现代化'，确定性只是支撑这些的底座之一。架构师把技术能力当标杆核心是典型工程师思维盲区。
  理由：业界标杆游戏引擎（Unity/Unreal）的标杆地位来自'能做出什么游戏'而非'确定性多强'。Bevy 确定性极强但游戏生态远不如 Unity。MUD 的标杆应该是'最 MUD 的现代游戏'（MUD 历史专家的定位我同意），确定性服务于这个目标而非反之。
- **ECS+Actor+DSL+Agent 四范式堆叠过度复杂，应收敛为 ECS（仿真核心）+DSL（创作契约）两范式，Actor 降级为 ECS 分布式传输层，Agent 是 DSL 生产端消费者**（游戏引擎资深专家）
  反对：从游戏设计视角部分挑战：范式收敛在技术上合理，但'Agent 是 DSL 生产端消费者'这个定位严重低估了 Agent NPC 作为游戏内实体的设计价值。引擎专家把 Agent 降级为'离线创作工具'，但真正的游戏设计标杆差异化是'LLM 驱动的游戏内 NPC'——能动态对话、根据玩家行为调整剧情、产生涌现叙事。这是 LPC 时代做不到而现在能做且无人做成的 MUD 现代化核心卖点。把 Agent 仅当离线工具等于放弃最大的差异化机会。范式可以收敛为 ECS+DSL 两个运行时范式，但 Agent 应有第三个维度：作为游戏内实体的运行时 Agent（走 ECS tick 驱动但行为由 LLM 实时生成）。
  理由：LPC 的 NPC 是固定对话树+脚本行为，现代化标杆应超越。Agent NPC 是'游戏设计'层面的标杆，不是'引擎架构'层面的范式。引擎专家从架构简洁性出发降级 Agent，但游戏设计上 Agent NPC 恰是应该升维的能力。

## 新涌现缺口
- 战斗手感验证维度在所有8位专家中完全缺失。架构师要确定性、分布式专家要 handoff 规约、AI 专家要 combat-sim、引擎专家要性能基准——但没有一个专家问'新引擎的战斗，玩家打起来感觉对不对'。战斗手感 = 副作用时序 + 文案输出节奏 + 戏剧性（guard_msg/catch_hunt_msg/winner_msg/damage_msg 多档文案池的时序与频率）。需要建'文本体验流 diff'验证：录制玩家战斗的 message_vision 输出顺序与代词替换，前端渲染后做'阅读体验 diff'，而非仅做状态快照 diff。这是游戏设计独有维度，技术专家不会自动覆盖。
- 游戏设计标杆维度在专家讨论中系统性缺席。架构师论标杆=确定性+热重载+工具链+性能；制作人论标杆=DSL-as-contract+开源生态；MUD 历史专家论标杆=文化保真。但没有人问最根本的问题：'新引擎做出的游戏，比原侠客行更好玩吗？更有深度吗？'标杆首先是游戏设计标杆，其次才是技术标杆。方案把大量精力投入'忠实迁移'却几乎不讨论'设计超越'——除了我提的 Agent NPC，没有任何专家提出'LPC 做不到但现在能做'的游戏设计创新。
- 经济系统规模化重新设计在所有专家中只有我提及但无人响应。moneyd.c MAX_CASHFLOW_ALLOWED=400000 按200人调参的通缩模型，放大到1万在线需完全重新设计货币产出/回收/通胀控制。这是独立游戏设计专项非迁移子任务，但方案和所有专家都把它当技术问题（children() 全库扫描的分布式化）。分布式化是工程问题，经济重新平衡是设计问题，两者不可混淆。
- 门派师徒关系状态机建模只有我提及。feng.c 实测：death_count 作为师徒关系版本号（343行 set students/id = death_count）、check_student 检测 death_count 变化则逐出师门（143行）、check_betrayal/revenge/improve_sword 完整状态机、19门派各有师徒逻辑。这是门派系统核心非数据部分，但方案只提'FamilyDefinition 门派数据声明 Prefab'，所有专家都没追问师徒关系状态机。这会影响门派系统迁移完整性。
- Agent NPC 作为游戏内实体无人接续讨论。我提出这是最大的差异化机会（LPC做不到、现在能做、无人做成），但7位专家无人支持也无人挑战。这要么说明它被低估，要么说明它超出技术专家视野。但从游戏设计看，这是'业界标杆'定位中最可能成功的创新点：保留 MUD 文学性 + 加 LLM 驱动的动态 NPC 对话与涌现叙事。需要专门立项设计 Agent NPC 的行为约束（LLM 生成对话但状态变更受 DSL 契约约束）、一致性缓存、与 ECS tick 的集成。

## 精炼立场
综合7位专家的交叉质证，我作为主策划的精炼立场是：这个项目的真正标杆定位不应是技术专家们争论的'最快的 MUD 引擎'或'最确定性的仿真核心'，而应是'最 MUD 的现代游戏'——保留侠客行玩法设计灵魂用现代技术放大其表现力与触达，同时加入 LPC 时代做不到的设计可能。

具体而言，标杆由三个游戏设计维度定义，而非技术维度：

第一，玩法内核保真红线（必须忠实，不可现代化简化）：战斗七步管线的副作用时序、skill_power 三段式累积公式（经源码验证非 level³/3 可配表达式）、门派隐藏大招与 hit_by 反击（70%+在脚本非数据）、死亡惩罚的反作弊精妙（death_count 师徒版本号经源码验证、free_rider 检测、shen 区间惩罚）、72 condition 的社会治理语义（killer/vote_suspension/city_jail 经源码验证非战斗 buff）。这些是'游戏手感与深度的数学骨架'，简化即平庸化。

第二，文本叙事灵魂保真（必须忠实，不可降级）：代词10变量7函数求值（经源码验证）、emote 7视角（经源码验证）、damage_msg 11种damage_type×6档分级文案池、guard_msg/catch_hunt/winner_msg 叙事池。这些是 MUD 文学性核心，降级即丢失灵魂。服务端必须求值 RANK_D 产出 PronounContext 下发，前端只做纯替换。

第三，设计超越红线（必须创新，不可只做考古）：Agent NPC 作为游戏内实体是最大的差异化机会——LPC 做不到动态对话/涌现叙事/玩家行为感知的剧情分支，现在能做且无人做成。这是'下一代 MUD'vs'用 Python 重写的 LPC'的分水岭。配合制作人的'垂直切片优先'建议：应在阶段-1 就验证'Agent NPC+DSL 约束'的可行性，而非等到14个月后。

对技术选型争议的立场：Rust vs Python、Ray vs asyncio、ES vs 无 ES——这些都是实现层决策，应由架构师基于性能基准决定，但不应被定位为'标杆分水岭'。标杆分水岭是游戏设计层面的：是否保留了战斗手感、是否保留了文本叙事、是否创造了 LPC 做不到的新体验。一个用 Python 但忠实实现 skill_power 三段式 + Agent NPC 的引擎，比用 Rust 但简化公式、无 Agent NPC 的引擎更配称标杆。

对 greenfield 重审的立场：greenfield 反而是游戏设计的解放——不再受'逐位对齐运行中 LPC'束缚，可以'从规格提取玩法设计意图，从零实现并创新'。但解放不等于简化：必须区分'必须忠实的玩法内核'（战斗/门派/师徒/死亡惩罚/文本叙事）与'可现代化的表层'（UI/并发/Agent NPC/大规模世界事件）。方案当前的缺陷是把'忠实迁移'和'业界标杆'当作对立面，实际上标杆=忠实内核+现代超越。

最后，所有专家都遗漏了一个根本的游戏设计问题：经济系统从200人调参的通缩模型（moneyd MAX_CASHFLOW_ALLOWED=400000）放大到目标万人在线需完全重新设计货币产出/回收/通胀控制。这不是迁移任务而是独立的游戏设计专项，方案和所有专家都未立项。

## 开放问题
- 战斗手感等价性如何验证？需要定义'文本体验流 diff'的具体方法：录制哪些战斗场景的 message_vision 输出、diff 的粒度（逐行 vs 语义等价）、可接受的偏差阈值。这是所有技术专家都没覆盖的游戏设计独有维度。
- skill_power 三段式公式的精确迁移规格如何形式化？需从 combatd.c 第288-333行提取完整契约：基础幂(level³/3)+状态修饰链(jingli_bonus动态50-150/apply修饰/fight-dodge累积态/busy减半)+属性系数(str/dex封顶300)。fight/dodge 累积态如何建模为 CombatState 瞬态字段而不进 YAML？
- death_count 师徒版本号设计在分布式下如何建模？feng.c 用 death_count 作为关系失效信号，分布式下师徒关系状态机需要专门设计。19门派各有不同师徒逻辑，是否需要统一的 MentorshipRelation 抽象？
- Agent NPC 作为游戏内实体的行为约束如何设计？LLM 生成对话的边界在哪（可自由对话但状态变更受 DSL 约束）？如何防止 LLM 幻觉破坏世界一致性？与 ECS tick 如何集成（LLM 调用延迟 vs tick 实时性）？
- 经济系统从200人到1万在线如何重新平衡？moneyd 通缩模型（MAX_CASHFLOW_ALLOWED=400000）的参数如何重调？分布式下用'分区货币总量+跨区贸易汇兑'替代 children() 全库扫描的具体设计？
- 门派隐藏大招的完整迁移评估：163个含*action的技能里有多少有类似 dugu-jiujian 玄铁重剑的条件触发隐藏招？迁移前需全量盘点所有 query_action 的非标准分支，否则 YAML 化会静默丢失门派灵魂。这是 UGC 表达力校准实验的关键输入。
- bwdh 武林大会（实测276文件）作为墙钟驱动的社区竞技系统如何在新引擎实现？它与 WorldClockSystem、CombatSystem、SocialGovernance 的关系如何设计？这是被完全遗漏的大型灵魂系统。
- 战斗文本叙事的分级文案池迁移优先级与 schema：combatd.c 的 damage_msg(11种damage_type×6档) / guard_msg / catch_hunt_human/beast/bird_msg / winner_msg 如何建模为可迁移的 themed 文本资产？前端如何渲染才能保留 MUD 文学性同时增加现代表现力？
