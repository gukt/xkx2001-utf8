# 交叉质证·资深游戏架构师（20年游戏服务器与引擎架构经验，主导过多个大型 MMO/MUD 类项目，对 LPC/MudOS/现代游戏引擎有深度理解，专注架构承重性、ECS/Actor 选型、分布式性能、技术债务长期代价、复用 vs 自研权衡）

## 同意的论断
- **纯 Python 3.12 作为引擎核心语言无法达到业界标杆--GIL 导致 tick 计算期间全网 I/O 冻结、GC 抖动无法消除、确定性回放在 PYTHONHASHSEED 随机化下脆弱**（游戏引擎资深专家）：我从架构承重性视角完全同意。实测 Python 3.12 asyncio 单线程事件循环在 CPU-bound tick 期间冻结所有网络 I/O，GIL 阻止用线程解决。do_attack 有 29 处 random() 调用 + 同步递归调用图，在 PYTHONHASHSEED 默认随机化下 set 迭代跨进程不稳定，CombatContext 确定性回放存在隐性分叉风险。业界标杆引擎（Bevy=Rust、Unity DOTS=C++/Burst、Godot=C++）全部用系统级语言做核心，纯 Python 路线在第一年就会撞上 GIL+GC 天花板。这是我独立复审中'Python 性能天花板未被正面回应'论断的源码级佐证。
- **引擎工具链完全缺失--场景编辑器/实体检视器/tick replay scrubber/profiler 是标杆引擎与玩具的分水岭**（游戏引擎资深专家）：完全同意，且与我的独立复审'缺可视化调试工具链'高度一致。我额外验证：方案仅在 UGC 评审工作台提了场景图/对话图可视化预览（用于审核而非开发），grep 确认无 entity inspector / tick profiler / combat replay viewer。没有 inspector 和 replay scrubber，调试 8400 文件迁移的行为偏差只能靠 printf，调试成本将失控。工具链应作为阶段 0 一等公民而非 backlog。
- **LPC 同步 call_other 与 Ray 异步 actor 消息根本不同构，do_attack 的同步递归+random 分支在 Ray 下大概率死锁或退化为单线程**（分布式系统资深专家）：我验证了 combatd.c line 772/777（riposte 递归 do_attack）、line 804/812（double_attack 递归）、line 823/831（follow_up 递归），全部由 random() 分支决定。Ray actor 默认 FIFO 非重入，actor A 调 B、B 回调 A 死锁；ray.get() 阻塞事件循环使分布式化为零。方案零处论证 68771 处同步调用如何映射到异步而不破坏调用栈语义与递归回溯。这与我独立复审中'Ray 是 ML 框架非游戏服务器框架'的论断交叉印证。
- **skill_power 不是'可配置表达式'而是状态依赖的累积函数，YAML 表达式无法忠实还原战斗手感**（资深游戏主策划）：源码验证 combatd.c skill_power() line 288-333：level³/3 仅一部分，叠加 jingli_bonus（实时随 jingli 消耗动态变化，50+jingli/max_jingli*50 封顶 150）、apply/attack、apply/defense、fight/dodge 累积态（level*((100+fight/dodge/10)/100)，每次命中重置）、str/dex 系数封顶 300。方案'level³/3 可配置表达式'严重简化。这直接影响 ECS 组件设计--fight/dodge 累积态必须建模为 CombatState 瞬态字段，不能进 YAML 表达式，否则战斗手感漂移。
- **门派特色核心逻辑根本不在 mapping *action 表里，YAML skill 表只能覆盖约 30% 的技能行为**（资深游戏主策划）：验证 dugu-jiujian.c：query_action 有玄铁重剑隐藏大招（str>=30&&int>=36&&random(skill)>80&&force>150&&武器特定，6 条件触发）、hit_by 有 4 级递进反击（random(ap+dp)>ap -> random(ap+2dp)>ap -> random(ap+3dp)>ap，直接 mutate neili/jingli/busy）、valid_learn/valid_enable 有门派绑定（master_id=='feng qingyang'）且未学过 12 种其他剑法。这些全在脚本。方案'技能=行为定义非数据'与'数据驱动 mapping *action 迁移 YAML'自相矛盾，主策划的三层划分（SkillData+SkillBehavior+SkillEffect）是更诚实的架构。
- **代词体系实测是 10 个变量，其中 6 个由 RANK_D 的 7 个动态业务函数求值，'代词替换纯函数下移前端'不成立**（MUD/LPC 历史与社区专家）：验证 emoted.c 注释明确列出 $N/$P/$C/$c/$R/$r/$S/$s 加上 message.c 的 $n/$p 共 10 个变量。RANK_D 的 query_respect/query_rude/query_self/query_self_rude/query_close/query_self_close 等函数依赖年龄/性别/职业/门派/官职/武功等级/善恶/鬼魂状态，是不可下沉的业务规则。这是架构事实不是文化选择--服务端必须为每条事件求值 PronounContext 下发，前端只做 $X 到字段的纯替换。方案 00 总纲 §4.8 的承重论断在代词维度不成立。
- **当前 MVP 验证引擎可行性而非产品价值，核心价值主张 DSL+Agent 创作闭环要到阶段 4-5（14+月后）才被验证，顺序反了**（资深游戏制作人）：从架构风险管控视角同意。引擎重建是确定性工程（风险已知可控），DSL+Agent 创作闭环是未验证的产品假设（风险最高）。把最高风险放最后验证是顺序错误。但我对'用最小 stub 运行时验证'有保留（见 challenged）。
- **以 DSL 为契约把 LLM 非确定性收敛到可验证终态--方向正确但被过度声称，能收敛的只有结构合法性+数值分布，收敛不了叙事连贯/对话语义死锁/任务绕过/经济平衡/趣味性**（AI/Agent 工程资深专家）：从架构契约视角同意。DSL schema（pydantic）收敛结构合法性、world-graph 收敛图可达性、combat-sim 收敛数值分布--这三类是可机器验证的。但 UGC 内容质量承重维度（叙事/任务逻辑/经济平衡/趣味性）全部在验证覆盖外。方案用'可验证终态'一词暗示'内容质量已验证'，实际只验证了'结构合法+数值在分布内'，这是验证覆盖度幻觉。
- **greenfield 下应直接选 PG 唯一权威+Redis 读穿透缓存，彻底消除双写复杂性**（分布式系统资深专家）：完全同意且与我的独立复审一致。greenfield 无旧系统数据迁移包袱，§14 在两选项间犹豫是没享受 greenfield 红利。cache-aside 是游戏服务器成熟模式，与 52985 处同步调用的 read-after-write 语义兼容，故障窗口可明确界定（Redis 降级=读延迟上升，不丢数据）。标杆引擎不应在基础持久化模式上留开放问题。
- **greenfield 0-1 框架下迁移适配层/双栈过渡/securityd 修复等增量假设必须去除**（多位专家共识（游戏架构师/分布式/制作人/MUD历史/AI））：这是 8 位专家中至少 5 位的压倒性共识。我验证：02 文档第 102 行明确有'迁移适配层'、00 阶段 0 明确'录制 LPC 命令流为 golden trace'、§30 明确'过渡期保留 LPC message_vision 经 Telnet 适配器作为真相源'、§31d 明确'双栈过渡一致性测试'--全部是增量概念。greenfield 不存在'适配层''双栈''过渡期''逐步替换''修旧 bug'，应改为'规格提取->按规格实现->行为等价验证'。

## 挑战的论断
- **greenfield 应严肃评估 Rust 核心(PyO3)+Python 脚本混合架构作为阶段 0 架构决策点**（游戏引擎资深专家）
  反对：分阶段 compiled core 策略而非一开始就上 Rust
  理由：方向正确但时机错误。第一，PyO3 跨语言边界开销：do_attack 有 20-30 次组件查询频繁穿越 Rust↔Python 边界，每次穿越有微秒级 marshalling 开销，可能抵消 Rust 性能优势。第二，单人/小团队引入双语言栈是认知负荷灾难--构建系统/调试/类型边界/CI 全部翻倍。第三，greenfield 的'先做对再做快'原则：应先用纯 Python 验证语义正确性和迁移可行性（这才是 8400 文件迁移的真正风险），性能瓶颈实证后再按需引入 compiled core。我的修正立场：阶段 0 纯 Python 原型 + 性能基准（单 do_attack μs 级 + tick 预算拆解）-> 阶段 1 按基准结果决定热路径是否用 Cython（比 Rust 更平滑的 Python 加速路径）或 Rust（仅当 Cython 不够）-> 不引入 compiled core 则诚实承认 Python 是原型验证语言。
- **垂直切片平台验证应在阶段 0 之前用最小 stub 运行时跑 DSL+Agent 创作闭环**（资深游戏制作人）
  反对：stub 运行时会制造 false confidence，应用真实引擎最小子集
  理由：制作人的核心洞察（最高产品风险应先验证）正确，但'stub 运行时'方案有致命缺陷--AI 专家已论证 combat-sim 独立模型与真实引擎脱节制造 false confidence。同样的问题适用于 stub 运行时：如果 stub 的语义与真实 ECS 引擎不一致（例如 stub 不做 do_attack 的同步递归+random 分支、不做 condition 位聚合、不做 Effect stacking），DSL+Agent 验证通过的内容上线后在真实引擎可能崩溃。验证信号失真比没验证更危险。我的修正：垂直切片应基于'真实引擎最小子集'--提取 resolve_attack 为无副作用纯函数（不依赖完整 ECS 运行时但保证验证与未来真实引擎同源），这是 combat 迁移阶段 2.4 的前置工作，提前做不浪费。
- **迁移即重建 RNG 契约，用符号执行/约束求解从 LPC 重建遗留内容的 RNG 调用序列**（分布式系统资深专家）
  反对：符号执行对 LPC 动态类型+闭包+call_other 反射不现实，用运行时插桩更务实
  理由：分布式专家的目标（迁移内容也要有确定性回放）正确，但手段不现实。LPC 是动态类型语言，含 1979 处闭包语法、call_other 字符串反射调用、query_entire_dbase 活引用直改（11+ 字段），符号执行工具链对这种语言几乎不存在。do_attack 的递归深度由 while(random(defense_factor) > my['combat_exp']) 动态决定，约束求解需要建模整个运行时语义。更务实的方式：greenfield 重写的 Python 代码本就用 seeded DeterministicRNG（这是新代码不是遗留代码），通过 property-based testing 验证'相同 seed+相同输入=相同输出'。§10 的'放弃位等价'只适用于'与原 LPC 逐位对照验证'，不应延伸为'运行时放弃确定性'--同意专家的这后半部分论断。
- **可视化场景编辑器应作为 UGC 阶段首要交付物，四层 DSL 作为其底层 IR**（UGC 平台产品专家）
  反对：引擎未定型前做可视化编辑器是本末倒置，但 DSL 门槛问题真实存在
  理由：UGC 专家的方向对（可视化编辑器是 UGC 平台护城河），但时机错误。可视化编辑器操作的数据模型--组件 schema、房间图拓扑、对话树结构、事件规则--全部依赖 ECS 引擎定型。在 ECS/SparseSet/组件设计未稳定前做编辑器，等于在不稳定的地基上盖楼，引擎 schema 一变编辑器全部返工。但 UGC 专家的'四层 DSL 对非程序员门槛过高'是真实问题。我的修正：先稳定数据模型（阶段 0-1），再做可视化编辑器（阶段 2+），期间 DSL 文本格式作为过渡创作方式。同时承认--编辑器应作为引擎工具链的一部分（与 entity inspector/tick profiler 同批交付），而非独立的 UGC 平台能力。
- **Agent 编排应从 LangGraph 迁移到 Temporal/DBOS durable execution**（AI/Agent 工程资深专家）
  反对：Temporal 对单人 MVP 过重，PG-backed state machine 先满足需求
  理由：AI 专家正确识别了 LangGraph 的局限（单进程状态机库、checkpoint 为崩溃恢复设计非创作工作流），但 Temporal 是分布式工作流引擎，引入它意味着额外的基础设施运维（Temporal Server/Worker/Namespace），对单人/小团队 MVP 过重。durable execution 的核心需求--中断-恢复-人类介入--可先用 PG-backed state machine 满足（状态持久化到 PG，checkpoint 支持 resume）。当多创作者并发需求出现时（阶段 3+）再评估 Temporal。LangGraph 作为过渡原型可接受，但需明确标注'非终态选型'。
- **Agent NPC 应作为游戏内实体走与玩家相同的 ECS 实体+tick 驱动，行为由 LLM 实时生成**（资深游戏主策划）
  反对：LLM 推理延迟（500ms-3s）与 tick 仿真（1s 周期）根本冲突，需异步架构而非同步 ECS tick
  理由：主策划的'超越原作的设计可能'方向正确且是标杆差异化点，但架构上不能走'与玩家相同的 ECS tick 驱动'。LLM 推理延迟典型 500ms-3s，而 tick 周期 1s--如果 NPC 在 tick 内同步调用 LLM，单个 NPC 的 LLM 调用就阻塞整个 tick 仿真。Agent NPC 必须走异步事件驱动架构：LLM 调用在独立协程/进程中发起，结果通过事件队列异步注入 ECS World。NPC 的'即时反应'（如战斗中的 perform）走确定性脚本（与 LPC 一致），'深度对话/涌现叙事'走异步 LLM。这是架构分层问题不是'加个 LLM 就行'。
- **业界标杆应定位于'最 MUD 的现代游戏'而非'最现代的 MUD'，文化保真应与差分测试行为保真并列为一等指标**（MUD/LPC 历史与社区专家）
  反对：文化保真是重要维度但不能凌驾于架构承重性之上，二者是不同层级的约束
  理由：MUD 历史专家的'想象力治理'（天雷 themed 反作弊、阴间 themed 死亡惩罚）作为差异化能力我完全同意--这确实是图形 MMO 无法复制的。但'文化保真度与差分测试行为保真并列为一等指标'是范畴混淆。差分测试/行为等价是架构验证层（可机器执行、可自动化、有明确通过标准），文化保真是产品体验层（主观判断、无明确通过标准、依赖人类评估）。架构师视角：代词 10 变量的求值必须留服务端是架构事实（rank 求值依赖业务状态），但'themed 文本美学保留率'作为验收指标无法机器执行。正确做法：架构层保证'代词求值能力完整迁移'（可验证），产品层保证'themed 文本美学保留'（人工评估）。

## 新涌现缺口
- 所有专家都在谈 greenfield 重审，但没人系统梳理'增量重构概念'在 5 份文档中的全部出现位置和统一清理方案。8 位专家中至少 5 位独立指出这个问题，但各自只举了 1-3 个例子。需要做一次全文扫描：'适配层/双栈/过渡期/逐步替换/修旧 bug/golden trace 录制运行中 LPC/差分测试对照运行系统'等概念在文档中的每一处出现，统一替换为 greenfield 语义（规格提取/行为等价验证/spec oracle）。这是承重级文档工程，不做完就无法进入实现。
- 跨专家共识盲区：8 位专家分别从各自视角发现了 greenfield 增量假设，但没有人指出 greenfield 的最大风险是'没有运行中的系统约束，容易过度设计'。greenfield 的自由度是双刃剑--没有'必须兼容旧系统'的约束，也没有'旧系统证明可行'的兜底。方案在没有性能基准、没有容量验证的情况下规划了 ECS+Actor+DSL+Agent 四范式堆叠，这正是'过度设计'的症状。greenfield 应从最小可行架构起步（单进程 asyncio + 普通对象 + 纯函数 resolve_attack），按实证瓶颈逐层引入复杂度。
- 没人讨论'迁移内容作为官方 StdLib'与'UGC 创作'的架构隔离。两者应走不同的代码路径：StdLib 是可信 Python（平台级作者维护，无沙箱限制，可访问全部引擎能力），UGC 是沙箱受限 Python（RestrictedPython + fuel 配额 + 能力令牌）。方案的 CPK 机制只区分了内容包层面，没区分代码执行路径。如果 StdLib 和 UGC 走同一沙箱，StdLib 的门派特色（如 dugu-jiujian 的 hit_by 递归反击）会被沙箱限制无法表达；如果不走沙箱，UGC 作者无法复用 StdLib 的能力。这是承重级架构决策。
- 规格提取的形式化程度未被任何专家讨论。主策划提到'隐藏大招触发条件全量盘点'，AI 专家提到'flag 状态机模型检查'，但没人回答：规格提取应该做多形式化？轻量级（JSON schema + 前置/后置条件断言 + property-based testing）还是重量级（TLA+/Alloy 形式化规约 + 模型检查）？形式化程度影响验证成本和团队负担--重量级对单人项目不可持续，轻量级可能不够覆盖 condition 叠加/effect stacking 的组合空间。需要在阶段 0 明确这个 trade-off。
- 多专家提到性能基准但没人定义'达标标准'。游戏引擎专家说'tick 预算 50ms 或 100ms'但没说这是否含网络 I/O；分布式专家说'单房间 50/100/500/1000 人战斗'但没说 p99 延迟阈值。没有达标标准，基准测试无法做 go/no-go 决策。需要定义：tick 预算（含/不含 I/O）、单分片并发目标（活跃实体数 vs 战斗实体数）、单房间容量上限、tick 抖动 p99/p999 阈值、确定性回放位等价验证标准。
- 门派师徒系统的状态机建模被主策划提出但没人深化。feng.c 的 check_student/check_betrayal/revenge 是完整的师徒关系状态机（拜师条件 13 项/死亡追踪/善恶判断/技能遗忘/叛师永久标记），19 门派各有师徒逻辑。这在分布式下是跨分片关系状态机（师徒可能在不同分片），需要专门的关系一致性协议。方案的 ReferenceStore 提了'关系类型'但没设计跨分片关系状态机。

## 精炼立场
综合 8 位专家的交叉质证与我的源码验证，我精炼的最终架构立场如下：

**一、承重级概念缺陷必须先修正，否则一切技术讨论都建在沙基上**

greenfield 与增量重构的概念混淆是 8 位专家中至少 5 位的压倒性共识，我验证了文档中每一处增量概念的具体位置（02 文档第 102 行'迁移适配层'、00 阶段 0'录制 LPC golden trace'、§30'过渡期保留 LPC message_vision 经 Telnet 适配器'、§31d'双栈过渡一致性测试'、§19'修 securityd.c bug'）。这不是措辞问题而是承重级框架缺陷。必须重写迁移路径：阶段 0 改为'LPC 规格提取与验证基建'（spec extraction + equivalence verification），删除所有'适配层/双栈/过渡期/逐步替换/修旧 bug'概念，securityd 不修旧代码而是从 securityd.c 提取权限规格后从零实现新 PermissionService，差分测试重定义为'规格提取+行为等价验证'（对照本地 MudOS spec oracle + 提取的形式化规格，而非运行中的生产 LPC）。

**二、性能天花板必须正面回应，但用'分阶段 compiled core'而非'一开始就上 Rust'**

游戏引擎专家和分布式专家都正确指出了 Python 纯解释器的三重天花板（GIL/GC/确定性），我完全同意标杆引擎不能回避这个问题。但我修正游戏引擎专家'阶段 0 就评估 Rust 核心'的立场--greenfield 单人/小团队引入双语言栈（Rust+Python）是认知负荷灾难（构建系统/调试/类型边界/CI 全部翻倍），且 PyO3 跨语言边界开销在 do_attack 的 20-30 次组件查询场景下可能抵消性能优势。正确策略是分阶段：阶段 0 纯 Python 原型 + 单 do_attack μs 级性能基准 + tick 预算拆解 -> 按基准结果决定热路径是否用 Cython（比 Rust 更平滑的 Python 加速路径，无需双语言栈）-> 仅当 Cython 不够时才引入 Rust（仅 CombatSystem/SkillSystem/TickSimulator 核心热路径）。不引入 compiled core 则必须诚实承认 Python 是原型验证语言而非标杆引擎实现语言。同时必须固定 PYTHONHASHSEED=0 + 禁用 set 迭代 + 序列化 key 排序，从环境层消除确定性分叉风险。

**三、标杆引擎三大核心能力必须在阶段 0 作为架构决策点确定，而非后期补丁**

全仿真确定性、逻辑热重载、可视化调试工具链--这三项是'够用引擎'与'标杆引擎'的分水岭，且它们不是'后期可选'而是'架构决策点'，因为它们影响从 ECS 设计到序列化协议的全部底层选择。（1）全仿真确定性：所有 System 的 RNG 走 seeded DeterministicRNG（不仅战斗），记录所有外部输入为 input log，支持任意 tick 边界的 state snapshot + input log = 完整重放。迁移内容在 greenfield 重写时本就用 seeded RNG，§10 的'放弃位等价'只适用于'与原 LPC 逐位对照验证'，不应延伸为'运行时放弃确定性'。（2）逻辑热重载：基于 importlib.reload + entity state migration，定义'热重载安全窗口'（无进行中战斗、无跨分片 handoff）。（3）可视化调试工具链：entity inspector / tick profiler / combat replay viewer / world graph visualizer，与引擎同步开发。

**四、分布式层从'假设同构'改为'分片内同步、分片间异步'**

分布式专家的'do_attack 同步递归+random 分支在 Ray 下死锁'论证经我源码验证成立（line 772/777/804/812/823/831 递归调用全由 random() 决定）。Ray 是 ML 分布式计算框架非游戏服务器框架，其 actor 间延迟（1-10ms）对 1s tick 仿真有负担，单进程阶段完全不需要 actor。正确架构：阶段 1 用纯 asyncio + 普通实体对象（同步方法调用与 LPC call_other 语义一致），完全不引入 Ray。同房间/同区域实体 colocate 到同一进程，call_other 退化为进程内同步函数调用。仅在跨分片边界用异步消息。Ray 推迟到阶段 3 真正需要分布式计算时引入，且仅用于无状态/可并行计算（Agent 编排、combat-sim），不做游戏实体运行时。同时必须正视 MUD 的'单房间串行'固有瓶颈--热门房间（主城/PK 场）可能聚集超单分片容量，需房间级水平扩展（多实例+玩家分桶）或下调'万人在线'为'万人注册+千人同时在线'的务实目标。

**五、方案最有价值的差异化点是 DSL-as-contract 的 Agent 协作创作闭环，应作为标杆叙事核心**

8 位专家中多位认可这个方向。但 AI 专家正确指出'验证覆盖度幻觉'--DSL 只收敛结构合法性+数值分布，收敛不了叙事连贯/对话语义死锁/任务绕过/经济平衡/趣味性。要成为真正的标杆，必须补齐'语义/叙事/平衡'三维评估矩阵（LLM-as-judge + 状态机模型检查 + 经济仿真 + 自动化 playtest）。同时制作人建议的'垂直切片验证'应采用'真实引擎最小子集'（提取 resolve_attack 为无副作用纯函数，保证验证与未来真实引擎同源）而非'stub 运行时'（避免 false confidence）。

**六、范式收敛：ECS + DSL 两范式聚焦，Actor 降级为分布式传输层**

游戏引擎专家的'ECS+Actor+DSL+Agent 四范式堆叠过度复杂'论证成立。标杆引擎聚焦少数范式做到极致（Bevy=纯 ECS、Unity DOTS=ECS+Job）。应收敛为：ECS（仿真核心）+ DSL（创作契约）两范式。Actor 退化为 ECS 的分布式传输层（非独立仿真范式），Ray actor 仅作为 ECS World 的跨进程容器。Agent 是 DSL 的生产端消费者（非运行时范式，LangGraph 编排器是 DSL 编译器的前端工具）。

**七、规格提取管线是 greenfield'LPC 作为 spec'的核心工序，方案完全缺失**

主策划、AI 专家和我的独立复审都指向同一个缺口：方案直接从'LPC 源码统计'跳到'Python 实现'，缺少中间的形式化规格提取层。应对每个子系统提取形式化规格契约（do_attack 的输入输出契约、condition 的状态转移图、heal_up 的恢复公式、各门派加成的数学模型），规格以可执行文档（hypothesis property / JSON schema）形式存储，既是实现依据也是验证基线。门派特色（如 dugu-jiujian 的隐藏大招/4 级反击/门派绑定）必须作为 SkillBehavior 策略对象单独提取，不能塞进 YAML 数据表。

**总结**：这个方案在技术调研深度上达到了业界水准（40 条修正多数经我源码验证属实），但要成为'业界标杆的现代化 MUD 引擎'，必须先修正承重级概念缺陷（清除增量重构假设），再正面回应性能天花板（分阶段 compiled core），最后补齐三大标杆核心能力（全仿真确定性/逻辑热重载/可视化工具链）。方案最有前景的差异化点--DSL-as-contract 的 Agent 协作创作闭环--应从阶段 4-5 提前到垂直切片验证，用真实引擎最小子集而非 stub 运行时。

## 开放问题
- compiled core 的引入时机与边界：阶段 0 纯 Python 原型性能基准结果达到什么阈值才触发 compiled core 决策？Cython（更平滑）还是 Rust（更彻底）？PyO3 边界开销在 do_attack 20-30 次组件查询场景下实测是多少？这需要 micro-benchmark 回答。
- do_attack 同步递归调用图在分布式下的正确处理方式：'分片内同步、分片间异步'是否足以覆盖所有递归场景？riposte（受害者反击，同房间）和 double_attack（同实体追加攻击）都是分片内同步可解决，但跨房间追杀/组队跨区域 PK 的战斗上下文如何在分片间传递？CombatContext 快照存哪、seed 如何协调？
- 代词 10 变量求值的服务端性能影响：RANK_D 7 个函数每个都做业务查询（query_skill/query('class')/query('dali/rank')/is_ghost() 等），每条事件对 me/you 都要求值。在 3000 并发、每 tick 数百条事件场景下，代词求值是否成为热路径？是否需要预计算+缓存？
- 规格提取的形式化程度选择：对 do_attack 的 29 处 random()、72 个 condition 的状态转移、effect stacking 的组合空间，property-based testing（hypothesis）能覆盖多少？剩余多少需要形式化模型检查？轻量级与重量级验证的覆盖率差异有多大？
- StdLib 与 UGC 的执行路径隔离如何在 ECS 层实现：是同一 World 不同 System 权限，还是不同 World 实例？StdLib 的门派特色（如 hit_by 递归反击直接 mutate neili/jingli/busy）在受限 Python 沙箱中如何表达？是否需要为 UGC 提供更高层的 Effect API 而非直接 mutate？
- 热重载安全窗口的精确定义：'无进行中战斗、无跨分片 handoff'是否充分？condition 堆叠、call_out 定时器、AOI 订阅在热重载期间如何处理？System 代码版本切换时，旧 System 的 in-flight 事件（如已入队未处理的 Effect）如何迁移到新 System？
- greenfield 的真正红利清单：除了'无旧系统数据迁移包袱可选 PG 唯一权威'和'技术选型自由度'，greenfield 还有哪些红利未被享受？例如'可从第一天就内建全仿真确定性'、'可从第一天就设计热重载协议'、'可从第一天就内建多题材 ThemeRegistry'--这些在增量重构下是改造代价，在 greenfield 下是设计自由。
