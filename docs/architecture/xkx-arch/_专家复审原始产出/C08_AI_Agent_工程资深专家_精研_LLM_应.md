# 交叉质证·AI/Agent 工程资深专家（精研 LLM 应用、Agent 编排、MCP、确定性控制流、LLM 成本与延迟优化、评估方法论）

## 同意的论断
- **把 DSL+Agent 创作闭环这个最高风险的产品假设放到阶段4-5（14+月后）才验证，而前14月全在验证工程可行性（能否重建引擎），顺序完全反了；应在阶段0前插入2-3月'垂直切片平台验证'。**（资深游戏制作人）：这是对我领域最承重的交叉论断。我本人在独立复审中指出'核心论点被过度声称'和'五角色分工缺评估方法论'，但只在验证维度内部打转。制作人的视角让我意识到：整个 Agent 创作闭环根本没被最小化验证过，连'是否收敛'都不知道，却在路线图里把它当成已成立的前提来排后续14个月的工程。作为 AI 主管，我会直接用 Claude+手动编辑先跑通'能否用 DSL 表达一个 5-10 房间世界 + Agent 产出是否可用'，2-3月内拿到真信号，再决定是否投入引擎。这正是'最便宜的方式学到最多的东西'。
- **skill_power 不是可配置表达式而是状态依赖的累积函数（jingli_bonus 实时动态 50-150、fight/dodge 累积态、apply 修饰、busy 减半、str/dex 系数封顶300），YAML 表达式无法还原战斗手感。**（资深游戏主策划）：我核实了 combatd.c 确实有 29 处 random() 与 query_entire_dbase 活引用直改。主策划的发现对我的领域有直接杀伤：方案把 Balancer 角色定位为'武功 action 数值、战斗模拟调参'并产出'层0 技能 + 数值表（YAML）'，但若 skill_power 是状态依赖的累积函数，Balancer 根本无法用 DSL 表达战斗手感--它只能调一张 YAML 数值表，真正的战斗数学骨架（状态修饰链/累积态/封顶）是平台级 Python。这意味着我的'五角色 Worker 分工不合理'论断比初判更严重：不仅缺评估方法论，Balancing 角色对战斗域几乎是无能力产出的虚构。
- **应立项 Agent NPC 作为游戏内实体（LLM 驱动动态对话/涌现叙事/玩家行为感知的剧情分支），这是 LPC 做不到而现在能做的真正差异化。**（资深游戏主策划）：方向我同意，且这正是'业界标杆'相对 Roblox/AI Dungeon 的真实护城河（我在独立复审的标杆评估里也点到实时 NPC 是缺失一环）。但我要在 challenged 里专门反击它'被当 feature 轻松加'的处理方式--游戏内 LLM NPC 是与离线创作截然不同且更难的非确定性管理问题，不能放进同一个 Agent 框架。
- **Agent 应作为'创作副驾驶'（copilot：人类在可视化编辑器中创作，Agent 实时辅助），而非全自动批量生成；人类始终是创作主体。**（UGC 平台产品专家）：部分同意。copilot 模式确实缓解了我的'false confidence'问题--不是靠更强的自动验证，而是把非确定性产物放回人类实时判断回路，绕过了'叙事/趣味性无法自动验证'的死结。但我反对由此推出'全自动模式只在批量迁移用'的二元划分，且要指出 copilot 并不消除验证缺口（见 challenged）。
- **greenfield 下迁移内容即重写为走 CombatRNG 契约（新代码本就用 seeded RNG），§10 的'放弃位等价'只适用于'与原 LPC 逐位对照验证'这一场景，不应延伸为'运行时放弃确定性'，否则反作弊基线对迁移内容是 0 覆盖。**（分布式系统资深专家）：这是对我领域的关键支撑。我的核心论点'收敛到可验证终态'隐含假设运行时是确定性的、Agent 产出能被 combat-sim 验证后真实生效。若迁移内容运行时放弃确定性，则 combat-sim 验证的是一个与实际运行行为漂移的基线--我的'验证信号失真'比初判更严重：不仅 combat-sim 独立模型脱节，连它验证的目标运行时本身都不确定。greenfield 重写即用 seeded RNG 是对的，应作为硬约束。

## 挑战的论断
- **Agent NPC 作为游戏内实体是'LPC 时代做不到、现在能做且无人做成的核心卖点'，应立项作为革命性体验。**（资深游戏主策划）
  反对：主策划把 in-game LLM NPC 当作'加一个设计专项'，严重低估其非确定性管理难度。离线创作 Agent 的非确定性是'批处理、可人工逐项审阅、失败可重来'的温和问题；游戏内 LLM NPC 的非确定性发生在与玩家的实时交互中，每一句 NPC 台词都是非确定的，无法逐句人工审阅，且会直接影响世界状态（NPC 给玩家道具/触发任务/修改关系）。这不是 feature 而是 research problem。可行的架构只能是：LLM 仅生成'对话文本'，所有状态变更（给道具/推进任务/改关系）必须经 DSL 契约的状态机 guardrail 显式授权--即'LLM 出文本、DSL 守状态'的硬隔离。若不这样设计，NPC 一句幻觉就能刷出无限金币或破坏任务逻辑。我的经济仿真验证缺口在此加倍放大：不仅要验 UGC 内容的经济平衡，还要验'LLM NPC 实时交互'不产生经济漏洞，这是组合爆炸级难题。
  理由：证据：hebi.c 双人和合已是'强状态多 actor 协调+递归 call_out+共享 set_temp'，连确定性脚本都难迁移；LLM NPC 是在其上叠加一层每交互一次的非确定性。AI 工程界做实时 LLM 角色的经验（如 Inworld/Convai）全部依赖'状态变更走确定性编排、LLM 只管对话表面'的分层，而非让 LLM 直接 mutate 游戏世界。主策划的提议价值在方向，危险在把 research 当 feature 排期。
- **将可视化场景编辑器列为 UGC 首要交付物，以编辑器+DSL 双向同步替代手写 DSL，能解决'四层 DSL 对非程序员门槛过高'问题。**（UGC 平台产品专家）
  反对：可视化编辑器解决的是'创作人机工程'，不是'产出正确性验证'。我同意编辑器对降低门槛必要，但反对它隐含解决我的验证覆盖度幻觉。你可以在最漂亮的拖拽编辑器里画出一个对话死锁（NPC 要求一个永不触发的 flag）、一个任务绕过、一个经济套利循环--编辑器让你画得更快，也让你错得更快。我的缺口是：叙事连贯/对话语义死锁/任务绕过/经济平衡/趣味性五个维度无自动验证手段，这是与编辑器正交的问题。把编辑器当验证问题的解，是范畴混淆。UGC 专家在承重论断里写'combat-sim+world-graph 只保证结构正确性不能保证内容质量'，却又把编辑器列为首要交付物暗示问题被解决，自相矛盾。
  理由：编辑器产出仍是 IR，IR 仍要过 SchemaValidator/CapabilityAuditor/ResourceBudgetChecker，而这些校验恰好是我指出的'只覆盖结构合法性'的那部分。编辑器不改变验证维度集合，只改变创作入口形态。
- **对迁移内容用符号执行/约束求解从 LPC 重建遗留内容的 RNG 契约，推导 random() 消耗与递归深度的关系，生成确定性边界。**（分布式系统资深专家）
  反对：do_attack 有 29 处 random()，且 do_attack 递归调用自身（riposte line 772/777、double_attack/follow_up line 804/812/823/831），含 while(random(defense_factor)>combat_exp) 循环（line 637）。29 处 random 的分支组合是 2^29 量级，叠加重归递归是路径爆炸，符号执行在此规模下计算不可行。这是把研究级方法当工程步骤呈现。务实得多的替代是：property-based testing（hypothesis 随机生成输入验证不变量）+ 统计回归（伤害分布而非逐位对齐），后者正是方案 §10 已认可的'行为等价'。分布式专家自己论证时也承认'greenfield 重写即用 seeded RNG'才是正解，那符号执行重建就是多余的--新代码直接调 DeterministicRNG 即可，无需从旧代码反推契约。
  理由：符号执行在游戏逻辑迁移上的已知失败模式：路径爆炸+LPC 动态分派（call_other 字符串分派 274 文件 this_object）使静态分析本身不可靠。这与主策划提出的 NewRandom 偏置函数精确复现（务实、可纳入差分断言）是两个量级的方案。
- **代词替换纯函数下移前端（00 总纲 §4.8/02 子系统9）。**（MUD/LPC 历史与社区专家（其论断本身我同意，但我挑战方案对该论断的派生假设））
  反对：这条我本不在 AI 域内，但它直接冲击 Agent 生成内容的正确性，所以从 AI 视角反驳：我核实 rankd.c 的 query_rank/query_respect/query_rude/query_self 全部依赖 ob->query_skill('buddhism'/'lamaism' 等)、ob->query('class')、ob->query('dali/rank')、ob->is_ghost()。这意味着同一 ($N) 在不同玩家身上求值出不同称谓。若 Agent 生成的 emote/对话模板把代词当纯前端字符串替换，它生成的'$P 笑了'在不同玩家身上可能渲染成'前辈笑了'或'秃驴笑了'，而 Agent 在创作时根本不知道最终求值结果--它无法验证自己写的代词在所有职业/官职/善恶组合下是否得体。这把我的'叙事风格一致性验证'缺口扩展为'代词得体性验证'新缺口：Agent 必须理解称谓是状态依赖的，且需对 10 变量×7 视角的组合做风格得体性检查，而这又是一个无自动解的语义维度。
  理由：历史专家的证据（rankd.c 7 函数动态求值）成立，但结论应是'代词求值留服务端'，对 Agent 域的连带影响是：Agent 生成文本时面对的是'未知求值结果的占位符'，这比生成纯文本更难保证质量，应作为新缺口而非仅渲染层问题。

## 新涌现缺口
- 验证基线本身的非确定性：分布式专家揭示迁移内容若运行时放弃确定性（§10），则 combat-sim 验证的目标运行时本身是非确定的--我之前说的'combat-sim 独立模型与真实引擎脱节制造 false confidence'被加码为'即使 combat-sim 调真实 resolve_attack，若迁移内容 RNG 未契约化，验证基线仍在漂移'。验证的根基不稳，这是比单点 false confidence 更深的问题。
- 游戏内 LLM NPC 需要全新的'实时非确定性管理'范式：我原来的评估矩阵只覆盖离线批处理创作（world-graph/combat-sim/叙事 judge/经济仿真/playtest）。看到主策划的 in-game NPC 提议后意识到，实时交互非确定性无法套用'生成-评审-修订'循环（玩家在等，不能迭代），需要的是'生成时硬约束'（structured output+状态机 guardrail）而非'生成后验证'。这是与离线验证完全不同的技术栈，我此前完全没纳入。
- '五角色 Worker'双重崩塌：我原判只指出缺评估方法论+5x context 成本。主策划的 skill_power 论证表明 Balancer 角色对战斗域基本无产出能力（战斗数学是平台级 Python 非 YAML），Continuity 角色的事后审查应用'生成时约束'替代。即五角色中至少两个的角色定位是错的，不是优化问题而是结构错误。需重新设计 Agent 能力清单而非'优化分工'。
- copilot+in-game NPC 双重叠加下的 LLM 成本失控：我原提的'缺成本模型'是针对离线批处理五角色。若同时采纳 UGC 专家的 copilot（高频小生成）和主策划的 in-game NPC（每玩家交互一次生成），LLM 调用从'每 CPK 数十万 token'膨胀到'每玩家每交互数千 token'，成本量级从'每内容'变'每交互'。这把我的成本缺口从'平台运营成本'升级为'直接决定商业模式可行性'的承重项。
- 代词得体性/文化风格一致性验证：历史专家证明代词是状态依赖业务逻辑、emote 是 7 视角带 wizard 署名的文化资产。Agent 生成新题材/emote 时，没有任何自动手段验证'生成的称谓在所有职业×官职×善恶×鬼魂状态下都得体'或'生成的 emote 符合武侠文本美学'。这是叙事质量验证之外的新增'文化/风格验证'维度，当前完全空白，且很可能无法全自动。
- 世界圣经（WorldBible）作为 Agent 创作根的 schema 与自洽性验证缺失：多个专家触及内容质量但无人定义世界圣经为可验证结构化规格。我原已列此为遗漏步骤，但 greenfield 下 Agent 从零生成新题材意味着世界圣经从零构建、无既有内容兜底--它是所有下游 Worker 的共享上下文，其错误会级联。需把它从'Worldbuilder 角色产出物'提升为'显式可验证里程碑'，定义门派关系图/时间线/能力体系的 schema 与自洽规则（如'互敌门派不能有师承关系'）。

## 精炼立场
综合交叉质证后，我对'以 DSL 为契约把 LLM 非确定性收敛到可验证终态'这个核心论点的最终立场是：方向正确，但当前措辞是'验证覆盖度幻觉'，且该幻觉被分布式专家揭示的'迁移内容运行时非确定性'进一步放大。要成为业界标杆，必须做三处不可妥协的修正：

第一，显式分层声明验证覆盖度。停止用'可验证终态'笼统宣称。明确区分'可机器自动验证维度'（结构合法性=pydantic/world-graph、数值分布=combat-sim、不变量回归=断言集）与'当前无自动手段、需半自动或人工的维度'（叙事连贯=LLM-as-judge、对话语义死锁=flag 状态机模型检查、任务绕过=nuXmv/SPIN、经济平衡=多 agent 玩家行为仿真、趣味性=playtest agent+人工抽样）。在文档中对后者逐项声明 gap 与目标覆盖度，而非掩盖。

第二，combat-sim 必须调用从 LPC 提取的真实 resolve_attack 纯函数，而非独立简化模型。我核实 combatd.c do_attack 有 29 处 random、递归 riposte/double_attack、while(random) 循环、query_entire_dbase 活引用直改 11+ 字段，hebi.c 有递归 check_fight+共享 set_temp+闭包 call_out。独立模型若简化这些机制，验证通过的内容上线即失衡--false confidence 比没有验证更危险。纯函数提取是 CombatContext 快照边界本就需要的前置工作，提前做不浪费。同时采纳分布式专家的'迁移即用 seeded RNG'硬约束，否则验证基线本身漂移，我的整个验证论点失效。

第三，把 Agent 创作闭环与 in-game LLM NPC 拆为两个截然不同的非确定性管理问题，分别验证。离线创作是批处理、可人工逐项审阅、失败可重来的温和问题，适用'生成-评审-修订'循环（但需补收敛协议：max_iterations、违反数单调下降断言、超限人工升级）。in-game LLM NPC 是实时、不可逐句审阅、直接影响世界状态的研究级问题，不能放进同一框架，其架构只能是'LLM 出对话文本、DSL 状态机 guardrail 守所有状态变更'的硬隔离--NPC 幻觉不能 mutate 游戏世界。这个分离是标杆叙事的前提：标杆不是'又多一个 Agent 编排框架'，而是'首个把确定性世界仿真（combat/经济）与 LLM 非确定性创作用验证契约深度耦合的 UGC 平台'。

对五角色 Worker，立场从'缺评估方法论'升级为'结构性错误'：Balancer 对战斗域无产出能力（skill_power 是平台级 Python）、Continuity 应改生成时约束。建议五角色降级为'能力清单'而非'固定进程'，单 Orchestrator 持世界圣经摘要（RAG 检索，解决 200k+ token 超窗口）按需唤起专项 worker。LangGraph 仅作过渡原型，中期迁 Temporal/DBOS durable execution 承载创作工作流的中断-恢复-人类介入。

最后，强烈支持制作人提议的'阶段-1 垂直切片平台验证'：2-3 月内在 stub 运行时+浏览器前端上用 DSL 定义 5-10 房间+2NPC+1 战斗+1 任务+1 对话，部分由 Agent 生成，验证三件事--DSL 能否表达、Agent 产出是否可用、玩家是否觉得好玩。这是用最小成本验证我领域最大不确定性的唯一正确顺序。若这一步不通过，后续14月引擎投入对'Agent 创作闭环'这一核心价值主张是沉没成本。

## 开放问题
- property-based testing（hypothesis）+ 统计回归能否在工程可接受成本下替代分布式专家提议的符号执行 RNG 重建？需做对照实验：对 do_attack 递归调用图分别用两种方法，比较不变量发现率与计算成本。
- 单 CPK 创作的 LLM 成本上限是多少？在 copilot+in-game NPC 双重叠加下成本量级如何变化？需建立 token×角色×轮次×交互频率的成本模型并定义创作者配额熔断阈值。
- in-game LLM NPC 的'LLM 出文本/DSL 守状态'硬隔离在工程上是否足以阻止所有状态泄露？需形式化定义'NPC 可触发但须经 DSL guardrail 授权的状态变更集合'，并对 guardrail 绕过做对抗测试。这是研究问题，需先判定是否 tractable 再决定是否排期。
- 可视化编辑器能否约束作者只能在'静态可校验的 DSL 子集'内创作（即编辑器只允许生成可被 flag 状态机模型检查的对话结构），从而把'编辑器产出可验证'从愿望变成类型系统约束？这是编辑器与验证两个缺口能否合并解的关键。
- 世界圣经的 schema 应如何定义才能既作为 Agent 共享上下文又作为可验证规格？需定义门派关系图/时间线/能力体系的结构化 schema 与自洽规则集，并用它做'生成时约束注入'取代事后 Continuity 审查。这是级联失败的根，优先级最高。
- '迁移即用 seeded RNG'硬约束下，对 8400 文件迁移的额外工时成本是多少？需抽样评估：多少文件的 random() 调用序列可机械替换为 DeterministicRNG、多少需因递归/条件分支做设计决策。这直接影响迁移可行性判断。
