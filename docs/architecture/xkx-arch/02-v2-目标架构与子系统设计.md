# 目标架构与子系统设计（v2）

> 本文档是 v2 子系统设计方案的整合，以 [[05-专家对抗复审报告]] 的 14 条共识和专家专项发现为依据。与 v1（[[02-目标架构与子系统设计]]）相比，删除所有增量重构/桥接/Ray 实体运行时假设，新增引擎工具链、世界观治理层、称谓系统、技能三层、CombatKernel 抽象等关键子系统。

---

## 总体架构分层

```
┌─ 接入层 ────────────────────────────────────────────────┐
│ WSGateway · AuthService · SessionManager · InputPipeline│  无状态·水平扩展
└────────────────────────────┬───────────────────────────┘
                             │ CommandMessage{verb,args,ctx,seq}
┌─ 游戏分片层 (WorldShard, 有状态) ──────────────────────┐
│ asyncio ActorRuntime · ECS World(SparseSet) · TickSim   │  按区域共同体分片
│ VerbRouter · CombatCoordinator · Region Supervisor       │
│ Systems: Combat/Skill/Condition/Heal/Equip/Movement/... │
│          WorldGovernance/Title/NPCAI/ToolChain         │
└────────────────────────────┬───────────────────────────┘
                             │ 事件 / 投影 (tick 边界聚合)
┌─ 状态与服务层 ────────────────────────────────────────┐
│ PostgreSQL(唯一权威+审计) · Redis(读穿透缓存) · NATS    │
│ SchedulerService · ProjectionStore · ReferenceStore     │
└────────────────────────────┬───────────────────────────┘
                             │
┌─ UGC 与创作层 ────────────────────────────────────────┐
│ DSL编译器(4层) · CPK+provenance · RestrictedPython/WASM  │
│ Agent 编排(PG-backed状态机) · MCP验证 · ThemeRegistry     │
└──────────────────────────────────────────────────────────┘
```

**寻址统一**：`world://<shard>/<region>/<room>#<entity-uuid>` 替代 LPC `file_name`。

## 子系统 1：网关与会话层

同 v1，保留无状态网关、JWT/argon2 认证、ring buffer 重连。v2 新增：

- **客户端延迟隐藏**：移动命令乐观执行（前端本地渲染 + `client_seq` 上报，服务端 ack，不一致发 rollback delta）。
- **编码中间件**：Big5 转换保留在 InputPipeline 应用层，传输层统一 UTF-8。

## 子系统 2：分布式世界服务器（v2 重大修正）

**理念**：分片内同步、分片间异步；单进程 asyncio + 普通对象起步，Ray 仅作计算层。

### 组件

- **WSGateway 网关层**：见子系统 1。
- **ShardSupervisor**：管理 WorldShard 生命周期。监督策略：玩家/房间 **one-for-one**，区域 **one-for-one + 持久化重建**（非 all-for-one）。
- **WorldShard**：有状态游戏节点。asyncio 事件循环 + SimulationTick（1s）+ AOIManager（同房间实体集合 + 邻域静态元数据只读缓存）。
- **PlacementService**：按 EntityID 寻址。热路径用一致性哈希环 `shard=hash(entity_id)` 无外部依赖解析；Redis 仅控制面（placement 变更通告）。
- **MigrationCoordinator**：跨区 handoff 采用**两阶段提交 + outbox 模式**：
  - 源分片将 handoff 作为持久化事务写入 outbox（PG）。
  - 目标分片从 outbox 拉取并确认；确认后源分片才删除旧实体。
  - handoff 期间实体"只读冻结"（写入命令排队）。
  - 组队批量 handoff 用一个 outbox 事务包裹全队，原子提交。
- **GlobalStateService**：32 守护进程职责重新设计：被 ECS System 取代 / 保留无状态服务 / 演进为新能力。
- **BoundaryRoomAgent**：分片边界房间跨分片事件代理。
- **CombatCoordinator**：region/战斗级 Coordinator 串行裁决回合。同房间参与者同节点亲和，**分片内同步调用保持 68771 处调用栈语义**。热门房间（>30 人）按战斗组分多个子 Coordinator；**热点房间专用分片**；明确"单房间串行是 MUD 固有硬约束"。

### 关键接口

- `EntityAddr: 'world://<shard>/<region>/<room>#<entity-uuid>'`
- 分片内：普通 Python 对象同步方法调用（与 LPC `call_other` 语义一致）。
- 跨分片：异步消息 `ActorRef.send(msg, ctx)` / `ActorRef.ask(msg, ctx, timeout) -> Result`。
- `HandoffRequest{entity_id, target_shard, snapshot, team_members[], seq}`，幂等 + 超时降级。
- `CombatRoundResolved{combat_id, participants[], actions[], rng_seed, ts}`（内嵌逐击明细数组）。

### 旧->新映射

- `file_name` 即身份又即路径 -> EntityID(UUID) 解耦。
- `object` 指针引用 this_object()/environment() -> 分片内直接对象引用，跨分片传 addr 或值快照。
- `call_other` 同步阻塞 -> 分片内同步函数调用，跨分片异步消息。
- `master.c crash()` 全服宕 -> ShardSupervisor 分层监督 + CRASHES 审计。
- `epilog/preload` 16 daemon -> 32 守护进程职责重新设计。
- `room.c` 门状态 find_object 双面协调 -> DoorState 共享存储键(Redis) + BoundaryRoomAgent。

## 子系统 3：ECS 实体系统

**理念**：自研 SparseSet ECS（单存储，不用 Archetype）替换 LPC inherit 编译期静态混入。

### 组件

- **ECS 运行时核心**：Entity 为 64 位整数 ID（高位 shard/region）。SparseSet 存储 O(1) 挂载/卸载/查询。
- **组件族**：Identity / Vitals / Attributes / DerivedAttributes / ModifierStack / SkillSet / CombatState(enemy_id[] + ThreatTable + **fight/dodge 累积态**) / EquipmentSlots / Conditions / Position / Inventory / Wallet / TeamRelation / ActionState / DynamicProps / **Effect** / **Progression** / **MentorshipRelation**。
- **System 调度器**：CombatSystem / VitalsSystem / HealSystem / ConditionSystem / AttributeSystem / SkillSystem / EquipmentSystem / MovementSystem / ActionSystem / FinanceSystem / NPCAISystem / **WorldGovernanceSystem** / **TitleSystem**。
- **EventBus**：替换 `this_object()->method()` 隐式耦合。战斗热路径绕过总线。
- **Prefab/Archetype**：替代 default_ob 单级回退，支持多级继承。

### 关键接口

- `create_entity(prefab_id, overrides) -> EntityID` / `query(*comp_types) -> SparseSetIter`
- `System.update(world, dt, ctx)` 纯函数
- `EffectInstance{effect_id, type, source_entity, target_entity, duration_ticks, mode: tick|wallclock, payload, stages, rng_seed, stacking_policy}`

### 旧->新映射

- inherit F_DBASE + dbase/tmp_dbase -> 类型化 Component schema + DynamicProps
- default_ob 单级回退 -> Prefab 多级原型链
- F_ATTACK enemy/killer object 数组 -> CombatState.enemy_id[] + ThreatTable
- receive_damage/receive_wound 直接 set dbase -> DamageEvent/HealEvent + VitalsSystem（热路径同步）
- query_str 等 apply 聚合 -> Attributes + ModifierStack(分三类) + DerivedAttributes dirty 缓存
- 472 个 SKILL_D 策略对象 -> SkillRef(skill_id, level) 组件 + **SkillData/SkillBehavior/SkillEffect 三层**
- 72 个 condition daemon -> Conditions 组件 + ConditionHandler 协议
- `query_entire_dbase()` 活引用直改 -> 先建字段→组件映射表 + Progression 组件

## 子系统 5：战斗与玩法引擎（v2 重大修正）

**理念**：保留 do_attack 七步管线 + AP/(AP+DP) 概率 + 三层资源，用 **CombatKernel 抽象**泛化，用全仿真确定性保证可回放。

### 组件

- **TickSimulator**：固定 1s tick（MudOS 默认）。按子系统注册 tick 回调。分级 tick 桶。
- **CombatKernel**：抽象接口 `CombatAction` / `CombatResolver` / `DamageModel`。武侠七步是其具体实现；航海/书院/现代格斗可复用同一骨架。
- **CombatSystem**：`resolve_attack()` 纯函数管线。输入 CombatContext 快照，输出 CombatRoundResult。
  - **CombatContext 快照边界**：战斗开始时对全体参与者 + 可观察环境做快照。
  - **管线副作用账本**：七步各自的 mutate 按 step 顺序显式 apply。
  - seeded DeterministicRNG。
- **DamageSystem**：apply_damage/apply_wound/apply_heal/apply_cure。三层资源不变量 0<=current<=eff<=max。
- **HealSystem**：战斗/非战斗不同恢复速率；魔法数字提取 HealRateConfig。
- **ConditionSystem**：ConditionStack + ConditionHandler 协议（on_tick 返回组合结构 continue/suppress_heal/expire）。
- **EffectSystem**：一等公民。stacking_policy(unique/refresh/stack/independent)；mode: tick|wallclock。
- **SkillSystem**：**SkillData(YAML) + SkillBehavior(Python 策略) + SkillEffect(Effect 组件)** 三层。UGC 只编辑 SkillData。
- **DeathSystem**：DeathPolicy 策略对象；阴间/轮回作为 DeathSystem 核心流程。
- **TargetingSystem**：ThreatTable 替代随机选择；MAX_OPPONENT=4 可配。
- **CombatRenderer**：文案抽离 damage_messages.json；**文本体验流 diff** 验证战斗手感。
- **DamageFormula**：三段式可组合公式管线（基础幂 + 状态修饰链 + 属性系数与封顶）；fight/dodge 累积态为 CombatState 瞬态字段。

### 数据模型示例

```yaml
# kungfu/skill/yuenu-jian.skill.yaml
id: yuenu-jian
name: 越女剑法
type: sword
valid_enable: [sword, parry]
actions:
  - action: "$N施展一招「白露横江」，$w横削$n$l"
    force: 100
    dodge: 30
    damage: 20
    damage_type: 割伤
# 平台级 SkillBehavior 文件（非 UGC 可编辑）
behavior:
  hidden_finisher:
    condition: "str>=30 && int>=36 && random(skill)>80 && force>150 && neili>400"
    damage: 200
    special: true
```

### 旧->新映射

- do_attack 直接读写 dbase -> resolve_attack 纯函数读组件快照返回事件
- char.c heart_beat 上帝方法 -> TickSimulator 分系统独立 tick
- attack.c static object *enemy -> CombatState.enemy_id[] + ThreatTable
- skill_power level³/3 -> DamageFormula 三段式管线
- s_combatd.c -> 提取阵法合击/自创连击规格，实现为 CombatModifier
- perform call_out 闭包递归 -> EffectSystem + Effect 多阶段 stages
- condition.c update_condition + CND_CONTINUE 位标志 -> ConditionSystem + 组合返回值
- human.c 门派 if-else 加成 -> FamilyDefinition 门派数据声明 Prefab
- MudOS random() 全局 -> DeterministicRNG + CombatContext 快照
- `NewRandom(i,20,level/5)` -> DSL `weighted_random` 原语

## 子系统 6：数据与状态层

**理念**：**PostgreSQL 唯一权威 + Redis 读穿透缓存**（享受 greenfield 红利）。

### 组件

- **SnapshotStore**：周期/关键节点全量快照。`@persist` 进快照，`@transient` 不进。
- **EventStore**：append-only 领域事件权威持久层。仅高价值事件 + PvP 战报归档。
- **HotStateCache**：Redis 读穿透缓存。在线实体内存态。apply/ 修饰符存为 modifier 链表/栈。
- **EntityStateManager**：load 最近快照 + 重放增量事件 = 当前状态。
- **SchemaRegistry**：pydantic v2 类型化组件。
- **ReferenceStore**：实体间关系存 ID 引用；PostgreSQL 关系表。
- **SchedulerStore**：APScheduler + Redis JobStore 单一后端。分层：短延迟(<30s)留内存，长周期走持久化。
- **ProjectionStore**：CQRS 读侧。Wallet 账本双式记账替代 children() 扫描。
- **WorldStateReconciler**：声明式 desired-state + reconcile。

### 关键接口

- `CommandBus.dispatch(aggregate_id, command, ctx) -> Result`
- `EventStore.append(aggregate_id, events[], expected_seq) -> seq`（乐观并发）
- `SnapshotStore.save/load`
- `EntityStateManager.activate/passivate`

### 旧->新映射

- save_object .o 全量覆盖 -> SnapshotStore + EventStore
- F_DBASE dbase 无 schema -> 类型化 Component + SchemaRegistry
- tmp_dbase static 瞬时 -> Redis TTL + @transient
- default_ob 回退 -> Prefab 查询实例→模板回退链
- autoload path 数组 -> inventory_refs 懒加载
- object 指针引用 -> world:// 统一地址 + ReferenceStore
- call_out -> SchedulerStore（分层）
- heart_beat 全局 tick -> 解耦 System 调度器
- reset 全量扫描 -> WorldStateReconciler 分片/按需 reconcile
- marryd 双方 dbase 双写无事务 -> ReferenceStore 关系事务原子提交

## 子系统 11：引擎工具链（v2 新增）

**理念**：标杆引擎与玩具的分水岭是工具链。阶段 0 一等公民，与引擎同步开发。

### 组件

- **Entity Inspector**：WebSocket 实时查任意实体组件树。支持按 entity_id / 房间 / 组件类型过滤。
- **Tick Profiler**：per-system 耗时火焰图；tick 超时告警。
- **Combat Replay Viewer**：CombatContext 快照可视化回放；支持逐 step 前进/后退/断点。
- **World Graph Visualizer**：房间出口图 + 实时实体分布；高亮热点房间 / 跨分片边界。
- **DSL Preview & Lint**：YAML/对话树实时预览 + 规则校验。
- **热重载控制台**：列出可热重载 System、当前版本、安全窗口状态。

### 关键接口

- `InspectorAPI.get_entity(entity_id) -> ComponentTree`
- `ProfilerAPI.get_tick_flame(tick_id) -> FlameGraph`
- `ReplayAPI.load(combat_id) -> CombatContextStream`

## 跨子系统的一致性约束

1. **EntityID 全局唯一**：`world://<shard>/<region>/<room>#<uuid>`。
2. **ActionContext 贯穿**：所有命令/系统调用携带 ctx。
3. **事件 seq 单调**：每会话/每聚合域单调递增。
4. **@persist/@transient 标注**：决定 snapshot 行为。
5. **Effect 可序列化**：所有持续效果可跨分片迁移、可崩溃恢复。
6. **fail-closed 安全**：PermissionService Redis 不可达默认拒绝。
7. **greenfield 语义**：无双栈/无过渡期/无适配层/无逐步替换 LPC。
8. **单房间串行硬约束**：目标下调为万人注册 + 千人同时在线。
9. **全仿真确定性**：所有 System 走 DeterministicRNG + input log。
10. **热重载安全窗口**：无进行中战斗、无跨分片 handoff。
