# 侠客行 MUD 现代化重构总纲（v2）

> 本文档是 v2 架构总纲。v1 五份文档保留为 [[00-重构总纲]]、[[01-关键修正与避坑清单]]、[[02-目标架构与子系统设计]]、[[03-DSL-UGC与Agent协作]]、[[04-迁移路径与里程碑]]，作为技术调研基线；v2 以 [[05-专家对抗复审报告]] 为修正方向，清除增量重构假设，将项目重新锚定为 greenfield 重写。
>
> **核心前提**：没有旧 LPC 系统需要"逐步迁移"或"桥接"。LPC 代码是规格源（spec/source），新系统从零按规格实现，通过静态契约 + 行为等价验证保证保真。

---

## 1. 愿景与目标

将经典 LPC MUD《侠客行》（8412 个 LPC 文件、6414 房间、21 门派）重构为：

| 维度 | 目标 |
|---|---|
| 后端 | Python 3.12 asyncio + uvloop（阶段 0-2），核心热路径按实测基准决定是否升级为 compiled core |
| 前端 | 现代 Web（React + TypeScript），文本为基线、图形为渐进增强 |
| 部署 | 单进程起步，分布式按需扩展；目标务实下调为万人注册 + 千人同时在线 |
| 平台 | UGC 平台——用户借助 DSL 配置剧情场景，创作者经济 Day1 预留 |
| 创作 | Agent 协作创作多题材世界；MVP 先验证 DSL+Agent 创作闭环是否成立 |

### 三条不可妥协的边界

1. **网关无状态、纯协议层**：WSGateway 只做帧编解码 / 握手 / 限流 / 心跳 / 会话粘附路由，绝不持有游戏状态。
2. **会话与游戏实体解耦**：Session 提升为一等实体，断线只会话挂起（SUSPENDED），角色体留在世界继续被仿真。
3. **渲染下沉前端**：服务端只产语义事件，message_vision 模板 / ANSI 颜色 / damage_msg 文案全部下移前端纯函数渲染层。

### greenfield 的语义澄清

| v1（增量重构口吻） | v2（greenfield 重写口径） |
|---|---|
| 迁移适配层 | 删除；按规格直接实现 |
| 双栈过渡一致性测试 | 删除；无 LPC 运行栈，只有规格 + 本地 MudOS oracle |
| 过渡期保留 LPC message_vision 作为真相源 | message_vision 1626 模板从 Day1 就是必须实现的规格 |
| 修 securityd.c bug 恢复运维可用 | 不修旧代码；从 securityd.c 提取权限规格后从零实现 PermissionService |
| 录制运行中 LPC 命令流为 golden trace | 一次性 golden trace + 单元级行为规约（静态契约为主） |
| s_combatd.c 废弃 | 不存在废弃，只提取阵法合击规格 |
| 32 守护进程全量分类迁移 | 32 守护进程职责重新设计（被 ECS System 取代 / 保留无状态服务 / 演进为新能力） |

## 2. 现状一句话总结

XKX 是"单进程对象世界 + 蓝图/克隆原型 + Feature/Mixin 组合 + 守护进程单例 + heart_beat 全局 tick + .o 平文件持久化"。这套体系在 90 年代是精巧的，但单进程单线程、`file_name` 即身份又即路径、`call_other` 同步阻塞、`object` 指针跨进程不可用，是分布式化的核心障碍。

> 关键认知：**现有实际并发上限是 `MAX_USERS=50`**（`include/login.h` 已验证），而非 `config.xkx` 的 150 或 LPC 注释的 230。不要为 50 用户的系统预先分布式化。

## 3. 总体目标架构（分层）

```text
┌──────────────────────────────────────────────────────────────────┐
│  现代 Web 前端 (React+TS)  语义事件渲染 / 代词上下文 / 状态投影     │
└──────────────────────────────┬───────────────────────────────────┘
                               │ WebSocket (MsgPack + JSON fallback)
┌──────────────────────────────▼───────────────────────────────────┐
│  无状态网关层  WSGateway / AuthService / RateLimiter / InputPipeline│
│  (aiohttp+uvloop, SO_REUSEPORT 多 worker, 会话路由表存 Redis)        │
└──────────────────────────────┬───────────────────────────────────┘
                               │ CommandMessage (verb+args+ctx)
┌──────────────────────────────▼───────────────────────────────────┐
│  游戏分片层 (WorldShard, 有状态)                                   │
│  ┌────────────┐ ┌──────────────┐ ┌────────────┐ ┌──────────────┐ │
│  │ ActorRuntime│ │ ECS World    │ │ TickSimulator│ │CommandRouter │ │
│  │ (asyncio)   │ │ (SparseSet)  │ │ (1s tick)    │ │ (中间件管线) │ │
│  └────────────┘ └──────────────┘ └────────────┘ └──────────────┘ │
│  CombatSystem / SkillSystem / ConditionSystem / HealSystem / ...  │
│  Region Supervisor (one-for-one) / CombatCoordinator 单进程内同步    │
└──────────────────────────────┬───────────────────────────────────┘
                               │ 事件 / 投影
┌──────────────────────────────▼───────────────────────────────────┐
│  状态与服务层                                                     │
│  PostgreSQL (权威状态+审计) │ Redis (读穿透缓存/路由/AOI) │ NATS (扇出) │
│  SchedulerService (持久调度) │ ProjectionStore (读模型)            │
└──────────────────────────────┬───────────────────────────────────┘
                               │
┌──────────────────────────────▼───────────────────────────────────┐
│  UGC 与创作层                                                     │
│  DSL 编译器(4层) │ CPK 内容包+provenance │ RestrictedPython/WASM 沙箱│
│  Agent 编排(PG-backed 状态机起步) │ MCP 验证(world-graph/combat-sim)  │
│  ThemeRegistry(多题材) │ ReviewPipeline(人机协作审批门)           │
└──────────────────────────────────────────────────────────────────┘
```

### 分层职责

- **网关层**：协议、认证、限流、会话路由。无状态水平扩展。
- **分片层**：有状态。asyncio + 普通对象 + SparseSet ECS + tick 仿真。分片内同步调用，跨分片异步消息。
- **状态层**：PG 唯一权威 + Redis 读穿透缓存 + NATS 扇出。**非全量事件溯源**。
- **创作层**：DSL → CPK → 沙箱执行 → MCP 验证 → 修订闭环。Agent 编排与人类协作。

## 4. 核心架构决策（v2 修正版）

### 4.1 对象模型：ECS 替换 inherit+feature

XKX 的 Feature 混入已是"组合优于继承"的雏形。用自研 **SparseSet ECS** 替换编译期静态混入，实现组件运行时动态挂载/卸载（UGC 多题材必需）。

> 只用 SparseSet，不做 Archetype 双存储。5000-8000 实体规模下 Archetype 聚簇收益微乎其微，徒增复杂度。详见 [[01-v2-关键修正与避坑清单]]。

**核心组件族**：Identity / Vitals(qi/jing/jingli+eff/max 三档) / Attributes / DerivedAttributes / ModifierStack / SkillSet / CombatState(enemy_id[]) / EquipmentSlots / Conditions / Position / Inventory / Wallet / TeamRelation / ActionState / DynamicProps(UGC 动态键) / **Effect**(持续效果) / **Progression**(战斗成长) / **MentorshipRelation**(师徒关系状态机)。

**System 替代 daemon+heart_beat**：CombatSystem / VitalsSystem / HealSystem / ConditionSystem / AttributeSystem / SkillSystem / EquipmentSystem / MovementSystem / ActionSystem / FinanceSystem / NPCAISystem / **WorldGovernanceSystem**（世界观治理层）/ **TitleSystem**（称谓系统）。非均匀 tick。

### 4.2 分布式：分片内同步、分片间异步；Ray 仅作计算层

> v2 重大修正：阶段 1 **不引入 Ray 做实体运行时**，用纯 asyncio + 普通实体对象。同房间/同区域实体 colocate 到同一进程，`call_other` 退化为进程内同步函数调用，仅在跨分片边界用异步消息。Ray 推迟到阶段 3 真正需要分布式**计算**时引入，且仅用于无状态/可并行计算（Agent 编排、combat-sim 批量、离线迁移工具）。详见 [[05-专家对抗复审报告]] 共识 3。

**分片单元=区域共同体**：对 6414 房间出口图跑 Louvain 社区检测得 5-8 个共同体作为初始分片（边界对齐社区割边使跨分片移动最小化）。AOI 退化为"同房间实体集合 + 邻域静态元数据只读缓存"（MUD 是房间作用域，非 MMO 连续空间 AOI）。

> 热路径去 Redis 化：会话/实体路由用一致性哈希环（`shard=hash(entity_id)`）无外部依赖解析，Redis 仅作读穿透缓存与控制面（placement 变更通告）。

### 4.3 战斗引擎：保留七步管线，tick 驱动，全仿真确定性

保留 do_attack 七步管线（选技能→取招式→AP/DP 命中→PP 招架→伤害计算→施加伤害→经验/反击）作为 `CombatSystem.resolve_attack()` 纯函数管线。AP/(AP+DP) 概率模型原样保留。

> v2 修正：
>
> - 战斗开始必须对全体参与者 + 可观察环境做状态快照 = **CombatContext**，seeded RNG 只是其一部分。回放从 CombatContext 起点重放，禁止中途引入未快照输入。
> - `resolve_attack` 同步返回 CombatRoundResult 并就地应用 + 通知同房间订阅者，仅在 tick 边界发一次聚合事件到总线供投影层。
> - Vitals 运行时用裸 dataclass + 断言，Pydantic 仅在 snapshot 落盘 / UGC 上传 / 跨分片反序列化时校验。
> - ConditionHandler.on_tick 返回组合结构 `(continue, suppress_heal, expire)`，显式定义聚合规则。

### 4.4 数据与状态：PG 唯一权威 + Redis 读穿透

> v2 修正：greenfield 无旧数据迁移包袱，直接选 **PG 唯一权威 + Redis 读穿透缓存（cache-aside）**。写先入 PG，再回填 Redis；Redis 降级 = 读延迟上升，不丢数据。故障窗口明确。

状态变更直接写 PostgreSQL entity_state 表（行级权威）+ Redis 读穿透；仅对高价值领域事件（死亡/交易/婚姻/升级/门派变更）写 append-only audit_events 表。ES 仅在需严格回放的子域（PvP 战报归档）单独应用。

### 4.5 命令与权限：显式管线 + 能力令牌

command_hook god-function 拆为显式中间件管线（解码→限流→历史→别名→路由→授权→执行→审计）。`previous_object()` 隐式信任链显式化为 ActionContext + CapabilityToken。securityd 的目录级 ACL 升级为按 (subject, action, resource) 求值的能力策略引擎。

>
> - `call_out` 迁移要分类：3570 处字符串方法名调用天然可序列化；144 处闭包逐一处理为 Effect。
> - PermissionService 必须明确 **fail-closed 默认策略**（Redis 不可达时拒绝所有命令，仅 look/inventory 等只读无副作用命令可选 fail-open），内置本地策略快照兜底。
> - 调度系统收敛为**单一后端**（APScheduler + Redis JobStore 或自研 ZSET worker 二选一）。

### 4.6 DSL 与 UGC：分层混合 DSL，声明为主脚本为辅

四层 DSL，全部编译为 JSON IR 作为唯一真相源：

- **层0**：YAML 声明式场景数据（exits/objects/item_desc/cost/outdoors，对应 2482 个 replace_program 纯数据房间）
- **层1**：事件规则 `condition→action`（对应 add_action/valid_leave/accept_object/chat_msg）
- **层2**：对话树（Ink 的 knot/stitch/divert/weave，对应 inquiry）
- **层3**：脚本逃生舱（RestrictedPython 用于 UGC / WASM 用于平台级已审计扩展）

> heart_beat 不可被事件驱动替代，故层1事件规则只覆盖触发器（进房/命令/对话），战斗回合/NPC AI/持续效果保留周期 tick 语义。

### 4.7 Agent 协作创作：DSL 为契约，PG-backed 状态机起步

以 DSL 为唯一契约层把 LLM 非确定性收敛到可验证终态。Orchestrator 持世界圣经摘要，按需唤起专项 worker 能力（Worldbuilder / Narrator / Behaviorist / Balancer / Continuity 降级为能力清单而非固定进程）。MCP 共享验证基底（world-graph 出口可达性 + combat-sim 伤害/胜率）。

> v2 重大修正：
>
> - 五角色 Worker 降级为"能力清单"，单 Orchestrator + RAG 检索解决 200k+ token 超窗口。
> - Balancer 对战斗域几乎无产出能力（skill_power 是平台级 Python），Continuity 改为"生成时约束"（structured output + 实体关系约束注入）替代事后审查。
> - combat-sim **直接调用从 LPC 提取的 `resolve_attack` 纯函数**，不依赖完整 ECS 运行时但保证验证与未来真实引擎同源。

### 4.8 前端与状态同步：语义事件流 + 客户端投影渲染

服务端维护单一权威状态树，变更通过对比前后权威树生成 JSON-Patch ops + 单调 seq。渲染层完全下沉前端。

> v2 修正：
>
> - dbase 有函数值/闭包键，必须服务端求值后下发，绝不进 snapshot。
> - message_vision 1626 模板从 Day1 就是必须实现的规格，不存在"过渡期保留 LPC"。
> - 代词体系是 10 变量（`$N/$n/$P/$p/$C/$c/$R/$r/$S/$s`），其中 6 个由 RANK_D 的 7 个动态业务函数求值。**服务端**必须为每条事件的 me/you 预先求值 RANK_D 7 函数，产出结构化 `PronounContext` 作为事件 payload 下发；前端只做 `$X -> context[X]` 的纯字符串替换。

### 4.9 全仿真确定性与逻辑热重载

> v2 新增标杆分水岭能力：
>
> - **全仿真确定性**：所有 System 的 RNG 走 seeded DeterministicRNG，记录所有外部输入为 input log。任意 tick 快照 + 输入日志 = 完整重放。
> - **逻辑热重载**：设计 System 热重载管线（`importlib.reload` + entity state migration + schema migration），定义"热重载安全窗口"（无进行中战斗、无跨分片 handoff）。

### 4.10 运维可观测与部署

K8s + Helm + Argo CD GitOps，OpenTelemetry 统一 metrics/logs/traces。

> - 有状态游戏分片不能套用无状态 HPA。改为"预测性 + 预留容量"模型。
> - 战斗/数值变更本质是"灰度即全量"，走蓝绿全量切换。
> - liveness 探针仅判"事件循环死锁"，绝不基于 tick p99/p50。
> - 日志后端用 OpenSearch/ClickHouse，不用 Loki 做高维检索。

## 5. 迁移路径（风险驱动，分阶段）

```text
阶段 -1  垂直切片平台验证（2-3 月）★ 新增，最高优先级
  ├─ 5-10 房间 + 2 NPC + 1 战斗 + 1 任务 + 1 对话，全部 DSL 定义
  ├─ Agent 部分生成（验证 copilot 模式 + 产出可用性）
  ├─ 跑在真实引擎最小子集（resolve_attack 纯函数 + stub 运行时）

阶段 0  规格提取与验证基建（2-4 月）★ 重定义
  ├─ LPC 规格提取管线（形式化规格契约：前置/后置条件、不变量、状态机、概率分布）
  ├─ FluffOS Linux 编译可行性验证（或降级为单元级行为规约）
  ├─ 单元级行为规约（go/move/combat 输入输出契约，greenfield 主门禁）
  ├─ 性能基准：单 do_attack μs 级 + tick 预算定义 + GC 基准 + PYTHONHASHSEED=0
  ├─ 引擎工具链 PRD（entity inspector / tick profiler / replay scrubber）
  ├─ 抽样校准实验：68771 调用点按难度分层抽 50-100 个实测工时
  └─ 灵魂系统盘点（阴间/武林大会276文件/vote/法院/intermud + 5 题材文化系统）
  ★ 不做：修 securityd 旧代码、录制运行中 LPC

M1  确定性回放引擎开源（3 月）

阶段 1  单进程 asyncio 核心循环（纯 Python，不引入 Ray）
  ├─ ECS 骨架（SparseSet，纯 asyncio + 普通对象，分片内同步）
  ├─ 全仿真确定性（所有 System seeded RNG + input log）
  ├─ 逻辑热重载协议（System 热替换 + entity state migration）
  ├─ 引擎工具链与引擎同步开发（entity inspector / replay scrubber）
  ├─ PG 唯一权威 + Redis 读穿透（享受 greenfield 红利）
  └─ combat-sim 调用真实 resolve_attack 纯函数

M2  DSL+Agent 创作闭环 demo（2-3 月）

阶段 2  按规格实现子系统（非"逐子系统迁移"）
  ├─ 规格提取 → 按规格实现 → 行为等价验证（对照静态契约 + 本地 MudOS oracle）
  ├─ 战斗手感验证：文本体验流 diff（message_vision 输出顺序 + 代词替换）
  └─ 技能三层（SkillData YAML + SkillBehavior Python 策略 + SkillEffect）
  ★ Combat 迁移前必须：CombatContext 快照边界 + 管线副作用账本 + Effect 一等公民

M3  单题材（武侠）完整可玩 demo（6-8 月）

阶段 3  分布式与多分片（按需，单节点达 80% 承载后）
  ├─ 分片内同步、分片间异步（Ray 仅计算层，不做实体运行时）
  ├─ handoff 两阶段提交 + outbox 模式
  ├─ 热点房间专用分片（非实例化分桶）
  └─ 务实目标：万人注册 + 千人同时在线

阶段 4  UGC 平台（Day1 设计，与迁移并行）
  ├─ 可视化场景编辑器（引擎定型后做，非 UGC 阶段首要）
  ├─ 分层内容审核 + 版权合规（金庸衍生内容 71 文件需清洗）
  ├─ 创作者经济 MVP（市场 + 评分 + 分账）
  └─ 沙箱配额 + per-CPK 资源隔离

阶段 5  Agent 协作 + 多题材
  ├─ 五角色降级为"能力清单"，单 Orchestrator + RAG 检索
  ├─ Agent NPC（严格分层：确定性 AI + 独立对话服务 + DSL guardrail）
  ├─ 评估矩阵（结构/数值/经济/任务逻辑/叙事/趣味 六维）
  └─ 武侠为旗舰深耕，其余题材为 UGC 扩展
```

### 阶段性硬约束（不可妥协）

1. **垂直切片通过**：阶段 -1 验证失败则不投入引擎重构。
2. **规格提取**：greenfield 核心工序，不可跳过。
3. **性能基准 + 工具链 PRD**：阶段 0 一等公民。
4. **combat-sim 调真实 resolve_attack 纯函数**，非独立简化模型。
5. **迁移即用 seeded RNG**，运行时全量确定性。
6. **handoff outbox 规约**，非承诺。
7. **单房间串行是硬约束**，目标务实下调为千人同时在线。

## 6. 业界标杆的五大差异化能力

v2 明确：标杆的护城河不是"把 LPC 迁到 Python"，而是以下能力：

1. **DSL-as-contract 的 Agent 协作创作闭环**：schema 强约束把 LLM 非确定性收敛到可验证终态，补齐语义/叙事/平衡三维评估矩阵。
2. **全仿真确定性 + 时间旅行调试**：任意 tick 快照 + input log = 完整重放，反作弊/调试/自动化测试共享底座。
3. **逻辑热重载不停服**：MUD 引擎 vs 普通 WebSocket 服务器的核心差异。
4. **themed 治理范式 + intermud 联邦演进**：天雷/阴间/法院/vote 是图形 MMO 无法复制的"想象力治理元宇宙"。
5. **Agent NPC 作为游戏内实体**：LPC 做不到的动态对话/涌现叙事，但先判定 tractable 再排期。

## 7. 技术选型总表

| 层 | 选型 | 说明 |
|---|---|---|
| 运行时 | Python 3.12 asyncio + uvloop | 统一异步运行时；热路径按 μs 基准决策是否升级 compiled core |
| WS 网关 | aiohttp/websockets + SO_REUSEPORT | 无状态多 worker |
| REST/UGC API | FastAPI | 独立扩缩容 |
| ECS | 自研 SparseSet（单存储） | 针对 MUD 访问模式优化 |
| 协议 | MsgPack（默认）+ JSON（UGC）+ Protobuf（仅瓶颈） | 统一 {type,version,seq,trace_id,payload} 帧 |
| 认证 | argon2-cffi + PyJWT(RS256) | 替代 DES crypt() |
| 权威存储 | PostgreSQL 16（状态+审计，非全量 ES） | WAL 强一致，JSONB 灵活 |
| 热状态 | Redis 7 Cluster | 读穿透缓存 / 路由 / AOI |
| 消息扇出 | NATS JetStream（可选，Redis Stream 起步） | 不过早引入 Kafka |
| 调度 | APScheduler + Redis JobStore（单一后端） | 分层：短延迟留内存 |
| 前端 | React + TypeScript + Zustand/Jotai + Immer | xterm.js 可选终端模式 |
| 序列化 | msgspec（高频）/ orjson | Pydantic v2 仅边界校验 |
| DSL | Lark（EBNF）+ PyYAML + pydantic v2 + jsonschema | 四层编译为 JSON IR |
| 沙箱 | RestrictedPython（UGC）/ WASM（平台级） | wasmtime-py 燃料计量 |
| Agent | PG-backed 状态机（MVP）+ Claude API | LangGraph 仅过渡原型，终态迁 durable execution |
| 可观测 | OpenTelemetry + Prometheus + OpenSearch/ClickHouse + Tempo | 非 Loki 做高维检索 |
| 编排 | K8s + Helm + Argo CD + Argo Rollouts | GitOps |
| 弹性 | HPA（无状态）+ KEDA（预测性+预留） | 有状态分片蓝绿不金丝雀 |
| 验证 | 单元级行为规约 + 一次性 golden trace + property-based testing | greenfield 主门禁 |
