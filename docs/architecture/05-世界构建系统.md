# 世界构建系统

## 概述

《侠客行》MUD 的世界构建系统是一个基于文件系统的、高度可扩展的虚拟世界架构。整个世界由数千个独立的房间对象组成，这些房间通过出口（exit）相互连接，形成一个庞大的武侠世界网络。世界构建系统与 [[01-架构总览]] 中的其他子系统紧密协作，共同构成完整的游戏体验。

## 一、世界组织结构

**少林寺的入口控制**：

少林寺山门（shanmen.c）对进入者有严格限制，体现了门派区域的特点：

```c
int valid_leave(object me, string dir) {
    object *inv;
    mapping myfam, mygen;
    int i;
    
    // 女性不得擅入，除非是罗汉堂获胜者
    if (me->query("gender") == "女性" && dir == "eastup" 
        && !me->query("luohan_winner")) {
        if (objectp(present("xu tong", environment(me))))
            return notify_fail("虚通伸手拦住你白眼一翻说道：千年以来，少林向不许"
                "女流擅入。\n虚通又道；" + RANK_D->query_respect(me) + "请下山去吧，免得"
                "自讨没趣。\n");
        else if (objectp(present("xu ming", environment(me))))
            return notify_fail("虚明迈步挡在你身前，双手合什说道：阿弥陀佛，"
                "这位" + RANK_D->query_respect(me) + "请留步，恕小寺不接待女"
                "客。\n虚明顿了一顿又说：女施主如果一心向佛，何妨去峨眉山随喜？\n");
    }
    
    // 非本派弟子必须放下兵刃
    myfam = (mapping)me->query("family");
    if ((!myfam || myfam["family_name"] != "少林派") && dir == "eastup") {
        inv = all_inventory(me);
        for (i = sizeof(inv)-1; i >= 0; i--)
            if (inv[i]->query("weapon_prop"))
                if (objectp(present("xu tong", environment(me))))
                    return notify_fail("虚通拦住你说道：这位" +
                        RANK_D->query_respect(me) + "请放下兵刃。少林千年的"
                        "规矩，外客不得持兵刃上山。\n");
                else if (objectp(present("xu ming", environment(me))))
                    return notify_fail("虚明迈步挡在你身前，双手合什说道：阿"
                        "弥陀佛，这位" + RANK_D->query_respect(me) + "请收起"
                        "兵刃。佛门清静之地，不便随意舞刀弄剑。还请施主鉴"
                        "谅。\n");
    }
    return ::valid_leave(me, dir);
}
```

**武当山的入口控制**：

武当山玄岳门（shanmen.c）对邪恶玩家有特殊限制：

```c
int valid_leave(object me, string dir) {
    mapping myfam;
    object wp, *inv;
    int i, j = 0;
    
    // ... 守卫任务相关逻辑 ...
    
    // 杀气重的邪恶玩家不得上山，除非是敬香
    if (dir == "southup" && (me->query("shen") < -99)
        && !objectp(present("incense", me))) {
        inv = all_inventory(this_object());
        
        for (i = 0; i < sizeof(inv); i++)
            if (userp(inv[i]) && (string)inv[i]->query("job/description") == "守卫山门") {
                j++;
                if (objectp(wp = inv[i]->query_temp("weapon")))
                    message_vision("$N对$n喝道：尔等邪魔歪道人物，要想过此山门，得先问问我手中的"
                        + wp->query("name") + "答应不答应！\n", inv[i], me);
                else
                    message_vision("$N对$n喝道：只要我在，尔等邪魔歪道人物，便休想过此山门！\n", inv[i], me);
            }
        
        if (objectp(present("zhike daozhang", environment(me))))
            return notify_fail("知客道长拔出长剑对着你：如不是上山敬香，即刻请回！\n");
        
        if (j) return notify_fail("\n");
    }
    
    return ::valid_leave(me, dir);
}
```

### 1.4 区域内部层次结构

典型的区域（尤其是门派区域）内部通常遵循以下层次结构：

```
第一层：入口（山门/海港/广场）
  ↓
第二层：过渡区域（道路/走廊/前院）
  ↓
第三层：核心区域（广场/大殿/主要建筑）
  ↓
第四层：功能区域（练功房/厨房/仓库）
  ↓
第五层：隐秘区域（密室/山洞/禁地）
```

这种设计既保证了区域的层次感，又便于设置安全检查和剧情触发点。

## 二、房间对象模型

### 2.1 房间继承体系

房间对象的核心继承链如下：

```
ROOM (/inherit/room/room.c)
  ├── 继承 F_DBASE（数据存储功能）
  ├── 继承 F_CLEAN_UP（清理功能）
  ├── BANK (/inherit/room/bank.c)
  ├── FERRY (/inherit/room/ferry.c)
  ├── HARBOR (/inherit/room/harbor.c)
  ├── HOCKSHOP (/inherit/room/hockshop.c)
  ├── SHIP (/inherit/room/ship.c)
  ├── P9ROOM (/inherit/room/p9room.c)
  └── PIGROOM (/inherit/room/pigroom.c)
```

绝大多数普通房间直接继承 ROOM 基类，特殊功能房间则继承相应的子类。

### 2.2 ROOM 基类核心属性

ROOM 基类定义了房间的基本属性和行为。以下是核心属性：

| 属性设置函数 | 用途 | 示例 |
|-------------|------|------|
| set("short") | 房间短名称 | "中央广场" |
| set("long") | 房间详细描述 | 长文本描述 |
| set("exits") | 出口映射 | ([ "east": ... ]) |
| set("objects") | 房间内的物品/NPC | ([ "npc/liumang": 1 ]) |
| set("item_desc") | 可查看物品的描述 | ([ "dong": ... ]) |
| set("cost") | 移动消耗的内力 | 1 |
| set("outdoors") | 是否为户外房间 | "city" |
| set("no_sleep_room") | 禁止休息 | 1 |
| set("no_drop") | 禁止丢弃物品 | 1 |
| set("day_shop") | 日间商店 | 1 |

#### 2.2.1 房间描述系统

房间描述是玩家感知世界的主要方式，分为短描述（short）和长描述（long）：

```c
set("short", "中央广场");
set("long", @LONG
城市的正中心是一个很宽阔的广场，地面上铺着青石，经常有艺人在这里表
演。一些游手好闲的人在这里溜溜达达。中央有一棵大槐树，盘根错节，据传已
有千年的树龄，是这座城市的历史见证。树干底部有一个很大的洞(dong)。你可
以看到北边有来自各地的行人来来往往，南边人声鼎沸，一派繁华景象，东边不
时地传来朗朗的读书声，西边则见不到几个行人，一片肃静。
LONG
);
```

长描述通常采用 `@LONG ... LONG` 的 heredoc 格式，便于书写多行文本。

#### 2.2.2 物品描述系统

房间中可查看的物品通过 `item_desc` 映射定义：

```c
set("item_desc", ([
    "dong" : "这是一个黑不溜湫的大洞，里面不知道有些什么古怪。\n",
]));
```

`item_desc` 也支持函数指针形式，可以动态生成描述：

```c
set("item_desc", ([
    "gaoshi" : (: look_gaoshi :),
]));

string look_gaoshi() {
    // 动态生成告示内容
    return str;
}
```

### 2.3 出口系统

出口系统是房间连接的核心机制，通过 `exits` 映射实现：

```c
set("exits", ([
    "east" : __DIR__"dongdajie1",
    "south" : __DIR__"nandajie1",
    "west" : __DIR__"xidajie1",
    "north" : __DIR__"beidajie1",
]));
```

#### 2.3.1 方向类型

常见的方向包括：

| 方向 | 说明 |
|------|------|
| east / west / south / north | 基本四个方向 |
| northeast / northwest / southeast / southwest | 斜向 |
| eastup / westup / southup / northup | 向上（通常用于上山） |
| eastdown / westdown / southdown / northdown | 向下 |
| enter | 进入某个建筑/洞穴 |
| out | 出去 |
| up / down | 上下楼 |

#### 2.3.2 条件出口与动态出口

出口可以在运行时动态添加或删除，例如华山的石门：

```c
// 打开石门时添加出口
set("exits/enter", __DIR__"mishi");

// 关闭石门时删除出口
delete("exits/enter");
```

#### 2.3.3 valid_leave 验证函数

`valid_leave` 函数用于检查玩家是否可以离开房间，是实现区域控制的重要手段：

```c
int valid_leave(object me, string dir) {
    // 检查条件
    if (/* 不允许通过 */) {
        return notify_fail("你不能从这里离开！\n");
    }
    
    // 调用父类的 valid_leave
    return ::valid_leave(me, dir);
}
```

### 2.4 门系统（DOOR）

ROOM 基类内置了门系统，用于处理可开关的门：

```c
// 创建门
void create_door(string dir, mixed data, string other_side_dir, int status);

// 门的状态
#define DOOR_CLOSED    1
#define DOOR_LOCKED    2
#define DOOR_SMASHED   4
```

门系统的主要函数：

| 函数 | 用途 |
|------|------|
| create_door() | 创建门 |
| open_door() | 开门 |
| close_door() | 关门 |
| look_door() | 查看门 |
| check_door() | 检查门状态 |
| query_doors() | 获取所有门 |
| query_door() | 获取特定门 |

### 2.5 房间类型与特殊房间

#### 2.5.1 BANK（钱庄）

钱庄继承自 BANK 类，提供货币兑换功能：

## 三、世界连接图

### 3.1 房间连接原理

房间通过 `exits` 属性互相连接，形成一个有向图。每个房间是图中的节点，每个出口是图中的边。

连接示例：

```
扬州广场 (guangchang.c)
    ├─→ 北大街1 (beidajie1.c)
    │       └─→ 北门 (beimen.c)
    │              └─→ 郊外...
    ├─→ 南大街1 (nandajie1.c)
    │       └─→ 南门 (nanmen.c)
    │              └─→ ...
    ├─→ 东大街1 (dongdajie1.c)
    │       └─→ 东门 (dongmen.c)
    │              └─→ ...
    └─→ 西大街1 (xidajie1.c)
            └─→ 西门 (ximen.c)
                   └─→ ...
```

### 3.2 区域间连接

区域之间通过特定的"枢纽房间"连接，主要的连接方式包括：

#### 3.2.1 道路连接

野外道路将不同区域连接起来，例如终南山道路连接峨嵋山：

```c
set("exits", ([
    "west" : __DIR__"xiaojing1",
    "northup" : __DIR__"shanlu1",
    "south" : "/d/emei/emroad4",  // 跨区域连接
    "east" : __DIR__"puguansi",

```

#### 3.2.2 水道连接

通过海港和船只连接岛屿和大陆，例如桃花岛海港：

```c
object ship = new("/clone/ship/seaboat");
ship->move(environment(this_object()));
this_object()->set("exits/enter", base_name(ship));
ship->set("exits/out", base_name(this_object()));
```

### 3.3 虚拟区域与 VIRTUAL_D

VIRTUAL_D（虚拟区域守护进程）位于 `/adm/daemons/virtuald.c`，设计用于处理动态生成的区域。当前版本的实现很简单：

```c
void create() {
    seteuid(getuid());
}

object compile_object(string file) {
    return 0;  // 返回 0 表示不使用虚拟对象
}
```

虽然当前未启用，但这一设计为将来的动态区域生成预留了扩展空间。

## 四、NPC 与物品分布机制

### 4.1 NPC 放置系统

NPC 通过 `set("objects")` 在房间中定义：

```c
set("objects", ([
    __DIR__"npc/liumang" : 1,         // 1 个流氓
    __DIR__"npc/liumangtou" : 1,      // 1 个流氓头
    CLASS_D("emei") + "/ji" : 1,      // 1 个峨嵋派纪姓 NPC
    "/kungfu/class/shenlong/zhang" : 1,  // 1 个神龙岛张姓 NPC
]));
```

NPC 定义支持以下形式：

| 形式 | 说明 |
|------|------|
| __DIR__"npc/name" | 区域内 NPC |
| CLASS_D("pai") + "/name" | 门派 NPC |
| "/kungfu/class/pai/name" | 武功教师 NPC |

### 4.2 物品生成与刷新机制

ROOM 基类的 `reset()` 函数负责物品和 NPC 的刷新：

```c
void reset() {
    mapping ob_list, ob;
    object *inv;
    string *list;
    int i, j;
    
    set("no_clean_up", 0);
    ob_list = query("objects");
    inv = all_inventory(this_object());
    
    if (!mapp(ob_list)) {
        // 清理多余的物品
        for (i = 0; i < sizeof(inv); i++) {
            if (inv[i]->is_character()
                || inv[i]->query("no_clean_up")
                || inv[i]->query("no_refresh")
                || !clonep(inv[i])) continue;
            destruct(inv[i]);
        }
        return;
    }
    
    // ... 检查并刷新物品/NPC ...
    
    for (i = 0; i < sizeof(list); i++) {
        // ... 生成物品/NPC ...
    }
}
```

`make_inventory()` 函数实际创建对象：

```c
object make_inventory(string file) {
    object ob;
    
    if (objectp(ob = new(file))) {
        // 避免门派 NPC 重复克隆
        if (strsrch(base_name(ob), "/kungfu/class/") == 0
            && sizeof(filter_array(children(base_name(ob)), (: clonep :))) > 1) {
            npc_clean_up(file);  // 清理旧的，创建新的
            ob = new(file);
            if (!ob) return 0;
        }
        
        ob->move(this_object());
        ob->set("startroom", base_name(this_object()));
        if (living(ob) && environment(ob))
            tell_room(environment(ob), ob->query("name") + "急急忙忙走了过来。\n");
        return ob;
    }
    
    return 0;
}
```

### 4.3 NPC 回家机制

如果 NPC 离开了初始房间，`reset()` 会尝试让它们回家：

```c
if (environment(ob[list[i]]) != this_object()
    && ob[list[i]]->is_character()) {
    if (!ob[list[i]]->return_home(this_object()))
        add("no_clean_up", 1);
}
```

NPC 类实现了 `return_home()` 函数，使其能够返回起始房间。

### 4.4 商店（Vendor）实现模式

商店通常是一个包含特殊 NPC 的房间，NPC 负责交易。典型商店包括：

- 打铁铺（datiepu.c）：铁匠出售武器
- 当铺（dangpu.c）：收购物品
- 酒馆（jiuguan.c）：出售食物和饮料
- 药铺（yaopu.c）：出售药品

商店房间通常设置 `day_shop` 属性，表示只在白天营业：

```c
set("day_shop", 1);
```

## 五、世界叙事设计

### 5.1 房间描述的写作风格

房间描述是构建武侠世界氛围的主要手段，具有以下特点：

#### 5.1.1 视觉描绘

详细描述环境，营造代入感：

```
城市的正中心是一个很宽阔的广场，地面上铺着青石，经常有艺人在这里表
演。一些游手好闲的人在这里溜溜达达。中央有一棵大槐树，盘根错节，据传已
有千年的树龄，是这座城市的历史见证。
```

#### 5.1.2 隐含线索

在描述中暗示可交互对象或隐藏出口：

```
树干底部有一个很大的洞(dong)。
```

这种写法提示玩家可以 `look dong` 查看，甚至 `enter dong` 进入。

#### 5.1.3 方向指引

在长描述中暗示出口方向：

```
你可以看到北边有来自各地的行人来来往往，南边人声鼎沸，一派繁华景象，
东边不时地传来朗朗的读书声，西边则见不到几个行人，一片肃静。
```

### 5.2 环境与天气系统

[[02-守护进程系统]] 中的 NATURE_D（自然守护进程）管理着游戏世界的时间和天气变化。

#### 5.2.1 时间段

游戏中的一天分为多个阶段，定义在 `/adm/etc/nature/day_phase`：

```c
// 来自 natured.c 的示例
string *weather_msg = ({
    "天空中万里无云",
    "几朵淡淡的云彩妆点著清朗的天空",
    "白云在天空中飘来飘去",
    "厚厚的云层堆积在天边",
    "天空中乌云密布",
});
```

#### 5.2.2 户外房间的环境描述

设置了 `outdoors` 属性的房间会在描述后自动添加天气信息：

```c
set("outdoors", "city");
```

NATURE_D 提供 `outdoor_room_description()` 函数返回当前天气描述：

```c
string outdoor_room_description() {
    return "    " + day_phase[current_day_phase]["desc_msg"] + "。\n";
}
```

#### 5.2.3 时间事件

特定时间会触发事件，例如日出时自动存档：

```c
void event_sunrise() {
    object link_ob, *ob;
    int i;
    
    ob = users();
    for (i = 0; i < sizeof(ob); i++)
        if (objectp(link_ob = ob[i]->query_temp("link_ob"))) {
            link_ob->save();
            ob[i]->save();
        }
}
```

### 5.3 动态世界事件

某些房间会根据游戏状态动态生成事件，扬州中央广场是一个典型例子：

```c
// 蒙面人随机袭击玩家事件
void determine_target() {
    object victim, maskman, room, *all, *all1;
    int i, j, k, num, attempt1, attempt2;
    
    // ... 选择目标的逻辑 ...
    
    // 从 locations 数组中随机选择房间
    string *locations = ({
        "/d/shaolin/bamboo1",        // 竹林
        "/d/shaolin/damodong",       // 达摩洞
        "/d/shaolin/beilin3",        // 地下室
        // ... 更多地点 ...
    });
    
    // ... 生成蒙面人并复制目标属性 ...
    copyvictim(maskman, victim);
    maskman->move(environment(victim));
}

private int copyvictim(object me, object victim) {
    // 根据目标等级调整蒙面人属性
    mapping skill_status;
    string *sname;
    int i, max = 0, max1, j, power;
    
    // ... 复制技能和属性 ...
    me->set("combat_exp", victim->query("combat_exp") + random(victim->query("combat_exp") / 10));
    // ... 更多调整 ...
    
    me->set_leader(victim);
    me->set("no_sing", 1);
    return 1;
}
```

这种设计让世界感觉更加生动和危险。

### 5.4 区域叙事风格

不同区域有不同的叙事风格：

| 区域类型 | 叙事特点 | 例子 |

| 城市 | 繁华、热闹、人烟稠密 | 扬州 |
| 门派 | 庄重、森严、等级分明 | 少林、武当 |

| 特殊区域 | 超现实、奇幻 | 地狱、仙界 |

## 六、优缺点分析

### 6.1 优点

#### 6.1.1 直观简单

基于文件系统的世界构建方式非常直观，每个房间一个文件，易于理解和维护。

#### 6.1.2 版本控制友好

每个房间都是独立文件，非常适合使用 Git 等版本控制系统：

- 易于追踪修改历史
- 易于合并多人贡献
- 易于回滚错误修改

#### 6.1.3 高度可扩展

继承体系让特殊房间的实现变得简单：

```c
inherit ROOM;  // 99% 的房间只需要这一行

void create() {
    // 设置属性
    setup();
}
```

#### 6.1.4 分布式开发

多个开发者可以同时在不同区域工作，互不干扰。

### 6.2 缺点

#### 6.2.1 手工编写的局限性

世界规模受限于手工编写的工作量：

- 6000+ 房间已经是巨大的工程
- 难以生成真正大规模的开放世界
- 难以快速迭代世界设计

#### 6.2.2 缺乏可视化编辑工具

没有图形化的房间编辑器，修改世界布局需要：

1. 手动编辑多个房间文件
2. 修改出口连接
3. 重新加载所有相关房间
4. 测试连接是否正确

#### 6.2.3 一致性维护困难

保持数千个房间的风格和质量一致需要大量工作：

- 描述风格不统一
- 房间连接容易出错
- 难以全局调整（如修改所有房间的消耗值）

#### 6.2.4 动态区域支持有限

虽然有 VIRTUAL_D 的设计，但当前实现非常简单，缺乏：

- 程序化生成地形
- 动态区域实例
- 实例副本系统

### 6.3 与现代游戏引擎关卡设计的对比

| 特性 | 《侠客行》MUD | 现代游戏引擎 |
|------|---------------|-------------|
| 编辑器 | 文本编辑器 | 可视化关卡编辑器 |
| 世界表示 | 文件系统目录 | 场景文件 + 资源数据库 |
| 连接方式 | 代码定义的出口 | 可视化的节点连接 |
| 动态内容 | 有限的脚本 | 完整的实体组件系统 |
| 版本控制 | 优秀（纯文本） | 困难（二进制场景文件） |
| 协作 | 容易 | 需要专门的协作工具 |
| 性能 | 极高（纯文本） | 需要优化 |
| 开发速度 | 慢（手工编写） | 快（可视化工具） |

## 八、最佳实践与设计模式

### 8.1 房间创建的标准模板

```c
// Cracked by [作者]
// Room: /[区域]/[房间].c
// [日期]

inherit ROOM;

void create() {
    set("short", "房间短名");
    set("long", @LONG
房间详细描述...
LONG
    );
    
    set("exits", ([
        "方向" : __DIR__"目标房间",
    ]));
    
    set("objects", ([
        __DIR__"npc/对象" : 数量,
    ]));
    
    set("outdoors", "区域名");
    set("cost", 1);
    setup();
}
```

### 8.2 区域入口设计模式

门派区域入口通常遵循以下模式：

```c
int valid_leave(object me, string dir) {
    // 1. 检查性别限制
    if (/* 性别不符合 */)
        return notify_fail("拒绝原因...\n");
    
    // 2. 检查身份限制
    if (/* 不是本派弟子且携带武器 */)
        return notify_fail("放下兵刃...\n");
    
    // 3. 检查善恶值
    if (/* 杀气太重 */)
        return notify_fail("邪魔外道...\n");
    
    return ::valid_leave(me, dir);
}
```

### 8.3 动态出口模式

```c
// 打开时
void do_open() {
    set("exits/enter", __DIR__"目标");
    set("long", "新的描述...");
    set_temp("open", 1);
    remove_call_out("auto_close");
    call_out("auto_close", 延迟时间, this_object());
}

// 自动关闭
void auto_close(object room) {
    if (room->query_temp("open")) {
        message("vision", "关闭消息...\n", room);
        room->set("long", "原描述...");
        room->delete("exits/enter");
        room->delete_temp("open");
    }
}
```

## 九、总结

《侠客行》的世界构建系统是一个成熟、实用的 MUD 世界架构，虽然在现代视角下有一些局限性，但其设计思想仍然值得借鉴：

1. **文件即对象**：每个房间是独立文件，简单直观
2. **继承与组合**：通过继承实现功能扩展
3. **数据驱动**：房间属性通过数据定义，而非硬编码
4. **分布式开发**：文件系统天然支持协作

这种基于文件系统的世界构建方式为 MUD 的发展做出了重要贡献，也为现代游戏的关卡设计提供了历史参考。
