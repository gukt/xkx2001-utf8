# 多人交互系统

## 概述

《侠客行》MUD的多人交互系统是游戏的核心魅力所在，它提供了丰富的玩家间互动方式，包括消息传递、频道聊天、组队协作、经济交易、社交关系、PK战斗等多种交互模式。这些系统相互配合，构建了一个生动的武侠世界。

本文档将详细解析多人交互系统的各个组成部分及其实现原理。

---

## 一、消息分发机制

### 1.1 消息系统架构

消息分发机制是多人交互的基础，主要由 `/feature/message.c` 实现，负责处理不同类型的消息传递。

#### 核心数据结构

```c
// 消息缓冲区
static string *msg_buffer = ({});

// 最大缓冲区大小
#define MAX_MSG_BUFFER 500
```

#### 核心函数

**receive_message()** - 消息接收主函数

```c
void receive_message(string msgclass, string msg)
```

该函数处理所有接收到的消息，支持多种消息分类：

1. **频道消息** (`channel:xxx`) - 按频道过滤
2. **户外消息** (`outdoor`) - 仅在户外显示
3. **天气消息** (`weather`) - 仅在启用天气的房间显示

消息接收流程：
1. 检查是否为交互式用户
2. 解析消息分类，进行过滤
3. 检查消息阻塞状态 `block_msg/all` 或 `block_msg/<type>`
4. 处理字符编码转换 (Big5/GB)
5. 根据用户状态决定是缓冲还是直接显示

**write_prompt()** - 输出缓冲消息

当用户结束输入或编辑状态时，输出之前缓冲的消息。

### 1.2 LPC 消息传递原语

MudOS 提供了多种消息传递函数：

| 函数 | 作用范围 | 用途 |
|------|----------|------|
| `tell_object(ob, msg)` | 单个对象 | 私聊消息 |
| `tell_room(room, msg, exclude...)` | 房间 | 房间广播 |
| `message(type, msg, targets...)` | 指定对象 | 分类消息 |
| `say(msg)` | 房间 | 说话 |
| `write(msg)` | 当前对象 | 输出给自己 |

### 1.3 消息颜色与格式化

颜色处理由 `/include/ansi.h` 定义：

```c
// 前景色
#define RED ESC+"[31m"      // 红色
#define HIR ESC+"[1;31m"    // 高亮红色
#define GRN ESC+"[32m"      // 绿色
#define HIG ESC+"[1;32m"    // 高亮绿色
#define YEL ESC+"[33m"      // 黄色
#define HIY ESC+"[1;33m"    // 高亮黄色
#define BLU ESC+"[34m"      // 蓝色
#define HIB ESC+"[1;34m"    // 高亮蓝色
#define MAG ESC+"[35m"      // 紫色
#define HIM ESC+"[1;35m"    // 高亮紫色
#define CYN ESC+"[36m"      // 青色
#define HIC ESC+"[1;36m"    // 高亮青色
#define WHT ESC+"[37m"      // 白色
#define HIW ESC+"[1;37m"    // 高亮白色

// 重置
#define NOR ESC+"[37;0m"    // 恢复正常
```

常见颜色使用场景：
- `HIR` - 系统警告、PK提示
- `HIY` - 巫师频道、重要提示
- `HIC` - 聊天频道
- `HIM` - 谣言频道
- `GRN` - 私聊消息

### 1.4 消息传递范围

| 范围类型 | 实现方式 | 示例 |
|----------|----------|------|
| 个人 | `tell_object()` | 私聊 (tell) |
| 房间 | `tell_room()` 或 `message()` | 说话 (say) |
| 队伍 | 遍历队伍成员 `team->receive_message()` | 队伍聊天 |
| 频道 | `CHANNEL_D` 筛选监听者 | 聊天频道 |
| 全局 | `users()` 过滤 | 系统公告 |

## 二、频道系统

### 2.1 频道管理架构

频道系统由 `/adm/daemons/channeld.c` (CHANNEL_D) 统一管理，它维护了所有可用频道的配置和状态。

#### 频道配置结构

```c
mapping channels = ([
    "sys": ([          // 系统频道
        "msg_speak": HIR "【系统】%s：%s\n" NOR,
        "wiz_only": 1
    ]),
    "wiz": ([         // 巫师频道
        "msg_speak": HIY "【巫师】%s：%s\n" NOR,
        "msg_emote": HIY "【巫师】%s" NOR,
        "wiz_only": 1
    ]),
    "chat": ([        // 聊天频道
        "msg_speak": HIC "【闲聊】%s：%s\n" NOR,
        "msg_emote": HIC "【闲聊】%s" NOR,
        "omit_address": 1,
        "filter": 1
    ]),
    "rumor": ([       // 谣言频道
        "msg_speak": HIM "【谣言】%s：%s\n" NOR,
        "msg_emote": HIM "【谣言】%s" NOR,
        "anonymous": "某人"
    ]),
    "menpai": ([      // 门派频道
        "msg_speak": HIW "【门派】%s：%s\n" NOR,
        "msg_emote": HIW "【门派】%s" NOR,
        "menpai_only": 1
    ]),
    "gwiz": ([        // 网际巫师频道
        "msg_speak": HIG "【网际巫师】%s：%s\n" NOR,
        "msg_emote": HIG "【网际巫师】%s\n" NOR,
        "wiz_only": 1,
        "intermud": GWIZ,
        "channel": "CREATOR",
        "filter": 1,
        "omit_address": 0,
        "intermud_emote": 1
    ]),
    "gchat": ([       // 泥潭闲聊（网际）
        "msg_speak": YEL "【泥潭闲聊】%s：%s\n" NOR,
        "msg_emote": YEL "【泥潭闲聊】%s" NOR,
        "intermud": GCHANNEL,
        "channel": "gchat",
        "intermud_emote": 1,
        "omit_address": 1,
        "filter": 1
    ]),
]);
```

#### 频道属性说明

| 属性 | 类型 | 说明 |
|------|------|------|
| `msg_speak` | string | 说话消息格式 |
| `msg_emote` | string | 表情消息格式 |
| `wiz_only` | int | 是否仅巫师可用 |
| `menpai_only` | int | 是否仅同门可用 |
| `anonymous` | string | 匿名显示名称 |
| `intermud` | object | 网际频道对象 |
| `channel` | string | 网际频道名称 |
| `filter` | int | 是否启用过滤 |
| `omit_address` | int | 是否省略地址 |
| `extra_listener` | object[] | 额外监听器 |

### 2.2 频道消息流程

#### 主函数 `do_channel()`

```c
varargs int do_channel(object me, string verb, string arg, int emote)
```

完整流程：

1. **检查频道表情** 
   - 检测动词后缀 `*` 表示频道表情
   - 例如：`chat* smile`

2. **验证频道有效性**
   - 检查频道是否存在于配置中

3. **处理参数和状态**
   - 检查全局频道阻塞 `chblk_on`
   - 谣言频道消耗精力 50 点
   - 检查单个频道阻塞 `chblk_<channel>`
   - 验证门派权限

4. **防刷屏机制**
   - 检查重复消息 `last_channel_msg`
   - 记录消息计数 `channel_msg_cnt`

5. **自动订阅**
   - 发言用户自动订阅该频道

6. **表情消息处理**
   - 调用 `EMOTE_D->do_emote()` 生成表情文本
   - 支持网际表情 `intermud_emote`

7. **发送消息**
   - 匿名处理
   - 调用 `filter_listener()` 筛选接收者
   - 发送消息给所有符合条件的用户

8. **额外监听器和网际转发**
   - 转发给 `extra_listener`
   - 网际频道通过相应 daemon 转发

#### 频道监听器筛选

```c
int filter_listener(object ppl, mapping ch)
```

筛选规则：
- 跳过未登录用户 (无 environment)
- 巫师频道仅巫师可见
- 门派频道仅同门可见 (包括巫师)
- 其他频道所有人可见

### 2.3 频道订阅管理

玩家的频道订阅状态存储在 `user->channels` 中，是一个字符串数组。

当用户在某频道发言时，会自动添加订阅：
```c
tuned_ch = me->query("channels");
if (!pointerp(tuned_ch))
    me->set("channels", ({ verb }));
else if (member_array(verb, tuned_ch) == -1)
    me->set("channels", tuned_ch + ({ verb }));
```

### 2.4 频道阻塞机制

提供了多级别的频道阻塞：

| 级别 | 设置方式 | 效果 |
|------|----------|------|
| 全局阻塞 | `set("chblk_on", 1)` | 阻塞所有普通频道 |
| 单频道阻塞 | `set("chblk_chat", 1)` | 阻塞指定频道 |

阻塞检查在 `do_channel()` 开始时执行。

### 2.5 网际频道 (Intermud)

系统支持跨 MUD 的频道通信，主要包括：

- **gwiz** - 网际巫师频道
- **gchat** - 泥潭闲聊频道

网际消息流程：
1. 本地发送到 `intermud` 对象 (GWIZ 或 GCHANNEL)
2. 通过网络协议转发到其他 MUD
3. 远程 MUD 接收并显示

## 三、组队系统

### 3.1 组队功能架构

组队系统由两部分组成：
- `/feature/team.c` - 组队功能模块（基础功能）
- `/cmds/std/team.c` - 组队命令实现（用户接口）

### 3.2 组队数据结构

```c
static object *team;           // 队伍成员列表
static object leader;          // 跟随对象
static object lord;            // 主人对象
```

#### 队伍属性说明

- `team` - 成员数组，`team[0]` 为队长
- `leader` - 跟随对象，用于 `follow` 命令
- `lord` - 主人对象，用于NPC

### 3.3 核心组队函数

#### 队伍管理

| 函数 | 功能 |
|------|------|
| `add_team_member(ob)` | 添加成员 |
| `join_team(ob)` | 加入他人队伍 |
| `set_team(team_array)` | 设置队伍 |
| `query_team()` | 获取队伍 |
| `dismiss_team(ob)` | 离开/解散队伍 |
| `is_team_leader()` | 是否为队长 |
| `have_team_member(ob)` | 是否在队伍中 |

#### 跟随系统

| 函数 | 功能 |
|------|------|
| `set_leader(ob)` | 设置跟随对象 |
| `query_leader()` | 获取跟随对象 |
| `follow_path(dir)` | 沿路径跟随 |
| `follow_me(ob, dir)` | 对象移动时触发跟随 |

### 3.4 组队命令详解

#### 基本命令

| 命令 | 格式 | 功能 |
|------|------|------|
| `team` | `team` | 查看队伍成员 |
| `team invite` | `team invite <玩家>` | 邀请加入 |
| `team join` | `team join <玩家>` | 加入队伍 |
| `team dismiss` | `team dismiss [玩家]` | 离开/踢出/解散 |
| `team talk` | `team talk <消息>` | 队伍聊天 |
| `team fight` | `team fight <目标>` | 集体攻击 |
| `team form` | `team form <阵法>` | 组队阵法 |

#### 邀请流程

```
1. 队长: team invite 玩家A
   → 显示邀请信息给A
   → 设置队长的 pending/team 为A

2. 玩家A: team join 队长
   → 验证 pending/team 状态
   → 调用 add_team_member()
   → 广播队伍消息
```

#### 队伍聊天

```c
// 普通聊天
team talk <消息>
// 或简写
tt <消息>

// 表情聊天
team talk* <表情>
```

### 3.5 队伍战斗协作

#### 集体攻击

队长可以使用 `team fight` 命令发起集体攻击：

```c
if (sscanf(arg, "fight %s", arg) == 1) {
    if (!pointerp(t = me->query_team()) || !me->is_team_leader())
        return notify_fail("只有队伍领袖可以发起集体战斗。\n");

    for (int i = 0; i < sizeof(t); i++)
        if (living(t[i]))
            KILL_CMD->main(t[i], arg);
    return 1;
}
```

#### 战斗中的协作

- 队员受伤队长可见
- 可通过 `team talk` 即时沟通战术
- 阵法系统提供战斗加成（见 [[06-武功与战斗系统]]）

### 3.6 跟随系统实现

跟随功能由 `/cmds/std/follow.c` 和 team.c 配合实现：

```c
int follow_me(object ob, string dir)
{
    if (!living(this_object()) || ob == this_object()) return 0;
    if (ob == leader || (query("pursuer") && this_object()->is_killing(ob->query("id")))) {
        if (random(ob->query_skill("move")) > this_object()->query_skill("move")) {
            remove_call_out("follow_path");
            call_out("follow_path", 1, dir);
            return 1;
        }
        return follow_path(dir);
    }
}
```

跟随有延迟机制，取决于双方的轻功技能等级。

### 3.7 队伍解散规则

| 场景 | 结果 |
|------|------|
| 队长 `team dismiss` | 全队解散 |
| 队员 `team dismiss` | 仅该队员离开 |
| 队长踢出队员 | 被踢队员离开 |
| 队长离线/死亡 | 次资深队员接任队长 |

## 四、交易与经济交互

### 4.1 货币体系

货币系统由 `/adm/daemons/moneyd.c` (MONEY_D) 管理，实现了三种货币单位：

| 货币 | 单位 | 兑换率 | 对象文件 |
|------|------|--------|----------|
| 黄金 | 两 | 10000文 | `/clone/money/gold.c` |
| 白银 | 两 | 100文 | `/clone/money/silver.c` |
| 铜钱 | 文 | 1文 | `/clone/money/coin.c` |

#### 货币转换

```c
// 货币显示格式
string money_str(int amount)
{
    // 输入单位为文
    // 输出格式如：三两黄金五两白银七十文
}

// 价格显示格式
string price_str(int amount)
{
    // 类似 money_str，但用 "又" 连接
}
```

#### 现金流控制

系统设置了最高现金流限制，防止通货膨胀：

```c
#define MAX_CASHFLOW_ALLOWED 400000  // 单位：银
```

相关函数：
- `query_avalible_xkx_cashflow()` - 查询可用现金流
- `query_total_xkx_cashflow()` - 计算当前总现金流

### 4.2 玩家间交易

#### 给予物品 - `/cmds/std/give.c`

```c
// 命令格式
give <物品> to <玩家>
give <玩家> <物品>
```

交易流程：
1. 验证物品存在和可交易性
2. 检查接收方状态
3. 移动物品所有权
4. 双方显示交易信息
5. 保存玩家数据

#### 丢弃与拾取

| 命令 | 文件 | 功能 |
|------|------|------|
| `drop` | `/cmds/std/drop.c` | 丢弃物品 |
| `get` | `/cmds/std/get.c` | 拾取物品 |

这两个命令也构成了一种间接的物品转移方式。

### 4.3 商店交易系统

商店系统由 `/feature/dealer.c` 实现，提供买卖功能。

#### 商店功能

| 命令 | 功能 |
|------|------|
| `list [类别]` | 列出商品 |
| `buy <物品> [数量]` | 购买物品 |
| `sell <物品>` | 出售物品 |
| `value <物品>` | 估价 |

商品类别：
- `weapon` - 武器
- `armor` - 防具
- `drug` - 药品
- `book` - 书籍
- `misc` - 杂货

#### 价格计算

```c
// 购买价格 = 原价 * 1.2 (val_factor = 12)
// 出售价格 = 原价 * 0.7 (70% 收购)

value * 70 / 100  // 收购价
value * 12 / 10   // 售价
```

#### 交易流程

**购买流程：**
1. 验证商品存在和库存
2. 检查玩家携带金额
3. 调用 `MONEY_D->player_pay()` 扣款
4. 生成/转移物品给玩家
5. 扣减库存

**出售流程：**
1. 验证物品可出售性
2. 检查商店现金流 `player_dealer_pay()`
3. 调用 `MONEY_D->pay_player()` 付款
4. 物品转移到商店

### 4.4 银行系统

银行功能由 `/inherit/room/bank.c` 实现：

```c
int do_convert(string arg)

    // 命令格式: convert <数量> <货币> to <货币>
    // 例如: convert 10 gold to silver

```

## 五、社交系统

### 5.1 表情动作系统

表情系统由 `/adm/daemons/emoted.c` (EMOTE_D) 提供支持。

#### 表情数据结构

每个表情定义包含三个消息模板：

```c
emote["smile"] = ([
    "myself": "$N微微一笑。",
    "target": "$N对你微微一笑。",
    "others": "$N对$n微微一笑。",
    "myself_target": "...",
    "others_target": "...",
]);
```

#### 变量替换

表情消息支持多种变量：

| 变量 | 说明 |
|------|------|
| `$N` | 主动者名称 |
| `$n` | 目标名称 |
| `$P` | 主动者代词 |
| `$p` | 目标代词 |
| `$C` | 对目标的亲密称呼 |
| `$c` | 目标的亲密称呼 |
| `$R` | 对目标的尊称 |
| `$r` | 对目标的蔑称 |
| `$S` | 自称 |
| `$s` | 谦称 |

#### 表情命令

| 命令 | 文件 | 功能 |
|------|------|------|
| `semote` | 内置 | 系统预设表情 |
| `emote` | `/cmds/std/emote.c` | 自定义表情 |

**自定义表情示例：**
```
emote 抱了抱拳。
→ 你抱了抱拳。
→ ->玩家A抱了抱拳。 (其他人看到)
```

### 5.2 婚姻系统

婚姻系统由 `/adm/daemons/marryd.c` (MARRY_D) 管理。

#### 婚姻数据结构

玩家的配偶信息存储在 `user->spouse` 映射中：

```c
spouse = ([
    "id": "player_id",
    "name": "玩家姓名",
    "title": "丈夫/妻子",
]);
```

附加数据：
- `married_times` - 结婚次数
- `last_spouse_id` - 前配偶ID
- 临时 `ring_worn` - 订婚戒指标记

#### 婚姻函数

| 函数 | 功能 |
|------|------|
| `setup_marriage(p1, p2, ring1, ring2)` | 结婚 |
| `break_marriage(breaker)` | 离婚 |
| `validate_marriage(me)` | 验证婚姻 |

#### 结婚流程

虽然 MARRY_D 提供了核心功能，但实际婚礼流程需要特定NPC实现，通常包括：
1. 双方订婚（交换戒指）
2. 举办婚礼（特定房间）
3. 证婚人见证
4. 调用 `setup_marriage()`
5. 系统公告

#### 离婚情形

- 主动离婚：`break_marriage()`
- 配偶死亡/删号：自动解除
- 登录验证失败：`validate_marriage()` 检测异常

### 5.3 公告板系统

公告板由 `/inherit/misc/bboard.c` 实现，是异步社交的重要方式。

#### 公告板数据结构

```c
// 留言存储
mapping *notes;  // 每条留言是一个 mapping

// 单条留言结构
note = ([
    "title": "标题",
    "author": "作者",
    "msg": "内容",
```

## 六、玩家间冲突与 PK 系统

### 6.1 战斗状态管理

战斗系统由 `/feature/attack.c` 提供基础功能。

#### 战斗数据结构

```c
static object *enemy = ({});    // 战斗对象列表
static string *killer = ({});   // 必杀对象ID列表
```

#### 核心战斗函数

| 函数 | 功能 |
|------|------|
| `fight_ob(ob)` | 开始战斗 |
| `kill_ob(ob)` | 开始攻击 (含杀意) |
| `is_fighting([ob])` | 是否在战斗 |
| `is_killing([id])` | 是否要必杀某人 |
| `remove_enemy(ob)` | 移除战斗对象 |
| `remove_killer(ob)` | 移除杀意 |

| `remove_all_killer()` | 清除所有杀意 |
| `select_opponent()` | 选择战斗目标 |
| `attack()` | 执行攻击动作 |

#### kill vs fight 的区别

| 特性 | fight | kill |
|------|-------|------|
| 杀意 | 无 | 有 (killer列表) |
| 伤害 | 较轻 | 可致命 |
| 对方反应 | 可选 fight_back | NPC 会 kill_back |
| PK标记 | 无 | 有 |

### 6.2 PK 机制详解

PK命令由 `/cmds/std/kill.c` 实现。

#### PK 限制规则

系统有多种 PK 限制保护新手：

```c
// 1. 禁止战斗区域
if (environment(me)->query("no_fight"))
    return notify_fail("这里不准战斗。\n");

// 2. 巫师不能杀普通玩家
if (SECURITY_D->get_status(me) == "(immortal)" &&
    SECURITY_D->get_status(obj) != "(immortal)")
    return notify_fail("你不能杀这个人！\n");

// 3. 刚刚被劝说的对象
if (me->query_temp("last_persuader") == obj->query("id"))
    return notify_fail("你不能杀这个人！\n");

// 4. 投降保护
if (me->query("id") == obj->query_temp("surrender/ownder"))
    return notify_fail("他/她已经投降了,你现在不能杀！\n");
if (me->query_temp("surrender/ownder") != 0)
    return notify_fail("你刚投降过别人，别老想着干坏事！\n");

// 5. PKer 状态冷却
if (userp(obj) && userp(me) &&
    (me->query_condition("pker") > 240 || obj->query("mud_age") < 18000))
    return notify_fail("你感到一丝内疚，手突然软了下来！\n");
```

#### PK 流程

```
1. 攻击者: kill 目标
   → 检查各种限制
   → 显示挑战信息
   → 调用 kill_ob()

2. 目标 (NPC):
   → kill_ob(攻击者) 反击

3. 目标 (玩家):
   → fight_ob(攻击者) (仅战斗，无杀意)
   → 提示: 若要性命相搏，请也用 kill
```

### 6.3 伤害与死亡系统

伤害系统由 `/feature/damage.c` 实现。

#### 伤害函数

| 函数 | 功能 |
|------|------|
| `receive_damage(type, amt, who)` | 受伤害 |
| `receive_wound(type, amt, who)` | 受创伤 |
| `receive_heal(type, amt)` | 恢复 |
| `receive_curing(type, amt)` | 治疗创伤 |

伤害类型：
- `qi` - 气血
- `jing` - 精神
- `jingli` - 精力

#### 死亡流程

`die()` - 死亡主函数

死亡过程：
1. 检查 `no_death` 房间
2. `COMBAT_D->announce()` 公告死亡
3. `death_penalty()` 执行死亡惩罚
4. 记录凶手 `my_killer`
5. `killer_reward()` 凶手奖励
6. 记录死亡日志 `PKILL_DATA`, `PLAYER_DEATH`
7. 生成尸体 `CHAR_D->make_corpse()`
8. 清理所有敌人
9. 玩家处理：
   - 保存数据
   - 变为鬼魂 (ghost = 1)
   - 移动到死亡房间 `DEATH_ROOM`
   - 解除婚姻
   - 可能解除师徒关系
10. NPC处理：销毁对象

#### 昏迷与复活

```c
void unconcious()  // 昏迷
void revive()      // 苏醒
```

昏迷在气血/精神降到 0 时触发，一段时间后自动苏醒。

### 6.4 PK 日志与监控

系统详细记录所有 PK 事件：

```c
// PKILL_DATA - 专门的 PK 日志
log_file("PKILL_DATA", sprintf("%s(%s) 被 %s 杀死了(PlayerKill) on %s。\n",
    me->query("name"), getuid(me),
    query_temp("last_eff_damage_from"), ctime(time())));

// PLAYER_DEATH - 所有死亡记录
log_file("PLAYER_DEATH", ...);
```

追踪数据：
- `last_damage_from` - 最后伤害来源
- `last_eff_damage_from` - 最后有效伤害来源（PK记录用）
- `last_fainted_from` - 最后打昏自己的人
- `my_killer` - 凶手ID

## 七、多部件系统 - `/feature/multi.c`

虽然名为 "multi"，这个模块实际用于实现由多个部件组成的复杂对象（如马车、机关等），但它也是实现某些多人交互场景的基础。

### 7.1 部件管理架构

#### 部件树结构

```
主对象
  ├── 部件1 (parent = 主对象)
  │     └── 子部件1a (parent = 部件1)
  ├── 部件2 (parent = 主对象)
  └── ...
```

#### 核心数据

```c
// 父对象中定义
set("components", ([
    "部件id": "部件文件",
    ...
]));

set("children", ([ ... ]));  // 加载后的部件对象
```

### 7.2 部件加载

```c
void load_parts()
{
    // 1. 读取 components 映射
    // 2. 调用 make_part() 创建每个部件
    // 3. 设置部件的 parent 和 id
    // 4. 部件 move 到父对象
    // 5. 保存到 children 映射
}

object make_part(string file, string id)
{
    object ob = new(file);
    ob->set("parent", this_object());
    ob->set_name(ob->query("name"), ({id}));
    ob->m_move(this_object());
    return ob;
}
```

### 7.3 批量操作函数

multi.c 提供了一系列 `m_*` 函数，用于对指定部件进行操作：

| 函数 | 对应基础函数 | 功能 |
|------|------------|------|
| `m_set(prop, val, id)` | `set()` | 设置属性 |
| `m_query(prop, id)` | `query()` | 查询属性 |
| `m_delete(prop, id)` | `delete()` | 删除属性 |
| `m_add(prop, val, id)` | `add()` | 增加值 |
| `m_set_temp(prop, val, id)` | `set_temp()` | 设置临时属性 |
| `m_query_temp(prop, id)` | `query_temp()` | 查询临时属性 |
| `m_delete_temp(prop, id)` | `delete_temp()` | 删除临时属性 |
| `m_add_temp(prop, val, id)` | `add_temp()` | 增加临时值 |

### 7.4 亲属查找函数

| 函数 | 功能 |
|------|------|
| `is_ancestor(ob, id)` | 是否为 ob 的祖先 |
| `find_offsprings(id)` | 查找所有后代 |
| `is_parent(ob)` | 是否为 ob 的直接父对象 |
| `find_owner()` | 查找根对象 |
| `find_relatives(id)` | 查找所有同 id 的亲属 |
| `all_offsprings()` | 所有后代 |
| `all_relatives()` | 所有亲属（包括自己） |
| `all_children()` | 所有直接子对象 |

### 7.5 部件移动

```c
varargs int m_move(mixed dest, int order)
{
    // dest 可以是对象或部件id
    // 如果是部件id，通过 find_relatives() 查找
    // 检查不能把自己移入后代中
}
```

### 7.6 动作代理系统

multi.c 提供了动作代理机制，允许父对象给子对象添加命令：

```c
// 1. 在父对象中添加动作到子对象
m_add_action("部件id", "回调函数", "命令动词", flag);

// 2. 子对象 init() 时设置
m_setup_actions();  // 添加所有代理命令

// 3. 玩家输入命令时
m_parse_action();   // 转发给父对象的回调函数
```

### 7.7 部件装卸

```c
varargs int m_remove(string id, int amount);  // 卸下部件
varargs int m_load(string owner, string id, int order);  // 装上部件
```

这两个函数提供了可视化的部件装卸消息。

## 八、优缺点分析

### 8.1 文字交互的优势

1. **沉浸感强**
   - 丰富的表情系统让文字充满情感
   - ANSI颜色突出重要信息
   - 详细的描述激发想象力

2. **社交系统简洁高效**
   - 频道系统实现大规模群体交流
   - 组队系统紧密协作
   - 公告板实现异步沟通

3. **架构设计清晰**
   - 消息分类机制灵活
   - Daemon 模式统一管理
   - Feature 模块可复用

4. **PK 系统平衡**
   - 多重保护机制防止恶意PK
   - 详细的日志记录便于追溯
   - 死亡惩罚合理适度

### 8.2 局限性

1. **缺乏图形界面**
   - 无法直观展示空间位置关系
   - 队伍位置不明确
   - 交易物品需要手动描述

2. **实时协作有限**
   - 没有共享地图/视野
   - 战术配合需要打字沟通
   - 缺乏实时状态同步界面

3. **经济系统较为简单**
   - 货币主要回收途径是商店
   - 缺乏玩家间的拍卖系统
   - 没有银行储蓄利息

4. **社交深度有限**
   - 婚姻系统较为基础
   - 缺乏结义、帮派等更深层社交组织
   - 好友列表功能缺失（早期版本）

## 十、总结

《侠客行》的多人交互系统虽然基于文字界面，但设计精妙、功能完整，提供了丰富的玩家互动方式。其核心优势在于：

1. **统一的消息架构** - 灵活的分类和过滤机制
2. **完善的频道管理** - 支持多种聊天场景和跨服通信
3. **实用的组队系统** - 跟随、聊天、战斗、阵法四位一体
4. **平衡的 PK 机制** - 既有保护新手的设计，又不阻碍江湖恩怨
5. **丰富的社交表达** - 表情动作、婚姻、公告板等多维度社交

这些系统相互配合，让玩家在文字世界中能够体验到真实的江湖互动，这正是 MUD 游戏经久不衰的魅力所在。
