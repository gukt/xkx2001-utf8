## 概述

守护进程（Daemon）是侠客行 MUD 的核心服务层，以单例对象形式存在，为整个游戏世界提供各种基础设施服务。与普通对象不同，Daemon 一旦加载就常驻内存，负责协调各个游戏系统之间的交互。

所有 Daemon 定义在 `/adm/daemons/` 目录下，并在 `include/globals.h` 中以宏定义形式提供引用，方便其他代码调用。

## Daemon 架构模式

### 为什么使用 Daemon 模式？

在 MUD 开发中，Daemon 模式是一种经典的设计选择，与直接函数调用相比具有以下优势：

1. **状态集中管理**：游戏中很多状态需要全局维护（如时间、天气、在线玩家列表等），Daemon 提供了天然的单例容器

2. **降低耦合度**：各系统通过 Daemon 提供的标准接口交互，而不是直接依赖具体实现

3. **便于调试和监控**：Daemon 作为独立对象，可以方便地添加日志、统计和调试信息

4. **资源共享**：避免重复加载相同数据或代码，节省内存

### Daemon 的加载机制

Daemon 的加载有两种主要方式：

1. **预加载**：在 `master.c` 的 `epilog()` 函数中，通过读取 `/adm/etc/preload` 文件列表来预加载指定的 Daemon

```c
// master.c 中的 epilog() 函数
string *epilog(int load_empty)
{
    string *items;
    items = update_file(CONFIG_DIR + "preload");
    return items;
}
```

2. **按需加载**：首次调用时由驱动程序自动加载。当代码执行类似 `LOGIN_D->logon(ob)` 时，若 Daemon 尚未加载，驱动会自动编译并加载 `/adm/daemons/logind.c`

### Daemon 的生命周期

Daemon 对象的生命周期具有以下特点：

- **常驻内存**：一旦加载，除非显式 destruct，否则一直存在
- **全局唯一**：每个 Daemon 只有一个实例（单例模式）
- **不保存**：大多数 Daemon 不使用 `save_object()` 保存状态，启动时重新初始化
- **权限**：通常 `seteuid(getuid())`，以 daemon 的自身权限运行

## 核心 Daemon 详解

### LOGIN_D - 登录守护进程

**文件**：`/adm/daemons/logind.c`

**职责**：
- 处理玩家连接和认证
- 新玩家角色创建流程
- 处理断线重连
- 登录后初始化工作

#### 登录状态机

玩家从连接到进入游戏经历以下状态：

```
    connect()
        ↓
    logon() - 欢迎界面、编码选择
        ↓
    get_id() - 输入用户 ID
        ↓
    ┌─────────┬───────────────┐
    ↓         ↓               ↓
  老用户   新用户       被封禁
    ↓         ↓               ↓
get_passwd confirm_id     destruct
    ↓         ↓
make_body   get_name
    ↓         ↓
enter_world new_password
              ↓
         confirm_password
              ↓
         random_gift
              ↓
         get_email
              ↓
         get_gender
              ↓
         init_new_player
              ↓
         make_body
              ↓
         enter_world
```

#### 核心 API

```c
// 主登录入口
void logon(object ob)

// 创建玩家身体
object make_body(object ob)

// 进入游戏世界
varargs void enter_world(object ob, object user, int silent)

// 重新连接
varargs void reconnect(object ob, object user, int silent)

// 验证 ID 合法性
int check_legal_id(string id, object ob)

// 验证中文名合法性
string check_legal_name(string name, object ob)

// 查找是否有人在创建同名角色
object find_login(object me)

// 查找已存在的玩家身体
object find_body(string name)
```

#### 重要数据结构

```c
// 禁止的用户名列表
string *banned_name = ({
    "你", "你", "我", "他", "她", "它", "它",
    "韦小宝", "某人", "您", "谣言", "蒙面人",
    "金庸", 
});

// 最大用户数限制
#define MAX_USERS 230
```

#### 登录流程详解

1. **连接处理**：`master.c` 的 `connect()` 创建 login 对象，调用 `LOGIN_D->logon(ob)`

2. **编码选择**：让玩家选择 GB 或 BIG5 编码

3. **ID 验证**：
   - 长度检查（3-8个字符）
   - 只允许小写字母
   - 检查是否被封禁
   - 检查是否超过人数上限
   - 检查巫师等级限制

4. **密码验证**：使用 `crypt()` 加密验证

5. **角色创建流程**：
   - 中文名验证（1-4个中文字）
   - 密码设置
   - 随机天赋分配
   - Email 输入
   - 性别选择

6. **天赋系统**：

```c
private void random_gift(mapping my)
{
    // 膂力、悟性、根骨、身法各 10 点基础，然后分配 50 个随机点
    // 耐力 = 100 - 其他四项之和
    // 各项最大值不超过 30
}
```

7. **进入游戏**：
   - 创建 user 对象
   - 恢复玩家数据
   - 检查并发放婚姻戒指
   - 登录数据检查（UPDATE_D）
   - 检查豹胎易筋丸效果
   - 根据门派给与服装
   - 移动到起始房间
   - 发送登录系统消息

### COMBAT_D - 战斗守护进程

**文件**：`/adm/daemons/combatd.c`

**职责**：
- 战斗核心仲裁
- 伤害计算与消息生成
- 战斗奖励与惩罚
- NPC 自动战斗触发

#### 战斗流程

战斗由 living 对象的 `heart_beat()` 驱动，每回合调用 `COMBAT_D->fight()`：

```
heart_beat()
    ↓
is_fighting() ?
    ↓
fight(me, victim)
    ↓
┌─────────────────────────────┐
│ 对方忙？→ yes → 快速攻击   │
│               ↓            │
│          do_attack()       │
│               ↓            │
│ 随机判断攻击/防御/观望     │
└─────────────────────────────┘
```

#### 核心 API

```c
// 主战斗循环
void fight(object me, object victim)

// 执行一次攻击
varargs int do_attack(object me, object victim, object weapon, int attack_type)

// 生成伤害描述消息
string damage_msg(int damage, string type)

// 生成状态描述消息
string eff_status_msg(int ratio)  // 有效状态
string status_msg(int ratio)      // 一般状态

// 报告战斗状态
varargs void report_status(object ob, int effective)

// 计算技能威力
varargs int skill_power(object ob, string skill, int usage)

// 自动战斗发起
void auto_fight(object me, object obj, string type)
void start_berserk(object me, object obj)
void start_hatred(object me, object obj)
void start_vendetta(object me, object obj)
void start_aggressive(object me, object obj)

// 战斗事件宣告
void announce(object ob, string event)

// 胜者奖励
void winner_reward(object killer, object victim)

// 死亡惩罚
void death_penalty(object victim)

// 凶手奖励
void killer_reward(object killer, object victim)
```

#### 攻击判定流程

```c
do_attack()
    ↓
选择攻击技能
    ↓
计算攻击方 AP (skill_power)
    ↓
计算防御方 DP (dodge)
    ↓
随机判定：random(AP + DP) < DP ?
    ↓
    ┌─── 闪避成功？
    ↓ yes
    └→ dodge_msg → 战斗结束
    ↓ no
计算格挡 PP (parry)
    ↓
random(AP + PP) < PP ?
    ↓
    ┌─── 格挡成功？
    ↓ yes
    └→ parry_msg → 战斗结束
    ↓ no
计算伤害
    ↓
应用伤害
    ↓
显示伤害消息
    ↓
report_status()
```

#### 伤害计算

伤害计算涉及多个因素：

1. **基础伤害**：武器伤害 + 临时伤害加值

2. **技能加成**：
   - 空手/武器技能等级
   - 内功等级加力

3. **属性加成**：
   - 膂力影响基础伤害
   - 精力影响加力效果

4. **战斗经验修正**：
   ```c
   defense_factor = victim->query("combat_exp");
   while(random(defense_factor) > me->query("combat_exp")) {
       damage -= damage / 3;
       defense_factor /= 2;
   }
   ```

5. **特殊装备/技能效果**

#### 伤害类型

| 类型 | 描述 |
|-----|------|
| "擦伤" | 轻微受伤 |
| "割伤" | 利器造成的伤口 |
| "刺伤" | 穿刺伤害 |
| "跌伤" | 摔倒、撞击 |
| "鞭伤" | 软兵器伤害 |
| "咬伤" | 野兽攻击 |
| "瘀伤" | 钝器打击 |
| "内伤" | 内家功夫伤害 |

#### 状态显示

根据有效气血比例显示不同状态：

| 气血比例 | 状态 |
|---------|------|
| 100% | 气血充盈，毫无损伤 |
| 95-99% | 似乎受了点轻伤 |
| 90-94% | 可能受了点轻伤 |
| 80-89% | 受了几处伤，不碍事 |
| 60-79% | 受伤不轻，状态不佳 |
| 40-59% | 气息粗重，动作散乱，伤势不轻 |
| 30-39% | 伤痕累累，勉力支撑 |
| 20-29% | 重伤，有生命危险 |
| 10-19% | 奄奄一息，命在旦夕 |
| <10% | 如同风中残烛 |

#### 死亡惩罚

```c
void death_penalty(object victim)
{
    // 增加死亡次数
    victim->add("death_times", 1);
    
    // 减少正气/戾气
    victim->add("shen", -shen / 20);
    
    // 减少行为经验
    victim->add("behavior_exp", -behavior_exp / 20);
    
    // 减少实战经验 (1%，最多 5000)
    victim->add("combat_exp", -amount);
    
    // 减少潜能
    victim->add("potential", -potential / 2);
    
    // 减少存款
    victim->add("balance", -amount / 2);
    
    // 清除 vendetta
    victim->delete("vendetta");
}
```

### COMMAND_D - 命令守护进程

**文件**：`/adm/daemons/commandd.c`

**职责**：
- 命令路径搜索
- 命令缓存管理
- 命令有效性验证

#### 核心 API

```c
// 查找命令
string find_command(string verb, string *path)

// 重新建立命令缓存
void rehash(string dir)
```

#### 命令查找机制

命令搜索按照给定的路径数组依次查找：

1. 检查该目录是否已缓存
2. 未缓存则调用 `rehash()` 扫描目录
3. 检查动词是否在该目录下存在
4. 验证命令权限（`SECURITY_D->valid_cmd()`）
5. 返回完整命令路径或 0

命令路径通常由用户的权限级别决定，如：
- `/cmds/std` - 普通玩家命令
- `/cmds/usr` - 注册用户命令
- `/cmds/imm` - immortal 命令
- `/cmds/wiz` - wizard 命令
- `/cmds/arch` - arch 命令
- `/cmds/adm` - admin 命令

### NATURE_D - 自然环境守护进程

**文件**：`/adm/daemons/natured.c`

**职责**：
- 管理游戏时间流逝
- 昼夜交替
- 触发定时事件
- 自动保存玩家数据

#### 时间系统

游戏时间与真实时间的对应关系：
- 真实 1 秒 = 游戏 1 分钟

这意味着：
- 游戏 1 小时 = 真实 1 分钟
- 游戏 1 天 = 真实 24 分钟
- 游戏 1 年 = 真实 6 天

#### 昼夜阶段配置

时间配置从 `/adm/etc/nature/day_phase` 读取：

```c
// 结构示例
mapping *day_phase = ({
    ([ "length": ..., "time_msg": ..., "desc_msg": ..., "event_fun": ... ]),
    // ...
});
```

#### 核心 API

```c
// 初始化昼夜阶段
void init_day_phase()

// 更新昼夜阶段
void update_day_phase()

// 日出事件（自动保存）
void event_sunrise()

// 通用事件检查
void event_common()

// 获取户外描述
string outdoor_room_description()

// 获取游戏时间
string game_time()

// 读取配置表
mapping *read_table(string file)

// 查询当前阶段
mapping *query_day_phase()
```

#### 定时事件

1. **日出事件** (`event_sunrise`)：
   - 自动保存所有在线玩家数据
   - 触发新一天的开始

2. **通用事件** (`event_common`)：
   - 检查并清理无环境的对象
   - 随机检查玩家背包（扒手、丢失物品）

#### 扒手机制

```c
void event_common()
{
    // 对经验 < 10000 的玩家，携带 > 50 两黄金可能被偷
    // 携带 unique 物品可能被偷
    // 背包物品 > 30 件可能丢失
}
```

#### 数字转换规则

- 0-10：直接映射（零、一、二...十）
- 11-19：十 + 个位（十一、十二...十九）
- 20-99：十位 + 十 + 个位（二十一、三十五...九十九）
- 100-999：百位 + 百 + 十位 + 个位（一百零一、三百五十...九百九十九）
- 1000-9999：千位 + 千 + ...
- 万以上：万、亿、兆等单位

#### 干支纪年

```c
string chinese_date(int date)
{
    // 天干：甲乙丙丁戊己庚辛壬癸
    // 地支：子丑寅卯辰巳午未申酉戌亥
    // 年：天干[年%10] + 地支[年%12]
    // 月：中文数字
    // 日：中文数字
    // 时：地支[(时+1)%24/2]
    // 刻：中文数字
}
```

#### 权限检查机制

1. **写权限检查流程**：
   - 检查是否是 Root
   - 检查是否是用户自己的目录
   - 检查是否被排除
   - 检查是否被信任

2. **读权限检查**：
   - 总是允许 file_size / stat
   - 允许读取 /data、/log、/adm/etc
   - Root 允许所有
   - 检查用户自己的目录
   - 检查排除/信任列表

3. **命令权限检查**：
   - Root 允许所有
   - 检查排除列表
   - 检查信任列表
   - /cmds/std /cmds/usr /cmds/skill 默认允许

## Daemon 通信机制

### call_other 机制

Daemon 之间以及 Daemon 与其他对象之间的通信主要通过 LPC 的 `call_other()` 机制：

```c
// 直接调用方式
result = LOGIN_D->make_body(ob);

// 等同于
result = call_other(LOGIN_D, "make_body", ob);
```

### 事件驱动 vs 轮询

侠客行的 Daemon 采用事件驱动为主，轮询为辅的模式：

1. **事件驱动**：
   - 玩家输入触发命令 → COMMAND_D
   - 战斗动作 → COMBAT_D
   - 频道聊天 → CHANNEL_D

2. **轮询**：
   - NATURE_D 使用 `call_out()` 定时更新时间
   - heart_beat() 中的战斗循环
   - 定期检查和清理任务

### 发布/订阅模式

虽然没有完整的发布/订阅系统，但 CHANNEL_D 提供了类似机制：

```c
// 注册频道中继
void register_relay_channel(string channel)

// 消息会广播到所有监听器
// filter_listener() 决定哪些对象接收
```

## 优缺点分析

### 单例 Daemon 模式的优点

1. **状态集中**：全局状态管理清晰，避免分散和不一致
2. **易于调试**：可以方便地在 Daemon 中添加日志和调试代码
3. **资源节省**：无需重复加载相同数据和代码
4. **接口统一**：所有使用者通过相同接口访问服务
5. **配置便利**：可以在一个地方管理配置项
6. **协作简化**：Daemon 之间可以直接调用，结构清晰

### 缺点和局限性

1. **单点瓶颈**：
   - Daemon 负载过高可能影响整个系统
   - Daemon 崩溃影响依赖它的所有功能

2. **耦合度高**：
   - Daemon 之间相互依赖可能形成复杂依赖网
   - 修改一个 Daemon 可能影响多个系统

3. **扩展性受限**：
   - 单例难以并行处理
   - 无法简单地分布到多个进程/服务器

4. **测试困难**：
   - Daemon 通常依赖游戏环境
   - 单元测试需要搭建完整环境

5. **重启影响**：
   - Daemon 重启会丢失内存状态
   - 需要精心设计哪些状态需要持久化

### 与现代微服务架构的对比

| 特性 | Daemon 模式 | 微服务 |
|-----|------------|-------|
| 部署单元 | 单进程内 | 独立服务 |
| 通信方式 | 直接函数调用 | 网络 IPC |
| 状态管理 | 内存单例 | 数据库/缓存 |
| 扩展能力 | 有限 | 良好 |
| 故障隔离 | 差 | 好 |
| 技术栈 | 统一 LPC | 可混合 |
| 开发复杂度 | 低 | 中/高 |
| 运维复杂度 | 低 | 中/高 |

## 总结

Daemon 系统是侠客行 MUD 的骨架，提供了游戏运行所需的基础服务。这种单例服务模式虽然简单，但在 MUD 这种中等规模的应用中非常有效。每个 Daemon 专注于自己的领域，通过清晰的接口协作，形成了一个有机的整体。

随着游戏发展，Daemon 模式可能面临的挑战包括：
- 如何更好地解耦 Daemon 之间的依赖
- 如何支持动态加载和更新 Daemon
- 如何提供更好的 Daemon 监控和调试工具
- 如何处理 Daemon 的状态持久化和恢复

但作为一个经典的 MUD 架构，它至今仍在很好地工作，这本身就证明了这种设计的合理性和生命力。
