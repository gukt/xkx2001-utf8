# 12. 武林大会与竞技系统

## 1. 系统概览

武林大会是《侠客行》中最为核心的玩家竞技玩法之一，由巫师（Wizard）手动发起并管控，属于阶段性开放的集体活动。与野外随时可能发生的 PK 不同，武林大会提供了一个**受控、公平、分级**的竞技环境，让所有满足条件的玩家都能在同等起跑线上切磋武艺、角逐排名。

### 1.1 活动性质

武林大会并非全天候开放，而是依赖巫师通过指令触发。在 `/d/bwdh/square.c` 中，巫师可以使用 `start` 命令同时启动 8 个年龄组的比武，也可以使用 `stop` 命令随时终止比赛：

```lpc
// /d/bwdh/square.c
int do_start(string arg)
{
    object me = this_player();
    if (!wizardp(me)) {
        tell_object(me,"你不是巫师，不能发动比武大会！\n");
        return 1;
    }
    this_object()->set("start",1);
    "/d/bwdh/kantai1.c"->do_start("25");
    "/d/bwdh/kantai2.c"->do_start("30");
    "/d/bwdh/kantai3.c"->do_start("35");
    "/d/bwdh/kantai4.c"->do_start("40");
    "/d/bwdh/kantai5.c"->do_start("45");
    "/d/bwdh/kantai6.c"->do_start("55");
    "/d/bwdh/kantai7.c"->do_start("65");
    "/d/bwdh/kantai8.c"->do_start("99");
    
    CHANNEL_D->do_channel(me, "chat*","高声宣布：武林大会现在开始！\n");
    return 1;
}
```

当比武开始后，系统会通过全服聊天频道广播消息，引导玩家从扬州客店（`/d/city/kedian`）入场参赛。

### 1.2 参与条件

武林大会按**年龄**划分为 8 个组别（25、30、35、40、45、55、65、99 岁以下），每个看台独立运作。玩家需满足以下基本条件方可报名：

- **年龄限制**：玩家年龄必须严格小于该擂台设定的年龄上限。例如 25 岁以下组拒绝 25 岁及以上玩家报名。
- **经验门槛**：在 `kantai.h` 实现的报名逻辑中，对玩家的实战经验（`combat_exp`）有明确要求：
  - 25 岁以下：经验需达到 `(age - 15) * 4000` 以上
  - 25 岁以上：经验需达到 `(age - 15) * 8000` 以上
  - 90 岁以上高龄组：经验不得低于 150 万
- **败场淘汰**：在个人赛模式（`kantai.c`）中，累计战败达到 3 场即被禁止再次报名；在高级版本（`kantai.h`）中，战败达到 16 场（高龄组为 100 场）才会被彻底淘汰。
- **同对手次数限制**：在 `kantai.h` 中，任意两名玩家之间最多交手 3 次，防止恶意刷分。

```lpc
// /d/bwdh/kantai.h
if( (age < 25 && me->query("combat_exp") < ( query("age")-15 ) * 4000 ) ||
    (age > 24 && me->query("combat_exp") < ( query("age")-15 ) * 8000 )
)
{
    tell_object(me,"主持人对你说：武功这么差，你当武林大会是街头把戏呀？\n");
    return 1;
}

if( me->query("bwdh/defeated") >= 100 )
{
    tell_object(me,"主持人对你说：你这个连输八场的败将，回家好好练去吧！\n");
    return 1;
}
```

## 2. 场地结构

武林大会的场地布局采用了**"广场 - 看台 - 擂台"**三层空间模型，实现了观众与参赛者的物理隔离，并通过 MudOS 的房间继承机制快速复刻出 8 套并行的比赛场地。这一设计与 [[05-世界构建系统]] 中提到的 ROOM 继承体系深度结合。

### 2.1 空间层次

| 层级 | 代表文件 | 功能描述 |
|------|----------|----------|
| 广场 | `/d/bwdh/square.c` | 总入口、龙虎榜展示、巫师控制台 |
| 看台 | `/d/bwdh/kantai1.c` ~ `kantai8.c` | 观众区、报名处、候赛区、下注区 |
| 擂台 | `/d/bwdh/leitai1.c` ~ `leitai8.c` | 实际战斗区域，禁止自由出入 |

广场（`square.c`）作为枢纽，通过数字出口 `1` ~ `8` 分别连接到 8 个看台。每个看台通过 `up` 出口通往对应的擂台，擂台通过 `down` 返回看台。这种设计在 LPC 中通过宏定义实现高度复用：

```lpc
// /d/bwdh/kantai1.c
#define LEITAI "/d/bwdh/leitai1"
#define KANTAI "/d/bwdh/kantai1"
#define KANTAI2 "/d/bwdh/kantai12"
#include "/d/bwdh/kantai.h"
```

`kantai.h` 是核心看台模板，定义了通用的报名、排名、自动检测逻辑；而 `kantai1.c` ~ `kantai8.c` 仅做宏替换，即可生成 8 个独立房间实例。擂台同理，由 `leitai.h` / `leitai.c` 提供模板。

### 2.2 观众与参赛者的区域分离

- **广场与看台**设置了 `no_fight = 1` 和 `no_practice = 1`，禁止任何战斗和练功行为，确保观众安全。
- **擂台**设置了 `no_death = 1`，玩家在此区域内不会真正死亡，而是进入昏迷状态，随后被系统移回看台。这是 `damage.c` 中特殊逻辑的直接体现。
- **看台出口管控**：`valid_leave` 函数会检查玩家的 `bwdh/admitted` 和 `bwdh/fighting` 标记。已报名且未被淘汰的选手在比武期间不能自行离开看台（`out` 方向被拦截），防止中途离场逃避比赛。

```lpc
// /d/bwdh/kantai.h
int valid_leave(object me, string dir)
{
    if( dir=="up" ) {
        if( wizardp(me) || me->query("judge") ) return 1;
        else return notify_fail("你不能自己上去。\n");
    }
    if( dir=="out" ) {
        if( wizardp(me) || me->query("judge") ||
            me->query("bwdh/defeated") >= 16 || !me->query("bwdh") || !query("start") )
            return 1;
        else
            return notify_fail("你是选手,现在不能离开。除非你认输(abandon)不干了。\n");
    }
    return ::valid_leave(me, dir);
}
```

### 2.3 试剑山庄（团体赛扩展）

在 `/d/bwdh/sjsz/` 及其变体 `sjsz2`、`sjsz3` 中，还存在着一套更为复杂的**团体赛**场地。该空间拥有独立的广场、东西厢房、赌博房、擂台群以及控制物件（`obj/control`），支持以门派或战队为单位的对抗赛。 square.c 在特定时段（游戏时间 2、4、6、8、10~12 点）会开放通往试剑山庄的入口（`northup`/`north`/`south`），实现个人赛与团体赛的场地联通。

## 3. 比武规则

### 3.1 对战匹配机制

武林大会采用经典的**"擂主守擂、挑战者攻擂"**机制：

1. **首位报名者成擂主**：当某看台尚无擂主（`host`）时，第一个 `baoming` 的玩家自动成为擂主，并被立即传送上擂台。
2. **后续报名者成挑战者**：第二名报名者为挑战者（`challenger`）。报名后系统通过 `call_out` 延迟约 25~60 秒，给予双方准备时间，随后锣响开战。
3. **排队候补**：若已有挑战者正在比赛，后续报名者会被记录在看台的 `boy` 映射中，按时间顺序排队。当前一场比赛结束后，`auto_check` 会自动从队列中取出下一位挑战者上场。

```lpc
// /d/bwdh/kantai.c
if( !(host=query("host")) )
{
    set("host", me->query("id"));
    call_out("let", 0, me->query("id") + " play " + me->query("id"));
    message_vision("主持人说：好，现在$N是擂主！\n", me);
    return 1;
}

if( query_temp("fighting") )
{
    me->set("bwdh/admitted", 1);
    set("boy/" + time(), me->query("id"));
    return 1;
}

set("challenger", me->query("id"));
call_out("start", 25, host, challenger);
```

### 3.2 胜负判定

胜负由 `auto_check` 定时器每 10~30 秒扫描一次，核心判定逻辑如下：

- **挑战者胜利**：若擂主昏迷（`qi <= 0`）、死亡、断线或离开，则挑战者晋升为擂主，原擂主败场 +1。
- **擂主胜利**：若挑战者昏迷、死亡、断线，则擂主守擂成功，胜场 +1。特别地，若擂主已连胜两场（`bwdh/once` 标记），系统会强制其下场休息，以平衡比赛节奏。
- **无人挑战**：若连续 3 次 `auto_check` 均无新挑战者，擂主被视为"威镇八方"，自动获胜并下场休息。
- **断线处理**：若挑战者断线，系统会创建一个临时对象（`USER_OB`）加载其存档，记录败场并保存，确保比赛数据不丢失。

```lpc
// /d/bwdh/kantai.h
if( query_temp("no_challenger") >= 3 && userp(ob1) && living(ob1) )
{
    delete("host");
    message_vision(HIM"\n主持人高声说：$N真是威镇八方，居然无人敢上场挑战。\n"NOR, ob1);
    ob1->set("bwdh/win", 1 + ob1->query("bwdh/win"));
    sort_rank(ob1, 1);
    call_out("move_me", 1, ob1, KANTAI);
}
```

### 3.3 伤害控制与 damage.c

在标准 MUD 中，玩家被击败后可能死亡并承受死亡惩罚（掉经验、掉技能等）。为了在武林大会中保护参赛者，`/d/bwdh/damage.c` 对基础伤害函数进行了覆盖，其核心目标是**在 `no_death` 环境下将死亡转化为昏迷**。

```lpc
// /d/bwdh/damage.c
void die()
{
    if( environment()->query("no_death") && userp(this_object()) )
    {
        unconcious();      // 改为昏迷
        remove_call_out("revive");
        return;
    }
    // 标准死亡流程...
}
```

此外，`damage.c` 还实现了 `receive_damage`、`receive_wound`、`receive_heal`、`receive_curing` 四个基础生命接口，并通过 `unconcious()` 和 `revive()` 管理昏迷与苏醒。这意味着在擂台上：

- 玩家不会因为战败而真正**死亡**，避免了死亡惩罚和掉装备。
- 昏迷后会由系统自动移回看台，并通过 `full_all()` 瞬间恢复满状态（满精、满气、满内力等），准备下一轮比赛。

### 3.4 禁用规则

为了保证公平竞技，擂台（`leitai.h` / `leitai.c`）对玩家行为做了严格限制：

```lpc
// /d/bwdh/leitai.h
void init()
{
    add_action("do_get", "get");
    add_action("do_disable", "save");
    add_action("do_disable", "kill");
    add_action("do_disable", "hit");
    add_action("do_disable", "steal");
    add_action("do_disable", "fight");
    add_action("do_disable", "sleep");
    add_action("do_disable", "throw");
    add_action("do_disable", "halt");
    add_action("do_disable", "quit");
    add_action("do_kickout", "kickout");
}
```

- **禁止主动退出**：`quit`、`save` 被屏蔽，防止玩家通过断线或存档逃避战斗。
- **禁止非比赛攻击指令**：`kill`、`hit`、`fight`、`steal` 被屏蔽，确保所有战斗由系统统一触发（`ob1->kill_ob(ob2)`）。
- **禁止中途叫停**：`halt` 被屏蔽，比赛一旦开始就必须分出胜负。
- **统一装备来源**：玩家不能携带个人装备进入擂台。上场前，系统会没收其全部物品存入仓库（`cangku.c`），并提示玩家从兵器架（`jia`）领取标准武器和护甲。

```lpc
// /d/bwdh/leitai.h
string look_jia()
{
    msg = "这是一个兵器架，十八般武器是应有尽有。\n";
    msg+= "目前存放有剑(sword)，刀(blade)，棒(stick)，杖(staff)，轮(falun)，\n鞭(whip)，萧(xiao)。还有一些铁甲(armor)。\n";
    msg+= "请用指令\"get 种类 from jia\"拿取你称手的兵器或甲胄。\n";
    return msg;
}
```

## 4. 奖励系统

### 4.1 排名与龙虎榜

每个看台都维护着独立的排名数据，通过 `sort_rank()` 函数按胜场数（`win`）降序排列，并在看台的 `long_desc` 中以 ASCII 艺术形式的"龙虎板"展示前 10 名。广场（`square.c`）的龙虎榜则汇总展示各年龄组的冠军。

```lpc
// /d/bwdh/kantai.h
void sort_rank(object ob, int n)
{
    int nRank = query("nRank");
    int i;
    string id = ob->query("id");
    
    if(n > 0)
    {
        for(i = nRank; i >= 1; i--)
        {
            if(query("rank/"+i+"/id") == id) break;
        }
        if(i < 1)
        {
            nRank++;
            set("nRank", nRank);
            set("rank/"+nRank+"/name", ob->query("name"));
            set("rank/"+nRank+"/id", ob->query("id"));
            set("rank/"+nRank+"/win", ob->query("bwdh/win"));
            set("rank/"+nRank+"/loss", ob->query("bwdh/defeated"));
            i = nRank;
        }
        // 冒泡排序上升...
    }
}
```

### 4.2 观众下注

在 `kantai.h` 版本中，系统支持观众对正在比赛的两位选手下注。报名后出现挑战者到锣响前的窗口期，观众可以使用 `bet <玩家ID> <数量> <gold|silver>` 指令下注。

```lpc
// /d/bwdh/kantai.h
int do_bet(string arg)
{
    if( query("fighting") )
        return notify_fail("对不起，现在比武开始了，不能再下注！\n");
    if( name != query("host") && name != query("challenger") )
        return notify_fail("对不起，你只能压赌注在比武的两人身上！\n");
    
    money->add_amount(-amount);
    me->add("bet/"+name, amount*(money->query("base_value")/100) );
    set("bet_amount/"+name, query("bet_amount/"+name)+amount*(money->query("base_value")/100) );
    return 1;
}
```

比赛结束后，`update_room()` 会根据赔率（总赌注 / 获胜方赌注）向猜中的观众发放金银奖励，并清空本轮赌注记录。这一机制与 [[07-多人交互系统]] 的经济子系统紧密关联。

### 4.3 仓库（cangku.c）的用途

`cangku.c` 并非普通仓库，而是武林大会的**装备仲裁中心**。其工作流程如下：

1. **寄存**：玩家进入擂台前，`let()` 函数遍历其全部物品，将除特殊物品（如 `yu di`）外的所有装备移入仓库，并按玩家 ID 建立 `basket` 数组。
2. **战斗**：玩家在擂台上只能使用兵器架提供的标准装备，杜绝了神兵利器对公平性的破坏。
3. **取回**：当玩家被移回看台时，`init()` 或 `restore_stuff()` 会自动从仓库取出该玩家寄存的所有物品归还。

```lpc
// /d/bwdh/cangku.c
inherit ITEM;
void create()
{
    set_name("雪领仓库", ({ "cangku", "ku"}) );
    set_weight(100000000);
    set_max_encumbrance(500000000);
    set("no_get", 1);
    set("no_clean_up", 1);
}
```

仓库对象被设置为不可拾取（`no_get`）且不可自动清理（`no_clean_up`），确保跨比赛周期的物品安全。

## 5. 组织与报名

### 5.1 报名流程

玩家在看台内输入 `baoming` 或 `报名` 即可触发 `do_baoming()` 函数。该函数由看台的 `init()` 动态绑定到玩家指令集。报名成功后，系统会检查年龄、经验、败场等条件，合格者被标记为 `bwdh/admitted`，随后通过 `let()` 函数传送至擂台。

```lpc
// /d/bwdh/kantai.h
void init()
{
    add_action("do_baoming", "报名");
    add_action("do_baoming", "baoming");
    add_action("do_start", "start");
    add_action("do_stop", "stop");
    add_action("do_bet", "bet");
}
```

### 5.2 裁判与主持人的实现

武林大会没有独立的裁判 NPC 对象，而是将**主持人逻辑直接嵌入看台房间**（`kantai.c` / `kantai.h`）中。这种做法在 LPC 架构中非常典型——房间本身即是一个具有状态和行为的控制器：

- 房间保存 `host`、`challenger`、`boy`（排队队列）、`rank`（排行榜）等关键状态。
- 房间通过 `call_out` 调度 `auto_check`、`start`、`let` 等函数，实现自动化比赛流程。
- 所有"主持人说：……"的广播，实际上都是由房间对象通过 `message_vision()` 或 `tell_room()` 发出。

在团体赛场地（`sjsz/`）中，则存在更复杂的控制物件（`obj/control`）和留言板（`bwboard`），用于管理战队信息、积分和赛程编排。

### 5.3 现场直播

为了让无法亲临现场的玩家也能观看比赛，`/d/bwdh/obj/live.c` 被放置在擂台房间中。该物件会拦截擂台内的所有消息（`receive_message`），并实时转发到看台的 `down` 和 `northdown` 出口房间。这是一种早期 MUD 中非常巧妙的"闭路电视"机制：

```lpc
// /d/bwdh/obj/live.c
void receive_message(string msgclass, string msg)
{
    mapping exits = environment(this_object())->query("exits");
    string where = environment(this_object())->query("short");
    if (msg[0..0] != "\n") msg = GRN"【"+where+"】"NOR + msg;
    else msg = replace_string(msg, "\n", "\n"GRN"【"+where+"】"NOR);
    tell_room(exits["down"], msg+"\n", this_object());
    tell_room(exits["northdown"], msg+"\n", this_object());
}
```

此外，`/d/bwdh/obj/camera.c` 提供了更专业的摄影师 NPC，支持将画面广播到任意指定房间，甚至跨房间直播。

## 6. 与其他系统的关联

武林大会并非孤立存在，它与游戏核心子系统有着深度耦合。

### 6.1 与 [[01-架构总览]] 的关联

武林大会是整个游戏社交与竞技生态的顶层应用之一。它利用了 MudOS 驱动提供的 `call_out` 定时器、`exec()` 玩家切换、`save_object` / `restore_object` 存档机制，以及 LPC 的继承体系（`inherit ROOM`、`#include` 模板复用），展现了 LPC MUD 典型的"房间即控制器"架构风格。

### 6.2 与 [[05-世界构建系统]] 的关联

比武场地完全基于 ROOM 继承树构建。广场、看台、擂台均继承自标准 ROOM，但通过设置 `no_fight`、`no_death`、`invalid_startroom` 等属性，改变了房间的基础行为。8 组场地的快速克隆则依赖于 `kantai.h` / `leitai.h` 的头文件模板机制，这是 [[05-世界构建系统]] 中模块化房间设计的典范。

### 6.3 与 [[06-武功与战斗系统]] 的关联

比赛中实际调用的是标准战斗接口：`kill_ob()`、`receive_damage()`、`unconcious()`。`damage.c` 通过覆盖这些函数，在不修改全局战斗逻辑的前提下，为武林大会场景定制了"免死+昏迷"的规则。兵器架提供的标准武器（钢剑、钢刀、铁甲）则保证了战斗只取决于玩家自身的武功修为，而非装备差距。

### 6.4 与 [[07-多人交互系统]] 的关联

武林大会是多玩家协同与对抗的集中体现：

- **频道广播**：比赛开始、结束、关键胜负均通过 `CHANNEL_D` 的 `chat*`、`xkx` 频道全服通告。
- **经济交互**：下注系统直接调用货币对象（`/clone/money/gold`、`/clone/money/silver`），涉及玩家间财富的再分配。
- **组队限制**：`damage.c` 中的 `dismiss_team()` 会在玩家昏迷时强制解散队伍，防止组队干扰单挑公平性。
- **社交仪式**："穆易比武招亲"（`/d/bwdh/npc/mu.c`）是武林大会机制在剧情社交中的延伸，将竞技系统与婚姻系统打通，玩家在擂台上技压群雄后可赢得配偶和嫁妆（白金戒指）。

---

综上，武林大会与竞技系统通过精心的场地隔离、严格的装备仲裁、自动化的比赛调度和完善的排名奖励机制，为《侠客行》构建了一个既热血又公平的武侠竞技舞台。其代码实现充分体现了 LPC MUD 在状态驱动、事件调度和模板复用方面的设计智慧。
