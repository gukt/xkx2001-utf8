# 迁移路径与避坑清单（v3 收缩版）

> 本路线图以第三轮专家对抗复审的收敛裁决为驱动，按"垂直切片优先、greenfield 重写、规格提取先行、单机验证优先、性能基准硬门禁"原则编排。相对 v2（见 `_archive/04-v2-迁移路径与里程碑.md`），核心变化：**砍分布式阶段的重排、存储收缩为内存+JSON、性能基准从"决策 compiled core"改为"go/no-go 硬门禁"、全仿真确定性与逻辑热重载后置、目标下调为单机千人+百并发**。

---

## 一、总体原则

1. **垂直切片优先**：阶段 -1 先验证 5-10 房间 + 2 NPC + 1 战斗 + 1 任务 + 1 对话的 DSL+Agent 闭环，不通过则不投入引擎重构。
2. **单机验证优先**：在验证 UGC 成立、改造顺利之前，不引入分布式/K8s/Rust/PG。先用单机纯 Python + 内存 JSON 跑通两件事：千人在线+百并发、DSL+Agent 创作闭环。
3. **greenfield 重写**：无双栈、无过渡期、无适配层、无逐步替换 LPC。LPC 代码是规格源。
4. **规格提取先行**：阶段 0 的核心工序是从 LPC 提取形式化规格契约，而非录制运行中系统。
5. **性能基准硬门禁**：阶段 0 必须做单 do_attack μs 基准 + 1s tick 预算 + 1000+100 负载压测 + PYTHONHASHSEED=0 + GC 基准，**不达标触发 kill criteria 而非继续推进**。
6. **目标务实**：单机 1000 人在线 + 100 并发（现状 MAX_USERS=50 的 20 倍）；单房间串行是 MUD 固有硬约束。
7. **收敛优先于完备**：识别过早抽象，砍掉为不存在的规模/受众建设的基础设施。

## 三、分阶段路线图

### 阶段 -1：垂直切片平台验证（2-3 月）★ 最高优先级

**目标**：在投入引擎重构前，验证核心价值主张"DSL+Agent 让非程序员创作可玩世界"。

```text
5-10 房间 + 2 NPC + 1 战斗 + 1 任务 + 1 对话
         │
         ▼
    全部 DSL 定义
         │
         ▼
    Agent 部分生成（copilot 模式）
         │
         ▼
    跑在真实引擎最小子集
    （resolve_attack 纯函数 + stub 运行时）
```

| 任务 | 产出 | 验收标准 |
|---|---|---|
| DSL 表达力验证 | 垂直切片全部 DSL 定义 | 不依赖逃生舱即可完成 5 房间+2 NPC+1 战斗+1 任务+1 对话 |
| Agent 产出可用性 | Agent 生成的 DSL 初稿 + 人工修订量度量 | 人工修订量可接受，无结构性错误 |
| 玩家可玩性验证 | 3-5 名目标玩家试玩反馈 | "觉得好玩"评分 ≥ 可继续投入阈值 |
| resolve_attack 纯函数提取 | 无副作用纯函数 + 属性测试 | 输入组件快照 + seed -> 输出 CombatRoundResult；与未来真实引擎同源 |
| **非武侠微场景验证 CombatKernel 主题无关性** | 非武侠微场景跑在 CombatKernel 上 | 核心引擎未硬编码武侠语义（硬门禁） |

**不可妥协**：本阶段失败则停止项目，不投入引擎重构。非武侠微场景无法验证内核主题无关性则暂停，先做内核主题无关性重构再继续。

```text
│  ├─ 命令 8 段中间件管线（Command 形式化） │
│  └─ 引擎工具链最小三件同步开发            │
├─────────────────────────────────────────┤
│ 单进程 WS 服务器 + 会话 + argon2/HS256    │
│  └─ 乐观执行 + rollback delta             │

```

| 里程碑 | 验收 |
|---|---|

| SchemaRegistry 类型化组件 | query 拼写错误启动期失败 |

| 内存权威 + JSON 存档 | 原子写 + offload + dirty-flag；崩溃冷重启从 checkpoint 恢复 |

## 四、kill criteria（止损标准）

1. **阶段 -1**：DSL+Agent 创作闭环验证失败（垂直切片无法用 DSL+Agent 完成且行为等价）-> **停项**，不投入引擎重构。
2. **阶段 -1**：非武侠微场景无法验证 CombatKernel 内核主题无关性（核心引擎硬编码武侠语义无法剥离）-> **暂停**，先做内核主题无关性重构。
3. **阶段 0**：micro-benchmark 单 do_attack 超阈值 -> 先优化热路径（对象池/避免分配/缓存）；1000 实体 tick 超预算 -> 降 tick 频率或裁剪 System；1000+100 经优化仍不达标且无 Rust 退路 -> **降级目标到 500 在线+50 并发**（仍是当前 10 倍）并重新评估 Rust/Go 热路径。
4. **阶段 0**：30 文件表达力校准显示层3 逃生舱使用率经原语扩充后仍 >15% -> **暂停 Agent 投入**，先扩层1-2 表达力（超标触发层1-2 迭代而非放松 KPI）。
5. **Agent 创作**：单 CPK 经 3 轮生成-修订迭代后人工修订量仍 >40% -> DSL+Agent 假设走弱，先扩 DSL 层1-2 表达力再继续；扩后仍 >30% -> **Agent 降级为辅助**（人工为主 Agent 建议为辅）；LLM token 预算耗尽且未产出通过闭环的 CPK -> 停止 Agent 投入回退人工创作 DSL。
6. **阶段 1**：单进程核心循环集成测试无法支撑 1000+100 -> **冻结功能范围**，纯做性能优化直至达标或触发目标降级。
7. **项目级**：18 个月未达 M3（单题材可玩 demo）-> **冻结迁移**，聚焦已迁移内容产品化。
8. **JSON 存储**：任何外部玩家测试开始前必须迁 PG（崩溃丢 30s 对外部玩家不可接受）--硬止损线非可选。
9. **性能后置触发**：do_attack 微基准与 1000+100 负载测试双失败且 Python 优化穷尽 -> 重新评估 Rust/Go 热路径（compiled core 重新引入的唯一触发条件）。

6. **阶段 1**：combat 确定性回放验证（CombatContext 快照 + seeded RNG + PYTHONHASHSEED=0 + 禁 set 迭代，任意 tick 快照 + 输入日志 = 完整重放）。

## 六、风险驱动的"不做"清单（v3 收缩版）

| 不做 | 理由 | 替代 / 何时重新引入 |
|---|---|---|
| 分布式全套（分片/Louvain/handoff/Ray/一致性哈希/placement） | 过早复杂度，验证 UGC 前不引入 | 单进程；单进程达 80% 承载且 UGC 验证成立后 |
| Redis 全部依赖（session/ring/限流/缓存/JobStore） | 单进程内全部内存化 | 进程内 dict/deque/令牌桶/EventBus/timer wheel |
| 分布式网关（SO_REUSEPORT 多 worker/NATS 跨节点扇出） | 单进程足够 | 单进程 WS 服务器 |
| K8s/Helm/ArgoCD/OpenSearch/ClickHouse/KEDA | 运维重型件后置 | OTel+Grafana+Langfuse；需编排 2+ 服务后 |
| Rust/Go compiled core | 暂纯 Python | 双失败且 Python 优化穷尽后 |
| PG/MongoDB 权威态 | 改造期内存+JSON | 外部玩家测试前必须迁 PG |
| `world://` URL 寻址 | 单机无 shard/region | entity_id(UUID)+进程内引用；分布式阶段加回 |
| uvloop 硬依赖 | 1000 连接 I/O 非瓶颈，C 扩展兼容性风险 | stdlib asyncio 优先；基准显示 I/O 瓶颈时启用 |
| JWT RS256+refresh+Redis 黑名单 | 单进程无跨进程鉴权 | HS256/内存 session token+内存吊销集合 |
| 题材包运行时热插拔（register/unload/version/isolation） | 过早抽象，1 题材+1 开发者 | 启动时静态加载；第二题材真实存在且需不停服切换后 |
| 独立自定义规则引擎 / RETE / OPA / Drools | 层1 已是规则表示层，再建是重复抽象 | 层1 IR + 薄求值子模块 + System 三分 |
| "每个状态变更（含 NPC/tick）都走 Command" | 破坏重放模型 + GC 压力 | Command 仅覆盖外部意图；System.update 覆盖派生变更 |
| Command 类层级 | 8 段管线已是 Command，新类冗余 | 管线信封形式化命名 |
| WASM 沙箱 | 单进程无多租户进程隔离需求 | RestrictedPython 统一两层；多租户生产后 |
| APScheduler+Redis JobStore 持久调度 | 单进程崩溃冷重启 | 进程内内存 timer wheel |
| 50ms/20Hz or 100ms/10Hz tick 框架 | LPC heart_beat 实测 1s，该框架解 MMO 不存在的问题 | tick=1s + compute<100ms |
| 全仿真确定性作为阶段1 里程碑 | 单机无分布式回放需求，对每个 System 征收确定性税 | combat-only 确定性；全仿真后置 M3 后 |
| 逻辑热重载协议（schema migration+entity state migration+'无跨分片 handoff'） | handoff 已不存在，"安全窗口"是分布式概念 | 维护窗口重启；有在线玩家不能容忍重启后 |
| 6 件工具链作为阶段0 一等公民 | 单机+1000 在线下过重 | 最小三件（inspector/profiler/replay viewer） |
| CombatKernel 跨 4 题材预先抽象 | 为泛化而平庸化武侠 70% 脚本灵魂 | 从武侠 do_attack 提取；非武侠微场景验证 |
| CombatRoundResult/Effect 对象池化与 GC 微优化 | 1s tick+8000 实体下 GC 是非问题 | __slots__ 廉价卫生可做；对象池用 tick profiler 实测后再决定 |
| 分布式燃料聚合器 | 单进程内 per-CPK fuel 即进程内计数器 | 进程内计数器 |
| CRDT/OT 多创作者 CPK 协作 | 单机验证不需要 | 单作者 CPK + fork-merge |
| 内容市场/分发/计费实现 | 阶段 -1~2 仅预留字段 | manifest 预留 author_id/revenue_share/price/title/tags |
| 双栈过渡 / 适配层 / 修 securityd 旧代码 | greenfield 不存在这些概念 | 规格提取后直接实现 |
| 录制运行中 LPC 命令流为 golden trace | 旧系统在当前 Linux 不可运行 | 单元级行为规约 + 一次性 golden trace |
| 全量事件溯源(CQRS+ES)作为默认持久化 | 与同步调用语义冲突 | 内存权威+JSON 存档（高价值事件 audit_event） |
| 全量事件驱动替代 heart_beat | heart_beat 是 tick 驱动器 | 保留周期 tick，层1 只覆盖触发器 |
| "秒级回滚无数据丢失" | schema 漂移导致不一致 | 逻辑回滚≠状态回滚 |
| LangGraph active-active 多副本编排 | 单进程库，重复分派 | 内存状态机 / 单编排器+故障转移 |

## 七、避坑重点（专家复审警示）

1. **文档与约束"两张皮"是最需先修**：v2 子系统文档 20 处分布式残留 + 9 处 PG/Redis/NATS 残留必须主动剪除，否则实现者按旧文档引入已砍复杂度。本文档集已剪除，实现时若发现残留须即时修正。
2. **tick 预算概念性错误（50ms/20Hz）**会从后门请回 compiled-core 紧迫感，必须用 1s tick 重写并附 `set_heart_beat(1)` 实证。
3. **战斗手感时序**："按 step 顺序 apply"措辞暗示可先算后 apply，必须显式改为"按文本与状态交织真实顺序记录"。
4. **存储收缩是隐性重新设计非"换存储"**，必须逐一标注丢失语义（事务原子性/崩溃恢复/并发写 CAS/关系完整性/append-only 可篡改）+ 止损线（外部玩家测试前必须迁 PG）。
5. **PronounContext 必须携带 viewer**（`rankd.c` 实证 `this_player()` 依赖），单机下 `this_player()` 映射 `ActionContext.actor` 需在 System tick 无"当前说话者"时定义回退。
6. **规则引擎否决但层1 原语蠕变风险**需 KPI + 判定标准护栏，薄求值器命名须避免"引擎"诱发 scope creep。
7. **Command 不新建类层级**，force_me 是保真让步须显式承认。
8. **5 灵魂系统降级为垂直切片 1-2 个代表性元素**，276 文件武林大会等推迟 M3 后，避免阶段2 范围过载。
9. **JSON 存档崩溃安全**：必须 write-temp+os.replace 原子写 + offload + dirty-flag 分摊，不得重蹈 LPC `save_object` 全量覆盖无原子写的覆辙。
10. **"暂不考虑"需附触发条件**：热插拔/分布式/PG/K8s/Rust 等后置项若无触发点会滑成"永不考虑"，本文档已为每项附触发条件。

## 八、决策检查点

每个阶段结束前，必须回答以下问题，任一为否则不推进：

**阶段 -1 -> 0**：

- DSL 能否表达垂直切片全部内容？
- Agent 产出的人工修订量是否可接受？
- 目标玩家是否觉得好玩？
- `resolve_attack` 纯函数是否提取成功且与未来引擎同源？
- **非武侠微场景是否验证 CombatKernel 主题无关性？**

**阶段 0 -> 1**：

- LPC 规格提取是否覆盖 go/move/combat 核心路径？
- FluffOS Linux 编译是否可行或已降级为单元级行为规约？
- **性能 micro-benchmark 是否达标（go/no-go）？**
- 引擎工具链 PRD（最小三件）是否评审通过？
- 30 文件表达力校准层3 是否 <15%？

**阶段 1 -> 2**：

- 单进程 asyncio 核心循环是否跑通？
- **1000 在线+100 并发是否达标？**
- combat 确定性是否可重放？
- Effect/ConditionHandler 契约是否落定？
- 内存权威 + JSON 存档是否稳定（原子写 + 崩溃恢复）？

**阶段 2 -> M3**：

- Combat 迁移是否通过行为等价验证 + 文本体验流 diff？
- 技能三层是否明确？
- 称谓系统、世界观治理层（代表性元素）是否落地？
- **门派内容包边界是否干净切割（核心引擎无武侠烙印）？**

**M3 -> 后置阶段**：

- 单进程容量是否实测达 80%？
- 是否需要外部玩家测试（触发 PG 迁移）？
- 第二题材是否真实存在（触发热插拔评估）？
