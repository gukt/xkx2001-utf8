## 整合遗留收纳（T10 前置）

T10 前收纳 Wave 2/3 整合遗留（[PROGRESS.md](../../PROGRESS.md) Wave 2/3 整合遗留）：

1. **CombatState 扩展**：加 `guarding`/`is_fighting`/`fight_dodge` 字段 + `to_snapshot` 传递（[ADR-0023](../adr/ADR-0023-combat-determinism-boundary-simplification-ledger.md) 决策 4 第 4/5 项，T6 用默认值兼容）
2. **Engine 统一 tick 循环**（[engine.py](../../engine/src/xkx/runtime/engine.py)）：`Engine` 类维护 `systems: list[SystemLike]`，每 tick 遍历调 `update`，用 `TickProfiler.measure_system` 包裹计时
3. **CombatBridge 适配器**：`CombatSystem` 的 `tick(snapshot, seed)` 接口与 `System.update(world, tick)` 不同，适配器在 runtime 层桥接（combat 包保持自包含）。按 `enemy_ids` 构建 input_log（O(活跃敌对对)），非 `CombatSystem.tick` 的 O(n²) 全两两遍历--后者 100 活跃战斗者下 9900 次攻击/tick 会超预算
4. **StorageSystem/ConnectionSystem 纳入** Engine.systems（`Engine.add_system` 自动挂 `world.storage_system` 供 mark_dirty）
5. **mark_dirty 整合**：CombatBridge/ConditionSystem update 末尾调 `storage.mark_dirty`（[ADR-0022](../adr/ADR-0022-json-save-crash-recovery-dirty-flag.md) §4 mutation 路径标记）

## 与 ADR-0012 呼应

[ADR-0012](../adr/ADR-0012-performance-microbenchmark.md) 阶段 0 μs 基准（resolve_attack hit 25.9μs）是 T10 的前置数据点。T10 在 1000+100 集成场景下验证：

- 50 对战斗 * ~106μs/对（含 snapshot 构建 + replay + apply_effects）= 5.3ms < 100ms
- 单次 resolve_attack 在集成场景下 ~106μs（μs 基准 25.9μs 的 ~4x，含 snapshot/apply 开销）
- 1000 实体下 CombatSystem 非 tick 瓶颈（5.3ms / 100ms 预算 = 5.3%）

## 瓶颈分析

CombatSystem 占 tick compute 92%（5.3ms / 5.8ms mean）。瓶颈定位：

- `to_snapshot`：每战斗实体构建 CombatantSnapshot（pydantic model 创建 + 组件查询），~50μs/entity
- `replay`：50 次 resolve_attack，~26μs/次
- `apply_effects`：flatten + apply，~10μs/次

1000 在线下 50 活跃战斗对是合理上界（5% 玩家同时战斗）。若战斗对增至 200（20% 玩家），CombatSystem ~21ms 仍 < 100ms。性能余量 8x。
