## 二、任务 1：LPC 规格提取管线（9 层分解）

### 方法论

见 [ADR-0010](../../docs/adr/ADR-0010-lpc-spec-extraction-methodology.md)。核心要点：

- **规格契约四要素**：签名 + 前置/后置条件 + 不变量 + 副作用 + 随机性
- **提取粒度**：函数级（不逐行翻译）
- **表示格式**：pydantic v2 模型，存于 `engine/src/xkx/spec/`
- **范围**：9 层约 4500-5000 行 LPC（feature/ 36 文件 + 核心 inherit + 核心 daemon）
- **辅助**：对难判路径运行旧 driver 录制 golden trace（[ADR-0009](../../docs/adr/ADR-0009-original-driver-runnable.md)）

## 七、规格提取后续补充计划

任务 1 的 9 层提取是**阶段 0 的一次性核心工序**，覆盖核心可玩循环约 7000 行 LPC（feature/ 36 文件 + 核心 inherit + 核心 daemon），不需要"重做"。但本次按 ADR-0010 §6 边界跳过的部分，未来按以下 3 类分阶段补充。**关键区分**：规格提取针对"引擎逻辑"，DSL+Agent 创作闭环（[03](03-DSL-UGC与Agent协作.md) / [07](07-agent-schema-mapping.md)）针对"内容数据"，两类工作性质不同不混用。

| 类别 | 范围 | 补充时机 | 方法 |
|---|---|---|---|
| **1. 引擎逻辑补充** | condition 具体状态类型（蛇毒/醉/失明等，本次只提取 apply/update/clear 框架）；第二梯队守护进程（CHANNEL_D/MONEY_D/UPDATE_D/RANK_D等）；riposte 递归 / s_combatd 阵法合击 / perform·exert 完整实现；hit_ob/hit_by mapping；门状态机运行时（do_knock/call_out 定时关/跨房间 exits 同步） | 阶段 0 后期或阶段 1 实现到该子系统时按需补 | 同本次规格提取方法（函数级契约 + pydantic 模型，追加到对应 `spec/layer_*.py`） |
| **2. 后置系统** | 阴间世界流程（黑白无常/还阳路径）；InterMUD；武林大会；坐骑交通；婚姻；PvP 通缉机制细节；尸体四阶段腐烂 | 阶段 1/2 实现该系统时才提取 | 规格提取（引擎逻辑部分）+ DSL 声明（内容部分） |
| **3. 阶段 2 内容迁移** | kungfu/（798 门派武学文件）；d/（6414 区域内容文件） | 阶段 2 子系统迁移 | **不用规格提取**，走 DSL+Agent 创作闭环（[03](03-DSL-UGC与Agent协作.md) / [07](07-agent-schema-mapping.md)，阶段 -1 验证的模式） |

**不提前提取的原则**：以上 3 类均"实现到时才补"，不提前批量提取。理由：（a）避免阶段 0 陷入穷尽细节违反收敛原则；（b）后置系统设计可能在阶段 1 演进，提前提取的规格易过期；（c）阶段 2 内容迁移用 DSL+Agent 模式，函数级契约不适用。

**规格细化（非补充）**：任务 3（单元规约）基于本次 9 层规格生成 hypothesis 属性测试时，如发现某函数规格不够精确（如前置条件遗漏边界 case），就地补充该函数的 FunctionSpec 细节，不视为"重做"。
