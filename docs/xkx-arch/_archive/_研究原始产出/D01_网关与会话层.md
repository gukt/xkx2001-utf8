# 方案设计·网关与会话层

## 设计概述
核心理念：把现有 MudOS 单进程里"连接入口(master.connect)+登录态机(login.c/logind.c input_to 回调链)+消息分发(F_MESSAGE)+断线恢复(net_dead/reconnect)"这套紧耦合的会话逻辑，拆解为一个独立、无状态、可水平扩展的网关与会话层，作为旧 LPC 世界与新分布式 Python 后端之间的"协议与状态接缝"。

三个不可妥协的边界：
1) 网关无状态、纯协议层：WSGateway 只做帧编解码、握手、限流、心跳、会话粘附路由，绝不持有游戏状态。会话状态（session_id->角色/世界/分片/last_seq/netdead/消息缓冲环）全部外部化到 Redis，使任一网关实例崩溃不丢会话定位，玩家重连可路由到正确分片。
2) 会话与游戏实体解耦：现有 login 对象=连接+登录态，user 对象=角色体，二者靠 link_ob/body_ob 临时指针绑定。新架构把"会话(Session)"提升为一等实体，用稳定 session_id 替代 file_name 既是身份又是路径的强绑定，Player Actor 用实体 ID 寻址而非对象指针，断线只会话挂起(SUSPENDED)而角色体留在世界继续被仿真。
3) 以 receive_message(msgclass,msg) 为渲染接缝保留 Telnet 兼容：这个签名是新旧两套传输共用的最小公约数。WS 路径把 msgclass 转成结构化事件 Frame（带 type/seq/payload），Telnet 路径把同一 msgclass 渲染回 ANSI+代词替换文本流。两条路径共享同一 Player Actor 出站，过渡期可旁路桥接器连 MudOS 旁车，降低迁移风险。

关键技术决策：MsgPack 为默认二进制协议（高频战斗帧紧凑），JSON 兜底 UGC 动态内容；统一帧封装 {type,version,seq,trace_id,payload}；状态同步复用 F_DBASE 的 dbase(持久 snapshot)/tmp_dbase(易失 delta) 两层分离--进场景/重连发 snapshot，变更发 JSON-Patch 风格 delta + 单调递增 seq；断线重连用 per-session ring buffer 重放 last_seq+1..current 缺序号段，缺口超环容量则退化全量 snapshot。心跳双层：WS Ping/Pong 20s 探活 + 应用层心跳带 last_seq；挂起窗口沿用现有 NET_DEAD_TIMEOUT=900s 语义，超时自动落盘退出。认证用 argon2 替代 DES crypt()，JWT(RS256) access 5-15min + refresh 7d，token 载 character_id/world 供跨分片路由与权限消费。

## 组件
- **WebSocket 接入网关 (WSGateway)**：承载海量长连接的协议层与流量入口。基于 aiohttp + uvloop 单进程事件循环，以 SO_REUSEPORT 多 worker 水平扩展（单机数万连接）。职责严格收窄到：(1) TLS 终结与 WS 握手；(2) 帧编解码（MsgPack/JSON 双模）；(3) origin 校验、per-IP/per-session 令牌桶限流（替代 alias.c 的对象内 cnt 计数器天雷机制，做全局视角限流）；(4) 会话粘附--按 token 中的 character_id/world 路由到对应游戏分片，session_id↔node 路由表存 Redis；(5) 心跳 Ping/Pong 20s 探活 + 应用层序号心跳；(6) 断线挂起而非销毁会话。网关无状态、可随意重启，不持有游戏状态。（技术：Python 3.11+ asyncio, aiohttp(WS+HTTP) 或 websockets 库, uvloop 事件循环加速, SO_REUSEPORT 多 worker, MsgPack(msgpack-python) 主协议, JSON 兜底）
- **会话管理器 (SessionManager)**：会话生命周期的权威持有者，是现有 master.c::connect() / login.c / user.c 的联合替代。维护 session_id->{character_id, world, shard_node, state, last_seq, netdead, msg_buffer_ring, created_at} 状态机。状态流转：CONNECTING(登录态)->AUTHENTICATED(已发 token)->ACTIVE(已绑定游戏分片)->SUSPENDED(断线挂起，复用 NET_DEAD_TIMEOUT=900s 窗口)->RESUMING(重连对齐中)->ACTIVE 或 CLOSED。对应 login.c 的 logon->time_out 链、user.c 的 net_dead->user_dump 链。会话状态外部化到 Redis（支持网关无状态重启不丢会话定位）。（技术：asyncio 状态机, Redis Hash 存会话记录, session_id 用 UUID v4/secrets.token_urlsafe）
- **认证服务 (AuthService)**：替代 logind.c 的 input_to 回调链登录与 securityd 的 crypt 校验。提供账号口令校验（argon2 替代 DES crypt()）、角色/能力解析（替代 wizhood 的路径表 switch）、JWT 签发与刷新、token 吊销黑名单。登录流程从现有同步 input_to 串行回调改为单次 REST POST /auth/login（FastAPI），返回 access_token(5-15min) + refresh_token(7d)。token 载 character_id/world/wiz_level，供网关跨分片路由与权限服务消费。保留现有 REGBAN_D IP 白名单/封禁（regband.is_banned）作为登录前置闸，迁为 AuthService 的 IP gate 中间件。（技术：FastAPI(REST), argon2-cffi 密码哈希替代 crypt(), PyJWT(RS256 非对称签名), Redis 黑名单(吊销), argon2 替代单轮 DES）
- **协议编解码器 (ProtocolCodec)**：统一出入站消息帧的序列化与语义封装。出站把游戏事件（对应 message(msgclass,msg,target) + 子类 channel:/outdoor:/weather:）编为结构化 Frame；入站把客户端 Frame 解为 Command 交给 Player Actor。帧规范：{type, version, seq, trace_id, payload}。type 区分 SnapshotFrame(全量)/DeltaFrame(增量 JSON-Patch 风格)/EventFrame(一次性事件)/AckFrame(序号确认)/HeartbeatFrame。直接对应 F_DBASE 的 dbase(持久 snapshot)/tmp_dbase(易失 delta) 两层分离。复用 receive_message 的 msgclass 路由语义：channel:->订阅过滤，outdoor:/weather:->前端按场景元数据决定渲染。（技术：msgpack-python 主序列化, JSON 兜底(UGC 动态内容), 可选 protobuf 仅在高频战斗压测瓶颈后局部引入, Frame 用 dataclass + pydantic 校验）
- **断线重连与状态对齐器 (ReconnectReconciler)**：解决断线后的状态漂移，是现有 net_dead->reconnect 的分布式增强版。重连三步：(1) 校验 token + session_id 有效性（SUSPENDED 窗口内）；(2) 客户端上报 last_seq，服务端从 per-session ring buffer 重放 last_seq+1..current_seq 的 delta（缺序号超 ring 容量则退化为全量 snapshot）；(3) 对齐后切 ACTIVE、清 netdead 标志、移除 user_dump 定时器（对应 user.c::reconnect 的 set_heart_beat(1)/remove_call_out）。客户端用指数退避重连（1s->2s->4s->上限 30s）。挂起窗口沿用 NET_DEAD_TIMEOUT=900s 语义，超时自动 quit 落盘。（技术：Redis 存 per-session ring buffer(有序集合 ZSet 按 seq 排序), TTL=挂起窗口, asyncio 重放协程）
- **Telnet 兼容适配器 (TelnetAdapter)**：过渡期兼容现有 MUD 客户端的桥梁，以 receive_message(msgclass, msg) 为渲染接缝。出站：把结构化 Frame 渲染回 ANSI 文本流（HIR/HIC/NOR/BOLD 颜色码 + 代词替换 $N/$n/$P/$p 经 message_vision 渲染），直接对应 feature/message.c 的 receive 路径与 languaged Big5 转码；入站：把 telnet 文本行解析为 Command（保留 history !/!N、alias 展开、process_input 管线语义）。可选桥接器旁路连 MudOS 旁车，降低迁移风险。WS 与 Telnet 共享同一 Player Actor，仅传输层不同。过渡完成后逐步下线。（技术：Python telnet 协议(telnetlib3 或自研 asyncio telnet handler), ANSI 渲染复用 message_vision 纯函数逻辑, languaged 映射表迁为 UTF-8 默认(去掉内联 Big5 转码)）
- **速率限制与安全中间件 (RateLimiter / SecMiddleware)**：把现有分散的安全机制收敛为网关中间件链。(1) 令牌桶限流替代 alias.c 的对象内 cnt 计数器（分布式可见、可配阈值，替换天雷伤害这种副作用式惩罚为明确的 429/限流反馈）；(2) per-session 消息配额（频道防刷屏，对应 channel_msg_cnt）；(3) origin 校验、WS 握手头校验；(4) 跨节点限流状态共享（Redis 计数器）。保留 force_me 的审计思想：特权动作走显式 API + 审计日志，而非 ROOT_UID 后门。（技术：Redis 令牌桶(原子 INCRBY + EXPIRE), asyncio 中间件管线, 结构化审计日志(log_file 的 PLAYER_DEATH/READ_LOG/WRITE_LOG 现代化)）

## 关键接口/事件
- 客户端->网关 WS 帧（入站）: Frame{type:'command'|'heartbeat'|'ack'|'resume', version, seq?, payload}. command 的 payload 含 verb+arg，等价现有 command(me, process_input(str)) 的动词模型；resume 携 last_seq 发起重连对齐
- 网关->客户端 WS 帧（出站）: Frame{type:'snapshot'|'delta'|'event'|'heartbeat', version, seq, trace_id, payload}. snapshot=全量状态投影(对应进场景/重连), delta=JSON-Patch 风格增量(对应 dbase 单键变更), event=一次性通知(对应 tell_object/shout)
- 消息子类路由契约（沿用 receive_message 语义）: Frame.payload.subclass ∈ {channel:<name>, outdoor, weather, vision, tell_object, tell_room, say, shout, write, system}. 网关/前端按 subclass 过滤渲染，对应现有 channel: 订阅检查、outdoor: 室内过滤、block_msg/* 阻断
- REST 认证接口（FastAPI）: POST /auth/login{login_name,password,client_caps}->{access_token,refresh_token,character_id,world}; POST /auth/refresh{refresh_token}->{access_token}; POST /auth/logout; GET /auth/session/{session_id}->SessionRecord
- WS 握手协议: 连接 URL 携 ?token=<JWT>&session_id=<可选,重连时>&client_caps=<编码能力/渲染层>. 网关验签->绑会话->回复 HelloFrame{server_version, heartbeat_interval, last_seq, snapshot?}
- 重连对齐协议: 客户端发 ResumeFrame{session_id,last_seq}->服务端校验 SUSPENDED 窗口->若 last_seq+1 在 ring 内则连发 delta 至 current_seq 再发 ResumeAck{current_seq}; 若缺序超容量则发 snapshot 再发 ResumeAck
- 心跳协议: 双层--WS Ping/Pong(20s, 传输层探活) + 应用 HeartbeatFrame{seq}(携带服务端 current_seq, 客户端回 AckFrame{ack_seq}); 超过 IDLE_TIMEOUT=1200s 无业务消息触发挂起(沿用现有发呆踢出语义)
- 事件总线跨节点消息: 跨分片 tell/频道/房间广播经 Redis Pub/Sub 或 NATS subject='world.<world>.channel.<name>' / 'world.<world>.room.<room_id>'. 网关订阅所属分片相关 subject 转发到对应 WS 会话(替代 channeld 的 filter_array(users()) 全量遍历)
- Telnet 适配器渲染接缝: TelnetAdapter.render(Frame)->str(经 message_vision 代词替换 + ANSI 颜色 + UTF-8/可选 Big5)；TelnetAdapter.parse(line)->CommandFrame（保留 !/!N 历史与 alias 展开）
- 能力令牌接口（替代 force_me 的 ROOT_UID 后门）: POST /privileged/action{action, target, reason}->审计日志+结果; 网关校验 token.capabilities 含所需能力，无则 403 并记审计

## 数据模型
会话层三类核心数据，均外部化以支持网关无状态与分布式：

1) SessionRecord（Redis Hash，key=session:{session_id}，TTL=挂起窗口+缓冲期）:
   - session_id: UUID（替代 login 对象的隐式身份）
   - account_id / character_id / world: 角色定位（替代 login.c 的 query("id") 与 user.c 的 geteuid）
   - shard_node: 当前所属游戏分片节点（替代单进程寻址）
   - state: CONNECTING|AUTHENTICATED|ACTIVE|SUSPENDED|RESUMING|CLOSED
   - last_seq: 服务端已发的最大序号（断线对齐基准，对应 receive 的顺序）
   - last_client_seq: 客户端已确认的序号（用于 ring buffer 裁剪）
   - netdead: bool + suspended_at: timestamp（对应 user.c set_temp("netdead")）
   - resume_deadline: 挂起超时点（沿用 NET_DEAD_TIMEOUT=900s 语义，超时触发 quit 落盘）
   - ip / client_caps: 客户端能力（替代 terminal_type 单字符串）

2) TokenClaims（JWT payload，RS256 签名）:
   - sub=character_id, account_id, world, wiz_level（替代 wizhood 路径表）
   - session_id（绑定会话，吊销走黑名单）
   - exp(5-15min access) / refresh_exp(7d)
   - capabilities: [read_world, say, move_self, ...]（替代 valid_cmd 的 ACL，能力令牌化）

3) DeltaRingBuffer（Redis ZSet，key=session:delta:{session_id}，member=序列化的 DeltaFrame，score=seq，TTL=挂起窗口）:
   - 按 seq 有序存放最近 N 条 delta（容量配置，缺序号超容量则退化为全量 snapshot）
   - 对应 feature/message.c 的 msg_buffer(MAX_MSG_BUFFER=500) 但语义升级：从'输入态暂存回放'变为'断线增量重放'

4) SnapshotProjection（PostgreSQL + Redis 缓存）:
   - character 的 dbase 快照（持久层，对应 save_object .o 的结构化版）
   - tmp_dbase 作为易失 delta 不持久化，重连时从 snapshot + ring 重建
   - 分片键沿用 data/user/%c/ 的首字母思想，作为 DB 索引或对象存储前缀

5) AuthAccount（PostgreSQL）:
   - account_id, login_name, password_hash(argon2),wiz_level,status, created_at, last_login_ip
   - 替代 login.c 的 .o 存档 + securityd 的 wiz_status 内存映射

## 旧->新映射
- master.c::connect() 每连接 new(LOGIN_OB) 创建 login 对象 -> **WSGateway 的 WS 连接回调，为每条连接创建 Session 记录（无游戏对象，仅会话态），状态机 CONNECTING**（解耦：连接对象不再承载游戏逻辑，会话与角色分离）
- clone/user/login.c (F_DBASE+F_SAVE) 登录态对象，receive_message 仅处理 write 类 -> **SessionManager 的 CONNECTING->AUTHENTICATED 状态 + AuthService，登录态不再是一个游戏内对象**（login.c 的 input_to 串行回调链迁为 REST 单次登录）
- adm/daemons/logind.c 的 input_to 状态机：logon->confirm_big5->get_id->get_passwd->confirm_relogin->enter_world/reconnect -> **AuthService POST /auth/login（账号校验+签 JWT）+ SessionManager 重建会话，异步单次完成**（现有 Big5 选择迁为传输层 UTF-8 统一，去掉内联编码选择）
- logind MAX_USERS=230 硬编码并发上限 -> **网关水平扩缩，按 worker 分片连接，无硬上限，仅按资源配额弹性扩容**（config.xkx 的 max users 字段废弃）
- DES crypt(pass,salt) 密码校验 (logind get_passwd / passwd.c) -> **argon2 哈希校验，存于账号库**（存量密码需迁移期双校验：旧 crypt 兼容 + 引导改密升级 argon2）
- logind reconnect(ob,user): exec(user,ob) 转移交互连接 + user.reconnect() -> **ReconnectReconciler：token 校验 + last_seq 对齐 + ring 重放 + 清 netdead/移 user_dump**（exec 的连接转移语义改为会话重绑定到新 WS 连接）
- user.c::net_dead(): set_heart_beat(0) + set_temp(netdead) + call_out(user_dump, NET_DEAD_TIMEOUT=900) -> **SessionManager 转 SUSPENDED + 挂起定时器(900s，沿用) + 通知 Player Actor 进入挂起态(停 tick 但保留世界位置)**（挂起而非销毁，netdead 标志语义保留为前端展示(断线中)）
- user.c::reconnect(): set_heart_beat(1)+清 netdead+remove_call_out(user_dump) -> **Reconciler 对齐完成后 SessionManager 转 ACTIVE + 恢复 Player Actor tick + 清挂起定时器**（一一对应，语义清晰）
- feature/message.c receive_message(msgclass,msg) + 子类 channel:/outdoor:/weather: + block_msg + BIG5 转码 + in_input 缓冲(MAX_MSG_BUFFER=500) -> **ProtocolCodec 出站 Frame（msgclass->type/subtype，block_msg->前端过滤，BIG5->移除统一 UTF-8）+ 输入态缓冲由 WS 客户端 ring 承载**（500 条缓冲下放客户端，服务端不再为每玩家存缓冲）
- simul_efun/message.c message_vision 的 $N/$n/$P/$p 代词替换 + ANSI 颜色码内嵌 -> **代词替换保留为纯函数渲染层，移到 TelnetAdapter（WS 路径发结构化 Frame，前端渲染）；颜色码不出现在 WS 协议**（展示逻辑与传输解耦）
- feature/alias.c process_input: cnt 洪水计数(天雷惩罚) + history(10) + alias 展开 + global_alias -> **网关 RateLimiter 令牌桶(全局限流，明确 429) + 输入预处理中间件(history/alias 保留为 UX，天雷副作用废弃)**（天雷这种'伤害玩家'的副作用式限流不健壮，改明确限流反馈）
- command.c command_hook: 出口捷径->命令->emote->channel 回退链 -> **网关入站 Frame->解析 Command->Player Actor 路由，回退链作为 Player Actor 内有序中间件管线(非网关职责)**（网关只解协议与路由，命令分发属游戏逻辑层）
- query_ip_name(ob) IP 追踪 + REGBAN_D IP 白名单/封禁 -> **AuthService 的 IP gate 中间件 + 网关连接级 IP 记录到 session，封禁表存 Redis**（IP 限制从进程内迁为可共享的网关前置闸）
- data/login/%c/%id.o + data/user/%c/%id.o 按首字母分片存档 -> **账号与角色存 PostgreSQL，session 存 Redis；首字母分片思想保留为 DB 索引或对象存储前缀**（.o 全量覆盖快照废弃，迁为事件溯源+快照）
- portbind (端口绑定封装) + config.xkx port 8888 -> **WSGateway 直接监听(可前置 Nginx/LB 做 TLS+负载均衡)**（单端口单进程替换为多 worker+LB）
- feature/command.c force_me 仅 ROOT_UID 后门授权 -> **显式特权动作 API + 审计日志，能力令牌替代 previous_object 隐式信任链**（隐式信任链在分布式异步下无法传递，必须显式化）
- login.c terminal_type(term) 设置终端类型 -> **WS 握手阶段协商客户端能力(capabilities: 渲染层/字符集/扩展)替代单一终端类型字符串**（WS 双向能力协商比 telnet 单向 terminal_type 更灵活）

## 分布式扩展策略
分布式扩展采用'无状态网关 + 会话路由表 + 消息总线扇出'三件套：

1) 网关无状态水平扩展：WSGateway 多实例（SO_REUSEPORT 多 worker + 多机），任一实例崩溃不影响其他；会话状态全在 Redis，网关重启后从路由表恢复 session->node 映射。负载均衡前置（Nginx/云 LB）做 TLS 卸载与连接分发。单机承载目标数万 WS 长连接。

2) 会话粘附与跨分片迁移：Redis 路由表 session:{session_id}->{shard_node, character_id, world}。玩家进入新 region 触发游戏分片 handoff（冻结->快照->目标分片 restore->源分片注销），同时原子更新路由表的 shard_node 指向。WS 连接保持不动（仍粘附原网关），仅游戏消息转发目标变更；玩家 WS 断线重连时按路由表定位正确分片。组队迁移按队形批量 handoff 保持顺序。

3) 跨节点消息扇出：替代 channeld 的 filter_array(users()) 全量遍历。频道/房间/tell 等广播经消息总线（Redis Pub/Sub 或 NATS JetStream）按 subject 路由：world.<world>.channel.<name>、world.<world>.room.<room_id>。各网关实例订阅其上有会话的相关 subject，收到消息后投递到对应 WS 连接。按题材分世界 shard（武侠/大航海/书院各一）做租户隔离。

4) 限流状态共享：per-IP/per-session 令牌桶与消息配额计数器存 Redis（原子操作），多网关实例共享同一限流视角，避免单进程视角的限流盲区。

5) 容量估算：单分片设计上限 5000-8000 并发（Python async+uvloop），1 万在线玩家约需 5-8 分片；网关层独立于游戏分片扩缩容，承载连接数与游戏负载解耦。

6) 断线重连的分布式一致性：ring buffer 存 Redis 而非网关内存，故玩家重连到任一网关实例都能恢复（不依赖粘附到原网关）；session 状态机在 Redis 用乐观锁/CAS 推进，防止并发重连导致状态错乱。

## 技术选型
- aiohttp + uvloop：WS 长连接服务器与事件循环，SO_REUSEPORT 多 worker 水平扩展，单机数万连接
- FastAPI：REST/JWT 签发与 UGC/管理 API，与 WS 网关独立扩缩容
- PyJWT(RS256 非对称签名)：access 5-15min + refresh 7d，token 载 character_id/world/wiz_level
- argon2-cffi：替代单轮 DES crypt() 的现代密码哈希，抗 GPU 暴力破解
- msgpack-python：默认二进制协议，紧凑高效；JSON 兜底 UGC 动态内容；Protobuf 仅在高频战斗压测瓶颈后局部引入（不过早引入）
- Redis：会话路由表(session_id->node)、per-session ring buffer(ZSet 按 seq)、限流令牌桶(原子 INCRBY)、JWT 黑名单、跨节点 Pub/Sub 扇出
- NATS JetStream（可选，达规模时）：跨分片消息扇出与持久化，subject='world.<world>.channel.<name>' 路由
- PostgreSQL：账号表(替代 login .o 存档)、结构化审计日志(替代 log_file 的 PLAYER_DEATH/READ_LOG/WRITE_LOG)、事件溯源 append-only 表
- telnetlib3 或自研 asyncio telnet handler：过渡期 Telnet 兼容适配器，以 receive_message 为渲染接缝
- pydantic + dataclass：Frame 与 SessionRecord 的类型化 schema（杜绝现有 query('xxx') 字符串键无检查问题）
- Python 3.11+ asyncio：统一异步运行时，与 Actor 运行时/ECS 共享事件循环

## 风险
- 现有密码用单轮 DES crypt() 存储，迁移到 argon2 需双校验过渡期：旧账号首次登录用 crypt 兼容校验后引导改密升级 argon2，存在兼容窗口与社工风险；过渡期 AuthService 需同时支持两套校验
- 断线重连的 ring buffer 容量与全量 snapshot 退化边界需压测确定：ring 太小则频繁退化为 snapshot（带宽与算力浪费），太大则 Redis 内存膨胀；建议按典型战斗 tick 频率(2s)与挂起窗口(900s)估算峰值 delta 条数定容量
- Telnet 适配器的 ANSI 渲染与 message_vision 代词替换逻辑需从 LPC 精确移植，任何代词($N/$n/$P/$p)/颜色码/性别处理的细微差异都会导致 telnet 老玩家体验割裂；建议将渲染逻辑抽为纯函数并用现有 LPC 文本做回归测试集
- WS 与 Telnet 双协议期，同一 Player Actor 的出站消息需同时走 Frame(TelnetAdapter 渲染)与结构化 Frame(WS)，两条路径的语义对齐是持续维护负担；建议 receive_message 接缝层严格统一输出抽象事件，适配器各自渲染
- 跨分片频道/广播依赖 Redis Pub/Sub 或 NATS，消息总线本身的高可用与延迟成为新单点；频道消息丢失或乱序会影响玩家体验，需监控总线积压与序号连续性
- JWT 刷新令牌(7d)与 access(5-15min) 的吊销依赖 Redis 黑名单，黑名单不可用时会话无法吊销（安全窗口）；需对 Redis 做主从与降级策略，吊销失败时 fail-open 还是 fail-closed 需明确（安全场景建议 fail-closed 但影响可用性）
- 现有 login.c 的 input_to 串行回调链含大量分支逻辑（重登录确认、自杀检查、机器人检测房间跳转等），迁为 REST 单次登录时易遗漏边界条件；需逐分支梳理 logind.c 的 get_id/get_passwd/confirm_relogin 全部分支做映射清单
- uvloop 在某些 Linux 环境与特定 Python 版本存在兼容性问题，且 uvloop 维护活跃度需关注；需准备退回默认 asyncio 事件循环的降级路径
- 增量 JSON-Patch 的客户端实现复杂度：前端需正确应用 patch 并处理序号乱序/重复，patch 算法实现不当会导致状态漂移；建议优先用成熟的 JSON-Patch 库并加幂等校验
- WS 网关水平扩展后，会话粘附路由表(session_id->node)的更新与一致性是关键：玩家迁移分片时路由表需原子更新，否则消息会发到旧节点；需分布式锁或原子的 Redis 事务保证路由切换一致性
- 过渡期若用桥接器旁路连 MudOS 旁车，LPC 与 Python 双系统状态同步会引入复杂性与潜在不一致；建议桥接器仅用于只读旁路或单一子系统验证，避免长期双写状态

## 🔍 对抗验证

**裁定**：sound 并简述理由：核心架构方向正确且站得住——把 master.connect+login.c/logind.c input_to 回调链+F_MESSAGE+net_dead/reconnect 拆为"无状态网关 + Redis 外部化会话 + JWT/argon2 现代认证 + ring buffer 增量重连 + receive_message 渲染接缝"是合理的，与源码核对后大部分映射成立（MAX_USERS 硬编码、crypt() DES、net_dead/reconnect 链、channeld filter_array(users())、force_me ROOT_UID 后门、exec 连接转移、message_vision 代词替换、MAX_MSG_BUFFER=500、NET_DEAD_TIMEOUT=900/IDLE_TIMEOUT=1200 均属实）。但方案对实际 LPC 代码存在多处事实性失真与一处核心语义自相矛盾（netdead 是否继续仿真），并漏掉了建号流程与登录封禁闸的真实分支，这些会在迁移期造成可执行的缺陷。属于"结构 sound、映射与保真度需返工修正"，故评 medium 而非直接 sound 通过。

**严重度**：medium

### 问题与修复
- **核心语义自相矛盾，且与源码冲突**
  - 影响：若按 summary 实现，玩家断线后角色继续掉血/中状态/死亡，属破坏性玩法变更；若按映射表实现，则 summary 误导下游实现者。三处必须一致。
  - 修复：明确统一语义：建议沿用现状（netdead 停 tick + remove_all_enemy，即'暂停仿真、退出战斗、保留世界位置'），删除 summary 中'继续被仿真'的表述。若确要改为挂起体继续仿真，须单列为一项玩法平衡变更并经数值评审，不能当作忠实延续。
- **新玩家角色创建流程在映射中整体缺失**
  - 影响：若仅实现 /auth/login，新玩家无法建号；或建号逻辑被错误地塞进登录接口导致耦合。团队自有文档已记录此流程，方案却未对齐，属一致性缺口。
  - 修复：在 currentToNewMapping 中补全整条建号链，并设计独立的多步角色创建 API（REST 多步或 WS 帧序列），与团队已文档化的状态机对齐。建号流程不应塞进单次 /auth/login。
- **BAN_D 与 REGBAN_D 被混淆，会丢失真正的登录封禁**
  - 影响：若仅迁移 REGBAN_D 作登录闸，已登录封禁策略丢失，被 BAN_D 拦截的恶意 IP 可正常登录，安全回退。
  - 修复：双迁移：BAN_D -> AuthService 登录前置 IP 闸；REGBAN_D -> 角色创建前置 IP 闸。两者规则源(banned_sites vs noreg_sites)与触发时机不同，须分别建模。
- **风险清单虚构了不存在的'机器人检测房间跳转'分支**
  - 影响：迁移梳理按虚构分支走会浪费工时并漏掉真实跳转；enter_world 的条件房间选择若丢失，高死亡低经验玩家/未注册玩家落点错乱。
  - 修复：用真实分支重写：列出 death_count>200&&combat_exp<50000 -> /d/death/block.c，registered!=yes -> /d/xiakedao/shatan1 这两个 enter_world 期房间跳转分支，并明确它们属于游戏层而非会话层。
- **dbase/tmp_dbase 类比失真，增量 JSON-Patch 被错误归源**
  - 影响：按错误类比实现，会把 tmp_dbase 的 link_ob/netdead/敌人列表等运行态当作 dbase 增量去 patch 持久状态，导致状态污染；或反过来把 dbase 当全量、tmp_dbase 当 diff 去做 JSON-Patch，模型错位。
  - 修复：修正类比：dbase->持久状态(snapshot 的数据源)，tmp_dbase->易失运行态(不进 snapshot/delta)。明确 DeltaFrame 的 JSON-Patch 是新引入机制、无 LPC 祖先；snapshot 序列化时须显式跳过 tmp_dbase 或单独投影为 runtime state，而非当作 dbase 的增量。
- **dbase 函数值与 default_ob 继承默认值无法被 snapshot 捕获**
  - 影响：角色投影不完整，前端/重连看到的属性缺函数计算值与继承默认值，引发静默状态漂移与玩家可见的属性丢失（如某些 query() 动态属性）。
  - 修复：snapshot 投影须显式枚举可序列化键并标记/求值函数值属性；default_ob 默认值在迁移期需先 flatten 进角色 snapshot，或在前端/Actor 层重建继承默认链。需做一次 dbase 全键扫描分类。
- **config.xkx 驱动级 maximum users:150 被遗漏，误以 LPC 230 为生效上限**
  - 影响：容量基线误判：以为旧系统上限 230，实际 150，迁移后的容量对照与扩缩容阈值设定会偏离。
  - 修复：理清并记录两套上限：驱动级 max users(150) 与 LPC MAX_USERS(230)。新系统用资源配额弹性扩缩替代两者，但迁移期须知道真实生效门槛是 150，用于容量基线对照。
- **状态机 CONNECTING 缺少登录超时（对应 LOGIN_TIMEOUT=300s）**
  - 影响：未完成登录的僵尸会话占据 Redis 与路由表，无清理路径；与现有 LOGIN_TIMEOUT=300s 语义脱节。
  - 修复：在状态机补全：CONNECTING 设 300s 超时(沿用 LOGIN_TIMEOUT)转 CLOSED 并清理未完成登录态；与 SUSPENDED(900s)、ACTIVE 发呆(1200s)三套超时分别显式映射。
- **enter_world 的游戏态引导职责在会话/游戏层边界上未澄清**
  - 影响：若会话层实现 enter_world 全部逻辑，会重新把游戏状态耦合进网关，违背'网关无状态纯协议层'的不可妥协边界。
  - 修复：在 enter_world 映射处明确切分：会话层只负责到 AUTHENTICATED/ACTIVE 状态切换；UPDATE_D->login_check、起始房间选择、tell_room、CHANNEL_D sys 广播等归游戏/Actor 层入口钩子，会话层通过事件交棒。
- **频道防刷屏的'自动禁言+公开点名'社会性惩罚被静默降级**
  - 影响：丢失社会性惩罚后，频道刷屏成本降低，刷屏行为可能回升；且若按'忠实迁移'预期验收会不通过。
  - 修复：显式保留该行为或显式废弃并告知产品方：要么在 RateLimiter 里实现'超阈值 -> 禁言 N 分钟 + 系统公告'的等价社会性惩罚，要么明确改为静默 429 并记录这是行为变更。
- **message_vision 三视角分叉渲染未在 Frame 模型表达**
  - 影响：WS 与 telnet 两路输出语义不对齐，telnet 老玩家看到的代词/称谓与 WS 新端不一致，过渡期体验割裂且难验收。
  - 修复：Frame 规范补'按观看者分叉'语义：event/tell_room/vision 类载荷携带 actor/target 引用与原始模板($N/$n/$P/$p 占位串)，由各端(TelnetAdapter 或 WS 前端)按观看者做性别化代词替换。保留 message_vision 纯函数做 telnet 兼容回归测试。
