# 方案设计·前端与状态同步

## 设计概述
核心理念：传输纯语义事件，渲染下沉前端。XKX 现有 receive_message(msgclass,msg) + message_vision 代词替换 + ANSI 颜色码是把"逻辑"和"展示"绑死在服务端的元凶——Web 前端无法复用。新架构以"语义事件流 + 客户端投影渲染"为骨架，把 message_vision 的 $N/$n/$P/$p 代词体系、ANSI 颜色、damage_msg 文案模板全部下移到前端纯函数渲染层，服务端只产结构化事件。

状态同步复用 F_DBASE 路径键模型：dbase(持久层)=snapshot 全量投影，tmp_dbase(瞬时层)+apply 修饰符堆叠=delta 增量补丁。进场景发 snapshot，变更发 JSON-Patch 风格 delta + 单调递增 seq，断线靠 ring buffer 重放。这直接对应现状分析中"dbase=持久 snapshot、tmp_dbase=易失 delta"的映射。

会话层以"断线挂起而非销毁"为核心：复用现有 msg_buffer + write_prompt 暂存回放思想，扩展为带 last_seq 的 ring buffer 重放 + 60-300s 挂起窗口，玩家重连后无缝续接。WS 网关无状态水平扩展，session↔分片路由表存 Redis，跨节点 tell/频道经 Pub/Sub 转发。

渲染双栈过渡：以 receive_message 为渲染缝，Telnet 适配器拼 ANSI、Web 适配器发结构化事件，过渡期可桥接器旁路连 MudOS 旁车，降低迁移风险。前端保留可选终端模式(xterm.js)与图形模式(场景图可视化)，文本渲染是基线、图形是渐进增强。

## 组件
- **WebSocket Gateway (WS网关)**：无状态长连接接入层。承担 WS 握手、JWT 校验(argon2 验密替代 crypt)、会话建立、令牌桶限流(替代 process_input 对象内 cnt 计数)、origin 校验、每会话消息配额。不做游戏逻辑，只把上行消息转发给玩家所在分片的 Player Actor，下行事件透传给客户端。支持 SO_REUSEPORT 多 worker 水平扩展，单机数万连接。（技术：Python: aiohttp 或 websockets + uvloop; JWT(RS256, access 5-15min/refresh 7d); Redis 会话路由表; 可选 uvloop 替代默认事件循环）
- **Session Manager (会话管理器)**：维护 session_id->{node,user_id,character_id,world,last_seq,ts} 路由表(存 Redis)。负责断线挂起(不销毁会话，保留 Player Actor 与挂起窗口 60-300s)、重连定位(按 session_id 路由到原分片)、跨分片 handoff 协调(玩家 go 跨区时快照迁移)。替代 logind.c 的 reconnect(ob,user) + netdead 标记。（技术：Python asyncio; Redis(路由表 + TTL 挂起窗口); session_id 入 token 便于跨分片）
- **Event Codec (事件编解码器)**：协议帧统一封装 {type,version,seq,ts,payload}。默认 MsgPack(高频战斗)，UGC/DSL 动态内容 JSON 兜底，仅高频战斗压测瓶颈后局部 Protobuf。统一版本号 version 支持协议演进与客户端兼容降级。替代 simul_efun 的隐式 write/printf 经 message() 路由。（技术：msgpack-python(默认); orjson(JSON 高性能); Pydantic v2(事件 schema 校验与版本); 仅瓶颈处 Protobuf）
- **State Sync Engine (状态同步引擎)**：核心：dbase 路径键即状态投影。进场景/重连发 snapshot(全量状态树)，变更发 JSON-Patch 风格 delta + 单调递增 seq。dbase=持久 snapshot、tmp_dbase=易失 delta。apply/ 修饰符堆叠作为 ModifierStack 增量。客户端按 seq 顺序应用补丁，支持 ring buffer 重放。替代 save_object .o 全量快照式同步。（技术：Python: 自研 snapshot/delta 编码器; JSON-Patch(RFC 6902 风格); 单调 seq + ring buffer; Pydantic 校验）
- **Projection Store (投影存储/前端状态树)**：前端镜像的服务端状态树。按 dbase 斜杠路径层级组织(family/master_name 等嵌套)。区分 persistent 投影(角色/背包/技能)与 ephemeral 投影(战斗态/位置/在线状态)。响应式更新驱动 UI 重渲染。是 dbase mapping + default_ob 原型回退在前端的等价物。（技术：前端: Zustand 或 Jotai(细粒度响应式); Immer(不可变更新); TypeScript 类型化 schema）
- **Render Adapter (渲染适配器/双栈缝)**：以 receive_message(msgclass,msg) 为渲染缝。Web 适配器把语义事件转为结构化 {type,payload} 下发；Telnet 适配器拼 ANSI 颜色码+文本。过渡期双栈并存，同一 Player Actor 可同时挂 WS 与 Telnet 适配器，降低迁移风险。msgclass 的 channel:/outdoor:/weather: 子类前缀在此转为订阅谓词。（技术：Python 适配器抽象; Telnet 端复用 languaged 仅做编码; Web 端纯语义事件）
- **Message Render Layer (消息渲染层/前端)**：前端纯函数渲染层。保留 message_vision 的代词替换体系 $N/$n/$P/$p(主格/宾格/自称/他称)做三视角渲染(我/你/旁观各看不同文案)。ANSI 颜色码 HIR/HIC 映射为主题样式 token。damage_msg/eff_status_msg/status_msg 等文案数组从 combatd 抽离为前端文案模板池(支持 i18n/换肤/A-B 测试)。（技术：前端: TypeScript 纯函数; React 组件; i18n(vue-i18n 或 react-i18next); CSS-in-JS 主题 token）
- **Reconnect Buffer (重连缓冲)**：复用现有 msg_buffer(MAX_MSG_BUFFER=500)+write_prompt 思想。服务端为每个会话维护 ring buffer 缓存最近 N 条 delta(按 seq)，断线重连时客户端上报 last_seq，服务端重放 last_seq+1..current 的 delta。挂起窗口内事件不丢，超时才钝化落盘。输入态/编辑态暂存消息回放保障玩家不丢上下文。（技术：Python: collections.deque ring buffer; Redis 挂起态; 客户端指数退避重连(1s->2s->4s->30s 上限)）
- **Input Pipeline (输入管线)**：command_hook god-function 拆为显式有序中间件管线：解析->预处理(alias/history/限流/编码)->路由->授权->执行->后处理(度量/审计)。动词->handler->Action(ctx)->result 统一命令契约。个人 alias + 全局 alias 合并为单一分层 alias 服务。洪水控制下沉网关层令牌桶。（技术：Python: 中间件模式(类 Starlette); 命令自注册元数据表(DB 持久化); 策略引擎授权）
- **Subscription & AOI Router (订阅与兴趣区域路由)**：channel:/outdoor:/weather: 子类过滤演化为发布订阅 topic 路由。频道改 Pub/Sub 替代 filter_array(users()) 全量遍历。AOI 默认 1 跳房间图邻域(匹配 tell_room/move 广播语义)，跨分片经边界房间代理转发。玩家进/离房间触发订阅增减，按区域元数据路由谓词。（技术：Python: NATS JetStream(扇出) / Redis Pub/Sub; 谓词路由; 边界房间代理）
- **Heartbeat & Liveness (心跳与存活检测)**：WS Ping/Pong 20s 探活 + 应用层心跳携带 last_seq。挂起窗口 60-300s 内断线不销毁会话。复用 heart_beat 2s 为确定性 tick 作为事件批边界(仿真 worker 推进、网关转发)，断线挂起而非销毁。query_idle>120 的'发呆中'状态由心跳驱动。（技术：Python asyncio 定时; WS 协议 Ping/Pong; 应用层心跳帧）
- **Display Projection (展示投影/前端 DisplayName)**：name.c short() 的 12+ 状态源聚合(colorname/nickname/title/apply 修饰符/pending 打坐/ghost/netdead/input/edit/idle/disable_type)拆为前端统一聚合的 DisplayName 投影组件。服务端发原始状态片段，前端按展示规则组合，避免服务端为每个视角重算展示文案。（技术：前端: React 组件; 状态片段响应式订阅; 规则化展示配置）

## 关键接口/事件
- EventFrame(协议帧): { type: string, version: int, seq: int64(单调递增), ts: int64, payload: object } -- 所有上下行统一封装; version 支持协议演进与客户端降级
- 下行 Snapshot: { type:'snapshot', seq:N, root: '<character_id>', state: {vitals:{qi,jing,jingli,max_qi,...}, attrs:{str,int,...}, position:{room_id,region}, equipment:{...}, skills:{...} } } -- 进场景/重连时全量发送
- 下行 Delta: { type:'delta', seq:N, base_seq:M, ops:[ {op:'replace',path:'/vitals/qi',value:80}, {op:'add',path:'/modifier/apply/damage',value:[...]} ] } -- JSON-Patch 风格; base_seq 指明基于哪个 snapshot
- 下行 Event(语义事件): { type:'event', event:'combat_round', seq:N, payload:{ attacker, victim, action, damage, hit, msg_key:'damage_slash_heavy', ctx:{...} } } -- 文案模板键(msg_key)而非渲染后文本，前端查模板池渲染
- 下行 Render Message(渲染缝): receive_message(msgclass, msg_event) -- msgclass 保留 channel:/outdoor:/weather: 子类语义转为订阅 topic; msg 为结构化事件而非 ANSI 字符串
- 上行 Command: { type:'command', verb:'go', args:['north'], seq:N, ctx:{session_id,character_id} } -- 替代 command_hook; 经 Input Pipeline 中间件链
- 上行 Heartbeat: { type:'ping', last_seq:M } -- 客户端上报已确认 seq; 服务端回复 {type:'pong', server_seq:N, replay:[...]}
- 上行 Reconnect: { type:'reconnect', session_id, token, last_seq:M } -- Session Manager 校验后路由到原分片，重放 M+1..current delta
- 下行 ReconnectResult: { type:'reconnected', character_id, world, current_seq:N, replay_buffer:[delta...], state_ok:true } -- 含挂起期间累积的事件
- 订阅协议 Subscribe: { type:'subscribe', topics:[ {topic:'channel:chat'}, {topic:'room:yangzhou_inn'}, {topic:'region:hangzhou'} ], predicates:{outdoors:true} } -- 替代 query('channels') + filter_array
- 发布协议 Publish(服务端内部): Publish(topic, event, filter_fn) -- 频道/房间广播经 Pub/Sub 路由替代 users() 全量遍历
- 代词渲染契约: render_message(template_key, actors:{me:{name,gender,id}, you:{...}, room:[...]}) -> {to_me, to_you, to_room} -- 三视角，前端纯函数; 模板用 $N/$n/$P/$p 占位
- 状态键契约: state path 沿用 dbase 斜杠路径 'family/master_name'、'vitals/qi'、'apply/damage' -- 前端 Projection Store 按此寻址
- 挂起协议 Suspend: { type:'suspended', session_id, resume_window:300, grace_until:ts } -- 断线后 Player Actor 保留，超时才钝化

## 数据模型
状态同步模型沿用 F_DBASE 的双层分离：dbase(持久) = Snapshot 树，tmp_dbase(易失) = Delta 补丁流。

服务端权威状态树(StateTree)：按 dbase 斜杠路径层级组织，如 /vitals/qi、/attrs/str、/family/master_name、/position/room_id、/equipment/weapon。default_ob 原型回退映射为前端 Projection Store 的模板继承(查不到则回退 prefab 默认值)。apply/ 命名空间修饰符堆叠映射为 ModifierStack(装备/buff 写增量，getter 聚合基础值+修饰符)，前端按此聚合展示值。

事件流(EventStream)：append-only，按 seq 单调。权威存储 PostgreSQL(events 表 append-only + snapshots 表周期快照)；热事件 Redis Stream；扇出 NATS JetStream。投影(背包/排行/银行流水/战报)独立部署为读模型，热更新=重建投影不触写侧。战斗不逐 tick 事件(会爆炸)，用回合聚合 CombatRoundResolved + 连续态周期快照(每 30s 或关键节点)。

前端投影状态(ProjectionStore)：persistent 投影(角色/背包/技能，进场景全量 snapshot) + ephemeral 投影(战斗态/位置/在线状态，仅 delta)。Immer 不可变更新，Zustand/Jotai 细粒度订阅，仅变化的路径触发重渲染。

会话记录(SessionRecord)：{session_id, node, user_id, character_id, world, last_seq, ts, suspend_state} 存 Redis，TTL=挂起窗口。断线不落盘，超时才触发 actor 钝化(写 snapshots 表 + 释放内存)。

重连缓冲(ReplayBuffer)：服务端每会话 ring buffer(deque maxlen=N)缓存最近 delta，重连按 last_seq 重放。挂起窗口内事件不丢。

文案模板池(MessageTemplates)：damage_msg/eff_status_msg/status_msg/winner_msg 等从 combatd 抽离为 YAML/JSON 模板，按 msg_key 索引，支持 i18n 多语言与多题材换肤。代词占位 $N/$n/$P/$p 由前端渲染时替换。

## 旧->新映射
- receive_message(msgclass, msg) 经 efun receive() 直接写 Telnet，ANSI 颜色码与 BIG5/GB 编码绑死 -> **Render Adapter 渲染缝：Web 适配器发结构化 {type,payload} 语义事件，Telnet 适配器拼 ANSI；编码统一 UTF-8 下沉传输层**（现状分析已指明 receive_message 是渲染缝，此处固化为双栈适配器接口）
- message_vision(msg, me, you) 的 $N/$n/$P/$p 代词替换 + 三视角渲染(me/you/room) -> **前端 MessageRenderLayer 纯函数 render_message(template_key, actors) -> {to_me,to_you,to_room}**（代词体系是中文武侠文本表现力核心，保留为纯渲染层解耦传输）
- ANSI 颜色码 HIR/HIC/HIM 等硬编码进文案字符串 -> **主题样式 token + CSS-in-JS，传输纯语义事件(msg_key + ctx)，颜色由前端主题决定**（颜色/编码移至 Web 前端渲染层，支持换肤/A-B 测试）
- dbase mapping 全量内存 + save_object .o 覆盖快照同步 -> **StateTree snapshot + JSON-Patch delta + 单调 seq + ring buffer 重放**（全量覆盖式同步改为增量补丁，dbase=snapshot、tmp_dbase=delta）
- tmp_dbase apply/ 修饰符堆叠(装备 apply/name, apply/short, apply/long) -> **ModifierStack 增量组件，前端按基础值+修饰符聚合展示值**（ECS 组件聚合雏形，保留为 Buff/Equipment 组件基础）
- default_ob 原型回退(query 查不到回退蓝图) -> **前端 Projection Store 模板继承 + Prefab 资源回退**（blueprint/clone 原型模式映射为 prefab + 实例差异）
- filter_array(users(), 'filter_listener') 频道广播 O(在线人数) -> **Pub/Sub topic 路由，按订阅分发，无全量遍历**（发布订阅替代 users() 全量扫描）
- msgclass 子类前缀 channel:/outdoor:/weather: 决定可见性 -> **Subscription topic + 谓词(outdoors/region/room) 路由，AOI 1 跳邻域**（消息子类订阅过滤思想演进为基于区域元数据的路由谓词）
- msg_buffer(MAX_MSG_BUFFER=500) + write_prompt 输入态暂存回放 -> **ReconnectBuffer ring buffer + last_seq 重放 delta，挂起窗口 60-300s**（输入态消息缓冲保障不丢上下文，扩展为断线重放）
- logind.c reconnect(ob,user) + exec 转移连接 + netdead 标记 -> **Session Manager 断线挂起(不销毁 Player Actor) + 重连定位 + 状态恢复**（断线挂起而非销毁会话）
- short() 12+ 状态源内联聚合(colorname/nick/title/apply/pending打坐/ghost/netdead/input/edit/idle/disable) -> **服务端发原始状态片段，前端 DisplayName 投影组件统一聚合展示**（拆分 name.c short 的内联展示逻辑）
- combatd 内嵌 damage_msg/eff_status_msg/status_msg/winner_msg 文案数组 -> **前端文案模板池(msg_key 索引，YAML/JSON，支持 i18n/换肤/多题材)**（战斗逻辑与文案解耦，国际化可行）
- process_input 单函数承载洪水控制+历史+Big5转码+alias 五职责 -> **Input Pipeline 显式中间件管线：解析->预处理(alias/history/限流/编码)->路由->授权->执行**（command_hook god-function 拆解）
- 对象内 cnt 洪水计数器 + 天雷惩罚 -> **网关层全局令牌桶限流 + per-session 消息配额**（分布式限流可共享状态）
- heart_beat 2s 单进程全局 tick 驱动战斗/治疗/条件/AI -> **确定性 tick 作为事件批边界，仿真 worker 推进、网关转发，断线挂起而非销毁**（保留 2s tick 思想但解耦为事件流）
- MAX_USERS=230 硬编码并发上限 -> **弹性分片容量(单分片 5000-8000 并发)，按 worker 分片连接**（WebSocket+分布式弹性扩缩）
- Big5->GB 编码转换内嵌每条输入/输出路径(languaged->toBig5) -> **传输层统一 UTF-8，编码无关内核**（编码下移 WebSocket 传输层）
- query('channels') + me->set('channels') 频道订阅列表存 dbase -> **Subscription 协议显式 subscribe/unsubscribe topic**（频道订阅显式化）
- telnet message(class,msg,target) 三参数原子原语 -> **Publish(topic, event, filter_fn) + 订阅者投影**（消息原语演化为发布订阅）

## 分布式扩展策略
连接层与游戏层分离扩展：无状态 WS 网关水平扩展(SO_REUSEPORT 多 worker)，按 session 粘附所在分片；游戏分片有状态持有房间/NPC/玩家/AOI。Redis 存 session_id->{node,user,character,world} 路由表支持断线重连定位与跨节点转发。

跨分片通信：频道/房间广播经 Redis Pub/Sub 或 NATS JetStream 路由，替代 filter_array(users()) 全量遍历。跨节点 tell/频道/房间广播经消息总线转发。按题材分世界 shard(武侠/大航海/书院各一)，互不干扰。

AOI 分片：默认 1 跳房间图邻域匹配现有 tell_room/move 广播语义。边界房间(区域共同体割边上)建立跨分片事件代理，AOI 跨分片订阅通过边界房间转发，避免每条消息走全局总线。玩家进入新 region 时迁移 Player Actor 到目标节点(先激活后注销，消息日志重放保证不丢)；同房间战斗参与者保持同节点亲和，COMBAT_D 实例化为 region/战斗级 Coordinator Actor 串行裁决回合，避免分布式事务。

单分片容量上限设计 5000-8000 并发(10 万房/20 万对象/Python async+uvloop)，1 万在线玩家需 5-8 分片。热区(city/扬州)做内部微分区，切房间簇成子分片并设城市入口路由。

投影独立部署：背包/排行/银行流水/战报投影作为独立读模型服务，与仿真 worker 解耦，可按读负载独立扩缩容。前端走 WebSocket 订阅投影而非直连仿真。

确定性随机：每场战斗/每个 region 分配独立 seeded RNG，回合结果写 append-only 事件日志，支持回放与反作弊审计——分布式下调试与公平性前提。

## 技术选型
- WS 网关: aiohttp 或 websockets + uvloop(SO_REUSEPORT 多 worker，单机数万连接)；不直接用 Akka/Orleans(非 Python 增加跨语言成本)
- REST/JWT/UGC API: FastAPI(独立扩缩容)；argon2 验密替代 crypt()；JWT RS256(access 5-15min + refresh 7d)
- 消息协议: msgpack-python(默认高频); orjson(JSON 高性能兜底 UGC 动态内容); Pydantic v2(事件 schema 校验与版本); 仅高频战斗压测瓶颈后局部 Protobuf
- 会话路由: Redis(session_id->{node,user,character,world,last_seq} + TTL 挂起窗口 + Pub/Sub 跨节点转发)
- 事件扇出: NATS JetStream(扇出与持久化); 热事件 Redis Stream; 权威存储 PostgreSQL(append-only events + snapshots 表)
- 前端框架: React + TypeScript(Zustand 或 Jotai 细粒度响应式 + Immer 不可变更新)
- 前端渲染: xterm.js(可选终端模式保留 MUD 文本体验); React Flow(场景图可视化); CSS-in-JS 主题 token 替代 ANSI 颜色码
- 前端 i18n: react-i18next(文案模板池多语言)
- 状态同步: 自研 snapshot/delta 编码器 + JSON-Patch(RFC 6902 风格); 单调 seq + collections.deque ring buffer
- 输入管线: 中间件模式(类 Starlette Pipeline); 命令自注册元数据表(DB 持久化)
- 限流: 网关层令牌桶(redis-py + Lua 原子脚本); per-session 消息配额
- 不选 esper/entt-py: 自研针对 MUD 访问模式优化，避免外部依赖风险(沿用现有架构研究结论)

## 风险
- 同步返回值迁移成本极高：LPC 大量依赖 move() 后立即检查 environment() 等同步返回值，改异步事件后需逐处重写，迁移面极大。建议先用 actor 适配器桥接保留同步语义，逐子系统替换而非全量重写
- 代词三视角渲染的上下文丢失：message_vision 的 me/you/room 三视角依赖运行时上下文(this_player/environment)，事件化后需在 payload 显式携带 actors 上下文，遗漏会导致渲染错乱。需建立 actors schema 强约束
- msgclass 子类语义迁移不完整：channel:/outdoor:/weather: 之外可能存在散落的子类约定未被识别，迁移时静默丢消息。建议全库扫描 receive_message 调用的 msgclass 取值建立清单
- 战斗逐 tick 事件爆炸：若误把每 tick 伤害都发事件，seq 与带宽爆炸。必须用回合聚合 CombatRoundResolved + 周期快照，连续态(qi/jing)用快照而非事件
- 断线重放的 seq 一致性：ring buffer 满后旧 delta 被驱逐，若客户端 last_seq 过旧则无法重放，需降级为全量 snapshot 重发。需明确重放窗口与降级策略
- 双栈过渡期的状态分裂：Telnet 与 Web 适配器并存时，若两者状态投影不一致(如 ANSI 颜色 vs 语义事件)，玩家体验割裂。需以语义事件为单一真相，Telnet 由其派生
- 前端文案模板池与游戏逻辑漂移：文案抽离后，若 combatd 仍硬编码文案字符串未同步删除，会出现双份维护。需建立 msg_key 的单一来源约束与 lint 检查
- AOI 跨分片订阅的延迟与一致性：边界房间跨分片事件代理若同步协调不当，会出现玩家'看到'已离开房间的人或漏看进入的人。需明确边界房间为单一归属 + 事件转发而非双写
- 协议版本演进的兼容性：version 字段虽支持降级，但若客户端旧版本不识别新 event type，静默丢弃会导致状态不同步。需客户端对未知 type 显式上报 unsupported 并降级
- UGC DSL 动态内容与类型化 schema 的张力：UGC 场景的事件 schema 动态多变，Pydantic 强类型校验可能过度约束，需在 schema 严格性与 UGC 灵活性间平衡(可选 schema + 运行时校验)
- Python 单进程 GIL 在高频战斗渲染下成为瓶颈：WS 网关虽多 worker，但单 Player Actor 的事件序列化/渲染若在主循环，高并发下阻塞。需把序列化与渲染计算卸载到线程池或独立 worker

## 🔍 对抗验证

**裁定**：risky — 核心方向正确(语义事件流+客户端投影渲染精准命中 receive_message/message_vision/ANSI 编码绑死逻辑与展示这一真痛点,且与团队自有架构文档 04/06/07 高度一致),但数据模型映射存在概念性错误(tmp_dbase≠delta)、渲染层迁移规模被严重低估(1626 处 message_vision)、战斗'回合聚合'前提不成立(实为 tick 驱动)、Python 单分片容量基准未验证。可执行,但需在动工前修正上述概念性问题并补基准测试,否则会沿错误模型投入大量人力。

**严重度**：high

### 问题与修复
- **核心数据模型映射概念错误:方案反复声称'dbase=snapshot、tmp_dbase=delta'是直接对应。但代码证实 dbase 与 tmp_dbase 是两个并行命名空间(分别经 query() / query_temp() 访问),tmp_dbase 不持有针对 dbase 的差分补丁,而是独立的瞬时键(如 apply/attack、pending/exercise、netdead)。apply/ 修饰符是在渲染/查询时叠加,不是 diff 流。把 tmp_dbase 当作'delta+seq'是新模型,不是忠实映射。**
  - 影响：若实现者按字面把 tmp_dbase 流式化为 patch,会构建出一个与真实状态变更方式不匹配的 delta 引擎。补丁生成逻辑需要从零设计,不能复用 tmp_dbase 结构。这是数据模型层的方向性错误。
  - 修复：放弃'tmp_dbase=delta'的映射话术。明确:dbase 与 tmp_dbase 是并行命名空间,不是 base+diff。StateTree+delta 应作为全新设计:服务端维护单一权威状态树(整合 dbase 与 tmp_dbase 的有效值),变更通过对比前后两次权威树生成 JSON-Patch ops,而非把 tmp_dbase 当作 delta 来源。文档需写清'delta 引擎是新建物,与 LPC 现状无直接对应'。
- **dbase 的函数值属性会击穿 snapshot 模型:dbase.c 第 51 行 query() 返回 evaluate(data, this_object()),实测 104+ 文件 set 了闭包值(如 set("long", (: long_desc :))、set("actions", (: call_other, WEAPON_D, "query_action" :)))。方案把 dbase 当纯数据快照树,但值可以是绑定 this_object() 的可执行闭包,无法跨网络序列化。**
  - 影响：snapshot 序列化会在闭包键上静默丢弃或崩溃。战斗 actions 正是要事件化的核心,却是最典型的闭包值场景,形成方案自身目标的内部矛盾。
  - 修复：建立 dbase 键的分类台账:data-valued(可快照)vs function-valued(服务端求值后下发)。战斗 actions 这类闭包必须在服务端 resolve 成动作描述符再进事件,绝不进 snapshot。对 query() 的 evaluate 路径做统一拦截,标记哪些键是 dynamic,快照时显式跳过或求值。
- **渲染层迁移规模被严重低估:message_vision 用于 1626 个文件、message() efun 1070 处、tell_room 317 处。每个 message_vision 调用点都把 $N/$n/$P/$p 模板内联在游戏逻辑里。抽离为 msg_key 索引的模板池需要触碰 1600+ 文件并为每处发明 msg_key,方案把它列为 currentToNewMapping 的一行却未做规模评估。**
  - 影响：渲染适配器桥接期会极长,方案自身风险栏列的'双份维护'近乎必然发生。真实的模板池抽离是数年级工作量,而 currentToNewMapping 把它呈现为一行映射,严重误导工时与风险预期。
  - 修复：补充分级迁移子计划:过渡期保留 LPC message_vision 路径经 Telnet 适配器作为真相源;仅先对最高频的战斗/社交事件抽 msg_key 模板;把 1626 处显式列为多年期 backlog,设优先级;文档明确'模板池是渐进替换,非一次性取代 message_vision',并配合 lint 防双份维护。
- **战斗是 heart_beat 驱动的 tick 制,不是可聚合的回合制:feature/attack.c 第 173 行确认攻击发生在 heart_beat() 内。方案假设存在离散'回合'可做 CombatRoundResolved 聚合,但 XKX 战斗是连续 tick(每个 heart_beat 一次攻击机会),没有天然的 round 边界。强行造 round 要么改变战斗时序语义,要么事件量接近 tick 级。**
  - 影响：要么改变战斗手感(把 tick 凑成 round 改变节奏),要么'聚合'实际上没怎么降量。连续态的 30s 快照思路正确,但离散战斗事件的 round 聚合前提站不住。
  - 修复：把战斗建模为 tick 流,显式定义聚合窗口(按防御方/按时间窗/N-ticks),不假设 round 存在。让 round 成为前端 UI 概念,不作为协议边界。连续态(qi/jing)的周期快照是对的,保留。
- **Python 单分片 5000-8000 并发 + GIL 的容量假设未经验证:方案自承 GIL 是风险,却仍把'单分片 5000-8000 并发(10万房/20万对象)'作为 5-8 分片估算的基准。没有基准测试。一个 Python async Player Actor 逐事件做 Pydantic 校验 + JSON-Patch 生成 + msgpack 序列化 + AOI 订阅扇出,在 GIL 下承载数千战斗者,是推测性的。uvloop 缓解 I/O 但不缓解方案自己标注为风险的 CPU 密集序列化/校验。**
  - 影响：5-8 分片的容量与成本规划建立在未验证的数字上。若真实单分片容量是 1000-2000(GIL 下逐事件校验+编码的合理上界),分片数与成本约翻 3 倍,且 AOI 跨分片流量会成为主导瓶颈。
  - 修复：在架构定型前做端到端基准:单 Player Actor 事件环(Pydantic v2 校验+delta 编码+msgpack+AOI 扇出)的 p99 延迟与吞吐。准备好降级路径(Rust/Cython 热路径、或更细的分片)。把架构与该具体数字解耦,容量规划按实测而非假设。
- **apply/ 修饰符语义被错误统一为'ModifierStack 增量':实测 apply/ 是重载的——apply/name、apply/short、apply/long 是数组栈(name.c 取 mask[sizeof-1],last-wins 覆盖),而 apply/attack、apply/damage、apply/armor 是标量累加槽(combatd.c 用 level += query_temp("apply/attack"),set_temp 覆盖、add_temp 累加)。方案的'ModifierStack 增量组件,前端按基础值+修饰符聚合'把这两类混为一谈。**
  - 影响：前端按单一聚合模型实现会导致展示覆盖类(应 last-wins)被错误累加,或战斗数值类被错误覆盖。两类 apply/ 必须分别建模。
  - 修复：统一建模为 Component:presentation 型(覆盖语义,last-wins)、stat 型(累加,add 语义)、transform 型(数组栈)。前端按类型分别聚合。不要用一个 ModifierStack 名词覆盖两种语义。
- **重连重放的降级路径未进协议:方案风险栏正确指出 ring buffer 满后旧 delta 被驱逐、客户端 last_seq 过旧需降级为全量 snapshot 重发,但 ReconnectResult 接口只有 replay_buffer 字段,没有'replay 不可用,改发 snapshot'的模式。客户端如何区分'重放成功'与'需丢弃本地状态改用全量'未定义。**
  - 影响：重连在缓冲驱逐时会卡住或状态不一致。这是分布式重连正确性的关键缺口。
  - 修复：增加明确的降级路径:服务端检测 last_seq < ring_buffer_oldest_seq 时,在 ReconnectResult 中发 type:'snapshot_resend' 而非 replay_buffer,客户端按全量快照重建。在协议接口中补这个分支并文档化触发条件。
- **双栈过渡的状态分裂无强制一致性测试:方案风险栏提到要以语义事件为单一真相、Telnet 由其派生,但若 Telnet 适配器在桥接期仍走 LPC message_vision 路径、Web 适配器发语义事件,这是两条会漂移的代码路径,没有不变量/契约测试保证 Telnet 输出是语义事件流的纯函数。**
  - 影响：玩家在两种客户端看到割裂的状态/文案,且无机制发现漂移,迁移期体验不可控。
  - 修复：把'每条 Telnet 输出 = render(某条语义事件)'固化为契约,加一致性测试(Telnet 与 Web 同一事件必须互为投影)。
- **洪水控制被当作纯限流,'天雷惩罚'的游戏性流失:alias.c 的 cnt 计数超阈值不只限流,还会对玩家造成 qi/jing 伤害(天雷劈)、unconcious、踢出游戏——这是 themed 惩罚玩法,不是纯基础设施限流。方案'网关层令牌桶替代 cnt 计数'把这层游戏设计一并抽象掉了。**
  - 影响：重构后丢失既有游戏机制且无人察觉,属迁移正确性问题。
  - 修复：把'天雷惩罚'显式建模为业务级反作弊/惩罚子系统(独立于网关限流),保留 themed 惩罚语义。网关令牌桶只负责基础设施级限流,业务级惩罚单独走授权后的游戏逻辑层。
- **heart_beat 实测为 1s 而非方案声称的 2s:代码用 set_heart_beat(1),MudOS 默认即 1s。方案把'2s tick 作为事件批边界'当作既定事实,影响 batch 边界与延迟的数学。**
  - 影响：若实际是 1s,batch 边界与延迟翻倍,事件聚合窗口与带宽估算需重算。
  - 修复：核实驱动配置实际值后再定 batch 边界数学;或明确'2s'是目标配置而非现状。
- **default_ob 被当作静态 prefab,实为活动主对象:LPC default_ob 是可挂 heart_beat、可被修改、会 add("no_clean_up",1) 副作用的 live master。方案映射为'前端 Projection Store 模板继承 + Prefab 资源回退',丢失了活动主对象语义。**
  - 影响：只读状态无碍,但任何依赖 default_ob 计算或运行时状态的默认值会静默丢失活性,前端拿到陈旧静态值。
  - 修复：default_ob 映射到'活动主对象代理':前端 prefab 仅作只读默认值快照;任何 default_ob 的可变/计算默认值必须由服务端 resolve 后再进投影,不交给前端静态 prefab。
