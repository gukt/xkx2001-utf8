# 现状分析·多人交互与经济系统（消息分发/频道/组队跟随/交易金融/婚姻/PK）

## 概述
XKX 的多人交互建立在 MudOS 单进程协作模型上：消息分发以 efun message() 为原子原语，叠加 tell_object/tell_room/say/shout 与 message_vision 三视角渲染；频道由 CHANNEL_D 集中管理，按 wiz_only/menpai_only/anonymous/intermud 策略过滤 users() 全量广播，含反刷屏与跨服 UDP 联邦。组队用内存 object 数组（team.c 指针共享，且 cmds/std/team.c 第28行 return 1 短路致指令失效）。经济采用三币制（金/银/铜=10000/100/1）实体钱对象，MONEY_D 用 children() 全量扫描做流通量上限阀（40万银）反通胀，钱庄存款收 10% 手续费为回收槽。PK 由 killer_reward/death_penalty 构成多维惩罚链（经验/金钱/技能/善恶）。婚姻双方 dbase 双写无事务。整体强耦合单进程对象指针，无法分布式扩展。

## 现有模式
- **消息分发-中心化 efun+收件人自过滤**：以驱动 efun message(class,msg,target,exclude) 为原子原语，其上封装 tell_object(单发)、tell_room(房间广播)、say(同房排除自身)、shout(全服)、message_vision(三视角模板渲染 $N/$n/$P/$p 性别代词)。feature/message.c 的 receive_message 按子类(channel:/outdoor:/weather:)做订阅过滤，并处理盲态、BIG5 转码、输入态缓冲。
- **频道系统-Daemon 配置表+全量过滤**：CHANNEL_D 持 channels 映射，每频道含 msg_speak/msg_emote 模板、wiz_only/menpai_only/anonymous/intermud/filter 等策略位。发消息时 filter_array(users(),filter_listener) 遍历全体在线玩家分发。门派频道按 family_name 动态生成标签，谣言频道匿名化为'某人'并扣 50 精力。
- **表情系统-EMOTE_D 代词替换**：do_channel 支持 verb* 后缀触发 emote，调用 EMOTE_D->do_emote 按 channel_emote 级别(1=chat/2=rumor/3=intermud)生成表情文本，支持 $N/$n/$P/$p/$C/$c/$R/$r/$S/$s 代词与称呼替换，文本含 ANSI 颜色码。
- **战斗-心跳节拍回合制**：heart_beat 每 tick 调 COMBAT_D->fight->do_attack。enemy(可解除)与 killer(持久)双数组，MAX_OPPONENT=4 随机选敌。AP/DP/PP 概率判定躲闪/招架/命中，伤害经 force/jiali/jiajin/武学/武器/护甲多重回调修正，结果以 message_vision 渲染战斗文本。
- **PK-条件状态机+多重惩罚**：玩家发起 kill 设 pking/<id> 临时标记，击杀累加 pker 状态(+120)。城内 PK 触发 killer 状态(官府通缉100)。killer_reward 统计 PKS/MKS、调整 shen(神)/behavior_exp、处理 vendetta 世仇。free_rider 检测防抢杀(受害者非你最后击伤则无奖励)。
- **经济-钱庄存取+货币兑换**：玩家属性 balance 记存款。deposit 按 base_value*amount 增余额并销毁实体钱；withdraw 扣余额并以 pay_player 发放 90%(10% 手续费=回收)。货币兑换 convert 按 base_value 比率。balance 上限 1000000，死亡减半超 10000 部分。
- **经济-流通量上限阀**：MAX_CASHFLOW_ALLOWED 400000(银本位)。query_total_xkx_cashflow 经 children(GOLD_OB/SILVER_OB/COIN_OB) 扫描全服所有钱对象求和。超额时 player_dealer_pay/player_bank_pay/player_job_pay 拒付，形成货币回收阀。
- **组队-内存对象指针数组**：队伍为 object *team 非持久数组，leader/lord 为对象指针。follow_me 按 move 技能概率 call_out 延迟跟随。成员变更经 map_array 广播 set_team 同步全队。注意 cmds/std/team.c 第28行 return 1 短路，组队指令实际被禁用，仅 F_TEAM 特性残留。
- **婚姻-双方 dbase 双写无事务**：MARRY_D 操作双方 spouse 映射(id/name/title 丈夫/妻子)、married_times、last_spouse_id。无中心登记，双存各自 dbase。break_marriage 单边解除+对在线方发 bad_news，validate_marriage 重登时用临时 USER_OB 校验离线配偶。死亡自动解除婚姻。
- **Intermud-UDP 联邦广播**：gchat/gwiz 频道经 DNS_MASTER->send_udp 发 || 分隔的 UDP 包到已知 mud 列表，incoming_request 校验 HOSTADDRESS 防伪，伪造则记 dns_fake 日志并回 DNS_WARNING。gtell 同理实现跨服私语。

## 痛点
- 全进程内存对象引用：队伍/leader/enemy/spouse 均为 object 指针，无法跨服，是分布式部署的根本障碍
- 频道分发 filter_array(users()) 对每条消息全量遍历在线玩家，O(在线人数)，高并发下成为单线程瓶颈
- 货币是 clone 出的实体 LPC 对象，流转靠 move()；流通量审计 query_total_xkx_cashflow 用 children() 扫描全服所有钱对象，O(n) 且每次 NPC/银行发放前都调用
- 无事务一致性：婚姻双方各自 dbase 双写、交易无原子提交、银行存取无回滚，状态可能不一致(break_marriage 单边解除即典型)
- 消息路径耦合 ANSI 颜色码(HIR/HIC)与 BIG5/GB 编码转换，展示逻辑渗入分发逻辑，Web 前端无法复用
-  cmds/std/team.c 第28行 return 1 短路，组队指令(invite/join/dismiss/fight/talk/form)全部失效，组队子系统处于半瘫痪态
- intermud 用 UDP+||分隔字符串，仅 HOSTADDRESS 校验，无加密无认证不可靠，易伪造
- PK/司法规则散落：kill.c 查 pker>240 与 mud_age<18000、combatd 加 pker/killer 状态、death_penalty 多维惩罚，无独立 PvP/司法服务，难扩展与配置
- pay_money 不足时直接 error() 抛异常而非优雅返回，非健壮设计
- 死亡惩罚硬编码于 COMBAT_D：combat_exp/金钱/技能/shen/behavior 多维直接改属性，不可配置，无法支持 UGC 多题材不同规则
- 消息缓冲(MAX_MSG_BUFFER=500)绑在玩家对象，输入/编辑态暂存，新架构应由 WebSocket 客户端承担以解耦服务端

## 应保留思想
- 频道分类策略表(wiz_only/menpai_only/anonymous/intermud/filter)以配置驱动而非硬编码，新架构应保留为可声明频道策略引擎，支撑 UGC 多题材自定义频道
- 表情代词替换体系($N/$n/$P/$p/$C/$c/$R/$r/$S/$s 称谓体系)是中文武侠文本表现力核心，应作为纯函数渲染层保留并解耦于传输
- 流通量上限阀(MAX_CASHFLOW_ALLOWED)作为反通胀货币政策思想值得保留，新架构用账本聚合监控货币供应并按世界配置阈值
- 死亡作为多维回收槽(经验/金钱/技能/善恶/行为)的复合惩罚设计，应保留为可声明策略以适应不同题材
- free_rider/抢杀检测(受害者非你最后击伤则无奖励)是公平 PvP 的关键防滥用机制
- 消息子类订阅过滤(channel:/outdoor:/weather: 按场景元数据决定可见性)的思想，对应新架构基于房间/区域元数据的路由谓词
- 输入态消息缓冲(编辑/输入时暂存再回放)保障玩家不丢失上下文，应由 WebSocket 客户端保留
- 反刷屏：channel_msg_cnt 计数+重复消息拦截+谣言精力消耗，作为频道服务速率限制与成本机制原型

## 应废弃设计
- 内存对象指针表示关系(队伍/leader/enemy/spouse)：无法跨进程跨服务器，新架构须改为 ID 引用+关系图服务
- filter_array(users()) 全量遍历在线玩家分发频道：改为发布订阅事件总线按 topic 路由
- 货币作为 clone 实体对象、流通审计靠 children() 扫描全服钱对象：改为账本服务双式记账，聚合查询替代全量扫描
- message(class,msg,target) 把 ANSI 颜色码与 BIG5/GB 编码绑死在传输与过滤逻辑中：颜色/编码移至 Web 前端渲染层，传输纯语义事件
- channeld 中 intermud 的 UDP+|| 分隔字符串+HOSTADDRESS 单点校验联邦协议：改为 mTLS+签名 JWT 的认证联邦消息总线
- marryd 双方 dbase 双写无事务的婚姻状态：改为关系图服务原子事务提交
- death_penalty/killer_reward 把惩罚与奖励硬编码在 COMBAT_D 单体并直接修改多维度属性：拆为可声明配置的死亡/击杀策略(支持 UGC 多题材不同规则)
- heart_beat 单线程串行 tick 驱动全部战斗与心跳逻辑：改为分布式调度+事件驱动，战斗结算可分区并行
- cmds/std/team.c 第28行 return 1 导致组队指令短路失效的半瘫痪态：新架构重新设计队伍为显式服务实体
- pay_money 不足时 error() 抛异常的非优雅处理：改为返回结果对象，交易原子化带回滚

## 复杂度热点
- /adm/daemons/combatd.c do_attack()：340-780 行的攻击结算流水线，嵌套 skill_power/hit_ob/hit_by 回调与多重 damage 修正，是新战斗/事件系统重构最复杂的单体
- /adm/daemons/moneyd.c query_total_xkx_cashflow()：通过 children(GOLD_OB/SILVER_OB/COIN_OB) 全量扫描所有钱对象计算流通量，O(n) 且每次发放前调用，规模化后是性能与一致性热点
- /adm/daemons/channeld.c do_channel()：单函数承担 emote 解析/匿名/反刷屏/门派动态标签/intermud 转发/监听者过滤，逻辑高度耦合，是频道服务拆分的难点
- /feature/team.c 的对象指针数组同步与 cmds/std/team.c 的 return 1 短路：组队子系统处于'特性在、入口断'的半瘫痪态，需重新设计分布式队伍模型
- /adm/daemons/network/services/gchannel.c 与 gtell.c：跨服消息靠 UDP + || 分隔字符串 + HOSTADDRESS 校验，无加密无认证，是联邦通信安全与可靠性的重构重点
- /feature/damage.c die()：死亡流程串联 death_penalty/killer_reward/make_corpse/break_marriage/dismiss_team/log，跨多个子系统副作用，是事件编排解耦的复杂点
- /d/city/npc/qian.c 钱庄 NPC：deposit/withdraw/convert 逻辑与单个 NPC 绑定且经 call_out 限流，银行服务需从 NPC 中抽离为账户/交易服务
