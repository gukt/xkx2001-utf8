# 方案设计·分布式世界服务器

## 设计概述
核心理念：将 LPC 单进程对象世界(6414 房间/41 区域/3450 处 environment() 直接指针引用)重构为基于虚拟 Actor 模型的分布式世界服务器，以 asyncio 自研轻量 Actor 运行时为主干，借鉴 Orleans 虚拟 actor 语义(按 ID 寻址、按需激活、钝化/激活、placement)。LPC 的对象世界与 Actor 模型高度同构：dbase=封装状态、feature=行为、call_other/find_object=位置透明寻址雏形、previous_object()=消息发送者、heart_beat/call_out=定时自消息、tell_room/message=位置相关广播、move()=对象迁移、Daemon 单例=服务 Actor、master.c connect/crash/valid_*=驱动层 supervisor 与安全边界、save_object/.o+autoload+clean_up=虚拟 actor 的钝化/激活。新架构以此同构映射为基础，但用可序列化的 EntityID 引用替换裸对象指针、用异步消息替换同步 call_other、用分片集群替换单进程地址空间。

分布式拓扑分四层：(1)无状态 WebSocket 网关层承载海量长连接与会话路由；(2)有状态 WorldShard 分片层，按区域共同体(Louvain 社区检测对 6414 房间出口图凝聚)切分，单分片承载 5000-8000 并发；(3)全局状态服务层，将原单进程内可变全局守护进程(channeld/natured/securityd)外部化到 Redis/PostgreSQL；(4)持久化与事件层，PostgreSQL 权威快照+事件溯源、Redis 热状态、NATS JetStream 扇出。跨区迁移复刻 go.c 流程并分布化(freeze->snapshot->activate->replay->deactivate)，节点容错靠分层 supervision(玩家/房间 one-for-one，区域 all-for-one)+WAL 重放+确定性 seeded RNG。世界状态分布遵循'逻辑实体+缓存逐出/回填'，保留 dbase(持久)vs tmp_dbase(瞬态)分层、default_ob 原型回退、heart_beat 按需 tick 思想。增量迁移路径：阶段一单进程 asyncio actor 重写核心循环验证语义，阶段二加多节点+placement+迁移，阶段三接 Ray/对象存储做 UGC 与 ML，避免一次性全量重写。

## 组件
- **WSGateway 网关层**：无状态 WebSocket 入口，承接海量长连接。职责：传输层认证(JWT 验签/argon2 验密替换 crypt)、会话路由(session_id->node 路由表写入 Redis，支持断线重连定位)、限流(网关层全局令牌桶替换 process_input 内联 cnt 洪水控制)、WS Ping/Pong 20s + 应用层心跳带 last_seq、消息协议(MsgPack 为主 JSON 兜底，统一 {type,version,seq,payload} 帧)。不持有任何游戏状态，水平扩展靠 SO_REUSEPORT 多 worker。origin 校验 + per-session 配额防刷屏(原 channel_msg_cnt 的分布式版)。（技术：aiohttp/websockets + uvloop + MsgPack）
- **ShardSupervisor 分片监督树**：对应 master.c 的驱动层监督职责(connect/crash/epilog)。管理每个 WorldShard 的生命周期：启动/停止/健康检查/崩溃重启。监督策略：玩家与房间 actor 为 one-for-one(单实体崩溃只重启自身)；Region 子树为 all-for-one(区域崩溃重启整个区域子树，类似 LPC 重载区域)。层级：顶层 ServiceSupervisor -> ShardSupervisor -> RegionSupervisor -> Room/NPC/Player Actor。crash() 回调(原 master.c 第44行)映射为 supervisor 捕获 actor 异常并落 CRASHES 审计日志。（技术：自研 asyncio supervision + 结构化日志(structlog)）
- **WorldShard 世界分片**：有状态游戏节点，持有一组区域共同体的房间/NPC/玩家/AOI。核心：运行 ActorRuntime(实体 actor 的激活/钝化/消息分派)、SimulationTick(每 2s 推进，对应 heart_beat)、AOIManager(1 跳房间图邻域订阅，对应 tell_room/move 广播语义)。单分片容量上限设计 5000-8000 并发(10万房/20万对象)。玩家会话粘附所在分片，跨分片仅迁移玩家实体与必要快照而非世界数据。（技术：自研 asyncio ActorRuntime + uvloop 事件循环）
- **PlacementService 放置与寻址服务**：实现虚拟 actor 语义：按 EntityID 寻址(对应 find_object 路径寻址的位置透明雏形)、按需激活(对应 load_object/new 的惰性加载)、钝化/激活(对应 swap/clean_up 的对象换出与重新 clone/restore)。维护 EntityID->ShardNode 路由表(Redis)。placement 策略：同房间战斗参与者保持同节点亲和(避免分布式事务)；玩家进入新 region 时迁移 Player Actor 到目标节点。禁止 actor 间传裸引用，仅传 addr 或值快照(沿用 save_object 不保留引用、靠 autoload 重建的安全特性)。（技术：Redis 路由表 + 自研 placement 策略）
- **MigrationCoordinator 迁移协调器**：跨区 handoff 的编排者，复刻 go.c 流程并分布化：freeze(冻结玩家 actor，停接受输入)->snapshot(dbase+inventory+combat+team 写入 Redis 附 TTL，对应 save_object 但结构化)->activate(目标分片 restore 并重建 autoload 引用，对应 restore_object)->replay(重放挂起期间排队消息，保证不丢)->deactivate(源分片注销)。组队迁移按队形批量 handoff 保持顺序(对应 all_inventory(env)->follow_me 的跟随传播)。门状态跨分片走共享存储保证一致(替换 room.c open_door/close_door 的 find_object 双面协调)。（技术：Redis 状态暂存 + WAL 消息日志重放）
- **GlobalStateService 全局状态服务**：将原单进程内可变全局守护进程状态外部化。三分类：(1)无状态可副本--weapond/emoted/chinesed，多副本缓存；(2)全局只读配置--natured 昼夜相位表、commandd 命令注册表，从 DB 读取多副本缓存；(3)全局可变--channeld(改 pub/sub 替代 filter_array(users()) O(n) 遍历)、securityd 鉴权策略、marryd 关系。channeld 频道消息经 Redis Pub/Sub 或 NATS 扇出，按 topic 路由替代全量广播。（技术：Redis Pub/Sub + NATS JetStream）
- **BoundaryRoomAgent 边界房间代理**：处理分片边界房间(区域共同体割边上的房间)的跨分片事件。AOI 跨分片订阅通过边界房间转发，避免每条消息都走全局总线。玩家跨区移动时，源边界房间广播 leave 事件、目标边界房间广播 enter 事件(对应 message vision 的 mout/min)。门状态查询经此代理读写共享存储(替换 find_object(exits[dir]) 协调另一面门，原 room.c 第185行若对侧房间未加载即静默失败的问题由此消除)。（技术：跨分片事件代理 + 共享门状态存储）
- **CombatCoordinatorActor 战斗协调 actor**：将 COMBAT_D(全库 531 处 COMBAT_D-> 调用)实例化为 region/战斗级 Coordinator Actor，串行裁决回合，避免分布式事务。同房间战斗参与者保持同节点亲和，战斗结算分区并行。为每场战斗分配独立 seeded RNG，回合结果写 append-only 事件日志，支持回放与反作弊审计(分布式下调试与公平性的前提)。回合聚合事件 CombatRoundResolved 而非逐 tick 事件，避免事件爆炸。（技术：独立 seeded RNG + append-only 事件日志）
- **SchedulerService 分布式调度器**：替换全库 3728 处 call_out(931 文件)的非持久单点定时器。特性：持久化(崩溃不丢，对应原 call_out 崩溃即静默丢失的痛点)、可取消(保留 remove_call_out 语义，如取消复活/关门计时)、可分布式调度(对应 natured 自递归 call_out 链的昼夜循环，单点 miss 即整日错乱)。reset(1800s)映射为按区域分片的事件驱动重生服务读取 spawn 配置(替换 room.c reset() 内联 make_inventory)。clean_up(180s)映射为带活跃度感知的缓存逐出(对应 usr_in() 检查 deep_inventory 是否有玩家)。（技术：Redis Stream 延迟队列 + APScheduler）

## 关键接口/事件
- EntityAddr: 'world://<shard>/<region>/<room>#<entity-uuid>' -- 统一寻址，替代 LPC file_name 字符串路径与 path#number 克隆标识。本地解析为对象引用，跨节点解析为透明代理(对应 find_object(path) 的位置透明雏形但显式化)
- ActorRef.send(msg: Message, ctx: CallContext) -> None -- 异步消息发送，替代同步 call_other。CallContext 携带调用者 addr + 能力令牌(替代 previous_object() 隐式信任链)，显式化授权链
- ActorRef.ask(msg: Message, ctx: CallContext, timeout) -> Result -- 带超时的请求-响应，替代依赖同步返回值的调用(如 move() 后立即检查 environment())。失败返回结果对象而非 error() 抛异常(对应 pay_money 不足时 error() 的非优雅处理改进)
- CommandMessage{actor_id, verb, args, seq, ctx} -> 玩家输入命令，对应 command_hook 分发后的 main(me, arg) 契约
- MoveEvent{actor_id, from_room, to_room, dir, mout_msg, min_msg} -- 玩家移动事件，替代 go.c 中 message('vision', name+mout/min) 的 ANSI 耦合广播。展示文案解耦为前端渲染
- HandoffRequest{entity_id, target_shard, snapshot, team_members[], seq} -- 跨区迁移请求，组队批量迁移保持队形顺序(对应 follow_me 跟随传播)
- HandoffComplete{entity_id, new_shard, ack_seq} -- 迁移完成确认，源分片收到后注销实体
- AOISubscription{subscriber_id, room_id, radius=1} / AOIBroadcast{room_id, event, recipients[]} -- 兴趣区域订阅与广播，对应 tell_room 语义但限定 1 跳邻域而非全量
- DoorStateGet/Set{room_id, dir} -> {status, other_side} -- 跨分片门状态读写，替代 room.c open_door 的 find_object(exits[dir]) 双面协调(原第185行对侧未加载即静默失败)
- CombatRoundResolved{combat_id, participants[], actions[], rng_seed, ts} -- 回合聚合事件，替代逐 tick 事件爆炸。可回放反作弊
- SnapshotSaved{entity_id, version, ts} / DeltaEvent{entity_id, patch, seq} -- 状态同步，对应 dbase(snapshot)/tmp_dbase(delta) 分层
- SpawnRequest{room_id, object_list} -- 房间重生，对应 room.c reset() 的 make_inventory 声明式清单(不可变基础设施 reconcile 语义)
- HealthCheck{shard_id} -> {alive, entity_count, tick_lag_ms} -- 分片健康探针，对应 master.c epilog 启动预加载清单的服务依赖健康检查演进
- ChannelPublish{topic, msg, filter_policy} / ChannelSubscribe{subscriber_id, topic} -- 频道发布订阅，替代 channeld filter_array(users()) 全量遍历。filter_policy 承载 wiz_only/menpai_only/anonymous 策略
- Supervise{child_spec, strategy} -- 监督策略声明(one_for_one/all_for_one)，对应 master.c crash() 的崩溃恢复但分层显式化

## 数据模型
核心数据模型与持久化策略，映射现有 dbase/tmp_dbase/save_object/heart_beat 体系：

【实体身份与地址】EntityID 为 UUID(解耦 file_name 即身份又即代码路径的强绑定)，地址 world://<shard>/<region>/<room>#<entity-uuid>。entity_type: player/npc/room/item。每个 EntityID 路由表记录 {entity_id, shard_node, status: active|passivated, last_seen}。

【状态分层 - 保留 dbase/tmp_dbase 分离】
- 持久层(dbase)：玩家角色属性/技能/经验/金钱/位置，对应 dataclass 标注 @persist 字段，写 PostgreSQL snapshots 表。房间蓝图配置走 Prefab(只读 JSON，对应 default_ob 原型回退)。
- 瞬态层(tmp_dbase)：战斗状态/修饰符堆叠(apply/ 命名空间)/AOI 订阅/忙碌标志，标注 @transient，写 Redis。这正是现有 dbase vs tmp_dbase(static 修饰)的边界保留。
- 快照 vs delta：dbase=snapshot 基线、tmp_dbase=易失 delta。进场景/重连发 snapshot，变更发 JSON-Patch 风格 delta + 单调 seq(复用 F_DBASE 路径键)。

【权威存储】PostgreSQL：entities 表(entity_id PK, type, region, shard, created, updated)、snapshots 表(entity_id FK, snapshot JSONB, version, ts)、events 表(append-only，event_id, entity_id, type, payload JSONB, seq, ts, rng_seed 用于战斗回放)、world_manifests 表(CPK 内容包清单+依赖声明+provenance 溯源链)。

【热状态】Redis：session 路由表(session_id->{node,user,character,world})、entity 路由表(entity_id->shard_node)、热实体状态缓存(玩家在线 dbase 热副本)、AOI 订阅集合、门状态共享键(door:<room_id>:<dir> = {status,other_side}，替换两房间静态 mapping)、迁移快照暂存(player:handoff:<id> TTL 60s)、频道 pub/sub。

【房间图模型】rooms 表(room_id PK, region, shard_hint, exits JSONB{dir->target_room_id}, cost, outdoors, doors JSONB, spawn_config JSONB)，exits 改为 room_id 引用而非绝对路径(解耦路径=ID 的硬编码命名约定，房间可在区域间迁移而不破坏引用)。

【分布式确定性】每场战斗/每 region 分配独立 seeded RNG，seed 存入事件日志，回合结果可回放重建。玩家 dbase 沿用 heart_beat tick 周期快照(每 30s 或关键节点如死亡/下线)，对应现有仅在 quit/death/save 触发存档的脆弱性改进。

## 旧->新映射
- file_name 字符串既是对象身份 ID 又是代码路径，克隆用 path#number 标识 -> **EntityID(UUID) 解耦身份与实现路径，地址 world://<shard>/<region>/<room>#<uuid>**（代码重构移动文件不再断裂存档引用/autoload 路径/find_object 调用；支持热重载与多版本共存(UGC 必需)）
- object 类型直接指针引用 this_object()/environment()，全库 3450 处 -> **ActorRef(addr) 透明引用 + 仅传 addr 或值快照，禁止传裸引用**（沿用 save_object 不保留引用、靠 autoload 重建的安全特性；跨进程通信改为异步消息）
- call_other 同步阻塞调用，大量依赖同步返回值(如 move() 后立即检查 environment()) -> **ActorRef.ask(msg, ctx, timeout) 带超时请求-响应，或 fire-and-forget send**（迁移面极大但必须改；返回结果对象而非 error() 抛异常）
- master.c connect() 创建 login 对象作为连接入口 -> **WSGateway 鉴权后向 PlacementService 激活 Player Actor，会话路由表写 Redis**（网关无状态可水平扩展，断线重连靠路由表定位玩家 actor 所在分片）
- master.c crash() 单进程崩溃即全服宕机 -> **ShardSupervisor 分层监督，玩家/房间 one-for-one，区域 all-for-one**（单实体崩溃只影响自身，区域崩溃重启子树而非全服；CRASHES 审计日志保留）
- master.c epilog/preload 显式服务清单启动预加载 -> **服务依赖描述 + 健康检查清单，GlobalStateService 按分类(无状态副本/只读/可变)启动**（preload 15 个守护进程映射为分类管理的服务集群）
- 6414 房间全内存单进程，reset() 每 1800s 全量扫描所有已加载房间 -> **区域共同体分片，按分片/按需/事件驱动重生(SpawnRequest 读 spawn 配置)**（可水平扩展，热区(city/beijing)做内部微分区）
- go.c 单函数承载移动/战斗/骑乘/阻挡/cost/valid_leave 高度交织 -> **迁移管线解耦：CommandMessage->MovementSystem->MigrationCoordinator(handoff)->AOIBroadcast**（跨区时复刻 go.c 流程并分布化(freeze/snapshot/activate/replay/deactivate)）
- room.c 门状态为静态 mapping + find_object 双面协调，对侧未加载即静默失败 -> **DoorState 共享存储键(Redis)，BoundaryRoomAgent 跨分片读写**（消除单进程内 find_object 依赖；双面门状态一致性由共享存储保证）
- feature/move.c move() 同步 call_other(dest) 加载 + move_object 指针迁移 -> **MigrationCoordinator 编排跨节点 handoff，本地 move 仍为 actor 内状态变更**（encumbrance 传播(weight/encumb)改为 room actor 聚合自身 inventory 重量）
- char.c heart_beat() 上帝方法处理战斗/治疗/状态/衰老/NPC聊天/频道清理/熄火 -> **SimulationTick 调度独立 System：CombatSystem/HealSystem/ConditionSystem/AISystem**（按需 set_heart_beat(0) 熄火映射为事件驱动仿真步进(仅活跃实体 tick)）
- call_out 全库 3728 处(931 文件)非持久单点定时器，崩溃即丢 -> **SchedulerService 持久化分布式调度器，保留 remove 取消语义**（复活/关门/昼夜循环(natured 自递归 call_out 链)不再单点错乱）
- save_object/restore_object 全量覆盖写 .o 文本快照，无历史/无并发/崩溃丢数据 -> **PostgreSQL snapshots 表(结构化快照) + events 表(append-only 事件溯源)**（玩家存档不再仅在 quit/death 触发；周期快照 + 事件可回放重建）
- data/user/<首字母>/<id>.o 按首字母目录分片 -> **PostgreSQL 按 entity_id 哈希分片 + Redis 热缓存，分片键对齐区域共同体**（天然分片思想保留但升级为数据库分片键，避免热点(原仅有 h/m 两个目录说明玩家少)）
- config.xkx max users:150 硬编码并发上限 -> **弹性扩缩容，按 worker 分片连接，单分片 5000-8000 并发**（WebSocket+分布式突破单进程上限）
- channeld filter_array(users()) 对每条消息全量遍历在线玩家 O(在线人数) -> **Redis Pub/Sub 或 NATS 按 topic 路由，频道策略表(wiz_only/menpai_only/anonymous)为订阅谓词**（高并发下不再单线程瓶颈；intermud UDP+白名单联邦升级为 mTLS+签名 JWT）
- previous_object()/geteuid 隐式信任链做权限校验 -> **CallContext 显式携带调用者 addr + 能力令牌(RBAC)**（force_me 的 ROOT_UID 后门改为带审计日志的显式特权动作 API）
- eval cost 600 亿单线程全局执行预算 -> **per-请求超时 + per-tenant 资源配额(内存/CPU/调用深度) + 熔断器**（分布式环境需 per-tenant 资源隔离；对齐 config.xkx 的 eval-cost/reset/clean-up 语义固化为 per-CPK 配额）

## 分布式扩展策略
分布式扩展与分片策略：

【分片单元：区域共同体】基于对 6414 房间/41 区域出口图分析，分片单元定为'区域共同体'(出口图社区凝聚)而非单区域或单房间。用 Louvain 社区检测对出口图跑社区检测得到 5-8 个共同体作为初始分片，边界对齐社区割边使跨分片移动最小化。注意现有跨区连接多为 car.c 快速旅行(17 个门派山门目的地)而非步行出口，步行 exits 大多区域内连通，故社区凝聚天然聚合门派地域。

【AOI 兴趣区域】默认半径=1 跳房间图邻域，匹配现有 tell_room/move 广播语义。户外野外区域在 DSL 补坐标后用九宫格。无需先做坐标化改造即可落地图 AOI。边界房间(社区割边上的房间)建立跨分片事件代理，AOI 跨分片订阅通过边界房间转发，避免每条消息走全局总线。

【热区微分区】city(扬州 441 房)、beijing(625 房)、shaolin、dali 等高连通枢纽做内部微分区，切房间簇成子分片并设城市入口路由。动态负载迁移以房间簇为粒度做'冻结迁移'。

【连接层水平扩展】无状态 WSGateway(SO_REUSEPORT 多 worker，单机数万连接)水平扩展，session<->分片路由表存 Redis 支持断线重连定位。游戏分片 WorldShard 有状态持有房间/NPC/玩家/AOI。

【全局状态解耦前置债务】将 32 个守护进程按三类解耦：无状态(weapond/emoted/chinesed 可副本)、全局只读(natured 昼夜相位/命令注册表)、全局可变(channeld/securityd/marryd 状态迁 Redis)。channeld 改 pub/sub 替代 users() 全局遍历。

【容量估算】单分片上限 5000-8000 并发(10 万房/20 万对象，Python async+uvloop)。1 万在线玩家需 5-8 分片。玩家会话尽量粘附所在分片。

【跨区 handoff】玩家进入新 region 时迁移 Player Actor：先激活后注销，消息日志重放保证不丢。同房间战斗参与者保持同节点亲和，把 COMBAT_D 实例化为 region/战斗级 Coordinator Actor 串行裁决回合，避免分布式事务。组队迁移按队形批量 handoff 保持顺序。

【多租户/多题材】每个题材世界(武侠/大航海/书院/穿越/现代)独立 supervision 与资源配额，Actor 边界即租户隔离边界。UGC 的 DSL 场景编译生成 region 子树 + room/npc actor 蓝图，Agent 协作创作作为向 region 投递'创作消息'的 actors。按题材分世界 shard。

【不要过度 Actor 化】无状态 HTTP(登录/查询/排行榜)、批量计算(地图/剧情生成)走传统微服务/批处理；仅在'有状态、可寻址、需激活与迁移'的实体上用 actor，保持架构分层清晰。

## 技术选型
- Actor 运行时：自研轻量 asyncio ActorRuntime(不依赖 entt-py/esper，不直接用 Akka/Orleans 非 Python)。借鉴 Orleans Virtual Actor 语义(按 ID 寻址/按需激活/钝化激活/placement)。Ray 作为可选执行后端与 ML/UGC 生成工作负载，游戏主循环 actor 自研以掌控延迟与消息序
- 网关：aiohttp 或 websockets+uvloop(SO_REUSEPORT 多 worker)，FastAPI 司 REST/JWT/UGC，独立扩缩容。MsgPack 为主、JSON 兜底 UGC 动态内容，统一 {type,version,seq,payload} 帧封装
- 会话认证：argon2 验密替代 crypt()，JWT(RS256，access 5-15min+refresh 7d)，session_id 入 Redis 路由表，token 载 character_id/world 便于跨分片
- 权威存储：PostgreSQL(entities/snapshots/events/world_manifests 追加表)。热状态：Redis(session/entity 路由表/热实体状态/AOI 订阅/门状态/迁移快照)
- 消息总线：Redis Pub/Sub(热事件扇出) + NATS JetStream(持久扇出与跨节点 tell/频道)。单区域 MUD 用 PG+NATS 足够，不过早引入 Kafka(仅在跨区域多租户达规模时)
- 调度器：Redis Stream 延迟队列(持久化 call_out)+ APScheduler(reset/昼夜循环)。保留 remove 取消语义
- 确定性随机：每场战斗/每 region 独立 seeded RNG(Python random.Random(seed))，回合结果写 append-only 事件日志
- ECS：自研 SparseSet ECS(不依赖外部库)，组件按领域聚合(Identity/Vitals/Attributes/SkillSet/CombatState/EquipmentSlots/ModifierStack/Position/Conditions/Inventory)，系统替代 daemon+heart_beat
- WASM 沙箱：UGC 脚本逃生舱用 WASM(内存安全+燃料计量+能力化+语言无关)，可信编辑者可用受限 Python。对应 eval cost 替换为 per-CPK 资源配额
- 可观测：structlog 结构化日志(替代 log_file 散落文本审计)+Prometheus 指标+分布式追踪。CRASHES/MONEY/PLAYER_DEATH/PKILL 等审计并入结构化事件流

## 风险
- 3450 处 environment() 直接指针引用是分布式化最大障碍，迁移面极大且大量依赖同步返回值(如 move() 后立即检查 environment())，无法一次性全量重写。缓解：阶段一单进程 asyncio actor 重写核心循环验证语义，阶段二加多节点，阶段三接 Ray/对象存储；先以 actor 适配器封装现有 LPC 行为逐子系统替换
- 战斗回合的强一致性与 random() 随机性是分布式化关键难点。分布式下 random() 不可重现会导致战斗结果分歧。缓解：每场战斗分配独立 seeded RNG 写入事件日志，回合结果可回放重建；战斗参与者保持同节点亲和避免跨节点事务，由 CombatCoordinatorActor 串行裁决
- go.c 移动逻辑与战斗/骑乘/阻挡/cost/valid_leave 高度交织(289 行单函数)，跨区迁移 handoff 复刻此流程分布化时极易遗漏隐式行为(如 exit_blockers 检查、playing 箫声阻挡、rided 骑乘状态)。缓解：先逐项拆解 go.c 流程为显式管线步骤再分布化，保留 new_valid_dest 动态重定向语义
- 3728 处 call_out(931 文件)迁移到持久化调度器工作量大，且许多闭包捕获对象引用无法序列化、对象销毁后悬空指针(如韦陀伏魔剑的 checking 递归 call_out)。缓解：先迁移无闭包的纯定时器，闭包型改为显式 Effect 对象可序列化可中断
- 自研 Actor 运行时与 ECS 调度器的工程成本高，需掌控延迟、消息序与确定性。缓解：借鉴 Orleans Virtual Actor 语义而非从零发明；不直接用 Akka/Orleans(非 Python 增加跨语言成本)；Ray 仅作可选执行后端与 ML/UGC 负载
- 房间 exit 路径硬编码绝对路径(如 /d/wudang/shanmen)在重构移动文件时会断裂大量引用。缓解：新架构解耦实体 ID 与代码路径，但迁移期需维护 path->EntityID 映射表保证兼容
- make_data_dir 是悬空调用(inherit/save.c:31 调用 MASTER_OB->make_data_dir() 但 master.c 从未定义)，save_data() 是死代码，说明保存子系统有未维护的腐化路径，迁移时需谨慎识别哪些持久化路径实际生效
- 单分片内 Python asyncio 仍是单线程事件循环，一个慢对象(如复杂 NPC AI chat)仍可能阻塞该分片 tick。缓解：eval cost 替换为 per-请求超时 + 资源配额 + 熔断器；重计算任务 offload 到进程池或 Ray；保持 set_heart_beat(0) 熄火思想仅活跃实体 tick
- securityd.c valid_cmd 存在 authorized_cmds["cmd"] 拼写 bug(应为 cmds)与硬编码目录，权限模型已损坏。分布式架构需先重建能力权限模型再开放 UGC 协作，否则跨包引用的安全边界无法成立
- 门状态跨分片协调若共享存储(Redis)故障会导致双面门不一致。缓解：门状态写入用 Redis 原子操作 + 短 TTL 缓存 + DB 落盘兜底；边界房间代理做幂等重试
- Louvain 社区检测结果依赖 exit 图质量，现有部分房间 exits 可能为空或断链(如 virtuald.c 空壳未生成虚拟对象)。缓解：分片前先校验 exit 图完整性，断链房间归入最近连通社区或单独处理

## 🔍 对抗验证

**裁定**：risky — 方案的代码库调研极其扎实（我逐项核验 master.c 第44行 crash、room.c 第185行门状态静默失败、save.c make_data_dir 未定义死代码、securityd 第757行 authorized_cmds["cmd"] 拼写 bug、config max users 150、go.c 289 行、natured 自递归 call_out、virtuald 19 行空壳、32 守护进程/41 区域/6414 房间/531 处 COMBAT_D 等几乎全部属实），LPC↔Actor 同构映射是真正有洞察力的理论基础，分阶段迁移方向正确。但它是 risky 而非 sound：(1) headline 迁移面被低估一个数量级——真实同步调用面是全库 68771 处 -> 方法调用 + 520 处 inventory 图遍历，而非标称的 3450 处 environment；(2) 自研 asyncio 虚拟 actor 运行时是多年级工程，方案却以'借鉴 Orleans 语义'轻描淡写带过；(3) 战斗确定性只解决了 RNG seed 一半，未在战斗开始时快照参与者 dbase，do_attack 是 447 行单函数持续读写实时状态，回放会分叉；(4) Redis 成为所有跨分片热路径的同步单点；(5) 区域 all-for-one 监督爆炸半径比原单进程更糟。这些是承重假设而非边缘细节，失败成本是多年期重写，故定 high。

**严重度**：high

### 问题与修复
- **迁移面被严重低估：方案标称'3450 处 environment() 直接指针引用'，实测 environment() 仅 664 处、this_object() 4214 处（二者合 4878），而真正的同步调用面是全库 68771 处 -> 方法调用（仅 /d 就 36333 处）+ 520 处 all_inventory/deep_inventory 同步图遍历 + 84136 处 query/query_temp/set/set_temp dbase 访问。headline 数字把迁移面低估了一个数量级，导致分阶段排期与工程量估计失真。move() 实测确有'调用后立即检查 environment()/this_object() 是否被析构'的同步后置条件（feature/move.c:109-119），是真实的硬骨头，但这只是 68771 处中的少数。**
  - 影响：排期与工作量被低估一个数量级，若按现有数字承诺交付期，阶段一几乎必然超期或被迫偷工减料；正确数字反而能迫使先做优先级排序（先迁移高频跨对象调用图，如 go/move/combat/channel），降低失败风险。
  - 修复：立即重测迁移面：单独统计 environment()=664、this_object()=4214、箭头方法调用 68771、all/deep_inventory=520、query/set dbase 访问 84136。按'跨 actor 必须改异步'vs'actor 内本地状态可保留同步'两分类重新估算，并以箭头调用面（而非 environment）作为主线指标驱动阶段一工作量与排期。
- **自研 asyncio ActorRuntime（含 placement/激活钝化/迁移 handoff/消息序/监督树）是本方案最大的工程风险，却被作为'借鉴 Orleans 语义而非从零发明'轻描淡写。借鉴语义不等于免实现：虚拟 actor 的按需激活、钝化一致性、handoff 不丢消息、消息重排序与超时语义、分布式死锁检测，每一项都是数月级工程。方案同时排斥 Akka/Orleans（非 Python）与 Ray（仅可选后端），等于在 8400 文件迁移之外再背一个自研分布式框架的债务。**
  - 影响：自研运行时是整个方案的成败支点。若按现方案自研，2-3 年内主循环稳定性、消息序、激活/钝化正确性都难以收敛，且无法借力成熟框架的容错与可观测性。改用 Ray（或先单进程 asyncio+actor 库）可把运行时风险外移，把团队精力集中在游戏语义迁移这一真正不可替代的工作上。
  - 修复：明确放弃'从零实现虚拟 actor 运行时'。阶段一用 Ray 的 actor 模型做最小可用验证（Ray 是 Python 原生、支持 placement/调度/容错），证明 LPC↔Actor 同构映射在语义层成立后再决定是否替换。游戏主循环可先跑在 Ray actor 上度量延迟，证明 P95 不可接受时再考虑自研薄壳。不要在未做原型前承诺自研运行时。
- **战斗确定性只解决了一半：'每场战斗独立 seeded RNG'只能保证随机数序列可重现，但 combatd.c 的 do_attack（340-787 行，约 447 行单函数）持续读写参与者 dbase/tmp_dbase（技能、装备、modifier 堆叠、内力/精力实时变化），这些输入在战斗期间会被其他异步消息（治疗、队友 buff、装备切换）修改。仅记录 RNG seed 而不在战斗开始时快照全体参与者状态快照，回放结果会与实际分叉。方案称'回合结果可回放重建'但未定义 CombatContext 快照边界。**
  - 影响：若以为 seed RNG 即可回放，上线后反作弊审计与 bug 复现都会失败（回放结果与实际不符），且这类失败极难发现（无人会去逐回合核对）。补齐 CombatContext 快照后，回放才真正可信。
  - 修复：将确定性边界明确为'战斗开始时对全体参与者+可观察环境做状态快照，写入 CombatContext'，seeded RNG 只是其一部分。回放时从 CombatContext 起点重放，禁止战斗中途引入未在快照中的输入。先抽样 2-3 场真实战斗导出其 do_attack 读取的 dbase 字段集合，据此定义 CombatContext 的最小边界。
- **Redis 成为所有跨分片操作的同步单点：会话路由、实体路由、AOI 订阅、门状态、迁移快照、频道 pub/sub 全部走 Redis。任何一次 Redis 抖动都会冻结全服跨区操作（门状态、迁移、频道全挂）。方案对'门状态 Redis 故障'给了原子操作+短 TTL+DB 兜底，但未覆盖 session/entity 路由表在 Redis 分区/脑裂时的处理，也未给出 Redis 整体不可用时的降级策略。对'高并发大规模'目标，每条跨分片消息都过 Redis 是吞吐上限。**
  - 影响：当前设计下 Redis 即游戏的单点命脉，既限吞吐又是爆炸半径上限。热路径去 Redis 化后，单分片可独立服务其粘附玩家，Redis 抖动只影响跨区操作，可用性与扩展性同时改善。
  - 修复：区分热路径与控制路径：热路径（会话路由、实体路由、AOI 订阅）用无外部依赖的一致性哈希环（shard=hash(entity_id)），Redis 仅做控制面（placement 变更通告、handoff 暂存）。门/迁移快照等强一致需求走 PostgreSQL 事务。明确写明 Redis 降级时游戏降级（仅本分片可玩）而非全停。
- **RegionSupervisor 用 all-for-one（区域崩溃重启整个区域子树）理由不足，且爆炸半径比原单进程更糟：原 LPC 单房间崩溃不会重启同区域其他房间（master.c crash() 是全进程级，但日常单房间 bug 走 catch/destruct 局部处理）。all-for-one 意味着一个房间 actor 异常会连带重置同区域数千房间/NPC/玩家，丢失内存状态，这在分布式下反而放大了单点故障。方案给的理由'类似 LPC 重载区域'不成立——LPC 并无'区域崩溃重启'语义。**
  - 影响：保留 all_for_one 会让分布式版在某些故障下比单进程更脆弱（单进程崩是全服宕但可快速重启；区域 all_for_one 是部分宕但重建慢、影响大）。改为 one_for_one+持久化重建后，单房间崩溃只影响该房间，符合方案'单实体崩溃只影响自身'的初衷。
  - 修复：区域层默认改 one-for_one，仅在能证明子实体间存在不可恢复的共享可变状态时才对该子树用 all_for_one，并显式记录触发条件。区域崩溃应触发从 PostgreSQL 快照+事件日志重建，而非丢弃在线状态。先确认当前 LPC 是否真的有'区域级共享状态'（多数区域状态其实是 per-room 或 per-npc，无真正区域共享），很可能根本不需要 all_for_one。
- **缺少行为等价性验证策略：8400 LPC 文件、531 处 COMBAT_D 调用、68771 处方法调用迁移到 Python actor，如何证明新版与旧版行为一致？方案全文未提差分测试、命令流回放、golden-output 比对。这是大型语义迁移最关键的风险兜底，缺失则行为漂移不可见，直到玩家发现战斗/经济/任务结果异常。**
  - 影响：无差分测试，行为漂移要等玩家投诉才暴露，届时已积累大量隐性 bug 且难以定位根因（是语义翻译错还是并发顺序变）。差分测试是把'不可见的正确性'变成可度量指标的唯一手段，对 8400 文件迁移是必需品而非可选项。
  - 修复：阶段一同步立项差分测试基建：录制 LPC 侧命令流（玩家输入序列+tick 边界）作为 golden trace，Python actor 版逐条重放并 diff 输出帧与状态快照。先覆盖 go/move/combat/channel 四条主线，再横向铺开。无差分测试通过的子系统不进入阶段二。
- **GlobalStateService 三分类未覆盖全部 32 守护进程：实测 /adm/daemons 下确为 32 个 .c，方案三分类只点名了 weapond/emoted/chinesed（无状态）、natured/commandd（只读）、channeld/securityd/marryd（可变），遗漏了 adsd/band/aliasd/fingerd/inquiryd/editord/ftpd/languaged/moneyd/pigd/profiled/rankd 等。这些守护进程的迁移归属与外部化策略未定义，迁移时易被遗漏或处理不一致。**
  - 影响：遗漏守护进程会导致迁移后部分子系统静默失效或行为异常。securityd 尤其关键——UGC 平台的安全边界完全依赖它，若带着已知损坏的权限模型上线，跨包引用的安全边界无法成立，UGC 协作无从谈起。
  - 修复：在 GlobalStateService 补一张'32 守护进程全量分类表'，逐个标注分类与迁移方式（多数是无状态或只读，可批量处理）。特别审视 securityd：先修第 757 行 typo，明确其权限模型是否已损坏到需重建再开放 UGC（方案自己也指出'权限模型已损坏'，这点应升为 UGC 前置阻塞项）。
- **freeze/snapshot/activate/replay/deactivate 迁移管线的部分失败语义未规约：目标分片在 activate 中途崩溃、源分片在 deactivate 前崩溃、replay 期间消息顺序错乱等场景未定义补偿与回滚。move() 已证明同步语义下'移动后立即析构/再移动'是常态（move.c 注释明说 this_object 可能被 move_object 析构），异步化后这些时序窗口被放大。WAL 重放保证'不丢'但未保证'不乱序/不双活'。**
  - 影响：不补齐 partial-failure 规约，迁移在生产环境会偶发丢消息/双活/状态不一致，且难以复现与定位。规约化后故障行为可预期、可恢复、可审计。
  - 修复：为 handoff 每一步定义超时、补偿动作与最终一致目标（如'deactivate 永远以目标分片已 ack 激活为前提，超时则回滚为双活并人工介入'）。状态重建统一走'从 PostgreSQL 最近快照 + 重放 Redis WAL 中 handoff 段'，而非多源仲裁。明确冻结期间输入的排队与丢弃策略（如战斗指令直接拒绝并提示）。

### 改进建议
- 立即重测迁移面并重排排期：以 68771 处箭头调用（而非 3450 处 environment）为真实工作量主线，先迁移 go/move/combat/channel 四条高频跨对象调用图作为阶段一里程碑，未覆盖子系统不进阶段二。
- 放弃自研虚拟 actor 运行式的从零路线，阶段一用 Ray actor（Python 原生、自带 placement/调度/容错）做最小可用原型，验证 LPC↔Actor 同构映射在真实负载下的延迟与消息序，再决定是否自研薄壳。不要在原型前承诺自研运行时。
- 把战斗确定性升级为 CombatContext 快照：战斗开始时快照全体参与者+可观察环境的 dbase/tmp_dbase，seeded RNG 仅是其一部分，回放从 CombatContext 起点重放。先抽样真实战斗导出 do_attack 读取的字段集以定义最小边界。
- 热路径去 Redis 化：会话/实体路由用一致性哈希环（shard=hash(entity_id)）无外部依赖解析，Redis 仅作控制面（placement 变更通告、handoff 暂存）；门/迁移强一致需求走 PostgreSQL 事务。明确 Redis 降级时游戏降级为仅本分片可玩而非全停。
- 区域监督默认改 one_for_one+持久化重建，而非 all_for_one；先确认是否存在真正区域级共享可变状态，多数区域状态实为 per-room/per-npc，很可能根本不需要 all_for_one。
- 阶段一同步立项差分测试基建：录制 LPC 命令流为 golden trace，Python 版逐条重放并 diff 输出帧与状态快照，先覆盖四条主线。无差分测试通过的子系统不进入阶段二。
- 补全 32 守护进程全量分类迁移表，逐个标注分类与外部化方式；securityd 先修第 757 行 typo 并评估权限模型是否需重建，将其列为 UGC 开放的前置阻塞项。
- 为 handoff 管线每一步定义超时、补偿与最终一致目标，状态重建统一走 PostgreSQL 快照+Redis WAL 重放而非多源仲裁，明确冻结期间输入的排队/丢弃策略。
- 容量上限改为'按并发玩家 actor 数'而非'房间数'约束（当前 6414 房远小于设计的 10 万房上限，房间数不是瓶颈）；先做原型实测单分片在 AOI 邻域全激活下的并发上限，再定分片数。
