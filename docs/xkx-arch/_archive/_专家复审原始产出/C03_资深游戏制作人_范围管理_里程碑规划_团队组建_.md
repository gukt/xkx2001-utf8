# 交叉质证·资深游戏制作人（范围管理/里程碑规划/团队组建/风险评估/商业化）

## 同意的论断
- **Greenfield 与增量重构的根本性概念混淆是承重级缺陷，'迁移适配层/双栈/过渡期'概念必须从路线图中清除**（资深游戏架构师）：作为制作人这是我最强烈的同意：这个混淆不只是措辞问题，它直接虚增了项目范围。'适配层/双栈/过渡期'是没有任何产品价值的工作--它不产出玩家可感知的功能，不产出可验证的资产，只为一个'逐步替换'的幻觉服务。我实测 04 文档第 59 行仍有'Ray actor 原型验证 LPC↔Actor 同构映射'作为阶段一硬约束、第 75 行有'逐子系统替换 LPC 行为'、§30 有'过渡期保留 LPC message_vision 经 Telnet 适配器作为真相源'--这些是必须删除的幽灵工作。清除后，68771 调用点的性质从'迁移对象'变为'按规格实现的需求来源'，这改变了整个工时估算的语义。
- **缺规格提取管线(spec extraction pipeline)，应从 LPC 源码静态提取形式化规格再据规格实现**（资深游戏架构师）：这条直接补齐了我'抽样校准实验'缺口的前提：你无法估算你没有规格化的东西。架构师说的'规格提取'和我说的'工时抽样校准'是同一洞察的两个面--他补了上游（先规格化才能抽样），我补了下游（抽样才能外推全量）。没有规格提取层，68771 调用点的迁移是'读懂 LPC 再凭直觉重写'的黑盒，每一点的工时方差极大（机械型 10 分钟 vs 设计决策型 1 天+），全量估算不可信。这是我同意的核心：规格提取是让项目从'不可估算'变为'可估算'的承重前置工序。
- **代词体系是 10 变量 7 视角（非方案所称 4 变量 3 视角），且 RANK_D 7 个函数是业务求值非纯渲染**（MUD/LPC 历史与社区专家）：我实测确认 emoted.c 注释列出 $N/$n/$P/$p/$C/$c/$R/$r/$S/$s 共 10 变量，rankd.c 确有 query_rank/query_respect/query_rude/query_self/query_self_rude/query_close/query_self_close 共 7 个动态求值函数。作为制作人，这暴露了一个被隐藏的范围：'渲染下沉前端纯函数'这条承重论断（00 总纲第 23 行三条不可妥协边界之一）在代词维度不成立。代词求值依赖年龄/性别/职业/门派/官职/善恶/鬼魂状态，是服务端业务逻辑。这意味着渲染层不是'下移前端'而是'服务端求值+前端替换'的分层--这是一个被错误简化为'纯前端'的子系统，其真实工作量被低估。隐藏范围=隐藏工时=里程碑失守。
- **5 个灵魂系统（阴间/武林大会/vote/法院/intermud）在 5 份文档中零提及，是 MUD 文化骨架非边缘玩法**（MUD/LPC 历史与社区专家）：我实测确认：d/death 17 文件、d/bwdh 276 个 .c 文件（远超专家所说的'40+ 房'）、cmds/std/vote + 2 子动作、d/wizard/courthouse.c、adm/daemons/network 41 文件全部存在且全部未出现在路线图的任何阶段。作为制作人这是最让我警觉的发现：路线图声称'32 守护进程全量分类迁移'（阶段3）但完全没有枚举这些 themed 社会治理系统。d/bwdh 一个系统 276 个文件--这比整个 d/city 试点（441 文件）小不了多少，却完全不在工时估算里。迁移面统计只数了'技术调用面'（68771 调用点），没数'文化系统面'。这是系统性范围遗漏。
- **引擎工具链（场景编辑器/实体检视器/tick 回放器/profiler/AI 调试器）完全缺失，是标杆引擎与玩具的分水岭**（游戏引擎资深专家）：作为制作人我再加一层：这不只是技术缺口，是团队扩展瓶颈。8400 文件迁移的调试，没有 inspector 和 replay scrubber，只能靠核心作者 printf 调试--这意味着调试能力无法分摊给协作者，核心作者成为单点故障。单人项目最大的风险不是'技术不够好'而是'只有一个人能推进时任何中断（生病/倦怠/离职）都是致命的'。工具链是让项目从'单人可推进'变为'团队可分摊'的前提，应作为阶段0一等公民而非 backlog。
- **四层 DSL 无可视化编辑器，门槛高于 Roblox 的 Lua+Studio，非程序员中文 MUD 爱好者无法上手**（UGC 平台产品专家）：这是与我'垂直切片'建议最强协同的论断。我的核心论点是'核心产品价值（DSL+Agent 创作闭环）要到阶段 4-5 才验证'，UGC 专家补了'即使验证了，创作门槛可能依然太高'这一层。四层 DSL（YAML+事件规则+Ink 对话树+受限 Python）要求创作者同时理解声明式数据、事件规则表达式、Ink 的 knot/stitch/divert 语义、Python 脚本--这确实比 Roblox Lua+Studio 高一个数量级。可视化编辑器不是'锦上添花'而是'创作能否发生的物理前提'。
- **'以 DSL 为契约收敛 LLM 非确定性'被过度声称，能收敛的只有结构合法性+数值分布，收敛不了叙事连贯/对话死锁/任务绕过/经济平衡/趣味性**（AI/Agent 工程资深专家）：作为制作人这是我最担心的'假阳性门禁'：如果 Agent 产出通过了 world-graph+combat-sim 验证就被当'可验证终态'发布，但实际叙事/平衡性不达标，等于用易验证指标冒充内容质量已验证。这比没有验证更危险--制造 false confidence 会让团队在错误信号上继续投入 14 个月。这条直接强化了我'阶段-1 垂直切片'的紧迫性：Agent 产出质量必须在投入引擎重构前就用真实内容验证，不能等到阶段 5 才发现'验证通过了但内容不好玩'。

## 挑战的论断
- **纯 Python 无法成为标杆引擎，必须 Rust 核心(PyO3)+Python 脚本混合架构**（游戏引擎资深专家）
  反对：Rust 核心在技术指标上正确（GIL/GC/确定性三重天花板成立），但作为制作人我必须挑战其落地的团队/人才维度：实测 git 历史只有 3 作者近期实质单人，'公司探索项目'背景下能否招募/留住 Rust 游戏引擎开发者是现实约束。Rust+Python 双语言栈使人才池缩窄一个数量级，且 PyO3 绑定层本身是维护负担。正确做法不是'否决 Rust'而是'用数据决策'：架构定型前先做单次 do_attack 的微基准（纯 Python vs Rust PyO3 原型），同时评估 Python 3.13t free-threaded(PEP 703)的成熟度。语言选型应作为阶段0的正式 go/no-go 决策点，附带人才可得性评估，而非由引擎专家单方面技术裁定。
  理由：引擎专家从纯技术角度得出的'必须 Rust'结论，忽略了制作人的核心约束：项目存活取决于团队能否持续推进。引入 Rust 是范围×人才的双重乘数风险，可能比 GIL 更快杀死项目。但同时我承认纯 Python 不做基准就承诺'2000-3000 并发'是乐观假设，引擎专家的'先测后定'方向正确。
- **Agent NPC 作为游戏内实体（LLM 驱动实时对话/涌现叙事）是真正的差异化能力，被严重低估，应立项**（资深游戏主策划）
  反对：作为制作人我强烈反对将其纳入近期范围。Agent NPC 是比离线 DSL+Agent 创作更大的未验证假设：它把 LLM 放进游戏实时循环，引入延迟（LLM 推理 100ms-数秒）、成本（每对话一次 API 调用）、一致性（LLM 幻觉破坏世界状态）三重实时风险。主策划说的对--它是差异化能力，但错在优先级：它应该是一个'远期 could-have'（阶段 6+），不是 MVP 组成。把一个更大的未验证假设塞进一个已经有最高风险假设（DSL+Agent 创作闭环）未验证的项目，是典型的'技术上完美但永远做不完'。先证明离线 Agent 创作可用，再谈实时 Agent NPC。
  理由：主策划从游戏设计角度看到 Agent NPC 的革命性，但制作人的职责是控制范围顺序。未验证的实时 LLM 游戏内集成是成本和风险都极高的特性，在核心闭环（离线创作）验证通过前不应进入范围。这是设计愿景与交付纪律的冲突。
- **intermud 联邦协议应保留并演化为 UGC 多世界互联的差异化能力**（MUD/LPC 历史与社区专家）
  反对：联邦是 MUD 文化基因没错，但作为制作人这是典型的范围陷阱：你在为一个还不存在的受众建设基础设施。UGC 平台在阶段 4 才开放，届时有几个世界需要互联？多世界联邦是'有 100 个活跃世界后才有意义'的能力，阶段 0-1 设计联邦协议是为 0 个用户建设。正确做法：在 EntityAddr 设计中保留 world:// 扩展性（廉价），但明确将联邦通信列为'阶段 6+ 未来选项'，不分配实现工时。MUD 文化保真不等于照搬 MUD 的所有历史特性。
  理由：历史专家从文化保真角度有价值，但制作人的边界是：文化保真的优先级必须服从项目存活。intermud 联邦是高工程量低近期价值的功能，在没有多世界存量时建设是资源错配。协议预留（设计文档级）几乎零成本，实现则要分配稀缺的工程工时。
- **创作者经济（分账/DevEx/付费 CPK）应从 Day 1 设计**（UGC 平台产品专家）
  反对：我同意'预留 schema 字段'（author_id/revenue_share/price 字段加进 manifest 几乎零成本），但强烈反对'阶段4交付市场 MVP+分账'。制作人的铁律：不要在没有内容的地方建市场。UGC 平台在阶段 4 才开放创作，届时有多少创作者？多少可付费内容？Roblox 的 DevEx 是在有数百万创作者后才建立的。提前建支付/分账基础设施是为空仓库装收银台。正确路径：(1) Day 1 预留 manifest 字段；(2) 阶段 4 只做'安装量+评分'排行（最小分发）；(3) 有了 50+ 活跃创作者后再建分账/付费。商业化的节奏必须跟内容存量走，不能跑在内容前面。
  理由：UGC 专家从平台繁荣角度正确地指出创作者经济是飞轮核心，但制作人的挑战是时机：创作者经济是'做了才繁荣'的能力，但在创作者和内容存量不足时提前建支付基础设施是沉没成本。区分'预留数据结构'（廉价，做）和'建交易系统'（昂贵，缓）。
- **全仿真确定性应作为引擎一等能力，所有 System 的 RNG 走 seeded DeterministicRNG + input log + snapshot**（资深游戏架构师 + 游戏引擎资深专家 + 分布式系统资深专家）
  反对：作为概念我同意确定性是标杆能力，但挑战其时机：全仿真确定性（所有 System RNG seeded + 全局 input log + 任意 tick snapshot 回放）是巨大工程量，涉及每个 System 的改造。阶段 0-1 应只做战斗级确定性（已有 CombatContext+seeded RNG），这是正确的 MVP 范围。全仿真确定性应是'设计时预留架构钩子（所有 RNG 接口可注入 seed），实现时分阶段覆盖'，而不是阶段 0 的交付门禁。把全仿真确定性列为 Phase 0 一等公民会让 Phase 0 进一步过载（它已经装不下 spec extraction+FluffOS+垂直切片+工具链+灵魂系统盘点）。
  理由：架构师和引擎专家从标杆完整性角度正确，但制作人的约束是阶段交付能力。'设计时预留'与'实现时交付'是两回事。确定性是正确目标但错误的阶段0门禁--它会阻塞所有后续工作。战斗级确定性作为 MVP 已足够验证方向。
- **LangGraph 是过渡原型，中期应迁移到 Temporal/DBOS durable execution**（AI/Agent 工程资深专家）
  反对：AI 专家正确指出 LangGraph 不适合长时创作工作流，但'迁移到 Temporal'是引入又一个重型基础设施承诺。制作人的视角：这又是'显得先进而非解决痛点'的选型。MVP 阶段甚至不需要 LangGraph--直接用 asyncio + Claude SDK + PG 持久化的最小状态机就够。引入 Temporal 是为'多创作者并发'这个尚未存在的需求建设。正确路径：(1) MVP 用裸 asyncio+PG（零框架债）；(2) 验证创作闭环可用后，按实测瓶颈决定是否需要 durable execution 框架。不要在验证假设之前就承诺基础设施。
  理由：AI 专家从工程严谨角度正确，但引入 Temporal 与项目'不要为不存在需求建设'的核心原则冲突。先验证创作闭环是否成立（最小 stub），再决定编排引擎。LangGraph 和 Temporal 都可能是过度选型。
- **多题材共享运行时降低创作成本是优点，武侠为旗舰深耕其余为 UGC 扩展**（资深游戏主策划）
  反对：主策划的'武侠旗舰'方向对，但制作人必须指出：文档的 ThemeRegistry 假设'核心主题无关'，而 CombatSystem/Vitals/Condition 的设计直接从武侠 do_attack 反推，带着经脉/内力烙印。这不是'共享运行时+题材皮肤'而是'武侠引擎+题材适配'。制作人的挑战是范围诚实：不要声称'通用文字世界引擎'却建一个武侠引擎。应在阶段-1 垂直切片就用一个非武侠题材（如书院的最简对话/课业）验证引擎内核是否真的主题无关，否则'多题材'承诺会在阶段 5 才暴露为需要大量重构。
  理由：主策划从设计深度角度正确，但制作人的补充是：多题材的'共享性'必须在投入武侠深度迁移前就用非武侠题材验证，否则会在 14 个月后发现内核不可泛化。这是把'假设'变'验证'的执行纪律。

## 新涌现缺口
- 路线图内部存在未解决的阻断性矛盾：04 文档第 59 行将'Ray actor 原型验证 LPC↔Actor 同构映射'列为阶段一硬约束，但架构师和分布式专家都明确建议'阶段一完全不引入 Ray，用纯 asyncio'。三个来源对同一阶段给出互斥的架构指令。作为制作人这是阻断项：阶段一到底用什么？必须在进入阶段一前敲定单一答案，否则会做一半推倒重来。
- 灵魂系统的迁移范围在路线图中完全缺失：d/bwdh 276 文件、d/death 17 文件、vote 自治系统、courthouse 审判、intermud 41 文件--这些 themed 社会治理系统在 04 文档的阶段 2-3 的'32 守护进程全量分类迁移'里根本没有枚举。迁移面统计只数了 68771 调用点（技术面），没数灵魂系统（文化面）。这是一个独立的范围缺口，可能新增数百文件工作量。
- 整个评审面板（含我）都是工程师视角，无人 scoped 实际的游戏设计工作：写对话文案、平衡战斗数值、设计任务链、创作分级文案池。8400 文件迁移是'重建既有内容'，但 UGC 平台和 Agent 创作需要'原创内容设计'能力--这是编剧/游戏设计师的工作，不是工程师的。垂直切片的内容设计稿（5-10 房间+2NPC+1战斗+1任务+1对话）需要制作人/编剧视角，但没有任何专家把它列为交付物。这是 all-engineer 评审的结构性盲区。
- 阶段 0 严重过载：架构师要 spec extraction、引擎专家要工具链、我要垂直切片、MUD 专家要灵魂系统盘点、还要 FluffOS Linux 编译+securityd 分析。这些无法全部塞进 1-2 月。需要制作人的残酷优先级裁决：哪些是'不做则后续全错'（spec extraction + 垂直切片验证），哪些可后置（工具链、灵魂系统盘点）。
- '标杆资产早期独立交付'被三位专家从不同角度提出但未统一：我说确定性回放引擎可独立开源（3 月）、架构师说引擎核心应开源建生态、引擎专家说工具链应一等公民。这是同一个洞察（先发可被外部感知的独立资产建影响力）的三种表达。应统一为一个'早期可交付资产路线图'，与 8400 文件迁移解耦。
- 现有侠客行玩家社区在多年构建期内的迁移策略完全缺失：是迁移老玩家、还是面向新受众？这决定 Telnet 兼容层是长期一等公民还是过渡产物（MUD 专家指出文档把它当过渡会低估维护成本），也决定 go-to-market 路径。无任何专家讨论'建造期间现有社区怎么办'。
- d/city 试点目的被定义为'校准迁移工时'，但无人设计'工时抽样校准实验'的具体方法论：按难度分层抽多少样本、如何定义难度分层、如何回归出工时模型。这是统计学问题，不是工程问题，需要实验设计。

## 精炼立场
综合交叉质证后，作为制作人我的精炼立场如下，按优先级排序：

第一，项目存活取决于一件事：以最低成本验证核心价值主张（DSL+Agent 能否让非程序员创作可玩世界）。这不是工程问题而是产品假设问题。当前路线图把这个最高风险假设放在阶段 4-5（14+ 月后）才验证，前面全是工程可行性验证--这是'把最高风险放最后'的顺序错误。必须在阶段 0 之前插入'阶段-1 垂直切片平台验证'（2-3 月）：5-10 房间+2NPC+1 战斗+1 任务+1 对话，全部 DSL 定义+部分 Agent 生成，跑在最小 stub 运行时上，验证三件事：DSL 能否表达、Agent 产出是否可用、玩家是否觉得好玩。通过后才投入引擎重建。这条与 UGC 专家（可视化编辑器是创作前提）、AI 专家（验证覆盖度幻觉）、主策划（战斗手感等价）的发现强协同：核心价值必须先验证。

第二，greenfield 定位澄清（架构师）是不可谈判的前提，必须立即应用到路线图。我实测确认 04 文档仍充斥'迁移适配层/双栈/过渡期/逐步替换/修 securityd 旧 bug'等增量概念。这不是编辑工作而是工作定义变更：清除后，68771 调用点从'迁移对象'变为'按规格实现的需求来源'，securityd 从'修旧 bug'变为'从规格提取权限模型后从零实现'。这条直接影响工时估算的语义基础。

第三，规格提取管线（架构师）是让项目从'不可估算'变为'可估算'的承重前置工序，与我的'工时抽样校准'是上下游关系。二者合并为阶段 0 的第一项真实工作：先规格化（提取形式化契约），再抽样（按难度分层实测工时），最后外推全量。不做这一步，14-24 月估算不可用于严肃承诺。

第四，'标杆资产早期独立交付'需要统一为一条交付线。三位专家（我说确定性回放开源、架构师说引擎开源建生态、引擎专家说工具链一等公民）从不同角度提出同一洞察：先发可被外部感知的独立资产建影响力。应整合为：M1 确定性战斗回放引擎（3 月，独立开源组件，全行业缺的反作弊/审计基建）；M2 垂直切片证明 DSL+Agent 创作闭环（2-3 月）；M3 单题材可玩 demo（6-8 月）。三者都与 8400 文件全量迁移解耦，各自可独立交付并获社区反馈。'业界标杆'靠被使用和被引用建立，不靠代码量大。

第五，性能问题（Rust vs Python / GIL / 容量）必须用微基准回答而非断言。引擎专家和分布式专家都同意'先测后定'。作为制作人我坚持：单次 do_attack 微基准+单 tick 不同实体规模 p99+单 Player Actor 事件环延迟，三者作为架构定型前的 go/no-go 门禁，同时附带人才可得性评估（Rust 开发者能否招募）。语言选型是阶段 0 正式决策点。

第六，灵魂系统（MUD 专家）的文化保真维度必须成为 tracked metric 而非 afterthought。我实测 d/bwdh 276 文件证实这些系统规模远超文档预估。路线图的'32 守护进程全量分类迁移'根本没有枚举阴间/武林大会/vote/法院/intermud。这既是范围遗漏也是文化保真缺口。成功指标必须增加'灵魂系统迁移完整率''themed 文本保留率'，与差分测试通过率并列为一等指标。

第七，每阶段必须加 kill criteria。当前所有'决策检查点'都是'继续推进'条件，无一'何时停止/转向'标准。多年期单人项目最大风险是'永远做不完'。我要求：阶段 2 若 6 个月内 Combat 迁移未通过差分测试，则放弃逐调用点忠实迁移，转向行为等价+原生新内容；项目级止损若 18 个月内未达成单题材可玩 demo，冻结迁移聚焦已迁移内容产品化。

最后回应'业界标杆'定位：真正能让它成为标杆的不是'把 LPC 忠实迁到 Python'（这是考古不是产品），而是三个差异化能力的闭环：(1) DSL-as-contract 把 LLM 非确定性收敛到可验证终态（AI 专家指出当前被过度声称，需补语义/叙事/平衡三维评估矩阵）；(2) 确定性仿真+时间旅行调试（架构师指出需从战斗级扩展到全引擎，但作为分阶段目标而非阶段0门禁）；(3) 引擎开源+创作者生态（UGC 专家指出可视化工具+创作者经济+审核是飞轮核心，但经济实现需跟内容存量走）。这个闭环应成为 MVP 北极星，而非埋在 8400 文件迁移之后。

## 开放问题
- DSL+Agent 创作闭环能否产出可玩内容？这是项目最高风险假设，14+ 月后才验证等于把它放在最后。必须在阶段-1 垂直切片（2-3 月）用最小 stub 验证，否则无法判断是否值得投入引擎重建。
- 68771 调用点的真实工时分布是什么？需按难度分层（机械型/理解型/设计决策型）抽样 50-100 个实测，回归工时模型。不做抽样校准，14-24 月估算不可信，可能偏差一个数量级。
- FluffOS 能否在现代 Linux(WSL2) 编译运行？driver 实测是 macOS Mach-O 二进制。若编译受阻，golden trace 录制无法进行，差分测试 P0 门禁阻塞。需在阶段 0 第一步验证（保守估 2-4 周）。
- Rust 核心(PyO3)的人才可得性如何？这是个人项目+公司探索项目，能否招募/留住 Rust 游戏引擎开发者决定语言选型可行性。技术最优不等于项目可交付。
- 灵魂系统的真实迁移面有多大？d/bwdh 一个系统 276 文件，d/death/vote/courthouse/intermud 合计未统计。这些 themed 治理系统在路线图中零枚举，可能新增数百文件工作量。
- 阶段一的 Actor 运行时到底用什么？路线图说 Ray、架构师说纯 asyncio、分布式专家说分片内同步分片间异步。三个互斥答案，必须在进入阶段一前敲定唯一路径。
- 多题材共享内核是否成立？CombatSystem 从武侠 do_attack 反推，是否真的主题无关？应在阶段-1 用非武侠题材（如书院对话）验证内核可泛化性，否则'多题材'承诺在阶段 5 才暴露为需大量重构。
- 项目级 kill criteria 是什么？所有专家的'决策检查点'都是'继续推进'的条件，无任何'何时停止/转向'标准。多年期单人项目最大风险是'永远做不完'，需为每阶段定义止损（如'阶段2 若 6 个月内 Combat 迁移未通过差分测试，则放弃逐调用点忠实迁移，转向行为等价+原生新内容'）。
