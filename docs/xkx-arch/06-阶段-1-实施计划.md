## 一、第一切片 S1 的目标与范围

### 目标

端到端跑通阶段 -1 的两条核心链路，证明核心假设成立：

1. **DSL 表达力链路**：层0（房间/NPC 数据）+ 层1（事件规则）能定义一个最小可玩场景，编译到 JSON IR，被运行时加载。
2. **resolve_attack 纯函数链路**：从 LPC `do_attack` 七步提取出无副作用纯函数，seeded RNG 可重放，副作用按交织顺序记入账本。

两条链路在 stub 运行时里汇合：玩家能 `go` 移动（触发 `valid_leave` 规则）+ 能 `kill` NPC（触发 `resolve_attack` 跑一回合）。

### 范围（推荐 B）

| 维度 | S1 含 | S1 不含（后续切片） |
|---|---|---|
| resolve_attack | dodge/parry/hit 三分支 + 副作用账本 + seeded RNG + exp/damage effect | hit_ob/hit_by 的 mapping 分支、riposte 递归、武器类型完整集、技能 action 描述完整 |
| DSL 层0 | RoomDef + NpcDef 最小 schema + IR 编译 | 技能/物品/门 schema、字段穷举 |
| DSL 层1 | valid_leave 事件 + 条件阻断规则 + deny-wins | accept_object/chat_msg/init 等其他钩子、priority 复杂排序 |
| 运行时 | 最小 ECS（dict 存储）+ 6 组件 + go/kill 两命令 + 场景加载 | SparseSet、8 段管线、1s tick 调度、存储层、认证 |
| 场景 | 2 房间 + 1 战斗 NPC + 1 valid_leave 规则 | 5-10 房间、任务、对话、多 NPC |
| Agent | 不含（S3） | LLM 生成 DSL、copilot 修订量度量 |
| 非武侠验证 | 接口设计留口，不实跑（S2） | CombatKernel 非武侠微场景硬门禁 |
| 玩家试玩 | 不含（S5） | 3-5 名试玩 |

> **备选范围**：A（更小，仅 resolve_attack 最小版 dodge/hit + 层0 + 端到端，1-2 session，先证链路通）/ C（更大，B + hit_ob/hit_by + riposte + 非武侠，一次到位但风险高）。推荐 B：是一个有意义的完整闭环，验证阶段 -1 两条核心链路，又不致过大。

## 三、实施步骤

1. **resolve_attack 纯函数 + seeded RNG**（最核心最难，先做）
   - `CombatContext` / `CombatRoundResult` / `Effect` 数据模型（pydantic 或 dataclass）
   - `DeterministicRNG`（注入 `random.Random(seed)`，封装 do_attack 的 18 处 random 用途）
   - `resolve_attack(ctx) -> CombatRoundResult`：七步提取，dodge/parry/hit 三分支；hit_ob/hit_by 先做 int 加成/覆盖（mapping 分支后置）；riposte 标记不递归
   - 副作用全部产出为 `Effect`（DamageEffect/ExpEffect/JingliEffect），按交织顺序入账本，**不直接 mutate**
   - hypothesis 测试：同 seed + 同输入 -> 同输出

2. **层0 schema + IR 编译**
   - `RoomDef` / `NpcDef` pydantic v2 模型（字段参照 chaguan.c / bing.c）
   - YAML 加载（PyYAML）-> pydantic 校验 -> IR dict
   - 测试：加载 wuxia_micro YAML -> IR 无误

3. **层1 事件规则 + 薄求值器**
   - `EventRule`（event/condition/action/priority）
   - 最小谓词集：`attr_lt`（属性小于）/ `present_npc`（NPC 在场）/ `has_flag`（玩家标记）
   - 薄求值器：求 condition -> deny-wins 返回 allow/deny
   - 测试：yamen 型（守卫在场+无标记则 deny）能正确求值

4. **最小 ECS + 组件 + 场景加载**
   - 6 组件 dataclass；实体用 `entity_id -> {comp_type: comp}` dict（SparseSet 后置）
   - `world.load_ir(ir)`：RoomDef/NpcDef -> 实体 + 组件
   - 玩家实体 stub

5. **命令管线最小版**
   - `go(dir)`：查 exits -> 求 valid_leave 规则 -> deny 则阻止+消息，allow 则改 Position
   - `kill(target)`：进入战斗态 -> 调 `resolve_attack` -> apply effects 到组件 -> 产消息

6. **最小场景数据**
   - rooms.yaml：茶馆 + 街道，exits 互连
   - npcs.yaml：1 官兵（attitude heroism，可战斗）
   - rules.yaml：1 valid_leave 规则（如"未满 18 岁不得离开街道"，仿 duchang）

7. **端到端测试 + 修 bug**
   - 加载 wuxia_micro -> 玩家 go（触发 valid_leave）-> kill 官兵 -> resolve_attack 一回合 -> 断言伤害/消息/effect 顺序

8. **收尾**
   - 更新 [PROGRESS.md](../../PROGRESS.md)：S1 完成，标 S2 入手点
   - 写 ADR-0002：resolve_attack 提取决策（哪些步做了/简化了/为何，关联 05 dissent）

## 四、验收标准（对应阶段 -1 子集）

- [ ] `resolve_attack` 同 seed + 同输入确定性输出（hypothesis 通过）
- [ ] 副作用按"文本与状态交织真实顺序"记入账本，代码中无"先算后批量 apply"（测试断言 effect 顺序）
- [ ] 18 处 random 全部走注入 RNG，无裸 `random()`
- [ ] 层0 YAML 能定义 2 房间 + 1 NPC，编译 IR 无误
- [ ] 层1 valid_leave 能表达条件阻断规则，deny-wins 生效
- [ ] 端到端：go 移动 + valid_leave 阻断/放行 + kill 触发 resolve_attack 一回合 + 伤害/消息/effect 正确
- [ ] 代码不引入已砍复杂度（无分布式/PG/Redis/Rust/uvloop；combat 确定性 combat-only）
- [ ] ruff lint/format 全过，pytest 全绿

## 五、风险与对齐（关联 [05](05-第三轮专家对抗复审报告.md) dissent）

| 风险 | 对齐 | S1 应对 |
|---|---|---|
| CombatKernel 抽象时机张力（dissent 1） | 从武侠提取 vs 非武侠验证 | resolve_attack 接口不硬编码武侠语义（武器类型用枚举传入，"经脉/内力"等武侠概念不进核心签名），为 S2 非武侠验证留口；但 S1 不实跑非武侠 |
| 七步副作用交织（dissent / 01 子系统5） | 不得"先算后 apply" | Effect 账本严格按交织顺序，测试断言顺序 |
| 18 处 random（combat 确定性） | seeded RNG | 全部注入，hypothesis 验证可重放 |
| 层1 原语蠕变（dissent 3） | KPI <15% | S1 谓词集严格限定 3 个，扩充需 ADR |
| 阶段 -1 kill criteria 1（DSL+Agent 闭环） | DSL 表达力 | S1 先证层0+层1 能表达最小场景；若不能，触发扩原语（先于 Agent 投入） |
