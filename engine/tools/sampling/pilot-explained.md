# 为什么要花 14 小时只测 13 个函数？

> 一次 MUD 重构里的"抽样校准"到底在做什么，为什么它这么折腾，以及它到底值不值。
>
> 本文是面向"完全不了解这个任务"的科普说明，不讲代码细节，只讲清楚来龙去脉、目标、好坏、为什么复杂。

## 一句话先说结论

我们要把一个有 8412 个源文件、近 7 万处函数调用的经典 MUD 从零重写成 Python。在动手大规模迁移之前，先用 13 个代表性函数做一次"工时抽样"，量出真实的工作量，给整个项目一个**有数据支撑的工时承诺**--而不是拍脑袋。

## 一个让人困惑的开场

站在这件事的外面看，会觉得荒唐：

- 项目要重写的东西有 **8412 个 LPC 文件**、**6414 个房间**、**21 个门派**；
- 扫描出来的同步函数调用点有 **近 7 万处（68771）**；
- 而我们现在的"阶段 0 验收硬交付物"，居然是 **花 14–20 小时，只迁移 13 个函数**。

14 小时，13 个函数，对比 7 万处调用点--这不是杯水车薪吗？这 14 小时到底在干什么，为什么不直接开干？

答案藏在一个很朴素的工程问题里：**这堆东西，到底要迁多久？**

## 问题：盲盒式重写的陷阱

重构一个老系统，最怕的不是"难"，而是"不知道有多难"。

侠客行是一个 20 多年的 LPC MUD。我们要把它从零重写成 Python（不是在老代码上修修补补，而是按规格重新实现）。这叫 greenfield 重写。

greenfield 重写有一个绕不开的问题：**老板、评审、自己都要问一句--这要多久？**

这个问题如果答不好，有三种死法：

1. **严重低估**：以为 3 个月能干完，结果 3 个月只迁了 10%，里程碑崩盘，团队士气崩塌。
2. **严重高估**：报 3 年，结果资源被砍、项目搁置，或被别的优先级插队。
3. **平均外推失真**：拿"平均一个函数 X 分钟"去乘总数--但工作量分布严重不均，平均值毫无意义（下面会讲）。

所以我们需要一个**有数据支撑的工时承诺**。不是精确到小时，而是一个实测出来的"区间"。

这就是"抽样校准"要解决的问题。

## 类比：老城改造怎么做预算

想象你要给一座有 8000 多栋老房子的老城做现代化改造，市长要你报一个工期和预算。

你有三种做法：

- **挨栋量**：8412 栋，每栋请勘测队量一遍--等你量完，市长都换届了。
- **拍脑袋**：凭感觉说"大概两年"--预算崩了你要负责。
- **抽样**：先做两件事--
  1. 先勘测哪些已经是现代标准、不用动；
  2. 剩下要动的，按房型（住宅/商铺/工厂）分层，每类挑几栋实测，再乘以各类的总数，得出全城预算。

抽样校准做的就是第三件事。在这个项目里：

| 老城改造 | 侠客行重构 |
|---|---|
| 老城 | LPC MUD（8412 文件） |
| 水电接线点 | 函数调用点（近 7 万） |
| 已是现代标准不用动 | 新引擎已实现等价行为 |
| 房型分层 | 难度分层（高/中/低 + 逻辑/数据） |
| 抽几栋实测 | 抽 13 个函数实测工时 |
| 乘总数报预算 | 分层外推全量工时承诺 |

## 让事情变复杂的三个真相

到这里，"抽样"听起来挺简单。但真做起来，有三个真相让方案变得不那么直接。

### 真相一：根本不是全城都要重建，只有 18%

这是最关键的一个发现，也是整个方法论被"修正"过一次的原因（[ADR-0047](../../../docs/adr/ADR-0047-greenfield-effort-semantics.md)）。

一开始的想法是：7 万处调用点，抽 80 个实测，外推全量。但仔细一查，新引擎其实已经实现了系统级的等价行为：

- 数据读写（dbase）--已有
- 技能系统--已有
- 战斗--已有
- 装备、条件、移动--已有

这六大类合计 **48848 处调用点（82.4%）** 的等价行为**新引擎已经实现了**，工时约等于零。真正待迁移的只有 **10422 处（17.6%）**，分属 1159 个函数。

如果对全 7 万处抽样实测，会大量重复测"已经做完的东西"，工时被严重高估。所以抽样面要从 7 万缩到 1 万多，只测真正没做的部分。这叫 **greenfield 工时语义**：工时只计"还没实现的"。

### 真相二：难度天差地别，平均值毫无意义

待迁移的 1159 个函数里，工作量分布严重不均：

- 有的函数就是一堆 `set()` 数据定义（NPC 名字、房间描述），5–15 分钟搞定，能批量。
- 有的函数像 `xue.c:main`（学艺命令），135 行、20 个分支、53 个调用点，包含随机数、跨对象状态修改、事务性回滚，**单个就要 2–3 小时**。

如果用"平均一个函数 X 分钟"乘 1159，结果会被那些巨型函数拉偏，完全失真。所以必须**按难度分层**，每层单独估，再加权求和。

这也是为什么 pilot 要"high-tier 过采样"：变异最大的那一层（难函数）是估算误差的主要来源，多测几个才能压住方差。

### 真相三：勘测表本身会骗你（所以要纠偏）

我们有一张"分类表"，标注每个函数是"已实现 / 待迁移"、"逻辑 / 数据"、"高 / 中 / 低难度"。但调研发现这张表**有系统性偏差**：

- 有的标了"已实现、工时≈0"的函数，其实还差好几个后置分支没做，实际要 7 分钟（不是 0）。
- 有的标了"数据类、5 分钟"的函数（比如 `snake.c:init`），打开一看里面有战斗触发和新命令，其实是逻辑类，要 30–45 分钟。

如果直接拿这张偏差表去外推，工时承诺就是错的。所以 pilot 特意**塞进几个"纠偏样本"**（比如 `snake.c:init`、`death_penalty`），用来测量"分类到底偏了多少"，偏太多就回调分类规则、重新分层。这叫**定量审计纠偏**（[ADR-0048](../../../docs/adr/ADR-0048-stage-b-degraded-interval-pilot.md) 决策 3）。

## 为什么不测 80 个，只测 13 个？

最早的方案（[ADR-0047](../../../docs/adr/ADR-0047-greenfield-effort-semantics.md)）确实是测 80 个函数，还要算窄置信区间。但启动前调研 + 一个三方架构评审团发现三个问题：

1. **80 个实测要 80–100 小时、10 多个 session**。光是 high-tier 那 58 个就约 76 小时。
2. 验收标准其实只要求"工时承诺有数据支撑"，**不要求窄置信区间**。窄 CI 是自加码。
3. 80 个对"点估总工时"也过宽，40–50 个其实够了。

于是方案被**降级**（[ADR-0048](../../../docs/adr/ADR-0048-stage-b-degraded-interval-pilot.md)）：

- 从"80 样本窄 CI"降为"**13 样本 pilot + 类比基准的区间承诺**"。
- 工时从 80–100 小时降到 **14–20 小时 / 2–3 个 session**，省掉 75–85%。
- 留一条退路：如果测下来分类偏差太大，再升级回分批实测。

省下来的 60–80 小时留给主线（外部玩家测试、PG 迁移、第二题材）。这是评审团的核心判断：**该砍的过度工程要砍，省下的时间花在更值钱的地方。**

## 为什么 AI 只能"铺路"，不能"代跑"？

这可能是最反直觉的一点。

我们手里有 Claude 这样的 AI，让它把 13 个函数的 Python 代码写出来，可能一两个小时就跑完了。那为什么 pilot 还要 14 小时？为什么要"AI 铺路 + 人工计时"？

因为 **工时承诺问的是"人迁完需要多久"，不是"AI 跑完需要多久"**。

如果让 AI 写代码，人只负责校验和计时，那记下来的工时是"人校验 AI 代码的时间"，不是"人从头迁移的时间"。这两个数差太多了。而且 LLM 的质量会随模型版本、prompt 漂移，今天测的工时明天就不可复现，更没法外推到"将来不用这个 LLM"的真实迁移场景。这叫**工时语义污染**。

评审团三方一致否决了"AI 辅助双层工时模型"，正是守住这条红线（[ADR-0048](../../../docs/adr/ADR-0048-stage-b-degraded-interval-pilot.md) 决策 7）。

所以 AI 在 pilot 里的合法角色被严格限定：

- ✅ **铺路**：提前建好多个函数共用的"桩"（占位实现，比如配偶判断、拜师认可）、写推算脚本、做"哪些 API 已有/部分/没有"的对照导航、写最终报告。
- ❌ **代跑**：替人写迁移代码、写测试、替人读规格计时。这些必须人工做，工时才真实。

一个具体发现印证了这个设计有多必要：`xue.c:main` 这个函数，新引擎**其实已经有一个简化版的 `learn()`**（覆盖主流程），pilot 真正要测的是"补齐那 8 个被后置的分支到行为等价"。正因为补全比从零写简单，AI 代劳的诱惑更大--也正因如此，更要守住"人工计时"的口径，否则工时承诺会系统性偏低。

## 有这个系统 vs 没这个系统

| | 没有抽样校准 | 有抽样校准 |
|---|---|---|
| 工时承诺 | 拍脑袋，或拿平均数瞎乘 | 有 13 个实测锚点 + 类比外推的区间 |
| 迁移面大小 | 以为 7 万处全要迁 | 量化出只有 17.6% 真待迁 |
| 难度分布 | 以为都差不多 | 知道 high-tier 单函数 2–3h、data 类 5 分钟 |
| 分类准确性 | 盲信分类表 | 实测纠偏，偏差超 30% 就回调 |
| 里程碑 | 容易过载或崩盘 | 有数据支撑，可对外承诺 |
| 工程量投入 | 要么不投（盲盒）要么全投（80–100h） | 14–20h 拿到结论，省 75% |

一句话：**没有它，整个项目就是在盲盒里赶工期；有了它，至少知道这个盲盒有多大、哪儿最难。**

## 现在到哪了，下一步

脚手架（pilot 的"工具台"）已经搭了一半：

- ✅ 工时记录格式（[schema.py](pilot/schema.py)）、操作手册（[pilot/README.md](pilot/README.md)）、推算脚本骨架（[estimate.py](pilot/estimate.py)）、扫描统计报告（[report.md](report.md)）都写好了。
- ❌ 共享桩（5–7 个，见 [pilot/stubs.py](pilot/stubs.py)）还没建、推算核心函数没实现、13 个样本的代码和工时记录（[effort_records.jsonl](pilot/effort_records.jsonl)）都还是空的。

下一步（待拍板方向）：

1. 开 `feat/sampling-pilot` 分支。
2. AI 并行铺路：13 个函数各做一份"已有/部分/没有"对照，汇总后一次性建好共享桩，实现推算脚本。
3. 人工计时：从 `xue.c:main` 起步，逐个迁移 + 记工时。
4. 推算区间承诺，写报告，校验分类偏差。

整个 pilot 端到端约 **14–20 小时 / 2–3 个 session**。AI 铺路部分约 3 小时（可藏在第一个 session 里），**真正的瓶颈是人工计时那 14 小时**--而这个瓶颈无法用 AI 绕开，因为绕开就毁了工时承诺本身的意义。

## 收尾

回到开头的荒诞感：14 小时测 13 个函数，对比 7 万处调用点，听起来像杯水车薪。但这 14 小时不是在"迁移 13 个函数"，而是在"校准一把尺子"--用这把尺子，去量出剩下 1 万多处调用点、1159 个函数的真实工作量。

这是工程估算里最朴素也最容易被跳过的一步：**在承诺一个庞大工程之前，先实测一小撮，校准你的直觉。** 跳过它的人，要么低估到崩盘，要么高估到搁置。我们选择花 14 小时，换一个敢对外承诺的数字。
